## 3. 权限准则（血训）

### 3.1 Android 13+ 通知权限（POST_NOTIFICATIONS）

**从 Android 13（API 33）开始，所有通知（包括前台服务通知）必须同时满足以下两个条件：**

1. **静态声明**：插件的 `AndroidManifest.xml` 必须显式声明：
   ```xml
   <uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
   ```
   缺少此声明时，系统会在设置中将应用的通知开关置为**灰色不可交互状态**，彻底剥夺通知权利。

2. **动态请求**：在代码中向用户动态请求该权限：
   ```kotlin
   if (Build.VERSION.SDK_INT >= 33) {
       if (activity.checkSelfPermission(android.Manifest.permission.POST_NOTIFICATIONS)
           != android.content.pm.PackageManager.PERMISSION_GRANTED
       ) {
           activity.requestPermissions(
               arrayOf(android.Manifest.permission.POST_NOTIFICATIONS), 1001
           )
       }
   }
   ```

> **血训**：即使前台服务通知理论上不受通知权限影响，但在 ColorOS/EMUI/HarmonyOS 等国产 ROM 上，如果缺少 `POST_NOTIFICATIONS` 声明，通知渠道会被系统直接禁用，导致 `startForeground()` 调用成功但通知完全不显示。

### 3.2 前台服务权限

```xml
<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_REMOTE_MESSAGING" />
```

### 3.3 服务声明

```xml
<service
    android:name=".service.StreamKeepaliveService"
    android:enabled="true"
    android:exported="false"
    android:foregroundServiceType="remoteMessaging"
    android:stopWithTask="false" />
```

## 4. 事件通道规范

### 4.1 三种事件通道对比

| 通道 | 前端监听方式 | 适用场景 | 当前使用 |
|------|-------------|----------|---------|
| `Plugin.trigger()` | `listen()` from `@tauri-apps/api/event` | Tauri 官方插件事件通道 | 未使用（trigger 与 listen 不互通） |
| `evaluateJavascript` + `CustomEvent` | `window.addEventListener()` | 需要前端 window 级别事件兼容性 | `vcp-keyboard-inset`、`vcp-lifecycle` |
| `PluginHandle.run_mobile_plugin` | 无需监听，直接 invoke | Rust 到 Kotlin 单向命令调用 | `startStreamingService`、`stopStreamingService` |

### 4.2 关键约束：trigger() 与 listen() 不互通

`Plugin.trigger()` 只能向通过 `registerListener` 注册的内部插件频道发射事件，而前端 `@tauri-apps/api/event` 的 `listen()` 使用全局事件总线。两者**不互通**。

因此，若前端使用 `window.addEventListener()` 监听，Kotlin 侧**必须**使用 `evaluateJavascript` 注入 `CustomEvent`。

### 4.3 evaluateJavascript 事件格式标准

```kotlin
val json = "{\"height\":$keyboardHeight,\"visible\":$isKeyboardVisible}"
val script = "window.dispatchEvent(new CustomEvent('vcp-keyboard-inset', { detail: $json }))"
webView.evaluateJavascript(script, null)
```

## 5. 插件开发规范

### 5.1 Rust 侧

- **状态管理**：使用 `app.manage()` 注入 `VcpMobileState<R>`，通过 `app.state::<VcpMobileState<R>>()` 获取
- **Android 插件注册**：在 `setup` 中调用 `register_android_plugin`，保存返回的 `PluginHandle` 供 `run_mobile_plugin` 调用
- **错误处理**：Tauri command handler 中严禁使用 `unwrap()` 或 `expect()`，必须转换为 `Result<T, String>`
- **条件编译**：Android 专用代码用 `#[cfg(target_os = "android")]` 包围

### 5.2 Kotlin 侧

- **类注解**：`@TauriPlugin` 必须标注在 `Plugin(activity)` 子类上
- **命令注解**：`@Command` 标注公开方法，方法参数固定为 `Invoke`
- **参数解析**：`invoke.parseArgs(ArgsClass::class.java)`，参数类必须标注 `@InvokeArg`
- **返回值**：成功调用 `invoke.resolve()`；失败调用 `invoke.reject(message)`

### 5.3 通知规范（前台服务）

| 属性 | 要求 | 说明 |
|------|------|------|
| 渠道重要性 | `IMPORTANCE_HIGH` | 确保在所有 OEM 上显式显示 |
| `setOngoing(true)` | 必须 | 通知不可滑动关闭 |
| `setSilent(true)` | **禁止** | 部分 ROM 会将其归类为静默通知而不显示 |
| `setPriority()` | **禁止** | Android 8+ 已废弃 |
| Small Icon | `applicationInfo.icon` | 使用应用自身品牌图标 |
| 前台服务类型 | `remoteMessaging` | Android 14+ 必需 |

## 6. 官方插件采用准则

### 6.1 三条铁律

| 优先级 | 规则 |
|--------|------|
| 1 | **官方插件优先**：文件打开用 `tauri-plugin-opener`，触觉反馈用 `tauri-plugin-haptics` |
| 2 | **自定义插件兜底**：Android 特有的前台服务、键盘 Insets、生命周期桥接，官方无覆盖，必须自建插件 |
| 3 | **Raw JNI 仅限一次性调用**：`FLAG_KEEP_SCREEN_ON` 只需一次 `addFlags()`/`clearFlags()`，维持 Rust 侧 JNI 裸调比新增 Kotlin 方法更简洁 |

## 7. 血训与不可变共识

### 7.1 插件命令静默拒绝

如果插件命令被调用但没有任何反应（无错误、无返回），**首先检查 `capabilities/default.json` 是否包含 `"vcp-mobile:default"`**。Tauri v2 的能力系统会静默拒绝未授权命令。

### 7.2 POST_NOTIFICATIONS 是通知的前提

Android 13+ 上，没有 `POST_NOTIFICATIONS` 静态声明 + 动态请求，**任何通知在 ColorOS/EMUI/HarmonyOS 上都不会显示**。

### 7.3 Plugin.load() 不可靠

`Plugin.load(webView)` 的调用时机取决于 JNI 注册完成的时机。如果需要在 WebView 初始化后立即执行的逻辑（如返回键拦截），应放在 `MainActivity.onWebViewCreate(webView)` 中，这是 Wry 保证调用的钩子。

### 7.4 PluginHandle.run_mobile_plugin 优先于 Raw JNI

对于 Rust 到 Kotlin 的复杂调用（如启动 Service），优先使用 `PluginHandle.run_mobile_plugin(command, payload)` 而非手动 JNI。前者使用 Tauri 官方桥接通道，类型安全且避免 `loadClass` 等 JNI 反射风险。

## A.4 选型速查口诀

> 与本文档第 6 节“三条铁律”保持一致：

1. **官方插件优先**：凡是官方已有覆盖（fs、dialog、geolocation、biometric、barcode-scanner、haptics、notification），直接使用 `@tauri-apps/plugin-*`，享受长期维护保障。
2. **社区插件补位**：Android 特有功能（电池优化、防截屏、Sharesheet、设备信息）选用社区成熟插件，评估指标：近 6 个月有提交、crates.io 周下载 > 50、文档完整。
3. **自建插件兜底**：当官方 + 社区均无法满足（如 VCP Mobile 的流式前台保活、键盘 Insets、生命周期桥接），按本文档第 5 节规范自建插件，禁止为了单一简单调用引入整个社区插件。

## B.2 现代请求范式（Activity Result API）

Android 官方已废弃 `ActivityCompat.requestPermissions()` + `onRequestPermissionsResult()` 的组合，推荐使用 **Jetpack Activity Result API**，生命周期更安全。

### B.2.1 在 Tauri Plugin Kotlin 侧的标准模板

```kotlin
// 文件：VcpMobilePlugin.kt 或其他自建 Plugin 类

import androidx.activity.result.contract.ActivityResultContracts
import androidx.core.content.ContextCompat
import android.Manifest
import android.content.pm.PackageManager
import android.os.Build
import android.net.Uri
import android.provider.Settings
import app.tauri.plugin.Plugin
import app.tauri.plugin.Invoke
import app.tauri.plugin.Command

@TauriPlugin
class VcpMobilePlugin(private val activity: Activity) : Plugin(activity) {

    // --- 权限 Launcher（必须在 Plugin 初始化前注册） ---
    private val permissionLauncher = activity.registerForActivityResult(
        ActivityResultContracts.RequestMultiplePermissions()
    ) { result ->
        val granted = result.filterValues { it }.keys
        val denied = result.filterValues { !it }.keys
        // 通过 evaluateJavascript 回传结果给前端
        val json = """{"granted":${granted.toJsonArray()}, "denied":${denied.toJsonArray()}}"""
        val script = "window.dispatchEvent(new CustomEvent('vcp-permission-result', { detail: $json }))"
        webView?.evaluateJavascript(script, null)
    }

    // --- 通用权限检查方法 ---
    private fun hasPermissions(vararg permissions: String): Boolean {
        return permissions.all {
            ContextCompat.checkSelfPermission(activity, it) == PackageManager.PERMISSION_GRANTED
        }
    }

    // --- 通用权限请求方法 ---
    private fun requestPermissions(vararg permissions: String) {
        val notGranted = permissions.filter {
            ContextCompat.checkSelfPermission(activity, it) != PackageManager.PERMISSION_GRANTED
        }.toTypedArray()
        if (notGranted.isNotEmpty()) {
            permissionLauncher.launch(notGranted)
        } else {
            // 已全部授权，直接回传成功
            val json = """{"granted":${permissions.toJsonArray()}, "denied":[]}"""
            val script = "window.dispatchEvent(new CustomEvent('vcp-permission-result', { detail: $json }))"
            webView?.evaluateJavascript(script, null)
        }
    }

    // 辅助：Array 转 JSON 字符串
    private fun Array<out String>.toJsonArray(): String {
        return "[" + joinToString(",") { "\"$it\"" } + "]"
    }

    // --- 示例：暴露给 Rust/前端的权限命令 ---
    @Command
    fun checkNotificationPermission(invoke: Invoke) {
        val granted = if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
            hasPermissions(Manifest.permission.POST_NOTIFICATIONS)
        } else {
            true // Android 12 及以下通知无需运行时权限
        }
        invoke.resolveObject(mapOf("granted" to granted))
    }

    @Command
    fun requestNotificationPermission(invoke: Invoke) {
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
            requestPermissions(Manifest.permission.POST_NOTIFICATIONS)
        }
        invoke.resolve()
    }
}
```

> **约束**：`registerForActivityResult` 必须在 Activity 状态变为 `CREATED` 之前调用。Tauri Plugin 的初始化时机通常满足此条件，但若出现 `LifecycleOwner stopped` 错误，请将权限请求逻辑延迟到 `MainActivity.onWebViewCreate()` 中初始化。

## B.3 分权限实战指南

### B.3.1 通知权限（POST_NOTIFICATIONS）

- **目标版本**：Android 13+（API 33+）
- **静态声明**：
  ```xml
  <uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
  ```
- **动态请求**：
  ```kotlin
  if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
      if (!hasPermissions(Manifest.permission.POST_NOTIFICATIONS)) {
          requestPermissions(Manifest.permission.POST_NOTIFICATIONS)
      }
  }
  ```
- **国产 ROM 血训**：ColorOS / EMUI / HarmonyOS 上，缺少 `POST_NOTIFICATIONS` 静态声明时，系统设置中的通知开关会变为**灰色不可交互**。即使前台服务通知理论上不受通知权限影响，这些 ROM 也会直接禁用通知渠道，导致 `startForeground()` 成功但通知不显示。

### B.3.2 位置权限（Location）

- **权限组**：
  ```xml
  <uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
  <uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
  <!-- 后台定位：Android 10+ 需单独声明 -->
  <uses-permission android:name="android.permission.ACCESS_BACKGROUND_LOCATION" />
  ```
- **版本分支**：
  ```kotlin
  val locationPermissions = mutableListOf<String>()
  locationPermissions += Manifest.permission.ACCESS_FINE_LOCATION
  locationPermissions += Manifest.permission.ACCESS_COARSE_LOCATION

  if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q) {
      // Android 10+ 若需后台定位，必须额外请求
      if (needBackgroundLocation) {
          locationPermissions += Manifest.permission.ACCESS_BACKGROUND_LOCATION
      }
  }

  requestPermissions(*locationPermissions.toTypedArray())
  ```
- **后台定位限制**：Android 10+ 要求 `ACCESS_BACKGROUND_LOCATION` 必须与前台定位权限分开请求，且 Google Play 要求提供合理的使用场景说明。国产 ROM 可能在此基础上增加额外限制（如 MIUI 的"始终允许"需二次弹窗确认）。

### B.3.3 存储与多媒体权限（Storage & Media）

Android 的存储权限经历了从**统一存储权限**到**细分媒体权限**再到**用户选择部分访问**的演进。

- **静态声明（按版本区分）**：
  ```xml
  <!-- Android 12 及以下 -->
  <uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE"
      android:maxSdkVersion="32" />
  <uses-permission android:name="android.permission.WRITE_EXTERNAL_STORAGE"
      android:maxSdkVersion="29" />

  <!-- Android 13+ 细分媒体权限 -->
  <uses-permission android:name="android.permission.READ_MEDIA_IMAGES" />
  <uses-permission android:name="android.permission.READ_MEDIA_VIDEO" />
  <uses-permission android:name="android.permission.READ_MEDIA_AUDIO" />

  <!-- Android 14+ 部分访问（用户选择） -->
  <uses-permission android:name="android.permission.READ_MEDIA_VISUAL_USER_SELECTED" />

  <!-- 全磁盘访问（特殊权限，需跳转到设置页授权） -->
  <uses-permission android:name="android.permission.MANAGE_EXTERNAL_STORAGE"
      tools:ignore="ScopedStorage" />
  ```
- **动态请求**：
  ```kotlin
  val storagePermissions = when {
      Build.VERSION.SDK_INT >= Build.VERSION_CODES.UPSIDE_DOWN_CAKE -> {
          // Android 14+：先请求 READ_MEDIA_VISUAL_USER_SELECTED
          arrayOf(Manifest.permission.READ_MEDIA_VISUAL_USER_SELECTED)
      }
      Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU -> {
          // Android 13+：细分权限
          arrayOf(
              Manifest.permission.READ_MEDIA_IMAGES,
              Manifest.permission.READ_MEDIA_VIDEO,
              Manifest.permission.READ_MEDIA_AUDIO
          )
      }
      else -> {
          // Android 12 及以下：传统存储权限
          arrayOf(Manifest.permission.READ_EXTERNAL_STORAGE)
      }
  }
  requestPermissions(*storagePermissions)
  ```
- **Scoped Storage 与 MANAGE_EXTERNAL_STORAGE**：
  - Android 10+ 默认启用 Scoped Storage，应用只能访问自己的目录和公共媒体集。
  - 若需访问非媒体文件（如 PDF、任意路径的文件），使用 `Storage Access Framework`（`ACTION_OPEN_DOCUMENT`）而非申请 `MANAGE_EXTERNAL_STORAGE`。
  - `MANAGE_EXTERNAL_STORAGE` 仅在文件管理器、杀毒软件等特殊场景下允许使用，Google Play 审核会要求提供视频声明。

### B.3.4 相机权限（Camera）

- **静态声明**：
  ```xml
  <uses-permission android:name="android.permission.CAMERA" />
  <uses-feature android:name="android.hardware.camera" android:required="false" />
  ```
- **动态请求**：
  ```kotlin
  requestPermissions(Manifest.permission.CAMERA)
  ```

### B.3.5 麦克风 / 录音权限（Record Audio）

- **静态声明**：
  ```xml
  <uses-permission android:name="android.permission.RECORD_AUDIO" />
  ```
- **动态请求**：
  ```kotlin
  requestPermissions(Manifest.permission.RECORD_AUDIO)
  ```
- **注意**：Android 11+ 支持一次性权限授予，用户选择"仅本次允许"后，应用退后台即失效。语音消息、实时语音等场景需在前台时保持活跃。

### B.3.6 前台服务权限（Foreground Service）

- **静态声明**：
  ```xml
  <uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
  <!-- Android 14+ 需声明具体类型 -->
  <uses-permission android:name="android.permission.FOREGROUND_SERVICE_REMOTE_MESSAGING" />
  ```
- **服务声明**：
  ```xml
  <service
      android:name=".service.YourForegroundService"
      android:foregroundServiceType="remoteMessaging"
      android:exported="false" />
  ```
- **Android 14+ 前台服务类型限制**：
  - `dataSync` → 已被废弃，建议使用 `WorkManager` 或 `userInitiatedDataTransfer`
  - `remoteMessaging`：适用于即时通讯类应用的后台消息同步
  - `mediaPlayback` / `mediaProjection` / `microphone` / `camera` / `location` 等按场景选择
  - 启动前台服务前，必须已通过对应权限的动态请求（如 `FOREGROUND_SERVICE_LOCATION` 需先拿到定位权限）。

### B.3.7 生物识别权限（Biometric）

- **静态声明**：
  ```xml
  <uses-permission android:name="android.permission.USE_BIOMETRIC" />
  ```
- **注意**：`USE_FINGERPRINT` 已被废弃，统一使用 `USE_BIOMETRIC`。
- **使用方式**：推荐直接使用 Tauri 官方 `tauri-plugin-biometric`，其内部已封装 `BiometricPrompt`。

### B.3.8 蓝牙权限（Bluetooth）

- **静态声明**：
  ```xml
  <!-- Android 12+ 新权限模型 -->
  <uses-permission android:name="android.permission.BLUETOOTH_SCAN"
      android:usesPermissionFlags="neverForLocation" />
  <uses-permission android:name="android.permission.BLUETOOTH_CONNECT" />
  <uses-permission android:name="android.permission.BLUETOOTH_ADVERTISE" />

  <!-- Android 11 及以下兼容 -->
  <uses-permission android:name="android.permission.BLUETOOTH"
      android:maxSdkVersion="30" />
  <uses-permission android:name="android.permission.BLUETOOTH_ADMIN"
      android:maxSdkVersion="30" />
  <uses-permission android:name="android.permission.ACCESS_FINE_LOCATION"
      android:maxSdkVersion="30" />
  ```
- **动态请求（Android 12+）**：
  ```kotlin
  if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
      requestPermissions(
          Manifest.permission.BLUETOOTH_SCAN,
          Manifest.permission.BLUETOOTH_CONNECT
      )
  } else {
      requestPermissions(Manifest.permission.ACCESS_FINE_LOCATION)
  }
  ```

## B.4 权限拒绝后的用户引导

### B.4.1 判断拒绝状态

```kotlin
when {
    hasPermissions(Manifest.permission.CAMERA) -> {
        // 已授权，执行功能
    }
    activity.shouldShowRequestPermissionRationale(Manifest.permission.CAMERA) -> {
        // 用户之前拒绝了，但未勾选"不再询问"
        // 应显示解释性 UI，说明为什么需要此权限
        showPermissionRationaleDialog(
            permission = Manifest.permission.CAMERA,
            message = "相机权限用于扫描二维码和拍摄头像，拒绝后将无法使用相关功能。",
            onConfirm = { requestPermissions(Manifest.permission.CAMERA) }
        )
    }
    else -> {
        // 用户勾选了"不再询问"或首次请求前
        // 引导用户前往系统设置手动开启
        showGoToSettingsDialog()
    }
}
```

### B.4.2 跳转到应用设置页

```kotlin
val intent = Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS).apply {
    data = Uri.fromParts("package", activity.packageName, null)
    addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
}
activity.startActivity(intent)
```

## B.5 Tauri 自定义插件中的权限封装建议

为保持 VCP Mobile 插件架构的一致性，建议将权限逻辑按以下模式封装：

### B.5.1 Plugin 内部工具方法

```kotlin
// 封装在 BasePlugin 或 VcpMobilePlugin 中
object PermissionHelper {
    fun check(invoke: Invoke, vararg permissions: String) { /* ... */ }
    fun request(invoke: Invoke, vararg permissions: String) { /* ... */ }
}
```

### B.5.2 向前端暴露的 API 设计

| Rust Command | Kotlin 方法 | 前端调用 | 说明 |
|--------------|-------------|----------|------|
| `check_permission` | `checkPermission(invoke)` | `invoke('check_permission', { name: 'camera' })` | 检查单一权限状态 |
| `request_permission` | `requestPermission(invoke)` | `invoke('request_permission', { names: ['camera', 'microphone'] })` | 请求权限数组 |
| `open_app_settings` | `openAppSettings(invoke)` | `invoke('open_app_settings')` | 跳转到系统设置 |

### B.5.3 权限结果回传前端

与本文档第 4.3 节 `evaluateJavascript` 事件格式保持一致：

```kotlin
val detail = """
    {
        "permissions": {
            "android.permission.CAMERA": "granted",
            "android.permission.RECORD_AUDIO": "denied"
        },
        "allGranted": false
    }
""".trimIndent()
val script = "window.dispatchEvent(new CustomEvent('vcp-permission-change', { detail: $detail }))"
webView.evaluateJavascript(script, null)
```

前端监听：

```typescript
window.addEventListener('vcp-permission-change', (e: CustomEvent) => {
    const { permissions, allGranted } = e.detail;
    console.log('权限变更:', permissions);
});
```
