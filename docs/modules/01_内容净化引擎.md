# 01_内容净化引擎（Context Sanitizer）

## 1. 概述

### 1.1 模块定位

`context_sanitizer.rs` 是 VCP Mobile 核心层（Rust 后端）的**上下文 HTML 标签转 Markdown 净化器模块**（Rust 重构版），位于 `src-tauri/src/vcp_modules/context_sanitizer.rs`（371 行）。该模块承担从桌面端（JS 实现）向移动端 Rust 核心迁移后的 HTML 内容标准化职责，是消息渲染流水线中的关键预处理环节。

### 1.2 职责边界

| 职责领域 | 具体行为 | 对应源码位置 |
|---------|---------|------------|
| HTML → Markdown 转换 | 将富文本 HTML 降级为 Markdown，降低前端渲染复杂度 | `html_to_vcp_markdown:138` |
| 多媒体保留 | `img`、`audio`、`video` 标签原样保留为 HTML 片段 | `process_node:154`（算法 B） |
| VCP 特殊块提取 | `data-raw-content` 零损提取、`TOOL_REQUEST`、`DailyNote` 标记透传 | `process_node:154`（算法 A） |
| 元思考链清理 | 明文 `[--- VCP元思考链 ---]` 与 `<think>` 标签的剥离或结构化 | `strip_thought_chains:130` |
| 缓存加速 | 基于 LRU + TTL 的净化结果缓存，避免重复解析 | `ContextSanitizer:33` |

### 1.3 调用入口

```text
Vue 3 前端（消息渲染层）
    ↓ Tauri IPC invoke
lib.rs（命令路由）
    ↓ 调用
chat_manager.rs / message_service.rs
    ↓ 调用
ContextSanitizer::sanitize_content(content, keep_thoughts)
    ↓ 命中/未命中缓存
html_to_vcp_markdown(html, keep_thoughts)
```

主要公共入口为 `ContextSanitizer::sanitize_content`，其内部先进行 HTML 存在性检查与缓存查找，再委派至 `html_to_vcp_markdown` 执行实际的树遍历转换。

## 2. 核心类型与数据结构

### 2.1 CacheItem

```rust
pub struct CacheItem {
    pub value: String,      // 净化后的 Markdown 结果
    pub timestamp: SystemTime,  // 写入时间戳（用于 TTL 判定）
}
```

`CacheItem` 是缓存存储的基本单元，位于 `context_sanitizer.rs:26`。采用 `SystemTime` 记录写入时刻，在读取时通过 `timestamp.elapsed()` 与 TTL（Time To Live，存活时间）比对，实现惰性过期（Lazy Expiration）。

### 2.2 ContextSanitizer

```rust
pub struct ContextSanitizer {
    pub cache: Mutex<LruCache<String, CacheItem>>,
    pub ttl: Duration,
}
```

| 字段 | 类型 | 说明 |
|-----|------|------|
| `cache` | `Mutex<LruCache<String, CacheItem>>` | 线程安全的 LRU 缓存，键为哈希组合键，值为净化结果 |
| `ttl` | `Duration` | 缓存项最大存活时间，默认 3600 秒 |

`ContextSanitizer` 位于 `context_sanitizer.rs:33`，实现了 `Default` trait（默认容量 100 条，TTL 1 小时）。由于 Tauri 的命令处理模型为单线程事件循环，但内部使用 Tokio 异步运行时，`Mutex` 选用 `std::sync::Mutex` 而非 `tokio::sync::Mutex`，因为缓存操作极短，无需跨 await 点持有锁。

### 2.3 全局正则表达式（lazy_static）

| 正则名称 | 编译位置 | 用途 |
|---------|---------|------|
| `THOUGHT_CHAIN_REGEX` | `context_sanitizer.rs:16` | 匹配 `[--- VCP元思考链: "title" ---]... [--- 元思考链结束 ---]` |
| `CONVENTIONAL_THOUGHT_REGEX` | `context_sanitizer.rs:18` | 匹配 `<think>...</think>`（不区分大小写） |
| `HTML_CHECK_REGEX` | `context_sanitizer.rs:20` | 快速预检字符串中是否包含 HTML 标签 |
| `MULTI_NEWLINE_REGEX` | `context_sanitizer.rs:22` | 将 3 个以上连续换行压缩为 2 个 |

所有正则均通过 `lazy_static!` 宏在首次访问时惰性编译，避免重复编译开销。

## 3. 算法详解

### 3.1 HTML 树遍历与节点处理（process_node 递归）

`process_node` 函数位于 `context_sanitizer.rs:154`，是整个净化引擎的核心递归处理器。其输入为 `ego_tree::NodeRef<Node>`（来自 `scraper`  crate 的 DOM 树节点引用）与可变字符串缓冲区 `out`。

```text
                  HTML 解析入口
                       │
                       ▼
            Html::parse_fragment(html)
                       │
                       ▼
          遍历 root().children()
                       │
              ┌────────┴────────┐
              ▼                 ▼
        Node::Text         Node::Element
        (直接追加)            (按标签分发)
              │                 │
              │    ┌────────────┼────────────┐
              │    ▼            ▼            ▼
              │  算法 A      算法 B       算法 C/D
              │ (特殊块)   (多媒体)    (思考链/基础标签)
              │    │            │            │
              │  零损提取    原样保留      递归子节点
              │    │            │            │
              └────┴────────────┴────────────┘
                               │
                               ▼
                     追加到 out 缓冲区
```

递归终止条件：
- 文本节点（`Node::Text`）：直接追加文本内容，无递归子调用。
- 元素节点（`Node::Element`）：根据标签名进入特定分支；对于未匹配标签，默认递归处理所有子节点（透传）。
- 其他节点类型（如注释、Doctype）：忽略（`_ => {}`）。

### 3.2 算法 A：特殊块零损提取

**目标**：对于已经过前端或其他系统预处理的 VCP 特殊块，避免二次解析导致格式损坏。

**触发条件**：元素节点包含 `data-raw-content` 自定义属性，位于 `context_sanitizer.rs:166`。

```text
process_node 进入元素节点
       │
       ▼
  检查 el.attr("data-raw-content")
       │
   ┌───┴────┐
   ▼        ▼
  存在     不存在
   │        │
   ▼        ▼
直接追加    继续算法 B/C/D
属性值到 out
（立即 return，
 不处理子节点）
```

**伪代码**：
```rust
if let Some(raw) = el.attr("data-raw-content") {
    out.push_str(raw);
    return;  // 关键：截断递归，子节点完全忽略
}
```

此设计确保即使 `data-raw-content` 内部的文本看起来像是 HTML，也不会被递归解析，实现**零损（Lossless）提取**。

### 3.3 算法 B：多媒体与特殊结构处理

#### 3.3.1 图片标签保留

| 属性 | 处理逻辑 | 源码位置 |
|-----|---------|---------|
| `src` | 必须非空；为空则跳过 | `context_sanitizer.rs:175` |
| `alt` | 透传，为空时默认 `""` | `context_sanitizer.rs:176` |

输出格式：`<img src="{src}" alt="{alt}">`（保持原 HTML 形式，不转 Markdown 图片语法）。

#### 3.3.2 音频/视频标签保留

| 场景 | 处理逻辑 | 源码位置 |
|-----|---------|---------|
| 元素自身含 `src` | 直接输出 `<audio src="..."></audio>` | `context_sanitizer.rs:183` |
| 元素无 `src` | 递归扫描第一个 `<source>` 子节点的 `src` | `context_sanitizer.rs:189` |

**降级策略**：若 `<audio>` 或 `<video>` 标签未直接携带 `src`，则遍历子节点寻找 `<source src="...">`，取第一个有效源地址。这是为了兼容部分编辑器生成的嵌套 source 结构。

#### 3.3.3 代码块与 VCP 原始块（pre）

```text
pre 标签到达
     │
     ▼
 collect_text(node, &mut text_content)
     │
     ▼
 文本包含特殊标记？
     │
  ┌──┴────┐
  ▼       ▼
 是      否
  │       │
  ▼       ▼
直接追加  包装为 Markdown 代码块
到 out   ```\n...\n```
```

特殊标记检测（`context_sanitizer.rs:210`）：
- `<<<[TOOL_REQUEST]>>>`：工具调用请求标记
- `<<<DailyNoteStart>>>`：日记块起始标记

若文本包含以上任一标记，则视为 VCP 内部协议块，**不添加 Markdown 代码围栏**，直接透传原始文本。

### 3.4 算法 C：元思考链结构化

**触发条件**：`<div>` 标签具有 CSS 类 `vcp-thought-chain-bubble`，位于 `context_sanitizer.rs:222`。

**行为分支**：

| `keep_thoughts` 参数 | 行为 |
|---------------------|------|
| `true` | 将 HTML 结构转换为 VCP 明文思考链格式 |
| `false` | 完全跳过该 div 及其所有子节点（不输出任何内容） |

**结构化输出格式**：
```text

[--- VCP元思考链: "{data-thought-title}" ---]
{递归处理子节点内容}
[--- 元思考链结束 ---]

```

当 `data-thought-title` 属性为空时，标题部分省略，输出 `[--- VCP元思考链 ---]`。

### 3.5 算法 D：基础 HTML → Markdown 转换

`process_node` 中对常见 HTML 标签的映射规则如下：

| HTML 标签 | Markdown 输出 | 源码位置 | 备注 |
|----------|--------------|---------|------|
| `p` | `\n{children}\n` | `:243` | 段落前后换行 |
| `br` | `\n` | `:250` | 强制换行 |
| `strong` / `b` | `**{children}**` | `:251` | 粗体 |
| `em` / `i` | `*{children}*` | `:258` | 斜体 |
| `h1`–`h6` | `\n{'#' × level} {children}\n` | `:265` | 标题层级由末尾数字决定 |
| `code` | `` `{children}` `` | `:277` | 行内代码 |
| `ul` | `\n{children}\n` | `:284` | 无序列表容器 |
| `li` | `- {children}\n` | `:291` | 列表项 |
| `a` | `[{children}]({href})` | `:298` | 超链接 |
| 其他 | 递归处理子节点 | `:306` | 默认透传 |

**特殊设计点**：
- `h1`–`h6` 的层级解析未使用查找表，而是通过 `tag.chars().last().unwrap().to_digit(10)` 直接解析标签名末尾字符，简洁但依赖输入合法性。
- `ul` 与 `li` 的映射未处理嵌套列表的缩进层级，当前实现为扁平化 `- ` 前缀。对于 VCP Mobile 的消息上下文场景，嵌套列表出现频率较低，此简化被接受。

### 3.6 缓存策略（LRU + TTL）

#### 3.6.1 缓存查找流程

```text
sanitize_content 被调用
        │
        ▼
   content.trim().is_empty()?
        │
   ┌────┴────┐
   ▼         ▼
  是        否
   │         │
   ▼         ▼
直接返回   contains_html(content)?
            │
       ┌────┴────┐
       ▼         ▼
      否        是
       │         │
       ▼         ▼
   直接返回  generate_cache_key(...)
                 │
                 ▼
         get_cached(&cache_key)
                 │
            ┌────┴────┐
            ▼         ▼
         命中       未命中/过期
            │         │
            ▼         ▼
        返回缓存值  html_to_vcp_markdown(...)
                      │
                      ▼
                  set_cached(...)
                      │
                      ▼
                  返回新计算值
```

#### 3.6.2 缓存键生成

`generate_cache_key` 位于 `context_sanitizer.rs:337`，使用 `std::collections::hash_map::DefaultHasher` 对原始内容与 `keep_thoughts` 布尔值进行组合哈希：

```rust
let mut hasher = DefaultHasher::new();
content.hash(&mut hasher);
keep_thoughts.hash(&mut hasher);
let hash = hasher.finish();
format!("sanitized_{}_{}", hash, content.len())
```

键格式：`sanitized_{hash}_{len}`。追加 `content.len()` 是为了降低哈希碰撞导致的误命中概率（生日攻击防护）。

#### 3.6.3 TTL 惰性过期

`get_cached` 位于 `context_sanitizer.rs:58`，采用**读取时清理**策略：

```rust
if let Some(item) = cache.get(key) {
    if let Ok(elapsed) = item.timestamp.elapsed() {
        if elapsed < self.ttl {
            return Some(item.value.clone());
        }
    }
    cache.pop(key);  // 过期项在读取时弹出
}
None
```

此设计无需后台定时任务，但可能在缓存中长期积累已过期却未被访问的"僵尸项"，直到 LRU 容量淘汰将其驱逐。

## 5. 与桌面端/JS 的历史对齐

### 5.1 迁移背景

文件注释中明确提到"对齐 JS 逻辑"（`context_sanitizer.rs:147`），表明本模块是桌面端 JavaScript 实现的 Rust 重构版本。迁移的主要动机包括：

1. **内存安全**：将 HTML 解析与正则处理从 JS 移至 Rust，避免移动端 V8 引擎的内存抖动。
2. **计算本地化**：减少跨 IPC 的大字符串传输，HTML 解析在 Rust 侧完成后，前端仅接收精简的 Markdown。
3. **统一缓存**：Rust 侧的 LRU 缓存可跨消息生命周期复用，JS 侧难以在页面刷新后保持缓存状态。

### 5.2 已确认对齐的行为

| 行为 | Rust 实现 | 原 JS 逻辑推断 | 对齐状态 |
|-----|----------|--------------|---------|
| 多余空行压缩为最多 2 个 | `MULTI_NEWLINE_REGEX.replace_all(result.trim(), "\n\n")` | 相同逻辑（注释明确提及） | ✅ 已对齐 |
| `pre` 标签内 VCP 标记透传 | 检测 `TOOL_REQUEST` 与 `DailyNoteStart` | 相同特殊块协议 | ✅ 已对齐 |
| `data-raw-content` 零损提取 | `el.attr("data-raw-content")` 直接追加 | 相同属性协议 | ✅ 已对齐 |
| `vcp-thought-chain-bubble` 结构化 | 转换为明文 `[--- VCP元思考链 ---]` | 相同输出格式 | ✅ 已对齐 |

### 5.3 已知差异

| 差异点 | Rust 侧行为 | 潜在影响 |
|-------|------------|---------|
| HTML 解析器 | 使用 `scraper`（基于 `html5ever`） | 与浏览器原生 `DOMParser` 在边缘 HTML（如未闭合标签）处理上可能存在细微差异 |
| 正则引擎 | `fancy-regex`（PCRE/Perl 兼容） | 与 JS `RegExp` 在 lookahead/lookbehind 行为上略有不同，但当前使用的正则均未涉及高级特性 |
| 缓存机制 | 进程级 LRU + TTL | JS 侧通常无持久缓存，此为 Rust 侧新增优化 |

## 7. 性能考量

### 7.1 HTML 解析开销

`html_to_vcp_markdown` 使用 `scraper::Html::parse_fragment` 对输入进行 HTML5 兼容解析。该解析器基于 Mozilla 的 `html5ever`，时间复杂度为 O(n)，其中 n 为输入字符串长度。对于典型聊天消息（< 10 KB），解析开销可忽略；但对于包含大量富文本的长消息（如带多篇附件引用的系统消息），需注意以下优化点：

| 优化机制 | 实现方式 | 效果 |
|---------|---------|------|
| 无 HTML 短路 | `contains_html` 正则预检，无标签时直接返回原字符串 | 避免对纯文本消息执行完整 DOM 解析 |
| 空内容短路 | `content.trim().is_empty()` 前置判断 | 消除空字符串的哈希与缓存开销 |
| 片段解析 | `parse_fragment` 而非 `parse_document` | 不构建完整文档树（无 `<html>`/`<body>` 包装），减少节点数量 |

### 7.2 缓存命中率与参数调优

默认配置（容量 100，TTL 3600 秒）基于以下假设设计：

- **容量 100**：在典型会话中，用户会在短时间内多次查看同一批历史消息。100 条缓存足以覆盖当前可视窗口及其附近上下文。
- **TTL 1 小时**：消息内容在发送后通常不可变，极长的 TTL 本身不会导致脏数据；设置上限是为了在极端长会话中允许内存 eventual 释放。

**缓存失效场景**：
- 同一条消息被连续渲染，但 `keep_thoughts` 参数切换（如用户展开/收起思考链）→ 生成不同缓存键，导致双份存储。
- 消息编辑功能引入后，内容变更但缓存键基于旧内容 → 需在外层（调用方）在编辑后手动使缓存失效，当前模块未提供 `invalidate` 接口。

### 7.3 锁竞争分析

`ContextSanitizer.cache` 使用 `std::sync::Mutex` 保护 `LruCache`：

```text
sanitize_content 调用
        │
        ▼
  get_cached: lock() → get() → unlock()
        │
        ▼
  html_to_vcp_markdown: 无锁（纯计算）
        │
        ▼
  set_cached: lock() → put() → unlock()
```

在 Tauri 的单线程命令派发模型下，绝大多数场景为串行访问，锁竞争可忽略。若未来引入并发消息预加载，可考虑将 `Mutex` 替换为 `parking_lot::Mutex` 以降低锁开销，或采用 `tokio::sync::Mutex` 支持跨 await 的异步锁（但当前无此需求）。

### 7.4 正则性能

| 正则表达式 | 复杂度 | 适用场景 | 潜在风险 |
|-----------|--------|---------|---------|
| `HTML_CHECK_REGEX` | O(n) 线性扫描 | 快速预检 | 对包含 `<` 的纯文本（如数学公式）可能产生假阳性，但假阳性仅导致一次不必要的 HTML 解析，不影响正确性 |
| `THOUGHT_CHAIN_REGEX` | O(n)（`(?s)` 单行模式） | 思考链剥离 | 输入极长且无匹配时，回溯可控；`.*` 在 `(?s)` 下跨行匹配，但起始与结束标记固定，实际回溯有限 |
| `MULTI_NEWLINE_REGEX` | O(n) 线性扫描 | 空行压缩 | `\n{3,}` 为贪婪量词，但目标字符单一，无灾难性回溯风险 |

### 7.5 内存使用特征

- **峰值内存**：解析阶段 `scraper` 构建完整 DOM 树，峰值约为输入 HTML 大小的 2–4 倍（含节点结构开销）。对于 20 KB 输入，峰值内存约 40–80 KB，远低于移动端 OOM 阈值。
- **缓存内存**：每条缓存项存储完整净化后字符串。假设平均消息 2 KB，100 条缓存约占用 200 KB + 哈希表开销，整体 < 1 MB。
