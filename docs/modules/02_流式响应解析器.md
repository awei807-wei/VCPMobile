# 流式响应解析器

## 1. 概述

### 1.1 模块定位

`stream_block_parser.rs`（流式块解析器，位于 `src-tauri/src/vcp_modules/stream_block_parser.rs`）是 VCP Mobile 对话渲染 pipeline 中的**增量解析层（Incremental Parsing Layer）**。其核心职责是在 SSE（Server-Sent Events）流式传输场景下，对持续累积的 `full_text` 进行轻量扫描，识别出语义上已闭合的特种块（Special Blocks）和未闭合的尾部文本（Tail），供前端实现增量式（Incremental）UI 更新。

与完整的非流式解析器 `content_parser.rs` 不同，本模块不追求一次性产出最终 AST，而是维护一个单调递增的 `processed_len` 游标，确保已确认闭合的块不会被重复解析，从而降低流式场景下的计算开销。

### 1.2 与 content_parser.rs 的关系

| 维度 | `content_parser.rs` | `stream_block_parser.rs` |
|------|---------------------|--------------------------|
| **触发时机** | 消息完全接收后（非流式） | 每次 SSE chunk 到达后（流式） |
| **解析策略** | 全量扫描，构建完整 `ContentBlock` 数组 | 增量扫描，仅处理未确认区域 |
| **输出类型** | `Vec<ContentBlock>` | `Vec<StreamBlock>` + `tail: String` |
| **AST 预渲染** | 部分块有 `nodes` 字段 | 所有 Markdown 与 Thought/Diary 均预渲染 AST |
| **Hash 计算** | ✅ 有（`Option<String>`，通过 `compute_hashes_recursively`） | ✅ 每块均带 `hash: String`（前端 `v-for` key） |
| **状态维护** | 无状态（纯函数） | 有状态（`StreamBlockParser` 持 `processed_len`） |

两者共享同一套**正则常量（Regular Expression Constants）**和**块类型判别逻辑（Block Type Discrimination Logic）**，但面向不同生命周期阶段。`content_parser.rs` 使用 `MASTER_START` 单一正则进行全量捕获；`stream_block_parser.rs` 使用 `find_earliest_start_marker` 进行竞争匹配（Competitive Matching），更适合增量决策。

### 1.3 流式渲染场景

前端（Vue 3）通过 Tauri IPC 接收 SSE 流中的文本增量。每次增量到达后：

1. 调用 `StreamBlockParser::process(full_text)` 获取 `(blocks, tail)`
2. 将 `blocks` 追加到已稳定的渲染列表（Stable Blocks）
3. 将 `tail` 显示为「正在输入…」的临时区域
4. 当 SSE 流结束（Stream End），调用 `finalize()` 将剩余 `tail` 强制封装为最后一个 Markdown 块

```
SSE Chunk 1: "Hello, this is a "
  → process() → ( [], "Hello, this is a " )
  
SSE Chunk 2: "Hello, this is a test.\n\n[--- VCP元思考链 ---]\nStep 1"
  → process() → ( [Markdown{test}], "[--- VCP元思考链 ---]\nStep 1" )
  
SSE Chunk 3: "...\n[--- 元思考链结束 ---]"
  → process() → ( [Markdown{test}, Thought{Step 1}], "" )
```

### 2.3 工厂方法设计（Factory Methods）

`StreamBlock` 和 `ContentBlock` 均通过 `impl` 块提供**工厂方法**，以避免在代码中直接书写 enum variant 的 struct literal。Rust enum variant 不支持 `..Default::default()` 语法，因此当 variant 新增字段（如 `hash`）时，工厂方法是唯一能在「新增字段」与「避免全局补全」之间取得平衡的方案。

```rust
impl StreamBlock {
    pub fn markdown(content: String, nodes: Option<Vec<MarkdownNode>>, hash: String) -> Self;
    pub fn thought(theme: String, content: String, is_complete: bool, nodes: Option<Vec<MarkdownNode>>, hash: String) -> Self;
    pub fn tool(tool_name: String, content: String, hash: String) -> Self;
    pub fn tool_result(tool_name: String, status: String, details: Vec<ToolResultDetail>, footer: String, hash: String) -> Self;
    pub fn diary(maid: String, date: String, content: String, nodes: Option<Vec<MarkdownNode>>, hash: String) -> Self;
    pub fn html_preview(content: String, hash: String) -> Self;
    pub fn role_divider(role: String, is_end: bool, hash: String) -> Self;
    pub fn style(content: String, hash: String) -> Self;
    pub fn button_click(content: String, hash: String) -> Self;
}
```

| 设计要点 | 说明 |
|----------|------|
| **Hash 必填** | `StreamBlock` 的所有工厂方法均要求传入 `hash: String`，确保流式输出的每一块都有稳定 key |
| **零默认字段** | 工厂方法内部不再使用 `hash: None` 之类的默认值；调用方必须在构造前通过 `HashAggregator::compute_content_hash(...)` 计算好 hash |
| **与 ContentBlock 的对称** | `ContentBlock`（`content_parser.rs`）同样提供工厂方法，但 hash 默认填 `None`，由后续的 `compute_hashes_recursively()` 统一写入 |

> **为什么不在 `StreamBlock` 也使用 `Option<String>`？** 流式解析器在构造块时已经拥有完整文本，hash 计算成本已支付。使用 `String` 而非 `Option<String>` 可以消除前端每次渲染时的 `unwrap` 开销，同时让 `v-for :key` 的类型更干净。

## 3. StreamBlockParser 结构体与状态机

### 3.1 状态定义

```rust
pub struct StreamBlockParser {
    processed_len: usize,
}
```

`StreamBlockParser`（流式块解析器状态机）是本项目流式渲染架构中唯一持状态的 Rust 结构体，定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:169-171`。其状态极简，仅维护一个单调递增的字节游标（Monotonically Increasing Byte Cursor）。

### 3.2 processed_len 的语义

`processed_len` 记录的是**已被确认为「已闭合块」的文本在 `full_text` 中的结束位置**（Exclusive End Offset）。它具有以下关键语义：

| 场景 | `processed_len` 行为 | 原因 |
|------|---------------------|------|
| 新段落完整闭合 | 更新为 `full_text.len()` | 全部文本已稳定 |
| 特种块完整闭合 | 更新为 `pos + content_start + end_end` | 跳过已处理区域 |
| 特种块未找到结束标记 | 回退到该块起始位置 `pos` | 该块及之后全部变为 tail，等待下一 chunk |
| Markdown 段落未完整闭合（无 `\n\n`） | 更新为 `pos + remaining.len() - tail.len()` | tail 部分尚未形成完整段落 |

> **关键不变量（Invariant）**：`processed_len` 永远不会超过 `full_text.len()`，且只在以下两种情况下回退：
> 1. 遇到未闭合的特种块起始标记；
> 2. Markdown 段落被特种块中断，且段落尾部不完整。

### 3.4 finalize() 的收尾逻辑

`finalize(&mut self, full_text: &str) -> Vec<StreamBlock>` 定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:244-253`，在 SSE 流结束时调用：

```
finalize(full_text)
  │
  ├─► (blocks, tail) = process(full_text)   // 先走常规增量逻辑
  │
  ├─► trimmed = tail.trim()
  │
  └─► trimmed 非空 ?
        ├─Yes─► nodes = parse_markdown_to_ast(trimmed)
        │       hash  = compute_content_hash(trimmed)
        │       blocks.push( StreamBlock::markdown(trimmed.to_string(), Some(nodes), hash) )
        │
        └─No──► 直接返回 blocks
```

`finalize` 的核心职责是**将流结束时的未闭合 tail 强制视为最终 Markdown 段落**，避免前端出现「永远输入中」的悬停状态。该段落同样会进行 AST 预渲染和 Hash 计算，确保与已闭合块的数据一致性。

### 3.5 reset() 方法

`reset(&mut self)` 将 `processed_len` 归零，用于新消息到达时复用解析器实例，避免重新分配内存。


## 4. 核心算法

### 4.1 find_earliest_start_marker：竞争匹配

```rust
fn find_earliest_start_marker(text: &str) -> Option<(usize, usize, BlockType)>
```

定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:266-292`。该函数对 10 种特种块起始标记进行**竞争匹配（Competitive Matching）**，返回最早出现的那一组 `(start_offset, end_offset, BlockType)`。

#### 正则常量来源表

| 索引 | 正则常量（来自 `content_parser.rs`） | 匹配模式 | 对应 `BlockType` |
|------|--------------------------------------|----------|----------------|
| 1 | `TOOL_START` | `<<<[TOOL_REQUEST]>>>` | `Tool` |
| 2 | `THOUGHT_START` | `[--- VCP元思考链:主题 ---]` | `Thought` |
| 3 | `THINK_START` | `<think>` / `<thinking>` | `Think` |
| 4 | `TOOL_RESULT_START` | `[[VCP调用结果信息汇总:` | `ToolResult` |
| 5 | `DIARY_START` | `<<<DailyNoteStart>>>` | `Diary` |
| 6 | `HTML_FENCE_START` | `\`\`\`html` | `HtmlFence` |
| 7 | `HTML_DOC_START` | `<!doctype html>` / `<html…>` | `HtmlDoc` |
| 8 | `ROLE_DIVIDER` | `<<<[ROLE_DIVIDE_XXX]>>>` / `<<<[END_ROLE_DIVIDE_XXX]>>>` | `RoleDivider` |
| 9 | `STYLE_TAG_START` | `<style…>` | `Style` |
| 10 | `GENERIC_CODE_FENCE_START` | `\`\`\`lang` | `CodeFence` |

> 注意：`HtmlFence` 与 `CodeFence` 在 `StreamBlock` 输出层均映射为 `HtmlPreview` 或 `Markdown`，但内部判别阶段保持独立，以避免 HTML 围栏与普通代码围栏的解析冲突。

算法伪代码：

```
earliest = None
for (regex, block_type) in checks:
    if regex.find(text) matches at position m:
        if earliest is None OR m.start < earliest.start:
            earliest = (m.start, m.end, block_type)
return earliest
```

所有正则均通过 `lazy_static!` 预编译并缓存，运行时仅执行 `find()`，避免重复编译开销。正则定义详见 `src-tauri/src/vcp_modules/content_parser.rs:76-126`。

### 4.4 split_markdown_paragraphs：段落分割

```rust
fn split_markdown_paragraphs(text: &str) -> (Vec<StreamBlock>, String)
```

定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:422-451`。该函数按 `\n\n`（双换行）将纯文本分割为已闭合的 Markdown 段落和未闭合的尾部。

```
split_markdown_paragraphs(text)
  │
  ├─► text 为空 ? → return ([], "")
  │
  ├─► 查找最后一个 "\n\n" 的位置 last_break
  │     ├─► 未找到 → return ([], text)        // 全部作为 tail
  │     └─► 找到   → stable = text[..last_break+2]
  │                   tail   = text[last_break+2..]
  │
  ├─► stable.split("\n\n") 遍历每个 para:
  │     ├─► trim 后为空则跳过
  │     ├─► nodes = parse_markdown_to_ast(trimmed)
  │     ├─► hash  = compute_content_hash(trimmed)
  │     └─► 生成 StreamBlock::Markdown { ... }
  │
  ├─► blocks = extract_inline_buttons(blocks)   // 内联按钮提取
  │
  └─► return (blocks, tail.to_string())
```

**为什么使用最后一个 `\n\n` 而非全部分割？** 在流式场景下，尾部段落可能尚未写完（例如用户正在输入一个长段落）。只将最后一个 `\n\n` 之前的文本视为稳定区域，确保不因文本截断而产生错误的段落分裂。

### 4.5 extract_inline_buttons：内联按钮提取

```rust
fn extract_inline_buttons(blocks: Vec<StreamBlock>) -> Vec<StreamBlock>
```

定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:453-514`。该函数扫描所有 `StreamBlock::Markdown` 的 `content` 字段，匹配 `BUTTON_CLICK` 正则（`\[\[点击按钮:(.*?)\]\]`，定义于 `content_parser.rs:227`），将 Markdown 块拆分为：

- 按钮前的 Markdown 块（若有）
- `StreamBlock::ButtonClick` 块
- 按钮后的 Markdown 块（若有）

```
输入: [ Markdown{ "请点击[[点击按钮:确认]]继续" }, ... ]
                        │
                        ▼
输出: [ Markdown{ "请点击" }, ButtonClick{ "确认" }, Markdown{ "继续" }, ... ]
```

每个拆分出的子块都会**重新执行 AST 预渲染和 Hash 计算**，保证数据完整性。非 `Markdown` 变体的块原样透传。


## 5. 各块类型的构建逻辑详解

本节深入分析 `build_stream_block` 中每个 `BlockType` 分支的特殊处理逻辑。所有分支均定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:320-420`。

### 5.1 Tool 块的双重人格：普通工具调用 vs 日记创建

Tool 块（`<<<[TOOL_REQUEST]>>>` … `<<<[END_TOOL_REQUEST]>>>`）存在一条**类型升级路径（Type Promotion Path）**：

```
┌─────────────────────────────────────────┐
│  BlockType::Tool                        │
│  inner_content 传入                     │
└─────────────┬───────────────────────────┘
              ▼
        ┌─────────────┐
        │ extract_    │
        │ tool_name() │
        └──────┬──────┘
               ▼
        ┌─────────────────┐
        │ is_daily_note_  │
        │ create(content) │
        └───────┬─────────┘
           Yes / \ No
              /   \
             ▼     ▼
    ┌──────────┐  ┌─────────────┐
    │ extract_ │  │ StreamBlock │
    │ diary_   │  │ ::Tool      │
    │ details()│  │ { tool_name,│
    └────┬─────┘  │  content,   │
         │        │  hash }     │
         ▼        └─────────────┘
    ┌──────────┐
    │ StreamBlock│
    │ ::Diary   │
    │ { maid,   │
    │  date,    │
    │  content, │
    │  nodes,   │
    │  hash }   │
    └──────────┘
```

- **升级条件**：`content.contains("DailyNote") && content.contains("create")`
- **升级后**：不再输出 `StreamBlock::Tool`，而是输出 `StreamBlock::Diary`，并提取 `maid`、`date`、`content` 三个字段
- **预渲染**：升级为 Diary 后，会对 `content` 执行 `parse_markdown_to_ast`，生成 AST 节点

### 5.2 Thought 与 Think 的区别

| 维度 | Thought（VCP 格式） | Think（通用格式） |
|------|---------------------|-------------------|
| **起始标记** | `[--- VCP元思考链:主题 ---]` | `<think>` / `<thinking>` |
| **结束标记** | `[--- 元思考链结束 ---]` | `</think>` / `</thinking>` |
| **主题提取** | 从起始标记正则捕获组提取 | 固定为 `"思维链"` |
| **正则锚定** | 行首锚定 `(?im)^[ \t]*` | 无行首锚定，可内联出现 |
| **输出变体** | `StreamBlock::Thought` | `StreamBlock::Thought`（复用同一变体） |
| **Hash 输入** | `theme:content` | `content` |

两者在 `StreamBlock` 层统一为 `thought` 类型（序列化标签 `thought`），前端无需区分渲染逻辑。差异仅在**主题来源**和**正则匹配策略**。

### 5.3 RoleDivider 的自闭合机制

`RoleDivider`（角色分隔器）是唯一在 `find_end_marker` 中**不查找结束正则**的类型：

```rust
BlockType::RoleDivider => {
    return Some((0, 0));  // 自闭合
}
```

这意味着 `build_stream_block` 接收到的 `inner_content` 为空字符串，`remaining[start_idx..end_idx]` 才是完整的标记文本。解析器从中提取：

- `is_end`：`caps.get(1).is_some()`，判断是否匹配到 `END_` 前缀
- `role`：`caps.get(2)`，取 `SYSTEM` / `ASSISTANT` / `USER`，并转为小写

### 5.4 CodeFence 的特殊处理：重构完整围栏

普通代码围栏（如 `\`\`\`rust` … `\`\`\``）在 `StreamBlock` 层被**降级（Demote）**为 Markdown 块：

```rust
let fence = &remaining[start_idx..end_idx];   // 包含起始 ```rust
let full_text = format!("{}\n{}\n```", fence, inner_content);
let nodes = parse_markdown_to_ast(&full_text);
```

原因：流式解析器不需要区分「带语言标识的代码块」与「普通 Markdown」，前者已经是 Markdown 标准语法的一部分。通过重构完整的围栏文本（包含起始和结束标记），可直接复用 `pre_renderer` 的通用 Markdown AST 解析能力。

### 5.5 HtmlFence 与 HtmlDoc 的统一

| 输入标记 | 内部 `BlockType` | 输出 `StreamBlock` | 内容范围 |
|----------|-----------------|-------------------|----------|
| `\`\`\`html` … `\`\`\`` | `HtmlFence` | `HtmlPreview` | `inner_content`（不含围栏标记） |
| `<!doctype html>` … `</html>` | `HtmlDoc` | `HtmlPreview` | `inner_content`（不含 HTML 标记） |

两者最终均映射为 `HtmlPreview`，前端可统一使用 iframe 或 WebView 进行渲染。`HtmlDoc` 的结束标记为 `</html>`（不区分大小写），`HtmlFence` 的结束标记为独立的 `\`\`\`` 行。

### 5.6 ToolResult 的 KV 解析

`ToolResult` 的构建调用 `parse_tool_result(inner_content)`，定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:562-607`。该函数采用**状态机逐行扫描**：

```
状态: IDLE ──匹配 KV 行──► CAPTURING(key)
                    │
                    ▼
         下一行匹配 KV ──► 提交上一 key，开启新 key
         下一行非 KV ──► 追加到 current_value_lines
         EOF ──► 提交最后一 key

未进入任何 key 的非空行 → footer_lines
```

KV 行格式：`- 工具名称: xxx` 或 `- 执行状态: yyy`，通过 `KV_REGEX`（`^-\s*([^:]+):\s*(.*)`）匹配。特殊 key「工具名称」和「执行状态」被提取到顶层字段，其余 key 放入 `details: Vec<ToolResultDetail>`。

### 5.7 Diary 的三字段提取

`extract_diary_details` 定义于 `src-tauri/src/vcp_modules/stream_block_parser.rs:500-520`，使用三个独立正则：

| 字段 | 正则 | 匹配模式 | 默认值 |
|------|------|----------|--------|
| `maid` | `MAID_REGEX` | `maid: 「始」…「末」` 或 `Maid: …` | `""` |
| `date` | `DATE_REGEX` | `Date: 「始」…「末」` 或 `Date: …` | `""` |
| `content` | `CONTENT_REGEX` | `Content: 「始」…[

]*「末」` 或 `Content: …` | `"[日记内容解析失败]"` |

所有正则均支持两种格式：带「始」「末」边界符的 VCP 内部协议格式，以及纯文本冒号格式。

### 6.2 BlockType 枚举的设计意图

`BlockType`（块类型判别枚举）是 `content_parser.rs` 的**内部（`pub(crate)`）类型**，不直接参与序列化，仅作为两个解析器之间的共享判别语言（Shared Discrimination Language）。

```rust
pub(crate) enum BlockType {
    Tool, Thought, Think, ToolResult, Diary,
    HtmlFence, HtmlDoc, Style, RoleDivider, CodeFence,

与 `StreamBlock`（面向前端序列化）和 `ContentBlock`（面向后端全量解析）不同，`BlockType` 更贴近「标记层面的物理类型」：
- `Thought` 与 `Think` 在输出层统一为 `StreamBlock::Thought`，但在判别层分离
- `HtmlFence` 与 `CodeFence` 在输出层分别走向 `HtmlPreview` 和 `Markdown`，但共享 `GENERIC_CODE_FENCE_END`

## 7. 性能与内存考量

### 7.1 增量处理避免重复计算

`processed_len` 是流式解析器的核心性能优化手段：

| 场景 | 全量解析（`content_parser.rs`） | 增量解析（`stream_block_parser.rs`） |
|------|--------------------------------|-------------------------------------|
| 每次 SSE chunk | 重新扫描整个 `full_text` | 仅从 `processed_len` 开始扫描 |
| 1KB → 10KB → 100KB | O(n²) 累积扫描 | O(n) 线性累积 |
| 已闭合块 AST | 每次都重新生成 | 永不重新生成 |

### 7.2 Hash 计算开销

每个 `StreamBlock` 均通过 `HashAggregator::compute_content_hash` 生成内容指纹，定义于 `src-tauri/src/vcp_modules/sync_hash.rs`。

```rust
HashAggregator::compute_content_hash(trimmed)  // 通常为 SHA-256 或类似算法
```

Hash 计算发生在以下时机：
- 每个已闭合的 Markdown 段落
- 每个已闭合的特种块
- `finalize()` 时的剩余 tail
- `extract_inline_buttons` 拆分后的每个子块

在流式高频调用场景下（SSE chunk 每 50-100ms 到达一次），Hash 计算是主要的 CPU 开销点之一。由于输入文本通常较短（单段落或单工具块），实际开销可控；但对于超长 Markdown 文档（>10KB），建议前端在稳定后切换至 `content_parser.rs` 的全量解析结果。

### 7.3 内存分配模式

```
process() 单次调用的典型分配路径:
  ├─> Vec<StreamBlock>  （blocks，预分配于栈）
  ├─> String::new()     （tail，预分配于栈）
  ├─> 每找到一个特种块 → StreamBlock 变体 + String 字段
  ├─> Markdown 段落    → Vec<MarkdownNode>（AST，堆分配）
  └─> Hash 字符串       → 固定长度 String（堆分配）
```

`StreamBlock` 的所有字段均拥有（Owned），无生命周期引用（No Lifetime References），这意味着：
- ✅ 输出可安全跨线程传递到前端（Tauri IPC 要求 `Serialize`）
- ⚠️ 高频 chunk 下会产生较多短生命周期堆分配，依赖 Rust 的默认分配器性能

### 7.4 正则匹配的性能特征

`find_earliest_start_marker` 中 10 个正则均使用 `lazy_static!` 预编译，但**每次调用仍独立执行 `find()`**：

```
最坏情况：剩余文本中无任何特种块标记
→ 10 个正则各自扫描整段文本
→ 时间复杂度 ≈ O(10 × n)，常数因子较大
```

实际场景中，特种块标记稀疏分布，且 `remaining` 随 `processed_len` 推进而迅速缩短，因此整体性能接近 O(n)。若未来出现性能瓶颈，可考虑：
1. 将 `MASTER_START`（`content_parser.rs` 中的单一合并正则）复用于流式场景的前置快速筛选
2. 对 `remaining.len() < threshold` 的短文本跳过正则扫描，直接作为 tail 返回

### 7.5 与前端增量渲染的协同

前端（Vue 3）通过 `:key="block.hash"` 使用 Hash 作为列表渲染的 Stable Key，确保：
- 已稳定块不会因新 chunk 到达而重新挂载（Re-mount）
- 仅 tail 区域和最新块需要重新渲染
- diff 算法的比较开销降至最低

这种「Rust 侧 Hash + Vue `v-for` key」的协同设计，是 VCP Mobile 流式对话保持 60fps 的关键架构决策之一。
