# 群聊发言策略引擎 (Group Speaking Policy)

## 1. 概述

### 1.1 模块定位

`group_speaking_policy.rs` 是 VCP Mobile 群聊系统的**发言决策核心**，负责在 `naturerandom`（自然随机）模式下，从群聊的活跃成员（Active Members）中筛选出应当参与回复的 AI Agent 列表。该模块实现了 **NatureRandom 决策引擎**，以多层优先级（Priority Tier）的方式模拟群聊中人类发言的自然性与随机性，避免所有 Agent 同时回复造成的"刷屏"问题，同时保证被提及或相关的 Agent 能够及时响应。

### 1.2 与群聊系统的交互

```
┌─────────────────┐
│   用户输入消息    │
└────────┬────────┘
         ▼
┌─────────────────────────────────────────────┐
│  chat_manager.rs / 群聊消息分发器              │
│  - 获取 active_members (AgentConfig[])        │
│  - 获取 history (ChatMessage[])               │
│  - 获取 group_config (GroupConfig)            │
└────────┬────────────────────────────────────┘
         │ 调用 determine_naturerandom_speakers()
         ▼
┌─────────────────────────────────────────────┐
│  group_speaking_policy.rs                    │
│  - 5 层优先级决策                             │
│  - 发言者排序优化                             │
└────────┬────────────────────────────────────┘
         │ 返回 Vec<AgentConfig> (有序发言者列表)
         ▼
┌─────────────────────────────────────────────┐
│  群聊执行器                                   │
│  - 按顺序向各 Agent 发起推理请求               │
│  - 渲染多轮回复到聊天界面                      │
└─────────────────────────────────────────────┘
```

### 1.3 决策输出

模块入口函数 `determine_naturerandom_speakers` 接收以下输入：

| 参数 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `active_members` | `&[AgentConfig]` | `agent_types.rs` | 群聊中所有活跃 Agent 配置 |
| `history` | `&[ChatMessage]` | `chat_manager.rs` | 完整聊天历史 |
| `group_config` | `&GroupConfig` | `group_types.rs` | 群组配置（含标签、匹配模式等） |
| `user_message` | `&ChatMessage` | 当前用户消息 | 触发本次决策的用户输入 |

输出为 `Vec<AgentConfig>`，按发言优先级排序（Rank 0 最先发言）。若最终列表为空，P5 保底策略将确保至少有一个 Agent 回复。

## 2. NatureRandom 算法总览

### 2.1 5 层优先级决策树

NatureRandom 引擎采用**瀑布式（Waterfall）**决策流程：高层优先级命中后，成员被加入发言列表并标记为 `spoken_this_turn`，后续低层优先级自动跳过该成员。若高层无命中，继续向下流淌至低层。

```
                              开始
                               │
                               ▼
                    ┌─────────────────────┐
                    │  P1: @角色名直接提及   │ ──命中?──┬─Yes──► 加入 speakers
                    │  (Direct Mention)    │          │       标记 spoken
                    └─────────┬───────────┘          │
                              │ No                   │
                              ▼                      │
                    ┌─────────────────────┐          │
                    │  P2: Tag 匹配        │ ──命中?──┼─Yes──► 加入 speakers
                    │  (Tag Matching)      │          │       标记 spoken
                    │  strict / natural    │          │
                    └─────────┬───────────┘          │
                              │ No                   │
                              ▼                      │
                    ┌─────────────────────┐          │
                    │  P3: @所有人         │ ──命中?──┼─Yes──► 全部加入
                    │  (Mention All)       │          │       标记 spoken
                    └─────────┬───────────┘          │
                              │ No                   │
                              ▼                      │
                    ┌─────────────────────┐          │
                    │  P4: 概率发言        │ ──命中?──┼─Yes──► 加入 speakers
                    │  (Probabilistic)     │  rng    │       标记 spoken
                    │  基础 15% / boost 85%│         │
                    └─────────┬───────────┘          │
                              │ No                   │
                              ▼                      │
                    ┌─────────────────────┐          │
                    │  P5: 保底发言        │ ──空?───┼─Yes──► 随机选 1 人
                    │  (Fallback)          │         │
                    │  相关成员优先/全局随机 │         │
                    └─────────┬───────────┘         │
                              │                     │
                              ▼                     │
                    ┌─────────────────────┐         │
                    │  发言者排序优化       │◄────────┘
                    │  (Rank 0/1/2)        │
                    └─────────┬───────────┘
                              ▼
                         返回 speakers
```

### 2.2 决策层级速查表

| 优先级 | 名称（英文） | 触发条件 | 命中结果 | 互斥性 |
|:------:|:-----------|:---------|:---------|:-------|
| P1 | Direct Mention（直接提及） | 用户消息包含 `@角色名` | 强制加入 | 已命中者不再参与下层 |
| P2 | Tag Matching（标签匹配） | 消息/上下文包含成员标签关键词 | 模式依赖 | 同上 |
| P3 | Mention All（@所有人） | 用户消息包含 `@所有人` | 全部加入 | 同上 |
| P4 | Probabilistic Speaking（概率发言） | 随机数 < 阈值 | 概率加入 | 同上 |
| P5 | Fallback Speaking（保底发言） | 前四层全部未命中 | 强制选 1 人 | N/A |

## 3. 各优先级详解

### 3.1 P1: @角色名直接提及（Direct Mention）

**定位**：最高优先级，不可被任何后续逻辑覆盖。

**触发逻辑**：

```
user_text = user_message.content.to_lowercase()
for member in active_members:
    if user_text.contains("@" + member.name.to_lowercase()):
        加入 speakers
        标记 spoken_this_turn
```

**关键规则**：

| 规则项 | 说明 |
|--------|------|
| 大小写敏感 | 不敏感（全部 `to_lowercase()` 后比较） |
| 格式要求 | 必须为 `@角色名`，无空格分隔要求 |
| 去重机制 | 通过 `HashSet<String>`（`member.id`）确保同一轮次中同一 Agent 只加入一次 |
| 日志输出 | 命中时打印 `[GroupSpeakingPolicy] @{name} triggered by direct mention.` |

**示例**：

| 用户输入 | Agent 名称 | 是否触发 P1 |
|---------|-----------|:----------:|
| `@Nova 你怎么看` | Nova | ✅ 是 |
| `我觉得@艾丽莎说得对` | 艾丽莎 | ✅ 是 |
| `@nova` | Nova | ✅ 是（大小写不敏感） |
| `@Nova ` | Nova | ✅ 是 |
| `Nova` | Nova | ❌ 否（缺少 `@`） |

---

### 3.2 P2: Tag 匹配（Tag Matching）

Tag（标签）是群组成员的语义特征关键词，存储于 `GroupConfig.member_tags`（`serde_json::Value` 类型，映射 `agentId → "tag1, tag2"`）。P2 根据 `group_config.tag_match_mode` 执行两种截然不同的匹配策略。

#### 3.2.1 Tag 解析规则

```
tag_string = group_config.member_tags[member.id]  // 例: "技术,编程,AI"
tags = tag_string.split(/[,，]/)                   // 中英文逗号分割
       .map(trim)
       .filter(not_empty)
```

#### 3.2.2 Strict 模式 vs Natural 模式

| 对比维度 | Strict 模式 | Natural 模式 |
|:--------|:------------|:-------------|
| **默认值** | ✅ 是（`tag_match_mode` 未设置时回退到 `"strict"`） | ❌ 需显式配置 `"natural"` |
| **核心哲学** | 关键词出现即触发（确定性） | 模拟人类注意力分散与话题延续（概率性） |
| **检查范围** | `user_text` + `context_text`（拼接后的最近 8 条历史） | 分层检查：`user_text` → 他人历史 → 自身历史 |
| **触发方式** | 标签在范围内**出现即命中**，100% 加入 | 取决于"谁的历史"和"随机数" |
| **P4 概率加成** | 是（命中 tag 的成员在 P4 概率从 15% 提升至 85%） | 否（P4 保持基础 15%） |
| **适用场景** | 需要精确控制 Agent 参与度的群聊 | 追求自然感、模拟真实群聊氛围 |

#### 3.2.3 Strict 模式详解

```
context_text = 最近 8 条历史消息的 content 拼接（去除发言头，转小写）

for member in active_members:
    if member 已在 spoken_this_turn: continue
    tags = 解析 member 的标签列表
    if tags.is_empty(): continue

    if any(tag in context_text or tag in user_text for tag in tags):
        加入 speakers
        标记 spoken_this_turn
```

**特点**：
- 只要标签关键词在用户输入或最近 8 条历史消息中出现，该 Agent **必定**参与回复。
- 历史消息清洗：去除 `[xxx的发言]: ` 前缀，避免发言者名称被误匹配为标签。

#### 3.2.4 Natural 模式详解

Natural 模式引入**三层注意力漏斗**，模拟人类群聊中"听到自己感兴趣的话题才插嘴"的行为：

```
for member in active_members:
    if member 已在 spoken_this_turn: continue
    tags = 解析 member 的标签列表

    // 第一层: 直接兴趣（强触发）
    tag_in_others = 他人消息含标签 或 用户输入含标签
    tag_at = 用户输入含 @标签
    if tag_in_others or tag_at:
        强制加入 speakers
        continue

    // 第二层: 自身历史延续（概率触发）
    tag_in_own = 自身历史消息含标签
    if tag_in_own:
        计算动态概率 speak_chance
        if rng < speak_chance:
            概率加入 speakers
```

**三层注意力判定表**：

| 层级 | 条件 | 触发结果 | 概率 |
|:-----|:-----|:---------|:-----|
| L1（直接兴趣） | 用户输入含 `@tag` | 强制加入 | 100% |
| L1（直接兴趣） | 用户输入含 `tag` **或** 他人历史消息含 `tag` | 强制加入 | 100% |
| L2（自身延续） | 仅自身历史消息含 `tag`，L1 未命中 | 动态概率 | 见第 4 节 |
| L3（无兴趣） | 任何位置均无 `tag` | 不触发 P2 | 0% |

---

### 3.3 P3: @所有人（Mention All）

**定位**：次高优先级，通常在 P1/P2 之后执行，但可覆盖未发言成员。

**触发条件**：`user_text.contains("@所有人")`

**行为**：将所有 `active_members` 中尚未在 `spoken_this_turn` 中的成员全部加入发言列表。

**注意**：该优先级**不跳过**已在 `spoken_this_turn` 中的成员（通过 `insert` 返回值去重），因此不会导致重复。

---

### 3.4 P4: 概率发言（Probabilistic Speaking）

**定位**：在前三层均未命中的成员中，以基础概率随机触发发言，增加群聊的"偶然参与感"。

**基础参数**：

| 参数 | 数值 | 说明 |
|:-----|:-----|:-----|
| `base_chance` | 0.15 (15%) | 每个未命中成员独立掷骰 |
| `strict_boost` | 0.85 (85%) | Strict 模式下，上下文含 tag 的成员大幅提升 |
| 随机源 | `rand::thread_rng()` | 每次循环独立调用 `gen::<f32>()` |

**概率公式**：

```
for member in active_members:
    if member 已在 spoken_this_turn: continue

    speak_chance = 0.15

    if tag_match_mode == "strict":
        tags = 解析 member 的标签
        if any(tag in context_text for tag in tags):
            speak_chance = 0.85

    if rng.gen::<f32>() < speak_chance:
        加入 speakers
        标记 spoken_this_turn
```

**模式差异**：

| 模式 | 基础概率 | Tag 加成 | 加成条件 |
|:-----|:---------|:---------|:---------|
| Strict | 15% | +70% → 85% | 该成员标签出现在 `context_text` 中 |
| Natural | 15% | 无 | N/A（Natural 的 tag 逻辑在 P2 已处理） |

---

### 3.5 P5: 保底发言（Fallback Speaking）

**定位**：安全网机制。若前四层均未产生任何发言者，强制选择至少一人回复，避免群聊"冷场"。

**策略**：

```
if speakers.is_empty() and active_members not empty:
    relevant_members = active_members 中标签匹配上下文的成员

    if relevant_members not empty:
        fallback = random_choice(relevant_members)
    else:
        fallback = random_choice(active_members)  // 全局随机

    speakers.push(fallback)
```

**回退优先级**：

| 条件 | 选择范围 | 策略说明 |
|:-----|:---------|:---------|
| 存在标签匹配上下文的成员 | `relevant_members` | **语义优先**：选择与当前话题相关的 Agent |
| 无相关成员 | `active_members` | **全局随机**：任意选一个 Agent 兜底 |

## 4. Natural 模式的动态概率公式

在 Natural 模式的 P2 第二层（自身历史延续）中，当成员的 tag **仅出现在自身历史消息**中（即未在用户输入或他人消息中出现）时，系统不强制触发，而是使用**动态概率公式**决定是否发言。该公式模拟"话痨效应"——一个 Agent 越 recently active，它继续插嘴的概率越高。

### 4.1 前置条件

满足以下全部条件时进入动态概率分支：
1. `tag_match_mode == "natural"`
2. 成员标签未在用户输入中命中（`@tag` 或 `tag`）
3. 成员标签未在他人历史消息中命中
4. 成员标签在**自身**的最近历史消息中命中（`tag_in_own == true`）

### 4.2 概率分支图

```
                    tag_in_own == true
                           │
                           ▼
              ┌────────────────────────┐
              │ 获取倒数第二条消息      │
              │ (跳过当前用户消息)      │
              │ 查找最后一个 assistant  │
              └───────────┬────────────┘
                          │
              ┌───────────┴───────────┐
              ▼                       ▼
    is_last_speaker == true    is_last_speaker == false
              │                       │
              ▼                       ▼
    ┌─────────────────┐      ┌─────────────────┐
    │  own_msg_count  │      │  speak_chance   │
    │  = 自身在窗口中  │      │  = 0.2 (20%)    │
    │   的发言次数     │      └────────┬────────┘
    └────────┬────────┘               │
             │                        │
             ▼                        │
    speak_chance =                    │
    (0.5 + own_msg_count * 0.1)       │
         .min(0.75)                   │
             │                        │
             └────────┬───────────────┘
                      ▼
              ┌───────────────┐
              │ rng < chance ? │
              └───────┬───────┘
                      │
            ┌─────────┴─────────┐
            ▼                   ▼
          Yes                 No
            │                   │
            ▼                   ▼
        加入 speakers        不加入
```

### 4.3 公式详解

#### 场景 A：该成员是最后发言者（`is_last_speaker == true`）

```
speak_chance = min(0.5 + own_msg_count * 0.1, 0.75)
```

| `own_msg_count` | 概率值 | 说明 |
|:---------------|:-------|:-----|
| 1 | 0.60 (60%) | 窗口内自身发言 1 次 |
| 2 | 0.70 (70%) | 窗口内自身发言 2 次 |
| ≥3 | 0.75 (75%) | 触及上限，不再增长 |

**设计意图**：最后发言者若自身历史消息中多次出现相关标签，说明它对该话题有持续兴趣（话痨），应给予更高的继续发言概率。

#### 场景 B：该成员不是最后发言者（`is_last_speaker == false`）

```
speak_chance = 0.20 (20%)
```

**设计意图**：非最后发言者即使自身历史有相关标签，也因"话题已冷"而被赋予较低概率，避免过度抢话。

### 4.4 参数速查表

| 参数 | 符号 | 值 | 来源 |
|:-----|:-----|:---|:-----|
| 基础概率（非最后发言者） | `p_base` | 0.20 | 硬编码 |
| 最后发言者基础值 | `p_last_base` | 0.50 | 硬编码 |
| 每次发言增量 | `δ` | 0.10 | 硬编码 |
| 概率上限 | `p_max` | 0.75 | 硬编码 |
| 随机数范围 | `rng` | [0.0, 1.0) | `rand::thread_rng().gen::<f32>()` |

## 5. 发言者排序优化（Speaker Ranking）

在 5 层决策全部完成后，模块对 `speakers` 列表进行一次**排序优化**，确保与当前用户消息最相关的 Agent 最先发言。该排序基于**用户最新发言对标签的命中程度**。

### 5.1 Rank 定义

```
speakers.sort_by_cached_key(|member| {
    tags = 解析 member 的标签

    if any(tag in user_text for tag in tags):
        return 0    // Rank 0: 最高优先级
    else if any(tag in context_text for tag in tags):
        return 1    // Rank 1: 中等优先级
    else:
        return 2    // Rank 2: 最低优先级
})
```

### 5.2 Rank 判定逻辑表

| Rank | 判定条件 | 优先级 | 典型来源 |
|:----:|:---------|:------:|:---------|
| 0 | 用户输入 (`user_text`) 包含成员标签 | 最高 | P2 Natural L1、P2 Strict、P4 Strict boost |
| 1 | 仅历史上下文 (`context_text`) 包含成员标签 | 中等 | P2 Natural L2（概率命中）、P4 基础随机命中 |
| 2 | 用户输入和上下文中均无成员标签 | 最低 | P3 @所有人、P5 保底、P4 纯随机命中 |

### 5.3 排序影响

排序结果直接影响**群聊执行器向各 Agent 发送请求的顺序**。Rank 0 的 Agent 会最先收到推理请求，其回复也会最先出现在聊天界面中，形成"最相关者先响应"的用户体验。

**注意**：同一 Rank 内部的成员保持它们在 5 层决策中被加入的原始顺序（`sort_by_cached_key` 是稳定排序）。

## 6. 上下文窗口（Context Window）与历史消息处理

### 6.1 CONTEXT_WINDOW 常量

```rust
const CONTEXT_WINDOW: usize = 8;  // 对齐桌面端 CONTEXT_WINDOW = 8
```

该常量定义了决策引擎观察的**历史消息深度**。值为 `8` 表示引擎仅关注最近 8 条历史消息（不含当前用户消息）。该值与桌面端实现保持一致，确保跨平台行为统一。

### 6.2 窗口截取逻辑

```
history_len = history.len()
start_idx = max(0, history_len - (8 + 1))
end_idx   = max(0, history_len - 1)     // 排除当前用户消息

recent_history = history[start_idx .. end_idx]
```

| 场景 | `history_len` | `start_idx` | `end_idx` | 窗口大小 |
|:-----|:-------------|:-----------|:----------|:---------|
| 历史不足 8 条 | 3 | 0 | 2 | 2 条 |
| 历史正好 8 条 | 9 | 0 | 8 | 8 条 |
| 历史超过 8 条 | 15 | 6 | 14 | 8 条 |
| 仅当前消息 | 1 | 0 | 0 | 0 条（空切片） |

### 6.3 消息清洗（Sanitization）

在拼接 `context_text` 前，每条历史消息的内容会经过**发言头去除**处理：

```
Regex: ^\[.*?的发言\]:\s*
```

**示例**：

| 原始内容 | 清洗后 |
|---------|--------|
| `[Nova的发言]: 我觉得这个方案可行` | `我觉得这个方案可行` |
| `[System]: 用户已加入群聊` | `用户已加入群聊` |

**目的**：防止 Agent 的显示名称（如"Nova"）或发言头格式被误匹配为标签关键词，导致无关 Agent 被错误触发。

### 6.4 上下文拼接格式

清洗后的消息以 `" \n "`（空格 + 换行 + 空格）连接成单个字符串 `context_text`，用于后续的 `contains` 子串匹配。

```
msg1 + " \n " + msg2 + " \n " + msg3 + ...
```

## 9. 近期架构对齐变更（2026-05）

以下变更已在 `group_chat_application_service.rs`、`group_types.rs`、`context_assembler.rs` 及相关模块中落地，本文档同步更新以反映最新状态。

### 9.1 Speaker Name Prefix 简化

`context_assembler.rs` 中 speaker name 的推导逻辑去除冗余 fallback，简化为：

- 优先使用 `msg.name`（非空时）
- 为空时按 `role` 降级：`"user"` → `"User"`，`"assistant"` → `"AI"`

该简化移除了此前可能存在的多层嵌套回退，使发言头格式 `[{speaker_name}的发言]: \n{content}` 的构造更直接。

### 9.2 `invite_prompt` 迁移为独立 User 消息

`group_chat_application_service.rs` 中，`invite_prompt` 从 system prompt 的一部分迁移为**独立 user 消息**：

- 在历史消息组装完成后，`invite_prompt` 以 `{"role": "user", "content": processed_invite}` 形式追加到 messages 列表末尾（system prompt 之前）
- 支持模板变量 `{{VCPChatAgentName}}`，在追加前替换为当前发言 Agent 的名称
- 该变更是 nudge trigger 对齐的一部分，使邀请提示词在对话序列中获得更合理的角色定位

### 9.3 `requestId` → `messageId` 重命名

群聊请求载荷中的唯一标识字段由 `requestId` 统一重命名为 `messageId`，与 `VcpRequestPayload` 结构保持一致。

### 9.4 `current_topic_id` 字段删除

`group_types.rs` 中 `GroupConfig` 已删除 `current_topic_id` 字段，相关 SQL 查询同步清理。当前话题 ID 由调用方（如 `GroupChatPayload.topic_id`）显式传入，不再在群组配置中冗余存储。

### 9.5 `group_context_assembler.rs` 删除

原 `group_context_assembler.rs`（11 行）已删除，其逻辑合并入 `context_assembler.rs`。上下文组装相关的工具函数现集中管理，减少模块碎片化。
