# 表情包管理系统

## 1. 概述

### 1.1 模块定位

`emoticon_manager.rs`（`src-tauri/src/vcp_modules/emoticon_manager.rs`）是 VCP Mobile 的**表情包库管理器**（Emoticon Library Manager），归属 Rust 核心层的媒体资源治理域。该模块承担以下四项核心职责：

| 职责 | 说明 |
|------|------|
| 远程同步 | 通过 `admin_api/emojis/list` 接口从 VCP 服务器拉取完整表情包目录结构 |
| 模糊匹配 | 基于 Levenshtein 编辑距离（Edit Distance）对损坏或过期的表情包 URL 进行智能修复 |
| 双层存储 | 内存缓存（Memory Cache）+ SQLite 持久化（Persistent Storage）双层架构 |
| URL 编码 | 对齐前端 JavaScript `encodeURIComponent` 行为的 percent-encoding 规则 |

### 1.2 使用场景

| 场景 | 触发方式 | 涉及接口 |
|------|---------|---------|
| 首次启动或手动刷新 | 用户在设置页点击"刷新表情包库" | `regenerate_emoticon_library` |
| 消息渲染时 URL 失效 | 前端图片加载失败，调用修复逻辑 | `fix_emoticon_url` |
| 表情包面板浏览 | 前端打开表情包选择器 | `get_emoticon_library` |

### 1.3 与前端的关系

该模块通过 Tauri Command（`#[tauri::command]`）向前端暴露三个调用点。前端不直接访问 SQLite，全部数据交互均经过 Rust 层的内存缓存或数据库加载：

```
┌─────────────────┐     Tauri IPC      ┌─────────────────────────────┐
│   前端 (Vue 3)   │  <─────────────>  │   emoticon_manager.rs       │
│                 │                    │   (Rust Core Layer)         │
│ regenerate_...  │ ─────────────────> │ refresh_emoticon_library_.. │
│ get_emoticon_.. │ ─────────────────> │ internal_load_library       │
│ fix_emoticon_.. │ ─────────────────> │ fix_emoticon_url            │
└─────────────────┘                    └─────────────────────────────┘
```

## 2. 核心数据结构

### 2.1 EmoticonItem

`EmoticonItem`（`src-tauri/src/vcp_modules/emoticon_manager.rs:23-31`）是表情包库中的原子记录结构，实现了 `Serialize` / `Deserialize` / `Clone`：

| 字段 | 类型 | 序列化别名 | 说明 |
|------|------|-----------|------|
| `id` | `Option<i64>` | — | 数据库主键，插入前为 `None`，加载后为 `Some(row_id)` |
| `category` | `String` | — | 表情包包名（Package Name），如 `"可爱表情"` |
| `filename` | `String` | — | 文件名（Filename），如 `"hello.png"` |
| `url` | `String` | — | 完整远程访问 URL，含 `pw=` 鉴权段 |
| `search_key` | `String` | `searchKey` | 小写检索键，格式为 `category/filename`，用于模糊匹配加速 |

### 2.2 EmoticonManagerState

`EmoticonManagerState`（`src-tauri/src/vcp_modules/emoticon_manager.rs:33-43`）是 Tauri 托管的全局单例状态（Managed State），承载运行时内存缓存：

| 字段 | 类型 | 说明 |
|------|------|------|
| `library` | `Arc<Mutex<Vec<EmoticonItem>>>` | 线程安全的内存级表情包库，`tokio::sync::Mutex` 保证异步上下文安全 |

该状态在 `lib.rs` 启动流程中通过 `app.manage(EmoticonManagerState::default())` 注入。所有 Tauri Command 通过 `State<'_, EmoticonManagerState>` 获取共享实例。

### 2.3 EmojiListResponse

`EmojiListResponse`（`src-tauri/src/vcp_modules/emoticon_manager.rs:139-142`）是远程 API 的原始响应结构，仅用于内部反序列化：

| 字段 | 类型 | 说明 |
|------|------|------|
| `data` | `HashMap<String, Vec<String>>` | 键为包名（`category`），值为该包下的文件名列表 |

### 3.4 URL 构造规则

每条表情包的完整 URL 遵循以下固定模式（`src-tauri/src/vcp_modules/emoticon_manager.rs:220-223`）：

```

```

| 片段 | 编码方式 | 说明 |
|------|---------|------|
| `base_url` | 无 | 来自设置，如 `https://vcp.example.com` |
| `file_key` | 无 | 原始密钥字符串，直接拼接 |
| `encoded_category` | `utf8_percent_encode(..., VCP_ENCODE_SET)` | 对齐 JS `encodeURIComponent` |
| `encoded_filename` | `utf8_percent_encode(..., VCP_ENCODE_SET)` | 对齐 JS `encodeURIComponent` |

### 3.5 数据库事务写入

同步流程采用**全量替换**策略（`src-tauri/src/vcp_modules/emoticon_manager.rs:205-249`）：

1. 开启 `sqlx::Transaction` 保证原子性；
2. 执行 `DELETE FROM emoticon_library` 清空旧数据；
3. 遍历远程返回的 `HashMap`，逐条 `INSERT`；
4. 提交事务 `transaction.commit()`；
5. 将新库写入内存缓存 `EmoticonManagerState.library`。

该策略的优点是实现简单、数据一致性高；缺点是同步期间前端若调用 `get_emoticon_library` 可能短暂读到空库（因事务隔离）。


## 4. 模糊匹配算法

### 4.1 Levenshtein 编辑距离（Edit Distance）

`edit_distance`（`src-tauri/src/vcp_modules/emoticon_manager.rs:46-82`）基于经典的**动态规划**（Dynamic Programming）实现，计算两个字符串之间最少的单字符编辑操作次数（插入、删除、替换）。

#### 算法伪代码

```
function edit_distance(s1, s2):
    chars1 = s1.chars().collect()    // 按 Unicode 字符拆分
    chars2 = s2.chars().collect()
    n = chars1.len()
    m = chars2.len()

    if n == 0: return m
    if m == 0: return n

    // 初始化 (n+1) × (m+1) 的二维 DP 表
    dp = [[0; m+1]; n+1]
    for i in 0..=n: dp[i][0] = i
    for j in 0..=m: dp[0][j] = j

    for i in 1..=n:
        for j in 1..=m:
            cost = 0 if chars1[i-1] == chars2[j-1] else 1
            dp[i][j] = min(
                dp[i-1][j]   + 1,    // 删除
                dp[i][j-1]   + 1,    // 插入
                dp[i-1][j-1] + cost  // 替换（或相同）
            )
    return dp[n][m]
```

#### 复杂度分析

| 指标 | 值 | 说明 |
|------|-----|------|
| 时间复杂度 | O(n × m) | n、m 分别为两个字符串的字符数 |
| 空间复杂度 | O(n × m) | 使用二维 `Vec<Vec<usize>>` 存储 DP 状态 |
| 字符粒度 | Unicode Scalar Value | `chars().collect()` 保证中文字符按单字计算 |

### 4.2 相似度计算

`get_similarity`（`src-tauri/src/vcp_modules/emoticon_manager.rs:84-93`）将编辑距离归一化为 0.0 ~ 1.0 的相似度分数：

```
similarity = (max_len - edit_distance(s1_lower, s2_lower)) / max_len
```

| 步骤 | 说明 |
|------|------|
| 小写化 | 调用 `to_lowercase()` 忽略大小写差异 |
| 取最长长度 | `longer_length = max(len(s1), len(s2))` |
| 边界处理 | 若两字符串均为空，直接返回 `1.0`（完全匹配） |

返回值为 `f64`，`1.0` 表示完全相同，`0.0` 表示完全不同。

## 5. URL 修复引擎

### 5.1 流程总览

`fix_emoticon_url`（`src-tauri/src/vcp_modules/emoticon_manager.rs:311-387`）用于修复消息历史中失效的表情包 URL。其核心逻辑为：在本地库中寻找与原始 URL 语义最接近的条目，若相似度超过阈值则替换为新 URL。

```
┌─────────────────────────────────────────────────────────────────────┐
│                        fix_emoticon_url                             │
├─────────────────────────────────────────────────────────────────────┤
│  输入: original_src (失效的原始 URL)                                 │
│         │                                                           │
│         ▼                                                           │
│  1. 懒加载内存缓存  ──>  若为空则从 SQLite 加载                      │
│         │                                                           │
│         ▼                                                           │
│  2. 完全匹配检查                                                     │
│         ├── 对 original_src 和库中 url 均做 percent_decode           │
│         └── 若存在完全匹配  ──>  直接返回 original_src               │
│         │                                                           │
│         ▼                                                           │
│  3. 关键词过滤                                                       │
│         └── 若 decoded_original 不含 "表情包"  ──>  返回原 URL        │
│         │                                                           │
│         ▼                                                           │
│  4. 提取信息                                                         │
│         └── extract_emoticon_info(original_src)                     │
│             ├── search_filename (文件名)                             │
│             └── search_package  (包名，可能为 None)                   │
│         │                                                           │
│         ▼                                                           │
│  5. 遍历本地库，计算加权评分                                          │
│         ├── package_score  = get_similarity(search_package, item.cat)│
│         ├── filename_score = get_similarity(search_filename, item.file)│
│         └── score = 0.7 × package_score + 0.3 × filename_score      │
│         │                                                           │
│         ▼                                                           │
│  6. 阈值判断                                                         │
│         └── 若 highest_score > 0.6  ──>  返回最佳匹配项的 url        │
│         └── 否则  ──>  返回 original_src                             │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 完全匹配检查（decodeURIComponent 对齐）

为对齐前端 JavaScript 的 `decodeURIComponent` 行为，模块使用 `percent_decode_str`（`src-tauri/src/vcp_modules/emoticon_manager.rs:333`）对 URL 进行解码后再比较。这能避免以下假阴性：

| 编码形式 | 解码后 | 说明 |
|---------|--------|------|
| `%E5%8F%AF%E7%88%B1` | `可爱` | 中文 URL 编码差异 |
| `hello%20world` | `hello world` | 空格编码差异 |

若解码后的原始 URL 已存在于库中，说明该 URL 并未真正失效，直接原样返回。

### 5.3 关键词过滤

`fix_emoticon_url` 仅处理含有关键词 `"表情包"` 的 URL（`src-tauri/src/vcp_modules/emoticon_manager.rs:342-344`）。这是业务层面的快速短路：非表情包类的外部图片链接不应进入修复流程，防止误替换用户分享的其他网络图片。

### 5.4 extract_emoticon_info：从 URL 提取文件名和包名

`extract_emoticon_info`（`src-tauri/src/vcp_modules/emoticon_manager.rs:96-137`）负责从原始 URL 中逆向解析出语义信息，供后续模糊匹配使用。

#### 解析步骤

| 步骤 | 操作 | 示例 |
|------|------|------|
| 1. 提取路径 | 查找 `/images/` 或 `/pw=.../images/` 后的部分 | `包名/文件名.png?w=100` |
| 2. 去除参数 | 以 `?` 和 `#` 分割，取首段 | `包名/文件名.png` |
| 3. Percent Decode | `percent_decode_str` 解码 UTF-8 | `包名/文件名.png` |
| 4. 分割路径 | 按 `/` 分割并过滤空段 | `["包名", "文件名.png"]` |
| 5. 提取信息 | 倒数第一段为 `filename`，倒数第二段为 `package_name` | `("文件名.png", Some("包名"))` |

若路径段数不足 2，则 `package_name` 返回 `None`，仅保留 `filename`。

### 5.5 加权评分策略

修复引擎对每条本地库记录计算综合相似度分数（`src-tauri/src/vcp_modules/emoticon_manager.rs:356-370`）：

| 维度 | 权重 | 计算方式 | 特殊处理 |
|------|------|---------|---------|
| 包名相似度（Package Score） | 70% | `get_similarity(search_package, item.category)` | 若两者均为空字符串，计为 `1.0`；若仅原始缺失，计为 `0.5` |
| 文件名相似度（Filename Score） | 30% | `get_similarity(search_filename, item.filename)` | 无特殊处理 |
| **综合得分** | **100%** | `0.7 × package_score + 0.3 × filename_score` | — |

**设计意图**：包名（目录/分类）比文件名具有更强的区分度。例如 `"可爱/hello.png"` 与 `"工作/hello.png"` 文件名相同，但包名不同；高包名权重可有效避免跨包误匹配。

### 5.6 阈值判断

最终得分需超过阈值 `0.6` 才视为有效修复（`src-tauri/src/vcp_modules/emoticon_manager.rs:379-384`）：

| 条件 | 结果 |
|------|------|
| `highest_score > 0.6` | 返回 `best_match.url`（修复成功） |
| `highest_score <= 0.6` | 返回 `original_src`（放弃修复，保留原链接） |

阈值 `0.6` 是经验值，与前端 JavaScript 实现保持一致。该值在修复率与误修率之间取得平衡：过低的阈值会导致大量无关替换，过高的阈值则会让轻微重命名的表情包无法被修复。


## 6. 内存缓存与数据库加载策略

### 6.1 双层存储架构

表情包库采用**内存优先、数据库兜底**的双层存储策略：

```
┌─────────────────────────────────────┐
│         前端请求 get_emoticon_library │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  内存缓存 (EmoticonManagerState)     │
│  Arc<Mutex<Vec<EmoticonItem>>>      │
│  ├─ 非空 ? 直接返回克隆              │
│  └─ 为空 ? 进入数据库加载            │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  SQLite (emoticon_library 表)        │
│  SELECT id, category, filename,     │
│         url, search_key             │
│  └─ 加载完成后回填内存缓存            │
└─────────────────────────────────────┘
```

### 6.2 懒加载机制

`get_emoticon_library`（`src-tauri/src/vcp_modules/emoticon_manager.rs:295-309`）实现了懒加载：

1. 首次获取内存锁；
2. 若 `library` 非空，直接 `clone()` 返回；
3. 若为空，调用 `internal_load_library` 执行 SQL 查询；
4. 将查询结果写入内存锁，再返回给前端。

`fix_emoticon_url`（`src-tauri/src/vcp_modules/emoticon_manager.rs:317-324`）同样内置了相同的懒加载逻辑：若内存为空，先尝试从数据库加载一次，避免修复流程因缓存未命中而提前退出。

### 6.3 缓存一致性保证

| 操作 | 内存缓存行为 | 数据库行为 |
|------|-------------|-----------|
| `refresh_emoticon_library_internal` | 事务提交后**全量替换** `library` | `DELETE` + `INSERT` 批量写入 |
| `internal_load_library` | 仅读取，不修改 | `SELECT` 全表扫描 |
| `get_emoticon_library` | 命中则返回，未命中则加载后返回 | 仅在未命中时读取 |

> **注意**：当前模块未实现增量更新。每次远程同步都会清空并重建全库。对于表情包这种规模有限（通常数百至数千条）且变更低频的数据集，全量替换的简洁性优于增量合并的复杂度。

## 7. URL 编码规则

### 7.1 VCP_ENCODE_SET

`VCP_ENCODE_SET`（`src-tauri/src/vcp_modules/emoticon_manager.rs:12-21`）是一个静态的 `AsciiSet` 常量，用于 `percent_encoding::utf8_percent_encode`，其设计目标是**逐字节对齐 JavaScript 的 `encodeURIComponent`**。

JavaScript 的 `encodeURIComponent` 保留以下字符不编码：

```
A-Z a-z 0-9 - _ . ! ~ * ' ( )
```

Rust 的 `percent_encoding` crate 提供 `NON_ALPHANERIC`（编码所有非字母数字字符），模块在此基础上依次移除上述保留字符：

```rust
const VCP_ENCODE_SET: &AsciiSet = &NON_ALPHANUMERIC
    .remove(b'-')
    .remove(b'_')
    .remove(b'.')
    .remove(b'!')
    .remove(b'~')
    .remove(b'*')
    .remove(b'\'')
    .remove(b'(')
    .remove(b')');
```

### 7.2 编码对照表

| 字符 | `encodeURIComponent` | `VCP_ENCODE_SET` | 是否一致 |
|------|---------------------|------------------|---------|
| `中` | `%E4%B8%AD` | `%E4%B8%AD` | ✅ |
| `空格` | `%20` | `%20` | ✅ |
| `&` | `%26` | `%26` | ✅ |
| `/` | `%2F` | `%2F` | ✅ |
| `-` | `-` | `-` | ✅ |
