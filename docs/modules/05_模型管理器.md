# 05. 模型管理器 (Model Manager)

## 1. 概述

### 1.1 模块定位

模型管理器（`model_manager.rs`，272 行）是 **VCP Mobile** Rust 核心层中负责 **LLM（Large Language Model，大语言模型）元数据生命周期管理** 的专用模块。其职责边界严格限定为：

- 从 VCP 服务器拉取模型列表（OpenAI 兼容格式 `/v1/models`）
- 维护双层缓存体系（内存 + SQLite）以降低重复网络开销
- 提供模型收藏（Favorite）与使用统计（Usage Stats）的持久化能力
- 基于使用频次生成热模型排行，辅助前端默认模型推荐

该模块**不**涉及模型推理本身，**不**管理模型权重文件，仅处理模型 ID、所有者、创建时间等元数据。

### 1.2 与设置系统的关系

模型管理器对 `settings_manager` 存在单向依赖：

```
┌──────────────────┐      read_settings()      ┌──────────────────┐
│  model_manager   │ ─────────────────────────> │ settings_manager │
│   (本模块)        │     获取 vcp_server_url    │   (设置系统)      │
│                  │      与 vcp_api_key       │                  │
└──────────────────┘                           └──────────────────┘
```

- **输入依赖**：通过 `read_settings()` 读取用户配置的 VCP 服务器地址（`vcp_server_url`）与 API 密钥（`vcp_api_key`），用于构造认证请求。详见 `src-tauri/src/vcp_modules/settings_manager.rs`
- **输出存储**：将模型列表序列化为 JSON 后写入 `settings` 表的 `cached_models` 键，复用设置系统的数据库基础设施
- **独立表**：收藏（`model_favorites`）与统计（`model_usage_stats`）使用独立表，不侵入设置系统的键值空间

## 2. 核心数据结构

### 2.1 ModelInfo

模型元数据的序列化/反序列化载体，严格对应 OpenAI `/v1/models` 响应格式中的单个模型对象。

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `id` | `String` | API `data[].id` | 模型唯一标识，如 `"gpt-4o"`、`"claude-3-5-sonnet"` |
| `object` | `String` | API `data[].object` | 对象类型，固定为 `"model"` |
| `created` | `u64` | API `data[].created` | Unix 时间戳（秒），模型创建时间 |
| `owned_by` | `String` | API `data[].owned_by` | 提供方，如 `"openai"`、`"anthropic"` |

> 定义位置：`src-tauri/src/vcp_modules/model_manager.rs` 第 12–18 行

`ModelInfo` 实现 `Serialize` / `Deserialize`，可直接通过 `serde_json::from_value()` 从 API 响应反序列化，也可序列化为 JSON 字符串存入 SQLite。

### 2.2 ModelManagerState

Tauri Managed State（托管状态），以单例形式注入应用生命周期，供所有 Tauri Command 共享。

| 成员 | 类型 | 作用 |
|------|------|------|
| `cached_models` | `Arc<RwLock<Vec<ModelInfo>>>` | 内存级模型列表缓存，多线程安全读写锁 |

> 定义位置：`src-tauri/src/vcp_modules/model_manager.rs` 第 20–30 行

```
内存缓存结构示意
┌─────────────────────────────┐
│   ModelManagerState         │
│  ┌───────────────────────┐  │
│  │  Arc<RwLock<...>>     │  │
│  │  ┌─────────────────┐  │  │
│  │  │ Vec<ModelInfo>  │  │  │
│  │  │ [ModelInfo #0]  │  │  │
│  │  │ [ModelInfo #1]  │  │  │
│  │  │ ...             │  │  │
│  │  └─────────────────┘  │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
```

- `Arc` 提供多所有权，允许跨异步任务共享
- `RwLock` 区分读锁（`.read().await`，允许多个并发读）与写锁（`.write().await`，独占访问）
- 初始化时为空向量，首次调用 `get_cached_models` 或 `refresh_models` 时填充

### 3.2 URL 构造（从 `vcp_server_url` 解析基础 URL）

VCP 服务器地址在设置系统中以完整 URL 形式存储（例如 `https://api.vcp.example.com:8443/chat`）。`refresh_models` 需从中提取**协议 + 主机 + 端口**，拼接出 `/v1/models` 端点。

伪代码：

```
url_object = Url.parse(vcp_url)

port_str = if url_object.port() exists
              then ":" + port
              else ""

host_with_port = url_object.host_str() + port_str
base_url = url_object.scheme() + "://" + host_with_port

models_url = if base_url.ends_with('/')
                then base_url + "v1/models"
                else base_url + "/v1/models"
```

**关键约束**：
- 使用 `url::Url` 库进行结构化解析，避免字符串拼接导致的路径遍历或格式错误
- 保留显式端口（如 `:8443`），忽略默认端口（80/443）时 `url.port()` 返回 `None`
- 原 URL 的路径部分（如 `/chat`）被**完全丢弃**，因为 `/v1/models` 是独立的基础端点

### 3.3 API 请求与认证

| 属性 | 值 | 说明 |
|------|-----|------|
| HTTP 方法 | `GET` | |
| 端点 | `{base_url}/v1/models` | OpenAI 兼容格式 |
| 认证头 | `Authorization: Bearer {vcp_api_key}` | API 密钥直接作为 Bearer Token |
| 超时 | 10 秒 | 移动端弱网场景下防止无限挂起 |
| 客户端 | `reqwest::Client` | 共享默认配置，单次请求后丢弃 |

### 3.4 响应解析与过滤

API 返回的 JSON 结构如下（OpenAI 兼容）：

```json
{
  "object": "list",
  "data": [
    { "id": "gpt-4o", "object": "model", "created": 1715367049, "owned_by": "openai" },
    ...
  ]
}
```

解析策略：

1. 先反序列化为 `serde_json::Value`
2. 提取 `data` 字段并确认为数组
3. 对数组每个元素执行 `serde_json::from_value::<ModelInfo>()`
4. 使用 `filter_map` 跳过无法解析的条目（容错设计，单条格式错误不导致整体失败）

### 3.5 双层缓存更新

模型列表解析成功后，同步执行两层缓存更新：

```
        解析成功
           │
           ▼
    ┌──────────────┐
    │  内存缓存    │──> *state.cached_models.write().await = models.clone()
    │  (RwLock)    │    立即生效，后续 get_cached_models 直接命中
    └──────────────┘
           │
           ▼
    ┌──────────────┐
    │  SQLite 缓存 │──> UPSERT settings SET value = json_str, updated_at = now
    │  (settings)  │    WHERE key = 'cached_models'
    └──────────────┘
```

| 层级 | 存储形式 | 生命周期 | 优势 | 劣势 |
|------|----------|----------|------|------|
| 内存缓存 | `Vec<ModelInfo>` | 应用进程存活期 | 零 I/O、微秒级读取 | 进程结束即丢失 |
| SQLite 缓存 | JSON 字符串 | 持久化 | 冷启动可用、跨会话 | 需要序列化/反序列化 |

SQLite 写入使用 `ON CONFLICT(key) DO UPDATE`（UPSERT）语义，`updated_at` 字段记录毫秒级 Unix 时间戳，便于未来实现缓存过期策略。

## 4. 收藏系统（`toggle_favorite_model`）

收藏系统允许用户将常用模型标记为收藏，前端通常以星标图标或置顶列表形式展示。

> 函数位置：`src-tauri/src/vcp_modules/model_manager.rs` 第 197–239 行

### 4.1 幂等切换逻辑

`toggle_favorite_model` 采用**查询后决定操作**的幂等（Idempotent）设计：传入同一个 `model_id` 多次调用，效果在"收藏"与"取消收藏"之间切换，不会产生重复记录或报错。

```
输入: model_id
         │
         ▼
    开启数据库事务 (BEGIN)
         │
         ▼
    SELECT model_id FROM model_favorites WHERE model_id = ?
         │
    ┌────┴────┐
    ▼         ▼
  存在      不存在
    │         │
    ▼         ▼
 DELETE    INSERT
    │         │
    ▼         ▼
 返回 false 返回 true
    │         │
    └────┬────┘
         ▼
    提交事务 (COMMIT)
```

- **返回 `true`**：模型被收藏（执行了 INSERT）
- **返回 `false`**：模型被取消收藏（执行了 DELETE）

### 4.2 数据库事务

整个操作包裹在显式事务（`pool.begin()` → `tx.commit()`）中，确保查询与写入的原子性：

| 步骤 | SQL | 说明 |
|------|-----|------|
| 1. 查询 | `SELECT model_id FROM model_favorites WHERE model_id = ?` | 判断当前状态 |
| 2a. 删除 | `DELETE FROM model_favorites WHERE model_id = ?` | 已存在时移除 |
| 2b. 插入 | `INSERT INTO model_favorites (model_id, created_at) VALUES (?, ?)` | 不存在时添加，记录毫秒时间戳 |
| 3. 提交 | `COMMIT` | 原子生效 |

> 事务的使用消除了并发场景下"重复收藏"的竞态条件（Race Condition）。

## 5. 使用统计（`record_model_usage`）

每次用户发送消息并选定模型后，前端通过 Tauri Command 调用 `record_model_usage`，累加该模型的使用计数。

> 函数位置：`src-tauri/src/vcp_modules/model_manager.rs` 第 241–267 行

### 5.1 UPSERT 逻辑

使用 SQLite 的 `ON CONFLICT(...)` 实现**插入或更新**（UPSERT）：

```sql
INSERT INTO model_usage_stats (model_id, usage_count, updated_at)
VALUES (?, 1, ?)
ON CONFLICT(model_id) DO UPDATE SET
    usage_count = usage_count + 1,
    updated_at = excluded.updated_at
```

| 场景 | 执行结果 |
|------|----------|
| 该模型首次使用 | 插入新行，`usage_count = 1` |
| 该模型已有记录 | 更新原有行，`usage_count` 自增 1，`updated_at` 刷新 |

### 5.2 时间戳语义

`updated_at` 字段记录的是**最后一次使用时间**，而非首次使用时间。这允许 `get_hot_models` 在同等频次下优先展示最近活跃的模型（当前实现仅按 `usage_count` 排序，但 Schema 保留了扩展空间）。

### 5.3 调用时机

`record_model_usage` 由前端在**消息成功发送后**异步调用，不与网络请求阻塞在同一路径上。即使统计写入失败，也不会影响聊天核心流程。

## 6. 热模型排行（`get_hot_models`）

热模型排行基于 `model_usage_stats` 表的 `usage_count` 字段降序排列，返回使用频率最高的模型 ID 列表。前端可利用此数据实现"常用模型快捷入口"或"默认模型智能推荐"。

> 函数位置：`src-tauri/src/vcp_modules/model_manager.rs` 第 150–173 行

### 6.1 查询逻辑

```sql
SELECT model_id
FROM model_usage_stats
ORDER BY usage_count DESC
LIMIT ?
```

| 参数 | 类型 | 说明 |
|------|------|------|
| `limit` | `usize`（绑定为 `i64`） | 返回的最大条数，由前端传入 |

### 6.2 结果特征

- **纯 ID 列表**：返回 `Vec<String>`，不包含使用次数。前端如需展示次数，需额外调用其他接口或自行维护映射
- **无数据时返回空数组**：不会报错，前端应做好空态处理
- **排序稳定**：相同 `usage_count` 的模型顺序取决于 SQLite 默认排序规则（通常按插入顺序或 RowID）

## 7. 缓存读取策略（`get_cached_models`）

`get_cached_models` 是前端获取模型列表的主要入口，实现**内存优先、数据库回退**的分级缓存策略。

> 函数位置：`src-tauri/src/vcp_modules/model_manager.rs` 第 32–63 行

### 7.1 读取流程（ASCII 图）

```
前端调用 get_cached_models
         │
         ▼
┌─────────────────────┐
│ 1. 读取内存缓存      │──> state.cached_models.read().await
│    (RwLock 读锁)     │
└──────────┬──────────┘
           │
      ┌────┴────┐
      ▼         ▼
   非空        为空
    │           │
    ▼           ▼
 直接返回    ┌─────────────────────┐
 克隆结果    │ 2. 查询 SQLite      │──> SELECT value FROM settings
            │    (settings 表)     │    WHERE key = 'cached_models'
            └──────────┬──────────┘
                       │
                  ┌────┴────┐
                  ▼         ▼
               有记录      无记录
                │           │
                ▼           ▼
        ┌─────────────┐  返回空数组
        │ JSON 反序列化 │
        │ 为 Vec<ModelInfo>│
        └──────┬──────┘
               │
               ▼
        ┌─────────────┐
        │ 回写内存缓存 │──> 防止下次重复读取 DB
        │ (RwLock 写锁)│
        └──────┬──────┘
               │
               ▼
          返回结果
```

### 7.2 策略要点

| 优先级 | 来源 | 触发条件 | 副作用 |
|--------|------|----------|--------|
| L1 | 内存 (`RwLock`) | 始终先尝试 | 无 |
| L2 | SQLite (`settings`) | 内存为空时 | 成功后将数据回填内存，提升下次命中率 |
| L3 | 空数组 | 两层均未命中 | 无 |

**回写机制（Write-Behind）**：当从 SQLite 成功反序列化模型列表后，立即通过写锁将数据写入 `cached_models`，确保后续读取直接命中内存，避免重复的 DB I/O。

## 8. 数据库 Schema

模型管理器依赖两张独立业务表（`model_favorites`、`model_usage_stats`）以及设置系统的 `settings` 表作为缓存载体。

### 8.1 `model_favorites`（收藏表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `model_id` | `TEXT` | `PRIMARY KEY` | 模型唯一标识 |
| `created_at` | `INTEGER` | | 收藏时间，毫秒级 Unix 时间戳 |

### 8.2 `model_usage_stats`（使用统计表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `model_id` | `TEXT` | `PRIMARY KEY` | 模型唯一标识 |
| `usage_count` | `INTEGER` | | 累计使用次数，默认从 1 开始 |
| `updated_at` | `INTEGER` | | 最后使用时间，毫秒级 Unix 时间戳 |

### 8.3 `settings` 表（缓存载体）

`settings` 表由设置系统统一管理，模型管理器仅使用其中一条记录：

| key | value | 说明 |
|-----|-------|------|
| `cached_models` | JSON 字符串（`Vec<ModelInfo>` 序列化结果） | 模型列表的持久化缓存 |

> 表定义位置：由 `db_manager.rs` 在应用初始化时执行 `CREATE TABLE IF NOT EXISTS`，详见 `src-tauri/src/vcp_modules/db_manager.rs`

## 10. 设计决策与注意事项

### 10.1 为何使用 `settings` 表而非独立表缓存模型列表？

模型列表是**全量替换**的键值型数据（一整份 JSON），天然适合键值存储。复用 `settings` 表减少了 Schema 复杂度，同时利用 `ON CONFLICT` UPSERT 语义实现简洁的更新逻辑。

### 10.2 为何 `init_model_manager` 为空实现？

收藏与统计数据存储于独立表中，无需在应用启动时加载到内存。模型列表本身通过 `get_cached_models` 的懒加载（Lazy Loading）机制按需回填内存。因此初始化函数无需预热操作，保留为扩展钩子。

### 10.3 错误处理策略

所有 Tauri Command 统一返回 `Result<T, String>`，错误信息以人类可读中文或英文描述传递至前端。网络层错误（超时、DNS 失败、HTTP 非 2xx）均被捕获并转换为描述性字符串，避免前端收到 Rust 内部 panic 信息。

### 10.4 并发安全

- `ModelManagerState` 通过 `Arc<RwLock<...>>` 实现 Send + Sync，可在多线程异步运行时中安全共享
- `toggle_favorite_model` 使用显式数据库事务，避免并发切换时出现状态不一致
- `record_model_usage` 的 UPSERT 操作依赖 SQLite 的行级冲突处理，无需应用层加锁
