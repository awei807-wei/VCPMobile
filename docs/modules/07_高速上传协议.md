# 高速上传协议（High-Speed Upload Protocol）

> 源文件：`src-tauri/src/vcp_modules/infra/high_speed_channel.rs`（213 行）

---

## 1. 概述

### 1.1 模块定位

`high_speed_channel.rs` 实现了一条**高速本地上传链路**（High-Speed Local Upload Link），用于前端通过 HTTP 直接向 Rust 后端传输大文件（如聊天附件）。该模块是 Tauri IPC 机制之外的**补充通道**：当文件体积较大时，避免了 Base64 编解码和 JSON 序列化带来的内存与性能开销。

### 1.2 为什么需要高速上传链路

在常规 Tauri 调用中，前端通过 `invoke` 将文件以 Base64 字符串形式传入 Rust，存在以下瓶颈：

| 瓶颈 | 说明 |
|------|------|
| 内存膨胀 | Base64 编码使体积增加约 33%，大文件易导致前端 OOM |
| IPC 序列化开销 | JSON 承载兆级字符串效率极低 |
| 缺乏流式能力 | 必须等待完整文件进入 Rust 进程后才能落盘 |

高速上传链路通过在前端与 Rust 之间建立**临时本地 TCP 服务器**，让前端以原生 `fetch`/`XMLHttpRequest` 直接发送二进制流，Rust 则边读边写，实现真正的**零拷贝流式落盘**。

在实际项目中，该链路的调用存在明确的平台差异与场景分工：

**a) 当前实际使用场景**

- **桌面端（非 Android）**：大文件（≥2 MB）的主力上传链路。前端 `attachmentStore.ts` 中明确以 2 MB 为界分流：小文件（<2 MB）走 `store_file` IPC，大文件走 `prepare_vcp_upload` 高速 TCP 链路。
- **Android 端**：`prepare_vcp_upload` 当前**未被调用**（即用户所称"轮空"）。Android 端主链路是 `plugin:vcp-mobile|pick_file`（原生文件选择器）→ `register_local_file`（零拷贝路径传输）。

**b) Android 端轮空原因**

Android 10+ 引入的 Scoped Storage 限制使得前端 HTML `<input>` 无法获得可靠物理路径。原生文件选择器（`Intent.ACTION_GET_CONTENT`）通过 Kotlin 侧流式拷贝到 app cache，再经 `register_local_file` 零拷贝入库，效率更高且规避了 Tauri IPC 的大文件 Base64 问题。因此高速 TCP 链路在 Android 端当前没有调用入口。

**c) 高价值定位：覆盖非路径式文件上传**

虽然 Android 端当前未启用该链路，但它保留了重要的战略价值——覆盖 Android 端**无法获得物理路径的文件上传场景**：

- 剪贴板粘贴图片（生成 Blob，无路径）
- 相机拍照直接生成的内存图像（无路径）
- 应用内部生成的临时文件（如录音、截图）

这些场景无法通过 `Intent.ACTION_GET_CONTENT` 获得物理路径，必须通过二进制流传输，高速链路是唯一可行的后端接收通道。

### 1.3 与常规 file_manager 上传的区别

| 维度 | 常规 file_manager 上传 | 高速上传链路（本模块） |
|------|----------------------|----------------------|
| 传输通道 | Tauri `invoke` IPC | HTTP POST 到 127.0.0.1 临时端口 |
| 数据格式 | Base64 JSON 负载 | 原始二进制流（Raw Binary Stream） |
| 适用场景 | 小图标、剪贴板片段 | 大附件、图片、视频 |
| 桌面端分流策略 | 小文件（<2 MB）走 `store_file` IPC | 大文件（≥2 MB）走高速 TCP 链路 |
| Android 端分流策略 | 小文件 `store_file` IPC；大文件走原生 `pick_file` → `register_local_file`（零拷贝） | 当前轮空；预留覆盖非路径式文件（剪贴板、相机、录音等） |
| 流式支持 | 否（全量内存缓冲） | 是（64 KiB 循环缓冲） |
| 鉴权方式 | Tauri 内置安全模型 | 一次性 UUID Token |
| 生命周期 | 同步调用 | 异步后台任务（20 秒超时） |

## 3. 高速上传协议时序图

整个协议分为三个阶段：**准备阶段**、**传输阶段**、**收尾阶段**。

```
  前端 (Vue)                          Rust 后端 (Tauri)
    |                                       |
    |  ① prepare_vcp_upload(metadata)       |
    | ------------------------------------> |
    |                                       |  启动 TcpListener (127.0.0.1:0)
    |                                       |  生成 UUID token
    |                                       |  spawn 后台异步任务
    |     UploadEndpoint {url, token}       |
    | <------------------------------------ |
    |                                       |
    |  ② fetch(url, {method: 'POST',       |
    |              body: FileStream})       |
    | ====================================> |  接受 TCP 连接
    |                                       |  解析 HTTP 头部
    |                                       |  流式写入临时文件
    |                                       |  计算 SHA-256
    |                                       |
    |  ← OPTIONS 预检（若跨域）             |
    | <==================================== |  返回 204 No Content
    |                                       |
    |  ②' 再次 POST 二进制流               |
    | ====================================> |
    |                                       |  接收完成 → finalize
    |     HTTP/1.1 200 OK {AttachmentData}  |
    | <==================================== |
    |                                       |
    |  ③ 前端拿到 attachment id，插入消息   |
    |                                       |
```

### 3.1 准备阶段：prepare_vcp_upload

前端通过 Tauri `invoke('prepare_vcp_upload', {metadata})` 触发：

1. **绑定随机端口**：`TcpListener::bind("127.0.0.1:0")`，操作系统分配临时端口（ephemeral port）
2. **生成 Token**：`uuid::Uuid::new_v4().to_string()`，作为一次性凭证
3. **派生长效任务**：通过 `tauri::async_runtime::spawn` 启动后台协程，持续监听 20 秒
4. **立即返回**：将 `url` 和 `token` 返回前端，不阻塞主线程

### 3.2 传输阶段：前端 HTTP POST → 后端流式接收

前端获得 `UploadEndpoint` 后，使用标准 Web API 发送文件：

```
POST http://127.0.0.1:<随机端口>/
Content-Type: <mime>
[原始二进制请求体]
```

Rust 后台任务进入 `while` 循环，以 **500 ms 为周期**轮询 `listener.accept()`，直到：
- 成功接收一次完整上传（`upload_finished = true`），或
- 20 秒超时（`start_time.elapsed() >= timeout`）

### 3.3 收尾阶段：finalize_high_speed_upload

当接收字节数 `bytes_count >= metadata.size` 时，调用 `finalize_high_speed_upload`：

1. 提取原始扩展名（extension）
2. 以 SHA-256 哈希值作为 `internal_name`
3. 将临时文件移动到附件根目录
4. 调用 `register_attachment_internal` 写入数据库
5. 将 `AttachmentData` 以 JSON 形式写入 HTTP 响应体返回前端

## 4. TCP 服务器实现详解

模块并未使用完整的 HTTP 框架（如 axum、hyper），而是基于 `tokio::net::TcpListener` 实现了一个**极简单连接 HTTP 处理器**。这种设计的优势是零依赖、极低开销、生命周期完全受控。

### 4.1 随机端口绑定

```pseudo
listener = TcpListener::bind("127.0.0.1:0")
port     = listener.local_addr().port
url      = "http://127.0.0.1:" + port
```

- 地址 `127.0.0.1` 强制限定为**本地回环**（loopback），外部网络无法访问
- 端口 `0` 表示由操作系统自动分配，避免端口冲突
- 端口信息随 `UploadEndpoint` 暴露给前端，生命周期与后台任务绑定

### 4.2 超时循环（20 秒、500 ms 轮询）

```pseudo
upload_finished = false
timeout         = 20s
start_time      = now()

while !upload_finished && elapsed < timeout:
    result = timeout(500ms, listener.accept())
    if result 为超时或错误:
        continue
    (socket, addr) = result
    // 进入单连接处理逻辑
```

| 参数 | 值 | 设计意图 |
|------|-----|---------|
| 总超时 | 20 秒 | 防止后台任务无限期挂起，覆盖绝大多数 4G/5G 单文件上传场景 |
| 轮询间隔 | 500 ms | 平衡 CPU 占用与连接响应延迟 |
| 循环退出条件 | `upload_finished == true` | 仅当一次完整上传成功后才终止，拒绝重复上传 |

> 实现位置：`src-tauri/src/vcp_modules/infra/high_speed_channel.rs` 第 39–51 行

### 4.3 HTTP 头部解析（\r\n\r\n 分隔）

模块采用**状态机方式**解析 HTTP 请求：

```pseudo
buffer       = [0u8; 65536]   // 64 KiB 读取缓冲
body_started = false
header_data  = Vec::new()     // 累积头部字节

loop:
    n = socket.read(buffer)
    if n == 0: break           // 连接关闭

    if !body_started:
        header_data.extend(buffer[..n])
        pos = header_data.find("\r\n\r\n")   // HTTP 头部分隔符
        if pos 找到:
            body_started = true
            header_str = header_data[..pos] 转为字符串
            // 进入头部解析分支
```

- 使用 `Vec::with_capacity(4096)` 预分配头部缓冲，避免小量 realloc
- `header_data.windows(4).position(|w| w == b"\r\n\r\n")` 是标准的 HTTP 头部终止探测模式

### 4.4 CORS 预检响应

前端 `fetch` 跨域请求到 `127.0.0.1` 时，浏览器会自动发送 **OPTIONS 预检请求**（Preflight Request）。模块对此进行显式处理：

```pseudo
if header_str.starts_with("OPTIONS"):
    is_options = true
    response =
        "HTTP/1.1 204 No Content\r\n" +
        "Access-Control-Allow-Origin: *\r\n" +
        "Access-Control-Allow-Methods: POST, OPTIONS\r\n" +
        "Access-Control-Allow-Headers: *\r\n" +
        "Access-Control-Max-Age: 86400\r\n" +
        "Connection: close\r\n\r\n"
    socket.write_all(response)
    break   // 关闭连接，等待浏览器正式 POST
```

| 响应头部 | 值 | 说明 |
|---------|-----|------|
| `Access-Control-Allow-Origin` | `*` | 允许任意源（安全前提下本地回环仅前端可访问） |
| `Access-Control-Allow-Methods` | `POST, OPTIONS` | 显式允许上传方法与预检方法 |
| `Access-Control-Allow-Headers` | `*` | 允许前端携带自定义头部（如 Content-Type） |
| `Access-Control-Max-Age` | `86400` | 缓存预检结果 24 小时，减少重复 OPTIONS 请求 |
| `Connection` | `close` | 单次连接即关闭，不保持长连接 |

### 4.5 流式 body 写入临时文件

当头部解析完成且请求方法为 POST 时，模块立即创建临时文件，并将后续所有读取到的数据直接写入：

```pseudo
file = File::create(temp_file_path)   // 位于 app_cache_dir/uploads/<session_id>.tmp

loop 继续读取:
    data = buffer[..n]                 // 或扣除头部后的剩余数据
    if data 非空:
        file.write_all(data)           // 异步写入磁盘
        hasher.update(data)            // 同步更新 SHA-256
        bytes_count += data.len()
        if bytes_count >= metadata.size:
            break
```

- **64 KiB 循环缓冲**：避免大文件全量进入内存
- **头部数据拆分逻辑**：若头部跨越多个 `read` 调用，需要精确计算当前缓冲中 body 的起始偏移（`header_len_in_current_buffer`）
- **边收边写边算哈希**：接收、落盘、哈希三操作在单次循环内完成，无二次遍历

> 实现位置：`src-tauri/src/vcp_modules/infra/high_speed_channel.rs` 第 72–124 行

## 5. SHA-256 校验与文件落盘

### 5.1 增量哈希计算

模块在接收数据的同时，使用 `sha2::Sha256` 进行**增量哈希**（Incremental Hashing）：

```pseudo
hasher = Sha256::new()

// 每次收到数据块:
hasher.update(data_chunk)

// 传输完成后:
hash = hex::encode(hasher.finalize())
```

- 不依赖文件落盘后再读取计算，减少一次 I/O
- 最终哈希值同时作为文件名和数据库 `hash` 字段，用于去重与完整性校验

### 5.2 finalize_high_speed_upload 落盘逻辑

```pseudo
function finalize_high_speed_upload(app_handle, pool, temp_path, metadata, hash, size):
    ext  = metadata.name 的扩展名（若无则为空串）
    internal_name = (ext 空) ? hash : hash + "." + ext
    dest = get_attachments_root_dir(app_handle)
    dest_path = dest / internal_name

    if dest_path 不存在:
        rename(temp_path, dest_path)    // 原子移动
    else:
        remove(temp_path)                // 哈希冲突 = 相同文件，直接删除临时文件

    return register_attachment_internal(
        app_handle, pool, hash,
        metadata.name, metadata.mime, size, dest_path
    )
```

| 步骤 | 说明 |
|------|------|
| 扩展名保留 | 从原始文件名提取扩展名，附加到哈希文件名后，便于操作系统识别类型 |
| 原子移动 | 使用 `fs::rename`（同分区下为原子操作），避免落盘过程中断导致文件损坏 |
| 哈希去重 | 若目标路径已存在，说明该文件此前已上传，直接删除临时文件，不重复存储 |

> 实现位置：`src-tauri/src/vcp_modules/infra/high_speed_channel.rs` 第 173–213 行

## 6. 与 file_manager.rs 的协作

### 6.1 get_attachments_root_dir

```pseudo
pub fn get_attachments_root_dir<R: Runtime>(
```

### 6.4 与 `register_local_file` 的对比

`register_local_file` 是 Android 端原生文件选择器链路的核心后端接口，与高速上传链路在定位上形成互补。以下从六个维度进行对比：

| 维度 | 高速上传链路 (`high_speed_channel`) | 零拷贝路径传输 (`register_local_file`) |
|------|-----------------------------------|----------------------------------------|
| 适用平台 | 桌面端大文件主力 / Android 非路径式文件备用 | Android 原生选择器文件主力 |
| 传输方式 | HTTP POST 原始二进制流 | 物理文件路径传递（无数据传输） |
| 前端调用 | `fetch` / `XMLHttpRequest` POST | `invoke('register_local_file')` |
| Rust 侧动作 | 启动 TCP 服务器，流式接收，计算 hash | 校验路径，`rename` 移动，计算 hash |
| 内存开销 | 64 KiB 循环缓冲 | 64 KiB 流式读取缓冲（仅用于 hash） |
| Base64 编码 | 无 | 无 |

## 7. 安全考量

### 7.1 本地回环绑定

- 监听地址硬编码为 `127.0.0.1`，不绑定 `0.0.0.0`
- 外部设备、局域网内其他应用均无法连接到临时上传端口
- 即使端口信息通过前端 JavaScript 泄露，攻击者也需在同一设备上才能利用

### 7.2 一次性 Token

- `prepare_vcp_upload` 生成 UUID v4 Token，与 `UploadEndpoint` 一并返回
- 当前版本 Token 仅生成，实际请求头中由前端通过 `X-Upload-Token` 携带，但后端处理逻辑中**尚未显式校验**该 Token
- 建议后续增强：后端在接收 HTTP 头部后显式比对 `X-Upload-Token` 或要求 `Authorization: Bearer <token>`，校验通过后才进入 body 接收逻辑

### 7.3 超时保护

| 超时类型 | 时长 | 保护目标 |
|---------|------|---------|
| 总生命周期 | 20 秒 | 防止后台协程永久挂起，泄漏 TCP socket 与内存 |
| accept 轮询 | 500 ms | 允许任务定期检查超时标志，优雅退出 |

- 超时后 `listener` 随后台任务 drop 自动关闭
- 若前端在超时后才发起 POST，连接将被拒绝（OS 层面端口已释放）

### 7.4 临时文件清理

| 场景 | 清理行为 |
|------|---------|
| 接收不完整（`bytes_count < metadata.size`） | 删除临时文件，返回 `400 Bad Request` |
| 目标文件已存在（哈希去重） | 删除临时文件，不重复存储 |
| 上传成功 | `rename` 移动临时文件至附件目录，原子完成 |

- 临时目录位于 `app_cache_dir/uploads/`，即使极端情况下未清理，也处于应用私有缓存范围内

### 7.5 CORS 响应的合理性

- `Access-Control-Allow-Origin: *` 看似宽松，但结合 `127.0.0.1` 绑定，实际暴露面仅限于本机
- 若应用未来支持 WebView2 等可被外部网页注入的场景，建议将 `*` 收窄为前端 WebView 实际 Origin
