# 11_多模态媒体处理器（Media Processor）

## 1. 概述

### 1.1 模块定位

`media_processor/` 是 VCP Mobile 核心层的**多模态输入预处理子系统**，位于 `src-tauri/src/vcp_modules/infra/media_processor/`（4 个 Rust 文件：`mod.rs` 模块入口 + 3 个提取器）。该子系统在 VCP 请求发出之前执行，负责将用户选择的本地文件（图片、视频、音频）转换为大型语言模型（LLM）可消费的**Base64 data URL**格式，并执行必要的压缩、抽帧、转码以降低 payload 体积。

> **架构变更**：原 `ffmpeg_cli.rs` 及嵌入的 `assets/ffmpeg_aarch64` / `assets/ffprobe_aarch64` 二进制文件已删除（约 40MB APK 体积缩减），`assets/` 目录整体移除。媒体处理现采用**降级策略**（Degradation Strategy）：通过 `run_mobile_plugin` 桥接 Kotlin 层，调用 Android 原生 API（`processImage`/`processVideo`/`processAudio`）进行硬件加速处理；若 Android 原生不可用，则检查文件大小限制后直接读取原始字节；超出限制时返回错误文本描述，不阻断请求流程。

### 1.2 职责边界

| 职责领域 | 具体行为 | 对应源码文件 |
|---------|---------|------------|
| 图片预处理 | 3 层降级：Kotlin processImage → 直接读取（<5MB） → 报错 | `image_extractor.rs` |
| 视频预处理 | Android 专属：Kotlin processVideo 抽帧（1280x720/300帧/18MB截断），无降级路径 | `video_extractor.rs` |
| 音频预处理 | 3 层降级：Kotlin processAudio → 直接读取（<5MB） → 报错 | `audio_extractor.rs` |

### 1.3 调用入口

```text
Vue 3 前端（附件选择器）
    ↓ 用户选择本地文件，生成 {type: "local_file", path: "file://..."}
    ↓ Tauri IPC invoke: sendToVCP
vcp_client.rs
    ↓ perform_vcp_request 阶段 0（数据验证与规范化）
    ↓ 识别 local_file 类型，按扩展名分发
    ↓ 通过 tokio::task::spawn_blocking 调用各提取器
    ├─→ image_extractor::convert_local_image_for_multimodal
    ├─→ video_extractor::process_video_for_multimodal
    └─→ audio_extractor::process_audio_for_multimodal
         ↓ 各提取器独立执行降级策略
              ├─ 1. Tier 1: run_mobile_plugin(processImage/processVideo/processAudio)
              │      → Kotlin 层 Android 原生 API (BitmapFactory/MediaCodec/MediaExtractor)
              ├─ 2. Tier 2: 直接读取文件（需检查大小限制: 图片/音频 < 5MB）
              │      → 原始字节 → base64 data URL
              └─ 3. Tier 3: 超限或失败 → 返回错误文本占位，不阻断请求流程
                  注：视频仅支持 Tier 1（Android 专属），非 Android 直接返回错误
```

## 2. 公共接口

### 2.1 image_extractor

```rust
pub fn convert_local_image_for_multimodal<R: Runtime>(
    app: &AppHandle<R>,
    path: &Path,
) -> Result<String, String>
```

- **输入**：`AppHandle` 引用（用于获取 VcpMobileState 插件句柄）+ 本地图片文件路径
- **输出**：`data:image/webp;base64,{...}`（Android 原生处理后为 WebP）或 `data:image/{format};base64,{...}`（降级直接读取时保留原始格式）或错误文本占位
- **处理流程（降级策略）**：
  1. **Tier 1 — Android 原生处理**：通过 `run_mobile_plugin("processImage", { path })` 调用 Kotlin 层 `VcpMobilePlugin.processImage()`，内部使用 `BitmapFactory` 硬件解码 + 保真缩放，输出压缩 WebP 临时文件 → 读取后立即删除临时文件 → base64 编码返回
  2. **Tier 2 — 直接文件读取**（Tier 1 失败或非 Android 平台）：检查文件大小 < 5MB，直接读取原始字节 → base64 编码（按扩展名推断 MIME 子类型：png/webp/gif/bmp/x-icon/svg+xml/avif/heic，默认 jpeg）
  3. **Tier 3 — 错误文本占位**（文件 ≥ 5MB 或所有方式均失败）：返回 `"[图片附件: {path}]"` 错误文本

### 2.2 video_extractor

```rust
pub fn process_video_for_multimodal<R: Runtime>(
    app: &AppHandle<R>,
    path: &Path,
) -> Result<Vec<String>, String>
```

- **输入**：`AppHandle` 引用 + 本地视频文件路径
- **输出**：`Vec<data:image/jpeg;base64,{...}>` 帧序列列表（每帧一个 data URL）或错误
- **处理流程**（详见 §3.2）：Android 专属——通过 `run_mobile_plugin("processVideo", { path })` 调用 Kotlin 层高并发异步抽帧（1280x720 包络、步长采样、降采样截断 300 帧上限），返回 JPEG 帧临时文件列表 → 逐帧 base64 编码，18MB Base64 字符硬截断 → 边处理边物理清理临时帧文件及父目录。非 Android 平台直接返回错误，不支持视频处理。

### 2.3 audio_extractor

```rust
pub fn process_audio_for_multimodal<R: Runtime>(
    app: &AppHandle<R>,
    path: &Path,
) -> Result<String, String>
```

- **输入**：`AppHandle` 引用 + 本地音频文件路径
- **输出**：`data:audio/aac;base64,{...}`（Android 原生处理后为 AAC）或 `data:audio/{format};base64,{...}`（降级直接读取时保留原始格式）或错误文本占位
- **处理流程（降级策略）**：
  1. **Tier 1 — Android 原生处理**：通过 `run_mobile_plugin("processAudio", { path })` 调用 Kotlin 层 `VcpMobilePlugin.processAudio()`，内部使用 `MediaExtractor` + `MediaCodec` 进行 PCM 解码 → 重编码为 16kHz 单声道 32kbps AAC → 写入临时文件 → 读取后立即删除临时文件 → base64 编码返回
  2. **Tier 2 — 直接文件读取**（Tier 1 失败或非 Android 平台）：检查文件大小 < 5MB，直接读取原始字节 → base64 编码（按扩展名推断 MIME：audio/mpeg、audio/wav、audio/ogg、audio/flac、audio/aac、audio/mp4、audio/opus，默认 audio/mpeg）
  3. **Tier 3 — 错误文本占位**（文件 ≥ 5MB 或所有方式均失败）：返回 `"[音频附件: {path}]"` 错误文本

## 3. 核心算法详解

### 3.1 降级策略（Degradation Strategy）

> **架构变更**：原 `ffmpeg_cli.rs` 及嵌入的 `ffmpeg_aarch64` / `ffprobe_aarch64` 二进制文件已删除。图片和音频处理采用三层降级策略；视频处理为 Android 专属——仅通过 Kotlin `processVideo` 通道处理，非 Android 平台直接报错，无降级路径。

#### 三层降级架构

> **适用范围**：图片（image_extractor）和音频（audio_extractor）遵循此 3 层降级架构。视频（video_extractor）仅走 Tier 1（Android 专属 Kotlin 通道），非 Android 直接返回错误，不存在 Tier 2/3 降级路径。

```
媒体文件输入（图片 / 音频）
       │
       ▼
┌──────────────────────────────────┐
│ 1. Android 原生解码               │ ──成功──► 输出 base64 data URL
│ run_mobile_plugin(processImage/  │              (图片→WebP, 音频→AAC)
│   processAudio) → Kotlin 层      │
│ (BitmapFactory / MediaCodec /    │
│  MediaExtractor)                 │
└──────────┬───────────────────────┘
    失败/不可用
           ▼
┌──────────────────────────────────┐
│ 2. 直接文件读取                   │ ──文件<5MB──► 输出 base64 data URL
│ (原始字节 → base64)               │              (保留原始格式)
└──────────┬───────────────────────┘
    超限 (≥5MB) / 失败
           ▼
┌──────────────────────────────────┐
│ 3. 错误文本占位                   │ ──► 返回 "[{类型}附件: {path}]"
│ (不阻断请求流程)                  │
└──────────────────────────────────┘
```

#### 各媒体类型的大小限制

| 媒体类型 | 直接读取上限 | 说明 |
|---------|------------|------|
| 图片 | 5 MB | 中小型图片可直接嵌入 payload |
| 视频 | 不支持直接读取 | 视频仅通过 Kotlin `processVideo` 处理（Android 专属），非 Android 平台直接返回错误，无文件直读降级路径 |
| 音频 | 5 MB | 完整短音频可嵌入 |

#### 降级行为表

| 场景 | 行为 | 对请求的影响 |
|------|------|------------|
| Android 原生解码成功 | 返回高质量 base64 data URL（图片→WebP，音频→AAC 16kHz 32kbps 单声道，视频→JPEG 帧序列） | 正常 |
| Android 原生不可用，文件 < 5MB（图片/音频） | 直接读取原始字节，base64 编码后返回 | 正常（无转码/压缩） |
| Android 原生不可用，文件 >= 5MB（图片/音频） | 返回 `"[附件文件: {path}]"` 文本占位 | 文件路径保留，LLM 可感知文件存在但无法访问内容 |
| 视频且非 Android 平台 | 直接返回错误 `"非 Android 物理端不支持视频多模态抽帧处理"` | 视频处理为 Android 专属功能 |
| 文件不可读/不存在 | 返回 `"[附件文件: {path}]"` 文本占位 | 同上 |

### 3.2 视频抽帧策略（Android 专属）

`process_video_for_multimodal` 为 Android 专属功能——通过 `run_mobile_plugin("processVideo", { path })` 桥接 Kotlin 层实现高并发异步视频帧提取与 JPEG 压缩。非 Android 平台直接返回错误，**不支持文件直读降级**。

**当前流程**：

```
输入：视频文件路径
│
├─→ [cfg(target_os = "android")] 通过 VcpMobileState 获取 plugin_handle
│   └─→ run_mobile_plugin::<ProcessVideoResult>("processVideo", { path })
│       │
│       │ Kotlin 层 VcpMobilePlugin.processVideo() 内部处理：
│       ├─ 1280×720 包络缩放（保持宽高比，长边不超过 1280 / 短边不超过 720）
│       ├─ 步长采样抽帧（按视频时长等间隔提取关键帧）
│       ├─ 降采样截断上限 300 帧（防止长视频 payload 爆炸）
│       └─ 每帧独立 JPEG 压缩 → 写入临时文件 → 返回临时文件路径列表
│
├─→ Rust 侧后处理循环：
│   ├─ 逐帧 std::fs::read → base64 编码 → 拼接 data:image/jpeg;base64,{...}
│   ├─ 18MB Base64 字符硬截断限额（SIZE_LIMIT: 18_000_000），超出部分停止编码
│   └─ 边处理边物理清理临时帧文件（remove_file）+ 父目录（remove_dir_all）
│
└─→ 返回 Vec<String> 帧序列
```

**非 Android 分支**：

```rust
#[cfg(not(target_os = "android"))]
{
    Err("非 Android 物理端不支持视频多模态抽帧处理".to_string())
}
```

> 原 10 步精细处理流水线（场景检测、均匀采样、时间戳去重、批量转码）依赖 ffmpeg/ffprobe，现已被 Kotlin 层 Android 原生实现取代。视频降级路径不复存在——若 Kotlin 插件未初始化或处理失败，直接返回错误，无文件直读 fallback。

### 3.3 音频处理策略（降级版）

音频处理在 ffmpeg 删除后采用简化的降级策略：

| 策略 | 条件 | 行为 |
|------|------|------|
| Android 原生解码 | `run_mobile_plugin("processAudio")` → Kotlin MediaExtractor + MediaCodec 可用 | PCM 解码 → 16kHz 单声道 32kbps AAC 重编码 → base64 |
| 直接文件读取 | 文件 < 5 MB | 原始字节 → base64 data URL（保留原始格式） |
| 错误文本占位 | 文件 >= 5 MB 或读取失败 | `"[音频附件: {path}]"` |

> 原 ffmpeg 精细化音频转码参数（32kbps MP3、16kHz 单声道、3500s 硬截断等）不再使用。未来可通过 Android MediaCodec 实现类似的质量优化。

## 5. 错误处理与边界情况

### 5.1 调用者层容错（vcp_client.rs）

`media_processor/` 的函数在 `vcp_client.rs` 中通过 `tokio::task::spawn_blocking` 异步调用（因各提取器需要同步访问 `AppHandle` 状态和文件系统）：

```rust
let app_clone = app.clone();
match tokio::task::spawn_blocking(move || {
    convert_local_image_for_multimodal(&app_clone, &path_buf_clone)
}).await {
    Ok(Ok(data_url)) => { /* 成功 */ }
    Ok(Err(e)) | Err(e) => { /* 降级为文本占位 */ }
}
```

- 任何层级的失败都不会导致请求整体失败
- 降级策略：保留 `[附件文件: {path}]` 文本描述，确保对话内容不丢失
- `spawn_blocking` 确保同步的 Android 插件调用不阻塞 async 运行时

### 5.2 媒体处理器内部错误

| 场景 | 行为 |
|------|------|
| Android 原生 API 不可用（图片/音频） | 静默降级到直接文件读取（若 < 5MB）或错误文本占位 |
| Android 原生 API 不可用（视频） | 直接返回错误——视频无降级路径 |
| 非 Android 平台（视频） | 返回 `"非 Android 物理端不支持视频多模态抽帧处理"` |
| 文件大小超过直接读取限制（图片/音频） | 返回文本占位 `"[{类型}附件: {path}]"` |
| VcpMobileState 插件句柄未初始化 | 返回 `"VCP Mobile Plugin handle not initialized"` |
| 文件不存在或不可读 | 返回文本占位 |
| 文件路径包含非 UTF-8 字符 | `path.to_str().ok_or("Invalid path")?` 提前失败 |

### 5.3 极端输入边界

| 场景 | 行为 |
|------|------|
| 空图片文件 | Android 原生解码失败 → 尝试直读降级 → 可能成功（文件虽空但 < 5MB） |
| 超大视频文件 | Android 端：Kotlin 抽帧 300 帧上限 + 18MB Base64 硬截断 → 返回部分帧；非 Android：直接报错 |
| 超大音频文件（>= 5MB） | 超出直接读取限制 → 返回文本占位 |
| 已被删除的临时文件 | 文件读取失败 → 返回文本占位 |
| 超大图像（如 8K 照片） | Android 端 Kotlin BitmapFactory 自动降采样避免 OOM |
| 视频帧提取后临时文件读取失败（部分帧） | 跳过该帧，继续处理其他帧 |

## 6. 性能特征

### 6.1 复杂度

| 操作 | 主导开销 | 大致耗时 |
|------|---------|---------|
| 图片处理（Android 原生） | BitmapFactory 解码 + 缩放 | 数十毫秒 |
| 图片处理（直接读取） | 文件 I/O + base64 编码 | 与文件大小成正比 |
| 音频/视频处理（Android 原生） | MediaExtractor + MediaCodec | 与媒体时长成正比 |
| 音频/视频处理（直接读取） | 文件 I/O + base64 编码 | 与文件大小成正比 |

### 6.2 并发模型

- 所有媒体处理函数均为**同步函数**（非 `async`），因为它们需要访问 `AppHandle` 状态和文件系统
- `vcp_client.rs` 通过 `tokio::task::spawn_blocking` 将同步调用卸载到阻塞线程池，避免阻塞 async 运行时
- 各提取器独立执行，无内部并发
- 视频处理中 Kotlin 层抽帧为异步实现（Kotlin 协程），但从 Rust 侧看仍为同步阻塞调用

### 6.3 内存特征

| 场景 | 峰值内存 | 说明 |
|------|---------|------|
| 图片处理（Android 原生） | Kotlin 层缩放后位图（自动释放）+ Rust 层临时 WebP 文件 + base64 字符串 | BitmapFactory 自动降采样避免 OOM；临时文件处理后立即删除 |
| 图片处理（直接读取） | 原始文件 + base64 字符串 | 上限约 6.7MB（5MB 文件 base64 编码后约 6.67MB） |
| 音频处理（Android 原生） | Kotlin 层 PCM 缓冲区（自动释放）+ Rust 层临时 AAC 文件 + base64 字符串 | 临时文件处理后立即删除 |
| 音频（直接读取） | 原始文件 + base64 字符串 | 上限约 6.7MB（5MB 文件 base64 编码后约 6.67MB） |
| 视频处理（Android 原生） | Kotlin 层帧缓存（自动释放）+ Rust 层逐帧 base64 编码（单帧处理完即释放） | 18MB Base64 硬截断，边处理边物理清理临时帧文件 |
| 降级失败 | 微小（仅错误字符串） | 不加载任何媒体数据 |

```
              └──────────┬───────────────┘
                         │ JNI / Kotlin 互调
                         ▼
              ┌──────────────────────────┐
              │  Kotlin VcpMobilePlugin   │
              │  processImage()  → WebP  │
              │  processVideo()  → JPEG  │
              │  processAudio()  → AAC   │
              │  (Android 原生 API:      │
              │   BitmapFactory /        │
              │   MediaMetadataRetriever/│
              │   MediaCodec /           │
              │   MediaExtractor)        │
              └──────────────────────────┘
```

- **上游**：`vcp_client.rs` 在请求预处理阶段按扩展名分发到三个提取器，以 `spawn_blocking` 方式调用。
- **下游**：各提取器通过 `run_mobile_plugin` 桥接 Kotlin 层 Android 原生 API。图片/音频支持 3 层降级（Kotlin → 直读 < 5MB → 错误占位），视频仅支持 Kotlin 通道（非 Android 直接报错）。不再依赖 ffmpeg 基础设施层。
