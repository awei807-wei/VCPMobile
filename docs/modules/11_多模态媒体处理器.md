# 11_多模态媒体处理器（Media Processor）

## 1. 概述

### 1.1 模块定位

`media_processor/` 是 VCP Mobile 核心层的**多模态输入预处理子系统**，位于 `src-tauri/src/vcp_modules/media_processor/`（4 个 Rust 文件 + assets/ 目录）。该子系统在 VCP 请求发出之前执行，负责将用户选择的本地文件（图片、视频、音频）转换为大型语言模型（LLM）可消费的**Base64 data URL**格式，并执行必要的压缩、抽帧、转码以降低 payload 体积。

### 1.2 职责边界

| 职责领域 | 具体行为 | 对应源码文件 |
|---------|---------|------------|
| 图片预处理 | 缩放 + 转 webp + base64 编码 | `image_extractor.rs` |
| 视频预处理 | 场景检测 + 均匀采样 + 去重 + 抽帧 + 缩放 | `video_extractor.rs` |
| 音频预处理 | 提取 16kHz 单声道 MP3 (32kbps) + base64 编码 | `audio_extractor.rs` |
| FFmpeg 命令封装 | 跨平台（Android/桌面）二进制发现与命令执行 | `ffmpeg_cli.rs` |
| Android 二进制管理 | 编译期嵌入 `ffmpeg_aarch64` / `ffprobe_aarch64` | `ffmpeg_cli.rs` |

### 1.3 调用入口

```text
Vue 3 前端（附件选择器）
    ↓ 用户选择本地文件，生成 {type: "local_file", path: "file://..."}
    ↓ Tauri IPC invoke: sendToVCP
vcp_client.rs
    ↓ perform_vcp_request 阶段 0（数据验证与规范化）
    ↓ 识别 local_file 类型，按扩展名分发
    ├─→ image_extractor::convert_local_image_for_multimodal
    ├─→ video_extractor::process_video_for_multimodal
    └─→ audio_extractor::process_audio_for_multimodal
         ↓ 均委托 ffmpeg_cli::run_ffmpeg
              ↓ get_ffmpeg_path() / get_ffprobe_path()
                   ├─ Android: 从 include_bytes 解压到 temp_dir
                   └─ 桌面: 搜索 PATH 与常见安装路径
```

## 2. 公共接口

### 2.1 image_extractor

```rust
pub fn convert_local_image_for_multimodal(path: &Path) -> Result<String, String>
```

- **输入**：任意格式的本地图片文件路径
- **输出**：`data:image/webp;base64,{...}`
- **处理流程**：
  1. `ffmpeg` 读取原图
  2. `scale='min(1120,iw)':'min(1120,ih)':force_original_aspect_ratio=decrease:flags=lanczos`
     - 长边缩放到 ≤1120px，保持宽高比，使用 Lanczos 重采样
  3. `-c:v libwebp -q:v 80 -f webp pipe:1`
     - 输出 webp 格式，质量 80，保留透明度通道
  4. stdout bytes → base64 → data URL 拼接

### 2.2 video_extractor

```rust
pub fn process_video_for_multimodal(path: &Path) -> Result<Vec<String>, String>
```

- **输入**：任意格式的本地视频文件路径
- **输出**：`Vec<data:image/jpeg;base64,{...}>`，每张帧一个 data URL
- **处理流程**（详见 §3.2）

### 2.3 audio_extractor

```rust
pub fn process_audio_for_multimodal(path: &Path) -> Result<String, String>
```

- **输入**：任意格式的本地音频文件路径
- **输出**：`data:audio/mp3;base64,{...}`
- **处理流程**：
  1. `ffmpeg` 读取原音频，增加 `-t 3500` 硬截断（约 58 分钟）以保证 payload < 20MB
  2. `-vn -c:a libmp3lame -b:a 32k -ar 16000 -ac 1 -f mp3 pipe:1`
     - 去除视频流，MP3 编码，32kbps 极低码率，采样率 16kHz，单声道
  3. stdout bytes → base64 → data URL 拼接（优化内存分配，避免中间拷贝）

## 3. 核心算法详解

### 3.1 FFmpeg 跨平台基础设施

#### Android 二进制嵌入

```rust
#[cfg(target_os = "android")]
mod android {
    pub const FFMPEG_BINARY: &[u8] = include_bytes!("assets/ffmpeg_aarch64");
    pub const FFPROBE_BINARY: &[u8] = include_bytes!("assets/ffprobe_aarch64");
}
```

- 编译时将 Android 平台的 ffmpeg/ffprobe 二进制文件嵌入 Rust 可执行体
- 运行时首次调用时解压到 `std::env::temp_dir()`（`/data/data/{pkg}/cache/`）
- 文件已存在则跳过写入，避免重复 I/O
- **安全检查**：若嵌入的二进制小于 1024 字节，返回错误提示开发者放置真实二进制

#### 桌面端二进制发现

```rust
#[cfg(not(target_os = "android"))]
for name in ["ffmpeg", "/usr/bin/ffmpeg", "/usr/local/bin/ffmpeg", "/opt/homebrew/bin/ffmpeg"] {
    if Command::new(name).arg("-version").output().is_ok() {
        return Ok(PathBuf::from(name));
    }
}
```

- 按优先级搜索：PATH → `/usr/bin` → `/usr/local/bin` → Homebrew (`/opt/homebrew`)
- ffprobe 使用相同策略

#### 命令执行封装

```rust
pub fn run_ffmpeg(args: &[&str]) -> Result<Vec<u8>, String>
pub fn run_ffprobe(args: &[&str]) -> Result<serde_json::Value, String>
```

- `run_ffmpeg` 返回 stdout 原始 bytes（用于图片/音频的 pipe 输出）
- `run_ffprobe` 自动解析 stdout 为 JSON（用于元数据读取）
- 两者均检查 `output.status.success()`，失败时将 stderr 内容返回给调用者

### 3.2 视频抽帧策略

`process_video_for_multimodal` 是子系统中最复杂的算法，分 10 个步骤：

```
输入：视频文件路径
│
├─→ Step 1: 获取时长（ffprobe）
│   └─→ duration: f64
│
├─→ Step 2: 确定采样率
│   └─→ duration ≤ 60s → fps = 1.0
│       duration > 60s → fps = 0.5
│
├─→ Step 3: 场景切换检测（ffmpeg select='gt(scene,0.3)'）
│   └─→ scene_times: Vec<f64>
│
├─→ Step 4: 均匀采样时间戳
│   └─→ uniform_times: 0.0, 1/fps, 2/fps, ... < duration
│
├─→ Step 5: 合并与排序
│   └─→ all_times = uniform_times ∪ scene_times，升序
│
├─→ Step 6: 时间戳去重
│   └─→ 相邻时间戳差 < 1.5s 视为重复，保留第一个
│   └─→ deduped: Vec<f64>
│
├─→ Step 7: 硬上限截断
│   └─→ 若 deduped.len() > 300，均匀降采样至 300 帧
│
├─→ Step 8: 批量提取均匀帧到临时目录（不加载到内存）
│   └─→ ffmpeg fps=...,scale='min(1280,iw)':-1 → temp_dir/frame_%04d.jpg
│
├─→ Step 9: 按需读取并执行 Base64 累加截断
│   ├─→ 循环遍历目标时间戳，计算文件名并从磁盘按需读取（避免 OOM）
│   ├─→ 若不存在（场景帧）→ extract_single_frame 单独提取
│   ├─→ 累加 Base64 字符串长度，若超过 18,000,000 字节则提前 break
│   └─→ 返回受限大小的帧序列（确保请求体 < 20MB）
│
└─→ Step 10: 清理临时目录
    └─→ remove_dir_all(&temp_dir)
```

**设计决策说明**：

1. **为什么先批量提取再匹配？**
   - 单独为每个时间戳调用 `ffmpeg -ss` 的 overhead 极高（每次启动进程、定位解码）
   - 先按固定 fps 批量提取所有均匀帧到磁盘，再按需读取，摊薄 I/O 成本
   - 场景帧若恰好落在均匀采样网格中则零额外成本；否则单独提取

2. **按需读取与 18MB 保护**
   - 之前版本曾尝试一次性将所有临时帧读入内存，导致长视频 OOM。
   - 现版本改为按需读取（Step 9），且引入 `SIZE_LIMIT` 累加器，一旦已生成的 Base64 数组总大小触碰 18MB 边界，立刻停止处理，确保整体 JSON 请求不会被 API 网关拒绝。

3. **场景检测参数 `gt(scene,0.3)`**
   - 使用 ffmpeg 的 `select` 滤镜，当相邻帧差异度 > 0.3 时视为场景切换
   - 输出通过 `showinfo` 滤镜写入 stderr，用正则 `pts_time:([\d.]+)` 提取时间戳
   - 0.3 是经验阈值：能捕获明显的镜头切换，同时避免光照微变导致的误触发

4. **720p 缩放策略**
   - 单帧提取后，通过 `image` crate 的 `DynamicImage::resize` 缩放到 1280×720 边界框内
   - 使用 Lanczos3 滤波器，在缩小场景下保持锐利度

### 3.3 音频转码策略

音频处理包含压缩与硬截断双重保护：

| 参数 | 值 | 目的 |
|------|-----|------|
| `-t 3500` | 3500 秒 | 硬截断（约 58 分钟），确保 Base64 后的 payload < 20MB |
| `-vn` | — | 去除视频流（若输入为带视频的文件） |
| `-c:a libmp3lame` | MP3 编码 | 相比 WAV 大幅降低体积，且兼容性极佳 |
| `-b:a 32k` | 32 kbps | 极低码率，在保证语音清晰度的前提下最小化体积 |
| `-ar 16000` | 16 kHz | 语音识别模型的标准输入采样率 |
| `-ac 1` | 单声道 | 减少数据量 |
| `-f mp3` | MP3 容器 | 流式输出，直接 pipe 到 Rust |

## 5. 错误处理与边界情况

### 5.1 调用者层容错（vcp_client.rs）

`media_processor/` 的函数在 `vcp_client.rs` 中被 `tokio::task::spawn_blocking` 包裹调用，形成两层防护：

```rust
match tokio::task::spawn_blocking(move || {
    convert_local_image_for_multimodal(&path_buf_clone)
}).await {
    Ok(Ok(data_url)) => { /* 成功 */ }
    Ok(Err(e)) => { /* ffmpeg 错误，降级为文本占位 */ }
    Err(e) => { /* spawn_blocking panic，降级为文本占位 */ }
}
```

- 任何层级的失败都不会导致请求整体失败
- 降级策略：保留 `[附件文件: {path}]` 文本描述，确保对话内容不丢失

### 5.2 媒体处理器内部错误

| 场景 | 行为 |
|------|------|
| ffmpeg/ffprobe 未找到 | 返回 `Err("ffmpeg not found in PATH")`，由调用者降级 |
| Android 嵌入二进制过小 | 返回明确的构建错误，提示放置真实二进制 |
| 视频临时目录创建失败 | 返回 `Err`，不继续处理 |
| 单帧提取失败 | 在 `frame_to_data_url` 中返回 `Err`，由 `process_video` 传播 |
| 文件路径包含非 UTF-8 字符 | `path.to_str().ok_or("Invalid path")?` 提前失败 |

### 5.3 极端输入边界

| 场景 | 行为 |
|------|------|
| 空图片文件 | ffmpeg 读取失败 → 降级为文本占位 |
| 0 秒视频 | `duration = 0`，`while t < duration` 不执行，`all_times` 仅含场景帧 |
| 无场景切换的长视频 | 仅依赖均匀采样，上限 300 帧 |
| 大量场景切换（>300 处） | 合并均匀采样后若仍超 300，均匀降采样；可能丢失部分场景帧 |
| 单帧尺寸极大（如 8K） | ffmpeg 批量提取时已缩放到 1280px 宽，二次缩放至 720p 边界框 |

## 6. 性能特征

### 6.1 时间复杂度

| 操作 | 主导开销 | 大致耗时 |
|------|---------|---------|
| 图片转换 | ffmpeg 启动 + 编码 | 数百毫秒（取决于原图尺寸） |
| 音频转换 | ffmpeg 启动 + 编码 | 与原音频时长成正比 |
| 视频场景检测 | ffmpeg 全量解码 + 帧差异计算 | 与视频时长成正比，约为实时播放的 2–5 倍 |
| 视频批量抽帧 | ffmpeg 启动 + 解码 + 编码 | 与目标帧数成正比 |
| 单帧缩放 + base64 | `image` crate 内存操作 | 每帧数十毫秒 |

### 6.2 并发模型

- 所有媒体处理函数均为**同步函数**（非 `async`）
- 在 `vcp_client.rs` 中通过 `tokio::task::spawn_blocking` 放入独立线程池执行
- 视频处理内部无并发：帧按顺序提取和编码

### 6.3 内存特征

| 场景 | 峰值内存 | 说明 |
|------|---------|------|
| 图片处理 | 原图 + webp 输出 | 大图片可能短暂占用较高内存 |
| 音频处理 | 原音频 + MP3 输出 | **已优化**：内存占用约等于最终 MP3 大小（约 4KB/s） |
| 视频处理 | 单帧 + 临时目录 | **已优化**：不保持全量帧在内存；每帧处理后立即释放 |
