# 12_附件存储与文件管理器（File Manager）

## 1. 概述

### 1.1 模块定位

`file_manager.rs`（617 行）是 VCP Mobile 核心层的**附件物理存储与上传管理中心**，位于 `src-tauri/src/vcp_modules/infra/file_manager.rs`。该模块负责所有本地附件的持久化存储、内容寻址去重、MIME 类型精修、文本提取调用以及跨平台路径安全解析。文本提取的具体实现已下沉至同目录的 `file_extractor.rs`（759 行）。它是前端 `attachmentStore.ts` 与 Rust 后端之间的关键桥梁，同时也是 Android Kotlin 侧原生文件选择器的直接对接方。

### 1.2 职责边界

| 职责领域 | 具体行为 | 对应源码位置 |
|---------|---------|------------|
| 附件根目录解析 | Android 外部存储 / 桌面端 app_config_dir 差异化处理 | `get_attachments_root_dir` L18–L39 |
| 缩略图目录解析 | 与附件目录同级的 thumbnails 目录 | `get_thumbnails_root_dir` L42–L49 |

## 3. 路径解析与安全

### 3.1 附件根目录解析

```rust
pub fn get_attachments_root_dir<R: tauri::Runtime>(
    app_handle: &tauri::AppHandle<R>,
) -> Result<std::path::PathBuf, String>
```

**平台差异化策略**（L18–L39）：

| 平台 | 解析逻辑 | 最终路径示例 |
|------|---------|------------|
| Android | `document_dir` → `pop()` → `push("attachments")` | `/storage/emulated/0/Android/data/<pkg>/files/attachments` |
| 桌面端 (Fallback) | `app_config_dir` → `push("data/attachments")` | `%APPDATA%/<pkg>/data/attachments` |

**设计决策**：Android 端将附件目录从 `app_config_dir/data/attachments`（内部存储）迁移到外部存储，解决内部存储空间受限、卸载即丢失、其他应用无法访问等问题。旧数据迁移逻辑（`migrate_legacy_attachments`）已在技术债务清理中移除，该迁移作为一次性操作已完成其历史使命。

### 3.2 缩略图根目录解析

```rust
    app_handle: &tauri::AppHandle<R>,
```

- 实现：获取附件目录后 `pop()` 弹出 `attachments`，再 `push("thumbnails")`（L42–L49）。
- 结果：缩略图与附件在同一父目录下平行存放，路径结构清晰。

### 3.3 安全文件重命名（safe_rename）

```rust
pub fn safe_rename<P: AsRef<std::path::Path>, Q: AsRef<std::path::Path>>(
    from: P,
    to: Q,
) -> std::io::Result<()>
```

**问题背景**：Android 系统中，应用的 `app_cache_dir`（内部缓存）与 `attachments_dir`（外部存储）可能位于不同的物理挂载分区。当 `fs::rename` 尝试跨分区移动文件时，操作系统返回 `EXDEV`（跨设备链接）错误，导致附件注册失败。

**实现策略**（L48–L60）：

```rust
pub fn safe_rename<P: AsRef<std::path::Path>, Q: AsRef<std::path::Path>>(
    from: P, to: Q
) -> std::io::Result<()> {
    let from = from.as_ref();
    let to = to.as_ref();

    if std::fs::rename(from, to).is_err() {
        // 如果是跨物理分区移动，执行物理复制 + 物理删除源文件以兜底
        std::fs::copy(from, to)?;
        let _ = std::fs::remove_file(from);
    }
    Ok(())
}
```

1. **首选原子 rename**：优先尝试 `std::fs::rename`，同分区下为操作系统级原子操作。
2. **EXDEV 降级兜底**：若 `rename` 失败（大概率是跨分区），降级为 `copy + remove_file` 组合。
3. **无数据残留保证**：`copy` 成功后才删除源文件；即使删除失败，也只是遗留一个临时副本，不影响目标文件完整性。

**使用位置**：`register_local_file` 中移动附件主文件和缩略图文件时统一调用。

### 3.4 路径安全校验

```rust
fn ensure_safe_path(app_handle: &AppHandle, path: &std::path::Path) -> Result<(), String>
```

**授权访问范围**（L216–L254）：

| 目录 | 来源 | 用途 |
|------|------|------|
| `app_config_dir` | Tauri Path API | 应用配置与内部数据 |
| `app_cache_dir` | Tauri Path API | 临时缓存（缩略图过渡） |
| `attachments_dir` | `get_attachments_root_dir` | 附件物理存储 |
| `thumbnails_dir` | `get_thumbnails_root_dir` | 缩略图物理存储 |

**校验逻辑**（commit `0119fde` 安全加固）：

1. **物理路径展开**：`std::fs::canonicalize(path)` 将目标路径的所有相对分量（`..`、符号链接等）解析为绝对物理路径，杜绝字符级前缀欺骗。
2. **白名单比对**：对 `canonicalize` 后的路径与四个授权目录的 `canonicalize` 版本进行 `starts_with` 比对。
3. **存在性前置检查**：若 `path.exists()` 为 `false`，直接阻断返回 `"非法路径访问：目标文件不存在"`，防止信息探测攻击。

```rust
fn ensure_safe_path(app_handle: &AppHandle, path: &std::path::Path) -> Result<(), String> {
    let canonical_path = if path.exists() {
        std::fs::canonicalize(path).map_err(|e| format!("路径规范化失败: {}", e))?
    } else {
        return Err("非法路径访问：目标文件不存在".to_string());
    };

    let config_dir = app_handle.path().app_config_dir().map_err(|e| e.to_string())?;
    let canonical_config = std::fs::canonicalize(&config_dir).unwrap_or(config_dir);
    // ... 同理处理 cache_dir, attachments_dir, thumbnails_dir

    if canonical_path.starts_with(&canonical_config)
        || canonical_path.starts_with(&canonical_cache)
        || canonical_path.starts_with(&canonical_attachments)
        || canonical_path.starts_with(&canonical_thumbnails)
    {
        Ok(())
    } else {
        Err("非法路径访问：禁止访问应用授权范围以外的文件".to_string())
    }
}
```

## 4. MIME 类型精修引擎

```rust
pub fn get_refined_mime_type(
    path: &std::path::Path,
    original_name: &str,
    initial_mime: &str,
) -> String
```

该函数实现三层递进式 MIME 类型判定，对齐桌面端 `fileManager.js` 的决策逻辑（L194–L284）。

### 4.1 第一层：强制修正

经验性覆盖已知有问题的 MIME 映射：

| 扩展名 | 返回值 | 原因 |
|--------|--------|------|
| `mp3` | `audio/mpeg` | 多数系统误报为 `audio/mp3` 或其他变体 |

### 4.2 第二层：扩展名路由

当 `initial_mime` 为空或等于 `application/octet-stream`（通用二进制流）时，按扩展名精确路由。覆盖 60+ 种扩展名：

**文本/代码类**（统一映射为 `text/plain` 以触发后续文本提取）：
- `js`, `mjs`, `bat`, `sh`, `py`, `java`, `c`, `cpp`, `h`, `hpp`, `cs`
- `go`, `rb`, `php`, `swift`, `kt`, `kts`, `ts`, `tsx`, `jsx`, `vue`
- `yml`, `yaml`, `toml`, `ini`, `log`, `sql`, `jsonc`, `rs`, `dart`, `lua`
- `r`, `pl`, `ex`, `exs`, `zig`, `hs`, `scala`, `groovy`, `d`, `nim`, `cr`

**文档类**：
- `txt` → `text/plain`；`json` → `application/json`；`xml` → `application/xml`
- `csv` → `text/csv`；`html` → `text/html`；`css` → `text/css`
- `pdf` → `application/pdf`；`doc`/`docx`/`xls`/`xlsx`/`ppt`/`pptx` → Office MIME

**图片/音频/视频类**：
- 图片：`jpg`, `jpeg`, `png`, `gif`, `webp`, `svg`, `bmp`, `ico`, `tiff`, `tif`, `heic`, `heif`, `avif`
- 音频：`wav`, `ogg`, `ogv`, `flac`, `aac`, `aiff`, `aif`, `m4a`, `opus`, `amr`
- 视频：`mp4`, `m4v`, `webm`, `mov`, `qt`, `avi`, `mkv`, `wmv`, `flv`, `3gp`, `3g2`, `mts`, `m2ts`

### 4.3 第三层：物理层嗅探

当扩展名也无法识别时，进入物理层检测（L268–L279）：

1. **魔数匹配**：`infer::get_from_path(path)` 读取文件前几个字节进行魔数（Magic Number）匹配。可识别被恶意篡改扩展名的二进制文件（如将 `.exe` 改为 `.jpg`）。
2. **文本启发式**：`is_likely_text_file(path)`（L166–L190）读取前 1024 字节，检查是否包含 NULL 字节（`0x00`）。若不含 NULL 字节，则极大概率是纯文本或代码文件，返回 `text/plain`。

**NULL 字节启发式的可靠性**：
- 所有主流二进制格式（图片、音频、视频、压缩包、可执行文件）在头部均会出现 `0x00`。
- 纯文本/代码文件的编码（UTF-8、GBK、Latin-1 等）均不会在正常内容中使用 `0x00`。
- 因此该检测在统计意义上具有极高的置信度。

### 4.4 兜底策略

若三层检测均未命中，则直接返回 `initial_mime` 原值（L283）。

### 5.2 大小拦截与内存保护

### 5.3 mmap 读取与编码检测

```rust
fn read_text_with_mmap(path: &std::path::Path) -> Option<String>
```

**实现细节**（`file_extractor.rs` L11–L28）：

1. **打开文件** → `std::fs::File::open`
2. **内存映射** → `memmap2::Mmap::map(&file)`（`unsafe` 块，但内存映射本身由操作系统保证安全）
3. **BOM 头检测** → `encoding_rs::Encoding::for_bom(&mmap)`
   - 优先检测 UTF-8 BOM、UTF-16 LE/BE BOM、UTF-32 等。
   - 若命中，使用对应编码解码并去除 BOM。
4. **统计编码检测**（无 BOM 时）→ `chardetng::EncodingDetector`
   - Firefox 浏览器同款库，基于字节频率统计模型。
   - `detector.feed(&mmap, true)` 输入全部字节，`guess(None, true)` 输出最可能编码。
5. **统一输出 UTF-8** → `decode_without_bom_handling` → `into_owned()`

**为什么用 mmap 而非普通 read？**
- 大文件读取时，mmap 由操作系统按需分页加载，不会一次性占用等于文件大小的物理内存。
- 配合 50 MB 上限，即使极端情况也仅在虚拟地址空间映射，实际物理内存占用由 OS 调度。

### 5.4 字符截断保护

```rust
const MAX_TEXT_CHARS: usize = 10_000_000; // 1000 万字符
```

- 提取文本超过 1000 万字符时，截取前 1000 万字符并追加 `……（文本过长已截断）`（`file_extractor.rs` L600–L605）。
- 对齐 2M 上下文长度的大模型输入约束（约 8–10M 字符为安全上限）。

### 5.5 结构化文档文本提取后端化

> 函数位置（均已迁移至 `file_extractor.rs`）：`extract_docx_text` L48–L236、`extract_pptx_text` L238–L309、`extract_xlsx_text` L311–L536、`extract_pdf_text` L538–L555、`simple_xml_unescape` L30–L36

#### 5.5.1 背景与动机

此前，`pdf`/`docx`/`xlsx`/`pptx` 的文本提取由前端 `useDocumentProcessor` 在消息发送前通过 JavaScript 库（如 `mammoth.js`、`pdf.js`）JIT 处理。该方案存在以下问题：

- **跨平台一致性差**：移动端 WebView 与桌面端 Chromium 的 WASM/JS 性能差异大，大文档易 ANR。
- **内存双份占用**：JS 侧提取一次，再序列化通过 Tauri IPC 传入 Rust，同一份文本在 JS Heap + Rust Heap 各存一份。
- **生命周期脆弱**：前端刷新或进程回收后，未持久化的提取结果丢失。

**决策**：将结构化文档文本提取完全 offload 到 Rust 后端，利用原生 ZIP/PDF 解析库直接读取物理文件，提取完成后即时闭环写入 SQLite。

#### 5.5.2 Cargo.toml 新增依赖

```toml
zip = { version = "2", default-features = false, features = ["deflate"] }
lopdf = { version = "0.33.0", default-features = false, features = ["pom_parser"] }
```

| 依赖 | 用途 | 裁剪说明 |
|------|------|---------|
| `zip` | 解析 Office Open XML 格式（docx/pptx/xlsx 均为 ZIP 包） | 仅启用 `deflate`，禁用默认特性以减少编译体积 |
| `lopdf` | 解析 PDF 文档结构并逐页提取文本 | 仅启用 `pom_parser`，禁用 `nom_parser` 以精简依赖树 |

#### 5.5.3 四个提取函数实现原理

> 当前实现已从早期基于 `fancy_regex` 的正则提取方案，演进为**单遍状态机解析器**，以应对复杂文档结构（表格、标题层级、幻灯片顺序、工作表名称等）。

**`extract_docx_text(path: &Path) -> Option<String>`**
- 以 ZIP 档案打开文件 → 定位 `word/document.xml`。
- 采用单遍字符流状态机解析，实时追踪以下结构：
  - **段落属性**（`w:pPr`）：识别 `w:pStyle` 的 `w:val` 属性，将 `Heading1`–`Heading4` 映射为 Markdown 标题（`# `–`#### `）。
  - **列表段落**（`w:numPr`）：标记为无序列表项（`- `）。
  - **表格**（`w:tbl` / `w:tr` / `w:tc`）：按行解析单元格文本，输出 Markdown 表格格式（`| cell1 | cell2 |` + 分隔符行）。
  - **加粗**（`w:b` / `w:bCs` 在 `w:rPr` 内）：对 `w:t` 文本段包裹 `**bold**`。
  - **特殊字符**：`w:tab` → `\t`，`w:br` / `w:cr` → `\n`。
- 对捕获文本执行 `simple_xml_unescape` 反转义 5 个标准 XML 实体（`&amp;`, `&lt;`, `&gt;`, `&quot;`, `&apos;`）。
- 末尾执行"加粗熔接"优化：将 `**学****号**` 合并为 `**学号**`。
- 若结果为空则返回 `None`。

**`extract_pptx_text(path: &Path) -> Option<String>`**
- 以 ZIP 档案打开文件 → 收集并按数字排序 `ppt/slides/slide*.xml`。
- 对每张幻灯片单遍状态机解析 `<a:t>` 文本节点，同张幻灯片内文本段以换行拼接。
- 每张幻灯片前插入 `--- Slide N ---` 分隔标记，保持幻灯片边界。
- 空结果返回 `None`。

**`extract_xlsx_text(path: &Path) -> Option<String>`**
- 以 ZIP 档案打开文件。
- **第一步**：读取 `xl/workbook.xml`，解析每个 `sheet` 标签的 `name` 和 `sheetId`，建立 ID→名称映射。
- **第二步**：读取 `xl/sharedStrings.xml`，提取所有共享字符串到索引表。
- **第三步**：按编号顺序遍历 `xl/worksheets/sheet*.xml`：
  - 追踪 `<c>` 单元格的 `r` 属性（如 `A1`、`B3`）解析列索引。
  - 区分普通值与共享字符串引用（`t="s"`）。
  - 按行收集单元格，按列索引排序后输出制表符分隔文本（`\t`），空列补位。
- 每个工作表前插入 `--- Sheet N: {名称} ---` 分隔标记。
- 空结果返回 `None`。

**`extract_pdf_text(path: &Path) -> Option<String>`**
- 使用 `lopdf::Document::load(path)` 加载 PDF 文档。
- 获取页码列表并排序，逐页调用 `doc.extract_text(&[page_id])`。
- 页间以换行分隔。
- 空结果返回 `None`。

**`simple_xml_unescape(s: &str) -> String`**
- 反转义 5 个标准 XML 预定义实体：`&amp;` → `&`，`&lt;` → `<`，`&gt;` → `>`，`&quot;` → `"`，`&apos;` → `'`。

#### 5.5.4 `try_extract_text` 扩展

`try_extract_text` 在原有"纯文本/代码"分支之后，新增第三个判定分支（L468–L485）：

```rust
// 3. 结构化文档 (PDF, Docx, Xlsx, Pptx)
let doc_text = match ext.as_str() {
    "docx" => extract_docx_text(path),
    "pptx" => extract_pptx_text(path),
    "xlsx" => extract_xlsx_text(path),
    "pdf"  => extract_pdf_text(path),
    _ => None,
};

if let Some(text) = doc_text {
    const MAX_TEXT_CHARS: usize = 10_000_000;
    if text.chars().count() > MAX_TEXT_CHARS {
        let truncated: String = text.chars().take(MAX_TEXT_CHARS).collect();
        return Some(format!("{}……（文本过长已截断）", truncated));
    }
    return Some(text);
}
```

- **1000 万字符截断** 同样保护结构化文档提取分支，防止超大 PDF/Excel 撑爆上下文窗口。
- 与纯文本分支独立截断，互不干扰。

#### 5.5.5 数据库 UPDATE 闭环

`register_attachment_internal` 在 SQLite UPSERT 和文本提取之后，新增即时 UPDATE 闭环（L637–L650）：

```rust
// 核心安全优化：在后端即时且闭环地将耗时提取出的重资产数据持久化写入数据库
// 杜绝大文本数据在前端 WebView 绕一圈所导致的数据丢失或内存积压泄漏！
if extracted_text.is_some() || thumbnail_path.is_some() {
    sqlx::query(
        "UPDATE attachments 
         SET extracted_text = ?, thumbnail_path = ?, updated_at = ? 
         WHERE hash = ?"
    )
    .bind(&extracted_text)
    .bind(&thumbnail_path)
    .bind(now as i64)
    .bind(&hash)
    .execute(pool)
    .await
    .ok();
}
```

**设计意图**：
- `extracted_text` 和 `thumbnail_path` 均为"重资产"：前者可能达数 MB，后者为文件系统路径。
- 若仅通过 `AttachmentData` 返回前端，一旦前端刷新、进程回收或 Tauri 通道异常，这些结果将丢失。
- 通过在注册阶段即时 UPDATE 数据库，确保**提取结果的生命周期与附件记录本身绑定**，不受前端 volatility 影响。
- 使用 `.ok()` 静默忽略错误：注册流程不应因 UPDATE 失败而阻断，后续前端仍可通过物理路径重新提取。

#### 5.5.6 IPC 大文本截断

`register_attachment_internal` 返回 `AttachmentData` 时，强制将 `extracted_text` 字段设为 `None`（L671）：

```rust
Ok(AttachmentData {
    // ...
    extracted_text: None, // 掐断大文本在前端的冗余中转传输
    thumbnail_path,
})
```

**设计决策**：
- Tauri IPC 序列化大字符串到 WebView 存在显著内存开销和延迟；对于 10MB 级别的提取文本，IPC 往返可能阻塞 UI 线程数百毫秒。
- 前端如需查看提取内容，通过 `internal_path` 发起独立 fetch（或未来新增专用 query 接口），而非在注册响应中携带。
- `thumbnail_path` 仍为 `Option<String>`，因为缩略图路径短小且前端展示急需。

### 5.6 `file_extractor.rs` 模块架构总览

`file_extractor.rs`（830 行）是独立于 `file_manager.rs` 的**纯函数库**，负责多模态文件文本提取。该模块无内部状态、无异步逻辑、无 Tauri 依赖，可被任何上层模块直接调用。

#### 扩展名分类体系

模块顶部通过 `lazy_static!` 维护两组扩展名集合：

| 集合 | 包含格式 | 用途 |
|------|---------|------|
| `TEXT_AND_CODE_EXTENSIONS` | `txt`, `md`, `csv`, `json`, `ts`, `tsx`, `rs`, `py`, `java`, `c`, `cpp`, `go`, `swift`, `kt`, `css`, `html`, `yaml`, `toml`, `sql`, `dart`, `lua`, `zig`, `scala` 等 60+ 种 | 判定是否触发 mmap + 编码检测分支 |
| `STRUCTURED_DOC_EXTENSIONS` | `docx`, `pdf`, `xlsx`, `pptx` | 判定是否触发 ZIP/PDF 结构化解析分支 |

三组判定函数：
- `is_text_or_code_extension(ext)`: 纯文本/代码判定
- `is_structured_doc_extension(ext)`: 结构化文档判定
- `is_extractable_extension(ext)`: 两者并集

#### 主入口 `try_extract_text`

```rust
pub fn try_extract_text(path: &std::path::Path, mime_type: &str) -> Option<String>
```

执行三段式分发：

1. **纯文本/代码分支**：若 MIME 以 `text/` 开头，或为 `application/json`/`application/javascript`，或扩展名命中 `TEXT_AND_CODE_EXTENSIONS`：
   - 先执行 **50 MB 硬上限拦截**，超限返回占位提示字符串。
   - 调用 `read_text_with_mmap`：内存映射 + BOM 检测 → `chardetng` 统计检测 → 统一输出 UTF-8。
   - 结果超过 **1000 万字符**则截断并追加提示。

2. **结构化文档分支**：按扩展名匹配 `docx`/`pptx`/`xlsx`/`pdf`，调用对应单遍状态机提取器。

3. **未识别格式**：返回 `None`。

#### 与 `file_manager.rs` 的协作边界

| 职责 | 归属 | 说明 |
|------|------|------|
| 文件路径安全校验 | `file_manager.rs` | `ensure_safe_path` 在调用提取前完成 |
| 扩展名/MIME 精修 | `file_manager.rs` | `get_refined_mime_type` 三层递进判定 |
| 物理文本提取 | `file_extractor.rs` | 纯函数，接收 `&Path` 和 `&str` mime |
| 结果持久化 | `file_manager.rs` | `register_attachment_internal` 执行 SQLite UPDATE |
| 大文本 IPC 截断 | `file_manager.rs` | 返回前端前 `extracted_text: None` |

`file_manager.rs` 仅在 `register_attachment_internal` 中通过 `use super::file_extractor::try_extract_text;` 引入并调用一次，两者通过函数签名解耦，便于独立测试和迭代。

## 6. 两条上传链路详解

### 6.1 链路 A：store_file IPC（小文件）

```rust
#[tauri::command]
pub async fn store_file(
    app_handle: AppHandle,
    db_state: State<'_, DbState>,
    original_name: String,
    file_bytes: Vec<u8>,
    mime_type: String,
) -> Result<AttachmentData, String>
```

**适用场景**：前端传来的极小内存数据，如录音片段、二维码、截图等（桌面端 <2MB 的文件也走此链路）。

**流程**（L467–L522）：

```text
接收 file_bytes: Vec<u8>
│
├─→ 0. 冗余兜底拦截（L477–L479）
│   └─→ 若 file_bytes.len() > 100MB → Err("文件过大，请使用高速链路上传")
│
├─→ 1. 计算 SHA-256（L482–L484）
│   └─→ hasher.update(&file_bytes) → hex::encode(finalize)
│
├─→ 2. 构造内部文件名（L486–L494）
│   └─→ {hash} 或 {hash}.{ext}
│
├─→ 3. 确保 attachments_dir 存在
├─→ 4. 内容寻址去重（L506–L508）
│   └─→ 若 internal_file_path.exists() → 跳过写入
│   └─→ 否则 fs::write(&internal_file_path, &file_bytes)
│   └─→ 文本提取调用：`try_extract_text`（位于 `file_extractor.rs`）异步执行
│
├─→ 5. MIME 精修（L511）
│   └─→ get_refined_mime_type(&internal_file_path, &original_name, &mime_type)
│
└─→ 6. 注册并返回（L512–L522）
    └─→ register_attachment_internal(...)
```

**关键特性**：
- `Vec<u8>` 直接通过 Tauri IPC 传递，前端使用 `Uint8Array`（Tauri v2 原生支持二进制传输）。
- 100 MB 兜底拦截是深层防御，正常情况下前端已将 >2MB 文件分流至高速链路。
- 内容寻址去重：同名同内容的文件仅存储一份物理副本。

### 6.2 链路 B：register_local_file（零拷贝路径传输）

```rust
#[tauri::command]
pub async fn register_local_file(
    app_handle: AppHandle,
    db_state: State<'_, DbState>,
    local_path: String,
    original_name: String,
    mime_type: Option<String>,
    thumbnail_path: Option<String>,
) -> Result<AttachmentData, String>
```

**适用场景**：Android 原生文件选择器（Kotlin 侧）返回的物理路径；或任何前端已掌握本地绝对路径的场景。

**核心设计目标**：全程无 Base64 编解码，无 `Vec<u8>` 全量缓冲，实现"零拷贝"物理路径传输。

**流程**（L443–L578）：

```text
接收 {local_path, original_name, mime_type, thumbnail_path}
│
├─→ 1. 源文件存在性检查（L454–L456）
├─→ 2. 路径安全校验 ensure_safe_path（L459）
├─→ 3. 异步读取元数据获取 size（L462–L465）
├─→ 4. 流式异步读取 + SHA-256 计算（L468–L483）
│   └─→ tokio::fs::File::open → loop { read(&mut buffer[65536]) → hasher.update(&buffer[..n]) }
│
├─→ 5. 构造目标路径 {hash}.{ext}（L486–L494）
├─→ 6. 内容寻址去重检查（L507–L509）
│   └─→ 若 dest_path.exists() → 删除源临时文件，复用已有副本
│
├─→ 7. 文件移动（L512–L522）
│   ├─→ 优先: `safe_rename`（内部先尝试 `fs::rename`，EXDEV 时降级为 `copy + remove_file`）
│   └─→ 跨越物理挂载分区时自动降级，无数据残留风险
│
├─→ 8. MIME 精修（L525–L526）
├─→ 9. 统一注册（L529–L538）
│   └─→ register_attachment_internal → 返回 AttachmentData
│
└─→ 10. 缩略图后处理（L541–L578）
    ├─→ 若 frontend 传入 thumbnail_path:
    │   ├─→ 确保 thumbnails_dir 存在
    │   ├─→ 目标路径: {hash}_thumb.webp
    │   ├─→ `safe_rename` 优先，EXDEV 降级为 copy+delete
    │   └─→ SQLite UPDATE attachments SET thumbnail_path = ? WHERE hash = ?
    └─→ 更新返回数据的 thumbnail_path 字段
```

**为什么需要前端传入缩略图？**
- Android Kotlin 侧在文件选择阶段已通过 `ThumbnailUtils.createImageThumbnail` 生成硬件加速缩略图。
- 将缩略图路径随主文件一并传入，Rust 侧只需执行 rename/copy 到统一目录并更新数据库，避免重复生成。
- 这是"缩略图下沉到 Kotlin 侧"架构的直接体现（见 §7）。

## 7. 附件注册与后处理

### 7.1 register_attachment_internal

```rust
pub async fn register_attachment_internal<R: tauri::Runtime>(
    app_handle: &tauri::AppHandle<R>,
    pool: &sqlx::SqlitePool,
    hash: String,
    original_name: String,
    mime_type: String,
    size: u64,
    internal_path: String,
) -> Result<AttachmentData, String>
```

**流程**（L295–L377）：

1. **获取当前时间戳**：`SystemTime::now().duration_since(UNIX_EPOCH).as_secs()`（L305–L308）。
2. **SQLite UPSERT**（L311–L324）：
   ```sql
   INSERT INTO attachments (hash, mime_type, size, internal_path, created_at, updated_at)
   VALUES (?, ?, ?, ?, ?, ?)
   ON CONFLICT(hash) DO UPDATE SET internal_path = excluded.internal_path
   ```
   - 冲突键为 `hash`，实现内容寻址去重的数据库层保障。
   - 若同 hash 文件已存在（可能路径不同），更新 `internal_path` 为最新值。
3. **文本提取**：调用 `file_extractor.rs` 的 `try_extract_text(&internal_file_path, &mime_type)`（L328）。
4. **缩略图生成**（L331–L335）：
   - 仅当 `mime_type.starts_with("image/")` 时调用 `generate_thumbnail`。
   - 当前 `generate_thumbnail` 直接返回 `None`（见 §7.2）。
5. **数据库 UPDATE 闭环**（L339–L352）：
   - 若 `extracted_text` 或 `thumbnail_path` 有值，执行 `UPDATE attachments SET extracted_text = ?, thumbnail_path = ? WHERE hash = ?`。
   - 将重资产数据即时持久化，避免前端 volatility 导致丢失。
6. **构造返回结构**：填充 `AttachmentData` 所有字段（L364–L375）。
   - **IPC 截断**：`extracted_text` 强制设为 `None`，避免超大文本通过 Tauri IPC 泄漏到 WebView。

### 7.2 缩略图策略

```rust
pub async fn generate_thumbnail<R: tauri::Runtime>(
    _app_handle: &tauri::AppHandle<R>,
    _original_path: &std::path::Path,
    _hash: &str,
) -> Option<String>
```

**当前实现**（L209–L215）：

```rust
pub async fn generate_thumbnail(...) -> Option<String> {
    None
}
```

**设计决策：缩略图生成已完全下沉到 Android Kotlin 侧**

- Android 端使用 `ThumbnailUtils.createImageThumbnail`，利用硬件加速和系统级缓存，性能远超 Rust `image` crate 的软件解码。
- 桌面端缩略图暂未实现，`generate_thumbnail` 作为占位保留。
- Rust 侧已移除 `image` crate 依赖，减少编译体积和依赖面。
- 缩略图路径由 `register_local_file` 的 `thumbnail_path` 参数传入，或未来桌面端实现后在此函数中生成。

## 9. 安全与性能

### 9.1 路径遍历三层防护

| 层级 | 机制 | 位置 |
|------|------|------|
| 第一层：白名单校验 | `ensure_safe_path` 限定只能在 `app_config_dir` / `app_cache_dir` / `attachments_dir` / `thumbnails_dir` 内，且使用 `canonicalize` 物理展开路径 | `file_manager.rs:218–254` |
| 第二层：路径规范化 | `PathBuf::from` + `starts_with` 天然抵御 `../` 穿越（未规范化前 `starts_with` 对未解析路径仍然有效，因为 Tauri API 返回的路径本身已是绝对规范路径） | 隐式 |
| 第三层：最小权限 | Android 外部存储目录为应用私有（`Android/data/<pkg>/files/`），其他应用无权限访问 | 系统级 |

### 9.2 OOM 三层防御

| 层级 | 机制 | 阈值 | 位置 |
|------|------|------|------|
| 第一层：前端拦截 | attachmentStore.ts 中对非 GIF 图片执行大小和分辨率拦截 | >10MB 或 >8K×8K | `attachmentStore.ts:172–201` |
| 第二层：IPC 兜底 | `store_file` 拒绝超过 100 MB 的 `Vec<u8>` | 100 MB | `file_manager.rs:477–479` |
| 第三层：流式处理 | `register_local_file` 使用 64 KiB 缓冲流式读取；文本提取使用 mmap + 50 MB 上限 | 50 MB / 64 KiB | `file_extractor.rs:587–594` |

### 9.3 内容寻址去重

- **物理层**：`store_file` 和 `register_local_file` 在写入/移动前检查 `internal_file_path.exists()`，存在则跳过写入或删除源文件复用已有副本。
- **数据库层**：`register_attachment_internal` 使用 `INSERT ... ON CONFLICT(hash) DO UPDATE`，确保同一 hash 仅一条记录。
- **注册本地文件层**：`register_local_file` 若发现目标文件已存在，直接删除源临时文件并复用已有副本。

### 9.4 安全文件移动

- `safe_rename` 优先尝试 `fs::rename`，同分区内为操作系统级原子操作。
- 若跨物理分区（EXDEV），降级为 `copy + remove_file`，确保无数据残留。
- 应用于：`register_local_file` 的源路径 → 目标目录、缩略图路径 → `thumbnails_dir`。
- 原子性保证：同分区下文件要么完整出现在目标位置，要么仍留在原位置；跨分区时 `copy` 成功后才删除源文件。

### 9.5 性能特征

| 操作 | 主导开销 | 大致耗时 |
|------|---------|---------|
| 小文件直存 (`store_file`) | SHA-256 计算 + fs::write | 与文件大小成正比，<2MB 通常 <100ms |
| 本地文件注册 (`register_local_file`) | 流式 SHA-256 计算 + safe_rename | 大文件主要开销在哈希计算（约 100–300 MB/s） |
| 文本提取 (`try_extract_text`) | 纯文本：mmap + 编码检测；结构化文档：ZIP 解压/PDF 解析 | 纯文本 50MB 约 50–100ms；PDF 取决于页数与密度 |
| MIME 精修 (`get_refined_mime_type`) | 扩展名匹配（O(1)）或魔数嗅探（前若干字节） | 通常 <1ms；魔数嗅探需一次磁盘读取 |

```text
      │ ├─→ Content-Type: application/octet-stream
      │ ├─→ X-Upload-Token: {token}

      │ └─→ xhr.send(file)  // 直接发送 File 对象，流式传输
      ▼

      │ └─→ 入库、文本提取、返回元数据

```

**双链路分流的设计理由**：

| 维度 | store_file IPC (<2MB) | 高速 TCP 链路 (≥2MB) |
|------|----------------------|----------------------|
| 传输机制 | Tauri IPC (`invoke`) | 本地回环 TCP (`127.0.0.1`) |
| 内存占用 | 前端 `Uint8Array` + 后端 `Vec<u8>` | 流式，无全量缓冲 |
| 进度反馈 | 无（一次性调用） | 有（`xhr.upload.onprogress`） |
| 适用场景 | 小文件、快速响应 | 大文件、需要进度条、避免 IPC 序列化开销 |
| 取消支持 | 无 | 可中断 XHR |

> **关键设计决策备忘**
>
> 1. **Android 附件目录迁移**：从 `app_config_dir/data/attachments`（内部存储）迁移到外部存储 `document_dir/../attachments`。旧迁移逻辑（`migrate_legacy_attachments`）已在 commit `3c5112a` 中移除，该一次性操作已完成历史使命。
> 2. **缩略图下沉**：`generate_thumbnail` 已移除 `image` crate 实现，完全返回 `None`。Android 端缩略图由 Kotlin 侧 `ThumbnailUtils.createImageThumbnail` 硬件加速生成，通过 `thumbnail_path` 参数传入 Rust 侧后执行 `safe_rename` 入库。桌面端缩略图暂未实现。
> 3. **零拷贝路径传输**：`register_local_file` 专为 Android 原生文件选择器设计，全程无 Base64 编解码，流式异步计算 SHA-256，`safe_rename` 优先实现毫秒级移动，EXDEV 跨分区自动降级为 copy+delete。
> 4. **结构化文档文本提取后端化**（commit `123abc8` 引入，commit `8050cf6` 固化）：docx/pptx/xlsx/pdf 的文本提取从前端 JIT 完全 offload 到 Rust 后端，实现代码下沉至独立 `file_extractor.rs`（759 行）。新增 `zip` + `lopdf` 依赖，通过 `fancy_regex` 解析 OOXML 标签提取文本。`register_attachment_internal` 新增即时 UPDATE 闭环持久化提取结果，并通过 IPC 截断（返回前 `extracted_text: None`）避免超大文本泄漏到 WebView。
> 5. **路径规范化安全加固**（commit `0119fde`）：`ensure_safe_path` 引入 `std::fs::canonicalize` 物理展开路径，杜绝字符级前缀欺骗导致的沙盒逃逸漏洞。
> 6. **分片上传 API 删除**（commit `bc20ed2`）：`init_chunked_upload` / `append_chunk` / `finish_chunked_upload` / `cancel_chunked_upload` 及 `UploadManagerState` / `UploadSession` 已完全移除。大文件上传统一走 `high_speed_channel.rs`（临时 TCP 服务器）或 `store_file` IPC（<2MB 小文件）。
