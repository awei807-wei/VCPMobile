# 15. 分布式节点能力 (Distributed Node)

## 1. 概述

### 1.1 模块定位

`distributed/` 是 **VCP Mobile** Rust 核心层中一个**完全独立**的模块，位于 `src-tauri/src/distributed/`，与 `vcp_modules/` 同级。其设计目标是将 VCP Mobile 物理设备转化为 VCPChat 分布式网络中的一个**可感知、可调用、可监控**的移动端节点。

该模块的核心职责包括：

- **节点身份注册**：通过 WebSocket 向 VCPToolBox 主服务器注册自身，上报设备名称、IP 地址与可用工具清单
- **工具能力暴露**：将 Android 设备的硬件传感器、系统状态、前端 API 能力封装为 14+ 个标准化工具，供主服务器远程调用
- **双向状态同步**：持续推送设备状态（电池、CPU、内存、位置等），并接收主服务器的工具执行指令
- **前端状态透传**：通过 Tauri 事件将连接状态实时反馈给 Vue 前端，支持 UI 层的在线/离线指示

### 1.2 与 `vcp_modules/` 的架构边界

`distributed/` 是项目中的**零依赖飞地**（Zero-Dependency Enclave）：

```
┌─────────────────────────────────────────────┐
│              src-tauri/src/                  │
│  ┌─────────────┐    ┌─────────────────────┐ │
│  │ distributed/│    │   vcp_modules/      │ │
│  │  (本模块)    │◄───┼─── 零依赖 ───►│ │
│  │             │    │                     │ │
│  │ • client    │    │ • db_manager        │ │
│  │ • tools     │    │ • sync_service      │ │
│  │ • registry  │    │ • chat_manager      │ │
│  └─────────────┘    └─────────────────────┘ │
│           ▲                                 │
│           │ 仅通过 lib.rs 的 app.manage()   │
│           ▼                                 │
│  ┌─────────────────────────────────────┐   │
│  │  lib.rs — Tauri State + Command 注册 │   │
│  └─────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

- **不导入任何 `vcp_modules/` 的代码**：`client.rs`、`tool_registry.rs`、全部 `tools/` 子模块的 `use` 语句仅限于 `std`、`tokio`、`tauri`、`serde`、`serde_json`、`futures_util`、`tokio_tungstenite`、`reqwest`、`async_trait`、`libc` 以及 `distributed/` 内部路径
- **不读写 `vcp_modules/` 管理的数据库**：所有传感器数据保存在内存中（通过 Android 原生 `SensorStatusManager.kt` → Plugin IPC 通道 `plugin:vcp-mobile|get_sensor_data` 传递，或 `sysfs_utils.rs` 的实时文件读取）
- **独立生命周期**：启动、停止、重连完全由 `distributed::DistributedState` 控制，不依赖 `bootstrap()` 的完成状态
- **可移除性**：如 `mod.rs` 注释所述——删除 `distributed/` 目录 + 移除 `lib.rs` 中 3 处引用后，`cargo check` 仍可通过

## 2. 节点客户端 (`client.rs`)

`client.rs`（507 行）是分布式节点的心脏，负责与 VCPToolBox 主服务器建立并维护 WebSocket 长连接。其设计直接镜像了 VCPChat 中的 `VCPDistributedServer.js`（`DistributedServer` 类）。

### 2.1 连接生命周期

```
前端调用 start_distributed_node
         │
         ▼
┌─────────────────────┐
│  stop() 清理旧连接   │──> 若已运行，先 abort 旧任务
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ spawn connection_loop│──> 在独立 tokio 任务中运行
└─────────────────────┘
           │
           ▼
    ┌──────────────┐
    │  连接主循环   │
    └──────┬───────┘
           │
    ┌──────┴──────┐
    ▼             ▼
  成功          失败
    │             │
    ▼             ▼
┌────────┐  ┌─────────────┐
│run_session│ │指数退避等待  │
│(会话处理) │ │(5s → 60s)   │
└───┬────┘  └─────────────┘
    │             │
    ▼             ▼
断开/错误      检查 shutdown
    │             │
    └──────┬──────┘
           ▼
    循环重连直到 stop()
```

### 2.2 连接 URL 构造

WebSocket 连接地址基于用户传入的 `ws_url`（如 `ws://192.168.1.100:5800`），拼接固定路径：

```
{ws_url}/vcp-distributed-server/VCP_Key={vcp_key}
```

- `ws_url` 中的尾部斜杠会被 `trim_end_matches('/')` 移除，避免双斜杠
- `vcp_key` 用于服务器端的节点认证，日志输出时会替换为 `***` 防止泄露

### 2.3 会话处理 (`run_session`)

成功建立 WebSocket 后，进入会话循环，通过 `tokio::select!` 同时监听三个事件源：

| 事件源 | 触发条件 | 处理逻辑 |
|--------|----------|----------|
| WS 接收 | 主服务器发来 Text/Ping/Close | Text → `handle_incoming()`；Ping → 自动回复 Pong；Close → 退出会话 |
| 定时推送 | 每 30 秒 | 调用 `push_static_placeholders()`，上报全部 StreamingTool 的当前快照值 |
| 关闭信号 | `shutdown_rx.changed()` | 收到 `true` 时主动关闭 WS 发送端并退出 |

### 2.4 入站消息处理 (`handle_incoming`)

所有入站消息先反序列化为 `IncomingEnvelope`，再匹配为 `IncomingMessage` 变体：

| 消息类型 | 对应 `IncomingMessage` | 动作 |
|----------|------------------------|------|
| `connection_ack` | `ConnectionAck { server_id, client_id }` | 更新状态为已连接，触发工具注册、IP 上报、初始占位符推送 |
| `execute_tool` | `ExecuteTool { request_id, tool_name, tool_args }` | 在 `ToolRegistry` 中查找并执行工具，通过 WS 返回 `tool_result` |
| 其他 | `Unknown(String)` | 记录 debug 日志，忽略 |

### 2.5 指数退避重连

连接失败或会话断开后，客户端不会立即重连，而是采用指数退避策略：

- 初始间隔：**5 秒**
- 每次失败后翻倍，上限：**60 秒**
- 成功连接后重置为 5 秒
- 退避期间通过 `tokio::select!` 同时监听 `shutdown_rx.changed()`，确保 `stop()` 能即时中断等待

### 2.6 状态透传

`DistributedClient` 维护一个 `Arc<RwLock<DistributedStatus>>` 状态对象。任何连接状态变化（连接成功、断开、错误）都会通过 Tauri 事件通道推送给前端：

```rust
app.emit("vcp-distributed-status", &status)
```

前端通过 `window.addEventListener("vcp-distributed-status", ...)` 或 Tauri 的事件监听 API 接收，用于更新 UI 中的节点在线指示器。

## 3. 工具注册表 (`tool_registry.rs`)

`tool_registry.rs`（158 行）是工具能力的中央调度器， mirrors VCPChat 中的 `Plugin.js`（`PluginManager` 类）。其核心设计是**三模式工具 trait 系统**。

### 3.1 三种执行模式

| 模式 | Trait | 特征 | 典型代表 |
|------|-------|------|----------|
| **OneShot** | `OneShotTool` | 调用即返回，无前端参与，纯 Rust 侧执行 | `MobileDeviceInfo`、`MobileClipboard` |
| **Interactive** | `InteractiveTool` | 需要前端 UI 参与（相机、生物识别等），触发 Tauri 事件 → Vue 展示 UI → 结果返回 | 预留，Phase 3 实现（如拍照） |
| **Streaming** | `StreamingTool` | 持续产生数据，通过 `update_static_placeholders` 定期推送到主服务器 | `MobileBatteryInfo`、`MobileCPUInfo`、`MobileLocation` |

三种 trait 的共同约束：

```rust
fn manifest(&self) -> ToolManifest;  // 工具元数据
```

OneShot 与 Interactive 额外要求：

```rust
async fn execute(&self, args: Value, app: &AppHandle) -> Result<Value, String>;
```

Streaming 额外要求：

```rust
fn placeholder_key(&self) -> &str;      // 占位符标识，如 "{{MobileBattery}}"
fn poll_interval_secs(&self) -> u64;    // 建议轮询间隔（元数据）
fn read_current(&self) -> Result<String, String>;  // 读取当前快照（必须快/非阻塞）
```

### 3.2 统一包装与注册

由于 Rust trait object 要求类型统一，`ToolRegistry` 内部使用 `ToolEntry` 枚举包装三种工具：

```rust
pub enum ToolEntry {
    OneShot(Arc<dyn OneShotTool>),
    Interactive(Arc<dyn InteractiveTool>),
    Streaming(Arc<dyn StreamingTool>),
}
```

`ToolRegistry` 以 `HashMap<String, ToolEntry>` 存储工具，键为 `manifest().name`。提供三类注册方法：

```rust
pub fn register_oneshot<T: OneShotTool + 'static>(&mut self, tool: T);
pub fn register_interactive<T: InteractiveTool + 'static>(&mut self, tool: T);
pub fn register_streaming<T: StreamingTool + 'static>(&mut self, tool: T);
```

### 3.3 动态发现与调用分发

- **`get_all_manifests()`**：提取全部工具的 `ToolManifest`，用于连接建立后的 `register_tools` 消息
- **`get_all_placeholder_values()`**：遍历所有 `StreamingTool`，调用 `read_current()`，构建 `placeholder_key → value` 映射，用于 30 秒定时推送
- **`execute(tool_name, args, app)`**：按名称查找工具，根据 `ToolEntry` 变体路由到对应执行器。若工具为 Streaming 类型，`execute` 返回当前 `read_current()` 的快照值（字符串包装为 `Value::String`）

## 4. 类型系统 (`types.rs`)

`types.rs`（183 行）定义了分布式节点与主服务器之间的完整协议消息契约，全部基于 `serde` 实现 JSON 序列化/反序列化。

### 4.1 出站消息 (`OutgoingMessage`)

```rust
#[serde(tag = "type", content = "data")]
pub enum OutgoingMessage {
    RegisterTools { server_name, tools },
    ReportIp { server_name, local_ips, public_ip },
    ToolResult { request_id, status, result, error },
    UpdateStaticPlaceholders { server_name, placeholders },
}
```

| 变体 | 序列化后 `type` 字段 | 发送时机 |
|------|---------------------|----------|
| `RegisterTools` | `"register_tools"` | 收到 `connection_ack` 后 |
| `ReportIp` | `"report_ip"` | 注册工具后 |
| `ToolResult` | `"tool_result"` | 完成 `execute_tool` 后 |
| `UpdateStaticPlaceholders` | `"update_static_placeholders"` | 初始连接后 + 每 30 秒定时 |

### 4.2 入站信封与消息 (`IncomingEnvelope` / `IncomingMessage`)

`IncomingEnvelope` 是原始 JSON 的弱类型表示：

```rust
pub struct IncomingEnvelope {
    pub msg_type: String,   // JSON 中的 "type"
    pub data: Option<Value>,
    pub message: Option<String>,
}
```

通过 `parse()` 方法转换为强类型的 `IncomingMessage`：

```rust
pub enum IncomingMessage {
    ConnectionAck { server_id, client_id },
    ExecuteTool { request_id, tool_name, tool_args },
    Unknown(String),
}
```

`Unknown` 变体确保协议前向兼容——主服务器未来新增消息类型不会导致节点崩溃。

### 4.3 工具清单 (`ToolManifest`)

```rust
pub struct ToolManifest {
    pub name: String,
    pub description: String,
    pub parameters: Value,        // JSON Schema 对象
    pub tool_type: String,        // 默认 "mobile"
}
```

`parameters` 字段遵循 JSON Schema 规范，描述工具的入参结构。例如 `MobileClipboard` 的 parameters 定义了 `action`（`"read"` / `"write"`）和可选的 `content`。

### 4.4 连接状态 (`DistributedStatus`)

```rust
pub struct DistributedStatus {
    pub connected: bool,
    pub server_id: Option<String>,
    pub client_id: Option<String>,
    pub registered_tools: usize,
    pub last_error: Option<String>,
}
```

该结构体通过 `#[derive(Serialize, Deserialize, Default, Clone)]` 支持直接作为 Tauri 事件载荷传递给 Vue 前端。

## 6. 关键工具详解

### 6.1 `device_info.rs` — 设备基本信息 (OneShot)

> 源码：`src-tauri/src/distributed/tools/device_info.rs`（36 行）

`MobileDeviceInfo` 是最简单的 OneShot 工具， mirrors VCPChat 的 `WindowSensor` 插件。它不读取任何传感器，仅返回编译期常量：

```rust
async fn execute(&self, _args: Value, _app: &AppHandle) -> Result<Value, String> {
    Ok(json!({
        "os": std::env::consts::OS,       // 如 "android"
        "arch": std::env::consts::ARCH,   // 如 "aarch64"
        "family": std::env::consts::FAMILY // 如 "unix"
    }))
}
```

**设计意图**：在节点首次注册时，让主服务器快速了解该节点的操作系统与指令集架构，为后续工具调用策略提供上下文（例如，知晓是 Android 后不会调用桌面端的 `WindowSensor`）。

---

### 6.2 `battery.rs` — 电池状态 (Streaming)

> 源码：`src-tauri/src/distributed/tools/battery.rs`（85 行）

`MobileBatteryInfo` 是典型的 **Phase 1 Streaming 工具**，完全通过 Android sysfs 读取，不依赖前端。

**读取路径**：

| 数据项 | sysfs 路径 | 处理逻辑 |
|--------|-----------|----------|
| 电量 | `/sys/class/power_supply/battery/capacity` | 原样输出，追加 `%` |
| 状态 | `/sys/class/power_supply/battery/status` | 映射为中文：`Charging`→充电中、`Discharging`→放电中、`Full`→已充满、`Not charging`→未充电 |
| 温度 | `/sys/class/power_supply/battery/temp` | 单位是 0.1°C（如 `320` → `32°C`），除以 10 后格式化 |

**降级策略**：若 sysfs 路径不存在（如部分 OEM 屏蔽电池节点），三个值均返回 `N/A` / `未知`，最终输出 `"电池信息不可用"`，避免 panic 或错误上报。

**占位符**：`{{MobileBattery}}`，推送间隔：60 秒。

---

### 6.3 `location.rs` — GPS 位置 (Streaming)

> 源码：`src-tauri/src/distributed/tools/location.rs`（39 行）

`MobileLocation` 是典型的 **Phase 2 Streaming 工具**，代表 Android 原生传感器协作的设计模式。

**为什么需要 Android 原生？** Android 的精确 GPS 定位需要 `ACCESS_FINE_LOCATION` 权限。通过 Android 原生 `SensorStatusManager.kt` 调用 `LocationManager` 获取位置，并通过 Plugin IPC 通道（`plugin:vcp-mobile|get_sensor_data`）传递给 Rust 层，比通过 WebView 的 JavaScript API 更可靠、权限管理更统一。

**数据流**：

```
Android 原生 (SensorStatusManager.kt)
    │ 调用 LocationManager.requestLocationUpdates()
    │ 突发采样（Burst Sampling）策略降低功耗
    │ 通过 Plugin IPC → invoke("plugin:vcp-mobile|get_sensor_data", { key: "location", value: ... })
    ▼
Rust location.rs::read_current()
    │ 直接调用 IPC 获取传感器快照
    ▼
主服务器请求占位符更新
    │ client.rs 调用 registry.get_all_placeholder_values()
    ▼
location.rs::read_current()
    │ 返回最新坐标或 "等待 Android 传感器采集..."
```

**最大容忍陈旧度**：300 秒（5 分钟）。GPS 在室内可能无法更新，过长的容忍期确保即使信号弱，仍能提供最近一次有效定位。

**占位符**：`{{MobileLocation}}`，推送间隔：120 秒。

---

### 6.4 Plugin IPC 通道 — Phase 2 传感器数据通道

> 原 `frontend_bridge.rs`（43 行）已删除。传感器数据通信现已升级为 Android 原生 Plugin IPC 通道。

`plugin:vcp-mobile|get_sensor_data` 是 Phase 2 传感器（`ambient_sensor`、`location`、`motion_sensor`、`device_status_summary`）的**统一数据通道**。其架构为 Android 原生采集 + IPC 透传：

**核心设计**：

```
Android 原生层 (SensorStatusManager.kt)
    │ 突发采样（Burst Sampling）— 2s 高频采集 + 28s 休眠
    │ LocationManager / SensorManager / 环境传感器
    ▼
Plugin IPC 通道 (CommType::Ipc)
    │ plugin:vcp-mobile|get_sensor_data
    ▼
Rust 工具层 (location.rs / motion_sensor.rs / ambient_sensor.rs)
    │ read_current() → 通过 IPC 获取最新快照
    ▼
主服务器 (update_static_placeholders)
```

**API**：

| 函数 | 调用方 | 作用 |
|------|--------|------|
| `get_sensor_data(key)` (IPC) | 各 Phase 2 StreamingTool | 通过 Plugin IPC 从 Android 原生层获取最新传感器读数 |
| `read_current()` (同步) | `get_all_placeholder_values()` 定时推送 | 在各工具的 `read_current()` 中直接调用 IPC 获取快照值 |

**关键约束**：IPC 调用在 `read_current()` 的同步上下文中执行，必须短时返回（零阻塞等待）。`SensorStatusManager.kt` 在 Android 原生层维护传感器数据的内存缓存，保证 IPC 调用立即返回最近一次采样的值而非阻塞等待新数据。

**时效性差异**：各工具对传感器数据的陈旧容忍度由其自身的 `read_current()` 实现自行管理（运动 120 秒、环境光 180 秒、位置 300 秒），而非在数据通道层统一控制。

### 6.5 `device_status_summary.rs` — 状态聚合摘要 (Streaming)

> 源码：`src-tauri/src/distributed/tools/device_status_summary.rs`（205 行）

`MobileStatusSummary` 是 **Phase 3 聚合工具**，不直接读取新的传感器，而是综合其他工具的数据源，生成一行人类可读的状态摘要。

**聚合策略**：

| 子系统 | 数据来源 | 摘要格式 |
|--------|----------|----------|
| CPU | `/proc/stat` 单点采样 + 热区温度 | `CPU:使用率%/温度°C` |
| GPU | Adreno `gpu_busy_percentage` 或 Mali `utilization` | `GPU:使用率%` |
| 内存 | `/proc/meminfo` | `内存:已用%` |
| 电池 | `/sys/class/power_supply/battery/*` | `电量:XX%[充电中/已满]` |
| 网络 | `/proc/net/route` 默认路由 | `WiFi` / `移动数据` / `离线` |
| 位置 | `location.rs` 通过 IPC 获取（max_age=300s） | 坐标简写 |
| 运动 | `motion_sensor.rs` 通过 IPC 获取（max_age=120s） | 运动状态简写 |

**设计意图**：主服务器若只想快速了解设备概况，无需分别调用 6 个工具，直接读取 `{{MobileStatus}}` 占位符即可。这显著降低了协议开销与 LLM 上下文占用。

**占位符**：`{{MobileStatus}}`，推送间隔：60 秒。

## 7. Tauri 命令接口

`distributed/mod.rs` 中注册了 4 个 Tauri Command，前端通过 `@tauri-apps/api` 的 `invoke()` 调用。

### 7.1 `start_distributed_node`

| 属性 | 值 |
|------|-----|
| 函数签名 | `async fn start_distributed_node(app, state, ws_url, vcp_key, device_name) -> Result<(), String>` |
| 参数 | `ws_url: String` — 主服务器 WebSocket 基础地址；`vcp_key: String` — 认证密钥；`device_name: String` — 节点显示名称 |
| 返回值 | `Ok(())` 或 `Err(String)` |
| 副作用 | 启动后台 WebSocket 连接任务；若已有连接，先停止旧连接 |
| 调用时机 | 用户在前端"分布式节点"设置页中输入服务器地址并点击"启动" |

### 7.2 `stop_distributed_node`

| 属性 | 值 |
|------|-----|
| 函数签名 | `async fn stop_distributed_node(state) -> Result<(), String>` |
| 参数 | 无（从 managed state 获取） |
| 返回值 | 恒为 `Ok(())` |
| 副作用 | 发送 shutdown 信号，abort 后台任务，重置 `connected`/`server_id`/`client_id` |
| 调用时机 | 用户点击"停止"，或应用退出时 |

### 7.3 `get_distributed_status`

| 属性 | 值 |
|------|-----|
| 函数签名 | `async fn get_distributed_status(state) -> Result<DistributedStatus, String>` |
| 返回值 | `DistributedStatus` 的克隆 |
| 副作用 | 无 |
| 调用时机 | 前端页面加载时查询当前状态；或定时轮询更新 UI |

### 7.4 `get_sensor_data` (Plugin IPC)

| 属性 | 值 |
|------|-----|
| 函数签名 | `fn get_sensor_data(key: String) -> Result<String, String>` |
| 参数 | `key` — 传感器标识（如 `"location"`、`"motion"`、`"ambient"`） |
| 返回值 | 传感器读数字符串或错误 |
| 副作用 | 通过 `CommType::Ipc` 调用 Android Plugin `plugin:vcp-mobile\|get_sensor_data` |
| 调用时机 | 各 StreamingTool 的 `read_current()` 方法被 `get_all_placeholder_values()` 定时调用时 |

> **架构变更**：原 `update_sensor_data` Tauri Command 已移除。传感器数据不再从前端 Vue 组件通过 `invoke()` 推送，改为 Rust 工具层通过 Plugin IPC 直接向 Android 原生层（`SensorStatusManager.kt`）拉取。`SensorStatusManager.kt` 使用突发采样策略（Burst Sampling）在 Android 原生层维护传感器数据缓存，IPC 调用返回缓存的最近一次采样值。

### 7.5 前端调用示例

```typescript
import { invoke } from '@tauri-apps/api/core';
import { listen } from '@tauri-apps/api/event';

// 启动节点
await invoke('start_distributed_node', {
  wsUrl: 'ws://192.168.1.100:5800',
  vcpKey: 'my-secret-key',
  deviceName: 'Pixel-8-Pro'
});

// 监听状态变化
listen('vcp-distributed-status', (event) => {
  const status = event.payload as DistributedStatus;
  console.log('Connected:', status.connected);
  console.log('Registered tools:', status.registered_tools);
});

// 传感器数据现在通过 Plugin IPC 通道自动获取，无需前端手动推送
// Android 原生 SensorStatusManager.kt 负责传感器采集和缓存
```

### 8.2 与 `vcp_modules/` 的零依赖证明

**没有出现任何以下路径**：

- `crate::vcp_modules::`
- `super::vcp_modules::`
- `crate::db_manager::`

这保证了 `distributed/` 可以单独理解、单独测试、甚至单独提取为子 crate 而不会破坏编译。

## 9. 术语速查表

| 术语 | 英文/缩写 | 定义 |
|------|-----------|------|
| 分布式节点 | Distributed Node | 物理设备（此处为 Android 手机）作为 VCPChat 分布式网络中的计算与感知终端 |
| 主服务器 | Main Server | VCPToolBox 中的 `VCPDistributedServer`，负责管理节点注册、工具调度与状态聚合 |
| OneShot 工具 | OneShot Tool | 调用后立即返回结果的工具模式，无需前端 UI 参与 |
| Interactive 工具 | Interactive Tool | 需要前端展示 UI、等待用户交互后才能返回结果的工具模式（Phase 3 预留） |
| Streaming 工具 | Streaming Tool | 持续产生数据，通过占位符机制定期推送至主服务器的工具模式 |
| 占位符 | Placeholder | 形如 `{{MobileBattery}}` 的模板标记，主服务器在构造 LLM Prompt 时替换为实时值 |
| Phase 1 传感器 | Phase 1 Sensors | 纯 Rust/sysfs 读取的硬件信息工具（电池、CPU、内存等），不依赖前端 |
| Phase 2 传感器 | Phase 2 Sensors | 依赖前端 Web API（Geolocation、DeviceMotion 等）采集并通过 `update_sensor_data` 推送的数据 |
| Phase 3 聚合 | Phase 3 Aggregation | 综合多源数据生成摘要的复合工具（如 `DeviceStatusSummaryTool`） |
| 插件 IPC 通道 | Plugin IPC Channel | `plugin:vcp-mobile\|get_sensor_data` 通 道，连接 Android 原生 `SensorStatusManager.kt` 传感器采集器与 Rust 工具 `read_current()` 方法 |
| 工具清单 | Tool Manifest | 向主服务器注册时提交的 JSON 元数据，含名称、描述、参数 Schema、类型 |
| 指数退避 | Exponential Backoff | 连接失败后等待时间从 5 秒开始逐次翻倍，上限 60 秒的重连策略 |
| VCP Key | VCP Key | 节点认证密钥，拼接在 WebSocket URL 中供主服务器校验 |
