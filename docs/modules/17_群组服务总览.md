# 17. 群组服务总览

## 1. 概述

### 1.1 领域定位

`group/` 是 VCP Mobile Rust 核心层中负责**群组（Agent Group）**完整生命周期管理与群聊工作流编排的独立领域目录。该领域将多 Agent 协作的复杂逻辑（配置管理、成员关系、发言调度、上下文组装）收敛到内聚的边界上下文内。

领域边界严格限定为：

- **群组配置与元数据**：CRUD、发言模式、成员管理、标签系统
- **群聊应用层编排**：将群组配置注入多 Agent 接力聊天流程，衔接消息服务与 VCP 请求客户端
- **群聊上下文组装**：合并 Agent 个人设定与群组全局设定，生成系统提示词

该领域**不**涉及：
- 具体的发言决策算法本身（`group_speaking_policy.rs` 负责，详见 `docs/modules/03_群聊发言策略引擎.md`）
- 模型推理与 SSE 流式解析本身（由 `vcp_client.rs` 负责）
- 消息持久化的通用逻辑（由 `message_service.rs` 负责）

### 1.2 模块构成

`src-tauri/src/vcp_modules/group/` 目录下包含 5 个模块 + 1 个入口文件：

| 文件 | 行数 | 职责 |
|------|------|------|
| `group_types.rs` | 58 | 群组核心类型定义与 serde 序列化契约 |

| `group_context_assembler.rs` | 42 | 群聊上下文组装器：合并 Agent 设定与群组设定（已合并入 `context_assembler.rs`） |

> 本文档聚焦前 4 个模块；`group_speaking_policy.rs` 已在 `docs/modules/03_群聊发言策略引擎.md` 中独立成篇。

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **类型即契约** | `GroupConfig` 通过 `#[serde(rename_all = "camelCase")]` 实现前端 camelCase 与 Rust snake_case 的零成本映射，所有默认值由独立函数提供 |
| **缓存优先** | `GroupManagerState` 使用 `DashMap` 实现无锁并发缓存，读取优先命中内存 |
| **按群组隔离锁** | 每个 Group ID 拥有独立的 `Arc<Mutex<()>>`，避免 A 的保存操作阻塞 B 的读取 |
| **事务化写入** | 所有涉及 `groups` 表、`group_members` 表与 `topics` 表的变更均包裹在 `sqlx::Transaction` 中 |
| **同步感知** | 写入时自动计算配置哈希，对比旧哈希后决定是否向 `sync_service` 发送变更通知 |
| **串行接力约束** | 群聊内部多 Agent 调度必须串行执行，一个 Agent 的流式请求完全结束后才启动下一个 |

## 2. 类型契约（group_types.rs）

`group_types.rs` 是整个 Group 领域的**类型契约层**。它仅包含 58 行代码，不做任何业务逻辑，只定义数据结构、序列化行为与默认值策略。

> 文件位置：`src-tauri/src/vcp_modules/group/group_types.rs`

### 2.1 GroupConfig 结构体

`GroupConfig` 是群组的完整配置载体，前端群组设置面板的所有字段均与此结构一一对应。

> 定义位置：`src-tauri/src/vcp_modules/group/group_types.rs` 第 17–58 行

| 字段 | 类型 | serde 默认值 | 说明 |
|------|------|-------------|------|
| `id` | `String` | `default` (空串) | 群组唯一标识，通常格式为 `____{ sanitized_name }_{ timestamp }` |
| `name` | `String` | `default_group_name` (`"Unnamed Group"`) | 显示名称 |
| `avatar_calculated_color` | `Option<String>` | `default` (`None`) | 头像主色调（十六进制），从 `avatars` 表动态获取 |
| `members` | `Vec<String>` | `default` (空数组) | 成员 Agent ID 列表 |
| `mode` | `String` | `default_group_mode` (`"sequential"`) | 发言模式：`sequential` / `naturerandom` / `invite_only` |
| `member_tags` | `Option<serde_json::Value>` | `default` (`None`) | 成员标签映射（`agentId → "tag1, tag2"`） |
| `group_prompt` | `Option<String>` | `default` (`None`) | 群组全局提示词，附加到每个成员的系统提示词之后 |
| `invite_prompt` | `Option<String>` | `default` (`None`) | 邀请发言提示词模板，支持 `{{VCPChatAgentName}}` 占位符 |
| `use_unified_model` | `bool` | `default` (`false`) | 是否强制所有成员使用统一模型 |
| `unified_model` | `Option<String>` | `default` (`None`) | 统一模型名称，仅在 `use_unified_model = true` 时生效 |
| `topics` | `Vec<Topic>` | `default` (空数组) | 该群组下的话题列表 |
| `tag_match_mode` | `Option<String>` | `default_tag_match_mode` (`"strict"`) | 标签匹配模式：`strict` / `natural` |
| `created_at` | `i64` | `default` (`0`) | 创建时间戳（毫秒级 Unix 时间） |

### 2.2 Serde 映射策略

```rust
#[derive(Debug, Serialize, Deserialize, Clone)]
#[serde(rename_all = "camelCase")]
pub struct GroupConfig { ... }
```

- **`rename_all = "camelCase"`**：Rust 侧使用 `group_prompt`，前端 JSON 使用 `groupPrompt`，无需手动映射。
- **自定义默认值函数**：`default_group_name`、`default_group_mode`、`default_tag_match_mode` 与 `group_service.rs` 中 `create_group` 的显式赋值保持同步。

### 2.3 与 Topic 类型的关系

`GroupConfig.topics` 的类型 `Vec<Topic>` 来自 `crate::vcp_modules::topic_types::Topic`。`Topic` 结构体由话题领域定义，`group_types.rs` 仅作为消费者引用。这体现了领域间的**单向依赖**原则：Group 领域知道 Topic，但 Topic 领域不感知 Group。

## 3. 群组 CRUD 服务（group_service.rs）

`group_service.rs` 是 Group 领域的**核心 Facade**，承担群组配置的生命周期管理、缓存治理与同步联动。全模块 494 行。

> 文件位置：`src-tauri/src/vcp_modules/group/group_service.rs`

### 3.1 GroupManagerState（全局托管状态）

`GroupManagerState` 被 Tauri 的 `app.manage()` 注入为全局单例，供所有 Group 相关的 Tauri Command 共享。

> 定义位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 17–39 行

```rust
pub struct GroupManagerState {
    /// 配置缓存: group_id -> GroupConfig
    pub caches: DashMap<String, GroupConfig>,
    /// 任务队列锁: group_id -> Mutex
    pub locks: DashMap<String, Arc<Mutex<()>>>,
}
```

| 成员 | 类型 | 作用 |
|------|------|------|
| `caches` | `DashMap<String, GroupConfig>` | 内存级配置缓存，无锁并发哈希表 |
| `locks` | `DashMap<String, Arc<Mutex<()>>>` | 按 Group ID 隔离的互斥锁，保证同一群组的并发写操作串行化 |

**`acquire_lock` 机制**：与 `AgentConfigState` 同源设计——通过 `DashMap` 的 `entry` API 按需创建 `Arc<Mutex<()>>`，按 ID 隔离锁避免全局锁瓶颈。

### 3.2 读取配置（read_group_config / read_group_config_internal）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 41–141 行

**三级读取策略**：

```
前端调用 read_group_config(group_id)
           │
           ▼
┌─────────────────────┐
│ 1. 命中内存缓存?     │──> DashMap.get(group_id)
│    (caches)         │    O(1) 无锁读取
└──────────┬──────────┘
      Yes  │      No
      ┌────┘        └────┐
      ▼                  ▼
 直接返回          ┌─────────────────────┐
 克隆结果          │ 2. 查询 SQLite      │
                  │    groups + avatars  │
                  │    + topics 三表联查 │
                  │    + group_members   │
                  └──────────┬──────────┘
                             │
                        ┌────┴────┐
                        ▼         ▼
                     有记录     无记录
                      │          │
                      ▼          ▼
               ┌─────────────┐  返回错误
               │ 反序列化为   │  "Group {} not found"
               │ GroupConfig  │
               │ 写入缓存     │
               └──────┬──────┘
                      ▼
                返回配置对象
```

**数据库查询细节**：

- **`groups` 表**：读取 `name`、`mode`、`group_prompt`、`invite_prompt`、`use_unified_model`、`unified_model`、`tag_match_mode`、`created_at`
- **`avatars` 表（LEFT JOIN）**：通过 `owner_id = group_id AND owner_type = 'group'` 关联，提取 `dominant_color`
- **`group_members` 表**：读取该群组的成员列表，按 `sort_order ASC` 排序，同时解析 `member_tag`
- **`topics` 表**：读取该群组下全部未删除话题，按 `updated_at DESC` 排序

### 3.3 保存配置（save_group_config）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 143–159 行

公共 Tauri Command，接收完整的 `GroupConfig` 对象，执行原子化写入。

1. 校验 `group_id` 非空
2. `acquire_lock(id)` → `.lock().await` 获取该群组的专属互斥锁
3. 委托 `internal_write_group_config` 实际持久化

### 3.4 增量更新（update_group_config）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 187–214 行

前端在修改单个字段（如仅调整 `group_prompt`）时，无需构造完整的 `GroupConfig`。`update_group_config` 提供 JSON Patch 风格的增量合并：

1. 获取锁：`acquire_lock` → `lock().await`
2. 读取当前配置：`read_group_config`
3. JSON 层级合并：将当前配置序列化为 `serde_json::Value`，遍历 `updates` 的每个键值对执行 `insert`
4. 反序列化回 `GroupConfig`
5. 原子写入：委托 `internal_write_group_config`

> 与 `agent_service.rs` 中的 `update_agent_config` 算法同源。

### 3.5 内部写入逻辑（internal_write_group_config）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 313–461 行

这是群组持久化的**唯一收敛点**，所有写入路径（`save_group_config`、`update_group_config`、同步导入）最终都调用此函数。

#### 3.5.1 同步哈希计算与变更检测

```rust
let dto = GroupSyncDTO::from(config);
let config_hash = HashAggregator::compute_group_config_hash(&dto);
```

- 将 `GroupConfig` 转换为 `GroupSyncDTO`（同步领域定义的传输对象）
- 若 `from_sync = false`（本地发起），查询旧哈希并对比：
  - 旧哈希不存在或不相等 → 向 `sync_service` 发送 `SyncCommand::NotifyLocalChange`
  - 哈希相等 → 跳过同步通知

#### 3.5.2 SQLite UPSERT（groups 表）

```sql
INSERT INTO groups (
    group_id, name, mode,
    group_prompt, invite_prompt, use_unified_model, unified_model,
    tag_match_mode, created_at, config_hash, updated_at
) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
ON CONFLICT(group_id) DO UPDATE SET
    name = excluded.name,
    mode = excluded.mode,
    group_prompt = excluded.group_prompt,
    invite_prompt = excluded.invite_prompt,
    use_unified_model = excluded.use_unified_model,
    unified_model = excluded.unified_model,
    tag_match_mode = excluded.tag_match_mode,
    config_hash = excluded.config_hash,
    updated_at = excluded.updated_at
```

#### 3.5.3 成员关系重建

```sql
DELETE FROM group_members WHERE group_id = ?
```

随后遍历 `config.members`，按索引写入 `sort_order`：

```sql
INSERT INTO group_members (
    group_id, agent_id, member_tag, sort_order, updated_at
) VALUES (?, ?, ?, ?, ?)
```

**全量重建策略**：不执行增量 Diff，而是先删除再插入。简化了成员顺序变更（拖拽排序）的处理，代价是写入量略大，但群组通常成员数极少（<20），性能可忽略。

#### 3.5.4 话题 Upsert

遍历 `config.topics`，对每个话题执行 `INSERT ... ON CONFLICT(topic_id) DO UPDATE SET title, locked, unread, updated_at`，并调用 `HashAggregator::bubble_topic_hash` 初始化/更新 Topic 自身哈希。

#### 3.5.5 聚合哈希冒泡与缓存刷新

事务提交后，若 `skip_bubble = false`，开启新事务调用 `HashAggregator::bubble_group_hash`，将群组配置哈希向上汇总。写入成功后立即更新 `DashMap` 缓存。

### 3.6 获取全部群组（get_groups）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 161–185 行

1. 查询 `groups` 表中所有 `deleted_at IS NULL` 的 `group_id`
2. 对每个 ID 调用 `read_group_config`（自动利用缓存）
3. 收集成功的结果，返回 `Vec<GroupConfig>`

> 前端群组侧边栏的数据源。

### 3.7 删除群组（delete_group）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 463–494 行

采用**软删除**策略：

```sql
UPDATE groups SET deleted_at = ? WHERE group_id = ?
```

副作用：
- 清除内存缓存：`state.caches.remove(&group_id)`
- 释放该群组的锁：`state.locks.remove(&group_id)`
- 通知同步中心：发送 `SyncCommand::NotifyDelete { data_type: Group, id: group_id }`

### 3.8 创建群组（create_group）

> 函数位置：`src-tauri/src/vcp_modules/group/group_service.rs` 第 216–311 行

**ID 生成策略**：

```rust
let base_id = name
    .chars()
    .filter(|c| c.is_alphanumeric() || *c == '_' || *c == '-')
    .collect::<String>();
let group_id = format!("____{}_{}", base_id, timestamp);
```

- 前缀 `____` 与 Agent ID 区分（Agent ID 无前缀）
- 从名称中提取合法字符，拼接毫秒级时间戳

**默认话题**：每个新群组自动创建一个默认话题：

| 属性 | 值 |
|------|-----|
| `topic_id` | `group_topic_{timestamp}` |
| `name` | `"主要群聊"` |
| `locked` | `true`（默认锁定，不可删除） |
| `unread` | `false` |

**默认配置**：

- `mode`: `"sequential"`
- `member_tags`: `{}`
- `invite_prompt`: 包含 `{{VCPChatAgentName}}` 的默认邀请模板
- `tag_match_mode`: `"strict"`

**事务边界**：整个创建过程（`groups` 表插入 + `topics` 表插入）包裹在单一事务中，失败时自动回滚。事务提交后额外执行聚合哈希冒泡。

## 4. 群聊消息编排流程（group_chat_application_service.rs）

`group_chat_application_service.rs` 是 Group 领域的**应用层编排器**，负责将群组配置注入群聊流程，衔接用户输入、历史上下文、发言决策与 VCP 请求客户端。全模块 414 行。

> 文件位置：`src-tauri/src/vcp_modules/group/group_chat_application_service.rs`

### 4.1 核心数据结构

#### GroupChatPayload

> 定义位置：第 20–28 行

```rust
#[derive(Debug, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct GroupChatPayload {
    pub group_id: String,
    pub topic_id: String,
    pub user_message: ChatMessage,

    pub vcp_api_key: String,

```

| 字段 | 说明 |
|------|------|
| `group_id` / `topic_id` | 路由信息，决定消息归属 |
| `user_message` | 用户发送的原始消息对象 |
| `vcp_url` / `vcp_api_key` | VCP 服务器端点与鉴权密钥 |

#### GroupChatParams

> 定义位置：第 30–37 行

内部扩展结构，在 `GroupChatPayload` 基础上增加 `stream_channel: Option<Channel<StreamEvent>>`，用于流式事件推送。`Option` 包裹使其可支持非流式调用场景（如重放/同步）。

### 4.2 公共命令：handle_group_chat_message

> 函数位置：第 379–414 行

Tauri IPC 入口，直接委托给 `internal_process_group_chat_message`，固定参数 `append_user_msg = true`。

### 4.3 内部编排逻辑（internal_process_group_chat_message）

> 函数位置：第 39–377 行

**6 阶段编排流水线**（串行接力约束）：

```
输入: GroupChatParams + stream_channel
       │
       ▼
┌─────────────────────────────────────────────────────────────┐
│ 0. 重置中断标记                                              │
│    cancelled_turns.0.remove(&topic_id)                       │
│    ──> 确保开启新回合，清除上一轮残留的中断信号                │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 1. 加载群组配置                                              │
│    read_group_config(group_id)                               │
│    ──> 获取 mode / members / unified_model / invite_prompt   │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 加载成员配置                                              │
│    for member_id in group_config.members:                    │
│        read_agent_config(member_id, allow_default=false)     │
│    ──> 仅保留成功读取的 Agent，过滤无效/已删除成员             │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. 持久化用户消息 (可选)                                     │
│    if append_user_msg:                                       │
│        message_service::append_single_message(...)           │
│    ──> 重新生成场景设为 false，避免重复入库                   │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 决策引擎：谁该说话？                                       │
│    if mode == "sequential": active_member_configs.clone()    │
│    if mode == "naturerandom":                                │
│        determine_naturerandom_speakers(...)                  │
│    else: return {"status": "no_ai_response"}                 │

                           ▼
┌─────────────────────────────────────────────────────────────┐

│        assemble_history_for_vcp(...)                         │
│        perform_vcp_request(...)                              │

└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. 回合结束信号                                              │
│    emit("vcp-group-turn-finished", {groupId, topic_id,      │
│                                       agentIds})             │
│    cancelled_turns.0.remove(&topic_id)                       │
│    ──> 无论成功/失败/中断，均发射信号给前端清理 UI 状态        │
└─────────────────────────────────────────────────────────────┘
```

#### 4.3.1 上下文加载策略

模块**两次加载历史记录**，服务于不同目的：

| 加载调用 | 深度限制 | `include_extracted_text` | 用途 |
|---------|---------|-------------------------|------|
| `load_chat_history_internal(..., Some(8), ..., false)` | 最近 8 条 | `false` | **发言决策**：`determine_naturerandom_speakers` 只需轻量上下文 |
| `load_chat_history_internal(..., None, ..., true)` | 全部 | `true` | **VCP 请求**：组装完整 messages 数组，需包含附件提取文本 |

#### 4.3.2 统一模型路由

```rust
let model_to_use = if group_config_inner.use_unified_model {
    if let Some(ref unified) = group_config_inner.unified_model {
        if !unified.is_empty() { unified.clone() }
        else { speaker.model.clone() }
    } else { speaker.model.clone() }
} else { speaker.model.clone() };
```

- 若群组启用统一模型且名称非空，所有成员强制使用该模型
- 否则回退到成员各自的 `model` 配置

#### 4.3.3 Invite Prompt 注入

`invite_prompt` 作为**独立 user 消息**追加到 messages 列表末尾（system prompt 之前）：

```rust
if let Some(invite_prompt) = &group_config_inner.invite_prompt {
    let processed_invite = invite_prompt.replace("{{VCPChatAgentName}}", &agent_name);
    messages.push(json!({ "role": "user", "content": processed_invite }));
}
```

- 支持模板变量 `{{VCPChatAgentName}}`，替换为当前发言 Agent 的名称
- 该设计使邀请提示词在对话序列中获得更合理的角色定位

#### 4.3.4 Tarven 规则注入

```rust
crate::vcp_modules::chat::context_injection::apply_tarven_pipeline(
    &db_pool, &topic_id, &agent_name, "group", &mut messages,
).await?;
```

在 messages 组装完成后、发起请求前，调用 Tarven 上下文注入引擎对消息列表进行后处理。

#### 4.3.5 串行接力与中断机制

```rust
for speaker in speakers {
    if cancelled_turns.0.contains(&topic_id) {
        println!("[GroupChatAppService] Group turn for topic {} was cancelled. Breaking loop.", topic_id);
        break;
    }
    // ... 组装上下文、发起请求、入库 ...
    full_history_for_context.push(ai_msg.clone());
}
```

**关键约束**：群聊内部必须串行。每个 Agent 的流式请求完全结束后，其生成的 `ai_msg` 被立即追加到 `full_history_for_context`，供下一个 Agent 的上下文组装使用。这种"实时追加"确保了多 Agent 接力时每个成员都能看到前面成员的回复。

**中断检查点**：每轮循环开始前检查 `CancelledGroupTurns`（`DashMap<String, ()>`）中是否存在当前 `topic_id`。若存在，立即 `break`，停止后续 Agent 的调度。

#### 4.3.6 Thinking 事件与流式生命周期

```rust
// 发射 thinking 事件，让前端为当前接力的 Agent 创建思考占位消息
if let Some(chan) = &stream_channel {
    let _ = chan.send(StreamEvent::thinking(message_id.clone(), context));
}

// 启动前台服务保活
start_stream_service_inner(&app_handle, &agent_name)

// 执行请求 (串行等待)
let res_result = perform_vcp_request(...).await;

// 停止前台服务
stop_stream_service_inner(&app_handle, &agent_name)
```

- **Backend-Driven**：`message_id` 由后端生成（`msg_group_{user_msg_id}_{agent_id}_{timestamp}`），通过 `StreamEvent::thinking` 推送给前端初始化气泡
- **前台服务**：每个 Agent 请求前后分别启动/停止 Android 前台保活服务，降低 OEM 杀后台概率
- **容错**：前台服务启动/停止失败仅打印日志，不阻断聊天流程

#### 4.3.7 回合结束信号

无论串行循环以何种方式退出（全部完成、中断 break、错误），均发射 `vcp-group-turn-finished` 事件：

```rust
let _ = app_handle.emit("vcp-group-turn-finished", json!({
    "groupId": group_id,
    "topic_id": topic_id,
    "agentIds": agent_ids
}));
```

前端据此清理 loading 状态、更新 UI。

## 5. 群聊上下文组装逻辑

> **架构变更**：原 `group_context_assembler.rs`（42 行）已合并入 `context_assembler.rs`（`src-tauri/src/vcp_modules/chat/context_assembler.rs`），其逻辑仍保留在 `assemble_group_context` 函数中，但代码物理位置已迁移。以下描述的是合并后的功能架构。

群聊上下文组装器负责将单个 Agent 的个人设定与群组全局设定合并，生成最终注入到 VCP 请求的系统提示词。其逻辑现在位于 `context_assembler.rs` 中。

### 5.1 核心函数：assemble_group_context

> 定义位置：第 6–41 行

```rust

    agent_config: &AgentConfig,
    group_config: &GroupConfig,
    active_members: &[AgentConfig],

```

**组装流程**：

```
输入: agent_config, group_config, active_members
       │
       ▼
┌─────────────────────────────────────────────┐
│ 1. 选取 Agent 系统提示词                      │
│    优先级: mobile_system_prompt > system_prompt│
│    ──> 若 mobile_system_prompt 非空则优先使用  │
└──────────┬──────────────────────────────────┘

└──────────┬──────────────────────────────────┘
           ▼
    返回 system_prompt
```

### 5.2 SessionWatcher 占位符

```rust
if final_group_prompt.contains("{{VCPChatGroupSessionWatcher}}") {
    let session_info = json!({
        "groupId": group_config.id,
        "groupName": group_config.name,
        "activeMembers": active_members.iter().map(|m| {
            json!({ "id": m.id, "name": m.name, "model": m.model })

```

当群主的 `group_prompt` 中包含 `{{VCPChatGroupSessionWatcher}}` 时，系统自动将当前群组的会话元数据（ID、名称、活跃成员列表）以 JSON 形式注入，供 Agent 感知群聊环境。

| 维度 | Agent 单聊 (`agent_chat_application_service`) | 群聊 (`group_chat_application_service`) |

| System Prompt 来源 | `mobile_system_prompt` > `system_prompt` | 同上 + 追加 `group_prompt` |

## 8. 设计决策与注意事项

### 8.1 为何 `group_types.rs` 与 `group_service.rs` 分离？

与 Agent 领域同源设计：将类型定义从业务逻辑中剥离，避免循环依赖。`group_chat_application_service.rs` 需要引用 `GroupConfig`，而 `group_service.rs` 又需要被 `group_chat_application_service.rs` 调用。分离后，`group_types` 成为所有模块的公共底座。

### 8.2 群组成员关系的全量重建策略

`internal_write_group_config` 在写入成员时采用 `DELETE + INSERT` 而非增量 Diff：

1. 先执行 `DELETE FROM group_members WHERE group_id = ?`
2. 再按 `config.members` 顺序重新插入

**原因**：
- 简化了拖拽排序（`sort_order` 变更）的处理逻辑
- 简化了成员增减的处理逻辑
- 群组成员数通常极少（<20），全量重建的 IO 开销可忽略
- 避免了复杂的 Diff 算法和边界情况处理

### 8.3 群聊的串行接力约束

`group_chat_application_service.rs` 明确约束：群聊内部多 Agent 调度必须**串行**执行。

```rust
for speaker in speakers {
    // ... 等待 perform_vcp_request 完全结束 ...
}
```

**原因**：
1. **上下文依赖**：后续 Agent 需要看到前面 Agent 的回复内容，`full_history_for_context` 在每次循环后追加
2. **资源控制**：并发发起多个 LLM 请求会导致移动端的网络、内存、电量压力剧增
3. **用户体验**：串行流式输出让用户能逐步看到各 Agent 的回复，避免同时涌入多条消息的混乱

### 8.4 中断机制的粒度

群聊中断以 **Topic** 为粒度，而非 Group 或全局：

```rust
if cancelled_turns.0.contains(&topic_id) { break; }
```

这意味着：
- 用户可以在群聊 A 的话题 T1 中中断，同时群聊 B 的话题 T2 不受影响
- 同一群聊的不同话题也相互隔离
- `CancelledGroupTurns` 使用 `DashMap<String, ()>`，Key 为 `topic_id`

### 8.5 Invite Prompt 的独立 User 消息定位

`invite_prompt` 从 system prompt 的一部分迁移为**独立 user 消息**，是 nudge trigger 对齐的一部分：

- 在历史消息组装完成后追加到 messages 列表末尾（system prompt 之前）
- 在对话序列中获得更合理的角色定位：它是对当前发言者的"邀请"，而非全局系统设定
- 支持 `{{VCPChatAgentName}}` 模板变量，在追加前替换为当前发言 Agent 的名称
