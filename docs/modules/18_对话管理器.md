# 18_对话管理器（Chat Manager）

## 1. 概述

### 1.1 模块定位

`chat_manager.rs` 是 VCP Mobile 核心层（Rust 后端）的**对话历史 Tauri 命令门面模块**（Facade Layer），位于 `src-tauri/src/vcp_modules/chat/chat_manager.rs`（211 行）。该模块本身不承担复杂业务逻辑，而是作为前端 Vue 3 对话引擎与后端 `message_service.rs` 之间的** IPC 路由与数据契约层**，负责定义对话相关的核心数据结构，并将 Tauri Command 调用无缝委托给下层业务服务。

其核心职责包括：
- 定义前端与后端共享的消息数据契约（`ChatMessage`、`Attachment`、`HistoryChunk`）
- 暴露 6 个 Tauri Command，覆盖历史记录的加载、追加、更新、删除与截断
- 支持**流式（Channel 逐条推送）**与**批量（一次性返回）**两种历史记录加载模式
- 将所有持久化与业务规则委托给 `message_service.rs`，保持自身薄层与可替换性

### 1.2 职责边界

| 职责领域 | 具体行为 | 对应源码位置 |
|---------|---------|------------|
| 数据契约定义 | `ChatMessage`、`Attachment`、`HistoryChunk` 的序列化/反序列化结构 | `Attachment:6`, `ChatMessage:34`, `HistoryChunk:77` |
| 流式历史加载 | 通过 `Channel<HistoryChunk>` 逐条向前端推送消息，避免大数组阻塞 IPC | `load_chat_history_streamed:87` |
| 批量历史加载 | 一次性返回 `Vec<ChatMessage>`，适用于前端初始数据直灌 | `load_chat_history:119` |
| 消息追加 | 将单条消息委托给 `message_service::append_single_message`，返回解析后的 `ContentBlock` 列表 | `append_single_message:141` |
| 消息修补 | 将单条消息的更新委托给 `message_service::patch_single_message` | `patch_single_message:161` |
| 消息删除 | 按 `topic_id` + `msg_ids` 批量软删除 | `delete_messages:182` |
| 历史截断 | 按时间戳截断指定话题的后续消息，用于上下文清理或分支回退 | `truncate_history_after_timestamp:191` |

### 1.3 调用入口

```text
Vue 3 前端（对话引擎 / chatHistoryStore）
    ↓ Tauri IPC invoke
lib.rs（命令路由，注册 chat_manager.rs 的 #[tauri::command]）
    ↓ 调用
chat_manager.rs（本模块：参数校验与类型转换）
    ↓ 委托
message_service.rs（业务逻辑：SQL 查询、附件对齐、渲染缓存）
    ↓ 操作
SQLite（messages / render_cache / attachments 表）
```

内部 Rust 调用者（不经过前端）：
```text
agent_chat_application_service.rs ──→ message_service.rs ──→ 单聊消息持久化
group_chat_application_service.rs ──→ message_service.rs ──→ 群聊消息持久化
sync_service.rs / sync_executor.rs ──→ message_service.rs ──→ 同步消息写入
```

## 2. 核心类型与数据结构

### 2.1 ChatMessage

```rust
#[derive(Debug, Serialize, Deserialize, Clone, Default)]
pub struct ChatMessage {
    pub id: String,
    pub role: String,
    pub name: Option<String>,       // serde(alias = "senderName")
    #[serde(skip_serializing_if = "String::is_empty")]
    pub content: String,
    pub timestamp: u64,
    #[serde(rename = "isThinking")]
    pub is_thinking: Option<bool>,
    #[serde(rename = "agentId", skip_serializing_if = "Option::is_none")]
    pub agent_id: Option<String>,
    #[serde(rename = "groupId", skip_serializing_if = "Option::is_none")]
    pub group_id: Option<String>,
    #[serde(rename = "topicId", skip_serializing_if = "Option::is_none")]
    pub topic_id: Option<String>,
    #[serde(rename = "isGroupMessage", skip_serializing_if = "Option::is_none")]
    pub is_group_message: Option<bool>,
    #[serde(rename = "finishReason", skip_serializing_if = "Option::is_none")]
    pub finish_reason: Option<String>,
    pub attachments: Option<Vec<Attachment>>,
    pub blocks: Option<serde_json::Value>,
    pub shell: Option<crate::vcp_modules::pre_renderer::MessageShell>,
    pub content_hash: Option<String>,
}
```

| 字段 | 类型 | 说明 |
|-----|------|------|
| `id` | `String` | 消息唯一标识（UUID），真理之源 |
| `role` | `String` | 角色：`user` / `assistant` / `system` |
| `name` | `Option<String>` | 发送者名称（单聊为 Agent 名，群聊为发言者名） |
| `content` | `String` | 消息原始文本内容。空字符串在序列化时被跳过 |
| `timestamp` | `u64` | Unix 时间戳（毫秒），排序与截断依据 |
| `is_thinking` | `Option<bool>` | 是否为 thinking 占位消息（Backend-Driven Streaming 架构） |
| `agent_id` | `Option<String>` | 所属 Agent ID |
| `group_id` | `Option<String>` | 所属群组 ID |
| `topic_id` | `Option<String>` | 所属话题 ID |
| `is_group_message` | `Option<bool>` | 是否群聊消息 |
| `finish_reason` | `Option<String>` | 流式结束原因：`completed` / `cancelled_by_user` / `error` / `null` |
| `attachments` | `Option<Vec<Attachment>>` | 附件列表 |
| `blocks` | `Option<serde_json::Value>` | 预渲染内容块（由 `content_parser.rs` 生成） |
| `shell` | `Option<MessageShell>` | 预渲染消息外壳（实验性字段） |
| `content_hash` | `Option<String>` | 内容哈希，用于同步冲突检测与去重 |

`ChatMessage` 是前后端对话系统的**核心数据原子**。前端 `chatHistoryStore` 中的消息对象即为此结构的 TypeScript 镜像。`serde` 属性密集，用于兼容前端 camelCase 命名习惯与空值省略策略。

### 2.2 Attachment

```rust
#[derive(Debug, Serialize, Deserialize, Clone, Default)]
pub struct Attachment {
    pub r#type: String,
    pub src: String,              // 物理存储路径（真理之源）
    pub name: String,
    pub size: u64,
    pub hash: Option<String>,
    pub status: Option<String>,
    pub internal_path: String,    // serde(rename = "internalPath")
    pub extracted_text: Option<String>,   // serde(rename = "extractedText")
    pub image_frames: Option<Vec<String>>, // serde(rename = "imageFrames")
    pub thumbnail_path: Option<String>,   // serde(rename = "thumbnailPath")
    pub created_at: Option<u64>,  // serde(rename = "createdAt")
}
```

| 字段 | 类型 | 说明 |
|-----|------|------|
| `r#type` | `String` | MIME 类型或简化类型标识：`image` / `audio` / `video` / `file` |
| `src` | `String` | 原始文件物理路径，用于超栈文件追踪与跨端同步参考 |
| `name` | `String` | 原始文件名 |
| `size` | `u64` | 文件大小（字节） |
| `hash` | `Option<String>` | 文件内容哈希（SHA-256），用于同步去重与完整性校验 |
| `status` | `Option<String>` | 附件处理状态：`pending` / `extracted` / `failed` |
| `internal_path` | `String` | 应用内部存储路径（实际读取位置） |
| `extracted_text` | `Option<String>` | 文件提取文本（如 PDF/DOCX/OCR 结果），大体积字段 |
| `image_frames` | `Option<Vec<String>>` | 视频抽帧后的 JPEG base64 列表 |
| `thumbnail_path` | `Option<String>` | 缩略图内部路径 |
| `created_at` | `Option<u64>` | 附件创建时间戳 |

`Attachment` 采用**双路径设计**：`src` 保留原始路径作为元数据与同步参考，`internal_path` 指向实际可读的内部存储位置。`extracted_text` 在列表加载时默认不返回（`include_extracted_text = false`），防止大文本撑爆 IPC。

### 2.3 HistoryChunk

```rust
#[derive(Debug, Serialize, Deserialize, Clone)]
pub struct HistoryChunk {
    pub message: ChatMessage,
    pub index: usize,
    pub is_last: bool,
}
```

`HistoryChunk` 专用于**流式历史加载**场景（`load_chat_history_streamed`）。前端通过 `Channel<HistoryChunk>` 逐条接收消息，`index` 表示当前消息在结果集中的位置，`is_last` 标记最后一条，便于前端实现渐进式列表渲染与加载状态收尾。

## 3. 核心函数说明

### 3.1 公共接口（Tauri Commands）

| 函数签名 | 返回值 | 功能简述 | 源码位置 |
|---------|--------|---------|---------|
| `load_chat_history_streamed(app_handle, owner_id, owner_type, topic_id, limit, offset, on_message)` | `Result<usize, String>` | 流式加载历史记录，通过 `Channel<HistoryChunk>` 逐条推送 | `:87` |
| `load_chat_history(app_handle, owner_id, owner_type, topic_id, limit, offset)` | `Result<Vec<ChatMessage>, String>` | 批量加载历史记录，一次性返回完整数组 | `:119` |
| `append_single_message(app_handle, db_state, owner_id, owner_type, topic_id, message)` | `Result<Vec<ContentBlock>, String>` | 追加单条消息，持久化后返回解析块 | `:141` |
| `patch_single_message(app_handle, db_state, owner_id, owner_type, topic_id, message)` | `Result<Vec<ContentBlock>, String>` | 更新单条消息内容，触发重新解析 | `:161` |

#### 3.1.1 load_chat_history_streamed

**流式加载模式**，适用于前端需要渲染大历史记录列表的场景，避免单次 IPC 传输 megabyte 级 JSON 导致 UI 卡顿。

```rust
pub async fn load_chat_history_streamed(
    app_handle: tauri::AppHandle,
    owner_id: String,
    owner_type: String,
    topic_id: String,
    limit: Option<usize>,
    offset: Option<usize>,
    on_message: tauri::ipc::Channel<HistoryChunk>,
) -> Result<usize, String>
```

- **委托路径**：调用 `message_service::load_chat_history_internal(..., false, false)`
- `include_extracted_text = false`：前端列表渲染不需要大体积提取文本
- 内部遍历返回的 `Vec<ChatMessage>`，通过 `on_message.send(HistoryChunk { ... })` 逐条推送
- 返回 `usize` 为总消息数，便于前端校验完整性

#### 3.1.2 load_chat_history

**批量加载模式**，适用于前端初始数据直灌或需要完整数组进行客户端计算的场景。

```rust
pub async fn load_chat_history(
    app_handle: tauri::AppHandle,
    owner_id: String,
    owner_type: String,
    topic_id: String,
    limit: Option<usize>,
    offset: Option<usize>,
) -> Result<Vec<ChatMessage>, String>
```

- 与 `load_chat_history_streamed` 共用同一底层实现 `load_chat_history_internal`
- 区别在于返回方式：直接返回 `Vec<ChatMessage>` 而非通过 Channel 流式推送
- 同样设置 `include_extracted_text = false`

#### 3.1.3 append_single_message

**消息追加**，前端发送用户消息或接收 Assistant 完成消息后调用。

```rust
pub async fn append_single_message(
    app_handle: tauri::AppHandle,
    db_state: tauri::State<'_, DbState>,
    owner_id: String,
    owner_type: String,
    topic_id: String,
    message: ChatMessage,
) -> Result<Vec<ContentBlock>, String>
```

- **返回值**：`Vec<ContentBlock>` 为 `content_parser.rs` 对消息内容的解析结果，前端可直接用于结构化渲染
- 内部委托 `message_service::append_single_message`，后者负责：
  1. 附件物理文件对齐与 `internal_path` 校正
  2. SQLite `messages` 表插入
  3. `render_cache` 预渲染缓存写入
  4. 触发内容解析生成 `ContentBlock` 列表

#### 3.1.4 patch_single_message

**消息修补**，用于流式过程中增量更新 Assistant 消息，或用户编辑历史消息后重新保存。

```rust
pub async fn patch_single_message(
    app_handle: tauri::AppHandle,
    db_state: tauri::State<'_, DbState>,
    owner_id: String,
    owner_type: String,
    topic_id: String,
    message: ChatMessage,
) -> Result<Vec<ContentBlock>, String>
```

- 内部委托 `message_service::patch_single_message(..., false)`
- 最后一个布尔参数 `force_recompile = false` 表示仅在内容发生实质变化时才触发重新解析与缓存更新

#### 3.1.5 delete_messages

**批量删除**，前端多选消息后调用。

```rust
pub async fn delete_messages(
    db_state: tauri::State<'_, DbState>,
    topic_id: String,
    msg_ids: Vec<String>,
) -> Result<(), String>
```

- 内部委托 `message_service::delete_messages`
- 实现为**软删除**：写入 `deleted_at` 时间戳，而非物理删除。便于同步子系统识别删除操作并传播到其他设备
- 同时清理关联的 `render_cache` 条目

#### 3.1.6 truncate_history_after_timestamp

**历史截断**，用于分支对话、上下文清理或用户主动清空某时间点后的记录。

```rust
pub async fn truncate_history_after_timestamp(
    app_handle: tauri::AppHandle,
    db_state: tauri::State<'_, DbState>,
    owner_id: String,
    owner_type: String,
    topic_id: String,
    timestamp: i64,
) -> Result<(), String>
```

- 内部委托 `message_service::truncate_history_after_timestamp`
- 删除条件：`timestamp > 给定值` 且 `deleted_at IS NULL`
- 同时清理被截断消息关联的 `render_cache` 与附件文件

### 5.1 与 message_service.rs 的关系

`chat_manager.rs` 与 `message_service.rs` 构成典型的 **Facade + Service** 双层架构：

| 维度 | chat_manager.rs | message_service.rs |
|------|-----------------|-------------------|
| 层级 | 门面层（Tauri Command 入口） | 业务服务层（领域逻辑） |
| 可见性 | 对前端暴露（Tauri IPC） | 仅内部 Rust 调用 |
| 职责 | 参数接收、类型转换、委托调用 | SQL 组装、附件对齐、缓存管理、内容解析触发 |
| 代码量 | 211 行 | 732 行 |

**双向类型依赖**：
- `chat_manager.rs` → `message_service.rs`：所有 Command 均委托给 `message_service`
- `message_service.rs` → `chat_manager.rs`：反向导入 `Attachment` 与 `ChatMessage` 作为输入/输出类型

**关键委托链**：

| chat_manager Command | message_service 对应函数 |
|---------------------|------------------------|
| `load_chat_history_streamed` | `load_chat_history_internal(..., false, false)` |
| `load_chat_history` | `load_chat_history_internal(..., false, false)` |
| `append_single_message` | `append_single_message(...)` |
| `patch_single_message` | `patch_single_message(..., false)` |
| `delete_messages` | `delete_messages(...)` |
| `truncate_history_after_timestamp` | `truncate_history_after_timestamp(...)` |

### 5.2 与 topic_service.rs 的关系

`chat_manager.rs` 不直接 `use` 或调用 `topic_service.rs`，但两者通过 **`topic_id` 字段**形成隐式关联：

- `topic_service.rs` 负责话题的 CRUD（创建、重命名、切换、删除）
- `chat_manager.rs` 的所有 Command 均以 `topic_id` 作为必填参数，将消息绑定到特定话题
- 当 `topic_service.rs` 删除一个话题时，由上层业务逻辑（而非 `chat_manager.rs`）级联调用 `delete_messages` 或 `truncate_history_after_timestamp` 清理该话题下的全部消息

### 5.3 与 content_parser.rs 的关系

- `chat_manager.rs` 导入 `ContentBlock`（`use crate::vcp_modules::content_parser::ContentBlock`）
- `append_single_message` 与 `patch_single_message` 的返回类型为 `Vec<ContentBlock>`
- 实际的解析工作由 `message_service.rs` 在持久化后触发 `content_parser.rs` 完成，`chat_manager.rs` 仅负责类型透传

### 5.4 与 message_repository.rs 的关系

`message_service.rs` 进一步将 SQL 操作委托给 `message_repository.rs`（Repository 模式）：

- `MessageRepository` 封装原始 SQL 查询与 `render_cache` 读写
- `ContentCompressor` 负责消息内容的 zstd 压缩/解压（`message_service.rs:63` 可见）
- `MessageRenderCompiler` 负责预渲染缓存的生成与读取

`chat_manager.rs` 对 `message_repository.rs` 无直接依赖，关系路径为：
```
chat_manager.rs → message_service.rs → message_repository.rs → SQLite
```

## 6. 设计考量与约束

### 6.1 门面层的薄层设计

`chat_manager.rs` 刻意保持薄层（211 行），所有业务规则下沉到 `message_service.rs`。这带来以下收益：

1. **单一职责**：本模块仅负责"接收 IPC 调用并委托"，易于理解
2. **前端契约稳定**：`ChatMessage` 与 `Attachment` 的结构变更集中管理
3. **测试友好**：业务逻辑在 `message_service.rs` 中，可脱离 Tauri 运行时进行单元测试（虽然当前测试覆盖率有限）

### 6.2 双模式历史加载

| 模式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 流式 (`streamed`) | 大历史记录列表（>50 条） | 渐进渲染，UI 不阻塞；可按 `index`/`is_last` 实现动画 | 实现复杂度高（前端需管理 Channel 监听器） |
| 批量 (`load`) | 初始数据直灌、小记录集 | 简单直接，一次性拿到完整数组 | 大数组 IPC 传输延迟明显，前端白屏时间长 |

当前前端默认使用**流式模式**加载聊天历史，批量模式保留给特定场景（如导出、同步批量推送）。

### 6.3 extracted_text 的默认排除

两个加载函数均硬编码 `include_extracted_text = false`：

```rust
load_chat_history_internal(..., false, false)
// 第四个参数：include_extracted_text
```

这是移动端**内存与 IPC 带宽防御**的关键设计。附件提取文本（如 PDF 全文、OCR 结果）可达数 MB，列表加载时不需要该字段。若前端需要查看附件详情，通过独立接口按需获取。

### 6.4 错误处理

所有 Command 均返回 `Result<T, String>`，符合 Tauri 命令处理规范（AGENTS.md §4.2 要求：严禁使用 `unwrap()` 或 `expect()`）。错误信息从 `message_service.rs` 逐层透传，前端统一显示为 Toast 通知。
