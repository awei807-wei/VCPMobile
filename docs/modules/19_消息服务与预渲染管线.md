# 19_消息服务与预渲染管线

## 1. 概述

### 1.1 模块定位

`message_service.rs`（位于 `src-tauri/src/vcp_modules/chat/message_service.rs`，~732 行）是 VCP Mobile 对话领域中的**消息业务逻辑中心（Message Business Logic Hub）**。它负责消息全生命周期的 CRUD 操作、批量查询、附件本地对齐、渲染缓存（`render_cache`）的读写管理，以及消息外壳属性（`MessageShell`）的触发计算。

`pre_renderer/`（位于 `src-tauri/src/vcp_modules/chat/pre_renderer/`）是**渲染编译管线的前置子系统（Pre-rendering Subsystem）**，负责将原始 Markdown 文本解析为结构化 AST（`MarkdownNode`/`InlineNode`），执行代码语法高亮、HTML 容器提取、LaTeX 公式预处理、VCP Magic 标记识别等。其产出被 `content_parser.rs` 和 `message_repository.rs` 消费，最终序列化为二进制缓存写入 SQLite。

两者的关系：**`message_service.rs` 是调度者，`pre_renderer` 是执行者**。前者决定"何时编译、缓存、清除"，后者决定"如何解析、高亮、结构化"。

### 1.2 文件总览表

| 文件 | 行数 | 职责 |
|------|------|------|
| `message_service.rs` | ~732 | 消息 CRUD、批量加载、附件对齐、渲染缓存读写、外壳预计算触发 |
| `pre_renderer/mod.rs` | ~8 | 子模块聚合与统一导出 |
| `markdown_parser.rs` | ~985 | Markdown → AST 解析主引擎（`pulldown-cmark` Event 驱动） |
| `markdown_ast.rs` | ~421 | AST 节点类型定义与哈希指纹计算 |
| `code_highlighter.rs` | ~39 | 代码块语法高亮（`syntect`） |
| `shell_precomputer.rs` | ~59 | 消息外壳属性预计算（头像色、显示名、用户判定） |

### 2.2 批量加载：`load_multi_topic_messages`

该函数是**前端初始化时的核心入口**，用于一次性拉取多个 topic 的全部消息（如侧边栏切换时的预加载）。

```rust
pub async fn load_multi_topic_messages(
    pool: &sqlx::SqlitePool,
    topic_ids: &[String],
) -> Result<HashMap<String, Vec<ChatMessage>>, String>
```

**查询策略**：
1. **主查询**：`SELECT ... FROM messages LEFT JOIN render_cache ... WHERE topic_id IN (...)`，动态生成 `?` 占位符，避免 N+1 查询。
2. **附件批量回填**：收集所有 `(topic_id, msg_id)` 对，通过 `(?, ?)` IN 子句一次性 JOIN `message_attachments` + `attachments`。
3. **内容解压**：`content` 字段使用 `ContentCompressor::decompress`（zstd）解压为原始文本。
4. **渲染字段解析**：`render_content` 通过 `parse_render_bytes` 反序列化为 `blocks: Option<Value>`。

> 批量加载场景**不填充附件的 `extracted_text`**，也不计算 `shell`，因为这些是按需懒计算属性，在批量 push 场景下可由前端后续单独请求。

### 2.3 历史查询：`load_chat_history_internal`

这是**聊天页面渲染的核心入口**，相比批量加载多了以下能力：

| 能力 | 实现方式 |
|------|---------|
| 分页控制 | `LIMIT ? OFFSET ?` 可选参数 |
| 懒渲染策略 | `render_cache` 命中则直接反序列化；未命中则实时编译并异步回写 |
| 附件文本提取 | `include_extracted_text` 为 true 时，调用 `file_manager::ensure_extracted_text` 进行 OCR/文本提取 |
| 外壳预计算 | 调用 `pre_renderer::precompute_shell`，基于 agents 配置和用户信息生成 `MessageShell` |
| 用户头像色 | 查询 `avatars` 表获取 `dominant_color` |

#### 懒渲染策略详解

```
render_content 存在 ?
  ├─ Yes ──► parse_render_bytes(render_content) → blocks
  │           content = decompress(content_bytes) [若 include_content]
  │
  └─ No  ──► decompress(content_bytes) → raw_text
             MessageRenderCompiler::compile(raw_text) → blocks
             MessageRenderCompiler::serialize(blocks) → binary
             tokio::spawn(异步 INSERT/UPDATE render_cache)
             content = raw_text [若 include_content] 或 ""
```

异步回写使用 `tokio::spawn` 将数据库操作抛入后台任务，**不阻塞消息加载流**。这是移动端保持 60fps 滚动的关键设计。

### 2.4 消息写入与更新

#### `append_single_message`

```
输入: ChatMessage (含 content, 可能含 blocks)
  │
  ├─► ensure_attachments_locally()   // 下载缺失附件
  │
  ├─► blocks 来源判定:
  │     ├─ message.blocks 存在 → 直接反序列化
  │     └─ 不存在 → MessageRenderCompiler::compile(content)
  │
  ├─► MessageRenderCompiler::serialize(blocks) → render_bytes
  │
  ├─► 事务开始
  │     ├─ MessageRepository::upsert_message(tx, message, topic_id, render_bytes, false)
  │     ├─ SELECT COUNT(*) 更新 topics.msg_count
  │     └─ UPDATE topics SET updated_at, msg_count
  │
  └─► 事务提交 → 返回 blocks
```

#### `patch_single_message`

与 `append_single_message` 逻辑几乎一致，区别在于：
- 支持 `skip_bubble` 参数，用于流式消息更新时控制是否触发前端气泡动画
- 不更新 `topics.msg_count`，仅更新 `updated_at`

### 2.5 删除与截断

删除操作执行**三级物理清除**，杜绝幽灵数据：

| 层级 | SQL 操作 | 目的 |
|------|---------|------|
| 消息层 | `UPDATE messages SET deleted_at = ?` | 软删除，保留审计 |
| 缓存层 | `DELETE FROM render_cache WHERE ...` | 物理清除预渲染缓存 |
| 关联层 | `DELETE FROM message_attachments WHERE ...` | 清除消息-附件关联 |

`truncate_history_after_timestamp` 额外清理：
- `DELETE FROM render_cache WHERE msg_id IN (SELECT ... WHERE timestamp > ?)`
- `DELETE FROM message_attachments WHERE msg_id IN (SELECT ... WHERE timestamp > ?)`

### 2.6 附件本地对齐：`ensure_attachments_locally`

该函数解决**多端同步场景下附件的物理存在性**问题。

```
遍历 message.attachments:
  │
  ├─► 计算本地路径: attachments_root_dir / {hash}.{ext}
  │
  ├─► 文件存在 ?
  │     ├─ Yes → 跳过
  │     └─ No  → 构造下载 URL: {sync_http_url}/api/mobile-sync/download-attachment?hash={hash}
  │              携带 x-sync-token 与 Authorization 头部
  │              下载成功后 fs::write(local_path, bytes)
  │
  └─► 路径对齐:
        ├─ src = "file://{local_path}" [若 src 为空]
        └─ internal_path = local_path [始终更新为本地绝对路径]
```

> 下载失败不阻断流程，UI 会显示裂图，用户可后续手动触发重新加载。

`markdown_ast.rs` 不依赖任何外部解析库，仅使用 `serde` 和 `std::hash`，是**纯粹的数据定义层**。所有节点均通过 `#[serde(tag = "type")]` 实现内部标签序列化，前端可直接根据 `type` 字段进行条件渲染。

### 4.4 哈希指纹机制

每个可变的块级/行内节点均携带 `hash: Option<u64>`，通过 `std::collections::hash_map::DefaultHasher` 计算。

```rust
impl MarkdownNode {
    pub fn compute_hash(&self) -> u64 { ... }
    pub fn set_hash(&mut self, h: u64) { ... }
    pub fn compute_hashes_recursively(&mut self) { ... }
}
```

**用途**：
- 前端 `v-for :key` 使用 `hash` 作为稳定键，避免不必要的 DOM 重排
- 流式渲染中识别"已稳定的段落"，仅重新渲染 tail 区域
- 同步子系统的块级差异检测（与 `sync_hash.rs` 协同）

## 5. markdown_parser.rs — Markdown 解析引擎

### 5.1 解析入口与 Panic 隔离

```rust
pub fn parse_markdown_to_ast(text: &str) -> Vec<MarkdownNode> {
    let raw_text = text.to_string();
    let result = std::panic::catch_unwind(move || {
        parse_markdown_to_ast_impl(&raw_text)
    });
    match result {
        Ok(nodes) => nodes,
        Err(e) => {
            log::error!("[PreRender] parse_markdown_to_ast panicked: {:?}", e);
            let mut fallback_node = MarkdownNode::paragraph(vec![InlineNode::text(text.to_string())]);
            fallback_node.compute_hashes_recursively();
            vec![fallback_node]
        }
    }
}
```

**设计决策**：使用 `catch_unwind` 包裹整个解析流程。`pulldown-cmark` 在处理极端畸形输入（如未闭合的深层嵌套标签、特殊 Unicode 组合）时理论上存在 panic 可能。一旦 panic，降级为**单行纯文本段落**，确保前端至少能看到原始内容而非崩溃。

### 5.2 LaTeX 预处理：`preprocess_latex_math`

在调用 `pulldown-cmark` 之前，先对原始文本进行**围栏感知的 LaTeX 公式替换**：

| 输入模式 | 输出模式 | 场景 |
|---------|---------|------|
| `\[ ... \]` | `$$ ... $$` | 独立显示公式 |
| `\( ... \)` | `$ ... $` | 行内公式 |

**围栏感知**：通过 `FENCE_RE`（`^[ \t]*\`\`\`[a-zA-Z0-9-]*[ \t]*$`）追踪代码块边界，仅在围栏外执行替换，防止代码示例中的 LaTeX 片段被误解析。

```
原始文本
  │
  ├─► 扫描代码围栏位置 → 建立 fence 范围列表
  │
  ├─► 围栏外: MATH_RE 捕获 → 替换为 $/$$
  │
  └─► 围栏内: 原样保留
```

### 5.3 HTML 容器提取：`extract_html_containers`

这是 VCP Mobile 支持**混合 HTML + Markdown** 内容的核心机制。

```
原始文本
  │
  ├─► 扫描 <div>, <section>, <article>, <header>, <footer>,
  │     <main>, <aside>, <figure>, <figcaption> 等容器标签
  │
  ├─► 找到开标签 ?
  │     ├─ Yes → find_matching_close_tag()（支持嵌套深度追踪，跳过围栏内标签）
  │     │         内部文本经 trim_common_leading_indent() 去缩进
  │     │         递归调用 parse_markdown_to_ast() 解析内部 Markdown
  │     │         替换为占位符 <!--VCP_HTML_CONTAINER:{idx}-->
  │     └─ No  → 保留原样
  │
  └─► 解析完成后 → replace_container_placeholders()
                    将占位符替换为 [raw_html(open_tag), ...inner_nodes, raw_html(close_tag)]
```

**`trim_common_leading_indent`** 是解析容器内部 Markdown 的关键辅助函数。由于 HTML 容器在原始文本中通常有统一的前导缩进（如嵌套在 `<div>\n    ...\n    ...\n  </div>`），直接传入 `pulldown-cmark` 会将缩进内容误判为**Indented Code Block**。该函数计算所有非空行的最小公共前导缩进（空格计 1，制表符计 4），并将其去除，保留相对结构。

### 5.4 核心解析循环：`parse_markdown_to_ast_impl`

基于 `pulldown-cmark::Parser` 的 Event 驱动状态机：

```rust
let parser = Parser::new_ext(
    &text,
    Options::ENABLE_MATH | Options::ENABLE_TABLES | Options::ENABLE_STRIKETHROUGH,
);

let mut stack: Vec<PartialNode> = Vec::new();

for event in parser {
    match event {
        Event::Start(tag) => stack.push(PartialNode::from_tag(tag)),
        Event::Text(text) => {
            let inlines = if 在 CodeBlock 内 { vec![Text] } else { process_text_magic(&text) };
            stack.last_mut().push_inlines(inlines);
        }
        Event::Code(code) => stack.last_mut().push_inline(InlineNode::code(...)),
        Event::InlineMath(math) => stack.last_mut().push_inline(InlineNode::inline_math(...)),
        Event::DisplayMath(math) => ...,
        Event::End(tag_end) => {
            let node = stack.pop().finalize(tag_end);
            if let Some(parent) = stack.last_mut() { parent.push_child(node); }
            else { nodes.push(node); }
        }
        Event::Html(html) => nodes.push(MarkdownNode::raw_html(...)),
        // ... 其他事件
    }
}
```

**关键点**：
- `Options::ENABLE_MATH` 配合预处理后的 `$...$`/`$$...$$` 实现 LaTeX 支持
- `Options::ENABLE_TABLES` 支持 GFM 表格
- `Options::ENABLE_STRIKETHROUGH` 支持 `~~删除线~~`

### 5.5 PartialNode 状态机

`PartialNode` 是解析过程中的**中间表示（Intermediate Representation）**，与 `MarkdownNode` 一一对应，但携带更多构造期状态。

| PartialNode | 累积方式 | finalize 产出 |
|-------------|---------|--------------|
| `Paragraph` | `push_inline` / `push_inlines` | `MarkdownNode::paragraph(children)` |
| `Heading` | 同上 | `MarkdownNode::heading(level, children)` |
| `CodeBlock` | `Text` Event 直接追加到 `code` 字符串 | `MarkdownNode::code_block(lang, code)` + 调用 `highlight_code_block` |
| `Blockquote` | `push_child` 接收 `MarkdownNode` | `MarkdownNode::blockquote(children)` |
| `List` / `Item` | `push_list_item` | `MarkdownNode::list(ordered, items)` |
| `Table` / `TableHead` / `TableRow` / `TableCell` | `push_table_cell` / `push_table_row` / `set_table_header` | `MarkdownNode::table(header, rows)`，自动设置 `wrapper_class` |
| `Link` / `Image` | `push_inline`（Link）/ `Text` 追加到 `alt`（Image） | `InlineNode::link(...)` / `InlineNode::image(...)`，再包装为 `MarkdownNode::paragraph` |
| `Strong` / `Emphasis` / `Strikethrough` | `push_inline` | 对应 `InlineNode`，再包装为 `MarkdownNode::paragraph` |

### 5.6 VCP Magic 标记处理：`process_vcp_magic`

在标准 Markdown 解析完成后，对纯文本节点执行二次扫描，识别 VCP 自定义标记：

| 正则捕获组 | 模式 | 产出 InlineNode |
|-----------|------|----------------|
| 组 1 | `["""](?:[^"""]|\\.)+?["""]`（引号包裹文本） | `QuotedText` |
| 组 2 | `@![^\s@!]+` | `AlertTag` |
| 组 3 | `@[^\s@]+` | `HighlightTag` |

**引号内容的二次解析**：`QuotedText` 内部的文本会调用 `parse_inline_standard` 进行轻量级标准 Markdown 行内解析（支持 strong/emphasis/strikethrough/code/link/image/math），但**不再次解析 VCP Magic**，避免无限递归。

### 5.7 轻量级内联解析器：`parse_inline_standard`

该函数专门用于解析已知不含块级语法的纯文本片段（如 `QuotedText` 内部、HTML 容器内部）。它将文本包装为 `"{text}\n"`，强制 `pulldown-cmark` 视为单一段落，仅提取行内事件，丢弃段落包装层。

## 6. code_highlighter.rs — 语法高亮

### 6.1 实现概要

```rust
lazy_static! {
    static ref SYNTAX_SET: SyntaxSet = SyntaxSet::load_defaults_newlines();
    static ref THEME_SET: ThemeSet = ThemeSet::load_defaults();
}

pub fn highlight_code_block(code: &str, lang: &str) -> Option<String> {
    // 语言查找优先级: token → token(原始大小写) → extension
    // 回退链: JavaScript → Plain Text
    let syntax = SYNTAX_SET.find_syntax_by_token(&lang_lower)
        .or_else(|| SYNTAX_SET.find_syntax_by_token(lang))
        .or_else(|| SYNTAX_SET.find_syntax_by_extension(lang))
        .unwrap_or_else(|| /* JavaScript fallback */);

    let theme = &THEME_SET.themes["base16-ocean.dark"];
    let html = highlighted_html_for_string(code, &SYNTAX_SET, syntax, theme).ok()?;

    // 统一添加 CSS 类
    if html.starts_with("<pre") {
        html.replacen("<pre", "<pre class=\"vcp-code-block vcp-scrollable\"", 1)
    } else {
        format!("<pre class=\"vcp-code-block vcp-scrollable\">{}</pre>", html)
    }
}
```

### 6.2 设计决策

| 决策 | 说明 |
|------|------|
| **静态加载** | `lazy_static!` 在首次调用时加载 syntect 默认语法集和主题集，后续调用零分配 |
| **固定主题** | 使用 `base16-ocean.dark`，与 VCP Mobile 暗色主题统一；不支持主题动态切换（减少移动端包体积） |
| **三层回退** | 指定语言 → JavaScript（类 C 语法通用高亮）→ Plain Text，确保任何语言标识都不会导致解析失败 |
| **CSS 类注入** | 强制添加 `vcp-code-block`（样式）和 `vcp-scrollable`（横向滚动）类名，确保移动端长代码块可横向滑动 |

### 7.2 预计算逻辑

precompute_shell(message, agents, user_name, user_avatar_color)

## 8. 渲染编译全流程

### 8.1 从原始内容到预渲染结果

```
原始 Markdown 文本
  │
  ▼
┌─────────────────────────┐
│ MessageRenderCompiler   │  （定义于 message_repository.rs）
│ ::compile(content)      │
└───────────┬─────────────┘
            │ Vec<ContentBlock>
            ▼
┌─────────────────────────┐
│ content_parser.rs       │  ← 特种块识别（Tool/Thought/Diary/...）
│ 其中 Markdown 块的分支   │     对纯 Markdown 文本调用:
└───────────┬─────────────┘
            │ &str (markdown text)
            ▼
┌─────────────────────────┐
│ preprocess_latex_math   │  ← \[...\] → $$...$$
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ extract_html_containers │  ← HTML 容器提取 + 内部递归解析
└───────────┬─────────────┘
            │ text_with_placeholders
            ▼
┌─────────────────────────┐
│ pulldown-cmark Parser   │  ← Event 驱动解析
│   + PartialNode 状态机   │
└───────────┬─────────────┘
            │ Vec<MarkdownNode>
            ▼
┌─────────────────────────┐
│ compute_hashes_recursively│ ← 全量哈希指纹计算
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ replace_container_      │  ← 占位符 → 实际 HTML 包装节点
│   placeholders          │
└───────────┬─────────────┘
            │ Vec<MarkdownNode> (最终 AST)
            ▼
┌─────────────────────────┐
│ content_parser.rs 组装   │  ← 包装为 ContentBlock::Markdown { nodes }
└───────────┬─────────────┘
            │ Vec<ContentBlock>
            ▼
┌─────────────────────────┐
│ MessageRenderCompiler   │
│ ::serialize(blocks)     │  ← serde_json::to_vec → zstd::bulk::compress
└───────────┬─────────────┘
            │ Vec<u8> (zstd 压缩后的 JSON)
            ▼
        render_cache 表
```

### 8.2 缓存命中路径（热路径）

```
load_chat_history_internal()
  │
  ├─► LEFT JOIN render_cache ON (topic_id, msg_id)
  │
  ├─► render_content IS NOT NULL ?
  │     ├─ Yes ──► MessageRenderCompiler::deserialize(bytes)
  │     │           → Vec<ContentBlock>
  │     │           → serde_json::to_value → blocks: Option<Value>
  │     │           [零编译开销，直接反序列化]
  │     │
  │     └─ No  ──► ContentCompressor::decompress(content_bytes)
  │                → MessageRenderCompiler::compile(raw_text)
  │                → MessageRenderCompiler::serialize(blocks)
  │                → tokio::spawn(异步写入 render_cache)
  │                [首次加载或缓存 miss 时的冷路径]
```

### 8.3 缓存未命中路径（冷路径）

当消息是新写入的、或 `render_cache` 被物理清除（删除/截断操作）时，触发实时编译：

1. **解压**：`ContentCompressor::decompress`（zstd，通常 3-10 倍压缩比）
2. **编译**：`MessageRenderCompiler::compile` → `parse_content` → `parse_markdown_to_ast`
3. **序列化**：`MessageRenderCompiler::serialize`（JSON + zstd，压缩级别 3）
4. **异步回写**：`tokio::spawn` + `INSERT ... ON CONFLICT(topic_id, msg_id) DO UPDATE`

### 8.4 缓存清除策略

`render_cache` 在以下场景被**物理删除**：

| 场景 | 触发函数 | SQL |
|------|---------|-----|
| 消息软删除 | `delete_messages` | `DELETE FROM render_cache WHERE topic_id = ? AND msg_id IN (...)` |
| 历史截断 | `truncate_history_after_timestamp` | `DELETE FROM render_cache WHERE topic_id = ? AND msg_id IN (SELECT ... WHERE timestamp > ?)` |
| 强制重渲染 | `re_render_message` | `INSERT ... ON CONFLICT DO UPDATE`（覆盖旧缓存） |

## 9. 性能与内存考量

### 9.1 懒渲染策略的收益

| 指标 | 无缓存（每次实时编译） | 有缓存（首次后直取） |
|------|----------------------|---------------------|
| 单次消息加载 | ~5-20ms（取决于 Markdown 复杂度） | ~0.1-0.5ms（zstd 解压 + JSON 反序列化） |
| 100 条历史记录 | ~500-2000ms | ~10-50ms |
| 内存占用 | 无额外持久化 | 每条约 0.5-5KB 压缩缓存 |

### 9.2 AST 构建的性能特征

| 阶段 | 复杂度 | 优化手段 |
|------|--------|---------|
| LaTeX 预处理 | O(n)，单次扫描 | 围栏感知，避免无效替换 |
| HTML 容器提取 | O(n + m×k)，n=文本长，m=容器数，k=平均嵌套深度 | 预收集围栏范围，跳过围栏内标签扫描 |
| pulldown-cmark 解析 | O(n) | 标准库优化，增量迭代器 |
| VCP Magic 扫描 | O(n) | `regex::Regex` 单次 `captures_iter` |
| 哈希计算 | O(n) | `DefaultHasher`，递归但每个节点只访问一次 |
| syntect 高亮 | O(code_len) × 语法复杂度 | `lazy_static!` 缓存 SyntaxSet/ThemeSet |

### 9.3 移动端特殊优化

1. **panic 隔离**：`parse_markdown_to_ast` 使用 `catch_unwind`，防止极端输入导致后端崩溃
2. **异步回写**：缓存 miss 时的数据库写入不阻塞消息列表返回
3. **压缩存储**：`render_cache` 使用 zstd 压缩，典型压缩比 3-10 倍，降低 SQLite 存储压力和读取 I/O
4. **预计算外壳**：`MessageShell` 在 Rust 侧预计算，避免前端在 `v-for` 中反复查找 agents 列表
5. **围栏感知的 HTML 匹配**：`find_matching_close_tag` 跳过代码块内的 HTML 标签，防止 Markdown 代码示例被误判为 HTML 容器
