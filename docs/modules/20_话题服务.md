# 20. 话题服务 (Topic Service)

## 1. 概述

### 1.1 模块定位

话题服务由三个文件协同构成，负责 VCP Mobile 中**会话话题（Topic）的全生命周期管理**：

| 文件 | 行数 | 职责 |
|------|------|------|
| `topic_types.rs` | 21 | 定义话题领域模型 `Topic` 结构体 |
| `topic_service.rs` | 461 | 话题的 CRUD、状态管理、重新生成、同步哈希冒泡 |
| `topic_summary_service.rs` | 137 | 基于 AI 的自动化话题标题生成与清洗 |

话题是 VCP Mobile 中组织对话历史的核心容器。每个话题隶属于一个 `owner`（Agent 或 Group），包含独立的消息时间线、锁定状态与未读计数。话题服务**完全面向 SQLite 数据库**，不依赖本地文件系统。

### 1.2 职责边界

- **创建与删除**：支持软删除（`deleted_at` 标记），保留数据完整性以支持未来恢复或同步冲突解决
- **查询与流式传输**：支持一次性拉取与分块流式传输两种模式，适配不同规模的话题列表
- **状态管理**：锁定（`locked`）、未读（`unread` / `unread_count`）的独立控制
- **AI 总结**：调用外部 LLM 根据最近对话生成简洁标题
- **重新生成**：基于某条用户消息截断后续历史，重新发起 Agent/Group 对话流
- **同步联动**：关键变更触发聚合哈希冒泡与同步通知

## 2. 核心数据结构

### 2.1 Topic

`Topic` 是话题领域的唯一序列化载体，实现了 `Serialize` / `Deserialize` / `Clone` / `Default`，可直接通过 Tauri IPC 往返于前后端。

> 定义位置：`src-tauri/src/vcp_modules/chat/topic_types.rs` 第 3–21 行

| 字段 | 类型 | Serde 映射 | 说明 |
|------|------|-----------|------|
| `id` | `String` | — | 话题唯一标识，格式为 `topic_{timestamp}` 或 `group_topic_{timestamp}` |
| `name` | `String` | — | 话题显示标题，用户可手动修改或由 AI 自动生成 |
| `created_at` | `i64` | `createdAt` | 创建时间，毫秒级 Unix 时间戳 |
| `locked` | `bool` | — | 锁定状态。锁定后禁止继续发送消息，用于归档 |
| `unread` | `bool` | — | 是否存在未读消息（布尔标记，兼容旧逻辑） |
| `unread_count` | `i32` | `unreadCount` | 未读消息精确计数 |
| `msg_count` | `i32` | `msgCount` | 该话题下消息总数（数据库维护，减少前端计数查询） |
| `owner_id` | `String` | `ownerId` | 所属 Agent 或 Group 的 ID |
| `owner_type` | `String` | `ownerType` | `"agent"` 或 `"group"` |

**设计要点**：
- `locked` 默认为 `true`（新建话题即锁定，首次发送消息时由前端或业务逻辑解锁）
- `unread` 与 `unread_count` 并存：`unread` 提供快速布尔判断，`unread_count` 提供精确数字展示
- 所有时间戳均为毫秒级 `i64`，与前端 JavaScript `Date.now()` 精度一致

## 3. 话题查询体系

### 3.1 一次性查询（`get_topics`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 55–90 行

根据 `owner_id` + `owner_type` 查询全部未删除话题，按 `created_at DESC` 排序。

```sql
SELECT topic_id, title, created_at, locked, unread, unread_count, msg_count
FROM topics
WHERE owner_id = ? AND owner_type = ? AND deleted_at IS NULL
ORDER BY created_at DESC
```

- **返回**：`Vec<Topic>`，完整话题列表
- **适用场景**：话题数量可控（< 100）的常规查询

### 3.2 流式查询（`get_topics_streamed`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 92–140 行

通过 `tauri::ipc::Channel<Vec<Topic>>` 分块推送结果，每块 15 条。

```
前端调用 get_topics_streamed
         │
         ▼
┌─────────────────┐
│  开启数据库游标  │──> sqlx::query(...).fetch(pool) 流式读取
└────────┬────────┘
         │
         ▼
┌─────────────────┐  达到 chunk_size=15?
│  累积到 chunk   │──> Yes ──► Channel.send(chunk)
└────────┬────────┘            chunk.clear()
         │
         ▼ (流结束)
┌─────────────────┐  剩余记录?
│  发送尾块       │──> Yes ──► Channel.send(剩余)
└─────────────────┘
         │
         ▼
    Ok(())
```

- **适用场景**：话题数量庞大的 Group 或长期使用的 Agent，避免一次性加载导致内存峰值与 UI 卡顿
- **前端配合**：前端监听 Channel，每收到一块立即渲染，实现渐进式列表加载

### 3.3 批量未读计数（`get_unread_counts`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 16–53 行

**核心优化目标**：替代前端对多个 owner 的 N+1 查询，一次聚合返回所有 owner 的未读状态。

```sql
SELECT owner_id,
       CAST(COALESCE(SUM(unread_count), 0) AS INTEGER) as total_count,
       MAX(CASE WHEN unread = 1 THEN 1 ELSE 0 END) as has_unread
FROM topics
WHERE deleted_at IS NULL
GROUP BY owner_id
```

**返回值语义**（`HashMap<String, i32>`）：

| 条件 | 返回值 | 含义 |
|------|--------|------|
| `total_count > 0` | `total_count` | 精确未读消息数 |
| `total_count == 0` 且 `has_unread == 1` | `-1` | 存在未读但计数缺失（兼容旧数据） |
| 否则 | 不插入 Map | 该 owner 无未读 |

## 4. 话题生命周期管理

### 4.1 创建话题（`create_topic`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 142–195 行

```
输入: owner_id, owner_type, name
         │
         ▼
┌─────────────────┐
│ 生成时间戳 now  │
└────────┬────────┘
         ▼
┌─────────────────┐  owner_type == "group"?
│ 构造 topic_id   │──> Yes ──► "group_topic_{now}"
└────────┬────────┘  No  ───► "topic_{now}"
         ▼
┌─────────────────┐
│  构造 Topic 对象 │──> locked=true, unread=false, unread_count=0, msg_count=0
└────────┬────────┘
         ▼
┌─────────────────┐
│ INSERT topics   │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 聚合哈希冒泡    │──> HashAggregator::bubble_from_topic(&mut tx, &id)
└────────┬────────┘
         ▼
    Ok(topic)
```

- **默认锁定**：新话题创建时 `locked = true`，由首次发送消息的业务逻辑解锁
- **哈希冒泡**：通过 `HashAggregator` 初始化话题自身的 `config_hash`，并向上更新 Agent/Group 的 `ContentHash`，确保同步系统感知新话题

### 4.2 软删除话题（`delete_topic`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 197–230 行

采用**软删除**策略，将 `deleted_at` 设为当前时间戳而非物理删除记录：

```sql
UPDATE topics SET deleted_at = ? WHERE topic_id = ?
```

**后续联动**：
1. **同步通知**：向 `sync_service` 发送 `SyncCommand::NotifyDelete`，通知远端该话题已删除
2. **哈希冒泡**：根据 `owner_type` 调用 `bubble_agent_hash` 或 `bubble_group_hash`，更新父级聚合哈希

> 软删除为同步冲突解决、数据恢复、审计追踪保留可能性。所有查询均通过 `deleted_at IS NULL` 过滤。

### 4.3 更新标题（`update_topic_title`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 232–255 行

直接 `UPDATE` 标题与 `updated_at` 时间戳。纯本地操作，不涉及同步或哈希冒泡（标题变更不触发配置哈希变化，属于展示层属性）。

### 4.4 切换锁定（`toggle_topic_lock`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 277–322 行

```
输入: topic_id, locked
         │
         ▼
┌─────────────────┐
│ UPDATE topics   │──> SET locked=?, updated_at=now
└────────┬────────┘
         ▼
┌─────────────────┐
│ 聚合哈希冒泡    │──> HashAggregator::bubble_from_topic
└────────┬────────┘
         ▼
┌─────────────────┐  SyncState 存在?
│ 发送同步通知    │──> Yes ──► NotifyLocalChange { Topic, topic_id, config_hash, ts }
└─────────────────┘
```

锁定状态属于**配置属性**，变更后需要：
1. 更新话题的 `config_hash` 并向上冒泡
2. 通过同步通道通知其他设备

### 4.5 设置未读（`set_topic_unread`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 324–348 行

更新 `unread` 布尔标记与 `updated_at`。不触发同步或哈希冒泡（未读状态是**本地会话状态**，不应跨设备同步）。

## 5. 话题重新生成（`regenerate_topic_response`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_service.rs` 第 350–461 行

这是话题服务中最复杂的命令，支持在已有话题中**基于某条用户消息重新生成 AI 回复**，相当于"从该消息开始重新回答"。

### 5.1 整体流程

```
输入: owner_id, owner_type, topic_id, target_user_msg_id, stream_channel
         │
         ▼
┌─────────────────────────────────────┐
│ 1. 获取目标用户消息完整内容          │──> message_service::fetch_raw_message_content
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│ 2. 获取消息元数据（timestamp, role） │──> 查询 messages 表
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│ 3. 截断该消息之后的所有历史          │──> message_service::truncate_history_after_timestamp
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│ 4. 构造 ChatMessage 逻辑对象         │──> chat_manager::ChatMessage { ... }
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│ 5. 读取 Settings 获取 VCP 配置       │──> settings_manager::read_settings
└─────────────┬───────────────────────┘
              ▼
        ┌─────┴─────┐
        ▼           ▼
   owner_type    owner_type
   == "agent"   == "group"
        │           │
        ▼           ▼
┌───────────────┐ ┌───────────────────┐
│ agent_chat_   │ │ group_chat_       │
│ application_  │ │ application_      │
│ service::     │ │ service::         │
│ internal_     │ │ internal_process_ │
│ process_      │ │ group_chat_message│
│ agent_chat_   │ │                   │
│ message       │ │                   │
└───────┬───────┘ └─────────┬─────────┘
        │                   │
        └─────────┬─────────┘
                  ▼
          stream_channel (SSE 流)
```

### 5.2 关键步骤详解

**步骤 3：历史截断**

```rust
truncate_history_after_timestamp(
    app_handle, pool, &owner_id, &owner_type, &topic_id, timestamp
)
```

删除数据库中该 `timestamp` 之后的全部消息（包括原来的 AI 回复），确保重新生成时不会看到旧回复。

**步骤 4：ChatMessage 构造**

```rust
chat_manager::ChatMessage {
    id: target_user_msg_id,       // 复用原消息 ID
    role: "user",
    content: user_msg,            // 从步骤 1 获取的原始内容
    timestamp: timestamp as u64,
    is_thinking: Some(false),
    agent_id, group_id,
    topic_id: Some(topic_id),
    is_group_message: Some(...),
    finish_reason: None,
    attachments: None,            // 上下文组装时会从 DB 重新拉取
    blocks: None,
    shell: None,
    content_hash: None,
}
```

**步骤 5：skip append**

两个内部处理函数均传入 `skip_append_user_msg = false`，表示**不重复插入用户消息**（因为该消息已在数据库中），直接发起 AI 生成流程。

## 6. 话题摘要生成（`topic_summary_service.rs`）

### 6.1 整体定位

`topic_summary_service.rs` 是话题服务的** AI 辅助子模块**，职责单一：根据最近对话内容生成简洁的话题标题，帮助用户快速识别话题内容。

### 6.2 核心常量

| 常量 | 值 | 说明 |
|------|-----|------|
| `DEFAULT_SUMMARY_PROMPT` | 见源码 | 要求 AI 仅返回简洁标题，≤10 汉字，无标点、无编号、无解释 |
| `DEFAULT_SUMMARY_MODEL` | `"gemini-3.1-flash-lite"` | 默认总结模型，轻量快速 |
| `AI_REQUEST_TIMEOUT_SECS` | `30` | HTTP 请求超时 |
| `AI_MAX_TOKENS` | `64` | 标题很短，64 token 绰绰有余 |

> **注意**：请求体**不主动发送 `temperature`** 参数，以兼容 o1 / Gemini thinking 等不支持该参数的模型。

### 6.3 摘要流程（`summarize_topic`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_summary_service.rs` 第 23–114 行

### 6.4 标题清洗（`clean_summarized_title`）

> 函数位置：`src-tauri/src/vcp_modules/chat/topic_summary_service.rs` 第 116–137 行

```rust
pub fn clean_summarized_title(raw: &str) -> String {
    // 1. 仅取第一行
    let first_line = raw.lines().next().unwrap_or("").trim();
    
    // 2. 移除所有非字母数字空白字符
    // 3. 移除常见噪音词："标题"、"总结"、"Topic"、":"、"："
    // 4. 移除所有空白字符
    // 5. 截断至 12 个字符
```

**清洗规则**：
- 仅保留第一行（防止 AI 返回多行解释）

- 最终截断至 **12 个字符**（比 Prompt 要求的 10 汉字略宽松，留有余量）

## 7. 与周边模块的协作关系

### 7.1 与 `message_service.rs` 的协作

| 调用点 | 被调用函数 | 用途 |
|--------|-----------|------|
| `regenerate_topic_response` | `fetch_raw_message_content` | 获取目标用户消息的完整原始内容（含附件引用等） |
| `regenerate_topic_response` | `truncate_history_after_timestamp` | 删除该消息时间戳之后的全部历史，实现"从该消息重新生成" |
| `summarize_topic` (summary service) | `load_chat_history_internal` | 加载最近 4 条消息用于标题生成 |

`message_service.rs` 是话题服务在**历史操作**上的主要依赖方。话题服务本身不直接操作 `messages` 表，所有消息级读写均委托给消息服务。

### 7.2 与 `chat_manager.rs` 的协作

`chat_manager.rs` 定义了 `ChatMessage` 结构体，是 VCP Mobile 中**消息的统一领域模型**。在 `regenerate_topic_response` 中，话题服务从数据库加载原始消息后，需要将其包装为 `ChatMessage` 对象，再传递给 `agent_chat_application_service` 或 `group_chat_application_service` 的内部处理函数。

```
topic_service.rs ──构造──► ChatMessage ──传入──► agent_chat_application_service
                                    ──传入──► group_chat_application_service
```

这种设计保证了：
- 话题服务不需要了解 SSE 流、上下文组装、Agent/Group 调度等复杂逻辑
- 重新生成流程与正常聊天流程在底层生成阶段**完全复用**同一套代码

### 7.3 与 `sync_service.rs` 的协作

| 触发场景 | 同步命令 | 说明 |
|----------|----------|------|
| `delete_topic` | `NotifyDelete { Topic, id }` | 话题被删除，通知远端同步删除 |
| `toggle_topic_lock` | `NotifyLocalChange { Topic, id, hash, ts }` | 锁定状态变更，通知远端增量同步 |

话题创建（`create_topic`）当前**不主动发送同步通知**，因为创建时话题为空（无消息），其哈希冒泡会在后续首次消息发送时自然触发同步。

### 7.4 与 `HashAggregator` 的协作

| 触发场景 | 调用 | 说明 |
|----------|------|------|
| `create_topic` | `bubble_from_topic(&mut tx, &id)` | 初始化话题哈希并向上更新父级 |
| `delete_topic` | `bubble_agent_hash` / `bubble_group_hash` | 父级聚合哈希重新计算（减去被删话题） |
| `toggle_topic_lock` | `bubble_from_topic(&mut tx, &id)` | 锁定状态属于配置，变更后重新计算话题哈希并冒泡 |

## 10. 设计决策与注意事项

### 10.1 为何采用软删除？

话题的软删除设计服务于三个目标：
1. **同步安全**：同步冲突解决时，如果一方删除了话题而另一方继续编辑，软删除标记比物理删除更容易实现合并策略
2. **数据恢复**：用户误删后可由管理员或未来功能恢复
3. **引用完整性**：其他表（如同步日志、审计记录）可能引用 `topic_id`，物理删除会导致外键悬空

### 10.2 流式传输 vs 一次性查询的选型

- `get_topics` 适用于绝大多数场景（Agent 话题通常 < 50）
- `get_topics_streamed` 专为**高频使用的 Group** 设计。Group 可能积累数百个话题，一次性 `Vec<Topic>` 会导致：
  - Rust 侧内存峰值
  - JSON 序列化/反序列化延迟
  - 前端列表渲染阻塞

Channel 分块传输将大列表拆分为 15 条的小批次，前端可渐进渲染并展示骨架屏。

### 10.3 `regenerate_topic_response` 的幂等性考量

重新生成命令**不是幂等的**：每次调用都会：
1. 物理删除 `timestamp` 之后的消息
2. 向 AI 发起新的生成请求
3. 将新的回复写入数据库

这意味着连续多次调用会在同一话题中产生**多条截断-重新生成链**。前端应通过 UI 状态（如禁用按钮、加载动画）防止用户重复触发。

### 10.4 话题摘要的模型兼容性

`topic_summary_service.rs` 刻意**省略 `temperature` 参数**，因为：
- o1 系列模型不支持 `temperature`
- Gemini thinking 模型可能因该参数报错
- 标题生成是确定性任务（不需要创意发散），模型默认行为已足够

如果用户配置的 `topic_summary_model` 为空字符串，则回退到 `DEFAULT_SUMMARY_MODEL`（`gemini-3.1-flash-lite`）。

### 10.5 错误处理策略

所有 Tauri Command 统一返回 `Result<T, String>`：
- 数据库错误通过 `map_err(|e| e.to_string())` 转换为描述性字符串
- AI 请求失败返回 HTTP 状态码或网络错误描述
- `regenerate_topic_response` 中任何步骤失败（获取消息、截断历史、构造对象）都会提前返回错误，不会发起无意义的 AI 请求

## 11. 总结

话题服务以约 **619 行**（`topic_service.rs` 461 + `topic_summary_service.rs` 137 + `topic_types.rs` 21）的精炼代码，承担了 VCP Mobile 对话体系中话题容器的全部治理职责。其设计亮点可归纳为：

1. **双层查询策略**：一次性查询 + Channel 流式传输，兼顾常规场景与大规模列表性能
2. **软删除与数据完整性**：`deleted_at` 标记保留恢复与同步冲突解决空间
3. **重新生成复用**：通过 `ChatMessage` 包装与内部处理函数委托，话题服务无需侵入聊天核心逻辑
4. **AI 摘要轻量化**：64 token 上限、省略 temperature、严格清洗，确保快速稳定的标题生成
5. **同步感知**：关键状态变更（删除、锁定）自动触发哈希冒泡与同步通知，未读状态则保持纯本地
