## 1. 生命周期管理器（Lifecycle Manager）

### 1.1 模块定位

`lifecycle_manager.rs` 是 VCP Mobile 核心层的**应用启动编排中心**。它负责在 Tauri `setup` 阶段后，以线性化、优先级分明的顺序初始化所有后台服务，并在初始化完成后向前端广播核心状态变更。该模块是连接 Tauri 应用生命周期与 Rust 业务服务生命周期的唯一枢纽。

### 1.2 核心状态机

```rust
#[derive(Debug, Serialize, Clone, Copy, PartialEq)]
#[serde(rename_all = "lowercase")]
pub enum CoreStatus {
    Initializing,
    Ready,
    Error,
}
```

| 状态 | 语义 | 触发条件 |
|------|------|---------|
| `initializing` | 启动序列执行中 | `bootstrap()` 开始后立即设置 |
| `ready` | 所有 P0/P1 服务初始化完成 | `bootstrap()` 成功结束时设置 |
| `error` | 关键服务初始化失败 | 数据库或配置读取失败时设置 |

`LifecycleState` 同时维护 `last_error: Arc<RwLock<Option<String>>>`，供前端在 `Error` 状态下获取具体失败原因。

### 1.3 启动序列（Bootstrap Sequence）

`bootstrap(app: &AppHandle)` 是唯一的启动入口，执行严格的**同步阻塞 + 后台异步**两阶段初始化：

```
┌─────────────────────────────────────────────────────────────┐
│                     bootstrap()                              │
└─────────────────────────────────────────────────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
    ┌───────────┐      ┌───────────┐      ┌───────────┐
    │  P0 同步   │      │  P1 同步   │      │  P2 后台   │
    │  阻塞阶段  │      │  阻塞阶段  │      │  异步阶段  │
    └─────┬─────┘      └─────┬─────┘      └─────┬─────┘
          │                  │                  │
    1. 数据库初始化      3. 配置预加载      5. 表情库加载
    2. 状态注册(lib.rs)  4. 同步服务初始化   6. 模型管理器初始化
                                              7. VCP Log 自动连接
                                              8. DeleteExecutor 定时清理
                                              9. 前端热更新检查
```

**P0 — 绝对基础（阻塞，失败即致命）**：
1. **数据库初始化** (`init_db`)：创建连接池、建表/迁移。失败时立即将状态置为 `Error`，发射致命错误事件并返回 `Err`。
2. **基础状态注册**：`lib.rs` 的 `setup` 阶段已提前完成 `LifecycleState` 等单例的 `app.manage()`，此处不重复注册以避免覆盖已有缓存。

**P1 — 前端强依赖（阻塞，失败即致命）**：
3. **配置预加载** (`read_settings`)：将 `settings_manager` 的读取前置，确保前端 `fetchSettings` 在 `Ready` 信号发出后必然成功。
3.5. **浮动助手本地服务器裁决** (`reconcile_local_server`)：根据 `settings.enable_assistant` 决定启动或停止本地 axum HTTP+WebSocket 服务器（端口 14202），供 Android 悬浮球 WebView 使用。
3.6. **分布式节点裁决** (`reconcile_distributed_node`)：根据 `settings.distributed_enabled` 和持久化的 WS URL/Key，自动建立或断开分布式节点连接。首次启动时加载 `distributed_tools.json` 配置文件。
4. **同步服务初始化** (`init_sync_service`)：创建 `SyncState` 并注册到 Tauri managed state。

**P2 — 非阻塞后台任务（spawn 后台协程，失败不影响启动）**：
5. **表情库加载与刷新** (`internal_load_library` + `refresh_emoticon_library_internal`)：先从 DB 加载本地缓存，再 best-effort 从服务器刷新。
6. **模型管理器初始化** (`init_model_manager`)：加载模型列表缓存。
7. **VCP Log 自动连接**：若设置中 `vcp_log_url` 和 `vcp_log_key` 均非空，自动建立 WebSocket 日志通道。
8. **DeleteExecutor 定时清理**：启动延时 10 秒后执行首航清理——完美避开冷启动黄金 IO 密集期，随后每 86400 秒（24 小时）循环一次，清理 30 天前的已删除记录。
9. **前端热更新检查**：延时 5 秒后后台静默检查前端资源更新。若有新版本则自动下载 ZIP、解压并应用，下次冷启动生效。整个流程 best-effort——任何步骤失败均记录日志但不影响核心功能。
10. **网络监控激活**：在 `CoreStatus::Ready` 发射**之后**，通过 `tauri::async_runtime::spawn` 异步启动 Android 原生网络状态监控（`start_network_monitoring`）。延迟激活是为了规避冷启动时 JNI WebView 未就绪导致的死锁——此时所有 Rust 业务服务已完全就绪，Native 回调可以安全地触发分布式节点自动重连。

### 1.4 系统快照查询

```rust
#[derive(Debug, Serialize, Clone)]
pub struct SystemSnapshot {
    pub core: CoreStatus,      // 本模块状态 (Initializing / Ready / Error)
    pub log: String,           // vcp_log_service 的连接状态
    pub sync: String,          // sync_service 的连接状态
    pub distributed: String,   // 分布式节点连接状态 (ConnectionState 序列化)
}
```

`get_system_snapshot` 聚合**四个**子系统的运行时状态，供前端 `NotificationStatusBar.vue` 和"系统状态"面板一键诊断。分布式状态通过 `try_state::<DistributedState>()` 获取——若 `DistributedState` 未注册则 fallback 为 `"disconnected"`。`get_core_status` 与 `get_last_error` 提供细粒度单项查询。

### 1.5 分布式节点生命周期裁决 (`reconcile_distributed_node`)

`reconcile_distributed_node(app, enabled, force_reconnect)` 是分布式节点连接的**幂等裁决器**，确保应用状态与设置项始终一致：

```
调用来源：
├── bootstrap() 步骤 3.6 —— 应用启动时自动恢复上次的连接状态
├── reconcile_distributed_node_cmd —— 前端保存设置后手动触发
└── settings_manager 变更监听 —— WS URL/Key/设备名变更时自动回调
```

**裁决逻辑**：

```rust
if force_reconnect && client.is_running() → stop() → 置 is_running = false

match (enabled, is_running):
    (true, false)  → 读 settings (ws_url, vcp_key, device_name)
                    → 校验 ws_url/vcp_key 非空（为空则 warn 跳过）
                    → registry.load_disabled_config()
                    → client.start(ws_url, vcp_key, device_name, registry)
    (false, true)  → client.stop()
    _              → 无操作
```

| 参数 | 含义 | 使用场景 |
|------|------|----------|
| `enabled` | 分布式功能总开关 | `settings.distributed_enabled` |
| `force_reconnect` | 是否强制停止旧连接后再启动新连接 | WS URL/Key 变更时为 `true`（需要新凭据建立连接） |

> **`is_running()` vs `is_connected()`**：裁决器使用 `client.is_running()` 而非 `is_connected()`。前者在 `Connecting`、`Connected`、`Disconnecting` 状态下均返回 true——如果在退避重连期间（状态为 Connecting），设置再次触发不应启动第二个连接任务。

### 1.6 Tauri 命令表

| Command | 用途 | 参数 |
|---------|------|------|
| `get_system_snapshot` | 四合一系统状态快照（core + log + sync + distributed） | 无 |
| `reconcile_local_server_cmd` | 即时启动/停止浮动助手本地服务器 | `enable: bool` |
| `reconcile_distributed_node_cmd` | 即时启动/停止分布式节点 | `enable: bool` |
| `get_core_status` | 获取 CoreStatus 枚举值 | 无 |
| `get_last_error` | 获取最近一次启动阶段的致命错误信息 | 无 |

## 2. 维护管理器（Maintenance Manager）

### 2.1 模块定位

`maintenance_manager.rs` 是 VCP Mobile 的**系统内务与垃圾回收中心**。它将低频但关键的维护任务（孤儿附件清理、WebView 缓存清理、数据库空间回收）聚合在同一模块，对齐前端 `MaintenanceSection` 领域。

### 2.2 孤儿附件清理

#### 2.2.1 全量清理（`cleanup_orphaned_attachments`）

清理逻辑遵循**四道防线**设计：

| 防线 | 机制 | 说明 |
|------|------|------|
| 防线一 | `attachments` 表全量扫描 | 获取所有已记录的 `(hash, internal_path)` |
| 防线二 | `message_attachments` ↔ `messages` 内连接 | 仅保留 `deleted_at IS NULL` 的有效引用 |
| 防线三 | GC 阶段关联自愈 | 清理前自动删除指向已删除消息的 `message_attachments` 记录 |
| 防线四 | 物理文件存在性校验 | 仅在文件确实存在时才执行删除与空间统计 |

清理范围包括：
- 附件物理文件（`attachments_dir/<hash>.<ext>`）
- 缩略图文件（`thumbnails/<hash>_thumb.webp`）
- 数据库 `attachments` 表中对应记录

返回结果包含**删除数量**与**释放空间（MB）**。

#### 2.2.2 精准单条清理（`cleanup_single_orphaned_attachment`）

供前端「取消暂存」场景调用。在单条清理前增加两道安全机制：

1. **引用校验**：检查该 `hash` 是否被任何有效消息引用，若被引用则跳过。
2. **5 分钟安全锁宽限期**：对比 `attachments.created_at` 与当前时间，若不足 300 秒则跳过，防止时序竞争导致「刚创建就被删除」。

> `created_at` 存在秒级与毫秒级两种历史数据，统一换算为秒级对比。

### 2.3 WebView 缓存清理

`clear_webview_cache` 调用 Tauri v2 原生 API `webview.clear_all_browsing_data()`，清除 HTTP 缓存、图片缓存、LocalStorage 等浏览数据。仅在能获取到 `"main"` 窗口句柄时执行。

### 2.4 自动维护调度

`init_automatic_maintenance` 在应用启动时调用，检查距离上次维护是否超过 **3 天**（`3 * 24 * 60 * 60` 秒）。若超时，自动执行：

| 步骤 | 操作 | 意图 |
|------|------|------|
| 1 | `clear_all_browsing_data()` | 回收 WebView 磁盘占用 |
| 2 | `PRAGMA incremental_vacuum(100)` | SQLite 物理空间回收，每次 100 页，避免长时 IO 阻塞 |
| 3 | 删除指向已删除消息的 `message_attachments` 记录 | 关联自愈，防止关联表膨胀 |
| 4 | `PRAGMA optimize` | 更新 SQLite 查询规划器统计信息 |
| 5 | 写入 `lastWebviewCacheClear` 时间戳 | 避免短期内重复触发 |

## 3. VCP 日志服务（VCP Log Service）

### 3.1 模块定位

`vcp_log_service.rs` 负责与桌面端 VCP 的 **VCPlog WebSocket 服务**建立并维持长连接，将系统事件、调试日志实时推送到桌面端聚合展示。该模块是移动端与桌面端之间除同步服务外的第二条实时数据通道。

### 3.2 核心全局状态

模块通过 `lazy_static!` 维护 6 组全局静态变量，实现跨命令调用的状态共享：

| 变量 | 类型 | 职责 |
|------|------|------|
| `HEARTBEAT_INTERVAL_MS` | `AtomicU64` | 心跳间隔，默认 15000 ms，运行时可通过 Command 动态修改 |
| `LOG_CONNECTION_ACTIVE` | `Arc<AtomicBool>` | 连接任务是否已启动的互斥标志 |
| `LOG_SENDER` | `Arc<Mutex<Option<mpsc::UnboundedSender<Value>>>>` | 供 `send_vcp_log_message` 向 WebSocket 循环发送消息 |
| `WS_URL_CHANNEL` | `(watch::Sender<Option<Url>>, watch::Receiver<Option<Url>>)` | URL 配置通道，支持运行时切换地址而无需重启任务 |
| `CURRENT_LOG_STATUS` | `Arc<RwLock<String>>` | 当前连接状态：`closed` / `connecting` / `open` / `error` |
| `HEARTBEAT_RESET_TX` | `Arc<Mutex<Option<mpsc::Sender<()>>>>` | 心跳重置信号发送端，用于动态调整间隔时立即生效 |

> 关键修复：`WS_URL_CHANNEL` 使用 `watch::channel` 而非 `mpsc`，是因为 `watch` 保留最近一个值，允许新订阅者立即获取当前 URL，防止任务重启时丢失配置。

### 3.3 连接生命周期

```
前端调用 init_vcp_log_connection(url, key)
           │
           ▼
    ┌──────────────┐
    │ URL 解析与校验 │ ← parse_log_url：补全 /VCPlog 路径、追加 VCP_Key 查询参数
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │ URL 写入通道   │ ← WS_URL_CHANNEL.0.send(Some(ws_url))
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │ 首次启动监听   │ ← CAS 检查 LOG_CONNECTION_ACTIVE，避免重复 spawn
    └──────┬───────┘
           │
           ▼
    start_vcp_log_listener(app_handle)
           │
    ┌──────┴───────┐
    ▼              ▼
 重试循环          URL 变更监听
    │              │
    ▼              ▼
 connect_async    收到新 URL → break 当前连接，进入下一轮重试
    │
    ▼
 tokio::select! {
     url_rx.changed()    → 配置变更，断开重连
     reset_rx.recv()     → 心跳间隔变更，重置定时器
     heartbeat_timer     → 发送 WebSocket Ping
     ws_read.next()      → 接收文本消息，转发给前端
     rx.recv()           → 发送待 outgoing 消息
 }
```

### 3.4 心跳机制

心跳采用 **WebSocket Ping 帧**（非应用层消息），间隔默认 **15 秒**：

```rust
_ = &mut heartbeat_timer => {
    ws_write.send(Message::Ping(vec![].into())).await?;
    heartbeat_timer.reset(now + Duration::from_millis(current_ms));
}
```

**动态心跳调整**：
- `set_vcp_log_heartbeat(interval_ms)` 将新值写入 `AtomicU64`，并通过 `HEARTBEAT_RESET_TX` 发送重置信号。
- 监听循环内通过 `tokio::select!` 捕获 `reset_rx.recv()`，读取最新原子值后调用 `Sleep::reset()` 重新校准定时器，**无需断开重连**。
- `App.vue` 根据 Android 生命周期事件自适应调整：前台 15s、后台 120s，以降低 OEM 杀后台概率。

### 3.5 重试与容错

| 场景 | 行为 |
|------|------|
| 连接失败 | 指数退避重试：初始 1s，每次翻倍，上限 60s |
| 请求构造失败 | 发射错误卡片（前端 Toast），提示检查 URL 格式 |
| 连接超时（10s） | 发射错误卡片，提示检查桌面端运行状态与局域网 |
| 连接被服务器关闭 | 状态置为 `closed`，进入退避重连 |
| URL 变更 | 立即中断当前连接，以新地址重连 |

所有连接状态变更均通过 `vcp-system-event`（`type: vcp-log-status`）广播到前端，并附带人类可读的错误卡片（`type: vcp-log-message`）辅助排查。

### 3.6 消息发送

`send_vcp_log_message(payload)` 通过 `LOG_SENDER` 向 WebSocket 循环的 `mpsc` 接收端投递 JSON 消息。若连接未建立，返回 `"VCPLog connection is not active"`。

## 4. 文件提取器（File Extractor）

### 4.1 模块定位

`file_extractor.rs` 是 VCP Mobile 的**多模态文件文本提取纯函数库**。它负责将用户上传的附件（纯文本、代码、Office 文档、PDF）转换为可供 LLM 消费的纯文本字符串。该模块被 `file_manager.rs` 在附件注册阶段调用，提取结果写入 `attachments.extracted_text` 字段。

### 4.2 格式覆盖矩阵

模块维护两份后缀白名单，决定哪些文件类型进入提取管道：

**纯文本 / 代码格式**（`TEXT_AND_CODE_EXTENSIONS`，59 种）：
`txt`, `md`, `csv`, `json`, `js`, `ts`, `tsx`, `jsx`, `vue`, `rs`, `py`, `java`, `c`, `cpp`, `h`, `hpp`, `cs`, `go`, `rb`, `php`, `swift`, `kt`, `kts`, `css`, `html`, `xml`, `yaml`, `yml`, `toml`, `ini`, `sql`, `log`, `jsonc`, `dart`, `lua`, `r`, `pl`, `ex`, `exs`, `zig`, `hs`, `scala`, `groovy`, `d`, `nim`, `cr` 等。

**结构化文档格式**（`STRUCTURED_DOC_EXTENSIONS`，4 种）：
`docx`, `pdf`, `xlsx`, `pptx`

### 4.3 纯文本提取：mmap + 自动编码检测

`read_text_with_mmap` 采用两层编码检测策略：

1. **BOM 头检测**（最可靠）：通过 `encoding_rs::Encoding::for_bom()` 识别 UTF-8/UTF-16 LE/UTF-16 BE/UTF-32 等 BOM，并自动移除 BOM 后解码。
2. **统计检测**（无 BOM 时）：使用 `chardetng::EncodingDetector`（Firefox 同款引擎）对全文件字节进行频率分析，猜测最可能的编码。

底层使用 `memmap2::Mmap` 将文件映射到虚拟地址空间，避免大文件全量载入内存。

### 4.4 结构化文档提取

#### 4.4.1 DOCX（`extract_docx_text`）

基于 `zip::ZipArchive` 解压 `word/document.xml`，以**单遍字符状态机**解析 XML，无需完整 DOM 树：

| 特性 | 实现方式 |
|------|---------|
| 普通段落 | `<w:p>` → `<w:t>` 文本收集，段末加 `\n` |
| 标题 | 解析 `<w:pStyle w:val="HeadingN">`，映射为 Markdown `#`/`##`/`###` |
| 无序列表 | 检测 `<w:numPr>`，前缀添加 `- ` |
| 表格 | `<w:tbl>` → `<w:tr>` → `<w:tc>`，输出 Markdown 表格格式（含 `---` 分隔行） |
| 加粗 | `<w:b>` / `<w:bCs>` 在 `<w:rPr>` 内时，文本包裹 `**`** |
| 加粗熔接 | 后处理将 `**学****号**` 优化为 `**学号**` |

> 标题匹配采用精确防误策略：仅当 `pStyle` 值以 `heading` 开头或精确匹配 `1`/`2`/`3`/`Heading1`/`标题 1` 等才判定为标题，避免 `ListParagraph3` 被误判为 H3。

#### 4.4.2 PPTX（`extract_pptx_text`）

解压 `ppt/slides/slide*.xml`，按幻灯片编号排序，逐页提取 `<a:t>` 文本内容。每页输出格式：

```
--- Slide N ---
[文本内容]
```

#### 4.4.3 XLSX（`extract_xlsx_text`）

三阶段提取：
1. **Sheet 名称解析**：从 `xl/workbook.xml` 提取 `sheetId` → `name` 映射。
2. **共享字符串表**：解析 `xl/sharedStrings.xml`，建立索引到文本的映射。
3. **单元格遍历**：逐 sheet 解析 `xl/worksheets/sheet*.xml`：
   - 解析 `<c r="A1" t="s">` 的 `r` 属性（单元格坐标），通过 `col_name_to_index` 将列名字母（如 `AB`）转为数字索引。
   - `t="s"` 表示共享字符串，从共享字符串表查值；否则取 `<v>` 中的原始值。
   - 空列用 `\t` 占位，保持表格结构对齐。

#### 4.4.4 PDF（`extract_pdf_text`）

使用 `lopdf::Document::load()` 载入 PDF，按页码顺序调用 `doc.extract_text()`。若某页提取失败，打印日志并继续下一页，不阻断整体流程。

### 4.5 主入口：try_extract_text

```rust
pub fn try_extract_text(path: &Path, mime_type: &str) -> Option<String>
```

分流逻辑：

```
输入 (path, mime_type)
    │
    ▼
┌─────────────────────────┐
│  MIME 以 text/ 开头     │
│  或属于文本/代码后缀    │ ──▶ mmap + 编码检测 ──▶ 返回纯文本
└─────────────────────────┘
    否
    ▼
┌─────────────────────────┐
│  扩展名 = docx/pptx/    │
│  xlsx/pdf               │ ──▶ 结构化提取器 ──▶ 返回纯文本
└─────────────────────────┘
    否
    ▼
   None（无法提取）
```

**安全边界**：
- **50 MB 文件硬上限**：超过此阈值的文本文件直接返回占位提示，防止 OOM。
- **1000 万字符截断**：提取后文本若超过 1000 万字符（约对应 2M Token 上下文窗口），尾部截断并附加 `……（文本过长已截断）`。

### 4.6 与 file_manager.rs 的协作

`file_extractor.rs` 不直接操作数据库或文件系统，仅以**纯函数**形式提供提取能力。协作边界如下：

```
┌─────────────────────┐      ┌─────────────────────┐      ┌─────────────────────┐
│   file_manager.rs   │ ──▶  │   file_extractor.rs │      │      SQLite         │
│                     │      │                     │      │                     │
│  register_local_file│      │  try_extract_text() │      │  attachments 表     │
│  store_file         │ ──▶  │  read_text_with_mmap│      │  extracted_text     │
│  get_refined_mime   │      │  extract_docx_text  │      │                     │
│                     │ ◀──  │  extract_pdf_text   │      │                     │
│  (调用后入库)        │      │  ...                │      │                     │
└─────────────────────┘      └─────────────────────┘      └─────────────────────┘
```

调用时机：
- `register_local_file`：零拷贝路径传输完成后，对新注册附件调用提取，结果写入 `extracted_text`。
- `store_file`：小文件直存完成后，同样触发提取。

`file_manager.rs` 负责 MIME 精修与路径安全，`file_extractor.rs` 仅接收已校验过的 `Path` 和 `mime_type`，不处理任何业务状态。

## 5. 基础工具包（Base Utils）

### 5.1 模块定位

`utils.rs` 是 VCP Mobile 基础设施层的**无状态通用原语工具包**。它沉淀纯算法级、无副作用的高复用底层工具，面向全后端模块提供跨领域共享。该模块不依赖任何业务状态、数据库连接或外部 I/O，所有函数均为纯计算型或协作调度型。

该模块通过 `infra/mod.rs` 中的 `pub mod utils;` 声明为子模块，但**未在 `vcp_modules/mod.rs` 中通过 `pub use` 扁平化再导出**——各消费模块通过 `crate::vcp_modules::infra::utils::*` 路径直接引用。

### 5.2 YieldCounter — 协作式 CPU 挂起出让计数器

```rust
pub struct YieldCounter {
    count: u32,
    threshold: u32,
}

impl YieldCounter {
    pub fn new(threshold: u32) -> Self  // 推荐默认阈值 150-200
    pub async fn tick(&mut self)        // 递增计数，达阈值时自动 yield_now()
}
```

- **设计目的**：在重 I/O 或超大循环遍历（如批量消息处理、大量文件扫描）中，防止长时间占用 CPU 导致前台 WebView 帧率下降或 UI 卡顿。
- **工作原理**：每调用一次 `tick()`，内部计数器递增。当计数器达到 `threshold` 时，计数器归零并调用 `tokio::task::yield_now().await` 主动挂起，出让当前 CPU 时间片给其他任务（包括 WebView 渲染循环）。
- **推荐阈值**：`150-200`——在每 150-200 次迭代后挂起一次，平衡 CPU 利用率与 UI 响应性。
- **使用模式**：

```rust
let mut yc = YieldCounter::new(200);
for item in large_collection.iter() {
    process(item);
    yc.tick().await; // 每 200 次迭代出让一次 CPU
}
```

### 5.3 is_valid_cas_hash — CAS 哈希指纹校验

```rust
pub fn is_valid_cas_hash(hash: &str) -> bool
```

- **用途**：校验字符串是否为合法的 Content-Addressable Storage (CAS) 哈希指纹——64 位十六进制 SHA-256 摘要。
- **校验规则**：长度精确为 64 字符且所有字符均为 ASCII 十六进制数字（`0-9a-fA-F`）。
- **消费场景**：同步服务 (`sync_service.rs`) 和同步哈希模块 (`sync_hash.rs`) 在校验服务端返回的哈希指纹时调用，防止格式异常导致比对逻辑失败。

### 5.4 now_secs / now_millis — 时钟回拨安全的时间戳工具

```rust
pub fn now_secs() -> i64   // 当前秒级 UNIX 时间戳
pub fn now_millis() -> i64 // 当前毫秒级 UNIX 时间戳
```

- **设计目的**：标准库 `SystemTime::now().duration_since(UNIX_EPOCH)` 在系统时钟被回拨（如 NTP 校时、用户手动调整）时可能返回 `Err`（溢出 panic）。本项目中有大量日志、事件、过期判断等场景需要频繁获取时间戳，每次手动处理 `unwrap_or_default()` 较为繁琐。
- **行为**：内部使用 `unwrap_or_default()` 吞下 `SystemTimeError`，在时钟异常时返回 `0`（UNIX 纪元），而非 panic。`0` 作为合法时间戳的下确界，在大多数业务逻辑（过期判断、时间差计算）中表现为"极早"而非错误。
- **使用建议**：所有需要获取当前时间戳的模块（`chat_manager`、`sync_service`、`lifecycle_manager`、`maintenance_manager` 等）均可优先使用此封装，避免散布 `unwrap_or_default()` 模式。

### 5.5 calculate_sha256 / calculate_sha256_slices — SHA-256 摘要工具

```rust
pub fn calculate_sha256(bytes: &[u8]) -> String
pub fn calculate_sha256_slices(slices: &[&[u8]]) -> String
```

- **用途**：计算字节数据的 SHA-256 十六进制摘要（统一小写输出），为所有需要哈希指纹的模块提供一致的计算方式。
- **差异化**：
  - `calculate_sha256`：单字节切片输入，适用于完整数据块（如文件内容）。
  - `calculate_sha256_slices`：多段不连续字节切片输入，依次 `hasher.update()` 后统一 `finalize`。适用于需要将散列在内存不同区域的元数据片段（如实体 ID + 名称 + 内容）合并计算哈希指纹的场景。
- **依赖**：使用 `sha2::Sha256` 和 `hex::encode`，与 `sync_hash.rs` 的高层聚合逻辑形成互补——`sync_hash.rs` 负责实体指纹策略（哪些字段参与哈希），`utils.rs` 提供底层的哈希计算原语。
- **消费场景**：同步管线的 `ManifestBuilder`（指纹构造）、`sync_hash.rs`（实体哈希聚合）、CAS 内容存储校验等。

### 5.6 设计原则

| 原则 | 说明 |
|------|------|
| 无状态（Stateless） | 所有函数均为纯函数或仅依赖输入参数，不持有内部状态（`YieldCounter` 的状态为显式管理，非全局隐式） |
| 无副作用（Side-effect Free） | 除 `YieldCounter::tick()` 的调度出让外，所有函数不执行 I/O、不修改外部状态 |
| 不 panic | `now_secs`/`now_millis` 使用 `unwrap_or_default()` 免疫时钟异常；`calculate_sha256` 等为标准库安全操作 |
| 零依赖传播 | 模块仅依赖 `sha2` 和 `hex`（已在项目中广泛使用），不引入新外部 crate |

## 7. reconcile 生命周期协调模式

The bootstrap sequence (§1.3) includes two reconcile functions that implement a declarative state reconciliation pattern:

### reconcile_local_server

```rust
pub async fn reconcile_local_server(app_handle, lifecycle, enable_assistant: bool)
```

Purpose: Aligns the floating assistant local server state with the `enableAssistant` setting.

| State | Action |
|-------|--------|
| enable=true, no server | Start local server via `local_server::start_server()` |
| enable=false, server running | Shutdown existing server via `handle.shutdown()` |
| State matches | No-op |

Uses `lifecycle.local_server_handle: Arc<Mutex<Option<ServerHandle>>>` to track server lifecycle.

### reconcile_distributed_node

```rust
pub async fn reconcile_distributed_node(app_handle, distributed_enabled: bool, force_reconnect: bool)
```

Purpose: Aligns the distributed node WebSocket connection with the `distributedEnabled` setting.

| State | Action |
|-------|--------|
| enable=true, not running, URL/Key valid | Spawn async task to start WS connection with settings-derived parameters |
| enable=true, not running, URL/Key empty | Log warning, skip (avoids connection attempt with empty config) |
| enable=false, running | Call `client.stop()` to disconnect |
| `force_reconnect=true`, running | Stop existing connection first, then re-spawn |

Key design details:
- Reads settings at reconciliation time (not cached) to pick up config changes
- Connection params derived from: `settings.distributed_ws_url`, `settings.distributed_vcp_key`, `settings.distributed_device_name`
- Device name defaults to "VCPMobile" if setting is empty string
- Connection start is spawned as a separate async task to avoid blocking the bootstrap or settings-save flow
- On start, loads disabled tools config via `registry.load_disabled_config(&app)`

### Tauri Command wrappers

Both reconcile functions are exposed as Tauri commands for frontend on-save triggers:
- `reconcile_local_server_cmd(app, state, enable: bool) -> Result<bool, String>`
- `reconcile_distributed_node_cmd(app, enable: bool) -> Result<bool, String>`

These are called by `settings_manager.rs` after persisting settings changes, ensuring immediate service state alignment without app restart.

### Call Sites

| Caller | Function | Trigger |
|--------|----------|---------|
| `bootstrap()` (step 3.6) | `reconcile_distributed_node(handle, enable_dist, false)` | App startup, if `distributedEnabled` was persisted as true |
| `bootstrap()` (step 3.5) | `reconcile_local_server(handle, lifecycle, enable)` | App startup, if `enableAssistant` was persisted as true |
| `settings_manager::save_settings()` | `reconcile_distributed_node(app, enabled, force_reconnect)` | User toggles distributed switch or changes WS URL |
| Frontend via `invoke()` | `reconcile_local_server_cmd` | User toggles floating assistant |

### Design Pattern

The reconcile pattern follows a **declarative desired-state** approach:
1. The caller declares the desired state (enabled/disabled)
2. The reconcile function determines the current state
3. It performs the minimum action to align current → desired
4. No redundant start/stop cycles (idempotent)

This is superior to imperative start/stop commands because:
- Handles edge cases (e.g., double-enable, double-disable) gracefully
- Survives partial failures (e.g., settings say enabled but server crashed)
- Unifies startup and runtime paths into a single function

## 8. 错误处理与边界情况

### 7.1 生命周期管理器

| 场景 | 行为 |
|------|------|
| 数据库初始化失败 | 状态置 `Error`，发射 fatal 事件，返回 `Err`，启动中断 |
| 配置读取失败 | 同上，视为致命错误 |
| 后台任务失败（表情/模型/日志） | 仅打印日志，不阻断启动，状态仍为 `Ready` |
| 前端热更新失败 | 仅打印日志，不触发任何用户可见错误 |

### 7.2 维护管理器

| 场景 | 行为 |
|------|------|
| 附件目录不存在 | 直接返回 `"没有附件需要清理"` |
| 索引库为空 | 返回 `"索引库为空，无需清理"` |
| 缩略图目录解析失败 | 回退到附件同级 `thumbnails/` 目录 |
| 单条清理遇到安全锁 | 返回宽限期提示，不执行删除 |
| WebView 窗口未找到 | 返回 `"未找到主窗口，无法执行清理"` |

### 7.3 VCP 日志服务

| 场景 | 行为 |
|------|------|
| URL/Key 为空 | 向通道发送 `None`，静默停止现有连接 |
| 连接已存在 | CAS 返回，避免重复 spawn 监听任务 |
| WebSocket 读取到非 JSON 文本 | 以 `type: raw_text` 透传前端 |
| 发送消息时通道未建立 | 返回 `"VCPLog connection is not active"` |

### 7.4 文件提取器

| 场景 | 行为 |
|------|------|
| 文件超过 50 MB | 返回占位提示，跳过提取 |
| 编码检测失败 | `chardetng` 会给出最可能猜测，极少返回 `None` |
| DOCX/PPTX/XLSX 内部 XML 缺失 | 打印日志，返回 `None` |
| PDF 某页提取失败 | 打印日志，跳过该页，继续后续页面 |
| 未知扩展名 | 返回 `None`，由调用方决定如何处理 |

### 7.5 基础工具包

| 场景 | 行为 |
|------|------|
| `now_secs`/`now_millis` 遇到时钟回拨 | `unwrap_or_default()` 返回 `0`（UNIX 纪元），不 panic |
| `is_valid_cas_hash` 输入长度不等于 64 | 返回 `false`，不 panic |
| `is_valid_cas_hash` 输入含非十六进制字符 | 返回 `false`，不 panic |
| `calculate_sha256`/`calculate_sha256_slices` 输入为空 | 返回空输入的 SHA-256 摘要（合法哈希值） |
| `YieldCounter::tick` 长时间循环 | 每 `threshold` 次迭代出让 CPU，不影响逻辑正确性 |

### 8.2 安全

- **生命周期状态机**：`CoreStatus` 使用 `Arc<RwLock<>>` 保证跨线程安全，`bootstrap` 中的顺序依赖确保服务不会在使用前被调用。
- **维护清理的路径安全**：仅删除数据库中记录的 `internal_path`，不遍历任意目录。`file_manager.rs` 的 `ensure_safe_path` 已在文件注册阶段完成路径校验。
- **VCP Log 的 URL 安全**：`parse_log_url` 自动补全路径并追加 `VCP_Key`，日志中输出时会将 Key 掩码为 `********`。
- **文件提取的内存安全**：50 MB 硬上限 + mmap 避免大文件载入内存；结构化文档解析使用单遍状态机，不构建完整 DOM。
- **工具包的无 panic 设计**：所有 utils 函数均不 panic——`now_secs`/`now_millis` 遇时钟异常返回 `0` 而非崩溃；`is_valid_cas_hash` 仅做字符扫描，不分配内存；SHA-256 计算为纯数学运算，无失败路径。
