# 22_本地服务器与浮动助手（Local Server & Floating Assistant IPC）

> 源文件：
> - `src-tauri/src/vcp_modules/infra/local_server.rs`（341 行）
> - `src/core/stores/floatingAssistant.ts`（352 行）
> - `src-tauri/plugins/vcp-mobile/android/src/main/java/com/vcp/mobile/FloatingWindowManager.kt`（259 行）
>
> 本文档覆盖 VCP Mobile 内置的 axum HTTP + WebSocket 本地服务器，该服务器为 Android 浮动助手窗口提供 IPC 通道，实现浮动 WebView 与 Rust 后端的实时双向通信。

---

## 1. 架构总览

### 1.1 模块定位

`local_server.rs` 是一个**嵌入式 HTTP + WebSocket 服务器**，基于 [axum](https://docs.rs/axum/) 框架构建，监听 `127.0.0.1:14202`。它的存在目的是：

1. **浮动窗口 IPC**：Android 悬浮 WebView 无法直接调用 Tauri `invoke()`（因为浮动窗口与主 Activity 处于不同的 Window 上下文），本地服务器充当"中继层"，将 WebSocket 消息桥接到 Rust 业务函数。
2. **静态资产服务**：为浮动窗口的 WebView 提供前端文件（HTML/JS/CSS），支持 Tauri 内置资源、开发代理、兜底回退三层策略。
3. **优雅关闭**：通过 `watch::channel` 实现信号驱动的优雅退出，确保服务器不在应用关闭时残留僵尸进程。

### 1.2 系统位置

```
┌─────────────────────────────────────────────────────────┐
│                   Android 系统                           │
│  ┌─────────────────┐       ┌──────────────────────────┐ │

│  │ chatView (组件A) │       │  WebView                 │ │
```

## 2. 核心类型与状态

### 2.1 AppState

```rust
#[derive(Clone)]
struct AppState {
    app_handle: AppHandle,
}
```

`AppState` 持有 Tauri `AppHandle` 的克隆引用，通过 axum 的 `State` 提取器注入每个路由处理器。它不包含任何可变状态——所有业务状态（设置、Agent 配置、活跃请求）通过 `app_handle.state::<T>()` 按需获取。

### 2.2 ServerHandle

```rust
pub struct ServerHandle {
    shutdown_tx: watch::Sender<bool>,
}
```

`ServerHandle` 是对外暴露的唯一控制句柄：

| 方法 | 行为 |
|------|------|
| `shutdown(self)` | 发送 `true` 信号到 `watch` 通道，触发 `with_graceful_shutdown` 退出 axum serve 循环 |

`watch::channel` 的选择理由是：
- 优雅关闭仅需一次信号，不需要 `mpsc` 的多条消息能力
- `watch::Receiver` 的 `changed().await` 语义天然适合"等待关闭信号"的场景
- 不会产生重复信号导致的不必要 panic

## 3. 路由表

服务器注册了 4 条路由：

| 路由 | 方法 | 处理器 | 用途 |
|------|------|--------|------|
| `/ws` | GET | `ws_handler` | WebSocket 升级入口，处理浮动窗口的所有业务消息 |
| `/` | GET | `index_handler` | 返回 `index.html`（主应用入口，供调试/回退） |
| `/floating` | GET | `floating_handler` | 返回 `floating.html`（浮动助手独立入口） |
| `/*` (fallback) | GET | `any_handler` | 兜底路由，服务所有未被显式匹配的静态资源请求 |

### 3.1 WebSocket 路由 (`/ws`)

`ws_handler` 负责完成 WebSocket 升级，然后将连接句柄交给 `handle_socket` 进入消息循环：

```
客户端连接 ws://127.0.0.1:14202/ws
           │
           ▼
    ws.on_upgrade(|socket| handle_socket(socket, state))
           │
           ▼
    ┌───────────────────────────┐
    │  receiver.next().await    │  ──► 等待 JSON 消息
    │  (Message::Text)         │
    └───────────┬───────────────┘
                │
                ▼
    serde_json::from_str(text) → 解析为 serde_json::Value
                │
                ▼
    match action:
      "handle_assistant_chat_stream" → 流式 Agent 对话
      "archive_assistant_chat"       → 归档临时会话
      "get_initial_config"           → 获取初始设置
      _                               → 忽略
```

### 3.2 静态文件路由 (`/`, `/floating`, `/*`)

三条静态路由最终都委托给 `serve_asset()`，后者实现三层资源查找策略（详见第 6 节）。

## 4. WebSocket 协议

### 4.1 消息格式

所有 WebSocket 消息均为 **JSON 文本帧**，方向包含 **客户端 → 服务器**（请求）和 **服务器 → 客户端**（响应/推送）。

#### 客户端 → 服务器（请求）

```json
{
  "action": "<action_name>",
  "payload": { ... }
}
```

其中 `payload` 的结构取决于 `action`。

#### 服务器 → 客户端（响应/推送）

```json
{
  "type": "<message_type>",
  // ... type-specific fields
}
```

### 4.2 Action: `handle_assistant_chat_stream`

**请求格式**：

```json
{
  "action": "handle_assistant_chat_stream",
  "payload": {
    "agentId": "uuid-of-agent",
    "tempMessages": [
      { "role": "user", "name": "User", "content": "...", "timestamp": 1717500000000 }
    ],
    "vcpUrl": "https://vcp-endpoint.com",
    "vcpApiKey": "sk-..."
  }
}
```

**处理流程**：

```
接收请求
    │
    ▼
1. 反序列化 payload → AssistantChatPayload
    │
    ▼
2. 创建 tauri::ipc::Channel 作为流式事件通道
    │
    ▼
3. spawn 异步任务:
   agent_chat_application_service::handle_assistant_chat_stream(...)
    │
    ▼
4. while let Some(event) = rx.recv().await
    │
    ▼
5. 将 InvokeResponseBody::Json 直接通过 WebSocket 文本帧转发
```

**响应消息流**（服务器推送，非请求-响应模式）：

| type | 触发时机 | 关键字段 |
|------|---------|----------|
| `thinking` | Agent 开始思考 | `messageId`, `message_id` |
| `data` | 流式内容块到达 | `messageId`, `chunk` (含 `choices[0].delta.content`) |
| `end` | 流式输出完成 | `messageId` |
| `error` | 发生错误 | `error` |

消息格式与主应用 Tauri Channel 完全一致（`InvokeResponseBody::Json`），因此浮动窗口的前端可以使用与主应用相同的 `handleWsMessage` 逻辑处理流式响应。

### 4.3 Action: `archive_assistant_chat`

**请求格式**：

```json
{
  "action": "archive_assistant_chat",
  "payload": {
    "ownerId": "uuid-of-agent",
    "ownerType": "agent",
    "tempMessages": [
      { "role": "user", "content": "..." },
      { "role": "assistant", "content": "..." }
    ]
  }
}
```

**处理流程**：

```
接收请求
    │
    ▼
反序列化 ownerId / ownerType / tempMessages
    │
    ▼
topic_service::archive_assistant_chat(app_handle, state, owner_id, owner_type, temp_messages)
    │
    ▼
┌──────────────┐       ┌──────────────┐
│ 成功          │       │ 失败          │
│ → { type:    │       │ → { type:    │
│     "archive │       │     "error", │
│     _success"│       │     error:   │
│     , topicId│       │     "..." }  │
│     : "..." }│       │              │
└──────────────┘       └──────────────┘
```

**关键设计**：归档后返回的 `topicId` 可被主应用用于定位并打开新话题。前端收到 `archive_success` 后清除浮动窗口内的临时消息列表。

### 4.4 Action: `get_initial_config`

**请求格式**：

```json
{
  "action": "get_initial_config"
}
```

**处理流程**：

```
接收请求
    │
    ▼
settings_manager::read_settings(app_handle, state)
    │
    ▼
┌──────────────────────────────────────────────────┐
│ 成功: 返回完整 settings 对象                      │
│ {                                                │
│   "type": "initial_config",                      │
│   "settings": {                                  │
│     "userName": "...",                           │
│     "vcpServerUrl": "https://...",               │
│     "vcpApiKey": "sk-...",                       │
│     "assistantAgentId": "uuid-of-agent",         │
│     ...                                          │
│   }                                              │
│ }                                                │
└──────────────────────────────────────────────────┘
│
▼
失败: { "type": "error", "error": "Failed to read settings: ..." }
```

**用途**：浮动窗口 WebSocket 连接建立后立即调用，获取 VCP 服务器连接信息、绑定的 Agent ID 等必要配置，避免在浮动模式下依赖 Tauri invoke。

## 5. 生命周期

### 5.1 启动

本地服务器在应用启动的 **bootstrap 阶段**通过 `reconcile_local_server_cmd` 命令启动：

```
bootstrap() (lifecycle_manager.rs)
    │
    ▼
P2 后台任务 (非阻塞)
    │
    ▼
reconcile_local_server_cmd(enable: bool)
    │
    ├── enable == true:
    │     ├── 创建 AppState { app_handle: app_handle.clone() }
    │     └── start_server(app_handle)
    │           ├── 绑定 127.0.0.1:14202
    │           ├── 注册 4 条路由
    │           ├── spawn axum::serve(...).with_graceful_shutdown(shutdown_signal)
    │           └── 返回 ServerHandle
    │
    └── enable == false:
          └── server_handle.shutdown()
```

启动时由 `AssistantSettingsSection` 的 `enableAssistant` 设置控制——用户开启悬浮助手功能时才启动服务器。

### 5.2 关闭

```
应用关闭 / 用户禁用悬浮助手
    │
    ▼
server_handle.shutdown()
    │
    ▼
shutdown_tx.send(true)
    │
    ▼
watch::Receiver 检测到信号
    │
    ▼
axum::serve 退出，清理 TCP 监听器
    │
    ▼
日志: "[LocalServer] Server stopped."
```

**优雅关闭保证**：
- axum 的 `with_graceful_shutdown` 不会暴力关闭正在处理的连接
- 关闭信号发出后，已建立的 WebSocket 连接自然结束（发送端 `.send()` 返回错误时 `break`）
- 不产生僵尸 TCP 端口

## 6. 资产服务策略（3-Tier）

`serve_asset()` 实现三层资产查找，按优先级排列：

```
请求: GET /path/to/asset?query
              │
              ▼
┌──────────────────────────────────────────┐
│ Tier 1: Tauri 内置资源                    │
│ app_handle.asset_resolver().get(path)    │
│                                          │
│ 命中: 返回内置资源字节 + 正确的 MIME       │
│ 未命中: 进入 Tier 2                       │
│                                          │
│ MIME 修复: 若 query 含 "import" 或 "t="   │
│ → 强制 application/javascript (Vite 兼容) │
└──────────────┬───────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────┐
│ Tier 2: 开发代理 (仅 debug_assertions)     │
│ GET http://127.0.0.1:1420/{path}?{query} │
│                                          │
│ 命中 (HTTP 200): 返回 Vite 开发服务器内容   │
│ 失败: 进入 Tier 3                         │
│                                          │
│ 用途: pnpm dev:usb + adb reverse 场景     │
└──────────────┬───────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────┐
│ Tier 3: 兜底 index.html                   │
│ asset_resolver().get("index.html")       │
│                                          │
│ 命中: 返回 index.html (SPA fallback)      │
│ 未命中: 返回 404 诊断页 (含调试指引)        │
└──────────────────────────────────────────┘
```

**Tier 2 的重要性**：在开发模式下（`pnpm tauri android dev`），Vite 前端热更新服务器运行在 `127.0.0.1:1420`。通过 `adb reverse tcp:1420 tcp:1420` 将设备端口映射到 PC，本地服务器的开发代理使浮动窗口的 WebView 也能享受 HMR，无需每次修改前端代码都重新构建 APK。

### MIME 类型修正

Vite 的 `?import` 和动态 import 请求 URL 包含特殊 query 参数。`serve_asset()` 检测这些参数并强制设置 `Content-Type: application/javascript`，防止 WebView 因 MIME 类型不正确而拒绝执行 JS 模块。

## 7. 安全性

### 7.1 本地回环绑定

```rust
let addr = SocketAddr::from(([127, 0, 0, 1], 14202));
```

服务器**仅绑定 `127.0.0.1`**（IPv4 本地回环地址），不接受任何外部网络连接。这意味着：
- 同一设备上的其他应用可以连接（因为 Android 为单用户系统，应用进程隔离由 Android 沙箱保证）
- 局域网内的其他设备无法访问
- 不需要额外的防火墙或 TLS 配置

### 7.2 无身份验证

本地服务器不要求任何认证（无 token、无 API key 校验）。设计理由：
- 仅本地回环可访问，外部攻击者无法接触端口
- 服务器的唯一调用者是同一应用进程内的悬浮 WebView
- 所有业务鉴权（VCP API Key）由调用链末端的业务服务（`agent_chat_application_service`、`vcp_client`）负责

## 8. 与生命周期管理器的集成

`reconcile_local_server_cmd` 是 Tauri Command，注册在 `lib.rs` 中：

```
前端: invoke("reconcile_local_server_cmd", { enable: bool })
    │
    ▼
Rust: 检查全局 ServerHandle 状态
    │
    ├── enable == true && server 未运行:
    │     start_server(app_handle)
    │     保存 ServerHandle 到全局状态
    │
    ├── enable == false && server 运行中:
    │     server_handle.shutdown()
    │     清除全局状态
    │
    └── 状态已匹配: no-op
```

该命令在以下场景被调用：
1. **应用启动**：`bootstrap()` 中若 `enableAssistant == true`
2. **设置变更**：`AssistantSettingsSection` 中切换悬浮球开关
3. **应用 resume**：`vcp-lifecycle` resume 事件中恢复悬浮球时

## 9. 错误处理

| 场景 | 行为 |
|------|------|
| TCP 端口绑定失败 | 打印错误日志并返回，不 panic |
| WebSocket 消息解析失败 | 打印日志，`continue` 跳过该消息（不关闭连接） |
| AssistantChatPayload 反序列化失败 | 通过 WebSocket 发送 `{ type: "error", error: "..." }` |
| Agent 配置读取失败 | WebSocket 错误消息提示配置未就绪 |
| VCP 网络请求失败 | 通过流式通道发送 `{ type: "error" }` 事件 |
| 资产文件未找到 | 返回 404 诊断 HTML 页面（含 USB 调试指引，生产环境仅开发者可见） |
| 重复启动服务器 | `reconcile_local_server_cmd` 检查状态，避免重复绑定 |

## 11. 性能特征

| 指标 | 数值/策略 | 说明 |
|------|----------|------|
| 绑定地址 | 127.0.0.1:14202 | 本地回环，零网络延迟 |
| 并发连接数 | 无硬限制（单个浮动窗口仅需 1 条 WS 连接） | axum 异步 I/O，连接数不受线程数限制 |
| WebSocket 帧 | 文本帧 JSON 序列化 | 无二进制帧需求 |
| 资产服务 | 内存字节直接返回（Tier 1） | 零文件 I/O，构建产物已载入内存 |
| 开发代理 | HTTP GET reqwest 转发 | Tier 2 仅在 debug_assertions 激活 |
| 优雅关闭 | watch::channel 信号驱动 | 无 busy-wait，零 CPU 空转 |
