# 插件初始化与命令路由

## 1. 功能概述

`lib.rs` 是 `tauri-plugin-vcp-mobile` 的**唯一入口**。职责：

1. 定义插件共享状态 `VcpMobileState<R>`
2. 注册全部 7 条 Tauri 命令
3. 在 `setup` 钩子中完成 Android 原生插件注册，并将 `PluginHandle` 注入状态

> **设计原则**：`lib.rs` 只负责"路由与状态"，所有业务逻辑下沉到 `screen`、`stream`、`system` 三个子模块。

## 3. 共享状态详解

### 3.1 `VcpMobileState<R>`

```rust
pub struct VcpMobileState<R: Runtime> {
    pub active_streams: std::sync::Mutex<Vec<(String, u32)>>,
    #[cfg(target_os = "android")]
    pub plugin_handle: std::sync::Mutex<Option<tauri::plugin::PluginHandle<R>>>,
    #[cfg(not(target_os = "android"))]
    _marker: std::marker::PhantomData<fn() -> R>,
}
```

| 字段 | 类型 | 用途 | 访问方式 |
|------|------|------|----------|
| `active_streams` | `Mutex<Vec<(String, u32)>>` | 流式保活服务的引用计数列表 | `app.state::<VcpMobileState<R>>().active_streams.lock()` |
| `plugin_handle` | `Mutex<Option<PluginHandle<R>>>` | Android 原生插件句柄，用于 `run_mobile_plugin` | 同上，仅在 `target_os = "android"` 时存在 |

#### 为什么用 `std::sync::Mutex` 而非 `tokio::sync::Mutex`？

- 锁内操作极短（仅为 Vec 的 push/remove 或 Option 的 as_ref），不存在异步 IO。
- `std::sync::Mutex` 在同步上下文中性能更优，且 Tauri command handler 本身是同步调用（异步仅由 Tauri 框架在外层包裹）。

#### 桌面端兼容

非 Android 编译目标下，`plugin_handle` 字段被替换为 `_marker: PhantomData`，确保泛型参数 `R` 仍被使用，避免编译器报错。

## 4. 命令注册表

```rust
.invoke_handler(tauri::generate_handler![
    screen::set_keep_screen_on,
    screen::clear_keep_screen_on,
    stream::start_stream_service,
    stream::stop_stream_service,
    system::check_all_permissions,
    system::request_android_permission,
    system::move_task_to_back,
])
```

| 命令标识（Rust 函数名） | 前端调用命令 | 模块 | 说明 |
|------------------------|-------------|------|------|
| `set_keep_screen_on` | `plugin:vcp-mobile\|set_keep_screen_on` | `screen` | 设置屏幕常亮 |
| `clear_keep_screen_on` | `plugin:vcp-mobile\|clear_keep_screen_on` | `screen` | 清除屏幕常亮 |
| `start_stream_service` | `plugin:vcp-mobile\|start_stream_service` | `stream` | 启动流式保活（引用计数 +1） |
| `stop_stream_service` | `plugin:vcp-mobile\|stop_stream_service` | `stream` | 停止流式保活（引用计数 -1） |
| `check_all_permissions` | `plugin:vcp-mobile\|check_all_permissions` | `system` | 检查通知/存储/电池优化权限 |
| `request_android_permission` | `plugin:vcp-mobile\|request_android_permission` | `system` | 请求指定权限 |
| `move_task_to_back` | `plugin:vcp-mobile\|move_task_to_back` | `system` | 将应用移至后台 |

> **注意**：Tauri v2 中前端通过 `invoke('plugin:vcp-mobile|command_name')` 调用，命令名使用**下划线**（Rust 风格），但前端命令字符串中使用**原始函数名**（不是 camelCase）。详见 [07_Guest JS API](07_Guest JS API.md)。

## 5. Setup 流程

```rust
.setup(|app, _api| {
    #[cfg(target_os = "android")]
    let plugin_handle = _api.register_android_plugin("com.vcp.mobile", "VcpMobilePlugin")?;

    app.manage(VcpMobileState::<R> {
        active_streams: std::sync::Mutex::new(Vec::new()),
        #[cfg(target_os = "android")]
        plugin_handle: std::sync::Mutex::new(Some(plugin_handle)),
        #[cfg(not(target_os = "android"))]
        _marker: std::marker::PhantomData,
    });

    Ok(())
})
```

### 5.1 `register_android_plugin`

- **包名**：`"com.vcp.mobile"` —— 对应 Kotlin 侧的 package 声明。
- **类名**：`"VcpMobilePlugin"` —— 对应 Kotlin 侧的 `@TauriPlugin class VcpMobilePlugin`。
- **返回**：`PluginHandle<R>`，后续通过 `run_mobile_plugin(command, payload)` 调用 Kotlin 方法。

### 5.2 `app.manage()`

将 `VcpMobileState` 注入 Tauri 应用状态容器。任何 command handler 都可以通过 `app.state::<VcpMobileState<R>>()` 获取共享状态。

## 6. 数据流全景

```
前端 Vue/TS
    │ invoke('plugin:vcp-mobile|xxx')
    ▼
Tauri Runtime (IPC)
    │
    ▼
lib.rs :: invoke_handler
    │ 路由到 screen / stream / system
    ▼
具体 command handler
    │
    ├── screen: 直接操作 Window flags (Raw JNI)
    │
    ├── stream: 操作 active_streams → run_mobile_plugin
    │            │
    │            ▼
    │       Kotlin VcpMobilePlugin
    │            │
    │            ▼
    │       StreamKeepaliveService
    │
    └── system: run_mobile_plugin
                 │
                 ▼
            Kotlin VcpMobilePlugin
                 │
                 ├── checkAllPermissions → PowerManager + ContextCompat
                 ├── requestAndroidPermission → ActivityCompat.requestPermissions
                 └── moveTaskToBack → activity.moveTaskToBack
```

## 7. 关键约束

1. **`plugin_handle` 为 `Option`**：在 `setup` 之前或桌面端运行时可能为 `None`。业务代码中必须 `.ok_or("Plugin handle not initialized")?` 处理。
2. **命令扩展方式**：新增命令需要在 `invoke_handler` 数组和 `guest-js/index.ts` 中同步添加。
3. **与规范文档的交叉引用**：关于插件命令被静默拒绝的情况，参见 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §7.1。
