# 屏幕常亮控制

## 1. 功能概述

在长时间操作（同步、聊天流式输出、数据库重建/压缩/升级等）时，防止系统自动熄屏打断用户体验。该功能通过 **Raw JNI** 直接调用 Android `Window.addFlags(FLAG_KEEP_SCREEN_ON)` 和 `clearFlags(...)`，不经过 Kotlin 插件层。

前端通过 `useScreenKeeper` 全局引用计数抽象管理保活生命周期，避免并发场景下互相干扰（如同步未完成时发送聊天消息）。

> **设计决策**：仅涉及一次性的 Window flag 操作，使用 Raw JNI 比新增 Kotlin 方法更简洁。参见 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §6.1 三条铁律之第 3 条。

## 4. JNI 调用链拆解

```rust
let window = app
    .get_webview_window("main")
    .ok_or("main window not found")?;

window
    .as_ref()
    .with_webview(|webview| {
        webview.jni_handle().exec(move |env, activity, _webview| {
            // 1. 调用 activity.getWindow()
            let window = env
                .call_method(activity, "getWindow", "()Landroid/view/Window;", &[])
                .unwrap(); // 内部处理错误

            // 2. 调用 window.addFlags(FLAG_KEEP_SCREEN_ON)
            env.call_method(
                window.l().unwrap(),
                "addFlags", "(I)V",
                &[JValue::Int(FLAG_KEEP_SCREEN_ON)],
            )
        });
    })
    .map_err(|e| format!("with_webview failed: {:?}", e))?;
```

| 步骤 | JNI 签名 | 说明 |
|------|----------|------|
| `getWindow` | `()Landroid/view/Window;` | 从 Activity 获取 Window 对象 |
| `addFlags` / `clearFlags` | `(I)V` | 接收一个 int 参数，无返回值 |
| `FLAG_KEEP_SCREEN_ON` | `0x00000080` | Android SDK 常量，保持屏幕唤醒 |

## 5. 错误处理

| 错误来源 | 错误信息 | 触发条件 |
|----------|----------|----------|
| `get_webview_window` | `"main window not found"` | 名为 `"main"` 的 WebView 窗口不存在 |
| `with_webview` | `"with_webview failed: ..."` | JNI 执行环境异常 |
| `getWindow` | 日志输出 `[VcpMobilePlugin] getWindow failed` | JNI 调用 `getWindow` 返回错误 |
| `addFlags`/`clearFlags` | 日志输出 `[VcpMobilePlugin] addFlags failed: ...` | JNI 调用 addFlags 抛出异常 |

> **注意**：内部 JNI 调用错误仅通过 `log::error!` 记录，不中断 `Result` 返回。也就是说，即使 addFlags 失败，前端收到的仍是 `Ok(())`。这是已知设计（一次性标志操作，失败不影响核心功能）。

## 6. 前端抽象层：`useScreenKeeper`

为避免并发长时间操作互相干扰（例如同步未完成时发送聊天消息），前端使用全局引用计数抽象：

```typescript
// src/core/composables/useScreenKeeper.ts
let refCount = 0;

export function acquireScreenKeep(): void  // refCount++，首次触发 setKeepScreenOn
export function releaseScreenKeep(): void  // refCount--，归零时触发 clearKeepScreenOn
export async function withScreenKeep<T>(fn: () => Promise<T>): Promise<T>  // try/finally 包装
```

### 6.1 覆盖场景

| 场景 | 使用方式 | 开始位置 | 结束位置 |
|------|----------|----------|----------|
| **手动同步** | `acquire` / `release` | `syncSession.startSync()` | 完成/错误/关闭面板 |
| **聊天流式输出** | `acquire` / `release` | `chatHistoryStore.triggerGeneration()` | `chatStreamStore.removeSessionStream()` 全局流归零 |
| **聊天重生成** | `acquire` / `release` | `chatHistoryStore.regenerateResponse()` | `chatStreamStore.removeSessionStream()` 全局流归零 |
| **预渲染重建** | `withScreenKeep` | `rebuildSession.startRebuild()` | invoke 完成或抛错 |
| **内容压缩** | `withScreenKeep` | `rebuildSession.startRebuild()` | invoke 完成或抛错 |
| **数据库升级** | `withScreenKeep` | `rebuildSession.startRebuild()` | invoke 完成或抛错 |
| **附件 GC** | `withScreenKeep` | `MaintenanceSection.cleanupAttachments()` | invoke 完成或抛错 |
| **缓存清理** | `withScreenKeep` | `MaintenanceSection.clearSystemCache()` | invoke 完成或抛错 |

### 6.2 单 async 调用场景（`withScreenKeep`）

```typescript
import { withScreenKeep } from '@/core/composables/useScreenKeeper';

const startRebuild = async () => {
  // ...
  await withScreenKeep(() => invoke('rebuild_all_pre_renders'));
  // ...
};
```

### 6.3 跨异步回调场景（手动 `acquire` / `release`）

聊天流式的结束事件通过 Tauri Channel 异步回调到达，无法使用 `try/finally`：

```typescript
// chatHistoryStore.ts
const triggerGeneration = async () => {
  // ...
  streamStore.addSessionStream(agentId, topicId, thinkingId);
  acquireScreenKeep();  // 流注册后立即保活
  // ...
};

// chatStreamStore.ts
const removeSessionStream = (ownerId, topicId, messageId) => {
  // ...移除流...
  if (didRemove && Object.keys(sessionActiveStreams.value).length === 0) {
    releaseScreenKeep();  // 所有流结束后释放
  }
};
```

## 7. 与 Kotlin 侧的对照

虽然 `screen.rs` 使用 Raw JNI，但 Kotlin 侧 `VcpMobilePlugin.kt` 也保留了等效方法：

```kotlin
@Command
fun setKeepScreenOn(invoke: Invoke) {
    activity.window.addFlags(android.view.WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON)
    invoke.resolve()
}
```

**当前未使用**：Rust 侧直接走 Raw JNI，Kotlin 侧方法作为备用存在。如果未来需要从前端直接通过 `evaluateJavascript` 触发屏幕控制，可启用 Kotlin 侧方法。
