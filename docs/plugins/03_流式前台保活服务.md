# 流式前台保活服务

## 1. 功能概述

当 Agent 正在流式生成回复时，启动 Android **前台服务**（Foreground Service）并显示常驻通知，向系统声明"用户感知的重要任务"，显著降低进程被 OEM（ColorOS/EMUI/HarmonyOS/MIUI）杀后台的概率。

> **核心设计**：引用计数 + 通知内容动态更新。多 Agent 同时流式输出时，通知栏显示当前活跃 Agent 列表，全部结束后服务自毁，绝不空占资源。

### 3.2 计数规则

| 操作 | 逻辑 | 结果示例 |
|------|------|----------|
| `start_stream_service("A")` | Agent "A" 首次出现 → push ("A", 1) | `[("A", 1)]` |
| `start_stream_service("A")` | Agent "A" 已存在 → count += 1 | `[("A", 2)]` |
| `start_stream_service("B")` | Agent "B" 首次出现 → push ("B", 1) | `[("A", 2), ("B", 1)]` |
| `stop_stream_service("A")` | Agent "A" count > 1 → count -= 1 | `[("A", 1), ("B", 1)]` |
| `stop_stream_service("A")` | Agent "A" count == 1 → remove | `[("B", 1)]` |
| `stop_stream_service("B")` | 列表为空 → 调用 `stopStreamingService` | `[]` |

### 4.3 Rust 侧控制逻辑实现

#### `start_stream_service_inner`

```rust
pub fn start_stream_service_inner<R: Runtime>(
    app: &AppHandle<R>,
    agent_name: &str,
) -> Result<(), String> {
    let state = app.state::<VcpMobileState<R>>();
    let active_names = {
        let mut streams = state.active_streams.lock().map_err(|e| e.to_string())?;
        // 更新计数或添加新 Agent
        if let Some(pos) = streams.iter().position(|(name, _)| name == agent_name) {
            streams[pos].1 += 1;
        } else {
            streams.push((agent_name.to_string(), 1));
        }
        // 格式化名字列表：最多显示 3 个，超出截断
        let names: Vec<&str> = streams.iter().map(|(name, _)| name.as_str()).collect();
        if names.len() > 3 {
            format!("{}...", names[..3].join("、"))
        } else {
            names.join("、")
        }
    };
    #[cfg(target_os = "android")]
    {
        let handle = state.plugin_handle.lock().map_err(|e| e.to_string())?;
        let plugin_handle = handle.as_ref().ok_or("Plugin handle not initialized")?;
        plugin_handle
            .run_mobile_plugin::<serde_json::Value>(
                "startStreamingService",
                serde_json::json!({ "agentName": active_names }),
            )
            .map_err(|e| format!("run_mobile_plugin failed: {}", e))?;
    }
    log::info!(
        "[VcpMobilePlugin] Stream started for '{}'. Current notification: {}",
        agent_name, active_names
    );
    Ok(())
}
```

#### `stop_stream_service_inner`

```rust
pub fn stop_stream_service_inner<R: Runtime>(
    app: &AppHandle<R>,
    agent_name: &str,
) -> Result<(), String> {
    let state = app.state::<VcpMobileState<R>>();
    let (should_stop, active_names) = {
        let mut streams = state.active_streams.lock().map_err(|e| e.to_string())?;
        if let Some(pos) = streams.iter().position(|(name, _)| name == agent_name) {
            if streams[pos].1 > 1 {
                streams[pos].1 -= 1;
            } else {
                streams.remove(pos);
            }
        }
        let names: Vec<&str> = streams.iter().map(|(name, _)| name.as_str()).collect();
        let formatted = if names.len() > 3 {
            format!("{}...", names[..3].join("、"))
        } else {
            names.join("、")
        };
        (streams.is_empty(), formatted)
    };
    #[cfg(target_os = "android")]
    {
        let handle = state.plugin_handle.lock().map_err(|e| e.to_string())?;
        let plugin_handle = handle.as_ref().ok_or("Plugin handle not initialized")?;
        if should_stop {
            plugin_handle
                .run_mobile_plugin::<serde_json::Value>(
                    "stopStreamingService",
                    serde_json::json!({}),
                )
                .map_err(|e| format!("run_mobile_plugin failed: {}", e))?;
        } else {
            // 更新通知为剩余 Agent
            plugin_handle
                .run_mobile_plugin::<serde_json::Value>(
                    "startStreamingService",
                    serde_json::json!({ "agentName": active_names }),
                )
                .map_err(|e| format!("run_mobile_plugin failed: {}", e))?;
        }
    }
    log::info!(
        "[VcpMobilePlugin] Stream stopped for '{}'. Current notification: {}",
        agent_name, active_names
    );
    Ok(())
}
```

#### 错误处理策略（宽容策略）

Rust 侧函数返回 `Result<(), String>`，但**调用方不强制要求成功**。在 `agent_chat_application_service.rs` 与 `group_chat_application_service.rs` 中，调用模式为：

```rust
if let Err(e) = start_stream_service_inner(&app_handle, &agent_name) {
    println!("[AgentChatAppService] Failed to start streaming service: {}", e);
}
```

这体现了**宽容策略（Graceful Degradation）**：
- 前台服务启动失败**仅打印日志**，不会阻断后续 VCP 请求的发起。
- 前台服务停止失败同样仅打印日志，不影响聊天结果的回写与消息入库。
- Kotlin 侧在 `VcpMobilePlugin.startStreamingService` 中对 `SecurityException` 做二次兜底：若 `startForegroundService()` 因权限被禁而失败，自动降级为 `startService()`，避免应用崩溃。

## 5. Kotlin 侧实现

### 5.1 `VcpMobilePlugin.kt` 中的命令路由

```kotlin
@Command
fun startStreamingService(invoke: Invoke) {
    try {
        val args = invoke.parseArgs(StartStreamArgs::class.java)
        val intent = StreamKeepaliveService.createIntent(activity, args.agentName)
        try {
            if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
                activity.startForegroundService(intent)
            } else {
                activity.startService(intent)
            }
        } catch (e: SecurityException) {
            Log.w(TAG, "POST_NOTIFICATIONS permission denied, degrading to normal service", e)
            activity.startService(intent) // 静默降级为普通 Service，防止闪退崩溃
        }
        invoke.resolve()
    } catch (e: Exception) {
        Log.e(TAG, "startStreamingService failed", e)
        invoke.reject(e.message ?: "Unknown error")
    }
}

@Command
fun stopStreamingService(invoke: Invoke) {
    try {
        val intent = StreamKeepaliveService.createIntent(activity, "")
        activity.stopService(intent)
        invoke.resolve()
    } catch (e: Exception) {
        Log.e(TAG, "stopStreamingService failed", e)
        invoke.reject(e.message ?: "Unknown error")
    }
}
```

| 版本分支 | 行为 |
|----------|------|
| Android 8+ (API 26+) | `startForegroundService(intent)` —— 必须在 5 秒内调用 `startForeground()` |
| Android 7 及以下 | `startService(intent)` —— 传统启动方式 |

#### Android 14+ 前台服务类型与 `startForeground` 兜底

```kotlin
try {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.UPSIDE_DOWN_CAKE) {
        startForeground(
            NOTIFICATION_ID,
            notification,
            ServiceInfo.FOREGROUND_SERVICE_TYPE_REMOTE_MESSAGING
        )
    } else {
        startForeground(NOTIFICATION_ID, notification)
    }
} catch (e: Exception) {
    Log.e(TAG, "Failed to startForeground, falling back to basic background service", e)
}
```

| 元素 | 声明位置 | 作用 |
|------|---------|------|
| `FOREGROUND_SERVICE_TYPE_REMOTE_MESSAGING` | `AndroidManifest.xml` + Kotlin 代码 | Android 14+ 强制要求声明具体类型；即时通讯类后台消息同步适用此类型 |
| `android:foregroundServiceType="remoteMessaging"` | `AndroidManifest.xml` | 系统在安装时校验，缺失则前台服务无法启动 |
| `try-catch` | `onStartCommand()` 内 | 捕获 `ForegroundServiceStartNotAllowedException` 等异常，防止服务启动崩溃导致聊天中断 |

#### 服务生命周期与重启策略

- **`START_STICKY`**：服务被系统杀死后自动重启（但 Intent 会丢失，需等待下一次流式触发重新带入 Agent 名称）。**不选择 `START_NOT_STICKY`** 的原因是：流式输出期间进程被杀属于"用户正在关注但系统资源紧张"的场景，`START_STICKY` 可在资源恢复后自动重建服务骨架，降低通知掉失概率。
- **`onCreate()` → `createNotificationChannel()`**：服务首次创建时立即建立通知渠道，确保 `onStartCommand()` 中通知可被正确投递。
- **`stopWithTask="false"`**（`AndroidManifest.xml`）：当用户从最近任务列表划掉应用时，前台服务**不被**强制停止，继续保护流式输出进程。

### 5.3 Android 14+ 适配与厂商兼容性

#### Android 14 更严格的前台服务限制

Android 14（API 34）对前台服务引入了以下额外限制：
1. **必须声明具体 `foregroundServiceType`**：在 `AndroidManifest.xml` 的 `<service>` 标签中明确声明 `android:foregroundServiceType="remoteMessaging"`，同时在代码中传入 `ServiceInfo.FOREGROUND_SERVICE_TYPE_REMOTE_MESSAGING`。两者缺一不可。
2. **后台启动限制**：若应用处于后台（无可见 Activity），调用 `startForegroundService()` 可能抛出 `ForegroundServiceStartNotAllowedException`。当前实现通过 `try-catch` 捕获该异常并降级为普通后台服务，避免闪退。
3. **特殊场景白名单**：目标 Activity 需在前台或应用具有 `SYSTEM_ALERT_WINDOW` 权限等才能后台启动前台服务。由于 VCP Mobile 的流式服务总是在用户主动发起聊天后、应用仍在前台或可见窗口期间启动，通常不受此限制。

#### `RECEIVE_BOOT_COMPLETED` 是否需要

**不需要。** 当前设计哲学是"流结束立即自毁"，服务生命周期严格绑定于流式输出过程。设备重启后不存在"未完成的流式任务"，因此无需注册 `RECEIVE_BOOT_COMPLETED` 广播来重启服务。`AndroidManifest.xml` 中亦未声明此权限。

#### 特殊厂商电池优化白名单

国内主流 OEM 对后台前台服务的实际存活效果有显著差异：

| 厂商 | 行为特征 | 应对策略 |
|------|---------|----------|
| **小米 (MIUI/HyperOS)** | 严格限制非白名单应用的后台服务，即使 `IMPORTANCE_HIGH` + `START_STICKY` 也可能在锁屏后被杀 | 引导用户在系统设置中将应用加入"自启动" + "省电策略无限制"白名单 |
| **华为 (EMUI/HarmonyOS)** | 默认开启"电池优化"，前台服务存活率较低 | 利用 `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` 权限，在应用内引导用户关闭电池优化 |
| **OPPO (ColorOS)** | 对后台服务激进清理，对常驻通知容忍度相对较高 | 确保通知渠道为 `IMPORTANCE_HIGH`，并引导用户关闭电池优化 |

**`REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` 权限**已在 `AndroidManifest.xml` 中声明，并通过 `VcpMobilePlugin.kt` 的 `checkAllPermissions` / `requestAndroidPermission("battery")` 暴露给前端，用于检测和请求忽略电池优化。

## 7. 关键约束与血训

1. **必须在 5 秒内调用 `startForeground()`**：Android 8+ 上 `startForegroundService()` 后若未及时调用 `startForeground()`，系统会抛出 `RemoteServiceException: Context.startForegroundService() did not then call Service.startForeground()`。
   - 当前实现：在 `onStartCommand()` 中**立即**调用 `startForeground()`，无异步间隙。

2. **通知不可设为 `setSilent(true)`**：部分 ROM 会将静默通知归类为不显示。当前实现使用 `IMPORTANCE_HIGH` + `setOngoing(true)`，符合 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §5.3 通知规范。

3. **`POST_NOTIFICATIONS` 是前提**：Android 13+ 上，缺少此权限声明会导致通知渠道被系统禁用。参见 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §3.1 / §7.2。

4. **引用计数泄漏风险**：如果前端调用 `startStreamService` 后未在异常路径中调用 `stopStreamService`，服务将永远存活。前端代码中应确保在 `try/finally` 或组件卸载时停止服务。

5. **错误处理宽容性**：Rust 侧与 Kotlin 侧均采用"失败仅打印日志，不阻断聊天"的策略。`startForegroundService()` 失败时 Kotlin 会降级为普通 `Service`；Rust 侧调用方使用 `if let Err(e)` 捕获并忽略错误。这是有意为之——前台服务是"增强体验"而非"功能必需"，不应因通知权限缺失或厂商限制导致聊天中断。
