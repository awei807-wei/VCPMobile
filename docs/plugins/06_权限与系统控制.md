# 权限与系统控制

## 1. 功能概述

提供 Android 运行时权限的查询与请求能力，以及系统级控制命令（返回桌面）。涵盖通知、存储、麦克风、电池优化四项权限，并处理 Android 版本差异（Android 13+ 细分媒体权限 vs 传统存储权限，Android 14+ 新增部分授权语义）。

### microphone（requestPermissionForAlias）

v1.0.3 将麦克风权限请求从原生 `ActivityCompat.requestPermissions` 迁移回 Tauri 的 `requestPermissionForAlias("microphone")` 机制，与 `notification`、`storage`、`camera`、`location` 共享统一的 `@PermissionCallback` 回调通道：

```
前端: invoke('request_android_permission', { type: 'microphone' })
    │
    ▼
Kotlin: requestAndroidPermission → requestPermissionForAlias("microphone", invoke, "onPermissionResult")
    │
    ▼
系统弹出麦克风权限请求对话框
    │
    ▼
用户选择 允许/拒绝
    │
    ▼
@PermissionCallback onPermissionResult(invoke)
    │
    ├── emitPermissionsToWebView()
    │   ├── 查询全部 7 项权限最新状态
    │   └── evaluateJavascript("window.dispatchEvent(new CustomEvent('vcp-permission-change', { detail: {...} }))")
    │
    ▼
前端状态实时刷新
```

**Manifest 声明**：
```xml
<uses-permission android:name="android.permission.RECORD_AUDIO" />
<uses-permission android:name="android.permission.MODIFY_AUDIO_SETTINGS" />
```

`MODIFY_AUDIO_SETTINGS` 用于语音交互模式中的音频路由控制（如扬声器/听筒切换）。

## 3. 权限检查

### 3.1 接口

```rust

pub struct PermissionStatus {
    pub notification: bool,
    pub storage: bool,
    pub battery: bool,
    pub microphone: bool,
    pub camera: bool,
    pub overlay: bool,
    pub location: bool,
}

#[tauri::command]
pub fn check_all_permissions<R: Runtime>(app: AppHandle<R>) -> Result<PermissionStatus, String>;
```

| 属性 | 说明 |
|------|------|
| 前端命令 | `plugin:vcp-mobile\|check_all_permissions` |
| 参数 | 无 |
| 返回值 | `PermissionStatus` JSON 对象（含 7 个字段） |
| 桌面端行为 | 始终返回所有字段为 `true` |

**新增字段说明（v1.0.3）**：

| 字段 | 含义 | 检查方式 |
|------|------|----------|
| `camera` | 相机权限 | `ContextCompat.checkSelfPermission(CAMERA)` |
| `overlay` | 悬浮窗权限 | `Settings.canDrawOverlays()` 系统设置查询 |
| `location` | 定位权限 | `ACCESS_FINE_LOCATION` 或 `ACCESS_COARSE_LOCATION` 任一授予即视为已授权 |

### 3.2 Kotlin 实现

```kotlin
@Command
fun checkAllPermissions(invoke: Invoke) {
    val pm = activity.getSystemService(Context.POWER_SERVICE) as PowerManager

    val notificationGranted = if (Build.VERSION.SDK_INT >= 33) {
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.POST_NOTIFICATIONS) == PackageManager.PERMISSION_GRANTED
    } else {
        true
    }

    val storageGranted = if (Build.VERSION.SDK_INT >= 34) {
        val hasAll = ContextCompat.checkSelfPermission(activity, android.Manifest.permission.READ_MEDIA_IMAGES) == PackageManager.PERMISSION_GRANTED
        val hasVisualSelected = ContextCompat.checkSelfPermission(activity, "android.permission.READ_MEDIA_VISUAL_USER_SELECTED") == PackageManager.PERMISSION_GRANTED
        hasAll || hasVisualSelected
    } else if (Build.VERSION.SDK_INT >= 33) {
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.READ_MEDIA_IMAGES) == PackageManager.PERMISSION_GRANTED
    } else {
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.READ_EXTERNAL_STORAGE) == PackageManager.PERMISSION_GRANTED
    }

    val microphoneGranted = ContextCompat.checkSelfPermission(activity, android.Manifest.permission.RECORD_AUDIO) == PackageManager.PERMISSION_GRANTED
    val cameraGranted = ContextCompat.checkSelfPermission(activity, android.Manifest.permission.CAMERA) == PackageManager.PERMISSION_GRANTED
    val locationGranted = ContextCompat.checkSelfPermission(activity, android.Manifest.permission.ACCESS_FINE_LOCATION) == PackageManager.PERMISSION_GRANTED ||
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.ACCESS_COARSE_LOCATION) == PackageManager.PERMISSION_GRANTED

    val batteryOptimizationIgnored = pm.isIgnoringBatteryOptimizations(activity.packageName)
    val overlayGranted = floatingWindowManager.hasOverlayPermission()

    val result = JSObject()
    result.put("notification", notificationGranted)
    result.put("storage", storageGranted)
    result.put("microphone", microphoneGranted)
    result.put("camera", cameraGranted)
    result.put("location", locationGranted)
    result.put("battery", batteryOptimizationIgnored)
    result.put("overlay", overlayGranted)
    invoke.resolve(result)
}
```

#### 版本分支详解

| 权限 | Android 14+ (API 34+) | Android 13 (API 33) | Android 12 及以下 |
|------|----------------------|---------------------|-------------------|
| **通知** | `POST_NOTIFICATIONS` 运行时检查 | `POST_NOTIFICATIONS` 运行时检查 | 无需运行时权限，始终 `true` |
| **存储** | `READ_MEDIA_IMAGES` **或** `READ_MEDIA_VISUAL_USER_SELECTED` | `READ_MEDIA_IMAGES` 细分权限 | `READ_EXTERNAL_STORAGE` 传统权限 |
| **电池优化** | `PowerManager.isIgnoringBatteryOptimizations()` | 同上 | 同上，API 无版本差异 |

> **Android 14 存储权限变化**：API 34 引入 `READ_MEDIA_VISUAL_USER_SELECTED`，允许用户仅授权部分图片/视频。`checkAllPermissions` 中同时检查 `READ_MEDIA_IMAGES` 和 `READ_MEDIA_VISUAL_USER_SELECTED`，任一满足即视为已授权。

> **注意**：电池优化检查**不是**标准的 Android 运行时权限模型，而是通过 `PowerManager` API 查询应用是否被列入系统电池优化白名单。

### 4.2 Kotlin 实现

```kotlin
@TauriPlugin(permissions = [
    Permission(strings = ["android.permission.POST_NOTIFICATIONS"], alias = "notification"),
    Permission(strings = ["android.permission.READ_MEDIA_IMAGES"], alias = "storage"),
    Permission(strings = ["android.permission.READ_EXTERNAL_STORAGE", "android.permission.WRITE_EXTERNAL_STORAGE"], alias = "storageLegacy"),
    Permission(strings = ["android.permission.RECORD_AUDIO"], alias = "microphone"),
    Permission(strings = ["android.permission.CAMERA"], alias = "camera"),
    Permission(strings = ["android.permission.ACCESS_FINE_LOCATION", "android.permission.ACCESS_COARSE_LOCATION"], alias = "location")
])
class VcpMobilePlugin(private val activity: Activity) : Plugin(activity) {
    // ...
}
```

`@TauriPlugin` 注解中通过 `@Permission` 定义了**五个权限别名**（v1.0.3 新增 `camera` 和 `location`），将 Android 权限字符串映射为易用的别名：

| 别名 | Android 权限字符串 | 适用版本 |
|------|-------------------|----------|
| `notification` | `POST_NOTIFICATIONS` | Android 13+ |
| `storage` | `READ_MEDIA_IMAGES` | Android 13+ |
| `storageLegacy` | `READ_EXTERNAL_STORAGE` + `WRITE_EXTERNAL_STORAGE` | Android 12 及以下 |
| `camera` | `CAMERA` | 全部版本 |
| `location` | `ACCESS_FINE_LOCATION` + `ACCESS_COARSE_LOCATION` | 全部版本 |

> **注意**：`READ_MEDIA_VISUAL_USER_SELECTED`（Android 14+）不在 `@TauriPlugin` 注解中声明，因为 `requestPermissionForAlias` 机制无法处理「部分授权」语义。`checkAllPermissions` 中通过 `ContextCompat.checkSelfPermission` 手动检测该权限。

#### 权限请求命令

```kotlin
@Command
fun requestAndroidPermission(invoke: Invoke) {
    val args = invoke.parseArgs(RequestPermissionArgs::class.java)
    when (args.type) {
        "notification" -> {
            if (Build.VERSION.SDK_INT >= 33) {
                requestPermissionForAlias("notification", invoke, "onPermissionResult")
            } else {
                emitPermissionsToWebView()
                invoke.resolve()
            }
        }
        "storage" -> {
            if (Build.VERSION.SDK_INT >= 33) {
                requestPermissionForAlias("storage", invoke, "onPermissionResult")
            } else {
                requestPermissionForAlias("storageLegacy", invoke, "onPermissionResult")
            }
        }
        "microphone" -> {
            requestPermissions(arrayOf(Manifest.permission.RECORD_AUDIO), 999)
            // 结果通过 onRequestPermissionsResult → emitPermissionsToWebView()
        }
        "battery" -> {
            try {
                val intent = Intent(Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS).apply {
                    data = Uri.parse("package:${activity.packageName}")
                }
                startActivityForResult(invoke, intent, "onBatteryOptimizationResult")
            } catch (e: Exception) {
                val intent = Intent(Settings.ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS)
                startActivityForResult(invoke, intent, "onBatteryOptimizationResult")
            }
        }
    }
}
```

| 权限类型 | 请求机制 | 回调 |
|----------|----------|------|
| `notification` | `requestPermissionForAlias("notification", invoke, "onPermissionResult")` | `@PermissionCallback` |
| `storage` | `requestPermissionForAlias("storage"/"storageLegacy", invoke, "onPermissionResult")` | `@PermissionCallback` |
| `microphone` | `requestPermissionForAlias("microphone", invoke, "onPermissionResult")` | `@PermissionCallback` |
| `camera` | `requestPermissionForAlias("camera", invoke, "onPermissionResult")` | `@PermissionCallback` |
| `location` | `requestPermissionForAlias("location", invoke, "onPermissionResult")` | `@PermissionCallback` |
| `battery` | `startActivityForResult(invoke, intent, "onBatteryOptimizationResult")` | `@ActivityCallback` |

#### 回调与事件推送

```kotlin
@PermissionCallback
fun onPermissionResult(invoke: Invoke) {
    emitPermissionsToWebView()
    invoke.resolve()
}

@ActivityCallback
fun onBatteryOptimizationResult(invoke: Invoke, result: ActivityResult) {
    emitPermissionsToWebView()
    invoke.resolve()
}

private fun emitPermissionsToWebView() {
    val pm = activity.getSystemService(Context.POWER_SERVICE) as PowerManager

    val notificationGranted = if (Build.VERSION.SDK_INT >= 33) {
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.POST_NOTIFICATIONS) == PackageManager.PERMISSION_GRANTED
    } else {
        true
    }

    val storageGranted = if (Build.VERSION.SDK_INT >= 34) {
        val hasAll = ContextCompat.checkSelfPermission(activity, android.Manifest.permission.READ_MEDIA_IMAGES) == PackageManager.PERMISSION_GRANTED
        val hasVisualSelected = ContextCompat.checkSelfPermission(activity, "android.permission.READ_MEDIA_VISUAL_USER_SELECTED") == PackageManager.PERMISSION_GRANTED
        hasAll || hasVisualSelected
    } else if (Build.VERSION.SDK_INT >= 33) {
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.READ_MEDIA_IMAGES) == PackageManager.PERMISSION_GRANTED
    } else {
        ContextCompat.checkSelfPermission(activity, android.Manifest.permission.READ_EXTERNAL_STORAGE) == PackageManager.PERMISSION_GRANTED
    }

    val microphoneGranted = ContextCompat.checkSelfPermission(activity, android.Manifest.permission.RECORD_AUDIO) == PackageManager.PERMISSION_GRANTED

    val batteryOptimizationIgnored = pm.isIgnoringBatteryOptimizations(activity.packageName)

    val json = """{"notification":$notificationGranted,"storage":$storageGranted,"microphone":$microphoneGranted,"battery":$batteryOptimizationIgnored}"""
    val script = "window.dispatchEvent(new CustomEvent('vcp-permission-change', { detail: $json }))"
    webViewRef?.evaluateJavascript(script, null)
}
```

- **`requestPermissionForAlias`**：调用 Tauri 框架封装的 `registerForActivityResult(ActivityResultContracts.RequestMultiplePermissions())`，生命周期安全，无需手动处理 `onRequestPermissionsResult`。
- **`startActivityForResult`**：同理使用 Tauri 框架的 `ActivityResultLauncher`，用户从电池优化设置页返回后触发 `@ActivityCallback`。
- **`emitPermissionsToWebView`**：回调内重新查询 notification / storage / microphone / battery 四项权限状态，通过 `evaluateJavascript` 注入 `vcp-permission-change` CustomEvent，前端 `window.addEventListener` 即可实时接收。

> **注意**：电池优化请求返回的 `ActivityResult.resultCode` 因 ROM 而异（部分系统对话框不返回标准 `RESULT_OK`），因此回调内**不依赖 resultCode**，始终执行 `checkAllPermissions` 逻辑并以实际查询结果为准。

### 4.3 BatteryStatusManager

`BatteryStatusManager.kt` 是独立的设备电量与系统省电状态检测模块，与权限系统共用 `PowerManager` 但职责分离：前者负责**检测**，后者负责**白名单状态查询与请求**。

#### 接口

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct BatteryStatus {
    pub level: i32,
    pub is_power_save_mode: bool,
}

#[tauri::command]
pub fn get_battery_status<R: Runtime>(app: AppHandle<R>) -> Result<BatteryStatus, String>;
```

#### 6 层 OEM 省电模式检测逻辑

```kotlin
class BatteryStatusManager(private val context: Context) {
    fun isPowerSaveMode(): Boolean {
        // 1. 原生 PowerManager 状态检测
        val powerManager = context.getSystemService(Context.POWER_SERVICE) as PowerManager
        if (powerManager.isPowerSaveMode) return true

        val resolver = context.contentResolver

        // 2. 原生 Android 广播全局低电量标记
        if (android.provider.Settings.Global.getInt(resolver, "low_power", 0) == 1) return true

        // 3. 通用系统省电标记
        if (android.provider.Settings.System.getInt(resolver, "power_save_mode", 0) == 1) return true

        // 4. 小米 (MIUI / HyperOS) 专用标记
        if (android.provider.Settings.System.getInt(resolver, "POWER_SAVE_MODE_OPEN", 0) == 1) return true

        // 5. 华为 (EMUI / HarmonyOS) 专用标记
        // SmartModeStatus: 1=正常, 2=省电, 3=超级省电, 4=性能
        val hwPowerSave = android.provider.Settings.System.getInt(resolver, "SmartModeStatus", 1)
        if (hwPowerSave == 2 || hwPowerSave == 3) return true

        // 6. Vivo / Oppo 等其他定制 ROM Global 标记
        if (android.provider.Settings.Global.getInt(resolver, "power_save_mode", 0) == 1) return true

        return false
    }
}
```

| 层级 | 检测目标 | 数据来源 | 说明 |
|------|---------|----------|------|
| 1 | 原生 Android | `PowerManager.isPowerSaveMode()` | AOSP 标准 API，Google / 三星 / 索尼等原生或类原生系统 |
| 2 | 全局低电量广播 | `Settings.Global.low_power` | Android 全局低电量模式标记 |
| 3 | 通用系统省电 | `Settings.System.power_save_mode` | 部分国产 ROM 复用的通用键 |
| 4 | 小米 MIUI/HyperOS | `Settings.System.POWER_SAVE_MODE_OPEN` | MIUI 私有 Settings 键，HyperOS 兼容 |
| 5 | 华为 EMUI/HarmonyOS | `Settings.System.SmartModeStatus` | 华为四态电源管理：1=正常, 2=省电, 3=超级省电, 4=性能 |
| 6 | Vivo/OPPO 等 | `Settings.Global.power_save_mode` | 部分 ROM 将省电标记放在 Global 命名空间 |

#### 与前端 `BatteryOptimizationGuide.vue` 的联动

前端通过 `get_battery_status` 获取 `isPowerSaveMode`，但当前 UI 层面并不直接阻断用户操作，而是作为**引导信息**展示：

- `PermissionGate.vue`（引导页 Step 2）提示用户「品牌电池白名单设置」，告知国产 ROM 有独立的电池管理策略。
- `BatteryOptimizationGuide.vue`（全局设置 → 电池优化）提供 5 大品牌的详细白名单配置步骤：OPPO/一加/realme（ColorOS）、小米/Redmi（HyperOS）、华为/荣耀（HarmonyOS）、vivo/iQOO（OriginOS）、三星（One UI）。

#### Kotlin 实现

```kotlin
@Command
fun openFile(invoke: Invoke) {
    val args = invoke.parseArgs(OpenFileArgs::class.java)
    val path = args.path
    
    fileIoExecutor.execute {
        val file = java.io.File(path)
        val ext = file.extension.lowercase()
        val mimeType = MimeTypeMap.getSingleton().getMimeTypeFromExtension(ext) ?: "*/*"

        // 借助 FileProvider 生成临时读取授权的 content:// URI
        val uri = try {
            FileProvider.getUriForFile(context, "${context.packageName}.fileprovider", file)
        } catch (e: Exception) {
            FileProvider.getUriForFile(context, "${context.packageName}.opener.fileprovider", file)
        }

        val intent = Intent(Intent.ACTION_VIEW).apply {
            setDataAndType(uri, mimeType)
            addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)
            addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
        }

        context.startActivity(intent)
    }
}
```

- **FileProvider**：生成 `content://` 临时授权 URI，避免直接暴露 `file://` URI（Android 7.0+ 禁止）。
- **双 authority 回退**：优先使用 `.fileprovider`，失败时回退到 `.opener.fileprovider`。
- **权限要求**：`openFile` 本身不需要运行时权限，因为 FileProvider 分享的是应用私有目录下的文件（如 `cacheDir`）。但应用 Manifest 中必须声明 `FileProvider`：
  ```xml
  <provider
      android:name="androidx.core.content.FileProvider"
      android:authorities="${applicationId}.fileprovider"
      android:exported="false"
      android:grantUriPermissions="true">
      <meta-data android:name="android.support.FILE_PROVIDER_PATHS" android:resource="@xml/file_paths" />
  </provider>
  ```

#### 与 `08_文件选择与缩略图生成.md` 的关系

| 功能 | 对应命令 | 职责 |
|------|---------|------|
| **选择文件** | `pick_file` → `pickFile` | 唤起 `Intent.ACTION_GET_CONTENT`，用户从系统文件管理器中选择文件，返回后拷贝至 `cacheDir` 并生成缩略图 |

两者使用相同的 `fileIoExecutor` 单线程池避免并发冲突，但生命周期完全不同：`pickFile` 是「摄入」，`openFile` 是「输出」。

## 7. 关键约束

1. **`p_type` 字符串硬编码**：前端传入 `"notification"` / `"storage"` / `"microphone"` / `"battery"`，拼写错误会导致 Kotlin `when` 分支未命中，静默无操作。
   - **防范**：前端封装为枚举或常量，如 `PermissionType.NOTIFICATION`。

2. **事件通道依赖 `webViewRef`**：`emitPermissionsToWebView()` 需要在 `load(webView)` 时保存 WebView 引用。若插件加载失败或 WebView 未初始化，事件将静默丢失（前端仍可通过 `visibilitychange` 和兜底轮询恢复）。

3. **权限请求的回调通道分为两类**：
   - `notification` / `storage` / `microphone` / `camera` / `location` 使用 `@PermissionCallback`（Tauri 框架的 `ActivityResultLauncher`），共享 `onPermissionResult` 回调。
   - `battery` 使用 `@ActivityCallback`（Tauri 框架的 `startActivityForResult`），触发 `onBatteryOptimizationResult` 回调。
   - 两种回调触发路径不同，但统一收敛到 `emitPermissionsToWebView()` 推送 `vcp-permission-change` 事件（含全部 7 项权限最新状态）。

4. **`open_file_native` 的权限边界**：
   - 该函数通过 `FileProvider` 分享应用私有文件，**不需要** `READ_EXTERNAL_STORAGE` 或 `MANAGE_EXTERNAL_STORAGE` 运行时权限。
   - 但如果试图打开的文件位于应用私有目录之外（如用户通过 `pick_file` 选择后未拷贝到 `cacheDir` 的文件），`FileProvider` 会抛出 `IllegalArgumentException`，此时会回退到第二个 authority 尝试。
   - 参见 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §3.1 关于 `FileProvider` 的路径配置规范。

5. **与规范文档的交叉引用**：
   - `POST_NOTIFICATIONS` 血训：参见 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §3.1 / §7.2。
   - Tauri 权限回调机制官方文档：参见 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §5.2。

6. **传感器采集生命周期**：
   - `start_sensor_collection` 在 `DistributedNode::start()` 成功后由 Rust 端自动调用，**不由前端直接触发**。
   - `stop_sensor_collection` 在 `DistributedNode::stop()` 时由 Rust 端自动调用，确保传感器随节点生命周期同步启停。
   - 前端仅通过 `get_sensor_data` 被动读取缓存值，不对传感器进行启停控制。

7. **定位权限检查边界**：
   - `SensorStatusManager.startLocationListening()` 在启动前检查 `ACCESS_FINE_LOCATION` 和 `ACCESS_COARSE_LOCATION`。**两项均未授予时，定位采集静默降级**（缓存值设为"未获得定位权限"），不阻塞运动和环境的采集。
   - 定位权限请求**不在** `SensorStatusManager` 内部触发，而是由上层调用方（如 `DistributedSettingsSection`）在连接前通过 `request_android_permission({ type: 'notification' })` 等现有权限通道统一处理。

## 8. SensorStatusManager —— Android 原生传感器采集

### 8.1 架构概览

`SensorStatusManager` 是 v1.0.3 新增的 Android 原生传感器采集管理器，替代了此前基于前端 Web API（`navigator.geolocation`、`devicemotion`、`AmbientLightSensor`）的传感器方案。它是一个自包含的单例式类，由 `VcpMobilePlugin.kt` 持有引用，通过 Plugin IPC 通道向 Rust 工具层和前端提供传感器遥测数据。

### 8.2 三级传感器采集体系

`SensorStatusManager` 管理三种物理传感器/定位源，各自采用不同的采集策略以适应功耗、时效性和准确性的需求：


#### 8.2.1 定位采集（Location）

```kotlin
private fun startLocationListening() {
    val hasFine = ContextCompat.checkSelfPermission(
        context, android.Manifest.permission.ACCESS_FINE_LOCATION
    ) == PackageManager.PERMISSION_GRANTED
    val hasCoarse = ContextCompat.checkSelfPermission(
        context, android.Manifest.permission.ACCESS_COARSE_LOCATION
    ) == PackageManager.PERMISSION_GRANTED

    if (!hasFine && !hasCoarse) {
        latestLocationStr = "位置信息: 未获得定位权限"
        return  // 静默降级，不阻塞运动与环境采集
    }

    // Network Provider（网络定位：基站/WiFi）
    if (locationManager.isProviderEnabled(LocationManager.NETWORK_PROVIDER)) {
        locationManager.requestLocationUpdates(
            LocationManager.NETWORK_PROVIDER,
            120000L,  // 最小更新间隔 120 秒
            10f,      // 最小位移变化 10 米
            locationListener,
            Looper.getMainLooper()
        )
        getLastKnownLocation(LocationManager.NETWORK_PROVIDER)  // 立即获取已知位置
    }

    // GPS Provider（卫星定位）
    if (locationManager.isProviderEnabled(LocationManager.GPS_PROVIDER)) {
        locationManager.requestLocationUpdates(
            LocationManager.GPS_PROVIDER,
            120000L,  // 最小更新间隔 120 秒
            10f,      // 最小位移变化 10 米
            locationListener,
            Looper.getMainLooper()
        )
        getLastKnownLocation(LocationManager.GPS_PROVIDER)  // 立即获取已知位置
    }
}
```

**双 Provider 策略**：

| Provider | 定位源 | 精度 | 功耗 | 适用场景 |
|----------|--------|------|------|----------|
| `NETWORK_PROVIDER` | 基站 + WiFi 指纹 | 50m--500m | 低 | 室内、快速首次定位 |
| `GPS_PROVIDER` | GPS 卫星 | 3m--30m | 高 | 室外精确定位 |

- 两个 Provider **并行注册**，任一有新数据即更新缓存。
- 启动时调用 `getLastKnownLocation()` 获取系统缓存的最近位置，避免等待下一个回调周期。
- 定位结果格式化：`坐标: xx.xxxx°N/S, xx.xxxx°E/W | 精度: 15m | 海拔: 50m`

#### 8.2.2 运动突发采样（Motion Burst Sampling）

运动传感器（加速度计）采用**突发采样（Burst Sampling）**策略以最小化功耗：

```
┌────────────────────────────────────────────────────────────┐
│  一周期 = 30 秒                                             │
│                                                            │
│  ┌──────────┐                                              │
│  │ 活跃 2s  │───────────────────────────────────────        │
│  │ 10Hz 采 │                  休眠 28s                      │
│  │ 样(~20  │                (传感器关闭)                     │
│  │ 个样本) │                                              │
│  └──────────┘                                              │
│  t=0       t=2s                                t=30s       │
│                                                            │
│  活跃期: SensorManager.registerListener(accel, 100ms)       │
│  休眠期: SensorManager.unregisterListener(accel)            │
│  过渡:   mainHandler.postDelayed(nextBurst, 28000)          │
└────────────────────────────────────────────────────────────┘
```

**关键参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| `BURST_ACTIVE_DURATION` | 2000ms | 每周期采样窗口长度 |
| `BURST_SLEEP_DURATION` | 28000ms | 每周期间隔休眠长度 |
| `SAMPLING_PERIOD_US` | 100000us (100ms) | 活跃期内采样间隔，等价 10Hz |
| 内联限流 | `now - lastSampleTime < 100ms --> return` | 双重保障，防止回调过于频繁 |

**运动状态分类**（基于窗口内加速度幅度的平均值）：

| 平均加速度 `avg` | 状态 | 阈值设计依据 |
|-----------------|------|-------------|
| `avg <= 9.5` | 静止 | 静止设备仅受重力（~9.8 m/s²）影响 |
| `9.5 < avg <= 10.5` | 轻微移动 | 设备轻微晃动 |
| `10.5 < avg <= 12.0` | 步行中 | 典型步行步态加速度 |
| `avg > 12.0` | 运动中 | 跑步或交通工具震动 |

**后处理输出格式**：`状态: <分类> | 平均加速度: xx.xx m/s² | 峰值: xx.xx m/s²`

**线程安全**：样本缓冲 `burstSamples` 的读写使用 `synchronized(burstSamples)` 保护，始终在主线程 (`mainHandler`) 上调度启停，避免与 `SensorEventListener.onSensorChanged()` 的回调线程竞态。

#### 8.2.3 环境持续监听（Ambient Continuous Listening）

环境光传感器（`TYPE_LIGHT`）和气压传感器（`TYPE_PRESSURE`）以**持续低频率模式**监听，功耗极低：

```kotlin
// 持续监听，不启用突发/休眠模式
if (lightSensor != null) {
    sensorManager.registerListener(ambientListener, lightSensor, SensorManager.SENSOR_DELAY_NORMAL)
}
if (pressureSensor != null) {
    sensorManager.registerListener(ambientListener, pressureSensor, SensorManager.SENSOR_DELAY_NORMAL)
}
```

- **`SENSOR_DELAY_NORMAL`**：~200ms 采样间隔（约 5Hz），Android 默认速率。环境传感器（光/气压）变化缓慢，此速率完全满足需求。
- 两个传感器共享同一个 `ambientListener` 实例，在 `onSensorChanged` 中通过 `event.sensor.type` 区分数据来源。
- 传感器数据变化时自动回调 `updateAmbientString()`，拼接当前的光照和气压描述。

**环境光描述语义**：

| 光照强度范围 (lux) | 描述 | 典型场景 |
|-------------------|------|----------|
| 0--49 | 暗 | 夜间、关闭灯光 |
| 50--199 | 室内 | 办公室、客厅照明 |
| 200--999 | 明亮 | 靠近窗户、商场 |
| >= 1000 | 户外 | 日光直射或强光环境 |

**输出格式**：`环境光: 150 lux (室内) | 气压: 1013 hPa`

### 8.3 线程安全与缓存机制

#### @Volatile 缓存字段

三个输出字符串使用 `@Volatile` 注解：

```kotlin
@Volatile private var latestLocationStr = "位置信息: 等待数据采集..."
@Volatile private var latestMotionStr = "运动状态: 静止"
@Volatile private var latestAmbientStr = "环境传感器: 设备不支持或权限未授予"
```

- **`@Volatile`** 确保一个线程写入后，其他线程立即看到最新值（JVM happens-before 语义）。
- 写入仅发生在各自的监听器回调线程（Android Sensor 框架的传感器事件线程或主线程）。
- 读取仅发生在 `getSensorData()` 方法中，该方法由 Plugin IPC 在工作线程调用。
- 这种单写-单读模式配合 `@Volatile` 保证了无需加锁的线程安全。

#### getSensorData() 方法

```kotlin
fun getSensorData(type: String): JSObject {
    val obj = JSObject()
    when (type) {
        "location" -> obj.put("value", latestLocationStr)
        "motion"   -> obj.put("value", latestMotionStr)
        "ambient"  -> obj.put("value", latestAmbientStr)
        "all"      -> {
            obj.put("location", latestLocationStr)
            obj.put("motion", latestMotionStr)
            obj.put("ambient", latestAmbientStr)
        }
    }
    return obj
}
```

| type 参数 | 返回值结构 | 使用方 |
|-----------|-----------|--------|
| `"location"` | `{ value: "坐标: ..." }` | `location.rs::read_current()` |
| `"motion"` | `{ value: "状态: ..." }` | `motion_sensor.rs::read_current()` |
| `"ambient"` | `{ value: "环境光: ..." }` | `ambient_sensor.rs::read_current()` |
| `"all"` | `{ location, motion, ambient }` | `DistributedView.vue`（前端遥测面板） |

- **零开销读取**：直接返回 `@Volatile` 字段的当前值，无 I/O、无阻塞、无额外系统调用。
- **即时返回**：Rust 工具层的 `read_current()` 以 30 秒周期调用此方法，每次调用的响应时间 < 1ms。

### 8.4 生命周期管理

```
┌─────────────────────────────────────────────────────────────┐
│  DistributedNode::start()  (Rust)                           │
│      │                                                      │
│      ▼                                                      │
│  system.rs::start_sensor_collection()                       │
│      │                                                      │

│  VcpMobilePlugin.kt::startSensorCollection(invoke)          │
```

## 10. 定位权限（ACCESS_FINE_LOCATION / ACCESS_COARSE_LOCATION）

### 10.1 Manifest 声明

v1.0.3 在 `AndroidManifest.xml` 中新增了两项定位权限声明：

```xml
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
```

| 权限 | 定位精度 | 数据来源 | 调用 API |
|------|---------|----------|----------|
| `ACCESS_FINE_LOCATION` | ~3m--30m | GPS 卫星 + 精确 WiFi | `LocationManager.requestLocationUpdates(GPS_PROVIDER)` |
| `ACCESS_COARSE_LOCATION` | ~50m--500m | 基站 + WiFi 指纹 | `LocationManager.requestLocationUpdates(NETWORK_PROVIDER)` |

**权限关系**：`ACCESS_FINE_LOCATION` **包含** `ACCESS_COARSE_LOCATION` 的全部能力。若用户仅授予 `ACCESS_COARSE_LOCATION`，GPS Provider 仍然无法使用，仅 Network Provider 可用。

### 10.2 权限检查流程

```
SensorStatusManager.start()
    │
    ▼
startLocationListening()
    │
    ├── checkSelfPermission(ACCESS_FINE_LOCATION) → hasFine: Boolean
    ├── checkSelfPermission(ACCESS_COARSE_LOCATION) → hasCoarse: Boolean
    │
    ├── !hasFine && !hasCoarse ?
    │   ├── YES → latestLocationStr = "位置信息: 未获得定位权限"
    │   │         Log.w(TAG, "Location permissions not granted.")
    │   │         return (静默降级，继续启动运动和环境的采集)
    │   │
    │   └── NO → 继续注册 Provider
    │            ├── Network Provider: requestLocationUpdates(120s, 10m)
    │            └── GPS Provider: requestLocationUpdates(120s, 10m)
    │
    ▼
运动和环境的采集正常启动（不依赖定位权限结果）
```

### 10.3 降级语义对照表

| 权限状态 | hasFine | hasCoarse | 定位行为 | latestLocationStr 内容 |
|----------|---------|-----------|----------|----------------------|
| 两项均未授予 | false | false | 不注册任何 Provider | `"位置信息: 未获得定位权限"` |
| 仅粗略位置 | false | true | Network Provider 工作，GPS 跳过 | 格式化坐标（精度较低） |
| 精确位置已授予 | true | true (隐式) | 双 Provider 均工作 | 格式化坐标（GPS 精度优先） |
| 任一 Provider 异常 | -- | -- | `catch SecurityException` | `"位置信息: 获取异常 (msg)"` |
| 定位服务未开启 | -- | -- | `isProviderEnabled() == false` | `"位置信息: 未开启定位服务"` |

> **与现有权限系统的关系**：定位权限不在现有的 `request_android_permission` 的 `when` 分支中处理，也不通过 `@Permission` 注解声明别名。定位权限由 Android 系统在应用首次访问 `LocationManager` 时弹窗请求，或在系统设置中手动授予。`SensorStatusManager` 遵循"被动检查 + 静默降级"模式，不做主动权限请求。

### 10.4 与分布式工具层的映射

| 权限 | 涉及的 Rust 工具 | 数据流 |
|------|-----------------|--------|
| `ACCESS_FINE_LOCATION` / `ACCESS_COARSE_LOCATION` | `MobileLocation` (`location.rs`) | Android `LocationManager` → `SensorStatusManager` 缓存 → Plugin IPC `getSensorData("location")` → Rust `read_current()` → 主服务器 `update_static_placeholders` |
| 无（硬件访问无需运行时权限） | `MobileMotion` (`motion_sensor.rs`) | Android `SensorManager(TYPE_ACCELEROMETER)` → Burst 后处理 → Plugin IPC `getSensorData("motion")` → Rust |
| 无（硬件访问无需运行时权限） | `MobileAmbient` (`ambient_sensor.rs`) | Android `SensorManager(TYPE_LIGHT, TYPE_PRESSURE)` → Plugin IPC `getSensorData("ambient")` → Rust |

> 运动传感器（加速度计）和环境传感器（光/气压）在 Android 上**不需要运行时权限**，仅需 Manifest 声明或默认可用。因此 `SensorStatusManager` 的运动和环境采集不受权限检查影响，总是能正常启动。

### 10.5 定位权限数据流

```
用户授予定位权限（系统设置或首次弹窗）
    │
    ▼
Android 系统记录权限状态
    │
    ▼
SensorStatusManager.startLocationListening()
    ├── ContextCompat.checkSelfPermission(ACCESS_FINE_LOCATION) → GRANTED
    ├── ContextCompat.checkSelfPermission(ACCESS_COARSE_LOCATION) → GRANTED
    │
    ▼
LocationManager.requestLocationUpdates() 注册双 Provider
    │
    ▼
onLocationChanged(location) 回调
    ├── updateLocationString(location) → @Volatile latestLocationStr 更新
    │
    │  (30 秒周期)
    ▼
Rust location.rs::read_current()
    │
    ▼
IPC: plugin:vcp-mobile|get_sensor_data({ type: "location" })
    │
    ▼
@Volatile latestLocationStr → JSObject { value: "坐标: ..." }
    │
    ▼
Rust client.rs → 主服务器 update_static_placeholders → LLM Context
```

```kotlin
class CpuStatusManager(private val context: Context) {
    fun getThermalStatus(): JSObject {
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q) {
            val powerManager = context.getSystemService(Context.POWER_SERVICE) as PowerManager
            val status = powerManager.currentThermalStatus
            val statusStr = when (status) {
                PowerManager.THERMAL_STATUS_NONE      -> "正常"
                PowerManager.THERMAL_STATUS_LIGHT     -> "轻微发热"
                PowerManager.THERMAL_STATUS_MODERATE  -> "中等发热"
                PowerManager.THERMAL_STATUS_SEVERE    -> "严重发热"
                PowerManager.THERMAL_STATUS_CRITICAL  -> "极热(限频)"
                PowerManager.THERMAL_STATUS_EMERGENCY -> "紧急(限流)"
                PowerManager.THERMAL_STATUS_SHUTDOWN  -> "即将关机"
                else -> "未知"
            }
            result.put("status", statusStr)
        } else {
            result.put("status", "不支持(API<29)")
        }
    }
}
```

| 热状态常量 | 中文描述 | 系统行为 |
|-----------|----------|----------|
| `THERMAL_STATUS_NONE` | 正常 | 无限制 |
| `THERMAL_STATUS_LIGHT` | 轻微发热 | 轻度限制，用户不可感知 |
| `THERMAL_STATUS_MODERATE` | 中等发热 | 开始降低 CPU/GPU 频率 |
| `THERMAL_STATUS_SEVERE` | 严重发热 | 大幅降频，屏幕亮度可能降低 |
| `THERMAL_STATUS_CRITICAL` | 极热(限频) | CPU 被强制限频，UI 卡顿 |
| `THERMAL_STATUS_EMERGENCY` | 紧急(限流) | 严重限制，后台任务被暂停 |
| `THERMAL_STATUS_SHUTDOWN` | 即将关机 | 系统即将因过热关机 |

> **API 版本要求**：`currentThermalStatus` 需要 API 29 (Android 10 Q) 及以上。低于此版本返回 `"不支持(API<29)"`。

## 12. GpuStatusManager —— GPU 硬件信息拉取

> 源码：`src-tauri/plugins/vcp-mobile/android/src/main/java/com/vcp/mobile/GpuStatusManager.kt`（84 行）

### 12.1 概述

`GpuStatusManager` 是 v1.0.3 新增的独立模块，通过**动态创建临时 EGL 上下文**读取 GPU 渲染器型号（例如 "Adreno (TM) 740"）。GPU 型号信息仅需查询一次，使用双重检查锁定（Double-Checked Locking）+ `@Volatile` 缓存避免重复的 EGL 初始化开销。

### 12.2 Rust 接口

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct GpuStatus {
    pub renderer: String,
    pub restricted: bool,
}

#[tauri::command]
pub fn get_gpu_status<R: Runtime>(app: AppHandle<R>) -> Result<GpuStatus, String>;
```

| 属性 | 说明 |
|------|------|
| 前端命令 | `plugin:vcp-mobile\|get_gpu_status` |
| 参数 | 无 |
| 返回值 | `GpuStatus { renderer, restricted }` —— `restricted` 固定为 `true`，表示 GPU 信息受限（无 sysfs 功耗/温度数据） |
| 桌面端行为 | 返回 `{ renderer: "PC Mock GPU", restricted: true }` |

### 12.3 EGL 临时上下文技术

GpuStatusManager 采用轻量级 EGL Pbuffer Surface 方案读取 GPU 型号，避免需要 GLSurfaceView 或 Activity：

```
fetchGpuRendererFromEgl()
    │
    ├── EGLContext.getEGL() as EGL10
    ├── eglGetDisplay(EGL_DEFAULT_DISPLAY)
    ├── eglInitialize(dpy, vers)
    │
    ├── eglChooseConfig(dpy, configAttr, configs, 1, numConfig)
    │   └── configAttr: [RED_SIZE=8, GREEN_SIZE=8, BLUE_SIZE=8]
    │
    ├── eglCreatePbufferSurface(dpy, config, [WIDTH=1, HEIGHT=1])
    │   └── 1x1 像素 Pbuffer Surface，最小化 GPU 资源占用
    │
    ├── eglCreateContext(dpy, config, EGL_NO_CONTEXT, [EGL_CONTEXT_CLIENT_VERSION=2])
    │   └── GLES 2.0 上下文
    │
    ├── eglMakeCurrent(dpy, surf, surf, ctx)
    │
    ├── GLES20.glGetString(GLES20.GL_RENDERER) → "Adreno (TM) 740"
    │
    └── 清理（严格按顺序）
        ├── eglMakeCurrent(dpy, EGL_NO_SURFACE, EGL_NO_SURFACE, EGL_NO_CONTEXT)
        ├── eglDestroyContext(dpy, ctx)
        ├── eglDestroySurface(dpy, surf)
        └── eglTerminate(dpy)
```

**为什么 `restricted` 固定为 `true`？**
在大多数 Android 设备上，`/sys/class/kgsl/` 或 `/sys/class/devfreq/` 等 GPU 性能计数器路径需要 Root 权限读取。`GpuStatusManager` 仅通过 EGL 读取渲染器名称字符串，无法获取 GPU 使用率、频率、温度等运行时指标。`restricted: true` 向前端明确告知此限制。

### 12.4 双重检查锁定缓存

```kotlin
@Volatile
private var cachedGpuRenderer: String? = null

fun getGpuRenderer(): String {
    cachedGpuRenderer?.let { return it }     // 第一次检查（无锁，快速路径）
    synchronized(this) {
        cachedGpuRenderer?.let { return it } // 第二次检查（持锁，防止竞争）
        val renderer = fetchGpuRendererFromEgl()
        cachedGpuRenderer = renderer
        return renderer
    }
}
```

| 场景 | 行为 |
|------|------|
| 首次调用 | `cachedGpuRenderer == null` → 进入 `synchronized` → 创建 EGL → 读取 GL_RENDERER → 写入缓存 |
| 并发调用（首次） | 第二个线程在 `synchronized` 外的 `?.let` 返回 `null` → 等待锁 → 持锁后第二次检查 → 发现已缓存 → 直接返回 |
| 后续调用 | `@Volatile` 保证可见性 → 无锁快速路径返回缓存值 → 零 EGL 开销 |

```kotlin
fun getNetworkStatus(): JSObject {
    val cm = context.getSystemService(Context.CONNECTIVITY_SERVICE) as ConnectivityManager
    val activeNetwork = cm.activeNetwork
    if (activeNetwork == null) {
        // 无活跃网络
        return { connected: false, type: "未连接", downSpeedKbps: 0, upSpeedKbps: 0, ip: "未分配" }
    }

    val capabilities = cm.getNetworkCapabilities(activeNetwork) ?: return /* 同上 */

    val type = when {
        capabilities.hasTransport(NetworkCapabilities.TRANSPORT_WIFI)     -> "WiFi"
        capabilities.hasTransport(NetworkCapabilities.TRANSPORT_CELLULAR) -> "移动数据"
        capabilities.hasTransport(NetworkCapabilities.TRANSPORT_ETHERNET) -> "以太网"
        else -> "未知"
    }

    val downSpeed = capabilities.linkDownstreamBandwidthKbps  // 下行估计带宽 (Kbps)
    val upSpeed   = capabilities.linkUpstreamBandwidthKbps    // 上行估计带宽 (Kbps)
    val ip        = getLocalIpAddress()                       // 本地 IPv4 地址
}
```

| 字段 | 来源 | 说明 |
|------|------|------|
| `connected` | `ConnectivityManager.activeNetwork` | `null` 表示无网络连接 |
| `type` | `NetworkCapabilities.hasTransport()` | 优先级：WiFi > 移动数据 > 以太网 |
| `downSpeedKbps` | `linkDownstreamBandwidthKbps` | 链路层估计值，非实时测速 |
| `upSpeedKbps` | `linkUpstreamBandwidthKbps` | 链路层估计值，非实时测速 |
| `ip` | `NetworkInterface.getNetworkInterfaces()` | 非回环 `Inet4Address` 的第一个地址 |

### 13.4 本地 IP 获取

```kotlin
private fun getLocalIpAddress(): String? {
    val en = NetworkInterface.getNetworkInterfaces()
    while (en.hasMoreElements()) {
        val intf = en.nextElement()
        val enumIpAddr = intf.inetAddresses
        while (enumIpAddr.hasMoreElements()) {
            val inetAddress = enumIpAddr.nextElement()
            if (!inetAddress.isLoopbackAddress && inetAddress is Inet4Address) {
                return inetAddress.hostAddress
            }
        }
    }
    return null
}
```

- 遍历所有网络接口，返回**第一个非回环 IPv4 地址**。
- 如无可用 IPv4 地址，`ip` 字段设为 `"未分配"`。

### 13.5 桌面端与异常 fallback

| 场景 | connected | type | downSpeedKbps | upSpeedKbps | ip |
|------|-----------|------|---------------|-------------|-----|
| 正常连接（WiFi） | true | "WiFi" | 链路估计值 | 链路估计值 | 实际 IPv4 |
| 正常连接（移动数据） | true | "移动数据" | 链路估计值 | 链路估计值 | 实际 IPv4 |
| 无活跃网络 | false | "未连接" | 0 | 0 | "未分配" |
| capabilities 为 null | false | "未连接" | 0 | 0 | "未分配" |
| 异常 | false | "未连接" | 0 | 0 | "未分配" |

## 14. Root 访问体系

```kotlin
                intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
```

| `me.tool.apatch` | APatch | 新一代内核级 Root 方案 |

### 14.5 AndroidManifest.xml 包可见性声明

Android 11 (API 30) 引入了包可见性限制，默认情况下应用无法查询其他已安装应用。为支持 `launch_root_manager` 中通过 `getLaunchIntentForPackage` 查询 Root 管理器，需在 `AndroidManifest.xml` 中声明 `<queries>`：

```xml

    <package android:name="com.topjohnwu.magisk" />
    <package android:name="me.weishu.kernelsu" />
    <package android:name="me.tool.apatch" />
</queries>
```

> **不声明 `<queries>` 的后果**：`getLaunchIntentForPackage()` 始终返回 `null`，`launch_root_manager` 会错误报告"未找到支持的 Root 管理器"，即使设备上已安装了 Magisk/KernelSU/APatch。

## 15. WakeLock/WifiLock 保活机制

> 源码：`VcpMobilePlugin.kt` (lines 430--480), `system.rs` (lines 749--785)

### 15.1 概述

v1.0.3 新增 WakeLock + WifiLock 双锁保活机制，用于长连接场景（如 SSE 流式对话、WebSocket 同步）中防止 CPU 进入深度休眠和 WiFi 断开。两种锁的获取和释放通过 `acquire_wake_lock` / `release_wake_lock` 命令成对调用。

### 15.2 acquire_wake_lock —— 获取双锁

```rust
#[tauri::command]
pub fn acquire_wake_lock<R: Runtime>(app: AppHandle<R>) -> Result<(), String>;
```

| 属性 | 说明 |
|------|------|
| 前端命令 | `plugin:vcp-mobile\|acquire_wake_lock` |
| 参数 | 无 |
| 桌面端行为 | 空操作 |
| 安全性 | 最大持有时间 5 分钟，防止电池耗尽 |

**Kotlin 实现**：

```kotlin
@Command
fun acquireWakeLock(invoke: Invoke) {
    // 1. CPU WakeLock
    val pm = activity.getSystemService(Context.POWER_SERVICE) as PowerManager
    if (wakeLock == null) {
        wakeLock = pm.newWakeLock(PowerManager.PARTIAL_WAKE_LOCK, "VcpMobile:WakeLock")
    }
    if (wakeLock?.isHeld == false) {
        wakeLock?.acquire(5 * 60 * 1000L) // 最大 5 分钟安全限制
    }

    // 2. WiFi Lock（best-effort）
    try {
        val wm = activity.applicationContext.getSystemService(Context.WIFI_SERVICE) as WifiManager
        if (wifiLock == null) {
            wifiLock = wm.createWifiLock(
                if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q)
                    WifiManager.WIFI_MODE_FULL_LOW_LATENCY  // API 29+ 低延迟模式
                else
                    WifiManager.WIFI_MODE_FULL_HIGH_PERF,   // 旧版高性能模式
                "VcpMobile:WifiLock"
            )
        }
        if (wifiLock?.isHeld == false) {
            wifiLock?.acquire()
        }
    } catch (wifiEx: Exception) {
        Log.w(TAG, "Failed to acquire WiFi Lock: ${wifiEx.message}")
    }

    invoke.resolve()
}
```

**双锁详细参数**：

| 锁类型 | 类/方法 | 模式 | 标签 | 安全措施 |
|--------|---------|------|------|----------|
| WakeLock | `PowerManager.newWakeLock()` | `PARTIAL_WAKE_LOCK` | `"VcpMobile:WakeLock"` | 超时 5 分钟 (`5*60*1000L`)，防止忘记释放导致电池耗尽 |
| WifiLock (API 29+) | `WifiManager.createWifiLock()` | `WIFI_MODE_FULL_LOW_LATENCY` | `"VcpMobile:WifiLock"` | Best-effort 获取（catch Exception 仅警告） |
| WifiLock (API <29) | `WifiManager.createWifiLock()` | `WIFI_MODE_FULL_HIGH_PERF` | `"VcpMobile:WifiLock"` | 同上 |

**重复获取保护**：
- 两个锁在获取前均检查 `isHeld`，已持有的锁不会重复获取。
- WakeLock 和 WifiLock 的实例仅在 `null` 时创建，复用同一实例。

## 16. 网络监控与自动重连

### 16.1 概述

v1.0.3 新增基于 Android `ConnectivityManager.NetworkCallback` 的原生网络状态监控，在网络可用/丢失/能力变化时自动触发事件推送到 WebView。相比前端 `navigator.onLine` + `online`/`offline` 事件，原生回调更可靠且能感知网络能力变化（如从 WiFi 切换到移动数据）。

### 16.2 start_network_monitoring —— 启动网络监控

**Kotlin 实现**：

```kotlin
private var networkCallback: ConnectivityManager.NetworkCallback? = null
private var lastConnected: Boolean? = null
private var isNetworkMonitoringStarted = false

@Command
fun startNetworkMonitoring(invoke: Invoke) {
    if (isNetworkMonitoringStarted) {
        invoke.resolve()
        return
    }
    val cm = activity.getSystemService(Context.CONNECTIVITY_SERVICE) as ConnectivityManager
    val request = NetworkRequest.Builder()
        .addCapability(NetworkCapabilities.NET_CAPABILITY_INTERNET)
        .build()
    networkCallback = object : ConnectivityManager.NetworkCallback() {
        override fun onAvailable(network: Network) {
            emitNetworkStatusToWebView()
        }
        override fun onLost(network: Network) {
            emitNetworkStatusToWebView()
        }
        override fun onCapabilitiesChanged(network: Network, caps: NetworkCapabilities) {
            emitNetworkStatusToWebView()
        }
    }
    cm.registerNetworkCallback(request, networkCallback!!)
    isNetworkMonitoringStarted = true
    invoke.resolve()
}
```

### 16.3 回调触发条件

| 回调方法 | 触发时机 | 典型场景 |
|----------|----------|----------|
| `onAvailable` | 满足 `NET_CAPABILITY_INTERNET` 的网络变为可用 | WiFi 连接成功、移动数据开启 |
| `onLost` | 之前可用的网络丢失 | WiFi 断开、飞行模式 |
| `onCapabilitiesChanged` | 网络能力发生变化 | WiFi → 移动数据切换、带宽限制变化 |

### 16.4 变化检测与事件推送

`emitNetworkStatusToWebView()` 在推送事件前进行连接状态变化检测，避免重复通知：

```kotlin
private fun emitNetworkStatusToWebView() {
    val status = networkStatusManager.getNetworkStatus()
    val connected = status.optBoolean("connected", false)
    if (connected != lastConnected) {
        lastConnected = connected
        trigger("vcp-network-status-changed", status)
    }
}
```

| 变量 | 类型 | 说明 |
|------|------|------|
| `lastConnected` | `Boolean?` | 上次已知的连接状态。初始 `null`，首次回调必定触发事件 |
| `vcp-network-status-changed` | Tauri Plugin `trigger` 事件 | 携带完整 `NetworkStatus` JSON 对象 |
