# 文件选择与缩略图生成

## 1. 概述

### 1.1 定位与职责边界

本插件是 **VCP Mobile Android 端的附件上传入口**，核心职责：

1. **打通 Scoped Storage 壁垒**：Android 10+ 引入 Scoped Storage 后，前端 WebView 无法直接访问用户选择文件的物理路径。本插件通过 `Intent.ACTION_GET_CONTENT` 获取系统授权 URI，在 Kotlin 侧完成 URI 到应用私有缓存的流式拷贝。
2. **零拷贝物理路径传输**：拷贝完成后将绝对路径通过 Tauri IPC 回传前端，前端调用 Rust `register_local_file` 将文件从缓存目录 `rename` 移至正式附件目录，实现跨层"零拷贝"。
3. **硬件加速缩略图**：对图片资源利用 Android Q+ `ThumbnailUtils.createImageThumbnail` 进行硬件级自适应缩放；低版本通过智能预采样防 OOM 软解。

### 1.2 与三层架构的关系

| 层级 | 角色 | 交互方式 |
|------|------|----------|
| **Vue 3 前端** | `attachmentStore.ts` 识别 Android 环境并拦截为原生调用 | `invoke("plugin:vcp-mobile\|pick_file")` |
| **Tauri IPC** | 命令路由与参数序列化 | Tauri Plugin Manager 自动处理 |
| **Kotlin 插件** | 唤起选择器、流式拷贝、SHA-256、缩略图生成、**进度上报、双轨通信** | `VcpMobilePlugin.kt` 第 221–388 行 |
| **Rust 后端** | 安全校验、独立 SHA-256 验证、rename 移动、SQLite 入库 | `file_manager.rs` 第 778–913 行 |

### 1.3 为什么需要原生文件选择器

前端 HTML `<input type="file">` 在 Android WebView 中返回 Blob/File 对象，无法获取物理路径；大文件通过 base64 传入 Rust 会撑爆 IPC payload；且 Scoped Storage 下 JS 无法直接读取外部存储。原生 `pickFile` 通过 `Intent.ACTION_GET_CONTENT` 获取系统授权 URI，Kotlin 侧以 64 KiB 缓冲流式拷贝，边拷贝边算 SHA-256，图片直接走系统硬件加速解码。

### 2.2 唤起系统选择器

```kotlin
// VcpMobilePlugin.kt : 221–234
fun pickFile(invoke: Invoke) {
    try {
        val intent = Intent(Intent.ACTION_GET_CONTENT).apply {
            type = "*/*"
            addCategory(Intent.CATEGORY_OPENABLE)
        }
        startActivityForResult(invoke, intent, "onPickFileResult")
    } catch (e: Throwable) {
        Log.e(TAG, "[pickFile] Failed to start activity for result", e)
        invoke.reject("Failed to start native file picker: ${e.message}")
    }
}
```

| 属性 | 值 | 说明 |
|------|-----|------|
| `action` | `Intent.ACTION_GET_CONTENT` | 标准内容选择器，返回 URI 的读权限 |
| `type` | `"*/*"` | 不限制 MIME 类型 |
| `category` | `Intent.CATEGORY_OPENABLE` | 强制要求返回文件可被 `openInputStream()` 打开 |

> 不使用 `ACTION_OPEN_DOCUMENT`，因为不需要长期持久化权限。本插件在回调内立即读取并拷贝至私有缓存，URI 权限在回调结束后即废弃。
>
> `pickFile` 已包裹 `try-catch`，若系统选择器无法启动（如 Activity 不可用），会立即通过 `invoke.reject` 返回错误，避免前端无限等待。

### 2.4 流式拷贝与同步 SHA-256

```kotlin
// VcpMobilePlugin.kt : 268–302
val tempFile = File(context.cacheDir, "pick_${System.currentTimeMillis()}_temp")
val digest = MessageDigest.getInstance("SHA-256")

contentResolver.openInputStream(uri).use { inputStream ->
    if (inputStream == null) {
        Log.e(TAG, "[onPickFileResult] openInputStream returned null")
        invoke.reject("Could not open input stream")
        return@Thread
    }
    FileOutputStream(tempFile).use { outputStream ->
        val buffer = ByteArray(65536)      // 64 KiB 缓冲
        var bytesRead: Int
        var totalRead = 0L
        var lastReportTime = System.currentTimeMillis()
        
        while (inputStream.read(buffer).also { bytesRead = it } != -1) {
            outputStream.write(buffer, 0, bytesRead)
            digest.update(buffer, 0, bytesRead)   // 边读边算哈希
            totalRead += bytesRead
            
            // 每 200 ms 上报一次进度
            val now = System.currentTimeMillis()
            if (now - lastReportTime > 200) {
                lastReportTime = now
                val progress = if (size > 0) ((totalRead.toDouble() / size) * 100).toInt() else 0
                val progressDetail = JSObject().apply {
                    put("loaded", totalRead)
                    put("total", size)
                    put("progress", progress)
                    put("name", originalName)
                    put("mime", mimeType)
                }
                val progressScript = "window.dispatchEvent(new CustomEvent('vcp-mobile-file-progress', { detail: $progressDetail }))"
                activity.runOnUiThread {
                    webViewRef?.evaluateJavascript(progressScript, null)
                }
            }
        }
    }
}
val hash = digest.digest().joinToString("") { "%02x".format(it) }
```

| 决策点 | 实现 | 理由 |
|--------|------|------|
| 缓冲大小 | 64 KiB | 平衡内存占用与 IO 吞吐量 |
| 同步哈希 | `MessageDigest.update()` 内嵌于读循环 | 避免拷贝完成后二次读取 |
| 进度上报 | 每 200 ms 通过 `evaluateJavascript` 注入 `CustomEvent` | 前端实时感知拷贝进度，200 ms 避免过度刷新 |
| 临时文件命名 | `pick_${System.currentTimeMillis()}_temp` | 带时间戳防并发冲突 |
| 输出目录 | `context.cacheDir` | 应用私有目录，无需存储权限 |

### 2.5 内容寻址重命名与去重

```kotlin
// VcpMobilePlugin.kt : 284–294
val fileExtension = File(originalName).extension.let {
    if (it.isEmpty()) "" else ".$it"
}
val finalTempFile = File(context.cacheDir, "$hash$fileExtension")

if (finalTempFile.exists()) {
    tempFile.delete()           // 缓存去重，复用已有文件
} else {
    tempFile.renameTo(finalTempFile)   // 原子重命名
}
```

- 文件名 = `SHA-256` + 原扩展名（如 `a1b2c3...f0.png`）
- 内容完全相同的文件第二次选择时，`finalTempFile.exists()` 为 `true`，直接删除临时拷贝
- Rust 侧 `register_local_file` 在正式附件目录执行相同去重，形成**双层去重**

### 2.6 结果组装与线程模型

```kotlin
// VcpMobilePlugin.kt : 333–364
val resultObject = JSObject()

resultObject.put("name", originalName)

// 双轨通信：主动推送最终结果给前端，穿透 JNI 断裂层
val pickedDetail = JSObject().apply {
    put("path", finalTempFile.absolutePath)
    put("name", originalName)

    put("size", finalSize)
    put("hash", hash)
    if (thumbnailPath != null) {
        put("thumbnailPath", thumbnailPath)
    } else {
        put("thumbnailPath", org.json.JSONObject.NULL)
    }
}
val pickedScript = "window.dispatchEvent(new CustomEvent('vcp-mobile-file-picked', { detail: $pickedDetail }))"
activity.runOnUiThread {
    webViewRef?.evaluateJavascript(pickedScript, null)
}

invoke.resolve(resultObject)
```

| 字段 | 类型 | 来源 | Rust 侧接收参数名 |
|------|------|------|-------------------|
| `path` | `string` | `finalTempFile.absolutePath` | `localPath` |
| `name` | `string` | `originalName` | `originalName` |
| `mime` | `string` | `contentResolver.getType(uri)` | `mimeType` |
| `size` | `number` | `OpenableColumns.SIZE` fallback `file.length()` | — |
| `hash` | `string` | Kotlin 侧同步 SHA-256 | — |
| `thumbnailPath` | `string?` | `generateNativeThumbnail()` 返回值 | `thumbnailPath` |

**线程模型**：文件 IO 全部在 `Thread { ... }.start()` 工作线程执行；`evaluateJavascript` 事件推送通过 `runOnUiThread` 回到 UI 线程调用。`invoke.resolve()` / `invoke.reject()` 直接在工作线程末尾调用。

### 2.7 双轨通信与前端事件

除 `invoke.resolve()` 标准返回外，本插件在关键节点通过 `evaluateJavascript` 向前端注入 `CustomEvent`，形成"双轨通信"：

| 事件名 | 触发时机 | 字段 | 用途 |
|--------|---------|------|------|
| `vcp-mobile-file-start` | 元数据提取完成后 | `name`, `size`, `mime` | 前端提前创建进度占位卡片 |
| `vcp-mobile-file-progress` | 拷贝循环中每 200 ms | `loaded`, `total`, `progress`, `name`, `mime` | 前端实时更新拷贝进度 |
| `vcp-mobile-file-picked` | 拷贝与处理全部完成后 | `path`, `name`, `mime`, `size`, `hash`, `thumbnailPath` | 穿透 JNI 断裂层，确保前端必收到结果 |

> **为什么需要双轨通信**：`invoke.resolve()` 依赖 Tauri 标准 IPC 通道，在极端场景（如 JNI 断裂、WebView 重载）下可能丢失。`evaluateJavascript` 直接注入 JS 事件，作为冗余通道兜底。

### 3.3 双路径缩略图生成

#### 路径 A：Android Q+（API 29+）硬件加速

```kotlin
// VcpMobilePlugin.kt : 333–335
ThumbnailUtils.createImageThumbnail(
    originalFile,
    Size(200, 200),
    null
)
```

| 特性 | 说明 |
|------|------|
| API 级别 | API 29+ |
| 底层实现 | 系统多媒体框架，可能走硬件解码器 |
| 自适应缩放 | 系统自动保持长宽比，短边 ≈ 200px |
| 内存安全 | 系统级优化，大图不直接全尺寸载入 Java Heap |

#### 路径 B：Android Q 以下软件解码（智能预采样防 OOM）

```kotlin
// VcpMobilePlugin.kt : 337–372
val options = BitmapFactory.Options().apply { inJustDecodeBounds = true }
BitmapFactory.decodeFile(originalFile.absolutePath, options)
val width = options.outWidth
val height = options.outHeight

var inSampleSize = 1
if (width > 200 || height > 200) {
    val halfHeight = height / 2
    val halfWidth = width / 2
    while (halfHeight / inSampleSize >= 200 && halfWidth / inSampleSize >= 200) {
        inSampleSize *= 2
    }
}

options.inJustDecodeBounds = false
options.inSampleSize = inSampleSize
val rawBitmap = BitmapFactory.decodeFile(originalFile.absolutePath, options) ?: return null

// 保持长宽比，短边缩放到 200px
val (newW, newH) = if (w >= h) {
    ((200f * w / h).toInt() to 200)
} else {
    (200 to (200f * h / w).toInt())
}
val scaled = Bitmap.createScaledBitmap(rawBitmap, newW, newH, true)
if (scaled != rawBitmap) rawBitmap.recycle()
```

**预采样算法示例**：

| 原始尺寸 | `inSampleSize` | 实际解码尺寸 | 内存节省 |
|----------|---------------|-------------|----------|
| 4000 × 3000 | 8 | 500 × 375 | 64× |
| 8000 × 6000 | 16 | 500 × 375 | 256× |
| 1920 × 1080 | 2 | 960 × 540 | 4× |

> `inSampleSize` 必须是 2 的整数幂。算法核心：将 `halfHeight / inSampleSize` 和 `halfWidth / inSampleSize` 与目标 200 比较，直到任一维度小于 200。

### 3.4 编码输出与内存释放

```kotlin
// VcpMobilePlugin.kt : 374–383
FileOutputStream(thumbFile).use { out ->
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.R) {
        bitmap.compress(Bitmap.CompressFormat.WEBP_LOSSY, 80, out)
    } else {
        @Suppress("DEPRECATION")

```

| 项目 | 说明 |
|------|------|
| 输出格式 | WEBP，体积远小于 JPEG，质量接近 PNG |
| 质量系数 | 80 —— 视觉损失不可感知，体积较 PNG 减少 60–80% |
| Android R+ | `WEBP_LOSSY`，编码速度更快 |
| 低版本 | `WEBP`，等同于有损压缩 |

### 3.5 硬件加速 vs 软件解码对比

| 维度 | Android Q+ `ThumbnailUtils` | Android Q 以下 `BitmapFactory` |
|------|---------------------------|------------------------------|
| 解码器 | 系统多媒体框架（可能硬件解码） | 软件 Skia 解码 |
| 大图安全 | 系统自动管理 | 依赖 `inSampleSize` 手动预采样 |
| 速度 | 快（硬件加速） | 中等（纯 CPU） |
| 兼容性 | API 29+ | 全版本 |

### 4.4 为什么 Kotlin 侧已算 SHA-256，Rust 侧还要重算？

| 层面 | Kotlin 侧哈希 | Rust 侧哈希 |
|------|--------------|-------------|
| **信任边界** | 应用内部代码 | 数据持久化前的最终把关 |
| **用途** | 临时缓存去重、缩略图命名 | 正式附件内容寻址、SQLite 入库 |
| **防御性设计** | — | 确保入库哈希与实际物理文件**逐字节一致**。若 Kotlin 侧因并发或内存问题导致哈希错误，Rust 侧独立重算可拦截 |
| **算法实现** | `MessageDigest.getInstance("SHA-256")` | `sha2::Sha256` |
| **缓冲策略** | 64 KiB 同步流式 | 64 KiB 异步流式 (`tokio::io::AsyncReadExt`) |

> 双层哈希不是冗余，而是**跨语言信任边界的防御性校验**。Kotlin 侧哈希用于前端体验优化，Rust 侧哈希才是数据层的事实标准。

## 5. 安全与性能

### 5.1 Scoped Storage 合规

| 合规项 | 实现 |
|--------|------|
| 不申请存储权限 | 仅使用 `Intent.ACTION_GET_CONTENT` + `CATEGORY_OPENABLE`，获取用户单次授权 URI |
| 不扫描外部存储 | 不遍历 `/sdcard/` 等公共目录 |
| 最小权限原则 | URI 权限仅在 `onPickFileResult` 回调期间有效 |
| 输出隔离 | 所有写入限制在 `context.cacheDir`，其他应用无访问权限 |

### 5.2 内存安全

| 风险点 | 防护措施 |
|--------|----------|
| 大文件拷贝 OOM | 64 KiB 固定缓冲流式读写，内存占用与文件大小无关 |
| 大图解码 OOM | Android Q+ 走 `ThumbnailUtils`（框架级内存管理）；低版本用 `inSampleSize` 预采样 |
| Bitmap Native 内存泄漏 | 显式调用 `bitmap.recycle()` |
| 临时文件堆积 | Rust 侧 rename 后自动清理源文件；Kotlin 侧去重时主动 `delete()` |

### 5.3 性能特征

| 指标 | 数值 / 行为 |
|------|------------|
| 流式缓冲 | 64 KiB（与 Rust 侧一致） |
| 缩略图目标尺寸 | 短边 200 px |
| 缩略图格式 | WEBP 质量 80，典型体积 8–15 KB |
| 缩略图缓存命中 | O(1) 文件系统检查，内容相同则完全跳过生成 |
| 双层去重 | Kotlin 缓存层 + Rust 正式层，第二次起几乎零 IO |
| 线程模型 | 文件 IO 全部在后台线程，UI 线程负责唤起选择器与 `evaluateJavascript` 事件推送 |
| 进度上报 | 每 200 ms 触发一次 `evaluateJavascript`，单次 payload < 200 B |

### 5.4 已知限制

1. **临时文件生命周期**：`cacheDir` 中的文件在系统低存储时可能被自动清理。若用户选择后长时间不发送消息（不调用 `register_local_file`），临时文件可能丢失。
2. **缩略图生成失败静默降级**：`generateNativeThumbnail` 返回 `null` 时，前端回退为使用原图路径展示，不影响功能。
3. **GIF 动图缩略图**：仅提取首帧，系 Android 平台限制。
4. **非图片文件无缩略图**：仅 `mimeType.startsWith("image/")` 触发。
5. **Rust 侧路径遍历防护**：`ensure_safe_path` 校验传入路径是否位于应用合法目录内。
6. **`evaluateJavascript` 时序**：双轨通信事件通过 `evaluateJavascript` 注入，若 WebView 在文件处理期间被销毁或重载，事件可能丢失；`invoke.resolve()` 作为标准 IPC 兜底仍可用。
