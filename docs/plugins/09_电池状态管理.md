# 电池状态管理

## 1. 功能概述

电池状态管理模块提供两套互补的电池信息采集路径：

1. **按需查询路径**：前端通过 Tauri Command `get_battery_status` 主动获取当前电量百分比与系统省电模式状态，由 Rust 侧的 `system.rs` 通过 `PluginHandle.run_mobile_plugin` 桥接到 Kotlin 侧的 `BatteryStatusManager`。
2. **分布式流式路径**：Rust 核心层的 `BatteryInfoTool` 与 `DeviceStatusSummaryTool` 直接读取 `/sys/class/power_supply/battery/` 下的 sysfs 节点，作为 Agent 环境感知能力的输入源，无需经过 Kotlin 层。

> **设计决策**：`BatteryStatusManager` 采用**纯查询式**设计，不维护后台广播监听器，也不在 `evaluateJavascript` 通道中主动推送电池变化事件。这是因为：
> - 电池变化频率低（1% 粒度），主动轮询开销远大于按需查询；
> - 系统省电模式的切换通常伴随 Activity 生命周期事件（如 `onResume`），前端可在恢复时顺带查询；
> - 保持与 `docs/ANDROID_PLUGIN_MANAGEMENT.md` §6.1 三条铁律之第 2 条一致：不引入不必要的后台监听器。

## 3. 接口详解

### 3.1 Kotlin 侧：`BatteryStatusManager`

#### `getBatteryLevel(): Int`

通过注册一个空接收器监听 `Intent.ACTION_BATTERY_CHANGED` 粘性广播，提取 `BatteryManager.EXTRA_LEVEL` 与 `EXTRA_SCALE`，计算百分比：

```kotlin
val level = batteryStatus?.getIntExtra(BatteryManager.EXTRA_LEVEL, -1) ?: -1
val scale = batteryStatus?.getIntExtra(BatteryManager.EXTRA_SCALE, -1) ?: -1
return if (level >= 0 && scale > 0) {
    ((level.toFloat() / scale.toFloat()) * 100).toInt()
} else { -1 }
```

| 属性 | 说明 |
|------|------|
| 返回值范围 | `0..100` 为正常电量，`-1` 表示获取失败 |
| 失败原因 | 广播异常、`level`/`scale` 缺失或非法 |
| 线程安全 | 无状态，可在任意线程调用 |

#### `isPowerSaveMode(): Boolean`

**六合一补强防线**，全面兼容国产定制 ROM：

| 层级 | 检测目标 | API / Settings Key | 说明 |
|------|---------|-------------------|------|
| 1 | 原生 PowerManager | `PowerManager.isPowerSaveMode()` | 官方标准 API，Android 5.0+ |
| 2 | 全局低电量标记 | `Settings.Global.getInt("low_power", 0)` | 系统全局低电量触发 |
| 3 | 系统省电标记 | `Settings.System.getInt("power_save_mode", 0)` | 通用 ROM 存放位置 |
| 4 | 小米 (MIUI/HyperOS) | `Settings.System.getInt("POWER_SAVE_MODE_OPEN", 0)` | MIUI 私有标记 |
| 5 | 华为 (EMUI/HarmonyOS) | `Settings.System.getInt("SmartModeStatus", 1)` | `2`=省电, `3`=超级省电 |
| 6 | 其他定制 ROM | `Settings.Global.getInt("power_save_mode", 0)` | Vivo/Oppo 等可能存放位置 |

> **注意**：`SmartModeStatus` 的语义为 `1=正常, 2=省电, 3=超级省电, 4=性能`，因此仅 `2` 和 `3` 判定为省电模式。

#### `getStatusJson(): JSObject`

将上述两个方法的结果打包为 Tauri 的 `JSObject`：

```kotlin
val result = JSObject()
result.put("level", getBatteryLevel())           // Int
result.put("isPowerSaveMode", isPowerSaveMode()) // Boolean
return result
```

### 3.2 Kotlin 侧：`VcpMobilePlugin.getBatteryStatus`

```kotlin
@Command
fun getBatteryStatus(invoke: Invoke) {
    try {
        val status = batteryStatusManager.getStatusJson()
        invoke.resolve(status)
    } catch (e: Exception) {
        Log.e(TAG, "getBatteryStatus failed", e)
        invoke.reject(e.message ?: "Unknown error")
    }
}
```

这是 Tauri v2 插件的标准 `@Command` 方法，前端通过 `invoke('plugin:vcp-mobile|getBatteryStatus')` 触发。

### 3.3 Rust 侧：`system.rs::get_battery_status`

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct BatteryStatus {
    pub level: i32,
    pub is_power_save_mode: bool,
}

#[tauri::command]
pub fn get_battery_status<R: Runtime>(app: AppHandle<R>) -> Result<BatteryStatus, String> {
    #[cfg(target_os = "android")]
    {
        let state = app.state::<VcpMobileState<R>>();
        let handle = state.plugin_handle.lock().map_err(|e| e.to_string())?;
        let plugin_handle = handle.as_ref().ok_or("Plugin handle not initialized")?;

        let status = plugin_handle
            .run_mobile_plugin::<BatteryStatus>("getBatteryStatus", serde_json::json!({}))
            .map_err(|e| format!("run_mobile_plugin failed: {}", e))?;
        Ok(status)
    }
    #[cfg(not(target_os = "android"))]
    {
        Ok(BatteryStatus { level: 100, is_power_save_mode: false })
    }
}
```

| 属性 | 说明 |
|------|------|
| 前端命令 | `plugin:vcp-mobile\|getBatteryStatus` |
| 参数 | 无 |
| 返回值 | `Result<BatteryStatus, String>` |
| 桌面端行为 | 固定返回 `level: 100, is_power_save_mode: false` |
| 序列化 | `#[serde(rename_all = "camelCase")]` 将 `is_power_save_mode` 序列化为 `isPowerSaveMode` |

## 4. Rust → Kotlin 调用链路拆解

```
前端 invoke()
    ↓
Tauri Runtime 路由
    ↓
system.rs::get_battery_status()
    ↓
VcpMobileState.plugin_handle.lock()
    ↓
PluginHandle.run_mobile_plugin::<BatteryStatus>("getBatteryStatus", {})
    ↓ (JNI 桥接)
VcpMobilePlugin.getBatteryStatus(invoke)
    ↓
BatteryStatusManager.getStatusJson()
    ├─→ getBatteryLevel()      [IntentFilter + BatteryManager]
    └─→ isPowerSaveMode()      [PowerManager + Settings 六重检测]
    ↓
invoke.resolve(JSObject { level, isPowerSaveMode })
    ↓ (JNI 反向桥接)
Rust 侧反序列化为 BatteryStatus
    ↓
前端收到 { level: i32, isPowerSaveMode: bool }
```

## 5. 分布式流式工具链路（sysfs 路径）

除了 Tauri 插件桥接路径外，Rust 分布式工具层有两条**绕过 Kotlin** 直接读取内核 sysfs 的流式路径：

### 5.1 `BatteryInfoTool`

位于 `src-tauri/src/distributed/tools/battery.rs`，实现 `StreamingTool` trait：

| sysfs 节点 | 说明 | 示例输出 |
|-----------|------|----------|
| `/sys/class/power_supply/battery/capacity` | 电量百分比 | `"78"` → `"78%"` |
| `/sys/class/power_supply/battery/status` | 充放电状态 | `"Charging"` → `"充电中"` |
| `/sys/class/power_supply/battery/temp` | 电池温度（0.1°C 单位） | `"315"` → `"32°C"` |

- **轮询间隔**：60 秒
- **占位符**：`{{MobileBattery}}`
- **降级策略**：若 sysfs 路径全部不可读，返回 `"电池信息不可用"`

### 5.2 `DeviceStatusSummaryTool`

位于 `src-tauri/src/distributed/tools/device_status_summary.rs`，在生成设备状态聚合摘要时复用相同的 sysfs 路径：

```rust
fn battery_brief(&self) -> String {
    let cap = read_sysfs("/sys/class/power_supply/battery/capacity");
    let status = read_sysfs("/sys/class/power_supply/battery/status");
    // ...
}
```

输出示例：`电量:78%充电中`

### 5.3 两条路径的职责边界

| 维度 | Tauri 插件路径 (`system.rs` → Kotlin) | 分布式流式路径 (`battery.rs` → sysfs) |
|------|--------------------------------------|--------------------------------------|
| **触发方式** | 前端按需 `invoke` | Agent 环境感知自动轮询（60s） |
| **数据来源** | Android Framework (`BatteryManager`) | Linux 内核 sysfs |
| **信息维度** | 电量 + 省电模式 | 电量 + 充放电状态 + 温度 |
| **是否经过 Kotlin** | 是 | 否 |
| **前端可见性** | 高（可主动查询） | 低（仅作为 Agent 上下文注入） |
| **国产 ROM 兼容** | 是（`isPowerSaveMode` 六重检测） | 不相关（ sysfs 为内核标准接口） |

## 6. 电池优化白名单（关联但独立）

`BatteryStatusManager` 仅负责**实时电量与省电模式**查询。与之相邻但独立的功能是**电池优化白名单状态**（`isIgnoringBatteryOptimizations`），由 `VcpMobilePlugin` 直接处理：

### 6.1 查询白名单状态

```kotlin
// VcpMobilePlugin.kt::checkAllPermissions()
val batteryOptimizationIgnored = pm.isIgnoringBatteryOptimizations(activity.packageName)
result.put("battery", batteryOptimizationIgnored)  // Boolean
```

> 注意：此处 JSON key 也为 `"battery"`，但语义是"是否已忽略电池优化"，与 `BatteryStatusManager` 的 `"level"` / `"isPowerSaveMode"` 不同。

### 6.2 请求忽略电池优化

```kotlin
// VcpMobilePlugin.kt::requestAndroidPermission("battery")
val intent = Intent(Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS).apply {
    data = Uri.parse("package:${activity.packageName}")
}
startActivityForResult(invoke, intent, "onBatteryOptimizationResult")
```

失败时降级到 `Settings.ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS` 通用设置页。

## 7. 错误处理

| 错误来源 | 错误信息 | 触发条件 |
|----------|----------|----------|
| `getBatteryLevel` | 返回 `-1` | `IntentFilter` 注册失败、`level`/`scale` 非法 |
| `isPowerSaveMode` | 返回 `false` | 任意 Settings 读取异常（安全降级） |
| `getBatteryStatus` (Kotlin) | `invoke.reject(e.message)` | `BatteryStatusManager` 抛出未预期异常 |
| `get_battery_status` (Rust) | `"Plugin handle not initialized"` | 插件句柄未在 `setup` 中挂载 |
| `get_battery_status` (Rust) | `"run_mobile_plugin failed: ..."` | JNI 桥接调用 Kotlin 方法失败 |
| sysfs 路径 (`battery.rs`) | `"电池信息不可用"` | `/sys/class/power_supply/battery/` 不存在或全部节点不可读 |

### 8.2 Kotlin：Settings Key 对照表

| Key | 命名空间 | ROM | 值语义 |

### 8.3 Kotlin：BatteryManager 常量

| 常量 | 说明 |

| `BatteryManager.EXTRA_LEVEL` | 当前电量原始值 |
| `BatteryManager.EXTRA_SCALE` | 电量满值（通常为 100） |
