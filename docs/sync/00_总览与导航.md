# 总览与导航

## 1. 知识库概览

本文档集是 **VCPMobile 同步模块（Sync Module）** 的完整技术知识库，覆盖移动端（Rust/Tauri）与桌面端（Node.js/VCPChat 插件）的双向增量同步体系。

| 指标 | 数值 |
|------|------|
| 总文档数 | **20** 份（00 导航 + 01–15 正文 + 附录 A–D） |
| 总行数 | **~11,600** 行 |
| 覆盖代码文件 | **30+** 个 Rust / Node.js 源文件 |
| 协议版本 | V2（三阶段增量同步 + 双哈希体系） |
| 适用版本 | 1.0.3 |

### 适用读者

| 读者类型 | 阅读目标 |
|---------|---------|
| **AI Agent** | 作为同步模块的物理记忆，用于代码生成、故障诊断与重构决策 |
| **Rust 开发者** | 理解移动端 Pipeline、Executor、Manifest、Hash 体系的实现细节 |
| **Node.js 插件开发者** | 理解桌面端索引、Diff 引擎、传输层路由与实体生命周期 |
| **维护者与架构评审者** | 掌握端到端数据流、冲突仲裁策略、性能瓶颈与边界行为 |

## 4. 阅读路径推荐

### 4.1 新手上路路径（第一次接触同步模块）

**目标**：建立端到端概念模型，不求细节，只求全貌。

```
步骤1: 01_架构总览与设计理念.md（理解双轨三层 + 三阶段协议）
    ↓
步骤2: 04_同步协议详解.md（理解 WS + HTTP 双通道交互）
    ↓
步骤3: 05_流水线与阶段控制.md（理解移动端内部状态机）
    ↓
步骤4: 附录D_同步场景流程图集.md（用具体场景验证理解）
```

预计阅读时间：**60–90 分钟**

---

### 4.2 问题排查路径（遇到具体问题时）

**目标**：快速定位根因，最小化无关阅读。

```
A. 同步失败/断开连接
   → 13（看日志）→ 05（看 Phase 状态）→ 11（看冲突规则）

B. 数据不一致（某条消息/Agent 双端不同）
   → 03（检查哈希冒泡）→ 06（检查 Manifest 比对）→ 07（检查 Diff 范围）
   → 11（检查冲突仲裁）→ 13（对照日志时间戳）

C. 同步性能差（慢或卡顿）
   → 12（性能优化策略）→ 08（检查批量尺寸与并发）→ 10（检查传输模式）

D. 新增实体后同步不生效
   → 15（开发指南）→ 02（检查 DTO 与类型）→ 03（检查哈希计算）
   → 14（检查双端映射）
```

---

### 4.3 扩展开发路径（需要修改同步逻辑时）

**目标**：保证修改在双端一致、不破坏现有协议。

```
阶段1: 需求分析
   → 01（确认是否涉及架构变动）→ 15（确认扩展点）

阶段2: 类型与数据模型
   → 02（修改/新增类型）→ 14（确认双端映射）

阶段3: 协议与流水线
   → 04（确认消息/端点变更）→ 05（确认 Phase 插入点）→ 附录A/B（更新参考表）

阶段4: 实现与验证
   → 03（确认哈希与变更检测）→ 08/09（确认执行器/插件实现）
   → 11（确认冲突与边界）→ 12（确认性能影响）→ 13（确认可观测性）
```

## 5. 术语速查表

| 术语（中文） | 英文 | 说明 |
|-------------|------|------|
| 双轨三层 | Double-Track 3-Tier | 移动端内部 Rust/IPC/Vue 三层 + 双端 WS/HTTP 双通道隔离 |
| 三阶段同步协议 | Three-Phase Sync Protocol | Phase 1 Owner Metadata → Phase 2 Topic Metadata → Phase 3 Messages |
| 清单 | Manifest | 双端交换的数据指纹集合，用于"先比对后传输" |
| 差异计算 | Diff | 比对双端 Manifest 后生成的 Pull/Push/Delete 动作列表 |
| 哈希体系 | Hash Hierarchy | L1 config_hash → L2 content_hash → L3 aggregated_hash |
| Merkle Root | Merkle Root | 子实体哈希集合排序后连接再哈希的聚合指纹 |
| 哈希冒泡 | Hash Bubble Up | 子实体变更后向上层重新计算聚合哈希的级联过程 |
| 最后写入胜出 | LWW (Last-Write-Wins) | 以绝对时间戳 `ts` 为仲裁依据的冲突解决策略 |
| 软删除 | Soft Delete | 以 `deleted_at` 时间戳标记删除，物理保留数据 |
| 双哈希 | Dual-Hash | V2 协议中 config_hash 与 content_hash 分离设计 |
| 流水线 | Pipeline | 将同步流程分解为可管理、可观测的 Phase 序列 |
| 执行器 | Executor | PullExecutor / PushExecutor / DeleteExecutor 的统称 |
| 数据传输对象 | DTO | 跨端同步的最小字段子集，排除本地 UI 状态 |
| 换行分隔 JSON | NDJSON | 每行一个独立 JSON 对象，用于消息流式传输 |
| 幂等键 | Idempotency Key | 确保同一操作只执行一次的令牌 |
| 写意图锁 | Write Intent Lock | 防止文件监听器与同步写入冲突的标记机制 |
| 数据库写入队列 | DbWriteQueue | 移动端 SQLite 串行化写入与事务合并机制 |
| 信号量 | Semaphore | `NetworkAwareSemaphore`，控制同步并发数以避免 OOM |
| 会话 | Session | 一次完整的 WS 连接生命周期（握手到断开） |
| 阶段 | Phase | 会话内的逻辑分段：owner_metadata / topic_metadata / messages |
| 操作 | Operation | Phase 内的最小原子单元（单条实体处理） |
| 索引数据库 | Index DB | 桌面端 `sync_state.db`，用于快速 Diff 与哈希比对 |
| 全量重置 | Full Reset | 当哈希永久分叉时，清空移动端数据库重新全量同步 |
| 确定性哈希 | Deterministic Hash | 稳定 JSON 序列化后计算的 SHA-256，消除实现差异 |
| 粗到精 | Coarse-to-Fine | Phase 2.5 话题级排除 → Phase 3 逐消息比对的策略 |
