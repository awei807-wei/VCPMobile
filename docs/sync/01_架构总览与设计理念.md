# 架构总览与设计理念

## 1. 引言

### 1.1 文档定位

本文档是 **VCPMobile 同步模块（Sync Module）** 的顶层架构知识库文档，面向需要理解同步系统全貌的开发者、维护者和架构评审者。它回答的核心问题是：

- 移动端与桌面端如何通过双通道（WebSocket + HTTP）协同完成数据同步？
- 三阶段同步协议（Three-Phase Sync Protocol）的每一步如何保证数据一致性？
- V2 版本相比 V1 做了哪些关键改进，解决了哪些结构性问题？

### 1.2 目标读者

| 读者类型 | 阅读重点 |
|---------|---------|
| Rust 后端开发者 | 第 2、4、7 节 — 双端分层与模块边界 |
| 前端/移动端开发者 | 第 6 节 — 数据流与 UI 刷新 |
| 插件开发者（Node.js） | 第 2、4 节 — 桌面端职责与协议 |
| 架构评审者 | 第 3、5 节 — 设计原则与演进路线 |

### 1.3 与 `SYNC_ARCHITECTURE.md` 的关系

`docs/SYNC_ARCHITECTURE.md` 是**协议规范级**文档（接口契约），定义 DTO、消息格式、通信协议。

本文档是**架构知识级**文档，侧重：双端代码的实际分层与职责映射、状态机的工程实现视角、V1→V2 的演进脉络与决策依据。

两者互补：规范文档回答"传什么"，本文档回答"怎么传、为什么这样传"。

## 2. 双端架构概览

### 2.1 整体分层架构

VCPMobile 同步系统采用**双轨三层（Double-Track 3-Tier）**物理隔离架构：Rust 核心层、Tauri IPC 桥接层、Vue 3 渲染层在移动端内部隔离；移动端与桌面端之间通过 WebSocket（信令）+ HTTP（数据）双通道隔离。

## 3. 同步的核心设计原则

### 3.1 确定性哈希（Deterministic Hashing）

**原则**：所有实体的指纹必须通过**规范 JSON 序列化（Canonical JSON Serialization）** + **SHA-256** 生成，确保相同数据在任何时刻、任何设备上产生完全相同的哈希值。

**代码体现**：

- 桌面端 DTO 提取后，按键名排序再序列化：`computeDtoHash(dto, fields)` (`VCPMobileSync/index.js`)
- 移动端消息指纹计算：`compute_message_fingerprint(content, attachment_hashes)`，附件哈希先排序再拼接 (`docs/SYNC_ARCHITECTURE.md`)
- 双哈希体系（V2）：`config_hash`（配置指纹）与 `content_hash`（子实体 Merkle 根）分离 (`sync_service.rs`)

**伪代码示例（规范序列化核心逻辑）**：

```rust
fn stable_stringify(value: &Value) -> String {
    match value {
        Value::Object(map) => {
            let mut keys: Vec<&String> = map.keys().collect();
            keys.sort(); // 键名排序是确定性的根基
            let mut res = String::from("{");
            for (i, k) in keys.iter().enumerate() {
                if i > 0 { res.push(','); }
                res.push_str(&format!("\"{}\":{}", k, stable_stringify(map.get(k))));
            }
            res.push('}');
            res
        }
        Value::Array(arr) => {
            // 数组维持原有顺序（消息队列不可乱序）
            let mut res = String::from("[");
            for (i, v) in arr.iter().enumerate() {
                if i > 0 { res.push(','); }
                res.push_str(&stable_stringify(v));
            }
            res.push(']');
            res
        }
        // 基础类型直接输出...
        _ => value.to_string(),
    }
}
```

**意义**：哈希不仅是数据完整性校验，更是**分布式环境下判断"数据是否相同"的唯一仲裁标准**。规范序列化通过"键排序 + 统一空值表示"消除了字段顺序、精度等噪声。

### 3.2 Merkle 树聚合（Merkle Tree Aggregation）

**原则**：层级结构（Agent/Group → Topic → Message）采用 Merkle 树（Merkle Tree）风格的层级哈希聚合，父节点的哈希由所有子节点哈希有序拼接后计算得出。

**代码体现**：

```rust
// 伪代码：Agent Root Hash 计算 (sync_hash/HashAggregator)
fn compute_agent_root_hash(agent_id) -> String {
    let config_hash = get_config_hash(agent_id);
    let topic_hashes = get_topic_content_hashes(agent_id); // ORDER BY topic_id ASC
    let mut all_hashes = vec![config_hash];
    all_hashes.extend(topic_hashes);
    all_hashes.sort(); // 确定性排序
    sha256_concat(all_hashes)
}
```

- 桌面端：`computeAggregatedHashes(db, logger)` 预加载所有 Topic 并按 Parent ID 分组，消除 N+1 查询 (`VCPMobileSync/index.js`)
- 移动端：`HashAggregator::bubble_topic_hash()` → `bubble_agent_hash()` (`sync_service.rs`)

**意义**：任何子节点的变更仅需重新计算其直接父节点，无需遍历整棵树，实现 O(log n) 级变更检测。同时，双哈希体系让"配置变更"与"内容变更"在父节点层面即可区分，避免不必要子树遍历。

### 3.3 幂等操作（Idempotent Operations）

**原则**：所有可能因网络抖动而重试的写操作必须具备幂等性（Idempotency），即多次执行与一次执行效果相同。

**代码体现**：

| 机制 | 实现位置 | 说明 |
|------|---------|------|
| 幂等键（Idempotency Key） | HTTP Header `x-idempotency-key` | `SHA256(action + type + id + minute_timestamp)`，同分钟内重试生成相同键 (`SYNC_ARCHITECTURE.md`) |
| 附件去重上传 | `SyncState.uploaded_hashes: Arc<RwLock<HashSet<String>>>` | 单次同步会话内已上传的附件哈希被记录，避免跨 Topic 重复上传 (`sync_service.rs`) |
| Phase3Tracker | `completed: Mutex<HashSet<String>>` | 同一 Topic 的完成状态用集合去重，防止双重递减下溢 (`sync_service.rs`) |
| DbWriteQueue 顺序消费 | `mpsc::UnboundedSender<DbWriteTask>` | 同一实体的多次写任务按队列顺序执行，后到达的任务覆盖先到达的 (`db_write_queue.rs`) |

**幂等键生成逻辑**：

```rust
fn generate_idempotency_key(action: &str, entity_type: &str, id: &str) -> String {
    let mut hasher = Sha256::new();
    hasher.update(action.as_bytes());
    hasher.update(entity_type.as_bytes());
    hasher.update(id.as_bytes());
    let minute = chrono::Utc::now().timestamp() / 60; // 按分钟分桶
    hasher.update(minute.to_string().as_bytes());
    format!("{:x}", hasher.finalize())
}
```

**按分钟分桶的设计意图**：同一分钟内重试生成相同键（不重复数据），跨分钟重试生成新键（防止服务端永久缓存）。

### 3.4 软删除（Soft Deletion）

**原则**：删除操作不物理移除数据，而是标记 `deleted_at` 时间戳，使删除信息本身可同步。

**代码体现**：

- 移动端：`DeleteExecutor::soft_delete_agent/group/topic/avatar()` (`sync_service.rs`)
- Diff 动作扩展：`DiffAction::Delete { deleted_at }` 和 `DiffAction::PushDelete { deleted_at }` (`SYNC_ARCHITECTURE.md`)
- 桌面端：`deleteEntity()` / `deleteMessage()` 写入删除索引 (`VCPMobileSync/index.js`)
- 垃圾回收：桌面端每小时执行 `cleanupOldDeletedRecords()`；移动端在 `Finalize` 阶段批量清理

**同步语义**：

```
远程 deleted_at > 本地 deleted_at  →  本地执行 DELETE
本地 deleted_at > 远程 deleted_at  →  发送 PUSH_DELETE 通知远程
```

### 3.5 网络感知并发（Network-Aware Concurrency）

**原则**：并发度不应是硬编码常数，而应根据设备能力动态调整，在"充分利用带宽"与"保护移动端 UI 响应"之间取得平衡。

**代码体现**：

```rust
// NetworkAwareSemaphore (sync_service.rs)
let cores = std::thread::available_parallelism().unwrap_or(4);
let concurrency = ((cores as f32) * 1.5).clamp(6.0, 12.0) as usize;
let semaphore = Arc::new(Semaphore::new(concurrency));
```

- IO 密集型任务的理论平衡点为 `cores × 1.5`
- 移动端严格限制上限为 **12**，防止过量并发导致 SQLite WAL 锁竞争和 UI 卡顿
- HTTP Pull/Push 均通过 `semaphore.acquire().await` 获取许可

## 4. 三阶段同步协议

### 4.1 协议总览

同步会话（Sync Session）是一次完整的端到端数据对齐过程，由移动端主动发起，通过 WebSocket 连接桌面端插件，按严格顺序执行以下相位（Phase）：

| 相位 | 名称 | 核心目标 | 数据粒度 | 代码触发点 |
|------|------|---------|---------|-----------|
| Phase 1 | Owner Metadata | 同步 Agent、Group、Avatar | 实体级 | `SyncCommand::StartManualSync` |
| Phase 2 | Topic Metadata | 拉取/推送有变动的 Owner 下的 Topic 配置 | Topic 级 | `PipelineCommand::StartTopicMetadata` |
| Phase 2.5 | Topic Validation (V2) | 双哈希批量比对，精准筛选需同步消息的 Topic | Hash 级 | `PipelineCommand::StartTopicValidation` |
| Phase 3 | Messages | 逐 Topic 同步消息内容 | 消息级 | `PipelineCommand::StartMessages` |
| Finalize | 收尾 | 全局 Hash 冒泡、计数更新、关闭连接 | 事务级 | `PipelineCommand::Finalize` |

### 4.2 PipelinePhase 状态机

```
                    ┌─────────────────┐
                    │      Idle       │
                    └────────┬────────┘
                             │ start_manual_sync
                             ▼
                    ┌─────────────────┐
                    │  Phase 1        │
                    │  Owner Metadata │
                    └────────┬────────┘
                             │ on_owner_metadata_done()
                             ▼
                    ┌─────────────────┐
                    │  Phase 2        │
                    │  Topic Metadata │
                    └────────┬────────┘
                             │ on_topic_metadata_pull_done()
                             ▼
                    ┌─────────────────┐
                    │  Phase 2.5      │
                    │  Topic Validation│  ← V2 新增，不改变 PipelinePhase 枚举值
                    └────────┬────────┘
                             │ on_topic_validation_done()
                             ▼
                    ┌─────────────────┐
                    │  Phase 3        │
                    │  Messages       │
                    └────────┬────────┘
                             │ on_messages_done()
                             ▼
                    ┌─────────────────┐
                    │    Completed    │
                    └─────────────────┘
```

状态机定义位于 `sync_pipeline/pipeline.rs`，枚举值为：

```rust
pub enum PipelinePhase {
    Idle,
    Phase2TopicMetadata { progress: PhaseProgress },
    Phase3Messages { progress: PhaseProgress },
    Completed,
}
```

注意：Phase 2.5 在逻辑上独立，但**不新增枚举变体**，而是复用 Phase 2 的底层状态，通过 `PipelineCommand::StartTopicValidation` 命令触发。

```
    │── 所有 topic mark_completed()              │
    │── pending_msg_topics.completed.len == total│
    │── Pipeline::on_messages_done()             │
    │                                            │
    │============= FINALIZE ====================│
    │                                            │
    │── write_queue.flush()                      │
    │── 批量 UPDATE topics SET msg_count, updated_at
    │── HashAggregator::bubble_topic_hash()      │
    │── HashAggregator::bubble_agent_hash()      │
    │── HashAggregator::bubble_group_hash()      │
    │── tx.commit()                              │
    │                                            │
    │── emit "vcp-sync-completed"                │
    │── PHASE_COMPLETED ────────────────────────>│
    │── ws_stream.close()                        │
    │                                            │
```

### 4.4 Phase 1: Owner Metadata 详解

**目标**：同步 Agent、Group、Avatar 三类实体的配置与存在性。

**流程**：

1. **Hash 初始化**：`HashInitializer::ensure_all_agent_hashes(&pool)` 和 `ensure_all_group_hashes(&pool)` 确保所有本地实体的 `config_hash` 已计算（对旧数据兜底）(`sync_service.rs`)
2. **Manifest 构建**：`Phase1Metadata::build_phase1_manifests(&pool)` 构建 Agent、Group、Avatar 三个 Manifest (`sync_service.rs`)
3. **Diff 计算**：桌面端对比本地 `entity_index` 与移动端 Manifest，输出 `SYNC_DIFF_RESULTS`
4. **任务执行**：移动端将 Diff 结果分类为 batch_pull、push_topics、other_items 三类并行执行 (`sync_service.rs`)
5. **V2 关键改进**：桌面端在 Diff 中标记 `mismatchedContent`，移动端据此填充 `changed_owners: HashSet<String>`，用于 Phase 2 的**靶向 Topic 同步** (`sync_service.rs`)

**任务分类策略**：移动端收到 Diff 后按三类批处理：batch_pull（Agent/Group chunk=50, Topic chunk=1000）、push_topics_to_fetch（先查 DB 再推送）、other_items（Avatar 单独处理）(`sync_service.rs`)

**Diff 动作语义**：

| 动作 | 移动端行为 | 桌面端行为 |
|------|-----------|-----------|
| `SKIP` | 无操作 | 无操作 |
| `PULL` | HTTP GET 拉取实体数据 | 返回实体 DTO |
| `PUSH` | HTTP POST 推送实体数据 | 写入本地文件/索引 |
| `DELETE` | 本地 soft_delete_* | 已在远程删除，无需动作 |
| `PUSH_DELETE` | 发送 `SYNC_DELETE_NOTIFY` | 标记本地删除索引 |

### 4.5 Phase 2: Topic Metadata 详解

**目标**：仅同步**内容有变动的 Owner** 下的 Topic 配置，而非全量 Topic。

**V2 改进**：在 V1 中，Topic Metadata 是全量比对；在 V2 中，利用 Phase 1 筛选出的 `changed_owners`，移动端仅构建这些 Owner 下的 Topic Manifest：

```rust
Phase1Metadata::build_targeted_topic_manifest(&db.pool, &changed_owners)
```

这极大减少了 Phase 2 的数据传输量，尤其是当用户拥有大量 Topic 但仅少数 Agent/Group 有变更时。

### 4.6 Phase 2.5: Topic Validation 详解（V2 新增）

**目标**：在消息同步前，通过**双哈希（Dual-Hash）**精准判断哪些 Topic 真正需要进入 Phase 3。

**问题背景**：V1 中 Topic 聚合哈希变化即触发全量消息同步，但配置变更（如 title）可能不影响消息。

**V2 解决方案**：

移动端发送 `SYNC_TOPIC_HASH_BATCH_V2`，携带每个 Topic 的**双哈希**：

```json
{
  "type": "SYNC_TOPIC_HASH_BATCH_V2",
  "hashes": {
    "topic_001": { "configHash": "abc...", "contentHash": "def..." },
    "topic_002": { "configHash": "ghi...", "contentHash": "jkl..." }
  }
}
```

桌面端逐 Topic 比对：
- `configHash` 不同 → 配置需同步（Phase 2 已处理）
- `contentHash` 不同 → **消息需同步**，加入 `changedTopics`

桌面端返回：

```json
{ "type": "SYNC_TOPIC_HASH_RESULTS", "changedTopics": ["topic_001", ...] }
```

**意义**：Phase 3 的输入从"所有 Topic"缩小到"消息真正有差异的 Topic"，避免大量无效的消息哈希比对。

### 4.7 Phase 3: Messages 详解

**目标**：同步真正有差异的 Topic 下的消息。`build_diff_batches()` 按 `MAX_MESSAGES_PER_BATCH = 10000` 分批，保证同一 Topic 不跨批次：

**执行顺序**：桌面端返回 `SYNC_DIFF_RESULTS_BATCH` 后，移动端严格执行 **Push 先于 Pull**：

1. `PushExecutor::push_messages_batch()` — 先推送本地新增/更新的消息
2. `PullExecutor::pull_messages_batch()` — 再拉取远程新增/更新的消息

此顺序确保"两端都有新消息的 Topic"（push_pull）在合并时不会丢失移动端数据。

**附件补传机制**：桌面端返回 `neededAttachmentHashes`，移动端通过 `uploaded_hashes` 会话级去重后补传 (`sync_service.rs`)

**Phase3Tracker 完成判定**：

```rust
struct Phase3Tracker {
    completed: Mutex<HashSet<String>>, // 已完成的 Topic 集合
    modified:  Mutex<HashSet<String>>, // 实际发生数据变更的 Topic 集合
    total: AtomicUsize,
}
```

- `mark_modified()`：在 Push/Pull 成功后调用，记录该 Topic 需要在 Finalize 阶段重新冒泡哈希
- `mark_completed()`：首次标记返回 true，重复标记返回 false，彻底杜绝 `AtomicU32` 双重递减导致的下溢
- 当 `completed.len() == total.load()` 时，自动触发 `SyncCommand::Finalize`

### 4.8 Finalize 详解

**目标**：完成所有数据写入后，确保层级哈希一致性，并向前端报告完成。

**流程**：

1. **DbWriteQueue Flush**：强制落盘所有缓冲的写任务 (`sync_service.rs`)
2. **批量更新 Topic 元数据**：一条 SQL 更新所有受影响 Topic 的 `msg_count` 和 `updated_at` (`sync_service.rs`)
3. **层级冒泡（Hash Bubble Up）**：
   - 对每个修改过的 Topic：`HashAggregator::bubble_topic_hash()`
   - 收集受影响的 Agent/Group：`HashAggregator::bubble_agent_hash()` / `bubble_group_hash()`
4. **事务提交**：所有冒泡操作在一个 SQLite 事务内完成
5. **前端事件**：发射 `vcp-sync-completed`，携带 `agentsChanged: true` 等标志，触发前端刷新 (`sync_service.rs`)
6. **连接关闭**：发送 `PHASE_COMPLETED` 后主动关闭 WebSocket

## 5. V1→V2 的演进改进清单

| 改进项 | V1 行为 | V2 行为 | 解决的问题 | 对应代码文件 |
|--------|--------|--------|-----------|-------------|
| **双哈希体系** | Topic 仅有单一 `aggregated_hash` | Topic 分离为 `configHash` + `contentHash`（双哈希） | 配置变更与消息变更无法区分，导致不必要的消息同步 | `sync_service.rs`, `VCPMobileSync/index.js` |
| **Phase 2.5 引入** | Phase 2 后直接全量进入 Phase 3 | 新增 Phase 2.5，通过双哈希精准筛选需同步消息的 Topic | 大量 Topic 仅配置变化时仍全量比对消息，浪费带宽与 CPU | `sync_service.rs`, `sync_pipeline/pipeline.rs` |
| **靶向 Topic 同步** | Phase 2 构建全量 Topic Manifest | Phase 2 仅构建 `changed_owners` 下的 Topic Manifest | Owner 数量多但变更少时，Phase 2 数据传输量过大 | `sync_service.rs` |
| **Owner 变更追踪** | Phase 1 不记录内容变更的 Owner | Phase 1 Diff 中通过 `mismatchedContent` 标记，填充 `changed_owners: HashSet` | 无法关联 Owner 变更与 Topic 变更，导致 Phase 2 盲目全量 | `sync_service.rs` |
| **消息分批 Diff** | 单批次发送所有 Topic 的消息哈希 | 按 `MAX_MESSAGES_PER_BATCH = 10000` 分批发送 | 超大 Topic 的 WS payload 导致内存峰值与传输失败 | `sync_service.rs` |
| **Phase3Tracker** | 使用 `AtomicU32` 计数，存在双重递减下溢风险 | 使用 `HashSet<String>` 去重，防下溢 | 并发完成回调导致计数器下溢，同步流程提前结束 | `sync_service.rs` |
| **看门狗机制** | 无 | 每 10 秒检查 pending 计数，60 秒停滞则强制推进相位 | 网络异常或任务丢失导致同步永久卡死 | `sync_service.rs` |
| **Push 先于 Pull** | Pull 与 Push 无严格顺序 | Phase 3 强制先执行 Push，再执行 Pull | push_pull Topic 在并发执行时可能出现数据覆盖 | `sync_service.rs` |
| **版本握手** | 无版本校验 | 连接后发送 `VERSION_CHECK`，不兼容则中断并提示用户 | 移动端与桌面端插件版本不一致导致协议解析错误 | `sync_service.rs`, `VCPMobileSync/index.js` |
| **批量元数据推送** | 逐个 Topic PUSH | 批量查询后分块推送（chunk_size=1000） | Topic 数量大时 HTTP 请求过多，网络开销高 | `sync_service.rs` |
| **聚合哈希 N+1 优化** | 逐 Entity 查询子 Topic 哈希 | 预加载所有 Topic 并按 Parent ID 分组到 Map | `computeAggregatedHashes` 在 Entity 数量大时性能极差 | `VCPMobileSync/index.js` |
| **Topic 空哈希对齐** | 缺失 `aggregated_hash` 的 Topic 为空字符串 | 统一写入 `computeAggregatedHash([])` 的标准空值 | 空 Topic 的哈希值两端不一致，导致虚假差异 | `VCPMobileSync/index.js` |
| **附件去重上传** | 每次 Push 都上传附件 | `uploaded_hashes: HashSet<String>` 会话级去重 | 同一附件被多个 Topic 引用时重复上传 | `sync_service.rs` |

## 8. 关键设计决策记录

### 8.1 为什么用 WebSocket + HTTP 双通道，而非纯 WebSocket？

WebSocket 负责**信令与实时通知**（Manifest、Diff 请求、相位推进），HTTP 负责** bulk 数据传输**（实体 JSON、消息数组、附件二进制）。分离的原因：

1. **二进制传输效率**：附件通过 HTTP 的 `application/octet-stream` 直接流式传输，避免 Base64 编解码开销
2. **消息大小限制**：WebSocket payload 过大可能导致内存峰值；HTTP 可配合 chunked transfer 和流式解析
3. **幂等性与重试**：HTTP 请求天然支持标准重试机制（idempotency key + status code），WebSocket 的消息确认需自定义

### 8.2 为什么移动端主动关闭连接，而非等待桌面端？

`Finalize` 阶段由移动端主动发射 `vcp-sync-completed` 事件并关闭 WebSocket。设计原因是：

- 桌面端返回的 `PHASE_COMPLETED` 在部分网络环境下可能丢失
- 移动端已完成所有本地事务（hash 冒泡、db commit），可以独立判定"同步完成"
- 前端 UI 不应依赖不可靠的外部信号来解除加载状态

### 8.3 为什么 Phase 2.5 不改变 PipelinePhase 枚举？

Phase 2.5（Topic Validation）在逻辑上是独立相位，但工程上复用 `Phase2TopicMetadata` 枚举值。原因：

- Phase 2.5 不产生新的"进度条阶段"，从前端视角仍属于"Topic 同步"
- 减少状态机复杂度，避免引入仅用于内部跳转的枚举变体
- 通过 `PipelineCommand::StartTopicValidation` 命令即可实现控制流分离

### 8.4 为什么看门狗（Watchdog）的触发条件是 60 秒？

看门狗每 10 秒检查一次 `pending_tasks` 计数，若连续 6 次（60 秒）无变化则判定为死锁：

```
10s → 检查 pending (stuck_count=1, 警告)
20s → 检查 pending (stuck_count=2)
30s → 检查 pending (stuck_count=3)
40s → 检查 pending (stuck_count=4)
50s → 检查 pending (stuck_count=5)
60s → 检查 pending (stuck_count=6, 强制推进相位)
```

60 秒的选取依据：批量操作可能需 30-40 秒，6 次采样可过滤偶发抖动，同时不让用户等待超过 1 分钟。

### 8.5 为什么桌面端使用 SQLite 而非内存 Map 做索引？

桌面端插件需要处理数千个 Agent/Group、数万个 Topic 和数十万条消息的索引。使用 SQLite 而非纯内存结构的原因：

1. **进程重启恢复**：重启后无需重新扫描 AppData
2. **复杂查询支持**：SQL 层完成 GROUP BY / ORDER BY
3. **增量更新**：chokidar 只需更新变更条目
4. **与移动端对齐**：两端共享 SQLite 调试经验

| 英文术语 | 中文翻译 | 定义 |
|---------|---------|------|
| Sync Session | 同步会话 | 一次从连接到完成的完整同步过程 |

| Idempotency Key | 幂等键 | 基于 action + type + id + 分钟时间戳的 SHA-256 值 |
| Targeted Sync | 靶向同步 | V2 特性，仅同步有变动的 Owner 的 Topic |
