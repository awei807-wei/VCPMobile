# 数据模型与类型系统

## 1. 引言

本文档定义 VCPMobile（移动端）与 VCPChat + VCPMobileSync 插件（桌面端）之间同步协议涉及的全部数据结构、类型契约与持久化布局。它是理解三阶段增量同步协议（指纹广播 → 差异协商 → 数据传输）的物理基础。

**文档范围**：
- Rust 端同步核心类型（`sync_types.rs`、`sync_dto.rs`）
- Node.js 端 DTO 白名单与默认值工厂（`dto/*.js`、`config/defaults.js`）
- 移动端 SQLite 数据库 Schema（`db_manager.rs`）
- 桌面端 AppData 目录文件布局
- 双端字段映射、默认值对齐策略

**阅读前提**：建议先通读 `docs/SYNC_ARCHITECTURE.md` 了解三阶段协议的整体流程。

## 2. 核心类型定义（Rust 端）

### 2.1 哈希与指纹基础设施

同步协议以内容的**确定性哈希（Deterministic Hash）**作为唯一真理标识。Rust 端提供以下核心函数：

| 函数签名 | 说明 | 来源 |
|---------|------|------|
| `compute_deterministic_hash<T: Serialize>(data: &T) -> String` | 将任意可序列化对象转为稳定 JSON 后计算 SHA-256 | `sync_types.rs` |
| `compute_merkle_root(hashes: Vec<String>) -> String` | 将多组哈希按字典序排序后连接，再计算聚合哈希（Merkle Root） | `sync_types.rs` |
| `stable_stringify(value: &serde_json::Value) -> String` | 递归 JSON 序列化，保证 Object 键按字典序输出、数组顺序不变，消除不同实现间的序列化差异 | `sync_types.rs` |

> **关键约束**：`stable_stringify` 是双端哈希一致性的根基。桌面端 Node.js 在计算 DTO 哈希时，必须通过 `JSON.stringify(obj, Object.keys(obj).sort())` 对齐此行为。

### 2.2 `SyncDataType` — 同步实体类型枚举

```rust
#[serde(rename_all = "lowercase")]
pub enum SyncDataType {
    Agent,    // 智能体配置
    Group,    // 群组配置
    Avatar,   // 头像二进制
    Topic,    // 话题元数据
    Message,  // 消息历史
}
```

| 变体 | 序列化值 | 说明 |
|------|---------|------|
| `Agent` | `"agent"` | 对应移动端 `agents` 表 / 桌面端 `Agents/{id}/config.json` |
| `Group` | `"group"` | 对应移动端 `groups` 表 / 桌面端 `AgentGroups/{id}/config.json` |
| `Avatar` | `"avatar"` | 对应移动端 `avatars` 表 / 桌面端 `avatar.{ext}` 文件 |
| `Topic` | `"topic"` | 对应移动端 `topics` 表 / 桌面端 `config.json` 内 `topics` 数组 |
| `Message` | `"message"` | 对应移动端 `messages` 表 / 桌面端 `history.json` |

### 2.3 `EntityState` — 状态向量（State Vector）

`EntityState` 是阶段一指纹广播的**最小可识别单元**，移动端与桌面端互相交换的全量/增量清单均由该结构组成。

| 字段 | Rust 类型 | JSON 键 | Option | 说明 |
|------|----------|---------|--------|------|
| `id` | `String` | `id` | 否 | 实体唯一标识。Agent/Group 为自身 ID；Topic 为 topic ID；Avatar 为 owner_id |
| `hash` | `String` | `hash` | 否 | 内容指纹（SHA-256）。V2 协议中 Agent/Group 优先使用 `config_hash` 和 `content_hash` 的聚合值 |
| `config_hash` | `Option<String>` | `configHash` | 是 | 配置内容指纹（V2 优化），用于区分配置变更与内容变更 |
| `content_hash` | `Option<String>` | `contentHash` | 是 | 内容聚合指纹（V2 优化），代表旗下话题/消息是否有变动 |
| `ts` | `i64` | `ts` | 否 | 绝对时间戳 / 逻辑时钟，LWW（Last-Write-Wins）裁决标准 |
| `deleted_at` | `Option<i64>` | `deletedAt` | 是 | 软删除时间戳。非空表示该实体已被逻辑删除，用于双向删除同步 |
| `owner_type` | `Option<String>` | `ownerType` | 是 | 仅用于 `topic` 类型，区分 `"agent"` 和 `"group"`，指导路由到 `AgentTopicSyncDTO` 或 `GroupTopicSyncDTO` |

> **设计意图**：`config_hash` 与 `content_hash` 的分离，使得 Agent 修改系统提示词（System Prompt）不会触发该 Agent 下所有 Topic 的消息重新比对，显著降低协商带宽。

### 2.4 `SyncManifest` — 同步清单

```rust
pub struct SyncManifest {
    pub data_type: SyncDataType,
    pub items: Vec<EntityState>,
}
```

| 字段 | 说明 |
|------|------|
| `data_type` | 清单内所有实体的统一类型。一次 Manifest 只携带一种类型 |
| `items` | 该类型的全量或增量 `EntityState` 数组 |

移动端发送 `SyncManifest` 给桌面端时，通常按 `Agent` → `Group` → `Topic` → `Message` → `Avatar` 的顺序分帧传输；桌面端同理回传。

### 2.5 同步 DTO 结构详表

所有 DTO 均定义于 `sync_dto.rs`，采用 `#[serde(rename_all = "camelCase")]`，保证 JSON 字段名为驼峰式（CamelCase），与桌面端 JavaScript 惯例对齐。

#### 2.5.1 `AgentSyncDTO`

```rust
pub struct AgentSyncDTO {
    pub name: String,
    pub system_prompt: String,
    pub model: String,
    pub temperature: f64,
    pub context_token_limit: i32,
    pub max_output_tokens: i32,
    pub stream_output: bool,
}
```

| 字段 | Rust 类型 | JSON 键 | Option | 说明 |
|------|----------|---------|--------|------|
| `name` | `String` | `name` | 否 | 智能体显示名称 |
| `system_prompt` | `String` | `systemPrompt` | 否 | 系统提示词 |
| `model` | `String` | `model` | 否 | 模型标识，如 `"gemini-2.5-flash"` |
| `temperature` | `f64` | `temperature` | 否 | 采样温度，范围通常为 0.0–2.0 |
| `context_token_limit` | `i32` | `contextTokenLimit` | 否 | 上下文 Token 上限 |
| `max_output_tokens` | `i32` | `maxOutputTokens` | 否 | 单次输出 Token 上限 |
| `stream_output` | `bool` | `streamOutput` | 否 | 是否启用流式输出 |

**`From<&AgentConfig>` 实现**：直接从 `agent_types.rs` 的 `AgentConfig` 提取上述 7 个字段，不涉及 `id`、`topics`、`current_topic_id` 等非同步字段。

#### 2.5.2 `GroupSyncDTO`

```rust
pub struct GroupSyncDTO {
    pub name: String,
    pub members: Vec<String>,
    pub mode: String,
    pub member_tags: Option<serde_json::Value>,
    pub group_prompt: Option<String>,
    pub invite_prompt: Option<String>,
    pub use_unified_model: bool,
    pub unified_model: Option<String>,
    pub tag_match_mode: Option<String>,
    pub created_at: i64,
}
```

| 字段 | Rust 类型 | JSON 键 | Option | 说明 |
|------|----------|---------|--------|------|
| `name` | `String` | `name` | 否 | 群组名称 |
| `members` | `Vec<String>` | `members` | 否 | 成员 Agent ID 列表 |
| `mode` | `String` | `mode` | 否 | 发言模式：`sequential`、`naturerandom`、`invite_only` |
| `member_tags` | `Option<Value>` | `memberTags` | 是 | 成员标签映射（`agentId → tags`） |
| `group_prompt` | `Option<String>` | `groupPrompt` | 是 | 群组全局提示词 |
| `invite_prompt` | `Option<String>` | `invitePrompt` | 是 | 邀请发言提示词模板 |
| `use_unified_model` | `bool` | `useUnifiedModel` | 否 | 是否强制使用统一模型 |
| `unified_model` | `Option<String>` | `unifiedModel` | 是 | 统一模型名称 |
| `tag_match_mode` | `Option<String>` | `tagMatchMode` | 是 | 标签匹配模式：`strict`、`fuzzy` |
| `created_at` | `i64` | `createdAt` | 否 | 创建时间戳（毫秒） |

#### 2.5.3 `AgentTopicSyncDTO` 与 `GroupTopicSyncDTO`

桌面端不区分 `AgentTopic` 与 `GroupTopic` 的存储结构（均在父级 `config.json` 的 `topics` 数组内），但移动端在 `topics` 表中通过 `owner_type` 区分。因此同步协议使用两个独立 DTO，消除分类讨论。

**`AgentTopicSyncDTO`**：

| 字段 | Rust 类型 | JSON 键 | Option | 说明 |
|------|----------|---------|--------|------|
| `id` | `String` | `id` | 否 | Topic 唯一 ID |
| `name` | `String` | `name` | 否 | Topic 显示名称 |
| `created_at` | `i64` | `createdAt` | 否 | 创建时间戳 |
| `locked` | `bool` | `locked` | 否 | 是否锁定（默认 `true`） |
| `unread` | `bool` | `unread` | 否 | 是否未读（默认 `false`） |
| `owner_id` | `String` | `ownerId` | 否 | 所属 Agent ID |

**`GroupTopicSyncDTO`**：

| 字段 | Rust 类型 | JSON 键 | Option | 说明 |
|------|----------|---------|--------|------|
| `id` | `String` | `id` | 否 | Topic 唯一 ID |
| `name` | `String` | `name` | 否 | Topic 显示名称 |
| `created_at` | `i64` | `createdAt` | 否 | 创建时间戳 |
| `owner_id` | `String` | `ownerId` | 否 | 所属 Group ID |

> **差异说明**：`GroupTopicSyncDTO` 不含 `locked` 与 `unread` 字段，因为桌面端群组话题不存储这两项状态，且移动端群组话题的 UI 不展示锁定/未读标记。

#### 2.5.4 消息与附件 DTO

| DTO | 关键字段 | 说明 | 来源 |
|-----|---------|------|------|
| `AttachmentSyncDTO` | `type`, `name`, `size`, `hash`, `status`, `extracted_text`, `image_frames`, `created_at` | 附件元数据，`hash` 为内容摘要，用于去重与路径推导 | `sync_dto.rs` |
| `UserMessageSyncDTO` | `id`, `role`, `name`, `content`, `timestamp`, `attachments` | 用户发送的消息，`role` 通常为 `"user"` | `sync_dto.rs` |
| `AgentMessageSyncDTO` | `id`, `role`, `name`, `content`, `timestamp`, `agent_id`, `is_thinking`, `finish_reason`, `avatar_color` | Agent 回复消息，必须携带 `agent_id` 与 `avatar_color` | `sync_dto.rs` |
| `GroupMessageSyncDTO` | `id`, `role`, `name`, `content`, `timestamp`, `agent_id`, `group_id`, `topic_id`, `is_group_message`, `avatar_color` | 群组消息，额外携带 `group_id` 与 `topic_id` | `sync_dto.rs` |

> **附件处理**：移动端附件以 `attachments` 表为真理之源（BLOB 存储）；桌面端附件存储于 `UserData/attachments/{hash}.{ext}`，通过 `file://` URI 引用。

## 3. DTO 字段白名单

桌面端采用**字段白名单**策略：仅提取和同步预定义字段，忽略桌面端特有的扩展字段（如 `disableCustomColors`、`useThemeColorsInChat`、`creatorSource` 等）。这保证了即使桌面端配置结构演进，也不会破坏移动端的解析稳定性。

### 3.1 Agent 同步字段白名单

| Rust 字段名（`AgentSyncDTO`） | Node.js 字段名（`AGENT_SYNC_FIELDS`） | 类型 | 桌面端默认值 | 说明 |
|-----------------------------|-------------------------------------|------|------------|------|
| `name` | `name` | `String` | `"Unnamed Agent"` | 显示名称 |
| `system_prompt` | `systemPrompt` | `String` | `""` | 系统提示词 |
| `model` | `model` | `String` | `"gemini-2.5-flash"` | 模型标识 |
| `temperature` | `temperature` | `f64` | `1.0` | 采样温度 |
| `context_token_limit` | `contextTokenLimit` | `i32` | `1000000` | 上下文 Token 限制 |
| `max_output_tokens` | `maxOutputTokens` | `i32` | `64000` | 最大输出 Token 数 |
| `stream_output` | `streamOutput` | `bool` | `true` | 是否流式输出 |

**类型归一化**：桌面端 `extractAgentDTO` 在提取字段时执行强制类型转换：`temperature` → `parseFloat`，`contextTokenLimit` / `maxOutputTokens` → `parseInt`。这是为了对齐 Rust 端的强类型约束，防止 `"1"`（字符串）导致哈希不一致。

### 3.2 Group 同步字段白名单

| Rust 字段名（`GroupSyncDTO`） | Node.js 字段名（`GROUP_SYNC_FIELDS`） | 类型 | 桌面端默认值 | 说明 |
|-----------------------------|-------------------------------------|------|------------|------|
| `name` | `name` | `String` | `"Unnamed Group"` | 群组名称 |
| `members` | `members` | `Vec<String>` | `[]` | 成员 ID 列表 |
| `mode` | `mode` | `String` | `"sequential"` | 发言模式 |
| `member_tags` | `memberTags` | `Option<Value>` | `{}` | 成员标签映射 |
| `group_prompt` | `groupPrompt` | `Option<String>` | `""` | 群组提示词 |
| `invite_prompt` | `invitePrompt` | `Option<String>` | 长模板字符串 | 邀请发言模板 |
| `use_unified_model` | `useUnifiedModel` | `bool` | `false` | 使用统一模型 |
| `unified_model` | `unifiedModel` | `Option<String>` | `""` | 统一模型名称 |
| `tag_match_mode` | `tagMatchMode` | `Option<String>` | `"strict"` | 标签匹配模式 |
| `created_at` | `createdAt` | `i64` | `Date.now()` | 创建时间戳 |

### 3.3 Agent Topic 同步字段白名单

| Rust 字段名（`AgentTopicSyncDTO`） | Node.js 字段名（`AGENT_TOPIC_SYNC_FIELDS`） | 类型 | 桌面端默认值 | 说明 |
|-----------------------------------|-------------------------------------------|------|------------|------|
| `id` | `id` | `String` | — | Topic ID |
| `name` | `name` | `String` | — | Topic 名称 |
| `created_at` | `createdAt` | `i64` | — | 创建时间戳（`parseInt` 归一化） |
| `locked` | `locked` | `bool` | `true` | 是否锁定 |
| `unread` | `unread` | `bool` | `false` | 是否未读 |
| `owner_id` | `ownerId` | `String` | — | 所属 Agent ID（注入） |

### 3.4 Group Topic 同步字段白名单

| Rust 字段名（`GroupTopicSyncDTO`） | Node.js 字段名（`GROUP_TOPIC_SYNC_FIELDS`） | 类型 | 桌面端默认值 | 说明 |
|-----------------------------------|-------------------------------------------|------|------------|------|
| `id` | `id` | `String` | — | Topic ID |
| `name` | `name` | `String` | — | Topic 名称 |
| `created_at` | `createdAt` | `i64` | — | 创建时间戳（`parseInt` 归一化） |
| `owner_id` | `ownerId` | `String` | — | 所属 Group ID（注入） |

> **Owner 注入机制**：桌面端 Topic DTO 提取函数 `extractTopicDTO(topic, ownerId, ownerType)` 在调用时将父级实体 ID 作为 `ownerId` 注入，因为桌面端 `config.json` 内的 `topics` 数组项本身不存储 `ownerId`。

## 4. 移动端数据库 Schema

移动端使用 SQLite 作为本地持久化引擎，启用 WAL（Write-Ahead Logging）模式以降低并发写入锁竞争。所有同步涉及的表定义于 `db_manager.rs` 的 `setup_tables` 函数。

### 4.1 `avatars` — 全局多态头像表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `owner_type` | `TEXT` | `NOT NULL`, PK(1) | 实体类型：`agent`、`group`、`user`、`system` |
| `owner_id` | `TEXT` | `NOT NULL`, PK(2) | 对应实体 UUID 或固定值 `user_avatar` |
| `avatar_hash` | `TEXT` | `NOT NULL` | SHA-256 摘要，用于 WebSocket 快速 Diff |
| `mime_type` | `TEXT` | `NOT NULL` | 如 `image/webp`、`image/png` |
| `image_data` | `BLOB` | `NOT NULL` | 物理二进制数据（真理之源） |
| `dominant_color` | `TEXT` | — | 预计算主色调（rgb/hex） |
| `updated_at` | `BIGINT` | `NOT NULL` | 逻辑时钟 |

### 4.2 `agents` — 智能体配置表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `agent_id` | `TEXT` | `PRIMARY KEY` | Agent 唯一标识 |
| `name` | `TEXT` | `NOT NULL` | 显示名称 |
| `system_prompt` | `TEXT` | `NOT NULL DEFAULT ''` | 系统提示词 |
| `model` | `TEXT` | `NOT NULL` | 模型标识 |
| `temperature` | `REAL` | `NOT NULL DEFAULT 1` | 采样温度 |
| `context_token_limit` | `INTEGER` | `NOT NULL DEFAULT 0` | 上下文 Token 限制 |
| `max_output_tokens` | `INTEGER` | `NOT NULL DEFAULT 0` | 最大输出 Token 数 |
| `stream_output` | `INTEGER` | `NOT NULL DEFAULT 1` | 是否流式（SQLite 无原生 bool，0/1） |
| `config_hash` | `TEXT` | `NOT NULL DEFAULT ''` | 配置内容指纹（V2） |
| `content_hash` | `TEXT` | `NOT NULL DEFAULT ''` | 聚合指纹：Config + Topics（V2） |
| `current_topic_id` | `TEXT` | — | 最后活跃话题 ID |
| `updated_at` | `BIGINT` | `NOT NULL` | 更新时间戳 |
| `deleted_at` | `BIGINT` | — | 软删除时间戳（非空 = 已删除） |

### 4.3 `groups` — 群组配置表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `group_id` | `TEXT` | `PRIMARY KEY` | Group 唯一标识（通常为 `____123` 格式） |
| `name` | `TEXT` | `NOT NULL` | 群组名称 |
| `mode` | `TEXT` | `NOT NULL DEFAULT 'sequential'` | 发言模式 |
| `group_prompt` | `TEXT` | — | 群组全局提示词 |
| `invite_prompt` | `TEXT` | — | 邀请发言提示词 |
| `use_unified_model` | `INTEGER` | `NOT NULL DEFAULT 0` | 使用统一模型（0/1） |
| `unified_model` | `TEXT` | — | 统一模型名称 |
| `tag_match_mode` | `TEXT` | — | 标签匹配模式 |
| `config_hash` | `TEXT` | `NOT NULL DEFAULT ''` | 配置内容指纹（V2） |
| `content_hash` | `TEXT` | `NOT NULL DEFAULT ''` | 聚合指纹（V2） |
| `current_topic_id` | `TEXT` | — | 最后活跃话题 ID |
| `created_at` | `BIGINT` | `NOT NULL DEFAULT 0` | 创建时间戳 |
| `updated_at` | `BIGINT` | `NOT NULL` | 更新时间戳 |
| `deleted_at` | `BIGINT` | — | 软删除时间戳 |

### 4.4 `group_members` — 群组成员关联表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `group_id` | `TEXT` | `NOT NULL`, PK(1) | 所属群组 ID |
| `agent_id` | `TEXT` | `NOT NULL`, PK(2) | 成员 Agent ID |
| `member_tag` | `TEXT` | — | 成员标签 |
| `sort_order` | `INTEGER` | `NOT NULL DEFAULT 0` | 展示排序 |
| `updated_at` | `BIGINT` | `NOT NULL` | 更新时间戳 |

> **归一化策略**：桌面端 `GroupConfig.members` 是扁平字符串数组；移动端拆分为 `group_members` 关联表，支持未来扩展成员级元数据（如 `member_tag`、`sort_order`）。同步时双向转换。

### 4.5 `topics` — 话题元数据表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `topic_id` | `TEXT` | `PRIMARY KEY` | Topic 唯一 ID |

| `owner_id` | `TEXT` | `NOT NULL` | 所有者 ID |
| `title` | `TEXT` | `NOT NULL` | Topic 显示名称（对应 DTO 的 `name`） |
| `created_at` | `BIGINT` | `NOT NULL` | 创建时间戳 |
| `updated_at` | `BIGINT` | `NOT NULL` | 更新时间戳 |
| `locked` | `INTEGER` | `NOT NULL DEFAULT 1` | 是否锁定（Agent Topic 有效） |
| `unread` | `INTEGER` | `NOT NULL DEFAULT 0` | 是否未读（Agent Topic 有效） |
| `unread_count` | `INTEGER` | `NOT NULL DEFAULT 0` | 未读消息计数 |
| `msg_count` | `INTEGER` | `NOT NULL DEFAULT 0` | 消息总数 |

## 5. 桌面端文件布局

桌面端（VCPChat）采用**文件系统即数据库**的架构：每个 Agent/Group 是一个独立目录，配置以 JSON 文件形式存储，消息历史按 Topic 分目录存储。VCPMobileSync 插件在启动时扫描这些文件并构建内存索引（SQLite），用于快速 Diff 与哈希计算。

### 5.1 根目录结构

假设项目根目录为 `G:\VCPChat`，则 AppData 根为：

```
G:\VCPChat\AppData\
├── Agents\                    # 智能体目录
│   └── {agentId}\
│       ├── config.json        # Agent 完整配置（含 topics 数组）
│       └── avatar.{ext}       # 头像文件（png/jpg/jpeg/webp/gif）
├── AgentGroups\               # 群组目录
│   └── {groupId}\
│       ├── config.json        # Group 完整配置（含 topics 数组）
│       └── avatar.{ext}       # 群组头像
└── UserData\                  # 用户数据目录
    ├── {agentId}\             # 按 Agent ID 组织消息历史
    │   └── topics\            # 话题目录
    │       └── {topicId}\     # 各话题子目录（default 话题被同步忽略）
    │           └── history.json   # 消息数组（JSON 格式）
    ├── attachments\           # 附件物理存储
    │   └── {hash}.{ext}       # 以内容哈希命名的附件文件
    ├── user_avatar.png        # 用户头像
    ├── forum.config.json      # 论坛配置（不参与同步）
    ├── emoticon_library.json  # 表情包库（不参与同步）
    ├── settings.json          # 全局设置（选择性同步）
    └── ...                    # 其他系统文件
```

### 5.2 各文件详细说明

| 文件/目录路径 | 存储内容 | 同步类型 | 对应移动端表 |
|-------------|---------|---------|------------|
| `Agents/{id}/config.json` | Agent 完整配置 + topics 数组 | Agent + Topic | `agents`、`topics` |
| `Agents/{id}/avatar.{ext}` | Agent 头像二进制 | Avatar | `avatars` |
| `AgentGroups/{id}/config.json` | Group 完整配置 + topics 数组 | Group + Topic | `groups`、`topics` |
| `AgentGroups/{id}/avatar.{ext}` | Group 头像二进制 | Avatar | `avatars` |
| `UserData/{agentId}/topics/{topicId}/history.json` | 消息数组（按 Topic 隔离） | Message | `messages` |
| `UserData/attachments/{hash}.{ext}` | 附件物理文件 | Attachment | `attachments` |
| `UserData/user_avatar.png` | 用户头像 | Avatar | `avatars`（owner_type=user） |

### 5.3 `config.json` 内部结构

桌面端的 `config.json` 是**嵌套式**存储：Agent/Group 配置与其下属的 Topic 数组共存于同一文件。

```json
// Agents/{id}/config.json 示例（简化）
{
  "name": "Assistant",
  "systemPrompt": "You are a helpful assistant.",
  "model": "gemini-2.5-flash",
  "temperature": 1.0,
  "contextTokenLimit": 1000000,
  "maxOutputTokens": 64000,
  "streamOutput": true,
  "topics": [
    {
      "id": "topic_xxx",
      "name": "General Discussion",
      "createdAt": 1715500000000,
      "locked": true,
      "unread": false
    }
  ],
  "disableCustomColors": true,
  "useThemeColorsInChat": true
}
```

> **注意**：`id` 字段在 Agent `config.json` 中**不写入**（由目录名推导），但在 Group `config.json` 中**显式存储**。这是桌面端的历史设计差异，同步协议已通过 DTO 提取屏蔽此差异。

## 7. 默认值系统

默认值的一致性直接决定双端哈希是否匹配。如果移动端序列化时某字段缺失并使用 Rust 默认值，而桌面端提取 DTO 时使用不同的默认值，则同一实体将被误判为不一致，触发不必要的传输。

### 7.1 Rust 端默认值来源

### 7.3 两端默认值的差异与对齐策略

#### 7.3.1 已知差异清单

| 差异点 | Rust 端行为 | Node.js 端行为 | 影响 | 缓解措施 |
|--------|-----------|--------------|------|---------|
| `GroupConfig.name` 默认值 | `"Unnamed Group"` (`default_group_name`) | `"Unnamed Group"`（`GROUP_DEFAULTS`） | ~~已对齐~~ | `group_types.rs` 通过 `#[serde(default = "default_group_name")]` 对齐 |
| `GroupConfig.mode` 默认值 | `"sequential"` (`default_group_mode`) | `"sequential"` | ~~已对齐~~ | `group_types.rs` 通过 `#[serde(default = "default_group_mode")]` 对齐 |
| `GroupConfig.tagMatchMode` 默认值 | `Some("strict")` (`default_tag_match_mode`) | `"strict"` | ~~已对齐~~ | `group_types.rs` 通过 `#[serde(default = "default_tag_match_mode")]` 对齐 |
| `AgentConfig.systemPrompt` 动态生成 | `""`（静态） | `你是 ${name}。`（动态） | 桌面端新建 Agent 时 systemPrompt 不为空 | 该差异仅影响**桌面端新建**的实体；移动端上传时 `systemPrompt` 为完整字符串，桌面端原样接收 |
| `Topic.locked`（Rust `Topic` 结构体） | `false` | `true`（Agent Topic） | 仅影响未显式设置的场景 | `AgentTopicSyncDTO` 通过 `#[serde(default = "default_locked")]` 覆盖为 `true`，与数据库 `DEFAULT 1` 一致 |

#### 7.3.2 对齐策略：三层防御

1. **提取层对齐**：桌面端 `extractXxxDTO` 在白名单循环中，对任何缺失字段都使用 `?? DEFAULTS[field]` 填充，保证输出的 DTO 永远包含完整字段集。

2. **类型归一化对齐**：桌面端 `extractAgentDTO` 和 `extractGroupDTO` 在赋值前执行 `parseFloat` / `parseInt`，消除 JavaScript 弱类型导致的 `"1"` vs `1` 差异。

3. **序列化对齐**：两端哈希计算前均使用**键排序的稳定 JSON 序列化**（Rust 的 `stable_stringify`、Node.js 的 `Object.keys(obj).sort()`），消除字段顺序差异。

4. **结构体默认值对齐**（V0.9.13 修复）：`group_types.rs` 中 `GroupConfig` 的三个字段通过 `#[serde(default = "...")]` 显式绑定默认值函数：`name` → `"Unnamed Group"`、`mode` → `"sequential"`、`tag_match_mode` → `Some("strict")`，确保 JSON 反序列化缺失字段时与桌面端 `GROUP_DEFAULTS` 完全一致。

#### 7.3.3 新增字段的扩展规范

如需在同步协议中新增字段，必须遵循以下步骤：

1. **Rust 端**：在 `sync_dto.rs` 的对应 DTO 中添加字段，同步更新 `From` 实现。
2. **Node.js 端**：在 `config/defaults.js` 的对应 `DEFAULTS` 常量中添加默认值；在 `dto/*.js` 的白名单数组中添加字段名。
3. **移动端数据库**：如需本地持久化，在 `db_manager.rs` 的对应表定义中添加字段，并编写 `ALTER TABLE` 兼容语句用于存量 DB 升级。
4. **哈希验证**：修改后必须运行两端全量同步测试，验证相同配置的哈希值是否完全一致。

> **纪律**：`config/defaults.js` 文件头部的注释明确指出——"新增字段：只需在此文件添加，无需改动其他模块"。这是桌面端的设计承诺，但实际工程中仍需同步更新 Rust 端 DTO 定义，以维持双端契约一致。

#### 7.4 派生字段与同步边界：`avatarColor` / `avatarCalculatedColor`

`avatarCalculatedColor`（桌面端）与 `dominant_color`（手机端 `avatars` 表）是**从头像文件派生**的主色调值，不参与实体 DTO 同步。但 `avatarColor` 以特殊方式参与消息同步。

| 字段名 | 类型 | 同步范围 | 说明 |
|--------|------|----------|------|
| `avatarCalculatedColor` / `dominant_color` | 派生值 | **不参与实体同步** | 从头像二进制文件提取主色调，各端独立计算；头像文件本身通过 `SyncDataType::Avatar` 同步 |
| `avatarColor`（消息 DTO） | 消息字段 | **随消息同步，但不入库** | Push 时实时查询填充，Pull 时丢弃；属于渲染层而非持久层字段 |

##### 7.4.1 手机端 Push → 桌面端

```rust
// push_executor.rs:370-382
let avatar_color = query_avatar_color(&db.pool, &msg.agent_id.clone().unwrap_or_default())
    .await
    .unwrap_or("#6B7280".to_string());  // fallback 灰色
let dto = GroupMessageSyncDTO::from_message(msg, avatar_color);
```

- 每条 AI 消息推送前，实时查询 `avatars.dominant_color`（`owner_type='agent'`）
- 填充到 `AgentMessageSyncDTO.avatar_color` / `GroupMessageSyncDTO.avatar_color`
- 桌面端接收后直接取用：`desktopMsg.avatarColor = msg.avatarColor || "rgb(128, 128, 128)"`

##### 7.4.2 桌面端 Pull → 手机端

```rust
// pull_executor.rs:143-144
obj.remove("avatarUrl");
obj.remove("avatarColor");
```

- 桌面端 `history.json` 中原样包含 `avatarColor`，随消息流发送
- 手机端 Pull 时**主动移除**这两个字段
- 原因：`db_manager.rs:170` 明确注释 "messages 表 — 已移除冗余 `avatar_url` 和 `avatar_color`"
- 手机端渲染时从 `avatars` 表重新动态查询（`pre_renderer/shell_precomputer.rs:44-46`）

##### 7.4.3 设计理由

| 设计决策 | 理由 |
|----------|------|
| 颜色不随实体同步 | 颜色是从头像文件**派生**的，头像文件本身已通过 `/upload-avatar` / `/download-avatar` 同步；两端独立计算可保证各自渲染一致性 |
| 颜色随消息同步（单向） | 桌面端 `history.json` 需要自包含渲染信息，避免每次读取消息都去查头像文件再计算颜色 |
| 手机端移除后重新查询 | 手机端 `messages` 表刻意精简（无 avatar 字段），颜色在渲染 Pipeline 中从 `avatars` 表动态 join |

##### 7.4.4 已知不一致点

- **fallback 颜色格式**：手机端 `"#6B7280"`（hex），桌面端 `"rgb(128, 128, 128)"`（rgb）
- **影响**：格式不同但显示效果一致（均为灰色），对用户体验无实质影响
- **缓解**：如需严格对齐，可将手机端 fallback 统一为 `"rgb(107, 114, 128)"` 或桌面端统一为 `"#6B7280"`
