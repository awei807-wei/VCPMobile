# 哈希体系与变更检测

## 1. 引言

### 1.1 为什么需要哈希体系

在分布式同步（Distributed Synchronization）场景中，VCPMobile（手机端）与 VCPChat（桌面端）需要在弱网、异步、双写冲突的环境下保持数据一致性。传统的"全量对比"方案在消息量达到数万条时会产生不可接受的带宽与计算开销。

哈希体系（Hash Hierarchy）的核心设计目标如下：

| 目标 | 说明 |
|------|------|
| **秒级差异定位** | 通过分层哈希，在 O(log N) 层级内定位到具体变更实体 |
| **带宽最小化** | 阶段一（Phase 1）仅传输 64 字节的哈希指纹，而非完整实体数据 |
| **冲突可裁决** | 结合 LWW（Last-Write-Wins，最后写入胜出）时间戳与哈希对比，自动裁决双向冲突 |
| **离线可恢复** | 哈希值持久化存储，应用重启后无需重新计算全量哈希 |

### 1.2 不变性假设（Immutability Assumptions）

本哈希体系建立在以下不变性假设之上，任何违反这些假设的修改都会导致哈希冒泡（Bubble Up）失效：

1. **消息不可变**：消息一旦创建，其 `content` 和 `attachments` 不会被修改，仅支持软删除（`deleted_at` 标记）。
2. **Topic 元数据可变**：Topic 的 `name`、`locked`、`unread` 等字段支持修改，修改后必须重新计算 `config_hash`。
3. **Owner（Agent/Group）配置可变**：Agent 的 `model`、`temperature` 等字段修改后必须重新计算 `config_hash`。
4. **哈希字段自身不可参与哈希**：`config_hash`、`content_hash` 等字段**绝不**出现在被序列化的对象中，防止循环依赖。
5. **排序决定 Merkle Root**：同一组子实体的哈希，无论以何种顺序收集，最终 Merkle Root 相同——因为计算前会先排序。

> 重要：若在消息层面违反"不可变"假设（如直接修改历史消息内容），双端哈希将永久分叉，必须通过全量重置修复。

## 2. 三级 Hash 体系

VCP 同步协议采用三级哈希架构（Three-Tier Hash Architecture），每一级承担不同的职责，形成自底向上的保护链。

### 2.1 三级哈希概览

| 层级 | 字段名 | 中文名 | 定义 | 存储位置 |
|------|--------|--------|------|----------|
| L1 | `config_hash` | 配置哈希 | 实体自身属性的确定性指纹 | `agents.config_hash`、`topics.config_hash` |
| L2 | `content_hash` | 内容哈希 | 子实体集合的 Merkle 根（Merkle Root） | `topics.content_hash`、`agents.content_hash`、`groups.content_hash` |
| L3 | `aggregated_hash` | 聚合哈希 | `SHA256(config_hash + content_hash)`， Owner 的最终指纹 | 桌面端 `entity_index.aggregated_hash` |

### 2.2 层级关系图

```
+-------------------------------------------------------------+
|                     Agent / Group (Owner)                    |
|  +---------------------+    +-----------------------------+ |
|  |   config_hash       |    |        content_hash         | |
|  |  (自身配置指纹)      |    |   (旗下 Topic 的聚合指纹)    | |
|  +----------+----------+    +--------------+--------------+ |
|             |                              |                |
|             +--------------+---------------+                |
|                            |                                |
|  desktop: aggregated_hash = SHA256(config + content)        |
|  mobile:  各自独立存储 config_hash + content_hash           |
+-------------------------------------------------------------+
                             |
         +-------------------+-------------------+
         |                   |                   |
         v                   v                   v
    +---------+        +---------+        +---------+
    | Topic A |        | Topic B |        | Topic C |
    | config  |        | config  |        | config  |
    | content |        | content |        | content |
    +----+----+        +----+----+        +----+----+
         |                  |                  |
         v                  v                  v
    +----------+      +----------+      +----------+
    | Message  |      | Message  |      | Message  |
    |Fingerprint|     |Fingerprint|     |Fingerprint|
    +----------+      +----------+      +----------+
```

### 2.3 config_hash（配置哈希）

**定义**：实体自身属性的确定性 SHA-256 哈希，**不包含**任何子实体信息。它是实体本身的"身份证指纹"。

**计算方式**：
1. 提取实体的同步字段（白名单机制，排除 `id`、`owner_id` 等不参与同步的字段）
2. 对字段值进行类型归一化（如 `temperature` 统一保留 2 位小数）
3. 使用 stable stringify（确定性 JSON 序列化）生成标准化 JSON 字符串
4. 对字符串计算 SHA-256

**适用对象**：

| 实体类型 | 参与字段 | 计算函数 |
|----------|----------|----------|
| Agent | `name`, `systemPrompt`, `model`, `temperature`, `contextTokenLimit`, `maxOutputTokens`, `streamOutput` | `compute_agent_config_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:54`） |
| Group | `name`, `members`, `mode`, `memberTags`, `groupPrompt`, `invitePrompt`, `useUnifiedModel`, `unifiedModel`, `tagMatchMode`, `createdAt` | `compute_group_config_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:68`） |
| Agent Topic | `id`, `name`, `createdAt`, `locked`, `unread` | `compute_agent_topic_metadata_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:30`） |
| Group Topic | `id`, `name`, `createdAt` | `compute_group_topic_metadata_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:43`） |

> 注意：Topic 的 `config_hash` 计算时**排除 `owner_id`**，因为 Topic 的归属变更属于元数据变动，不应影响 Topic 自身的身份指纹。同时确保与桌面端字段白名单 `AGENT_TOPIC_SYNC_FIELDS` / `GROUP_TOPIC_SYNC_FIELDS` 严格一致。

### 2.4 content_hash（内容哈希）

**定义**：子实体集合的 Merkle Root（Merkle 树根哈希），代表该实体"旗下所有子项"的内容状态。

**计算方式**：
1. 收集所有**直属**子实体的哈希指纹（按固定规则排序）
2. 将所有子哈希字符串排序后，逐个（或连接后）输入 SHA-256
3. 若集合为空，返回 `""`（空字符串）

**适用对象**：

| 实体类型 | 子实体 | 排序规则 | 计算函数 |
|----------|--------|----------|----------|
| Topic | Messages | `timestamp ASC, msg_id ASC` | `compute_topic_root_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:86`） |
| Agent | Topics | `topic_id ASC` | `compute_agent_root_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:105`） |
| Group | Topics | `topic_id ASC` | `compute_group_root_hash`（`src-tauri/src/vcp_modules/sync_hash.rs:127`） |

**特殊设计——Agent/Group 的 content_hash**：

不同于 Topic 只收集消息指纹，Agent/Group 在计算 `content_hash` 时，同时收集每个 Topic 的 `config_hash` **和** `content_hash`，并将两者**交替排列**作为叶子节点（`src-tauri/src/vcp_modules/sync_hash.rs:118`）：

```
hashes = [topic1_config_hash, topic1_content_hash, topic2_config_hash, topic2_content_hash, ...]
```

这种设计确保：
- Topic 元数据变更（改名、锁定）会向上冒泡
- Topic 消息变更也会向上冒泡
- 两个变更不会相互掩盖

### 2.5 aggregated_hash（聚合哈希）

**定义**：桌面端特有的二级封装哈希，将 `config_hash` 与 `content_hash` 压缩为单一指纹，用于阶段一广播。

**计算公式**（桌面端 `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/hash.js:119`）：

```
aggregated_hash = SHA256( sorted_child_hashes.join("") )
```

> 双端差异：手机端**没有** `aggregated_hash` 这一概念。手机端在阶段一广播时直接发送独立的 `config_hash` 和 `content_hash`（`EntityState` 结构体，`src-tauri/src/vcp_modules/sync_types.rs:99`）。桌面端为了兼容旧索引结构，在 `entity_index` 表中维护 `aggregated_hash`，其语义等同于手机端的 `content_hash`（对 Owner 而言）或 `config_hash`（对 Topic 而言）。

### 2.6 桌面端与手机端字段映射

| 手机端字段 | 桌面端字段 | 语义对齐说明 |
|------------|------------|--------------|
| `agents.config_hash` | `entity_index.hash` (type=agent) | Agent 自身配置指纹 |
| `agents.content_hash` | `entity_index.aggregated_hash` (type=agent) | 旗下 Topic 聚合指纹 |
| `topics.config_hash` | `entity_index.hash` (type=topic) | Topic 元数据指纹 |
| `topics.content_hash` | `entity_index.aggregated_hash` (type=topic) | 消息聚合指纹 |

## 3. 确定性 JSON 序列化规则

双端哈希一致性完全依赖于**逐字节一致的 JSON 序列化**。任何字段顺序、空格、数字格式的差异都会导致哈希分裂。为此，双端均实现了自定义的 stable stringify（确定性 JSON 序列化）函数。

### 3.1 手机端 stable_stringify

实现位于 `src-tauri/src/vcp_modules/sync_types.rs:34`。

**核心规则**：

| 数据类型 | 序列化行为 | 与标准 JSON 的差异 |
|----------|-----------|-------------------|
| Object | 按键名字典序（A-Z）排列；键值对之间用 `,` 连接，无空格 | 标准 JSON 不保证字段顺序 |
| Array | 按元素顺序；元素之间用 `,` 连接，无空格 | 与标准 JSON 一致 |
| String | 使用 `serde_json::to_string(s)` 转义 | 与标准 JSON 一致 |
| Number | `n.to_string()` | 取决于 serde_json::Number 内部表示 |
| Bool | `true` / `false` | 与标准 JSON 一致 |
| Null | `null` | 与标准 JSON 一致 |

**关键代码逻辑**：

```rust
pub fn stable_stringify(value: &serde_json::Value) -> String {
    match value {
        serde_json::Value::Object(map) => {
            let mut keys: Vec<&String> = map.keys().collect();
            keys.sort();                          // ① 按键名排序
            let mut res = String::new();
            res.push('{');
            for (i, k) in keys.iter().enumerate() {
                if i > 0 { res.push(','); }       // ② 逗号分隔，无空格
                res.push_str(&format!("\"{}\":{}", k, stable_stringify(map.get(*k).unwrap())));
            }
            res.push('}');
            res
        }
        // ... 其他分支
    }
}
```

### 3.2 桌面端 stableStringify

实现位于 `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/hash.js:13`。

**核心规则**：

| 数据类型 | 序列化行为 |
|----------|-----------|
| Object | 键排序；`"key":value` 格式，无空格 |
| Array | `[elem1,elem2]` 格式，无空格 |
| String | `JSON.stringify(obj)` 转义 |
| Number | `obj.toString()`；**特殊**：`temperature` 字段且为整数值时强制 `toFixed(1)` |
| Bool | `true` / `false` |
| Null | `null` |

### 3.3 字段排序规则

双端均采用**字典序（Lexicographical Order）**对 Object 键排序：

```
// 原始对象
{ "z": 1, "a": 2, "m": 3 }

// stable stringify 后
{"a":2,"m":3,"z":1}
```

**嵌套对象**：递归地对每一层 Object 的键排序，Array 保持原有顺序不变。

### 3.4 数字格式化规则

这是双端对齐中最危险的区域。

#### 3.4.1 temperature 的特殊处理

`temperature` 字段在 Agent 配置中为浮点数，但桌面端（Electron/Node.js）使用 IEEE-754 f64，手机端 Rust 历史上曾使用 f32，存在精度差异。

**统一策略**：双端均对 `temperature` 执行**保留 2 位小数**的截断：

| 端 | 处理方式 | 代码位置 |
|----|----------|----------|
| 手机端 | `(dto.temperature * 100.0).round() / 100.0` | `sync_hash.rs:60` |
| 桌面端 | `Math.round(dto.temperature * 100) / 100` | `hash.js:106` |

**示例**：

| 原始值 | 统一后 | stable stringify 输出 |
|--------|--------|------------------------|
| `1.0` | `1.0` | 手机端 `"1"` 或 `"1.0"`；桌面端 `"1.0"`（特殊分支） |
| `0.66666` | `0.67` | `"0.67"` |
| `1.234` | `1.23` | `"1.23"` |

> 重要：`"1"` 与 `"1.0"` 的差异是双端对齐的已知风险点。手机端 `serde_json::Number::to_string()` 对整数形态的浮点数（如 `1.0`）可能输出 `"1"`，而桌面端通过 `temperature` 特殊分支强制输出 `"1.0"`。在实际运行中，由于 Rust 端通过 `json!` 宏插入的是 `f64` 类型，内部表示保留小数信息，因此通常输出 `"1.0"`。但**不应依赖此隐式行为**，未来应在 Rust 端也显式格式化到 2 位小数字符串。

#### 3.4.2 其他数字

对于非 `temperature` 的数字：

- **整数**：直接输出，无小数点（如 `"1000000"`）
- **浮点数**：`toString()` 自然输出（如 `"1.5"`）
- **大整数**：若超过 `f64` 安全范围，可能产生差异——同步协议中不涉及此类场景

### 3.5 null 值处理规则

| 端 | 行为 | 代码位置 |
|----|------|----------|
| 手机端 | `serde(skip_serializing_if = "Option::is_none")` 跳过字段；显式 `null` 值输出 `"null"` | `sync_dto.rs` 各字段注解 |
| 桌面端 | `computeDtoHash` 中 `val === undefined || val === null` 时跳过字段 | `hash.js:100` |

**关键对齐点**：桌面端的 `computeDtoHash` 在白名单过滤后，**完全跳过** `undefined` 和 `null` 值。这意味着如果某个可选字段为 `null`，它不会出现在序列化对象中。手机端通过 `Option<T>` 和 `skip_serializing_if` 实现相同语义。

**示例**：

```js
// Group 的 unifiedModel 为 null/undefined 时
// 双端序列化结果均不包含该键
{"name":"Test","members":[],"mode":"sequential",...}
```

### 3.6 数组与嵌套对象规则

- **数组（Array）**：保持元素顺序，元素之间用 `,` 连接，无空格。嵌套对象内部仍需递归排序键。
- **嵌套对象**：每一层独立排序，不受外层影响。

**示例**：

```json
// 原始对象
{
  "members": [{"id": "b"}, {"id": "a"}],
  "name": "Group"
}

// stable stringify 后（手机端 & 桌面端）
{"members":[{"id":"b"},{"id":"a"}],"name":"Group"}
// 注意：数组元素顺序不变，但每个元素的键已排序
```

### 3.7 双端对齐的注意事项

| 风险点 | 严重程度 | 当前状态 |
|--------|----------|----------|
| temperature 整数格式 `"1"` vs `"1.0"` | 高 | 桌面端已加特殊分支，手机端依赖 serde_json 隐式行为 |
| 字段白名单不一致 | 高 | 已显式枚举，双方需同步更新 |
| 默认值差异 | 中 | 桌面端 `extractAgentDTO` 使用 `AGENT_DEFAULTS` 填充缺失字段 |
| 字符串转义差异 | 低 | 双端均使用标准 JSON 字符串转义 |
| 布尔/Null 格式 | 低 | 完全一致 |
| 中文字符编码 | 低 | 双端均使用 UTF-8 |

## 4. 各类 Hash 计算详解

### 4.1 Agent config hash

**参与字段**（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/dto/agent.dto.js:8`）：

| 字段 | 类型 | 默认值 | 特殊处理 |
|------|------|--------|----------|
| `name` | String | `"Unnamed Agent"` | 无 |
| `systemPrompt` | String | `""` | 无 |
| `model` | String | `"gemini-2.5-flash"` | 无 |

```rust
let meta = serde_json::json!({

    "systemPrompt": &dto.system_prompt,
    "model": &dto.model,
    "temperature": (dto.temperature * 100.0).round() / 100.0,
    "contextTokenLimit": dto.context_token_limit,
    "maxOutputTokens": dto.max_output_tokens,
    "streamOutput": dto.stream_output,
});
compute_deterministic_hash(&meta)
```

**桌面端计算**（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/hash.js:95`）：

```js
const filtered = {};
AGENT_SYNC_FIELDS.forEach((field) => {
  const val = dto[field];
  if (val !== undefined && val !== null) {
    filtered[field] = val;
  }
});
if (typeof filtered.temperature === "number") {
  filtered.temperature = Math.round(filtered.temperature * 100) / 100;
}
return SHA256(stableStringify(filtered));
```

### 4.5 消息指纹（Message Fingerprint）

消息指纹是同步体系中最基础的哈希单元。每条消息的指纹基于其**内容**和**附件哈希**计算。

**参与字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `content` | String | 消息文本内容 |
| `attachmentHashes` | String[] | 附件的 SHA-256 哈希列表（可选） |

**关键规则**：
- 附件哈希按字典序排序后参与计算
- 若消息无附件，`attachmentHashes` 字段**不出现**在序列化对象中
- 附件哈希取自 `attachment.hash` 或 `attachment._fileManagerData.hash`

**手机端计算**（`src-tauri/src/vcp_modules/sync_hash.rs:11`）：

```rust
pub fn compute_message_fingerprint(content: &str, attachment_hashes: &[String]) -> String {
    let mut sorted_hashes = attachment_hashes.to_vec();
    sorted_hashes.sort();

    let mut fingerprint_map = serde_json::Map::new();
    fingerprint_map.insert("content".to_string(), serde_json::Value::String(content.to_string()));
    if !sorted_hashes.is_empty() {
        fingerprint_map.insert("attachmentHashes".to_string(), serde_json::to_value(sorted_hashes).unwrap());
    }

    compute_deterministic_hash(&serde_json::Value::Object(fingerprint_map))
}
```

**桌面端计算**（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/hash.js:65`）：

```js
function computeMessageFingerprint(msg) {
  const attachmentHashes = (msg.attachments || [])
    .map((att) => att.hash || att._fileManagerData?.hash || "")
    .filter((h) => !!h)
    .sort();

  const fingerprintObj = { content: msg.content || "" };
  if (attachmentHashes.length > 0) {
    fingerprintObj.attachmentHashes = attachmentHashes;
  }

  return crypto.createHash("sha256").update(stableStringify(fingerprintObj)).digest("hex");
}
```

## 5. Merkle Root 计算伪代码

Merkle Root（Merkle 树根哈希）在本协议中的实现是**简化版**：不构建二叉树，而是将所有叶子哈希排序后**串联计算单一 SHA-256**。

### 5.1 手机端 compute_merkle_root

```rust
// src-tauri/src/vcp_modules/sync_types.rs:22
pub fn compute_merkle_root(mut hashes: Vec<String>) -> String {
    if hashes.is_empty() {
        return "".to_string();
    }
    hashes.sort();                          // ① 排序（字典序）
    let mut hasher = Sha256::new();
    for h in hashes {
        hasher.update(h.as_bytes());        // ② 逐个 update
    }
    format!("{:x}", hasher.finalize())      // ③ 输出 hex
}
```

### 5.2 桌面端 computeAggregatedHash

```js
// VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/hash.js:119
function computeAggregatedHash(hashes) {
  if (!hashes || hashes.length === 0) {
    return "";
  }
  const sorted = [...hashes].sort();        // ① 排序（字典序）
  return crypto.createHash("sha256")
    .update(sorted.join(""))               // ② 连接后一次性 update
    .digest("hex");
}
```

### 5.3 双端等价性证明

SHA-256 的 `update` 操作具有**线性累积性**：

```
SHA256_update(a); SHA256_update(b); SHA256_update(c);
== SHA256_update(a + b + c);
```

因此，手机端的"逐个 update"与桌面端的"join 后一次 update"在数学上完全等价。只要**输入字符串集合相同且排序一致**，输出必定相同。

### 5.4 边界条件

| 场景 | 输入 | 输出 | 说明 |
|------|------|------|------|
| 空集合 | `[]` | `""` | 双端一致返回空字符串 |
| 单元素 | `["abc"]` | `SHA256("abc")` | |
| 重复元素 | `["a", "a"]` | `SHA256("aa")` | 重复元素保留 |
| 含空字符串 | `["", "b"]` | `SHA256("b")` | 空字符串不影响结果 |

## 6. Hash 冒泡（Bubble Up）机制

### 6.1 冒泡的定义

当底层数据发生变更时，其 Hash 变化必须**逐层向上传播**，直至 Owner 级别。这一机制称为 Hash 冒泡（Hash Bubble Up）。

### 6.2 冒泡触发条件

| 变更类型 | 受影响的 Hash | 冒泡路径 |
|----------|--------------|----------|
| 新建/删除消息 | 消息指纹 -> Topic content_hash | Topic -> Agent/Group content_hash |
| 修改 Topic 名称 | Topic config_hash | Topic -> Agent/Group content_hash |
| 修改 Topic locked/unread | Topic config_hash | Topic -> Agent/Group content_hash |
| 修改 Agent 配置 | Agent config_hash | 无需冒泡（自身变更） |
| 新建/删除 Topic | Topic config_hash + content_hash | Agent/Group content_hash |

### 6.3 冒泡流程图

```
消息变更（新增/删除/软删除）
    |
    v
+------------------------------------+
|  compute_message_fingerprint()     |
|  更新 messages.content_hash        |
+-------------+----------------------+
              |
              v
+------------------------------------+
|  compute_topic_root_hash()         |
|  收集该 Topic 下所有消息指纹        |
|  计算 Merkle Root                  |
|  更新 topics.content_hash          |
+-------------+----------------------+
              |
              v
+------------------------------------+
|  bubble_topic_hash()               |
|  重新计算 Topic config_hash        |
|  （元数据可能未变，但统一刷新）      |
+-------------+----------------------+
              |
              v
+------------------------------------+
|  compute_agent_root_hash()         |
|  或 compute_group_root_hash()      |
|  收集 Owner 下所有 Topic 的        |
|  config_hash + content_hash        |
|  计算 Merkle Root                  |
|  更新 agents.content_hash          |
|  或 groups.content_hash            |
+------------------------------------+
```

### 6.4 手机端冒泡实现

核心实现位于 `src-tauri/src/vcp_modules/sync_hash.rs:148`。

```rust
pub async fn bubble_from_topic(tx, topic_id) -> Result<(), String> {
    // 1. 更新 Topic 级别
    Self::bubble_topic_hash(tx, topic_id).await?;
    //    - 计算 topic content_hash（消息聚合）
    //    - 计算 topic config_hash（元数据）
    //    - 写入 topics 表

    // 2. 获取 Owner 信息
    let topic_row = sqlx::query("SELECT owner_id, owner_type FROM topics WHERE topic_id = ?")
        .bind(topic_id)
        .fetch_one(&mut **tx).await?;

    // 3. 向上冒泡到 Owner
    if owner_type == "agent" {
        Self::bubble_agent_hash(tx, &owner_id).await?;
    } else if owner_type == "group" {
        Self::bubble_group_hash(tx, &owner_id).await?;
    }
    Ok(())
}
```

### 6.5 性能优化——批量冒泡

在高频消息写入场景（如批量拉取历史消息），逐条冒泡会导致严重的 N+1 查询问题。手机端通过 `db_write_queue.rs` 实现**延迟冒泡**：

1. 批量写入消息时标记 `skip_bubble = true`
2. 写入完成后，收集所有受影响的 `topic_id` 去重
3. 对每个受影响的 Topic 执行一次批量冒泡

```rust
// db_write_queue.rs 简化逻辑
let mut affected_topics = HashSet::new();
// ... 批量写入消息，收集 affected_topics ...

// 批量冒泡
for topic_id in affected_topics {
    HashAggregator::bubble_from_topic(&mut tx, &topic_id).await?;
}
```

## 7. 双端 Hash 对齐的血泪史

### 7.1 temperature 精度差异（f32 vs f64）

**问题根源**：桌面端 JavaScript 的 Number 是 IEEE-754 f64，而手机端 Rust 早期版本曾使用 `f32` 存储 `temperature`。当值为 `0.7` 时：

| 类型 | 内存表示 | `*100/100` 后 |
|------|----------|---------------|
| f32 | `0.699999988079071` | `0.7` |
| f64 | `0.69999999999999996` | `0.7` |

看起来都是 `0.7`，但 serde_json 序列化时：
- f32 的 `0.7` 可能序列化为 `"0.7"`
- f64 的 `0.7` 可能序列化为 `"0.69999999999999996"`

**修复方案**：
1. 手机端将 `temperature` 字段统一改为 `f64`（`sync_dto.rs:17`）
2. 双端统一执行 `(val * 100).round() / 100` 截断
3. 桌面端对 temperature 整数值加 `toFixed(1)` 保护

### 7.2 序列化行为差异（serde_json vs JS JSON.stringify）

| 差异点 | serde_json（Rust） | JSON.stringify（JS） | 影响 |
|--------|-------------------|---------------------|------|
| 字段排序 | 不保证 | 不保证 | 双端均自定义 stable stringify |
| 浮点数精度 | 使用内部字符串表示 | 使用内部字符串表示 | temperature 已特殊处理 |
| 空字段 | `skip_serializing_if` | 无内置机制 | 桌面端 `computeDtoHash` 手动过滤 |
| undefined | 无此类型 | 被忽略或转为 `null` | 桌面端手动过滤 `undefined` |
| 特殊 Unicode | `\u{0000}` 转义 | `\u0000` 转义 | 无实际影响 |

### 7.3 字段白名单漂移

**问题**：桌面端配置文件中可能包含大量桌面端特有字段（如 `disableCustomColors`、`useThemeColorsInChat`、`creatorSource`），如果这些字段意外参与 Hash 计算，双端将永远不一致。

**修复方案**：
1. 手机端使用 `serde_json::json!` 宏**显式构造**参与 Hash 的对象（如 `sync_hash.rs:33`）
2. 桌面端使用 `AGENT_SYNC_FIELDS` / `GROUP_SYNC_FIELDS` **白名单过滤**（`dto/agent.dto.js:8`）
3. 任何新增同步字段必须**双端同步更新**白名单

### 7.4 默认值不一致

**问题**：桌面端旧配置可能缺失某些字段（如早期版本无 `streamOutput`），手机端读取数据库时这些字段有默认值。

**修复方案**：桌面端 `extractAgentDTO` 使用 `AGENT_DEFAULTS` 填充缺失字段（`dto/agent.dto.js:27`）：

```js
AGENT_SYNC_FIELDS.forEach((field) => {
  let val = config[field] ?? AGENT_DEFAULTS[field];
  // ...
  dto[field] = val;
});
```

### 7.5 如何确保逐字节一致

1. **白名单枚举**：禁止"序列化整个对象"，必须显式列出参与字段
2. **类型归一化**：数字类型在 DTO 提取阶段统一转换（`parseInt` / `parseFloat`）
3. **精度截断**：浮点数字段在 Hash 计算前统一截断到固定位数
4. **空值过滤**：`null` / `undefined` / `Option::None` 必须被跳过
5. **排序确认**：Object 键排序、Array 元素收集顺序必须文档化并双端对齐
6. **变更审查**：任何涉及 Hash 计算的代码修改必须通过 `pnpm check` 和双端集成测试

## 9. 遗留数据 Hash 初始化

### 9.1 问题背景

在 V0.9.13 之前的数据库或桌面端配置中，不存在 `config_hash` 和 `content_hash` 字段。首次启用同步时，必须为已有数据生成初始哈希，否则阶段一清单广播将发送空哈希，导致全量冲突。

### 9.2 手机端初始化策略

手机端在数据库 Schema 中已为 `config_hash` 和 `content_hash` 设置默认值 `''`（空字符串）。初始化流程：

1. **启动检测**：应用启动时检查是否存在 `config_hash = '' OR config_hash = 'PENDING'` 的 Agent/Group
2. **懒加载计算**：发现缺失时，读取完整配置并计算 Hash
3. **持久化写入**：将计算结果写回数据库
4. **Topic 级联**：Agent/Group Hash 初始化不自动级联到 Topic，Topic 的 Hash 在首次访问时独立计算

**核心实现**（`src-tauri/src/vcp_modules/sync_hash.rs:236`）：

```rust
pub struct HashInitializer;

impl HashInitializer {
    pub async fn ensure_agent_hashes(tx, agent_id) -> Result<(), String> {
        let row = sqlx::query("SELECT config_hash FROM agents WHERE agent_id = ?")
            .bind(agent_id)
            .fetch_optional(&mut **tx).await?;

        if let Some(r) = row {
            let config_hash: String = r.get("config_hash");
            if config_hash.is_empty() || config_hash == "PENDING" {
                let dto = Self::load_agent_dto(tx, agent_id).await?;
                let new_hash = HashAggregator::compute_agent_config_hash(&dto);
                sqlx::query("UPDATE agents SET config_hash = ? WHERE agent_id = ?")
                    .bind(&new_hash).bind(agent_id)
                    .execute(&mut **tx).await?;
            }
        }
        Ok(())
    }
}
```

### 9.3 批量初始化

对于已有大量数据的老用户，提供批量初始化入口（`sync_hash.rs:299`）：

```rust
pub async fn ensure_all_agent_hashes(pool: &sqlx::SqlitePool) -> Result<(), String> {
    let rows = sqlx::query(
        "SELECT agent_id FROM agents WHERE config_hash = '' OR config_hash IS NULL OR config_hash = 'PENDING'"
    ).fetch_all(pool).await?;

        let agent_id: String = row.get("agent_id");

}
```

### 9.4 桌面端初始化策略

桌面端在 `reconcileLocalFiles`（`index.js`）阶段完成初始化：

1. 启动时扫描 `AppData/Agents` 和 `AppData/AgentGroups` 目录
2. 读取每个 `config.json`，提取 DTO，计算 Hash
3. 将结果写入 `sync_state.db` 的 `entity_index` 表
4. 执行 `computeAggregatedHashes` 计算所有 Owner 的聚合哈希

### 9.5 首次同步的 Hash 对齐

当手机端与桌面端首次建立同步连接时：

1. 双方均已独立完成本地 Hash 初始化
2. 阶段一（Phase 1）交换 `EntityState` 清单
3. 若双方对同一实体的 `config_hash` 或 `content_hash` 不同，则进入阶段二拉取差异
4. **首次同步的特殊风险**：若初始化算法存在细微差异（如字段顺序、默认值处理），可能导致大规模假性冲突（False Conflicts）。建议在首次同步前，使用测试数据验证双端 Hash 一致性。

### 9.6 PENDING 标记机制

`PENDING` 是一个特殊标记值，用于指示某实体的 Hash 正在计算中或需要重新计算：

| 场景 | 行为 |
|------|------|
| 新建 Agent/Group | 初始写入时 `config_hash = 'PENDING'`，后台异步计算 |
| 配置更新 | 将 `config_hash` 设为 `'PENDING'`，触发重新计算 |
| 同步接收 | 接收对方实体后，标记 `PENDING` 以强制重新评估 |

## 附录 A：术语表

| 英文术语 | 中文翻译 | 定义 |
|----------|----------|------|
| config_hash | 配置哈希 | 实体自身属性的确定性 SHA-256 指纹 |
| content_hash | 内容哈希 | 子实体集合的 Merkle Root |
| aggregated_hash | 聚合哈希 | 桌面端使用的 `SHA256(config_hash + content_hash)` |
| fingerprint | 消息指纹 | 单条消息的内容 + 附件哈希的 SHA-256 |
| Merkle Root | Merkle 树根哈希 | 多个子哈希排序后串联计算的单一 SHA-256 |
| stable stringify | 确定性 JSON 序列化 | 按固定规则（字段排序、无空格、统一数字格式）生成标准化 JSON 字符串 |
| bubble up | 冒泡 | 子实体 Hash 变更向上传播到父实体的机制 |
| LWW | 最后写入胜出 | Last-Write-Wins，以时间戳为仲裁依据的冲突解决策略 |
| DTO | 数据传输对象 | Data Transfer Object，同步双方交换的标准化数据结构 |
| Phase 1 / 阶段一 | 清单交换阶段 | 同步协议的第一阶段，仅交换哈希指纹和时间戳 |
