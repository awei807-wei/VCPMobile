# 同步协议详解

## 1. 引言

### 1.1 协议概述

VCP Mobile 同步协议（以下简称本协议）是连接移动端（Mobile）与桌面端（Desktop）的数据同步桥梁，采用 WebSocket + HTTP REST 双协议架构（Dual-Protocol Architecture）。该设计兼顾了实时控制信令的低延迟需求与大容量数据传输的可靠性需求：

- WebSocket 层：承担同步会话的生命周期管理、阶段控制、差异比对指令交换等轻量级控制面职责。所有控制消息以 JSON 文本帧（Text Frame）传输。
- HTTP REST 层：承担实体数据、消息历史、附件文件等大容量载荷的传输职责。移动端使用 reqwest 客户端（`src-tauri/src/vcp_modules/sync/sync_service.rs`）向桌面端 Express 服务器发起调用。

双协议层共享同一套认证令牌（Sync Token），但传输方式存在差异：WebSocket 采用 Query Param 传递，HTTP 采用 Header 传递。

### 1.2 同步数据类型

本协议涉及五种核心同步数据类型（SyncDataType），定义于 `src-tauri/src/vcp_modules/sync/sync_types.rs`：

| 类型枚举 | 英文标识 | 说明 |
|---------|---------|------|
| Agent | agent | 智能体配置与元数据 |
| Group | group | 群组配置与元数据 |
| Avatar | avatar | 头像图片文件 |
| Topic | topic | 话题（会话）配置 |
| Message | message | 消息历史记录 |

### 1.3 核心设计哲学

- LWW（Last-Write-Wins，最后写入胜出）：以绝对时间戳 ts（毫秒级 Unix 时间戳）作为冲突裁决依据。
- 双哈希体系（Dual-Hash System）：V2 协议引入 configHash（配置指纹）与 contentHash（内容聚合指纹，即 Merkle Root）的分离设计，避免全量内容比对。contentHash 用于快速判定某 Agent/Group 旗下话题/消息是否发生深层变更，从而决定是否进入 Phase 3 消息同步。
- 分阶段流水线（Phased Pipeline）：同步过程被划分为四个严格顺序的阶段，通过 WebSocket 的 PHASE_START / PHASE_COMPLETED / PHASE_ACK 机制驱动流转。

## 2. 协议数据模型

### 2.1 实体状态向量（EntityState）

EntityState 是同步协议中最核心的数据结构，定义于 `src-tauri/src/vcp_modules/sync/sync_types.rs`。它代表任意同步实体在某一时刻的完整状态快照，是清单交换与差异比对的唯一依据。

| 字段 | 类型 | 序列化名 | 必填 | 说明 |
|------|------|---------|------|------|
| id | String | - | 是 | 实体唯一标识（如 agent_id、group_id、topic_id） |
| hash | String | - | 是 | 状态指纹（SHA-256 Hash）。V2 中对于 Agent/Group/Topic 等价于 configHash |
| config_hash | Option<String> | configHash | 否 | 配置内容指纹（V2 引入），代表该实体配置字段的哈希 |
| content_hash | Option<String> | contentHash | 否 | 内容聚合指纹（V2 引入），代表子实体集合的 Merkle Root |
| ts | i64 | - | 是 | 最后更新时间戳（毫秒级 Unix 时间戳），LWW 裁决依据 |
| deleted_at | Option<i64> | deletedAt | 否 | 软删除时间戳。存在该字段表示实体已被删除 |
| owner_type | Option<String> | ownerType | 否 | 仅 Topic 类型使用，区分 agent 和 group |

哈希计算规则：
- Agent/Group 配置哈希：提取 DTO 中的同步字段子集（AGENT_SYNC_FIELDS / GROUP_SYNC_FIELDS），通过确定性 JSON 序列化后计算 SHA-256（compute_deterministic_hash）。
- Topic 配置哈希：同理提取 AGENT_TOPIC_SYNC_FIELDS 或 GROUP_TOPIC_SYNC_FIELDS。
- Content Hash（聚合哈希）：对子实体哈希列表按字典序排序后连接，再次计算 SHA-256（compute_merkle_root）。Agent/Group 的 contentHash 为其旗下所有 Topic 的 configHash + contentHash 的聚合；Topic 的 contentHash 为其旗下所有消息的 contentHash 的聚合。

### 2.2 同步清单（SyncManifest）

SyncManifest 是移动端向桌面端发送的批量状态向量容器：

```rust
pub struct SyncManifest {
    pub data_type: SyncDataType,
    pub items: Vec<EntityState>,
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| data_type | SyncDataType | 清单中包含的实体类型 |
| items | Vec<EntityState> | 该类型的全量实体状态列表 |

Phase 1 通常发送 3 个 SyncManifest（agent、group、avatar），Phase 2 发送 1 个 SyncManifest（topic，且附带 targetedOwners 过滤）。

### 2.3 同步数据类型枚举（SyncDataType）

```rust
pub enum SyncDataType {
    Agent,
    Group,
    Avatar,
    Topic,
    Message,
}
```

序列化时采用 rename_all = lowercase，故 JSON 中对应的字符串值为 agent、group、avatar、topic、message。

## 3. 连接建立

### 3.1 WebSocket 连接流程

移动端在启动手动同步时，由 start_manual_sync 命令（`src-tauri/src/vcp_modules/sync/sync_service.rs`）触发同步会话。核心连接流程如下：

1. 读取配置：从 settings_manager 获取 sync_server_url（WebSocket 地址）、sync_http_url（HTTP 基地址）和 sync_token（同步令牌）。
2. 构造 WS URL：将 sync_token 以 Query Param 形式附加到 WebSocket URL 上，形如 ws://<host>:<port>?token=<sync_token>。
3. 建立 TCP 连接：通过 tokio_tungstenite::connect_async 发起异步握手。
4. 重试策略：连接失败时执行指数退避重试，最多 3 次，初始延迟 500ms，最大延迟 5s（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。

桌面端由 startWsServer（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/websocket.js`）在指定端口（默认 5975）监听 WebSocket 连接。

### 3.2 Token 验证机制

桌面端对同步令牌的验证存在两个入口，采用双轨验证策略：

WebSocket 入口（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/websocket.js`）：

| 位置 | 字段名 | 格式 |
|------|--------|------|
| Query Param | token | ws://host:port?token=<sync_token> |

验证逻辑提取桌面端源码核心逻辑：连接时解析 URL Query 中的 token 参数，与 syncToken 比对，不匹配则返回 WebSocket Close Code 4001。

HTTP 入口（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js`）：

| 优先级 | 位置 | 字段名 | 说明 |
|--------|------|--------|------|
| 1 | Header | x-sync-token | 自定义同步令牌头 |
| 2 | Header | Authorization | Bearer <sync_token> 标准格式 |
| 3 | Query Param | token | 兼容旧版或调试场景 |

移动端在发起 HTTP 请求时，会同时携带 x-sync-token 和 Authorization: Bearer <token> 两个头部（`src-tauri/src/vcp_modules/sync/sync_executor/push_executor.rs`），以确保桌面端无论采用哪种校验方式均可通过。

### 3.3 版本校验（VERSION_CHECK / VERSION_ACK）

连接建立后，同步会话的第一条消息必须是版本握手。该机制确保移动端 App 与桌面端插件的版本兼容性。

握手时序：

```
Mobile                                  Desktop
  |                                        |
  | ---- WS: VERSION_CHECK --------------> |
  |     { type: VERSION_CHECK,             |
  |       mobileVersion: 0.9.13 }          |
  |                                        |
  | <--- WS: VERSION_ACK ----------------- |
  |     { type: VERSION_ACK,               |
  |       version: 0.9.13 }                |
  |                                        |
```

移动端校验逻辑（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

| 常量 | 值 | 说明 |
|------|-----|------|
| EXPECTED_PLUGIN_VERSION | 0.9.13 | 期望的桌面端插件版本 |
| VERSION_CHECK_TIMEOUT | 5s | 等待 VERSION_ACK 的超时时间 |

校验规则：
- 若桌面端返回的版本与 EXPECTED_PLUGIN_VERSION 一致，握手成功，同步流程继续。
- 若版本不一致，移动端立即断开连接，向前端报告错误，并提示用户下载最新插件。
- 若超时未收到 VERSION_ACK，视为桌面端插件过旧或不支持版本校验，同样断开连接。

## 4. WebSocket 消息类型完整参考

以下表格覆盖同步会话中所有 WebSocket 消息类型。方向列中 M-D 表示 Mobile-Desktop，D-M 表示 Desktop-Mobile。

### 4.1 控制面消息

| 消息名称 | 方向 | 触发时机 | Payload 关键字段 | 处理函数（Mobile） | 处理函数（Desktop） |
|---------|------|---------|-----------------|-------------------|-------------------|
| VERSION_CHECK | M-D | WS 连接建立后，由移动端主动发送 | mobileVersion: string | run_sync_session 中直接构造发送 | index.js switch-case 处理，返回 VERSION_ACK |
| VERSION_ACK | D-M | 桌面端收到 VERSION_CHECK 后回复 | version: string | run_sync_session 中校验版本 | index.js 读取 plugin-manifest.json 返回 |
| PHASE_START | M-D | 各阶段开始时由移动端发送 | phase: string | run_sync_session 中通过 ws_stream.send 发送 | index.js 记录日志，返回 PHASE_ACK |
| PHASE_COMPLETED | M-D | 各阶段完成后由移动端发送 | phase: string（可选） | SyncCommand::Finalize 等命令触发 | index.js 记录日志，返回 PHASE_ACK |
| PHASE_ACK | D-M | 桌面端确认收到阶段控制消息 | phase: string | 移动端仅接收，无显式处理（日志输出） | index.js 统一返回 |
| SYNC_LOG_EVENT | D-M | 桌面端主动上报日志事件 | level: string, message: string | 通过 emit_sync_log 转发到前端 | 桌面端内部 logger 触发 |
| DESKTOP_PHASE_START | D-M | 桌面端报告自身阶段开始 | phase: string | 日志输出：[Desktop] Phase X started | 桌面端内部 logger 触发 |
| DESKTOP_PHASE_PROGRESS | D-M | 桌面端报告阶段进度 | phase: string | 日志输出：[Desktop] Phase X in progress | 桌面端内部 logger 触发 |
| DESKTOP_PHASE_COMPLETE | D-M | 桌面端报告自身阶段完成 | phase: string | 日志输出：[Desktop] Phase X completed | 桌面端内部 logger 触发 |

### 4.2 清单与差异比对消息

| 消息名称 | 方向 | 触发时机 | Payload 关键字段 | 处理函数（Mobile） | 处理函数（Desktop） |
|---------|------|---------|-----------------|-------------------|-------------------|
| SYNC_MANIFEST | M-D | Phase 1 / Phase 2 开始时，移动端发送本地清单 | data: EntityState[], dataType: string, phase: number, targetedOwners: string[] (V2) | SyncCommand::StartManualSync 或 PipelineCommand::StartTopicMetadata 触发 | handleSyncManifest (sync/manifest.js) |
| SYNC_DIFF_RESULTS | D-M | 桌面端完成清单比对后返回 | data: DiffResult[], dataType: string, phase: number | run_sync_session WS 处理器中解析，派发 Pull/Push/Delete 任务 | handleSyncManifest 返回 |
| SYNC_DIFF_RESULTS_BATCH | D-M | 桌面端完成消息差异批量比对后返回 | results: Record<topicId, { toPull: string[], toPush: boolean }> | run_sync_session 中调用 PushExecutor::push_messages_batch 和 PullExecutor::pull_messages_batch | handleSyncMessageDiffBatch (sync/diff.js) |
| SYNC_TOPIC_HASH_BATCH | M-D | 已废弃：V1 协议中 Phase 2 发送 topic 内容哈希 | hashes: Record<topicId, contentHash> | 旧版代码保留 | handleSyncTopicHashBatch (sync/diff.js) |
| SYNC_TOPIC_HASH_BATCH_V2 | M-D | Phase 2.5 发送 topic 双哈希 | hashes: Record<topicId, { configHash, contentHash }> | PipelineCommand::StartTopicValidation 触发构造发送 | handleSyncTopicHashBatchV2 (sync/diff.js) |
| SYNC_TOPIC_HASH_RESULTS | D-M | 桌面端返回哈希比对结果 | changedTopics: string[] | 接收后设置 changed_topics，触发 SyncCommand::StartMessages | handleSyncTopicHashBatchV2 返回 |
| SYNC_MESSAGE_DIFF_BATCH | M-D | Phase 3 分批发送消息哈希 | topics: Record<topicId, { topicHash, messages: Record<msgId, hash> }> | PipelineCommand::StartMessages 触发，按 MAX_MESSAGES_PER_BATCH（10000 条）分批次发送 | handleSyncMessageDiffBatch (sync/diff.js) |

### 4.3 实时变更通知消息

| 消息名称 | 方向 | 触发时机 | Payload 关键字段 | 处理函数（Mobile） | 处理函数（Desktop） |
|---------|------|---------|-----------------|-------------------|-------------------|
| SYNC_ENTITY_UPDATE | M-D | 移动端检测到本地实体变更（如通过文件监听器） | id, dataType, hash, ts | SyncCommand::NotifyLocalChange 触发发送 | index.js 调用 upsertEntityIndex 更新索引 |
| SYNC_DELETE_NOTIFY | M-D | 移动端实体被软删除 | id, dataType | SyncCommand::NotifyDelete 触发发送 | index.js 调用 deleteEntity / deleteMessage |
| SYNC_ERROR | D-M | 桌面端遇到不可恢复错误 | code: number, message: string | 移动端记录错误日志，更新同步状态为 error | 桌面端内部错误处理触发 |
| SYNC_ACK | D-M | 桌面端确认收到更新/删除通知 | id: string | 移动端不处理（可选日志） | index.js 统一返回 |

### 4.4 废弃与兼容性消息

| 消息名称 | 方向 | 状态 | 说明 |
|---------|------|------|------|
| GET_MESSAGE_MANIFEST | M-D | 已废弃 | V1 协议中单话题消息清单请求，V2 已被 SYNC_MESSAGE_DIFF_BATCH 取代 |
| MESSAGE_MANIFEST_RESULTS | D-M | 已废弃 | 对应 GET_MESSAGE_MANIFEST 的响应 |
| PHASE_MANIFESTS | D-M | 已废弃/忽略 | 桌面端在旧版协议中用于下发阶段清单，V2 中移动端主动发送 SYNC_MANIFEST，桌面端不再下发 PHASE_MANIFESTS。移动端 WS 处理器中显式忽略此消息类型 |
| SYNC_TOPIC_HASH_BATCH | M-D | 已废弃 | V1 单哈希批量比对，V2 由 SYNC_TOPIC_HASH_BATCH_V2 替代 |

### 4.5 新增内部命令（v0.9.14）

| 命令/消息 | 方向 | 触发时机 | 说明 |
|----------|------|---------|------|
| `SyncCommand::Cancel` | 内部 | `stop_sync` Tauri 命令被调用 | 向 WebSocket 循环发送取消信号，关闭流并中断主循环。用于用户主动关闭同步面板时立即终止后端同步会话。 |
| `SyncCommand::SendWsMessage(Value)` | 内部 | `BatchDiffHandler` 处理完一批后需要发送下一批 | 允许内部任务将 outbound WS 消息重新入队，用于 `pending_diff_batches` 的链式触发。 |
| `stop_sync` | Tauri Command | 前端调用 `closeSyncSession()` | 设置 `is_syncing = false`，发送 `SyncCommand::Cancel`，更新连接状态。前端在关闭同步面板时自动调用。 |

### 4.5 消息 Payload 字段详解

#### SYNC_MANIFEST Payload 结构

```json
{
  type: SYNC_MANIFEST,
  data: [
    {
      id: agent_001,
      hash: sha256_abc...,
      configHash: sha256_cfg...,
      contentHash: sha256_cnt...,
      ts: 1715000000000,
      deletedAt: null,
      ownerType: null
    }
  ],
  dataType: agent,
  phase: 1,
  targetedOwners: [agent_001, agent_002]
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| type | string | 是 | 固定为 SYNC_MANIFEST |
| data | EntityState[] | 是 | 本地实体状态清单 |
| dataType | string | 是 | 实体类型：agent / group / avatar / topic |
| phase | number | 是 | 阶段标识：1 表示 Phase 1（Owner Metadata），2 表示 Phase 2（Topic Metadata） |
| targetedOwners | string[] | 否 | V2 优化：仅针对特定所有者的话题清单 |

#### SYNC_DIFF_RESULTS Payload 结构

```json
{
  type: SYNC_DIFF_RESULTS,
  data: [
    { id: agent_001, action: PULL, ownerType: agent },
    { id: agent_002, action: PUSH, mismatchedContent: true },
    { id: agent_003, action: DELETE, deletedAt: 1714000000000 },
    { id: agent_004, action: PUSH_DELETE, deletedAt: 1714000000000 }
  ],
  dataType: agent,
  phase: 1
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 实体唯一标识 |
| action | string | 是 | 操作类型：PULL / PUSH / DELETE / PUSH_DELETE / SKIP |
| ownerType | string | 否 | 所有者类型，topic 类型时用于区分 agent_topic 和 group_topic |
| mismatchedContent | boolean | 否 | V2：内容哈希不匹配标记，用于引导后续 targeted topic sync |
| deletedAt | number | 条件 | DELETE / PUSH_DELETE 时必须，软删除时间戳 |

#### SYNC_MESSAGE_DIFF_BATCH Payload 结构

```json
{
  type: SYNC_MESSAGE_DIFF_BATCH,
  topics: {
    topic_001: {
      topicHash: sha256_topic...,
      messages: {
        msg_001: sha256_msg1...,
        msg_002: sha256_msg2...
      }
    }
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| topics | object | 是 | Key 为 topicId，Value 包含 topicHash（话题级聚合哈希）和 messages（消息哈希映射表） |

#### SYNC_DIFF_RESULTS_BATCH Payload 结构

```json
{
  type: SYNC_DIFF_RESULTS_BATCH,
  results: {
    topic_001: {
      toPull: [msg_003, msg_004],
      toPush: false
    },
    topic_002: {
      toPull: [],
      toPush: true
    }
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| toPull | string[] | 是 | 桌面端有、移动端无（或哈希不同）的消息 ID 列表 |
| toPush | boolean | 是 | 移动端有而桌面端无的消息时，标记为 true |

## 5. HTTP 端点完整参考

所有 HTTP 端点统一挂载在 /api/mobile-sync 路径前缀下（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js`）。请求必须使用 x-sync-token 或 Authorization: Bearer 头部进行认证。

### 5.1 实体下载端点

#### GET /download-entity

| 属性 | 值 |
|------|-----|
| 方法 | GET |
| 用途 | 单实体下载（Pull 单条） |
| Query 参数 | id: string, type: string（agent / group / agent_topic / group_topic） |
| 响应格式 | JSON（实体 DTO）或 404 Not Found |
| Body 限制 | 无 |
| 认证方式 | Header x-sync-token 或 Authorization: Bearer |
| 对应处理函数 | downloadEntity (sync/entity.js) |

#### POST /download-entities

| 属性 | 值 |
|------|-----|
| 方法 | POST |
| 用途 | 批量实体下载（Pull 批量，Phase 1/2 优化） |
| 请求格式 | {requests: [{id: ..., type: ...}]} |
| 响应格式 | JSON 数组，每个元素为对应实体的 DTO 或 null |
| Body 限制 | 默认 Express JSON 限制 |
| 认证方式 | Header |
| 对应处理函数 | downloadEntities (sync/entity.js) |
| 分批策略 | Mobile 端按类型分块：Agent/Group 每批 50，Topic 每批 1000（sync_service.rs） |

### 5.2 实体上传端点

#### POST /upload-entity

| 属性 | 值 |
|------|-----|
| 方法 | POST |
| 用途 | 单实体上传（Push 单条） |
| 请求格式 | {id: ..., type: ..., data: {...}} |
| 响应格式 | JSON（操作结果） |
| Body 限制 | 5mb |
| 认证方式 | Header |
| 幂等性 | 支持，通过 x-idempotency-key 头部 |
| 对应处理函数 | uploadEntity (sync/entity.js) |

#### POST /upload-entities-batch

| 属性 | 值 |
|------|-----|
| 方法 | POST |
| 用途 | 批量实体上传（Topic 归口优化） |
| 请求格式 | {items: [{id: ..., type: ..., data: {...}]} |
| 响应格式 | {success: true, results: [...]} |
| Body 限制 | 10mb |
| 认证方式 | Header |

## 6. 差异比对算法详解

### 6.1 清单差异计算（Manifest Diff）

桌面端收到 SYNC_MANIFEST 后，由 handleSyncManifest（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）执行差异计算。算法流程如下：

1. 加载本地清单：根据 dataType 查询桌面端本地索引数据库（entity_index 表），获取对应实体的 hash、configHash、contentHash、ts、deletedAt。
2. 遍历远程清单：对移动端发送的每个 EntityState：
   - 若 deletedAt 存在且本地未删除，输出 DELETE 动作
   - 若本地不存在，输出 PUSH 动作（移动端需要推送给桌面端）
   - 若本地已删除但远程未删除，输出 PUSH_DELETE 动作（告知移动端该实体在桌面端已被删除）
   - 若 configHash 不一致，按时间戳裁决：远程较新则 PUSH，否则 PULL
   - 若 contentHash 不一致（仅限 Agent/Group），标记 mismatchedContent = true，引导后续 Targeted Topic Sync
3. 遍历本地剩余：对远程清单中未包含的本地实体：
   - 若本地已删除，输出 PUSH_DELETE
   - 否则，输出 PULL（桌面端需要推送给移动端）

输出动作类型完整语义：

| 动作 | 方向 | 含义 | 后续操作 |
|------|------|------|---------|
| PULL | D-M | 桌面端数据较新或仅桌面端有，需推送给移动端 | 移动端调用 HTTP Pull 端点下载 |
| PUSH | M-D | 移动端数据较新或仅移动端有，需推送给桌面端 | 移动端调用 HTTP Push 端点上传 |
| DELETE | D-M | 移动端标记已删除，桌面端需同步删除 | 桌面端执行软删除 |
| PUSH_DELETE | M-D | 桌面端已删除，需通知移动端同步删除 | 移动端执行软删除，并发送 SYNC_DELETE_NOTIFY |
| SKIP | - | 数据一致，无需操作 | 无 |

### 6.2 消息差异计算（Message Diff）

Phase 3 的消息差异由 handleSyncMessageDiffBatch（sync/diff.js）计算，采用双层快速路径策略：

第一层：Topic 级快速路径
- 比较移动端发送的 topicHash 与桌面端 entity_index 中的 aggregated_hash
- 若完全一致，直接判定该 Topic 无需任何操作（toPull: [], toPush: false）
- 该路径避免了逐消息比对，在大部分话题未变更时极大降低计算量

第二层：消息级详细比对
- 对未命中快速路径的 Topic，读取桌面端 message_index 中该话题的全部消息哈希
- 构建桌面端哈希映射 Map<msgId, hash>
- 遍历桌面端消息：若移动端无此消息或哈希不同，加入 toPull
- 遍历移动端消息：若桌面端无此消息，设置 toPush = true

结果处理策略：
- toPull 非空：移动端调用 POST /download-messages-stream 批量拉取
- toPush 为 true：移动端调用 POST /upload-messages-batch 批量推送
- 对于同一 Topic 同时存在 toPush 和 toPull 的情况，移动端先执行 Push 再执行 Pull，确保桌面端数据合并后再拉取完整视图

## 7. 并发控制与流量整形

### 7.1 网络感知信号量（NetworkAwareSemaphore）

移动端使用自定义的 NetworkAwareSemaphore（`src-tauri/src/vcp_modules/sync/sync_service.rs`）控制并发 HTTP 请求数量，防止 IO 密集型任务耗尽系统资源或导致移动端 UI 卡顿。

并发数计算公式：

```rust
let cores = std::thread::available_parallelism()
    .map(|p| p.get())
    .unwrap_or(4);
let concurrency = ((cores as f32) * 1.5).clamp(6.0, 12.0) as usize;
```

| 参数 | 说明 |
|------|------|
| 基准值 | CPU 核心数 |
| 乘数 | 1.5（IO 密集型任务的平衡系数） |
| 下限 | 6 |
| 上限 | 12（移动端严格上限，保护 UI 响应） |

所有 HTTP Pull/Push 任务在发起前必须 acquire() 信号量许可，任务完成后自动释放。

### 7.2 移动端数据库写入队列（DbWriteQueue）

为避免高频并发写入 SQLite 导致锁竞争，移动端引入 DbWriteQueue（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）：

- 批量缓冲：Pull 下来的实体数据先进入内存队列，而非直接写入数据库
- 定时刷盘：队列在以下时机执行批量写入：
  1. 队列长度达到阈值
  2. 阶段切换时（PHASE_COMPLETED 触发 flush()）
  3. 同步完成时
- WAL 模式：SQLite 启用 Write-Ahead Logging，进一步降低并发写入的锁竞争

### 7.3 桌面端流式传输

Phase 3 的大批量消息传输采用 NDJSON 流式格式，避免移动端内存溢出：

- 下载流（/download-messages-stream）：桌面端逐条读取消息，序列化为 JSON 后写入响应流，每行一条记录
- 上传流（/upload-messages-batch）：移动端以流形式逐条写入消息，桌面端逐行读取并处理
- 流式传输的 Body 不限制总大小，仅受网络层和内存缓冲区限制

Phase 3 中由于涉及大量 Topic 的并行 Push/Pull，移动端使用 Phase3Tracker（`src-tauri/src/vcp_modules/sync/sync_service.rs`）精确追踪每个 Topic 的完成状态：

- 断线重连保护：pending_diff_batches 在 Phase 3 启动前被清空，防止断线重连后发送过时的差异批次。

## 10. 幂等性机制

### 10.1 设计目的

在移动网络环境下，HTTP 请求可能因网络抖动而超时，但服务器端实际已处理完成。若客户端盲目重试，可能导致数据重复写入。幂等性机制通过唯一操作标识（Idempotency Key）确保同一操作在 TTL 窗口内只执行一次。

### 10.2 幂等 Key 生成公式

移动端在发送 POST /upload-entity 请求时，通过 generate_idempotency_key 函数生成 Key（`src-tauri/src/vcp_modules/sync/sync_executor/push_executor.rs`）：

```rust
fn generate_idempotency_key(action: &str, entity_type: &str, id: &str) -> String {
    let mut hasher = Sha256::new();
    hasher.update(action.as_bytes());
    hasher.update(entity_type.as_bytes());
    hasher.update(id.as_bytes());
    let now = chrono::Utc::now().timestamp() / 60;  // 当前时间戳（分钟级）
    hasher.update(now.to_string().as_bytes());
    format!("{:x}", hasher.finalize())
}
```

公式解析：

| 组成部分 | 来源 | 作用 |
|---------|------|------|
| action | 操作类型，如 push | 区分同一实体的不同操作 |
| entity_type | 实体类型，如 agent / group | 防止跨类型 ID 冲突 |
| id | 实体唯一标识 | 定位目标实体 |
| timestamp / 60 | UTC 时间戳按分钟取整 | 时间窗口切片，每分钟生成不同的 Key |

设计权衡：使用分钟级时间窗口而非全局唯一 UUID，是为了在正常的单分钟重试场景内实现幂等保护，同时允许跨分钟的自然重试（例如用户一分钟后再次点击同步）。

### 10.3 TTL 策略

桌面端幂等性控制实现于 `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/idempotency.js`：

| 属性 | 值 |
|------|-----|
| TTL 时长 | 300000ms（5 分钟） |
| 存储结构 | Map<opId, { timestamp, result }> |
| 清理周期 | 每 60 秒执行一次自动清理 |
| 清理逻辑 | 遍历 recentOperations，删除 now - timestamp > IDEMPOTENCY_TTL 的条目 |

### 10.4 生效端点

目前幂等性机制仅在以下端点生效：

| 端点 | 方法 | 幂等头部 |
|------|------|---------|
| /upload-entity | POST | x-idempotency-key |

检测与返回逻辑（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js`）：

桌面端接收到请求后，提取 x-idempotency-key 头部，调用 checkIdempotency(opId)。若判定为重复请求，直接返回上次缓存的结果，不执行业务逻辑；若为新请求，执行业务后调用 recordOperation(opId, result) 记录结果，供后续幂等命中。

## 11. 版本兼容性策略

### 11.1 版本协商机制

本协议采用严格版本匹配策略，不支持向前/向后兼容的松散协商。

协商流程：

1. 移动端在 WebSocket 连接建立后，立即发送 VERSION_CHECK，携带自身的 CARGO_PKG_VERSION（编译期嵌入）。
2. 桌面端读取 plugin-manifest.json 中的 version 字段返回 VERSION_ACK。
3. 移动端对比两端版本字符串，必须完全相等（字符串精确匹配），否则视为不兼容。

### 11.2 版本不匹配处理

| 场景 | 移动端行为 | 桌面端行为 |
|------|-----------|-----------|
| 版本一致 | 继续同步流程 | 正常响应 |
| 版本不一致 | 断开 WS，状态置为 error，提示用户更新插件 | 不处理（版本校验由移动端主导） |
| 超时未收到 VERSION_ACK | 断开 WS，状态置为 error，提示插件过旧 | 不处理（旧版插件可能不识别 VERSION_CHECK） |

### 11.3 演进注意事项

- 版本号采用 SemVer（语义化版本）格式，但校验逻辑为字符串完全匹配，非语义化比较。这意味着 0.9.13 与 0.9.13-beta 会被视为不兼容。
- 桌面端插件版本硬编码于 plugin-manifest.json，移动端期望版本硬编码于 sync_service.rs 的 EXPECTED_PLUGIN_VERSION 常量。任何版本升级必须双端同步修改。
- 在 CI/CD 流程中，建议将 EXPECTED_PLUGIN_VERSION 与 plugin-manifest.json 的版本通过构建脚本自动对齐，避免人为遗漏。
