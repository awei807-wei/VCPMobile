# 流水线与阶段控制

## 1. 引言

在移动端与桌面端的数据同步过程中，涉及的数据类型众多（Agent、Group、Topic、Message、Avatar），操作类型复杂（Pull、Push、Delete、Push-Delete），且各阶段之间存在严格的因果依赖关系。若将所有逻辑平铺在一个巨大的异步函数中，不仅会导致代码难以维护，还会使错误定位、重试恢复和进度追踪变得极为困难。

**同步流水线（Sync Pipeline）** 的设计目标，是将复杂的端到端同步流程分解为一系列**可管理、可观测、可回退的阶段（Phase）**。每个阶段只负责一类数据的对齐工作，阶段之间通过显式的状态转换和命令通道（Command Channel）进行衔接。这种分层架构使得：

- 每个阶段的输入、输出和副作用清晰可控；
- 前端可以精确感知当前所处的同步阶段及进度；
- 某个阶段失败时，可以按策略回退或重试，而不影响其他阶段的数据一致性；
- 新增阶段（如 Phase 2.5 双哈希验证）时，只需在现有流水线的节点间插入即可，无需重构整体逻辑。

Pipeline 的核心实现位于 `sync_pipeline` 模块（`src-tauri/src/vcp_modules/sync/sync_pipeline/mod.rs`），而完整的会话生命周期管理由 `sync_service.rs` 中的 `run_sync_session` 函数驱动。

## 2. PipelinePhase 状态机

### 2.1 状态枚举定义

`PipelinePhase` 是流水线的核心状态枚举，定义于 `src-tauri/src/vcp_modules/sync/sync_pipeline/pipeline.rs`（v0.9.14 将 `pipeline_state.rs` 合并入 `pipeline.rs`），它以 `serde` 序列化，可直接通过 Tauri 事件透传给前端。枚举定义如下：

| 枚举值 | 含义 |
|--------|------|
| `Idle` | 空闲态，流水线尚未启动或已重置。 |
| `Phase1Metadata` | **Phase 1：Owner 元数据对齐**。处理 Agent、Group、Avatar 等 Owner 级别的数据差异。携带 `PhaseProgress` 进度。 |
| `Phase2TopicMetadata` | **Phase 2：Topic 元数据补全**。基于 Phase 1 筛选出的变更 Owner，拉取或推送相关的 Topic Config。携带 `PhaseProgress` 进度。 |
| `Phase3Messages` | **Phase 3：消息差异计算与对齐**。基于 Phase 2.5 筛选出的变更 Topic，进行消息级别的 Diff、Pull 和 Push。携带 `PhaseProgress` 进度。 |
| `Completed` | 所有阶段执行完毕，数据已全局对齐。 |
| `Failed` | 某阶段发生不可恢复错误，携带错误描述与失败阶段名称。 |

虽然 `PipelinePhase` 枚举仅定义了上述 6 个核心阶段状态，但在 `sync_service.rs` 的实际会话驱动中，同步流程还包含**连接建立**和**版本握手**两个前置隐含状态。下文状态流转图将展示完整的端到端生命周期。

### 2.3 错误与重试分支

```
    Connecting / VersionCheck / OwnerMetadata / TopicMetadata / Messages
                               |
                               | [发生可恢复错误]
                               v
                         +-----------+
                         |  Error    |  <-- retry_count < MAX_RETRIES (3)
                         +-----+-----+
                               |
                    [指数退避后重连]
                               v
                         +-----------+
                         |Connecting |  <-- 回到连接态，重新执行
                         +-----------+
                               |
                               | [retry_count >= MAX_RETRIES]
                               v
                         +-----------+
                         |  Fatal    |  <-- 发布 error 状态，终止会话
                         +-----------+
```

### 2.4 每个状态的可能下一个状态和触发条件

| 当前状态 | 可能的下一个状态 | 触发条件 |
|----------|------------------|----------|
| `Idle` | `Connecting` | 前端调用 `start_manual_sync` Tauri Command（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。 |
| `Connecting` | `VersionCheck` | `tokio_tungstenite::connect_async` 成功返回 `ws_stream`。 |
| `Connecting` | `Error` | 连接失败，且 `retry_count < MAX_RETRIES`（最大重试 3 次）。 |
| `VersionCheck` | `OwnerMetadata` | 收到 `VERSION_ACK`，且 `plugin_version == EXPECTED_PLUGIN_VERSION`（"0.9.13"）。 |
| `VersionCheck` | `Error` | 版本不匹配或握手超时（`VERSION_CHECK_TIMEOUT = 5s`）。 |
| `OwnerMetadata` | `TopicMetadata` | 所有 Phase 1 Manifest 的 `SYNC_DIFF_RESULTS` 已收到，且 `pending_tasks == 0`（由 Watchdog 监控）。 |
| `OwnerMetadata` | `Error` | 数据库 Hash 初始化失败，触发 `continue` 重连。 |
| `TopicMetadata` | `TopicValidation` | 所有 Phase 2 Manifest 的 `SYNC_DIFF_RESULTS` 已收到，且 `pending_tasks == 0`。 |
| `TopicValidation` | `Messages` | 收到 `SYNC_TOPIC_HASH_RESULTS`，提取 `changedTopics` 列表。 |
| `TopicValidation` | `Finalizing` | `changedTopics` 为空，直接跳过 Messages 阶段。 |
| `Messages` | `Finalizing` | 所有 topic 均通过 `Phase3Tracker::mark_completed` 标记完成，或 `changed_topics` 为空。 |
| `Messages` | `Error` | 获取消息哈希失败，直接发送 `Finalize` 命令优雅退出。 |
| `Finalizing` | `Completed` | 写队列 `flush`、全局 Hash 冒泡、WebSocket 关闭均完成。 |
| `Completed` | `Idle` | 会话结束，`is_syncing` 原子标志重置为 `false`。 |

## 3. PipelineState 结构

### 3.1 PhaseProgress

`PhaseProgress`（定义于 `src-tauri/src/vcp_modules/sync/sync_pipeline/pipeline.rs`）是每个阶段内部细粒度进度追踪的载体：

| 字段 | 类型 | 说明 |
|------|------|------|
| `total` | `u32` | 该阶段预期处理的总任务数（如 Manifest 数量、Topic 数量）。 |
| `completed` | `u32` | 已完成的任务数。 |
| `pending` | `Vec<String>` | 当前仍在等待处理的任务标识列表（如 Topic ID、Entity ID）。 |

`PhaseProgress::new()` 初始化时，三个字段均归零。在 `sync_service.rs` 中，进度信息并不直接写入 `PipelinePhase` 的 `PhaseProgress`，而是依赖独立的原子计数器（`pending_tasks`、`total_tasks`）和 `Phase3Tracker` 进行实时追踪，最终通过 `vcp-sync-progress` 事件发射给前端。

### 3.2 SyncPipeline 结构

`SyncPipeline`（`src-tauri/src/vcp_modules/sync/sync_pipeline/pipeline.rs`）是阶段转换的协调器：

| 字段 | 类型 | 说明 |
|------|------|------|
| `state` | `Arc<RwLock<PipelinePhase>>` | 受读写锁保护的当前流水线状态，支持跨异步任务安全访问。 |
| `command_tx` | `mpsc::UnboundedSender<PipelineCommand>` | 无界命令发送端，用于向内层事件循环投递阶段转换指令。 |

`SyncPipeline` 本身不执行业务逻辑，它只负责：
1. **更新 `PipelinePhase` 枚举状态**；
2. **通过 `command_tx` 发送 `PipelineCommand`**，触发 `sync_service.rs` 中 `tokio::select!` 的对应分支。

### 3.3 进度追踪机制

移动端同步的进度追踪采用**双层设计**：

1. **Pipeline 层（粗粒度）**：`PipelinePhase` 枚举本身标识当前处于哪一个大阶段（Phase 1/2/3）。
2. **Service 层（细粒度）**：
   - `AtomicU32` 类型的 `pending_tasks` 和 `total_tasks` 用于 Phase 1 和 Phase 2 的任务计数（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。
   - `Phase3Tracker` 结构用于 Phase 3 的 topic 级别完成追踪，内部使用 `Mutex<HashSet<String>>` 分别记录 `completed` 和 `modified` 的 topic，避免并发下重复计数或下溢。
   - 每完成一个子任务（或一批次），即时发射 `vcp-sync-progress` 事件，携带 `{ phase, total, completed, message }`。

## 4. 各阶段详解

### 4.1 Phase 1：Owner Metadata（Owner 元数据对齐）

**定位**：对齐移动端与桌面端在 Agent、Group、Avatar 三个数据类型上的差异。

**输入**：
- 移动端本地 SQLite 数据库中已有的 Agent、Group、Avatar 记录及其哈希值。
- 桌面端通过 `SYNC_DIFF_RESULTS` 返回的差异分析结果。

**核心逻辑**（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：
1. 会话建立后，首先调用 `HashInitializer::ensure_all_agent_hashes` 和 `ensure_all_group_hashes`，确保所有 Owner 记录的哈希字段已初始化。
2. 调用 `Phase1Metadata::build_phase1_manifests`（`src-tauri/src/vcp_modules/sync/sync_pipeline/phase1_metadata.rs`）构建 Phase 1 Manifest，生成 Agent、Group、Avatar 三类 `SyncManifest`。
3. 通过 WebSocket 发送 `SYNC_MANIFEST` 消息，`phase` 字段显式标记为 `1`。
4. 桌面端比对后返回 `SYNC_DIFF_RESULTS`，移动端根据 `action` 字段分类处理：
   - `PULL`：通过 `PullExecutor` 从桌面端 HTTP 接口拉取完整数据，经 `DbWriteQueue` 异步写入本地数据库。
   - `PUSH`：通过 `PushExecutor` 将本地数据推送到桌面端。
   - `DELETE` / `PUSH_DELETE`：通过 `DeleteExecutor` 执行软删除（`soft_delete_*`），`PUSH_DELETE` 额外通知对端同步删除。

**V2 关键增强**：在解析 `SYNC_DIFF_RESULTS` 时，若数据类型为 Agent 或 Group，且 `action` 为 `PULL`/`PUSH` 或 `mismatchedContent == true`，则将该 Owner ID 加入 `changed_owners` 集合。此集合将作为 Phase 2 的输入，实现**靶向 Topic 同步（Targeted Topic Sync）**，避免无差别扫描全部 Topic。

**输出**：
- 本地数据库与桌面端在 Agent、Group、Avatar 维度达成一致。
- 填充 `changed_owners: HashSet<String>`，供 Phase 2 使用。
- 触发 `SyncCommand::StartTopicMetadata`。

### 4.2 Phase 2：Topic Metadata（Topic 元数据补全）

**定位**：基于 Phase 1 识别出的变更 Owner，仅对这些 Owner 下属的 Topic 进行元数据对齐。

**输入**：
- `changed_owners` 集合（来自 Phase 1 的靶向筛选结果）。

**核心逻辑**（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：
1. Pipeline 收到 `PipelineCommand::StartTopicMetadata` 后，从 `changed_owners` 中读取 Owner 列表。
2. 若 `changed_owners` 为空，说明无 Owner 发生内容变更，直接跳过 Phase 2，发送 `SyncCommand::StartTopicValidation`。
3. 否则，调用 `Phase1Metadata::build_targeted_topic_manifest`（`src-tauri/src/vcp_modules/sync/sync_pipeline/phase1_metadata.rs`），传入 `owners` 切片，仅构建这些 Owner 下属的 Topic Manifest。
4. 通过 WebSocket 发送 `SYNC_MANIFEST`，`phase=2`，并携带 `targetedOwners` 字段告知桌面端本次比对范围。
5. 桌面端返回 `SYNC_DIFF_RESULTS`，处理方式与 Phase 1 类似，数据类型为 Topic：
   - `PULL`：调用 `PullExecutor::pull_entities_batch` 批量拉取 Topic 配置（分块大小 1000）。
   - `PUSH`：查询本地 Topic 元数据，组装 DTO，调用 `PushExecutor::push_entities_batch` 批量推送。

**输出**：
- 变更 Owner 下属的 Topic Config 在两端达成一致。
- 触发 `SyncCommand::StartTopicValidation`。

### 4.3 Phase 2.5：Topic Validation（Topic 双哈希批量验证）

**定位**：V2 协议新增的阶段，用于在消息级同步之前，通过**双哈希（Dual-Hash）**机制精确识别哪些 Topic 的消息内容真正需要同步，进一步减少无效流量。

**输入**：
- 与 Phase 2 相同的 `changed_owners` 列表（因为哈希验证仍以 Owner 为范围）。

**核心逻辑**（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：
1. Pipeline 收到 `PipelineCommand::StartTopicValidation`。
2. 调用 `Phase3Message::get_targeted_topic_hashes`（`src-tauri/src/vcp_modules/sync/sync_pipeline/phase3_message.rs`），批量查询这些 Owner 下所有有效 Topic（`deleted_at IS NULL`）的 `config_hash` 和 `content_hash`。
3. 组装为 JSON Map：`{ topic_id: { configHash, contentHash } }`。
4. 通过 WebSocket 发送 `SYNC_TOPIC_HASH_BATCH_V2` 消息到桌面端。
5. 桌面端比对后返回 `SYNC_TOPIC_HASH_RESULTS`，其中包含 `changedTopics` 数组——即双哈希中至少有一个不匹配的 Topic ID 列表。

**输出**：
- `changed_topics: Vec<String>`，精确标识需要进入 Phase 3 消息同步的 Topic。
- 若 `changedTopics` 为空，则直接跳过 Phase 3，进入 Finalize。
- 否则触发 `SyncCommand::StartMessages`。

### 4.4 Phase 3：Messages（消息差异计算与对齐）

**定位**：对 Phase 2.5 筛选出的变更 Topic，进行消息级别的 Diff、Pull 和 Push。

**输入**：
- `changed_topics` 列表（来自 Phase 2.5）。

**核心逻辑**（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：
1. Pipeline 收到 `PipelineCommand::StartMessages`。
2. 调用 `Phase3Message::get_topic_message_hashes`（`src-tauri/src/vcp_modules/sync/sync_pipeline/phase3_message.rs`），批量获取每个 Topic 的：
   - `topic_hash`（Topic 级别的 content_hash）
   - `messages` Map（`msg_id → content_hash`）
3. 构建 Diff Batch。为避免单条 WebSocket 消息过大（移动端内存和带宽敏感），采用**消息数量分块策略**：每批最多 `MAX_MESSAGES_PER_BATCH`（10,000 条消息，约 1.5–2 MB JSON），由 `build_diff_batches` 函数实现。
4. 发送第一批 `SYNC_MESSAGE_DIFF_BATCH`，剩余批次进入 `pending_diff_batches` 队列。
5. 桌面端返回 `SYNC_DIFF_RESULTS_BATCH`，结果按 Topic 归类：
   - `toPush: true`：该 Topic 需要向桌面端推送本地消息。
   - `toPull: [msg_id, ...]`：该 Topic 需要从桌面端拉取指定消息。
6. 移动端执行策略为**先 Push 后 Pull**：
   - 先调用 `PushExecutor::push_messages_batch` 批量推送，确保 push-pull 型 Topic 的本地数据已合并到桌面端。
   - 再调用 `PullExecutor::pull_messages_batch` 批量拉取。
7. 每个 Topic 处理完毕后，调用 `Phase3Tracker::mark_completed`。当所有 Topic 完成时，自动触发 `SyncCommand::Finalize`。

**输出**：
- 两端消息内容完全一致。
- `Phase3Tracker.modified` 集合记录实际发生数据变更的 Topic，供 Finalize 阶段进行全局 Hash 冒泡。

### 4.5 Finalize（收尾与全局一致性校验）

**定位**：清理资源、更新派生字段、通知前端刷新。

**触发条件**：Phase 3 所有 Topic 完成，或 `changed_topics` 为空直接跳过 Phase 3。

**核心逻辑**（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：
1. 调用 `DbWriteQueue::flush()`，确保所有异步写操作落盘。
2. **全局 Hash 冒泡**：对 `Phase3Tracker.modified` 中的 Topic，在一个数据库事务中依次执行：
   - 批量更新受影响 Topic 的 `msg_count` 和 `updated_at`。
   - 逐个调用 `HashAggregator::bubble_topic_hash` 重新计算 Topic 哈希。
   - 根据 Topic 的 `owner_type` 收集受影响的 Agent 和 Group。
   - 调用 `HashAggregator::bubble_agent_hash` 和 `bubble_group_hash` 向上层传播哈希变更。
   - 提交事务。
3. 更新 `PipelinePhase` 为 `Completed`。
4. 发射 `vcp-sync-completed` 事件，通知前端刷新数据。
5. 向桌面端发送 `PHASE_COMPLETED`，并关闭 WebSocket 连接。

## 5. 阶段间依赖关系

### 5.1 为什么必须按顺序执行

同步流水线采用**严格顺序执行（Sequential Execution）**模型，而非并发模型，原因如下：

1. **数据归属依赖**：Topic 必须归属于某个 Agent 或 Group（Owner）。若 Phase 1 尚未对齐 Owner 元数据，Phase 2 中拉取的 Topic Config 可能指向一个不存在的 Owner，导致外键约束失败或数据孤儿。
2. **哈希一致性依赖**：Phase 2.5 的双哈希验证依赖于 Phase 2 完成后本地 Topic Config 已处于最新状态。若 Topic Config 尚未同步完成就进行哈希比对，比对基准本身就是过时的，会产生大量假阳性差异。
3. **消息归属依赖**：消息（Message）必须归属于某个 Topic。若 Topic 元数据（如 `topic_id`、`owner_id`）未对齐，Phase 3 中拉取的消息无法正确插入本地数据库。
4. **写队列顺序性**：`DbWriteQueue` 是全局单例，按 FIFO 顺序执行写入。若多个阶段并发提交写任务，可能导致后到的 Phase 1 Owner 删除操作覆盖了先到的 Phase 3 消息写入，破坏数据一致性。

### 5.2 每个阶段的完成条件

| 阶段 | 完成条件 |
|------|----------|
| Phase 1 | 所有 Manifest 的 `SYNC_DIFF_RESULTS` 已接收，且 `pending_tasks == 0`（所有 PULL/PUSH/DELETE 异步任务执行完毕）。由 Watchdog 监控防止死锁。 |
| Phase 2 | 同上。Phase 2 的 Manifest 预期数量为 1（`expected_manifest_count = 1`），因为 targeted manifest 是聚合后的单条消息。 |
| Phase 2.5 | 收到桌面端返回的 `SYNC_TOPIC_HASH_RESULTS` 消息。 |
| Phase 3 | `Phase3Tracker.completed.len() == Phase3Tracker.total`，即所有变更 Topic 都已通过 `mark_completed` 标记。 |
| Finalize | 写队列 flush 完成，全局 Hash 冒泡事务提交成功，WebSocket 关闭完成。 |

### 5.3 阶段间数据传递

阶段之间的数据传递通过**共享内存结构**实现，而非函数参数传递，因为阶段触发是由异步消息驱动的：

```
Phase 1 (Owner Metadata)
   │
   │ 填充 changed_owners: Arc<Mutex<HashSet<String>>>
   │ （在 SYNC_DIFF_RESULTS 处理逻辑中，识别 action=PUSH/PULL 或 mismatchedContent=true 的 Agent/Group）
   v
Phase 2 (Topic Metadata)
   │
   │ 读取 changed_owners → 构建 targeted manifest → 同步 Topic Config
   │ （若 changed_owners 为空，直接短路到 Phase 2.5）
   v
Phase 2.5 (Topic Validation)
   │
   │ 读取 changed_owners → 查询本地双哈希 → 发送 SYNC_TOPIC_HASH_BATCH_V2
   │ 收到 changedTopics → 填充 changed_topics: Arc<Mutex<Vec<String>>>
   v
Phase 3 (Messages)
   │
   │ 读取 changed_topics → 查询消息哈希 → 构建 diff batches
   │ 执行 Pull/Push → 填充 Phase3Tracker.modified
   v
Finalize
   │
   │ 读取 Phase3Tracker.modified → 全局 Hash 冒泡
   v
Completed
```

## 6. 错误处理与恢复

### 6.1 某阶段失败时的回退策略

同步流水线并未实现传统意义上的"阶段回滚（Rollback）"，而是采用**断点续传式重连**策略：

1. **连接层错误**：若 WebSocket 断开或连接失败，`run_sync_session` 的外层 loop 会进行最多 3 次重试（`MAX_RETRIES = 3`），退避时间从 500ms 开始指数增长，上限 5 秒（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。
2. **版本验证失败**：若桌面端插件版本与移动端期望版本（`EXPECTED_PLUGIN_VERSION`）不匹配，属于不可恢复错误，直接终止会话并提示用户更新插件。
3. **数据库初始化失败**：Phase 1 开始时若 `HashInitializer` 失败，不终止会话，而是执行 `continue` 回到外层 loop，触发重连。这给予数据库临时锁竞争恢复的机会。
4. **阶段内部任务失败**：如 `PullExecutor` 或 `PushExecutor` 某个子任务失败，错误被记录到 `SyncLogger` 并通过 `emit_sync_log` 发送给前端，但**不会阻塞整个阶段**。失败的任务会使 `pending_tasks` 递减，阶段最终仍会尝试推进到下一状态。这种设计基于"部分成功优于完全停滞"的移动端务实原则。
5. **死锁检测（Watchdog）**：Phase 1 和 Phase 2 依赖 `pending_tasks` 归零来触发下一阶段。若某异步任务忘记递减计数器，会导致永久阻塞。系统部署了 Watchdog 守护任务（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：每 10 秒检查一次 `pending_tasks`，若连续 6 个周期（60 秒）未变化，则判定为死锁，强制发送下一阶段的 `SyncCommand` 进行恢复。

### 6.2 重连后如何恢复状态

当前版本（v0.9.13）的同步流水线采用**无状态重连（Stateless Reconnect）**设计：

- 每次重连都视为一次全新的同步会话，从 `VersionCheck` 开始重新执行完整流程。
- `SyncPipeline` 的 `state` 字段在重连时不会保留上一次会话的中间状态。
- 这种设计的合理性在于：
  - 同步协议本身是**幂等**的（基于哈希比对，重复同步同一份数据不会产生副作用）。
  - 移动端数据量可控，完整重同步的成本低于维护复杂的断点状态机。
  - 避免了重连后前后状态不一致导致的逻辑灾难。

**注意**：Phase 3 的 `pending_diff_batches` 队列在每次进入 Phase 3 时会被显式清空（`src-tauri/src/vcp_modules/sync/sync_service.rs`），防止断线重连后发送过时的 Diff 数据。

### 7.1 事件总览

### 7.3 事件 Payload 结构

#### `vcp-sync-status`

贯穿连接、进度、完成、错误四种状态：

```json
{
  "status": "connecting | open | error | completed",
  "message": "同步服务连接中...",
  "source": "Sync"
}
```

该事件同时通过 `vcp-system-event`（`type: vcp-sync-status`）和直接事件 `vcp-sync-status` 两种通道发射，确保兼容不同前端监听方式（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。

#### `vcp-sync-progress`

用于进度条实时更新：

```json
{
  "phase": "owner_metadata | topic_metadata | messages",
  "total": 42,
  "completed": 17,
  "message": "Syncing Messages: 17/42"
}
```

#### `vcp-sync-completed`

同步会话完全结束后发射，前端据此刷新数据列表：

```json
{
  "source": "Sync",
  "agentsChanged": true,
  "groupsChanged": true,
  "topicsChanged": true,
  "messagesChanged": true
}
```

注意：在 `Finalize` 阶段，移动端会**主动**发射此事件，而不是等待桌面端的 `PHASE_COMPLETED` 响应。这是因为移动端在发送 `PHASE_COMPLETED` 后会立即关闭 WebSocket，无法保证收到服务器的回执（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。

#### `vcp-log`

通用日志事件，供前端的同步日志终端渲染：

```json

  "id": "info_1234567890000000000",

  "message": "[Desktop] Phase owner_metadata started"

```

#### `vcp-system-event` (type: vcp-log-message)

用于侧边栏 Mini Log Terminal 的结构性提示：

```json

  "type": "vcp-log-message",

    "id": "vcp_sync_connection_status",
    "status": "success | error",
    "tool_name": "Sync",
```

### 7.4 桌面端反向事件

移动端也会接收并处理桌面端发送的日志事件：

- `SYNC_LOG_EVENT`：桌面端日志透传，前缀标记 `[Desktop]`，通过 `vcp-log` 事件展示给前端。
- `DESKTOP_PHASE_START` / `DESKTOP_PHASE_PROGRESS` / `DESKTOP_PHASE_COMPLETE`：桌面端各阶段生命周期通知，仅作为日志展示，不影响移动端状态机。

## 8. 关键设计决策摘要

1. **严格顺序优于并发**：虽然移动端网络为 IO 密集型场景，但同步流水线选择顺序执行阶段，以牺牲理论吞吐为代价，换取数据一致性和逻辑可维护性。
2. **靶向同步（Targeted Sync）**：通过 Phase 1 识别 `changed_owners`，Phase 2 仅同步这些 Owner 的 Topic，Phase 2.5 通过双哈希进一步筛选，避免全量消息 Diff，显著降低移动端计算和网络负载。
3. **无状态重连**：不重试单个阶段，而是回退到会话起点重新执行。利用哈希协议的幂等性，简化故障恢复逻辑。
4. **Watchdog 防死锁**：通过独立的异步任务监控 `pending_tasks`，在极端异常下强制推进状态，防止用户界面永久卡在"同步中"。
5. **主动完成通知**：移动端在 `Finalize` 阶段主动发射完成事件并关闭连接，不依赖服务器回执，避免网络抖动导致的完成事件丢失。

## 9. PipelineCommand 命令协议

`PipelineCommand`（`src-tauri/src/vcp_modules/sync/sync_pipeline/pipeline.rs`）是 `SyncPipeline` 向内层事件循环投递阶段转换指令的枚举。与 `SyncCommand`（定义在 `sync_service.rs` 中）不同，`PipelineCommand` 仅关注流水线阶段本身的推进，不携带具体业务数据：

| 命令 | 含义 | 触发源 |
|------|------|--------|
| `StartTopicMetadata` | 启动 Phase 2。`SyncPipeline::on_owner_metadata_done` 在更新 `PipelinePhase` 为 `Phase2TopicMetadata` 后发送此命令。 | `SyncPipeline` |
| `StartTopicValidation` | 启动 Phase 2.5。`SyncPipeline::on_topic_metadata_pull_done` 发送此命令（注意：该命令不改变 `PipelinePhase` 枚举值，因为 Phase 2.5 在状态机中复用 `Phase2TopicMetadata` 的抽象）。 | `SyncPipeline` |
| `StartMessages` | 启动 Phase 3。`SyncPipeline::on_topic_validation_done` 在更新 `PipelinePhase` 为 `Phase3Messages` 后发送此命令。 | `SyncPipeline` |
| `Finalize` | 进入收尾阶段。`SyncPipeline::on_messages_done` 在更新 `PipelinePhase` 为 `Completed` 后发送此命令。 | `SyncPipeline` |

在 `sync_service.rs` 的 `tokio::select!` 中，`pipeline_rx` 分支专门处理上述命令，而 `rx` 分支处理 `SyncCommand`。这种**双通道设计**实现了流水线状态机（`SyncPipeline`）与会话执行器（`run_sync_session`）的解耦：前者只决定"何时进入下一阶段"，后者决定"该阶段具体做什么"。

## 10. 并发控制与资源保护

### 10.1 NetworkAwareSemaphore

同步会话中的大量子任务（Pull、Push、Delete）通过 `tauri::async_runtime::spawn` 并发执行。为防止并发过高导致移动端网络拥堵或 UI 卡顿，`sync_service.rs` 中引入了 `NetworkAwareSemaphore`（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

- 该信号量根据设备 CPU 核心数动态计算并发上限：`(cores * 1.5).clamp(6.0, 12.0)`。
- 在移动端（通常为 4–8 核），实际并发数会被 clamp 在 6 到 12 之间，平衡 IO 吞吐与系统负载。
- 每个异步任务在执行网络请求前必须 `acquire()` 一个许可，任务完成后自动释放。

### 10.2 Phase3Tracker 的互斥设计

Phase 3 中，多个 Topic 的 Push/Pull 任务并发执行，完成后都需要调用 `mark_completed`。若使用 `AtomicU32` 计数，存在**双重递减下溢**风险（同一 Topic 被多个并发任务同时标记完成）。`Phase3Tracker` 使用 `Mutex<HashSet<String>>` 存储已完成的 Topic ID，利用 `HashSet::insert` 返回的 `is_new` 布尔值确保：

- 同一个 Topic 无论被多少并发任务标记，只会计数一次；
- 当 `completed.len() == total` 时，才触发下一阶段的 `Finalize` 命令。

## 11. 模块导出关系

`sync_pipeline` 模块的公开接口由 `mod.rs`（`src-tauri/src/vcp_modules/sync/sync_pipeline/mod.rs`）统一聚合：

```
sync_pipeline/
├── mod.rs                  # 公开导出：Phase1Metadata, Phase3Message, SyncPipeline
├── pipeline.rs             # SyncPipeline 结构 + PipelineCommand 枚举 + PipelinePhase/PhaseProgress
├── phase1_metadata.rs      # Phase1Metadata 静态方法（直接 SQL 查询构建 Manifest）
└── phase3_message.rs       # Phase3Message 静态方法（哈希查询）
```

这种扁平导出策略使得 `sync_service.rs` 只需一条 `use` 语句即可引入全部 Pipeline 相关类型：

```rust
use crate::vcp_modules::sync_pipeline::{Phase1Metadata, Phase3Message, SyncPipeline};
```
