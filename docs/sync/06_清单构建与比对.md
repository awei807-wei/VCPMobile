# 清单构建与比对

## 1. 引言

在 VCP Mobile 的分布式同步架构中，**Manifest（清单）** 是"先比对，后传输"（Hash-First Sync）策略的核心数据结构。它的作用类似于 Git 的索引（Index）：在真正传输实体数据之前，双端先交换各自的数据指纹（SHA-256 Hash），通过比对指纹差异精确识别需要同步的条目，从而避免无意义的整表传输。

Manifest 贯穿同步流程的三个阶段：

| 阶段 | Manifest 类型 | 比对范围 | 说明 |
|------|--------------|---------|------|
| Phase 1 | Owner Metadata Manifest | Agent / Group / Avatar | 比对所有者元数据（`src-tauri/src/vcp_modules/sync/sync_manifest/manifest_builder.rs`） |
| Phase 2 | Topic Metadata Manifest | Topic（按 Owner 过滤） | 比对话题配置与聚合哈希（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`） |
| Phase 3 | Message Manifest | Message（按 Topic 分批） | 比对消息内容哈希（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js`） |

整个比对流程遵循**移动端发起、桌面端裁决**的模型：移动端构建并发送自己的 Manifest，桌面端将其与本地 `sync_state.db` 中的状态进行 Diff（差异比对），生成 Action 列表返回给移动端执行。

## 2. SyncManifest 结构

### 2.1 数据定义

清单由两个核心结构组成，定义于 Rust 类型系统（`src-tauri/src/vcp_modules/sync/sync_types.rs`）：

```
SyncManifest
├── data_type: SyncDataType   // agent | group | topic | avatar | message
└── items: Vec<EntityState>

EntityState
├── id: String                // 实体唯一标识
├── hash: String              // 兼容性主哈希（V2 中等同于 config_hash）
├── config_hash: Option<String>   // 配置指纹（V2 新增）
├── content_hash: Option<String>  // 内容聚合指纹（V2 新增）
├── ts: i64                   // 最后更新时间戳（毫秒级 Unix Epoch）
├── deleted_at: Option<i64>   // 软删除时间戳
└── owner_type: Option<String>    // 仅 topic 使用："agent" | "group"
```

**字段语义说明**：

| 字段 | 语义 | 适用数据类型 | 版本 |
|------|------|------------|------|
| `hash` | 向后兼容的单一哈希，V2 中映射为 `config_hash` | 全部 | V1/V2 |
| `config_hash` | 实体静态配置的 SHA-256 指纹（名称、模型参数、排序等） | Agent/Group/Topic | V2 |
| `content_hash` | 动态内容聚合指纹（Agent/Group 下所有 Topic 的 Merkle Root；Topic 下所有 Message 的 Merkle Root） | Agent/Group/Topic | V2 |
| `ts` | LWW（Last-Write-Wins，最后写入优先）仲裁的时间基准 | 全部 | V1/V2 |
| `deleted_at` | 非空表示该实体已被软删除，参与删除同步判定 | 全部 | V1/V2 |
| `owner_type` | 用于区分话题属于 Agent 还是 Group，指导 Pull/Push 路由 | Topic | V2 |

### 2.2 移动端 Manifest 构建

移动端（Rust 侧）通过 `ManifestBuilder`（`src-tauri/src/vcp_modules/sync/sync_manifest/manifest_builder.rs`）直接从 SQLite 主数据库查询构建清单。

#### Agent / Group Manifest

```
SELECT agent_id, config_hash, content_hash, updated_at, deleted_at FROM agents
SELECT group_id, config_hash, content_hash, updated_at, deleted_at FROM groups
```

查询结果逐行映射为 `EntityState`：
- `id` ← `agent_id` / `group_id`
- `hash` / `config_hash` ← `config_hash`
- `content_hash` ← `content_hash`
- `ts` ← `updated_at`
- `deleted_at` ← `deleted_at`
- `owner_type` ← `None`

#### Avatar Manifest

Avatar 的标识规则较为特殊，采用组合键：

```
SELECT owner_id, owner_type, avatar_hash, updated_at FROM avatars
```

- `id` ← `format!("{}:{}", owner_type, owner_id)`，例如 `"agent:agent_001"`
- `hash` ← `avatar_hash`
- `config_hash` / `content_hash` ← `None`（Avatar 无 V2 双哈希）

#### Topic Manifest（Targeted 模式）

Phase 2 中，Topic Manifest 并非全量构建，而是基于 Phase 1 筛选出的 **`changed_owners`** 列表进行过滤：

```
SELECT topic_id, config_hash, content_hash, updated_at, owner_type, deleted_at
FROM topics WHERE owner_id IN (?, ?, ...)
```

- 过滤条件：`owner_id IN (owners)`
- 特殊排除：`topic_id == "default"` 的条目被跳过（默认话题为系统保留，不参与同步）
- `owner_type` 直接从数据库列读取（区别于桌面端需要从文件路径推导）

#### Phase 1 批量构建

```rust
// build_phase1_manifests 的调用顺序
manifests.push(agent_manifest);
manifests.push(group_manifest);
manifests.push(avatar_manifest);
```

三个 Manifest 分三条独立 WebSocket 消息发送，桌面端逐个处理并返回对应的 `SYNC_DIFF_RESULTS`。

### 2.3 桌面端 Local Manifest 构建

桌面端（Node.js 插件侧）通过 `getLocalManifest`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）从 `sync_state.db` 构建本地清单。

#### Avatar 查询

```sql
SELECT owner_id, owner_type, hash, updated_at, deleted_at FROM avatar_index
```

- `id` 同样使用 `"owner_type:owner_id"` 格式
- 返回字段映射与移动端一致

#### Agent / Group / Topic 查询

```sql
-- 普通模式（Agent / Group）
SELECT * FROM entity_index WHERE type = ?

-- Topic Targeted 模式
SELECT * FROM entity_index
WHERE type = 'topic'
  AND (file_path LIKE ? OR file_path LIKE ?)
```

**Topic 的 `ownerType` 推导**：桌面端 `entity_index` 不存储显式的 `owner_type` 列，而是通过 `file_path` 字符串匹配推导：

| 路径特征 | ownerType |
|---------|-----------|
| `file_path` 包含 `"AgentGroups"` | `"group"` |
| `file_path` 包含 `"Agents"` | `"agent"` |

**Targeted 过滤的防冲突设计**：早期实现使用 `file_path LIKE '%${id}%'` 导致 prefix 冲突（如 `agent_1` 同时匹配 `agent_11`）。V2 修复后采用精确路径模式：

```
%Agents%${id}%config.json
%AgentGroups%${id}%config.json
```

使用 `%` 作为分隔符以兼容 Windows 反斜杠与 Unix 正斜杠。

#### 哈希字段映射

桌面端 `entity_index` 表的字段与移动端略有差异：

| 移动端字段 | 桌面端字段 | 说明 |
|-----------|-----------|------|
| `config_hash` | `hash` | 实体配置指纹 |
| `content_hash` | `aggregated_hash` | 内容聚合指纹 |

在 `getLocalManifest` 中完成映射：`configHash = r.hash`、`contentHash = r.aggregated_hash || ""`。

## 3. Diff 算法详解

Diff 算法的完整实现位于桌面端 `handleSyncManifest`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）。该函数接收移动端发送的远程清单（`remoteItems`），与桌面端本地清单（`localItems`）进行逐条比对，输出 Action 列表。

### 3.1 算法总体框架

算法采用**两轮遍历 + 标记集合**的设计：

1. **第一轮**：遍历所有 `remoteItems`，针对每个条目查找对应的本地状态，判定 Action，并将已处理 ID 记入 `processedIds`。
2. **第二轮**：遍历所有 `localItems`，仅处理未出现在 `processedIds` 中的条目（即远程清单未覆盖的本地条目）。

```
function handleSyncManifest(payload):
    remoteItems ← payload.data        // 来自移动端的清单
    localItems  ← getLocalManifest(dataType, targetedOwners)
    results     ← []
    processedIds ← Set()

    // ── 第一轮：以 remote 为基准 ──
    for each remote in remoteItems:
        local ← localItems.find(it => it.id === remote.id)
        remoteDeleted ← remote.deletedAt != null

        if remoteDeleted:
            handleRemoteDeleted(local, remote, results, processedIds)
        else if local == null:
            results.push({ id: remote.id, action: "PUSH" })
        else if local.deletedAt != null:
            results.push({ id: local.id, action: "PUSH_DELETE", deletedAt: local.deletedAt })
            processedIds.add(local.id)
        else:
            handleBothExist(local, remote, dataType, results, processedIds)

    // ── 第二轮：处理 remote 未覆盖的 local ──
    for each local in localItems:
        if processedIds.has(local.id): continue
        if remoteItems.find(r => r.id === local.id): continue

        if local.deletedAt != null:
            results.push({ id: local.id, action: "PUSH_DELETE", deletedAt: local.deletedAt })
        else:
            results.push({ id: local.id, action: "PULL" })

    return { type: "SYNC_DIFF_RESULTS", data: results, dataType, phase }
```

### 3.2 逐条比对逻辑

#### 分支 A：远程条目已删除（`remoteDeletedAt != null`）

```
if remote.deletedAt:
    if local == null || local.deletedAt == null:
        results.push({ action: "DELETE", deletedAt: remote.deletedAt })
    processedIds.add(remote.id)
```

| 本地状态 | 判定结果 | 逻辑解释 |
|---------|---------|---------|
| 不存在 | `DELETE` | 桌面端无此条目，但远程已标记删除。为保持一致性，桌面端也执行删除（幂等操作）。 |
| 存在且未删除 | `DELETE` | 桌面端仍保留该条目，需响应远程的删除指令，执行软删除。 |
| 存在且已删除 | 无操作 | 双端均已删除，无需任何 Action。仅标记为已处理。 |

#### 分支 B：远程条目存在，本地不存在（`local == null`）

```
results.push({ id: remote.id, action: "PUSH", ownerType: remote.ownerType })
```

桌面端缺失此条目，需从移动端**拉取**（PUSH 表示"移动端推送给桌面端"，从移动端视角看是 Push，从桌面端视角看是接收）。

#### 分支 C：远程存在，本地已删除（`local.deletedAt != null`）

```
results.push({ id: local.id, action: "PUSH_DELETE", deletedAt: local.deletedAt, ownerType: local.ownerType })
processedIds.add(local.id)
```

桌面端已删除，但移动端未删除。桌面端需要：
1. 将本地的删除状态**推送**给移动端（让移动端也删除）。
2. 该 Action 在移动端触发 `DeleteExecutor` + `NotifyDelete`（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。

#### 分支 D：双端均存在且均未删除（核心比对）

进入 V2 双哈希比对流程：

```
remoteConfig ← remote.configHash || remote.hash
remoteContent ← remote.contentHash || ""
localConfig ← local.configHash || local.hash
localContent ← local.contentHash || ""
itemChanged ← false

// Step 1: 比较 config_hash
if localConfig !== remoteConfig:
    if remote.ts > local.ts:
        results.push({ action: "PUSH" })
    else:
        results.push({ action: "PULL" })
    itemChanged ← true

// Step 2: 比较 content_hash（仅 Agent/Group）
if (dataType === "agent" || dataType === "group") && localContent !== remoteContent:
    existing ← results.find(r => r.id === remote.id)
    if existing:
        existing.mismatchedContent ← true
    else:
        results.push({ action: "SKIP", mismatchedContent: true })

processedIds.add(remote.id)
```

### 3.3 边界情况处理

| 边界场景 | 处理策略 | 代码位置 |
|---------|---------|---------|
| `dataType` 非法 | 直接返回 `null`，记录 warn 日志 | `manifest.js:94-101` |
| `payload.data` 非数组 | 返回 `null`，记录 error 日志 | `manifest.js:103-106` |
| `topic_id == "default"` | 移动端构建时直接跳过 | `manifest_builder.rs:96-98` |
| Avatar ID 解析失败 | 移动端使用 `"type:id"` 格式；桌面端按同样规则解析 | 双端一致 |
| 时间戳为负数或零 | 直接参与数值比较，`remote.ts > local.ts` 判定依然有效 | `manifest.js:146` |
| `configHash` 一方为 `null` | 回退到 `hash` 字段；若仍缺失则视为 `""` | `manifest.js:137-140` |
| `contentHash` 一方为 `null` | 回退到 `""` | `manifest.js:138, 140` |

## 4. Action 判定矩阵

以下矩阵完整覆盖所有状态组合。状态维度定义：

- **远程状态**：`存在`（未删除）、`不存在`（清单中无此 ID）、`已删除`（`deletedAt != null`）
- **本地状态**：`存在`（未删除）、`不存在`（清单中无此 ID）、`已删除`（`deletedAt != null`）

### 4.1 第一轮遍历矩阵（remote 为基准）

| 远程状态 | 本地状态 | Action | 附加字段 | 进入 processedIds | 说明 |
|---------|---------|--------|---------|------------------|------|
| 已删除 | 不存在 | — | — | ✓ | 双端一致（均无有效数据），无需操作 |
| 已删除 | 存在 | `DELETE` | `deletedAt` | ✓ | 桌面端需同步删除 |
| 已删除 | 已删除 | — | — | ✓ | 双端均已删除，无需操作 |
| 存在 | 不存在 | `PUSH` | `ownerType` | — | 桌面端缺失，从移动端拉取 |
| 存在 | 已删除 | `PUSH_DELETE` | `deletedAt`, `ownerType` | ✓ | 桌面端已删，需通知移动端同步删除 |
| 存在 | 存在 | *见下方子矩阵* | | ✓ | 进入双哈希细粒度比对 |

### 4.2 双端均存在时的子矩阵（Hash 比对）

此子矩阵仅适用于 Agent / Group / Topic 类型。Avatar 无 V2 双哈希，仅比较单一 `hash`。

#### 4.2.1 Config Hash 不一致

| configHash 比较 | 时间戳比较 | Action | 附加字段 |
|----------------|-----------|--------|---------|
| `localConfig ≠ remoteConfig` | `remote.ts > local.ts` | `PUSH` | `ownerType` |
| `localConfig ≠ remoteConfig` | `remote.ts ≤ local.ts` | `PULL` | `ownerType` |

> **LWW 裁决**：以时间戳更大的版本为"更新"方。时间戳相等时默认走 `PULL`（桌面端覆盖移动端），这构成一种隐式的 Hash 仲裁机制（`src-tauri/src/vcp_modules/sync/sync_service.rs` 中若收到 `PULL` 则从桌面端拉取数据覆盖本地）。

#### 4.2.2 Content Hash 不一致（仅 Agent / Group）

| configHash | contentHash | Action | `mismatchedContent` | 说明 |
|-----------|-------------|--------|-------------------|------|
| 一致 | `localContent ≠ remoteContent` | `SKIP` | `true` | 配置未变但内容聚合不一致，触发 targeted topic sync |
| 不一致 | `localContent ≠ remoteContent` | `PUSH` 或 `PULL` | `true` | 附加到 Step 1 已产生的 Action 上 |
| 一致 | `localContent = remoteContent` | — | — | 完全匹配，不产生任何 Action |

### 4.3 第二轮遍历矩阵（local 为基准，未处理项）

第二轮仅处理**未出现在第一轮 `processedIds` 中**且**远程清单无对应 ID** 的本地条目。

| 本地状态 | 远程状态 | Action | 附加字段 | 说明 |
|---------|---------|--------|---------|------|
| 存在 | 不存在 | `PULL` | `ownerType` | 移动端缺失，从桌面端拉取 |
| 已删除 | 不存在 | `PUSH_DELETE` | `deletedAt`, `ownerType` | 桌面端已删，通知移动端同步删除 |

### 4.4 Action 类型汇总

| Action | 语义 | 执行端 | 触发条件 |
|--------|------|--------|---------|
| `PUSH` | 移动端数据更新，推送给桌面端 | 桌面端接收 | 桌面端缺失，或桌面端数据较旧 |
| `PULL` | 桌面端数据更新，拉取到移动端 | 移动端执行 | 移动端缺失，或移动端数据较旧 |
| `DELETE` | 桌面端执行软删除 | 桌面端执行 | 远程已删除，桌面端未删除 |
| `PUSH_DELETE` | 桌面端已删除，通知移动端同步删除 | 双端执行 | 本地已删除，远程未删除（或远程不存在） |
| `SKIP` | 无需操作，仅标记观察 | 无 | Config Hash 一致，仅 Content Hash 不一致 |

## 5. V2 双哈希比对逻辑

V2 协议引入**双哈希机制**（Dual-Hash Scheme），将实体的"静态配置"与"动态内容"解耦，使得配置微调和内容大量变更可以被独立感知。这对 Agent/Group 类型尤为关键，因为其 `content_hash` 是旗下所有 Topic 消息树的 Merkle Root，任何子话题的消息变动都会向上冒泡（Bubble Up）改变父级的 `content_hash`。

### 5.1 双哈希的语义分层

| 哈希类型 | 代表内容 | 变更频率 | 冒泡来源 |
|---------|---------|---------|---------|
| `configHash` | 实体静态配置（名称、描述、系统提示词、模型参数、排序权重等） | 低 | 用户直接编辑 |
| `contentHash` | 动态内容聚合指纹（Agent/Group → Topic Merkle Root；Topic → Message Merkle Root） | 高 | 子实体变更冒泡 |

### 5.2 Config Hash 不一致的处理

当 `localConfig ≠ remoteConfig` 时，双端至少有一方的静态配置发生了变更。此时直接启用 **LWW（Last-Write-Wins，最后写入优先）** 时间戳仲裁：

```
if remote.ts > local.ts:
    Action = PUSH   // 移动端更新较新，桌面端接收
else:
    Action = PULL   // 桌面端更新较新（或时间戳相等），移动端拉取
```

**时间戳碰撞（Timestamp Collision）的 Hash 仲裁**：

当 `remote.ts == local.ts` 时，代码走 `else` 分支，默认判定为 `PULL`。这并非真正的随机仲裁，而是一种**保守策略**——在无法判断谁更新的情况下，优先以桌面端（Server）数据为准。该策略避免了双端反复互推同一配置的振荡风险。

### 5.3 Content Hash 不一致的处理

`contentHash` 不一致本身**不直接触发 PUSH/PULL**，而是作为**元数据信号**引导 Phase 2 的 targeted topic 同步。这是 V2 协议的关键优化：如果仅需同步消息，则不必重传整个 Agent/Group 配置。

```
if dataType in ["agent", "group"] and localContent ≠ remoteContent:
    existing ← results.find(r => r.id == remote.id)
    if existing:
        // configHash 已不一致，已有 PUSH/PULL Action
        existing.mismatchedContent = true
    else:
        // configHash 一致，仅 contentHash 不一致
        results.push({
            id: remote.id,
            action: "SKIP",
            mismatchedContent: true,
            ownerType: remote.ownerType
        })
```

**`mismatchedContent` 的下游影响**：

移动端 `sync_service.rs` 处理 `SYNC_DIFF_RESULTS` 时，对 Agent/Group 类型会检查 `mismatchedContent`：

```rust
if data_type == SyncDataType::Agent || data_type == SyncDataType::Group {
    let is_mismatched = item["mismatchedContent"].as_bool().unwrap_or(false);
    if action == "PUSH" || action == "PULL" || is_mismatched {
        changed_owners.lock().await.insert(id.clone());
    }
}
```

无论 Action 是 `PUSH`、`PULL` 还是 `SKIP`，只要 `mismatchedContent == true`，该 Owner 就会被加入 `changed_owners` 集合，触发后续 Phase 2 的 targeted Topic Manifest 比对。

### 5.4 Phase 2.5：Topic Hash 批量验证

在 Phase 2 的 Topic Metadata 拉取完成后，双端进入 Phase 2.5 的 **Topic Hash 批量验证**（`SYNC_TOPIC_HASH_BATCH_V2` / `SYNC_TOPIC_HASH_RESULTS`）。

移动端发送：

```json
{
  "type": "SYNC_TOPIC_HASH_BATCH_V2",
  "hashes": {
    "topic_001": { "configHash": "abc...", "contentHash": "def..." },
    "topic_002": { "configHash": "ghi...", "contentHash": "jkl..." }
  }
}
```

桌面端 `handleSyncTopicHashBatchV2`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js`）执行比对：

```
for each (topicId, remoteHashes) in payload.hashes:
    topicRow ← db.query("SELECT hash, aggregated_hash FROM entity_index WHERE id = ? AND type = 'topic'")
    if !topicRow:
        changedTopics.push(topicId); continue

    localConfig  ← topicRow.hash || ""
    localContent ← topicRow.aggregated_hash || ""
    remoteConfig  ← remoteHashes.configHash || ""
    remoteContent ← remoteHashes.contentHash || ""

    if localConfig == remoteConfig && localContent == remoteContent:
        matchCount++
    else:
        changedTopics.push(topicId)
```

桌面端返回：

```json
{
  "type": "SYNC_TOPIC_HASH_RESULTS",
  "changedTopics": ["topic_001", "topic_003"]
}
```

只有出现在 `changedTopics` 中的话题才会进入 Phase 3 的消息级同步，其余话题被完全跳过。

## 6. targetedOwners 过滤机制

### 6.1 启用条件

targeted 同步并非始终启用。它仅在以下条件下触发：

| 条件 | 说明 |
|------|------|
| 当前处于 Phase 2 | Phase 1 完成后，`changed_owners` 集合非空 |
| `dataType == "topic"` | 仅对 Topic 类型启用 targeted 过滤 |
| `targetedOwners` 数组有效且非空 | 由移动端在 `SYNC_MANIFEST` 消息中显式携带 |

如果 `changed_owners` 为空（Phase 1 中所有 Agent/Group 的 configHash 和 contentHash 均一致），则 Phase 2 被跳过，直接进入 Phase 2.5/3。

### 6.2 targetedOwners 列表的计算

`targetedOwners` 在移动端 `sync_service.rs` 中动态计算：

```rust
// Phase 1 处理 SYNC_DIFF_RESULTS 时
if data_type == SyncDataType::Agent || data_type == SyncDataType::Group {
    if action == "PUSH" || action == "PULL" || is_mismatched {
        changed_owners.lock().await.insert(id.clone());
    }
}

// Phase 2 开始前
let owners = changed_owners.lock().await.iter().cloned().collect::<Vec<_>>();
```

即：**所有在 Phase 1 中发生 configHash 变更或 contentHash 变更的 Agent/Group ID**。

### 6.3 对 Manifest 构建的影响

#### 移动端（构建侧）

```rust
// build_targeted_topic_manifest
if owners.is_empty():
    return SyncManifest { data_type: Topic, items: [] }

let placeholders = owners.iter().map(|_| "?").collect::<Vec<_>>().join(", ");
let sql = format!(
    "SELECT topic_id, config_hash, content_hash, updated_at, owner_type, deleted_at
     FROM topics WHERE owner_id IN ({})", placeholders
);
```

- 查询范围从**全表**缩小到**特定 Owner 的话题子集**
- 参数化查询防止 SQL 注入

#### 桌面端（接收侧）

```javascript
// getLocalManifest 中
if (dataType === "topic" && Array.isArray(targetedOwners) && targetedOwners.length > 0) {
    const likeClauses = targetedOwners.map(() => "(file_path LIKE ? OR file_path LIKE ?)").join(" OR ");
    const params = [];
    targetedOwners.forEach(id => {
        params.push(`%Agents%${id}%config.json`);
        params.push(`%AgentGroups%${id}%config.json`);
    });
    rows = db.prepare(`SELECT * FROM entity_index WHERE type = 'topic' AND (${likeClauses})`).all(...params);
}
```

### 6.4 性能影响

targetedOwners 机制将 Phase 2 的 Topic Manifest 规模从 **O(全量 Topic)** 降低到 **O(变更 Owner 的 Topic)**。在大型知识库场景下（数千话题），这一优化可将清单传输量减少 **80%-95%**。

| 场景 | 全量 Manifest | Targeted Manifest | 缩减比例 |
|------|--------------|-------------------|---------|
| 100 Agents × 50 Topics | 5,000 条 | 50 条（仅 1 Agent 变更） | 99% |
| 100 Agents × 50 Topics | 5,000 条 | 500 条（10 Agents 变更） | 90% |
| 小型库（10 Agents × 5 Topics） | 50 条 | 5-50 条 | 0%-90% |

## 7. 软删除在 Diff 中的处理

VCP Mobile 采用**软删除（Soft Delete）**策略：实体被删除时并不从数据库移除，而是将 `deleted_at` 字段设为当前时间戳。这使得删除操作本身可以被同步——即"删除同步"（Delete Sync）。

### 7.1 deleted_at 的参与规则

`deleted_at` 在 Diff 算法中拥有最高优先级：**只要任一方的 `deleted_at` 非空，就优先处理删除状态，跳过 Hash 比对**。

```
// 在 handleSyncManifest 中
if remote.deletedAt:
    // 直接进入删除分支，不进行 configHash/contentHash 比较
    ...
else if local.deletedAt:
    // 直接进入 PUSH_DELETE 分支
    ...
else:
    // 双方均未删除，才进入 Hash 比对
    ...
```

### 7.2 DELETE 与 PUSH_DELETE 的区别

| Action | 触发条件 | 执行方 | 数据流向 | 是否通知对端 |
|--------|---------|--------|---------|------------|
| `DELETE` | 远程已删除，本地未删除 | 桌面端执行 | 单向：远程 → 本地 | 否（桌面端被动响应） |
| `PUSH_DELETE` | 本地已删除，远程未删除（或远程不存在） | 双端执行 | 单向：本地 → 远程 | 是（移动端执行后会发送 `SYNC_ENTITY_DELETE` 通知桌面端） |

**在移动端 `sync_service.rs` 中的差异化处理**：

```rust
if action == "DELETE" || action == "PUSH_DELETE" {
    // 1. 执行本地软删除
    match data_type {
        Agent => DeleteExecutor::soft_delete_agent(&h, &id).await,
        Group => DeleteExecutor::soft_delete_group(&h, &id).await,
        Topic => DeleteExecutor::soft_delete_topic(&h, &id).await,
        Avatar => DeleteExecutor::soft_delete_avatar(&h, type, owner_id).await,
        _ => {}
    }

    // 2. PUSH_DELETE 额外通知桌面端
    if action == "PUSH_DELETE" {
        tx_internal.send(SyncCommand::NotifyDelete { data_type, id });
    }
}
```

### 7.3 删除同步的边界情况

| 场景 | 移动端状态 | 桌面端状态 | 桌面端返回 Action | 移动端行为 |
|------|-----------|-----------|------------------|-----------|
| 双端均未删除 | `deleted_at: None` | `deletedAt: null` | Hash 比对决定 | 正常同步 |
| 移动端已删，桌面端未删 | `deleted_at: Some(ts)` | `deletedAt: null` | `PUSH_DELETE` | 软删除本地（幂等）+ 发送删除通知 |
| 移动端未删，桌面端已删 | `deleted_at: None` | `deletedAt: Some(ts)` | `DELETE` | 执行本地软删除 |
| 双端均已删除 | `deleted_at: Some(ts1)` | `deletedAt: Some(ts2)` | 无 | 无操作 |
| 桌面端不存在，移动端已删 | `deleted_at: Some(ts)` | 不存在 | `PUSH_DELETE` | 软删除本地 + 发送通知（桌面端已删或从未存在） |
| 移动端不存在，桌面端已删 | 不存在 | `deletedAt: Some(ts)` | `DELETE` | 无操作（移动端无此数据） |

## 8. Diff 结果的数据结构

### 8.1 桌面端返回格式

桌面端通过 WebSocket 返回统一的 `SYNC_DIFF_RESULTS` 消息：

```json
{
  "type": "SYNC_DIFF_RESULTS",
  "data": [
    { "id": "agent_001", "action": "PUSH", "ownerType": "agent" },
    { "id": "agent_002", "action": "PULL", "ownerType": "agent" },
    { "id": "agent_003", "action": "SKIP", "mismatchedContent": true, "ownerType": "agent" },
    { "id": "group_001", "action": "DELETE", "deletedAt": 1715500000000 },
    { "id": "topic_007", "action": "PUSH_DELETE", "deletedAt": 1715600000000, "ownerType": "group" }
  ],
  "dataType": "agent",
  "phase": 1
}
```

结果数组中每条记录的可能字段：

| 字段 | 类型 | 出现条件 | 说明 |
|------|------|---------|------|
| `id` | String | 始终 | 实体唯一标识 |
| `action` | String | 始终 | `PUSH` / `PULL` / `DELETE` / `PUSH_DELETE` / `SKIP` |
| `ownerType` | String | Topic / Agent / Group | 用于路由到正确的 Pull/Push Executor |
| `deletedAt` | Number | `DELETE` / `PUSH_DELETE` | 软删除时间戳 |
| `mismatchedContent` | Boolean | 仅 Agent/Group | 标记 contentHash 不一致，用于触发 targeted sync |

### 8.2 移动端消费方式

移动端 `sync_service.rs` 收到 `SYNC_DIFF_RESULTS` 后，执行三层分类派发：

```rust
let mut batch_pull_requests = Vec::new();   // 批量拉取
let mut push_topics_to_fetch = Vec::new();  // 需要推送的 Topic 元数据
let mut other_items = Vec::new();           // 单条处理（删除、Avatar 等）

for item in items_clone {
    let id = item["id"].as_str().unwrap_or_default().to_string();
    let action = item["action"].as_str().unwrap_or_default().to_string();

    // V2: 收集 changed_owners
    if data_type == Agent || data_type == Group {
        if action == "PUSH" || action == "PULL" || mismatchedContent {
            changed_owners.lock().await.insert(id.clone());
        }
    }

    if action == "PULL" && data_type in [Topic, Agent, Group] {
        batch_pull_requests.push(json!({ "id": id, "type": type_str }));
    } else if action == "PUSH" && data_type == Topic {
        push_topics_to_fetch.push((id, owner_id, owner_type));
    } else {
        other_items.push(item);
    }
}
```

#### 分类处理策略

| 分类 | 数据类型 | 处理方式 | 批大小 |
|------|---------|---------|--------|
| `batch_pull_requests` | Agent / Group / Topic | `PullExecutor::pull_entities_batch` | Agent/Group: 50；Topic: 1000 |
| `push_topics_to_fetch` | Topic | 先查本地 DB 组装 DTO，再 `PushExecutor::push_entities_batch` | 1000 |
| `other_items` | Avatar / 删除 / SKIP | 并发逐条处理（`for_each_concurrent(15)`） | 单条 |

#### 进度追踪与死锁防护

移动端维护两个原子计数器：

```rust
pending_tasks: AtomicU32   // 剩余未完成的任务数
total_tasks: AtomicU32     // 总任务数
```

每条 Action（除 `SKIP` 外）都会使 `pending_tasks` 递增。任务完成后递减。当所有 Manifest 响应收到且 `pending_tasks == 0` 时，自动触发下一阶段（Phase 1 → Phase 2 → Phase 2.5 → Phase 3）。

为防止 Executor 死锁导致同步停滞，系统还部署了 **Watchdog（看门狗）** 机制：每 10 秒检查一次 `pending_tasks`，若连续 60 秒（6 个 tick）无变化，则强制发送阶段推进命令（`src-tauri/src/vcp_modules/sync/sync_service.rs:915-943`）。

### 8.3 日志与可观测性

桌面端在 Diff 完成后输出结构化统计日志：

```
[owner_metadata] diff agent success: push=3 pull=2 delete=1 push_delete=0
```

移动端在收到 Diff 后同样记录：

```rust
let msg = format!("[{}] Diff: pull={} push={} delete={} push_delete={}",
    data_type, pull_count, push_count, delete_count, push_delete_count);
```

Skip 数量通过独立计算获得（双端 Hash 完全一致且未删除的条目），不纳入 `pending_tasks` 统计。

## 附录：Diff 算法完整伪代码

```
function DiffAlgorithm(remoteManifest, localManifest, dataType):
    results       ← empty list
    processedIds  ← empty set

    // ── Round 1: Iterate over remote items ──
    for remote in remoteManifest.items:
        local ← findLocalById(remote.id)

        if remote.deletedAt != null:
            if local == null || local.deletedAt == null:
                results.add({ id: remote.id, action: DELETE, deletedAt: remote.deletedAt })
            processedIds.add(remote.id)
            continue

        if local == null:
            results.add({ id: remote.id, action: PUSH, ownerType: remote.ownerType })
            continue

        if local.deletedAt != null:
            results.add({ id: local.id, action: PUSH_DELETE, deletedAt: local.deletedAt, ownerType: local.ownerType })
            processedIds.add(local.id)
            continue

        // Both exist and not deleted → V2 dual-hash compare
        rCfg ← remote.configHash || remote.hash
        rCnt ← remote.contentHash || ""
        lCfg ← local.configHash || local.hash
        lCnt ← local.contentHash || ""
        changed ← false

        if lCfg ≠ rCfg:
            if remote.ts > local.ts:
                results.add({ id: remote.id, action: PUSH, ownerType: remote.ownerType })
            else:
                results.add({ id: local.id, action: PULL, ownerType: local.ownerType })
            changed ← true

        if dataType ∈ [agent, group] and lCnt ≠ rCnt:
            existing ← results.find(r => r.id == remote.id)
            if existing:
                existing.mismatchedContent ← true
            else:
                results.add({ id: remote.id, action: SKIP, mismatchedContent: true, ownerType: remote.ownerType })

        processedIds.add(remote.id)

    // ── Round 2: Iterate over local items not in processedIds ──
    for local in localManifest.items:
        if processedIds.contains(local.id): continue
        if remoteManifest.hasId(local.id): continue

        if local.deletedAt != null:
            results.add({ id: local.id, action: PUSH_DELETE, deletedAt: local.deletedAt, ownerType: local.ownerType })
        else:
            results.add({ id: local.id, action: PULL, ownerType: local.ownerType })

    return results
```
