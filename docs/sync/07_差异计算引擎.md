# 差异计算引擎（Diff Engine）

## 1. 引言

差异计算引擎（Diff Engine，下文简称"引擎"）是 VCP Mobile 同步协议中承上启下的核心组件，负责在移动端与桌面端之间快速识别数据不一致的精确范围。引擎横跨 **Phase 2.5**（话题哈希批量验证）与 **Phase 3**（消息级差异计算）两个同步阶段，其设计目标是在保证正确性的前提下，最大限度地减少网络传输量与数据库查询次数。

在同步管道的整体架构中，引擎的定位如下：

| 阶段 | 名称 | 引擎职责 | 输入来源 | 输出去向 |
|------|------|----------|----------|----------|
| Phase 2 | Topic Metadata Pull | 非引擎范畴：拉取缺失话题配置 | Phase 1 筛选的变更 Owner | 本地数据库写入队列 |
| **Phase 2.5** | **Topic Hash Validation** | **双哈希批量比对，判定哪些话题需要进入消息同步** | 本地 `topics` 表双哈希 | 桌面端 `diff.js` |
| **Phase 3** | **Message Diff & Sync** | **消息级 Fast Path / Detailed Path 差异计算** | 本地 `messages` 表逐条哈希 | 桌面端 `diff.js` |
| Phase 4 | Finalize | 非引擎范畴：Hash 冒泡与一致性校验 | Phase 3 修改记录 | 本地数据库事务 |

引擎的运作遵循一个基本原则：**先粗后精（Coarse-to-Fine）**。即在 Phase 2.5 通过话题级哈希快速排除完全一致的话题，仅在必要时才在 Phase 3 展开逐条消息比对。这一策略在典型场景下（少量消息变更）可将计算复杂度从 O(N_messages) 降低到 O(N_topics)。

本文档详细描述引擎涉及的三种批量差异计算协议：Topic Hash Batch V1、Topic Hash Batch V2（当前主力版本）以及 Message Diff Batch。

### 1.1 移动端差异处理器（v0.9.14 重构）

v0.9.14 将原先内联在 `sync_service.rs` 中的差异处理逻辑提取为两个独立处理器，使 WebSocket 消息循环与差异计算逻辑解耦：

| 处理器 | 源文件 | 处理消息类型 | 核心职责 |
|--------|--------|-------------|---------|
| `DiffHandler` | `sync_executor/diff_handler.rs` | `SYNC_DIFF_RESULTS` | Phase 1/2/2.5 的单批次差异结果处理：op 计数、watchdog 启动、任务分类（batch pull / push topics / other items）、并发任务分发 `for_each_concurrent(15, ...)` |
| `BatchDiffHandler` | `sync_executor/batch_diff_handler.rs` | `SYNC_DIFF_RESULTS_BATCH` | Phase 3 的多批次差异结果处理：push-before-pull 排序、per-topic 成功/失败追踪、`pending_diff_batches` 队列链式触发 |

**提取动机**：`sync_service.rs` 原内联了约 900 行的 diff 处理逻辑（`SYNC_DIFF_RESULTS` 约 400 行 + `SYNC_DIFF_RESULTS_BATCH` 约 200 行 + Phase 3 收尾约 180 行），导致服务循环文件超过 1600 行，难以维护。提取后：
- `sync_service.rs` 聚焦 WebSocket 连接管理、命令分发和状态机驱动；
- `diff_handler.rs` 和 `batch_diff_handler.rs` 各自独立演进，便于单元测试和代码审查；
- Phase 3 收尾逻辑进一步提取至 `sync_finalize.rs`（见 `docs/sync/05_流水线与阶段控制.md`）。


## 2. Topic Hash Batch V1

Topic Hash Batch V1（话题哈希批量比对第一版）是同步协议早期采用的粗粒度差异检测机制。该版本已在生产环境中被 V2 取代，但理解其设计有助于把握演进动机。

### 2.1 消息格式

移动端向桌面端发送 `SYNC_TOPIC_HASH_BATCH` 消息，载荷（Payload）结构如下：

```json
{
  "type": "SYNC_TOPIC_HASH_BATCH",
  "hashes": {
    "topic-uuid-1": "a1b2c3d4...",
    "topic-uuid-2": "e5f6g7h8..."
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 固定为 `SYNC_TOPIC_HASH_BATCH` |
| `hashes` | object | Key 为 `topic_id`，Value 为单哈希字符串（`content_hash` / `aggregated_hash`） |

此处的哈希值由移动端 `phase3_message.rs` 中的 `get_targeted_topic_hashes` 早期版本提供，仅包含单一内容哈希（`content_hash`），对应数据库 `topics` 表的 `content_hash` 字段（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js` 中记为 `aggregated_hash`）。

### 2.2 比对逻辑

桌面端插件 `diff.js` 中的 `handleSyncTopicHashBatch` 函数执行以下逻辑：

```pseudo
function handleSyncTopicHashBatch(payload):
    changedTopics = []
    for each (topicId, localHash) in payload.hashes:
        topicRow = db.query(
            "SELECT aggregated_hash FROM entity_index 
             WHERE id = ? AND type = 'topic' AND deleted_at IS NULL"
        )
        if topicRow exists AND topicRow.aggregated_hash == localHash:
            continue  // 一致，无需同步
        else:
            changedTopics.append(topicId)  // 不一致或不存在
    return { type: "SYNC_TOPIC_HASH_RESULTS", changedTopics }
```

**关键判定规则**：

| 场景 | 桌面端状态 | 判定结果 |
|------|-----------|----------|
| 话题存在，哈希相等 | `aggregated_hash === localHash` | 跳过（无变更） |
| 话题存在，哈希不等 | `aggregated_hash !== localHash` | 加入 `changedTopics` |
| 话题不存在 | `topicRow` 为 `undefined` | 加入 `changedTopics`（保守处理） |
| 查询异常 | SQL 报错 | 加入 `changedTopics`（保守处理） |

### 2.3 输出格式

桌面端返回 `SYNC_TOPIC_HASH_RESULTS`：

```json
{
  "type": "SYNC_TOPIC_HASH_RESULTS",
  "changedTopics": ["topic-uuid-1", "topic-uuid-2"]
}
```

移动端 `sync_service.rs` 在收到该响应后，将 `changedTopics` 写入 `changed_topics` 共享状态，随即触发 Phase 3 的消息差异计算：

```rust
// sync_service.rs 节选
Some("SYNC_TOPIC_HASH_RESULTS") => {
    manifest_phase.store(3, Ordering::SeqCst);
    if let Some(changed) = payload["changedTopics"].as_array() {
        let changed_ids: Vec<String> = changed.iter()
            .filter_map(|v| v.as_str().map(|s| s.to_string()))
            .collect();
        {
            let mut guard = changed_topics.lock().await;
            *guard = changed_ids;
        }
        let _ = tx_internal.send(SyncCommand::StartMessages);
    }
}
```

### 2.4 V1 的局限性

V1 的核心缺陷在于**仅使用单一哈希值（`aggregated_hash` / `content_hash`）代表整个话题的状态**。这导致以下问题：

| 局限性 | 具体表现 |
|--------|----------|
| 配置变更被掩盖 | 话题标题、锁定状态、未读标记等元数据变更不会改变 `content_hash`，导致移动端无法感知配置更新 |
| 伪同步（False Sync） | 若桌面端消息未变但话题配置变了，V1 会误判为一致，造成双端配置漂移 |
| 粒度不足 | 无法区分"配置变更"与"消息变更"，Phase 3 必须对所有变更话题进行全量消息比对 |

这些局限性直接催生了 V2 的双哈希设计。


## 3. Topic Hash Batch V2（重点）

Topic Hash Batch V2（话题哈希批量比对第二版）是当前同步协议的主力实现，通过引入**双哈希架构（Dual-Hash Architecture）**——`configHash`（配置哈希）与 `contentHash`（内容哈希）——精确区分话题元数据变更与消息内容变更，从根本上解决了 V1 的配置漂移问题。

### 3.1 消息格式

移动端发送 `SYNC_TOPIC_HASH_BATCH_V2`：

```json
{
  "type": "SYNC_TOPIC_HASH_BATCH_V2",
  "hashes": {
    "topic-uuid-1": {
      "configHash": "sha256-of-config...",
      "contentHash": "sha256-of-messages..."
    },
    "topic-uuid-2": {
      "configHash": "sha256-of-config...",
      "contentHash": "sha256-of-messages..."
    }
  }
}
```

| 字段 | 类型 | 来源 | 覆盖范围 |
|------|------|------|----------|
| `configHash` | string | `topics.config_hash` | 话题标题、锁定状态、未读标记、Owner 关系等元数据 |
| `contentHash` | string | `topics.content_hash` | 话题下所有消息的集合哈希（通常基于 Merkle Tree 或排序后串联哈希） |

移动端通过 `phase3_message.rs` 中的 `get_targeted_topic_hashes` 批量查询双哈希：

```pseudo
function get_targeted_topic_hashes(pool, owners):
    result = {}
    rows = sqlx_query(
        "SELECT topic_id, config_hash, content_hash 
         FROM topics 
         WHERE owner_id IN (owners) AND deleted_at IS NULL"
    )
    for row in rows:
        result[row.topic_id] = (row.config_hash, row.content_hash)
    return result
```

该查询使用动态占位符（`placeholders`）实现安全的 `IN` 子句批量查询，避免 N+1 问题。

### 3.2 双哈希比对逻辑

桌面端 `diff.js` 中的 `handleSyncTopicHashBatchV2` 执行比对：

```pseudo
function handleSyncTopicHashBatchV2(payload):
    changedTopics = []
    for each (topicId, remoteHashes) in payload.hashes:
        topicRow = db.query(
            "SELECT hash, aggregated_hash FROM entity_index 
             WHERE id = ? AND type = 'topic' AND deleted_at IS NULL"
        )
        if topicRow is null:
            changedTopics.append(topicId)
            continue
        
        localConfig  = topicRow.hash || ""
        remoteConfig = remoteHashes.configHash || ""
        localContent = topicRow.aggregated_hash || ""
        remoteContent = remoteHashes.contentHash || ""
        
        if localConfig == remoteConfig AND localContent == remoteContent:
            continue  // 完全一致
        else:
            changedTopics.append(topicId)  // 任一哈希不匹配
    return { type: "SYNC_TOPIC_HASH_RESULTS", changedTopics }
```

**双哈希判定矩阵**：

| 桌面端 `configHash` | 桌面端 `contentHash` | 移动端 `configHash` | 移动端 `contentHash` | 判定结果 | 实际含义 |
|---------------------|----------------------|---------------------|----------------------|----------|----------|
| 匹配 | 匹配 | — | — | **跳过** | 双端完全一致 |
| 匹配 | 不匹配 | — | — | **变更** | 消息内容不一致，需 Phase 3 消息同步 |
| 不匹配 | 匹配 | — | — | **变更** | 话题配置不一致，需元数据同步 |
| 不匹配 | 不匹配 | — | — | **变更** | 配置与消息均不一致 |
| 不存在 | — | — | — | **变更** | 桌面端无此话题，保守处理 |

### 3.3 输出格式

V2 的返回格式与 V1 保持一致（向后兼容）：

```json
{
  "type": "SYNC_TOPIC_HASH_RESULTS",
  "changedTopics": ["topic-uuid-1", "topic-uuid-2"]
}
```

移动端处理逻辑也复用同一分支（`SYNC_TOPIC_HASH_RESULTS`），无需额外协议字段即可区分 V1/V2。这种设计确保了协议升级的无缝过渡。

### 3.4 V2 的关键改进

| 改进维度 | V1 | V2 | 收益 |
|----------|-----|-----|------|
| 哈希维度 | 单哈希（`aggregated_hash`） | 双哈希（`configHash` + `contentHash`） | 可独立检测配置变更与消息变更 |
| 配置同步精度 | 低：配置变更无法被感知 | 高：配置哈希独立比对 | 消除话题元数据漂移 |
| 误判率 | 较高 | 显著降低 | 减少不必要的 Phase 3 全量消息比对 |
| 数据库查询 | 查询 `aggregated_hash` | 查询 `hash` + `aggregated_hash` | 查询成本增加约 1 个字段，可忽略 |
| 协议兼容性 | — | 返回格式与 V1 相同 | 平滑升级，无需修改移动端接收逻辑 |

**工程实践提示**：桌面端 `entity_index` 表的 `hash` 字段对应移动端 `topics` 表的 `config_hash`，`aggregated_hash` 字段对应 `content_hash`。两端字段命名差异源于历史原因，实际语义完全一致。


## 4. Message Diff Batch

Message Diff Batch（消息差异批量计算）是 Phase 3 的核心协议，负责在双端话题级哈希不匹配的前提下，精确计算出需要拉取（Pull）和推送（Push）的消息集合。该协议同样遵循**先粗后精**原则，引入 Fast Path 与 Detailed Path 两级加速机制。

### 4.1 消息格式

移动端向桌面端发送 `SYNC_MESSAGE_DIFF_BATCH`：

```json
{
  "type": "SYNC_MESSAGE_DIFF_BATCH",
  "topics": {
    "topic-uuid-1": {
      "topicHash": "topic-level-content-hash...",
      "messages": {
        "msg-uuid-a": "msg-hash-a...",
        "msg-uuid-b": "msg-hash-b..."
      }
    },
    "topic-uuid-2": {
      "topicHash": "topic-level-content-hash...",
      "messages": {
        "msg-uuid-c": "msg-hash-c..."
      }
    }
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 固定为 `SYNC_MESSAGE_DIFF_BATCH` |
| `topics` | object | Key 为 `topic_id`，Value 为该话题的本地状态 |
| `topicHash` | string | 该话题的内容哈希（`content_hash`），用于 Fast Path |
| `messages` | object | Key 为 `msg_id`，Value 为该消息的 `content_hash` |

移动端通过 `phase3_message.rs` 中的 `get_topic_message_hashes` 批量构建此载荷。该函数执行两步批量查询：

```pseudo
function get_topic_message_hashes(pool, topic_ids):
    // Step 1: 批量查询话题级 content_hash
    topic_rows = sqlx_query(
        "SELECT topic_id, content_hash FROM topics 
         WHERE topic_id IN (topic_ids) AND deleted_at IS NULL"
    )
    result = {}
    for row in topic_rows:
        result[row.topic_id] = { topic_hash: row.content_hash, messages: {} }
    
    // Step 2: 批量查询消息级 content_hash
    msg_rows = sqlx_query(
        "SELECT topic_id, msg_id, content_hash FROM messages 
         WHERE topic_id IN (topic_ids) AND deleted_at IS NULL"
    )
    for row in msg_rows:
        if result.has(row.topic_id):
            result[row.topic_id].messages[row.msg_id] = row.content_hash
    
    return result
```

**批次拆分策略**：为避免单次 WebSocket 载荷过大（约超过 2MB 可能导致部分网关截断），`sync_service.rs` 中的 `build_diff_batches` 将话题按消息数量分批次打包，每批上限为 **10,000 条消息**：

```pseudo
const MAX_MESSAGES_PER_BATCH = 10000

function build_diff_batches(topic_states):
    batches = []
    current_batch = {}
    current_msg_count = 0
    
    for (topic_id, state) in topic_states:
        msg_count = state.messages.length
        
        if current_msg_count > 0 AND current_msg_count + msg_count > MAX_MESSAGES_PER_BATCH:
            batches.append(current_batch)
            current_batch = {}
            current_msg_count = 0
        
        current_batch[topic_id] = {
            topicHash: state.topic_hash,
            messages: state.messages
        }
        current_msg_count += msg_count
    
    if current_batch not empty:
        batches.append(current_batch)
    
    return batches
```

批次以队列形式管理（`pending_diff_batches`），首批立即发送，后续批次在前一批的 `SYNC_DIFF_RESULTS_BATCH` 响应处理完毕后自动触发。

### 4.2 Fast Path 与 Detailed Path

桌面端 `diff.js` 中的 `handleSyncMessageDiffBatch` 实现了双路径差异计算：

#### 4.2.1 Fast Path（快速路径）

```pseudo
for each (topicId, localState) in payload.topics:
    topicRow = db.query(
        "SELECT aggregated_hash FROM entity_index 
         WHERE id = ? AND type = 'topic' AND deleted_at IS NULL"
    )
    
    if topicRow exists AND topicRow.aggregated_hash == localState.topicHash:
        results[topicId] = { toPull: [], toPush: false }
        fastPathCount++
        continue  // 快速跳过，无需逐条比对
```

Fast Path 的触发条件极为严格：话题级 `aggregated_hash`（桌面端）与 `topicHash`（移动端）必须**完全相等**。这意味着该话题下所有消息的集合哈希在双端一致，因此可以安全地跳过逐条消息比对。

**Fast Path 适用场景**：
- 话题消息数量大但无实际变更（历史话题）
- 仅话题配置变更（已通过 Phase 2.5 的 `configHash` 不匹配筛选进入 Phase 3，但消息实际未变）

#### 4.2.2 Detailed Path（详细路径）

当 Fast Path 未命中时，进入逐条消息哈希比对：

```pseudo
// 读取桌面端该话题的全部消息哈希
remoteRows = db.query(
    "SELECT msg_id, hash FROM message_index WHERE topic_id = ?"
)
remoteMap = Map(remoteRows.map(r => [r.msg_id, r.hash]))
localMap = localState.messages || {}

toPull = []
toPush = false

// 判定需要拉取的消息：桌面端有但移动端没有，或哈希不匹配
for (msgId, remoteHash) in remoteMap:
    localHash = localMap[msgId]
    if localHash is undefined:
        toPull.append(msgId)  // 移动端缺失此消息
    else if localHash !== remoteHash:
        toPull.append(msgId)  // 消息内容不一致

// 判定是否需要推送：移动端有而桌面端没有
for msgId in localMap.keys():
    if not remoteMap.has(msgId):
        toPush = true
        break  // 只需标记布尔值，具体消息列表由移动端在推送时生成

results[topicId] = { toPull, toPush }
detailedCount++
```

**Detailed Path 判定规则**：

| 桌面端状态 | 移动端状态 | 动作 |
|-----------|-----------|------|
| 有消息，哈希匹配 | 有消息，哈希匹配 | 无操作 |
| 有消息，哈希不匹配 | 有消息 | `toPull` 加入该消息 ID |
| 有消息 | 无消息 | `toPull` 加入该消息 ID |
| 无消息 | 有消息 | `toPush = true` |

### 4.3 输出格式

桌面端返回 `SYNC_DIFF_RESULTS_BATCH`：

```json
{
  "type": "SYNC_DIFF_RESULTS_BATCH",
  "results": {
    "topic-uuid-1": {
      "toPull": ["msg-uuid-a", "msg-uuid-b"],
      "toPush": false
    },
    "topic-uuid-2": {
      "toPull": [],
      "toPush": true
    },
    "topic-uuid-3": {
      "toPull": ["msg-uuid-c"],
      "toPush": true
    }
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `toPull` | string[] | 需要移动端从桌面端拉取的消息 ID 列表 |
| `toPush` | boolean | 移动端是否需要向桌面端推送该话题的消息 |

移动端 `sync_service.rs` 收到后，将话题分类为三种操作模式：

```pseudo
for (topicId, result) in results:
    toPullIds = result.toPull
    toPush = result.toPush
    
    if not toPush AND toPullIds.is_empty:
        mark_completed(topicId)  // 无操作，直接标记完成
        continue
    
    if toPush:
        push_topic_ids.append(topicId)
    if toPullIds not empty:
        pull_batch.append((topicId, toPullIds))
    // push-only 的话题在 push 后直接标记完成
```

操作顺序遵循 **先 Push 后 Pull** 原则，确保双端对同一话题的修改先合并移动端数据，再拉取桌面端数据，避免拉取后又被推送覆盖的竞态条件。


## 5. 变更话题的精确判定

变更话题的判定是差异计算引擎中"正确性"与"效率"的博弈点。VCP Mobile 采用**分层递进**策略，在不同阶段使用不同精度的判定标准，确保既不漏掉真实变更，也不产生大量无效的同步任务。

### 5.1 三层判定体系

| 层级 | 阶段 | 判定依据 | 精度 | 作用 |
|------|------|----------|------|------|
| L1 | Phase 1 | Owner 级哈希（Agent/Group Hash） | 粗 | 筛选出"可能"有内容变更的 Owner，排除完全一致的 Agent/Group |
| L2 | Phase 2.5 | 话题级双哈希（V2） | 中 | 在已筛选的 Owner 内，精确判定哪些话题需要进入消息同步 |
| L3 | Phase 3 | 消息级逐条哈希 | 精 | 在已判定为变更的话题内，精确识别每条消息的 Pull/Push 需求 |

### 5.2 Phase 1 → Phase 2.5 的衔接

Phase 1 通过 `SYNC_MANIFEST` 交换 Owner（Agent/Group）级哈希。桌面端在返回 `SYNC_DIFF_RESULTS` 时，对于 `action` 为 `PUSH`、`PULL` 或带有 `mismatchedContent: true` 标记的 Owner，会被移动端加入 `changed_owners` 集合：

```rust
// sync_service.rs 节选
if data_type == SyncDataType::Agent || data_type == SyncDataType::Group {
    let is_mismatched = item["mismatchedContent"].as_bool().unwrap_or(false);
    if action == "PUSH" || action == "PULL" || is_mismatched {
        let mut owners = changed_owners.lock().await;
        owners.insert(id.clone());
    }
}
```

`changed_owners` 直接决定了 Phase 2.5 需要比对哪些 Owner 下属的话题，将话题比对范围从"全量话题"缩小到"可能变更的 Owner 的话题"。

### 5.3 Phase 2.5 → Phase 3 的衔接

Phase 2.5 输出的 `changedTopics` 是 Phase 3 的唯一输入。这一衔接的严格性保证了：

- **若话题未进入 `changedTopics`**：Phase 3 完全不会对其进行任何消息级操作，节省数据库查询与网络传输
- **若话题进入 `changedTopics`**：Phase 3 会展开完整的 Fast Path / Detailed Path 计算，确保不遗漏任何消息级差异

### 5.4 保守性原则（Conservative Strategy）

引擎在多个节点采用保守性设计，宁可误判为"有变更"，也不漏掉真实变更：

| 场景 | 处理方式 | 代码位置 |
|------|----------|----------|
| 桌面端查询话题哈希时数据库未初始化 | 返回空 `changedTopics` | `diff.js: handleSyncTopicHashBatchV2` |
| 桌面端查询具体话题时抛出异常 | 将该话题加入 `changedTopics` | `diff.js: handleSyncTopicHashBatchV2` |
| 消息差异计算时数据库异常 | `toPull: [], toPush: false, error: msg` | `diff.js: handleSyncMessageDiffBatch` |
| 桌面端某话题不存在 | 视为有变更 | `diff.js: handleSyncTopicHashBatchV2` |

这种保守策略的代价是偶尔产生不必要的 Phase 3 Detailed Path 计算，但收益是**绝不因哈希比对错误导致数据静默不一致**。

### 5.5 变更话题的状态流转

一个话题从 Phase 1 到 Phase 3 完整的状态流转如下：

```pseudo
Phase 1: Owner Manifest Diff
    ↓ Owner 被标记为 mismatched / PUSH / PULL
Phase 2: Topic Metadata Pull (如需要)
    ↓ 该 Owner 下所有话题被纳入 Phase 2.5 比对范围
Phase 2.5: Topic Hash Batch V2
    ├─ configHash 不匹配 → 加入 changedTopics
    ├─ contentHash 不匹配 → 加入 changedTopics
    └─ 双哈希均匹配 → 跳过
    ↓ changedTopics 非空
Phase 3: Message Diff Batch
    ├─ Fast Path: topicHash 匹配 → 无操作，标记完成
    └─ Detailed Path: 逐条比对 → 生成 toPull / toPush
    ↓
Phase 4: Hash Re-calculation & Bubble
```


## 6. 性能优化策略

差异计算引擎在移动端（SQLite + ARM 架构）与桌面端（Node.js + 可能的大型数据库）两端均面临性能约束。以下从 Fast Path 命中率、批量处理优势与内存占用三个维度分析其优化设计。

### 6.1 Fast Path 命中率

Fast Path 是 Phase 3 中最重要的性能优化手段，其命中率直接决定了消息同步的平均时间复杂度。

**命中条件**：桌面端 `entity_index.aggregated_hash` == 移动端 `topicHash`（即 `topics.content_hash`）

**典型场景与命中率预估**：

| 场景 | Fast Path 命中率 | 原因分析 |
|------|------------------|----------|
| 首次同步 | ~0% | 双端话题均无历史哈希，全部进入 Detailed Path |
| 日常增量同步（少量消息变更） | ~70-90% | 大部分历史话题消息未变，仅活跃话题需要 Detailed Path |
| 仅话题配置变更 | ~100% | Phase 2.5 的 `configHash` 不匹配导致进入 Phase 3，但 `contentHash` 一致，Fast Path 命中 |
| 大批量消息导入后 | ~10-30% | 大量话题消息发生变更，Detailed Path 比例上升 |

**V2 对 Fast Path 的增益**：在 V1 时代，若话题配置变更（如重命名），`contentHash` 可能未变，但 V1 的单哈希设计无法区分这一情况。V2 的 Phase 2.5 通过 `configHash` 独立筛选，确保仅因配置变更进入 Phase 3 的话题能以 100% Fast Path 命中快速通过，无需 Detailed Path 开销。

### 6.2 批量处理优势

引擎在多个层面采用批量处理（Batching）降低往返时延（Round-Trip Time, RTT）与数据库查询次数：

| 优化点 | 具体实现 | 性能收益 |
|--------|----------|----------|
| 话题哈希批量查询 | `get_targeted_topic_hashes` 使用单条 `IN` 查询替代 N 次单话题查询 | 查询次数从 O(N) 降至 O(1) |
| 消息哈希批量查询 | `get_topic_message_hashes` 两步批量查询：先查话题哈希，再查消息哈希 | 两趟数据库查询覆盖全部目标话题 |
| 差异批量发送 | `build_diff_batches` 将多个话题打包为单条 WS 消息 | 减少 WS 帧数量，降低 RTT 影响 |
| 差异结果批量处理 | 桌面端一次性返回所有话题的 `toPull` / `toPush` 结果 | 移动端可并行调度 Push/Pull 任务 |
| Push/Pull 批量执行 | `PushExecutor::push_messages_batch` 与 `PullExecutor::pull_messages_batch` | 减少 HTTP 请求次数 |

**批次大小权衡**：

`MAX_MESSAGES_PER_BATCH` 设定为 **10,000 条消息**，对应约 1.5-2MB 的 JSON 载荷。此数值的选取基于以下考量：

- **上限约束**：部分 WebSocket 网关/代理对单帧大小有限制（通常 1-4MB），超过可能导致连接断开
- **下限约束**：批次过小会增加 RTT 往返次数，在移动网络高延迟场景下显著拖慢同步速度
- **内存平衡**：10,000 条消息的哈希字符串（每条约 64 字节 Hex）约占用 640KB 内存，加上 JSON 开销约 1.5MB，处于移动端安全内存预算内

### 6.3 内存占用

差异计算引擎在内存使用上采取**流式与分片**策略，避免一次性加载全量数据：

| 组件 | 内存策略 | 峰值占用估算 |
|------|----------|-------------|
| `get_targeted_topic_hashes` | 仅加载目标 Owner 的话题双哈希 | ~Owner 数量 × 平均话题数 × 128B |
| `get_topic_message_hashes` | 仅加载 Phase 2.5 筛选出的变更话题的消息哈希 | ~变更话题数 × 平均消息数 × 96B |
| `build_diff_batches` | 分片输出为 `VecDeque`，不保留完整副本 | 当前批次 ≤ 10,000 条消息 |
| `pending_diff_batches` | 异步队列，当前批次处理完毕后释放 | 剩余批次 × 平均批次大小 |

**关键内存安全设计**：

- **批量查询优先**：`phase3_message.rs` 中的所有查询均使用 `IN` 子句批量执行，避免 N+1 查询带来的内存与 CPU 双重压力
- **队列清理**：`sync_service.rs` 在 Phase 3 启动前显式清空 `pending_diff_batches`，防止断线重连后发送过期的差异数据：

```rust
// sync_service.rs 节选（Phase 3 启动时）
{
    let mut pending = pending_diff_batches.lock().await;
    pending.clear();  // 清空可能残留的旧批次
}
```

- **并发控制**：`NetworkAwareSemaphore` 根据设备核心数动态限制并发数（`cores × 1.5`， clamp 在 6-12 之间），防止同时发起过多 HTTP 请求导致内存与网络资源耗尽

### 6.4 性能监控与可观测性

引擎内置多层次的性能日志，便于后续分析与调优：

| 日志位置 | 内容 | 用途 |
|----------|------|------|
| `diff.js: handleSyncTopicHashBatchV2` | `total=N match=M changed=K` | 监控 Phase 2.5 的跳过率 |
| `diff.js: handleSyncMessageDiffBatch` | `topics=N fast_path=F detailed=D` | 监控 Phase 3 Fast Path 命中率 |
| `diff.js: handleSyncMessageDiffBatch` | `toPull=X toPush=Y`（逐话题） | 定位消息变更热点话题 |
| `sync_service.rs` | `Phase3 diff split into N batches` | 监控批次拆分合理性 |

**调优建议**：
- 若日常同步中 Phase 3 `detailed` 占比持续高于 50%，建议检查 Hash 冒泡机制是否正常工作，可能存在话题级哈希未正确更新的情况
- 若 `pending_diff_batches` 经常拆分为 3 批次以上，说明单个同步会话涉及的变更消息量过大，可考虑提升 `MAX_MESSAGES_PER_BATCH` 或优化哈希存储格式
