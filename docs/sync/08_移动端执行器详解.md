# 移动端执行器详解

## 1. 引言

在 VCP Mobile 的**双轨三层（Double-Track 3-Tier）**同步架构中，`sync_executor` 模块承担着"最后一公里"的数据搬运职责。它将同步管道（`sync_pipeline`）解析出的抽象指令转化为具体的 HTTP 请求与数据库操作，是连接"决策层"与"物理 IO 层"的关键枢纽。

本模块由三个执行器（Executor）组成，各自负责单向或双向的数据流动：

| 执行器 | 职责 | 数据流向 | 核心源文件 |
|--------|------|----------|-----------|
| **PullExecutor** | 从桌面端拉取数据到移动端 | 桌面端 → 移动端 | `pull_executor.rs` |
| **PushExecutor** | 将移动端数据推送到桌面端 | 移动端 → 桌面端 | `push_executor.rs` |
| **DeleteExecutor** | 处理双向删除传播 | 双向（以移动端为准触发） | `delete_executor.rs` |

三者的协作关系可概括为以下伪代码：

```
sync_cycle:
    // 阶段1：拉取桌面端变更
    results_pull = PullExecutor::pull_entities_batch(...)
    results_msg  = PullExecutor::pull_messages_batch(...)
    
    // 阶段2：推送移动端变更
    results_push = PushExecutor::push_entities_batch(...)
    results_msg  = PushExecutor::push_messages_batch(...)
    
    // 阶段3：清理（按需触发）
    if deletion_queue.not_empty():
        DeleteExecutor::soft_delete_*(...)
    
    // 阶段4：周期性物理清理
    if schedule.due:
        DeleteExecutor::cleanup_old_deleted_records(days=30)
```

**归口优化（Batch Consolidation）** 是本模块的核心设计哲学：无论是 Pull 还是 Push，优先使用批量端点（`/download-entities`、`/upload-messages-batch` 等），将分散的实体操作聚合成单次 HTTP 往返，以降低移动网络下的 TLS 握手开销与蜂窝电台唤醒成本。

## 2. PullExecutor

### 2.1 职责边界

`PullExecutor`（拉取执行器）负责将桌面端（Desktop）的最新数据复制到移动端本地数据库。其处理范围涵盖：

- **实体（Entity）**：Agent、Group、AgentTopic、GroupTopic
- **消息（Message）**：按 Topic 分帧的流式消息批量下载
- **头像（Avatar）**：Agent / Group 的二进制头像图片
- **附件（Attachment）**：消息关联的文件资源（在消息处理管线中间接涉及）

所有拉取到的数据**不直接写入数据库**，而是通过 `DbWriteQueue`（数据库写入队列）异步排队，以串行化 SQLite 写入、避免 WAL（Write-Ahead Logging）模式下的并发锁竞争。

### 2.2 HTTP 调用链

#### 2.2.1 批量实体下载

```
POST /api/mobile-sync/download-entities
Body: { "requests": [{ "id": "...", "type": "agent|group|agent_topic|group_topic" }] }
```

对应实现为 `PullExecutor::pull_entities_batch`（`pull_executor.rs`）。服务端返回 JSON 数组，每个元素包含 `id`、`type`、`data` 三个字段。执行器根据 `type` 将 `data` 反序列化为对应的 DTO（Data Transfer Object，数据传输对象），并提交到 `DbWriteQueue`。

**Topic 过滤规则**：对于 `agent_topic` 和 `group_topic` 类型，`id == "default"` 的条目会被静默跳过。这是为了避免桌面端可能存在的默认占位 Topic 污染移动端数据。

#### 2.2.2 单实体下载（降级路径）

当批量端点不可用或需要单独更新某个实体时，提供以下单实体接口：

| 方法 | 端点 | 说明 |
|------|------|------|
| `pull_agent` | `GET /download-entity?id={id}&type=agent` | 单 Agent 下载 |
| `pull_group` | `GET /download-entity?id={id}&type=group` | 单 Group 下载 |
| `pull_agent_topic` | `GET /download-entity?id={id}&type=agent_topic` | 单 Agent Topic 下载，404 静默跳过 |
| `pull_group_topic` | `GET /download-entity?id={id}&type=group_topic` | 单 Group Topic 下载，404 静默跳过 |
| `pull_avatar` | `GET /download-avatar?id={id}&type={owner_type}` | 头像二进制下载，内置指数退避重试 |

#### 2.2.3 流式消息批量下载

```
POST /api/mobile-sync/download-messages-stream
Body: { "requests": [{ "topicId": "...", "msgIds": ["..."] }] }
```

对应实现为 `PullExecutor::pull_messages_batch`（`pull_executor.rs`）。这是消息同步的核心路径，具备以下技术特征：

| 特征 | 实现细节 |
|------|----------|
| 传输格式 | **NDJSON**（Newline Delimited JSON，换行分隔 JSON），逐 Topic 分帧 |
| 流式消费 | 使用 `res.bytes_stream()` 逐 chunk 读取，不等待完整响应 |
| 并发处理 | `Semaphore(20)` 控制最大并发 Topic 处理数为 20 |
| 进度反馈 | `mpsc::unbounded_channel` 实时推送每 Topic 处理结果 |
| 错误隔离 | 单 Topic 的 `_error` 帧或解析失败不中断整体流 |

**NDJSON 解析流程**：

```
while chunk = stream.next():
    buffer.extend(chunk)
    while line = buffer.extract_line():
        topic_data = parse_json(line)
        if topic_data._error:
            send_error_result(topic_data.topicId)
            continue
        
        // 通过 Semaphore 获取许可后 spawn 异步任务
        permit = sem.acquire_owned().await
        spawn process_topic_messages(topic_data)
```

#### 2.2.4 消息处理管线

`process_topic_messages`（`pull_executor.rs`）是 PullExecutor 中最复杂的内部函数，承担从原始 JSON 到可持久化消息的完整转换。其管线分为 4 个阶段：

| 阶段 | 操作 | 优化策略 |
|------|------|----------|
| 1. 附件路径预查 | 批量收集所有消息的附件 hash，一次性 SQL 查询 `attachments` 表 | 替代 N+1 查询 |
| 2. 强类型反序列化与路径填充 | `TopicNDJSONFrame` 强类型反序列化替代手动 `Value` 处理；用 `path_map` 填充 `internalPath` 和 `src` | 编译期类型安全 |
| 3. 预渲染与 Zstd 压缩 | 对每个消息调用 `MessageRenderCompiler::compile` + `serialize`，并计算 `ContentCompressor::compress` Zstd 压缩文本 | `catch_unwind` 防 panic |
| 4. 提交写入队列 | 组装 `DbWriteTask::TopicMessages` 提交到 `DbWriteQueue` | `skip_bubble = true` 避免同步期间重复冒泡 |

**强类型反序列化（Type-Safe Deserialization）**（commit `5dbaa2b` 重构）：

此前使用 `serde_json::Value` 手动规范化桌面端字段类型（`normalize_desktop_message`），存在运行时类型不匹配导致解析失败的风险。重构后引入 `TopicNDJSONFrame` 强类型结构：

```rust
#[derive(serde::Deserialize)]
struct TopicNDJSONFrame {
    #[serde(rename = "topicId")]
    topic_id: String,
    messages: Vec<crate::vcp_modules::chat_manager::ChatMessage>,
    #[serde(rename = "_error")]
    error: Option<String>,
}
```

- `ChatMessage` 类型直接由 `serde` 反序列化驱动，字段类型在编译期保证一致。
- 彻底消除了 `normalize_desktop_message` 的运行时类型修复开销和遗漏风险。
- 解析失败计数从过去的 `failed_count > 0` 降为恒定的 `0`，所有 NDJSON 帧要么全量成功，要么在更早的 HTTP/JSON 层失败。

### 2.3 重试机制

PullExecutor 在 `pull_avatar` 中实现了完整的**指数退避（Exponential Backoff）**重试策略（`pull_executor.rs`）：

```
参数:
    max_retries = 3
    initial_delay_ms = 200
    backoff_multiplier = 2

逻辑:
    loop:
        发送 HTTP 请求
        if 成功:
            返回 Ok
        else if 是解码错误 AND retries < max_retries:
            retries += 1
            sleep(delay_ms)
            delay_ms *= 2
            continue
        else if 是网络错误 AND retries < max_retries:
            retries += 1
            sleep(delay_ms)
            delay_ms *= 2
            continue
        else:
            返回 Err
```

**注意**：当前实现未包含 **Jitter（抖动）** 机制，即退避延迟为纯指数序列 `200ms → 400ms → 800ms`。在极端并发场景下，多个重试请求可能形成"惊群效应"。建议未来引入随机抖动：`delay = delay_base * (1 + random_factor)`。

### 2.4 并发控制

`pull_messages_batch` 的并发控制由两层机制组成：

| 层级 | 机制 | 限流值 | 作用 |
|------|------|--------|------|
| 第一层 | `tokio::sync::Semaphore` | 20 | 限制同时处理 Topic 的并发任务数 |
| 第二层 | `DbWriteQueue` 内部串行化 | 1 | SQLite 写入天然串行，避免 WAL 锁竞争 |

**网络感知调节**：当前版本尚未实现基于网络类型（WiFi / 5G / 4G）的动态并发度调整。在 WiFi 环境下，Semaphore 的 20 许可通常不会成为瓶颈；但在弱网蜂窝环境下，过大的并发可能导致请求超时风暴。

### 2.5 批量操作的归口优化

PullExecutor 的归口优化体现在三个层面：

1. **实体层**：`pull_entities_batch` 将多个 Agent / Group / Topic 的下载请求聚合为单次 HTTP POST，避免多次 TLS 握手。
2. **消息层**：`pull_messages_batch` 将多个 Topic 的消息请求聚合为单次 NDJSON 流式响应，利用 HTTP/1.1 或 HTTP/2 的持久连接。
3. **附件路径层**：`process_topic_messages` 在解析前批量查询所有附件的本地路径，将 N 次 SQL 查询降为 1 次。

### 2.6 核心函数输入输出签名表

| 函数 | 输入参数 | 输出 | 所在文件 |
|------|----------|------|----------|
| `PullExecutor::pull_agent` | `app`, `client`, `http_url`, `sync_token`, `agent_id`, `write_queue` | `Result<(), String>` | `pull_executor.rs` |
| `PullExecutor::pull_group` | `app`, `client`, `http_url`, `sync_token`, `group_id`, `write_queue` | `Result<(), String>` | `pull_executor.rs` |
| `PullExecutor::pull_entities_batch` | `app`, `client`, `http_url`, `sync_token`, `requests: Vec<Value>`, `write_queue` | `Result<(), String>` | `pull_executor.rs` |
| `PullExecutor::pull_avatar` | `app`, `client`, `http_url`, `sync_token`, `owner_type`, `owner_id`, `write_queue` | `Result<(), String>` | `pull_executor.rs` |
| `PullExecutor::pull_agent_topic` | `app`, `client`, `http_url`, `sync_token`, `topic_id`, `write_queue` | `Result<(), String>` | `pull_executor.rs` |
| `PullExecutor::pull_group_topic` | `app`, `client`, `http_url`, `sync_token`, `topic_id`, `write_queue` | `Result<(), String>` | `pull_executor.rs` |
| `PullExecutor::pull_messages_batch` | `app`, `client`, `http_url`, `sync_token`, `requests: &[(String, Vec<String>)]`, `write_queue` | `Result<Vec<BatchPullResult>, String>` | `pull_executor.rs` |

## 3. PushExecutor

### 3.1 职责边界

`PushExecutor`（推送执行器）负责将移动端本地产生或修改的数据上传到桌面端。其处理范围涵盖：

- **实体**：Agent、Group 的配置变更
- **消息**：按 Topic 聚合的聊天记录
- **头像**：Agent / Group 的头像图片
- **附件**：消息关联的文件资源（按需去重上传）

与 PullExecutor 不同，PushExecutor **直接读取本地数据库**（通过 `agent_service`、`group_service`、`message_service`），而非接收外部传入的数据。

### 3.2 HTTP 调用链

#### 3.2.1 批量实体上传

```
POST /api/mobile-sync/upload-entities-batch
Body: { "items": [{ "id": "...", "type": "agent|group", "data": { ... } }] }
```

对应实现为 `PushExecutor::push_entities_batch`（`push_executor.rs`）。调用方需预先构建好 `items` 数组，每个元素包含实体的 `id`、`type` 和序列化后的 `data`。

#### 3.2.2 单实体上传（降级路径）

| 方法 | 端点 | 特殊头 |
|------|------|--------|
| `push_agent` | `POST /upload-entity` | `x-idempotency-key` |
| `push_group` | `POST /upload-entity` | `x-idempotency-key` |

**幂等性（Idempotency）保障**：单实体上传使用 `x-idempotency-key` 请求头，其值为基于 action、entity_type、id 和当前时间戳（分钟级）生成的 SHA-256 哈希。这意味着同一分钟内重复推送同一实体将被服务端视为同一次操作，防止网络超时后的客户端重试导致数据重复。

```
fn generate_idempotency_key(action, entity_type, id):
    hash = SHA256(action + entity_type + id + floor(now / 60))
    return hex(hash)
```

#### 3.2.3 头像上传

```
POST /api/mobile-sync/upload-avatar?id={owner_id}&type={owner_type}
Content-Type: {mime_type}
Body: <binary image_data>
```

对应实现为 `PushExecutor::push_avatar`（`push_executor.rs`）。直接从 `avatars` 表读取 `image_data` 和 `mime_type`，以原始二进制形式 POST。

#### 3.2.4 流式消息批量上传

```
POST /api/mobile-sync/upload-messages-batch
Content-Type: application/x-ndjson
Body: <NDJSON lines>
```

对应实现为 `PushExecutor::push_messages_batch`（`push_executor.rs`）。这是消息推送的核心路径，完整流程如下：

```
输入: topic_ids[], uploaded_hashes (Arc<RwLock<HashSet>>)

1. 批量查询 topics 表，获取每个 topic 的 owner_id + owner_type
2. 调用 load_multi_topic_messages() 一次 SQL 加载所有 topic 的消息
3. 对每个 topic:
     - 根据 owner_type 构建对应 DTO (UserMessageSyncDTO / AgentMessageSyncDTO / GroupMessageSyncDTO)
     - 查询 sender 的 avatar_color 填充到 DTO
     - 生成 NDJSON 行: { topicId, messages }
4. POST NDJSON 到 /upload-messages-batch
5. 解析 NDJSON 响应，收集 neededAttachmentHashes
6. 对 neededAttachmentHashes 去重，筛选出尚未上传的 hash
7. 以 chunks(3) 的并发度上传附件
8. 将上传成功的 hash 写入 uploaded_hashes 跟踪集
9. 返回 Vec<PushBatchResult>
```

**DTO 构建策略**：消息根据 `role` 和 `owner_type` 映射到不同的 DTO 类型：

| role | owner_type | DTO 类型 | 特殊处理 |
|------|-----------|----------|----------|
| `user` | any | `UserMessageSyncDTO` | 无 avatar_color |
| `assistant` / `system` | `group` | `GroupMessageSyncDTO` | 需查询 sender avatar_color |
| `assistant` / `system` | `agent` | `AgentMessageSyncDTO` | 需查询 sender avatar_color |

`avatar_color` 的查询兜底值为 `"#6B7280"`（中性灰色），确保无头像记录时仍能正确渲染。

#### 3.2.5 附件上传

```
POST /api/mobile-sync/upload-attachment?hash={}&type={mime_type}&name={display_name}
Content-Type: application/octet-stream
Body: <binary file_data>
```

对应实现为 `upload_attachment`（`push_executor.rs`）。从 `attachments` 表读取 `mime_type` 和 `internal_path`，从 `message_attachments` 表读取 `display_name`，以原始二进制上传。

### 3.3 与 DbWriteQueue 的交互

PushExecutor **不直接操作** `DbWriteQueue`，它的数据源是已经由前端或本地逻辑写入数据库的"稳定状态"。两者的交互关系是间接的：

```
前端操作 → DbWriteQueue 写入 SQLite → 同步服务检测到变更 → PushExecutor 读取并上传
```

这意味着推送的数据总是数据库中已提交的版本，不存在"读取未提交"的脏读问题。

### 3.4 批量操作的归口优化

PushExecutor 的归口优化体现在：

1. **实体层**：`push_entities_batch` 聚合多个实体为单次 HTTP POST。
2. **消息层**：`push_messages_batch` 将多个 Topic 的消息编码为 NDJSON，单次上传。
3. **附件层**：响应中的 `neededAttachmentHashes` 经去重后，以 3 并发分块上传，避免重复传输相同文件。

**附件去重机制**：`uploaded_hashes: Arc<RwLock<HashSet<String>>>` 是一个跨调用共享的状态跟踪器。在同一次同步周期内，同一附件 hash 不会被上传两次。这依赖于 `Arc` 的引用计数和 `RwLock` 的读写锁实现线程安全的共享状态。

### 3.5 核心函数输入输出签名表

| 函数 | 输入参数 | 输出 | 所在文件 |
|------|----------|------|----------|
| `PushExecutor::push_agent` | `app`, `client`, `http_url`, `sync_token`, `agent_id` | `Result<(), String>` | `push_executor.rs` |
| `PushExecutor::push_group` | `app`, `client`, `http_url`, `sync_token`, `group_id` | `Result<(), String>` | `push_executor.rs` |
| `PushExecutor::push_entities_batch` | `app`, `client`, `http_url`, `sync_token`, `items: Vec<Value>` | `Result<(), String>` | `push_executor.rs` |
| `PushExecutor::push_avatar` | `app`, `client`, `http_url`, `sync_token`, `owner_type`, `owner_id` | `Result<(), String>` | `push_executor.rs` |
| `PushExecutor::push_messages_batch` | `app`, `client`, `http_url`, `sync_token`, `topic_ids: &[String]`, `uploaded_hashes` | `Result<Vec<PushBatchResult>, String>` | `push_executor.rs` |
| `generate_idempotency_key` | `action`, `entity_type`, `id` | `String` | `push_executor.rs` |
| `build_message_dtos` | `app`, `history`, `owner_type` | `Vec<Value>` | `push_executor.rs` |
| `upload_attachment` | `app`, `client`, `http_url`, `sync_token`, `hash` | `Result<(), String>` | `push_executor.rs` |

## 4. DeleteExecutor

### 4.1 职责边界

`DeleteExecutor`（删除执行器）负责处理同步场景下的删除传播。它采用**软删除（Soft Delete）**为主、**物理删除（Physical Delete）**为辅的两级策略：

- **软删除**：将 `deleted_at` 字段设置为当前时间戳，数据仍保留在数据库中，可被恢复或用于冲突解决。
- **物理删除**：由 `cleanup_old_deleted_records` 定期清理超过保留期的软删除记录，释放存储空间。

DeleteExecutor 的操作全部在本地数据库完成，**不直接发起 HTTP 请求**。删除指令的上传由上层同步服务通过 PushExecutor 或独立协议完成。

### 4.2 级联删除逻辑

当前 DeleteExecutor 的实现**不自动级联**删除子资源。每个实体的软删除是独立操作，级联语义由调用方（上层同步服务或业务逻辑）显式控制。具体行为如下：

| 操作 | 目标表 | 是否级联子资源 | Hash 冒泡 |
|------|--------|---------------|-----------|
| `soft_delete_agent` | `agents` | 否（Topic/Message 由调用方处理） | `bubble_agent_hash` |
| `soft_delete_group` | `groups` | 否（Topic/Message 由调用方处理） | `bubble_group_hash` |
| `soft_delete_topic` | `topics` | 否（Message 由调用方处理） | 根据 `owner_type` 冒泡 |
| `soft_delete_avatar` | `avatars` | 否 | 否 |

**Hash 冒泡（Hash Bubbling）**：软删除后，通过 `HashAggregator` 重新计算受影响父实体的同步哈希值。这确保桌面端在下一轮同步时能够检测到变更：

```
soft_delete_agent(agent_id):
    UPDATE agents SET deleted_at = now WHERE agent_id = ?
    BEGIN TRANSACTION
    HashAggregator::bubble_agent_hash(tx, agent_id)
    COMMIT
```

对于 Topic，需要先查询其 `owner_id` 和 `owner_type`，再决定调用 `bubble_agent_hash` 还是 `bubble_group_hash`。

### 4.3 软删除与物理删除的边界

#### 4.3.1 软删除的触发时机

软删除通常在以下场景触发：

1. **用户主动删除**：用户在移动端删除 Agent / Group / Topic，本地先执行软删除，随后通过同步协议将删除标记传播到桌面端。
2. **同步冲突解决**：桌面端指示某实体已被删除，移动端执行对应的软删除。
3. **数据一致性修复**：发现孤儿记录（如 owner 已不存在的 Topic）时，后台执行软删除。

#### 4.3.2 物理删除的触发时机

```
DELETE FROM {table}
WHERE deleted_at IS NOT NULL
  AND deleted_at < (now - days * 24 * 60 * 60 * 1000)
```

对应实现为 `DeleteExecutor::cleanup_old_deleted_records`（`delete_executor.rs`）。参数 `days` 由调用方决定（建议值 30 天）。该操作会清理以下四张表：

| 表名 | 清理条件 | 返回值 |
|------|----------|--------|
| `agents` | `deleted_at IS NOT NULL AND deleted_at < threshold` | `rows_affected()` |
| `groups` | 同上 | `rows_affected()` |
| `topics` | 同上 | `rows_affected()` |
| `messages` | 同上 | `rows_affected()` |

**注意**：物理删除**不级联**。如果一张表中的记录被物理删除，而其关联的附件或头像记录未被清理，可能产生存储泄漏。建议未来增加对 `attachments` 和 `avatars` 表的孤儿清理。

### 4.4 核心函数输入输出签名表

| 函数 | 输入参数 | 输出 | 所在文件 |
|------|----------|------|----------|
| `DeleteExecutor::soft_delete_agent` | `app`, `agent_id` | `Result<(), String>` | `delete_executor.rs` |
| `DeleteExecutor::soft_delete_group` | `app`, `group_id` | `Result<(), String>` | `delete_executor.rs` |
| `DeleteExecutor::soft_delete_topic` | `app`, `topic_id` | `Result<(), String>` | `delete_executor.rs` |
| `DeleteExecutor::soft_delete_avatar` | `app`, `owner_type`, `owner_id` | `Result<(), String>` | `delete_executor.rs` |
| `DeleteExecutor::cleanup_old_deleted_records` | `app`, `days: i64` | `Result<(), String>` | `delete_executor.rs` |

## 5. Executor 间的协作

### 5.1 Pull 与 Push 的交替执行

在典型的同步周期中，Pull 和 Push 按以下顺序交替执行：

```
sync_cycle():
    // 阶段 A：Pull（先获取桌面端最新状态）
    // 目的：避免基于过时的本地状态进行推送
    
    A1. 通过 sync_pipeline 计算需要拉取的实体清单
    A2. PullExecutor::pull_entities_batch(实体清单)
    A3. PullExecutor::pull_messages_batch(消息清单)
    A4. 等待 DbWriteQueue 排空（本地数据已更新）
    
    // 阶段 B：Push（再推送本地变更）
    // 目的：将 A 阶段之后仍存在的本地新变更上传到桌面端
    
    B1. 扫描本地 dirty 标记，构建推送清单
    B2. PushExecutor::push_entities_batch(本地变更实体)
    B3. PushExecutor::push_messages_batch(本地变更消息)
    B4. 上传缺失的附件
```

**为什么先 Pull 后 Push？**

1. **冲突避免**：如果桌面端和移动端同时修改了同一实体，先 Pull 可以让移动端获得最新版本，从而基于最新状态决定是否需要覆盖或合并。
2. **删除传播**：如果桌面端已删除某实体，先 Pull 可以确保移动端在 Push 前感知到该删除，避免将已删除实体的子资源重新上传。
3. **一致性**：Push 的数据总是经过 Pull 更新后的"最终本地状态"，减少三方不一致的概率。

### 5.2 Delete 的触发时机

DeleteExecutor 的触发独立于 Pull/Push 的主循环，通常在以下节点执行：

| 触发场景 | 执行的操作 | 调用方 |
|----------|-----------|--------|
| 用户在 UI 删除 Agent | `soft_delete_agent` + `soft_delete_avatar` | Agent 管理模块 |
| 用户在 UI 删除 Group | `soft_delete_group` + `soft_delete_avatar` | Group 管理模块 |
| 用户在 UI 删除 Topic | `soft_delete_topic` | Topic 管理模块 |
| 同步协议收到删除指令 | 对应的 `soft_delete_*` | 同步服务 |
| 应用启动 / 定时任务 | `cleanup_old_deleted_records(30)` | 生命周期管理器 |

**删除传播到桌面端的流程**：

```
移动端 soft_delete_agent(agent_id)
    → HashAggregator::bubble_agent_hash() 更新 agent_hash
    → 下一轮 sync: sync_pipeline 检测到 agent_hash  mismatch
    → PushExecutor 将带有 deleted_at 的 AgentSyncDTO 推送到桌面端
    → 桌面端解析 DTO，执行对应的本地删除
```

### 5.3 批量操作的协调

三个执行器的批量操作在数据层面通过 `DbWriteQueue` 间接协调：

| 执行器 | 批量操作 | DbWriteQueue 交互 | 并发模型 |
|--------|----------|-------------------|----------|
| PullExecutor | `pull_entities_batch` | 提交 `DbWriteTask::AgentBatch`, `GroupTopicBatch` 等 | 并行解析，串行写入 |
| PullExecutor | `pull_messages_batch` | 提交 `DbWriteTask::TopicMessages` | Semaphore(20) 限制解析并发 |
| PushExecutor | `push_entities_batch` | 无直接交互（读取已提交数据） | 单次 HTTP 请求 |
| PushExecutor | `push_messages_batch` | 无直接交互 | 3 并发附件上传 |
| DeleteExecutor | 各 `soft_delete_*` | 直接执行 SQL（事务内） | 串行（SQLite 特性） |

## 6. 错误处理策略

### 6.1 单个操作失败时的回退

#### 6.1.1 Pull 侧回退

| 失败场景 | 回退行为 | 影响范围 |
|----------|----------|----------|
| `pull_agent` / `pull_group` HTTP 失败 | 返回 `Err(String)`，由调用方决定是否重试 | 单个实体 |
| `pull_avatar` 网络/解码失败 | 指数退避重试 3 次后返回 `Err` | 单个头像 |
| `pull_messages_batch` 流级错误 | 立即返回 `Err`，整个批次失败 | 整个消息批次 |
| `pull_messages_batch` 单 Topic 错误 | `_error` 帧标记为失败，其他 Topic 继续处理 | 仅该 Topic |
| `process_topic_messages` 解析失败 | `TopicNDJSONFrame` 强类型反序列化消除此类失败；若发生则整个 Topic 失败 | 整个 Topic |

**关键设计**：`pull_messages_batch` 中的流级错误（如 `"_stream_error"` 帧）会立即中断整个批次，而单 Topic 的 `_error` 帧仅影响该 Topic，通过 `mpsc channel` 将错误结果异步发送给接收协程，不阻塞 NDJSON 流的继续消费。

#### 6.1.2 Push 侧回退

| 失败场景 | 回退行为 | 影响范围 |
|----------|----------|----------|
| `push_entities_batch` HTTP 失败 | 返回 `Err(String)` | 整个实体批次 |
| `push_messages_batch` HTTP 失败 | 返回 `Err(String)` | 整个消息批次 |
| 单 Topic 上传失败（响应中 `success=false`） | 记录到 `PushBatchResult`，不影响其他 Topic | 单个 Topic |
| 附件上传失败 | 该 hash 不加入 `uploaded_hashes` 跟踪集，下次同步会重试 | 单个附件 |

**幂等性兜底**：`push_agent` 和 `push_group` 使用 `x-idempotency-key` 头，即使客户端因超时重试，服务端也能识别并去重。批量接口（`push_entities_batch`、`push_messages_batch`）当前未显式携带幂等键，依赖服务端自身的去重逻辑。

### 6.2 批量操作中部分失败的隔离

**Pull 侧**：`pull_messages_batch` 返回 `Vec<BatchPullResult>`，每个元素包含 `topic_id`、`success`、`parsed_count`、`failed_count`（重构后恒为 `0`）和 `error`。调用方可根据此结果精确识别哪些 Topic 需要重试。

**Push 侧**：`push_messages_batch` 返回 `Vec<PushBatchResult>`，每个元素包含 `topic_id`、`success` 和 `error`。附件上传的失败通过 `uploaded_hashes` 跟踪集隔离，确保仅失败附件会被重试。

```
// 部分失败隔离的调用范式
let results = PullExecutor::pull_messages_batch(...).await?;
for r in results:
    if !r.success:
        failed_topics.push(r.topic_id)
        log_error(r.error)
    else:
        success_topics.push(r.topic_id)

// failed_topics 可在下一轮同步中单独重试
```

### 6.3 网络中断时的状态保留

同步执行器在网络中断场景下遵循以下状态保留原则：

| 层级 | 状态保留机制 | 说明 |
|------|-------------|------|
| HTTP 请求层 | `reqwest::Client` 内部连接池 | 短连接，无长连接状态 |
| 数据解析层 | 无中间状态 | `pull_messages_batch` 的 NDJSON 解析是流式的，网络中断会导致整个批次失败，已处理的部分已提交到 `DbWriteQueue` |
| 数据库层 | SQLite 事务 + WAL | `DeleteExecutor` 的每个操作都在显式事务内；`PullExecutor` 通过 `DbWriteQueue` 串行写入，天然原子 |
| 附件上传层 | `uploaded_hashes` 跟踪集 | 已上传的 hash 被持久化到内存中的 `HashSet`，应用进程存活期间不会重复上传 |

**重要约束**：当前执行器的设计**不保证跨进程重启的断点续传**。如果应用在网络中断后被系统回收或用户手动关闭，下次启动时将从头开始新的同步周期。这是由移动端的资源受限环境决定的——维护复杂的断点状态机会增加内存和存储开销。

## 7. 设计决策与权衡

### 7.1 为什么使用 DbWriteQueue 而非直接写入

`PullExecutor` 和 `process_topic_messages` 选择将数据提交到 `DbWriteQueue` 而不是直接执行 `INSERT`/`UPDATE`，这一决策基于以下考量：

| 方案 | 优点 | 缺点 | 本模块选择 |
|------|------|------|-----------|
| 直接写入 | 简单、时序直观 | SQLite 并发写入会触发 WAL 锁竞争，导致 `database is locked` | ❌ |
| `DbWriteQueue` 串行写入 | 消除锁竞争、统一事务边界、便于日志追踪 | 引入异步延迟，数据非即时落盘 | ✅ |
| 批量事务写入 | 性能最优 | 实现复杂，失败回退成本高 | 部分采用（Queue 内部） |

`DbWriteQueue` 本质上是一个单消费者（Single-Consumer）的异步通道，所有写入任务按提交顺序执行。这种"漏斗"设计牺牲了极低的写入延迟，换取了移动端恶劣环境下的稳定性。

### 7.2 流式 NDJSON vs 传统 JSON 数组

`pull_messages_batch` 和 `push_messages_batch` 均采用 NDJSON 格式，而非传统的 JSON 数组 `[]`。这一选择的权衡如下：

| 维度 | NDJSON | JSON Array |
|------|--------|------------|
| 内存占用 | 低（逐行解析，无需缓冲整个响应） | 高（需完整加载到内存） |
| 解析启动延迟 | 零（收到第一行即可开始处理） | 需等待 `]` 闭合 |
| 错误隔离 | 单行损坏不影响其他行 | 整体解析失败 |
| 服务端实现 | 需支持流式序列化 | 简单（一次性 `json.dumps`） |
| 带宽效率 | 略低（每行重复字段名） | 略高（可共享结构） |

对于移动端而言，**内存占用**和**启动延迟**是决定性因素。一条 Topic 可能包含数千条消息，传统 JSON 数组很容易导致 OOM（Out of Memory，内存溢出）。

### 7.3 为什么 Push 使用 `let _ = client.post(...)` 忽略响应

观察 `push_agent`、`push_group`、`push_avatar` 的实现，会发现它们使用 `let _ = client.post(...).await` 主动丢弃响应结果，仅在 `push_entities_batch` 和 `push_messages_batch` 中检查 HTTP 状态码。这一"宽松"策略的考量是：

1. **乐观同步**：移动端的本地数据被视为"权威"，服务端返回的错误（如版本冲突）在移动端缺乏有效的自动解决策略，不如留待下一轮同步周期处理。
2. **减少阻塞**：严格检查每个响应会显著增加同步周期的总耗时，尤其在网络高延迟场景下。
3. **幂等兜底**：`x-idempotency-key` 的存在使得即使请求被重复提交，也不会造成数据损坏。

但这也带来风险：服务端静默拒绝（如权限不足）不会被及时发现。`push_entities_batch` 和 `push_messages_batch` 之所以检查状态码，是因为批量操作的失败成本更高（一次失败意味着多个实体/Topic 未同步）。

### 7.4 软删除的存储成本权衡

`DeleteExecutor` 采用软删除而非即时物理删除，是为了支持以下场景：

- **误删恢复**：用户删除后短期内可撤销。
- **冲突检测**：桌面端和移动端同时操作同一实体时，软删除标记可作为冲突信号。
- **审计追踪**：保留历史记录的删除时间戳。

代价是存储空间的持续增长。`cleanup_old_deleted_records` 作为补偿机制，以 30 天为周期清理过期软删除记录，在可恢复性和存储效率之间取得平衡。

## 8. 性能特征与优化建议

### 8.1 当前性能特征

基于代码分析，各执行器的理论性能特征如下：

| 操作 | 时间复杂度 | 网络往返数（RTT） | 主要瓶颈 |
|------|-----------|------------------|----------|
| `pull_entities_batch` | O(n) n=实体数 | 1 | 服务端序列化 + 网络带宽 |
| `pull_messages_batch` | O(m) m=消息总数 | 1 | NDJSON 解析 + 预渲染 CPU |
| `pull_avatar` | O(1) | 1~4（含重试） | 网络稳定性 |
| `push_entities_batch` | O(n) | 1 | 本地数据库读取 |
| `push_messages_batch` | O(m) | 1 + k（附件上传） | 附件文件 IO + 网络带宽 |
| `upload_attachment` | O(1) | 1 | 文件读取 + 网络带宽 |

### 8.2 已知瓶颈与优化方向

1. **预渲染的 CPU 开销**：`process_topic_messages` 中对每条消息调用 `MessageRenderCompiler::compile`，这是纯 CPU 密集型操作。当前通过 `catch_unwind` 防 panic，但未利用多线程并行。由于 `tokio::spawn` 的任务仍在异步运行时中执行，CPU 密集型任务会阻塞同线程的其他 IO 操作。建议考虑使用 `tokio::task::spawn_blocking` 将预渲染 offload 到独立线程池。

2. **附件查询的 N+1 问题已解决**：`process_topic_messages` 中的批量附件路径查询已有效消除了 N+1 问题。但 `push_messages_batch` 中的 `build_message_dtos` 对每条消息调用 `query_avatar_color`，虽然 `agent_id` 去重后实际查询数有限，但在极端场景下（数千条消息来自不同 Agent）仍可能产生大量查询。建议增加内存缓存层。

3. **NDJSON 的内存缓冲**：`pull_messages_batch` 使用 `Vec<u8>` 作为行缓冲。在服务端返回超大 chunk（如 10MB）时，频繁的 `extend_from_slice` 和 `drain` 可能导致内存重新分配。建议预分配合理容量的 buffer（如 64KB）。

## 9. 安全考量

### 9.1 认证与授权

所有执行器的 HTTP 请求均携带两个认证头：

| 请求头 | 用途 | 来源 |
|--------|------|------|
| `x-sync-token` | 同步服务专用令牌 | 同步握手阶段生成 |
| `Authorization: Bearer {token}` | 标准 JWT（JSON Web Token）认证 | 用户登录会话 |

双头策略提供了冗余：即使其中一个被截获，攻击者仍需伪造另一个才能通过服务端校验。

### 9.2 路径遍历防护

`upload_attachment` 从数据库读取 `internal_path` 后直接传递给 `tokio::fs::read`。由于路径来源于受控的数据库记录（而非用户输入），且数据库写入受 `file_manager.rs` 中的 `ensure_safe_path()` 保护，因此不存在路径遍历漏洞。但需注意：如果数据库已被恶意篡改（如设备被 root），`internal_path` 可能指向任意文件。建议在读取前增加路径前缀校验。

### 9.3 内存安全

`process_topic_messages` 中使用了 `std::panic::catch_unwind` 包裹 `MessageRenderCompiler::compile`。这是 Rust 中处理不可信输入的 C-FFI（Foreign Function Interface，外部函数接口）边界时的标准做法。如果渲染编译器遇到恶意构造的内容字符串而 panic，`catch_unwind` 可防止整个同步任务崩溃，将错误隔离到单条消息。

## 10. 监控与可观测性

三个执行器均内置了 `println!` / `eprintln!` 日志输出，用于开发阶段的问题追踪。关键日志模式如下：

| 执行器 | 日志前缀 | 关键事件 |
|--------|----------|----------|
| PullExecutor | `[PullExecutor]` | 批量完成、Topic 处理进度、解析失败诊断、重试次数 |
| PushExecutor | `[PushExecutor]` | 批量完成、NDJSON 解析错误、附件上传成功 |

**注意**：当前日志直接输出到 stdout/stderr，未接入结构化日志框架（如 `tracing` 或 `slog`）。在生产环境中，建议：

1. 将 `println!` 替换为分级日志（`info!`、`warn!`、`error!`）。
2. 增加 Prometheus 风格的指标计数器（成功/失败/重试次数）。
3. 对 `pull_messages_batch` 的每个 Topic 增加耗时直方图，用于发现慢 Topic。

以下伪时序图展示了一个完整的同步周期中三个执行器的协作：


## 附录 D：错误码与处理矩阵

| 错误场景 | HTTP 状态码 | 执行器行为 | 调用方应对 |
|----------|------------|-----------|-----------|
| 实体不存在 | 404 | `pull_agent_topic` 静默跳过；其他返回 `Err` | 记录日志，继续下一项 |
| 认证失效 | 401 | 返回 `Err` | 触发重新登录或同步令牌刷新 |
| 服务端流错误 | 200 + `_stream_error` 帧 | 立即返回 `Err` | 标记批次失败，延迟重试 |

| 消息解析失败 | N/A（本地错误） | `TopicNDJSONFrame` 强类型反序列化后不再发生；HTTP/JSON 层错误在更早阶段返回 `Err` | 分析日志，排查网络或序列化问题 |
| 附件读取失败 | N/A（本地 IO 错误） | 返回 `Err` | 检查文件系统完整性 |
| 网络超时 | N/A | `reqwest` 内部超时后返回 `Err` | 指数退避重试 |
| 数据库锁超时 | N/A | `map_err` 后返回 `Err` | 减小并发度，延长超时 |

## 11. 常见问题排查指南

### 11.1 PullExecutor 相关

#### 现象：消息解析失败率高（历史问题，已修复）

> **状态**：`normalize_desktop_message` 手动类型修复逻辑已在 commit `5dbaa2b` 中删除，`TopicNDJSONFrame` 强类型反序列化彻底消除了消息级解析失败。

**诊断步骤**（若仍观察到类似症状，通常源于更上层的错误）：

1. 检查日志中是否有 `[PullExecutor] Failed to parse NDJSON frame` 或网络层错误。
2. 确认 HTTP 响应的 Content-Type 是否为 `application/x-ndjson`。
3. 对比桌面端返回的 NDJSON 帧与 `TopicNDJSONFrame` 结构体，定位字段不匹配。

**常见根因**：

| 根因 | 症状 | 修复方式 |
|------|------|----------|
| 桌面端新增必填字段 | `TopicNDJSONFrame` 反序列化报错 "missing field" | 在 `ChatMessage` 中添加 `#[serde(default)]` |
| HTTP 响应被代理篡改 | NDJSON 帧格式损坏 | 检查中间层（CDN、网关）是否修改响应体 |
| 网络中断导致流截断 | 最后一条 NDJSON 帧不完整 | 属于流级错误，由 `pull_messages_batch` 统一处理 |

#### 现象：`pull_messages_batch` 返回空结果但服务端有数据

**诊断步骤**：

1. 检查 `requests` 参数中的 `msgIds` 是否为空数组。空数组表示"拉取全部消息"，但服务端可能因权限过滤返回空。
2. 检查 NDJSON 流的换行符格式。服务端必须严格使用 `\n` 而非 `\r\n`。
3. 检查 `buffer` 中是否存在残留的未闭合 JSON（流被提前中断）。

### 11.2 PushExecutor 相关

#### 现象：桌面端未收到移动端发送的消息

**诊断步骤**：

1. 检查 `push_messages_batch` 的日志，确认 HTTP 状态码是否为 200。
2. 检查响应中对应 Topic 的 `success` 字段是否为 `true`。
3. 如果 `success = false`，检查 `error` 字段中的服务端错误信息。
4. 如果 `success = true` 但桌面端仍无消息，检查 `neededAttachmentHashes` 是否非空。若附件上传失败，桌面端可能因缺少附件而拒绝入库。

#### 现象：附件重复上传

**诊断步骤**：

1. 检查 `uploaded_hashes` 跟踪集是否在同步周期内被正确共享。若每次调用 `push_messages_batch` 都创建新的 `HashSet`，则去重失效。
2. 检查 `Arc<RwLock<HashSet<String>>>` 的克隆方式。应使用 `Arc::clone(&uploaded_hashes)` 共享同一实例，而非 `uploaded_hashes.clone()`（后者克隆的是 `RwLock` 本身，内部数据仍共享，但语义上易混淆）。

### 11.3 DeleteExecutor 相关

#### 现象：已删除的 Agent 在同步后重新出现

**诊断步骤**：

1. 确认 `soft_delete_agent` 是否被正确调用。仅删除 UI 而未调用执行器会导致本地数据库未标记删除。
2. 检查 `HashAggregator::bubble_agent_hash` 是否在事务内成功执行。若事务回滚，桌面端不会感知到删除。
3. 检查桌面端是否正确处理了带有 `deleted_at` 字段的 `AgentSyncDTO`。

#### 现象：数据库体积持续增长

**诊断步骤**：

1. 查询各表的 `deleted_at IS NOT NULL` 记录数：
   ```sql
   SELECT 'agents' as table_name, COUNT(*) as deleted_count FROM agents WHERE deleted_at IS NOT NULL
   UNION ALL
   SELECT 'groups', COUNT(*) FROM groups WHERE deleted_at IS NOT NULL
   UNION ALL
   SELECT 'topics', COUNT(*) FROM topics WHERE deleted_at IS NOT NULL
   UNION ALL
   SELECT 'messages', COUNT(*) FROM messages WHERE deleted_at IS NOT NULL;
   ```
2. 若数值异常高，检查 `cleanup_old_deleted_records` 的调度频率。当前版本未内置自动调度，需由生命周期管理器定期触发。

## 附录 E：`TopicNDJSONFrame` 强类型结构

（替换已删除的 `normalize_desktop_message` 速查表）

```rust
#[derive(serde::Deserialize)]
struct TopicNDJSONFrame {
    #[serde(rename = "topicId")]
    topic_id: String,
    messages: Vec<crate::vcp_modules::chat_manager::ChatMessage>,
    #[serde(rename = "_error")]
    error: Option<String>,
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `topic_id` | `String` | Topic 唯一标识符，对应请求中的 `topicId` |
| `messages` | `Vec<ChatMessage>` | 该 Topic 下的消息列表，由 `serde` 直接反序列化为强类型结构 |
| `error` | `Option<String>` | 若服务端处理该 Topic 出错，此字段携带错误描述；成功时为 `None` |

**优势**：
- 编译期类型安全：`ChatMessage` 的所有字段类型在编译时确定，无需运行时手动转换。
- 零解析失败：强类型反序列化要么成功（全部消息可用），要么在 `serde` 层失败（整个 Topic 标记为失败），不会出现部分消息解析成功、部分失败的中间态。
- `avatarUrl` 和 `avatarColor` 等桌面端专属字段已不在 `ChatMessage` 中定义，自然被 `serde` 忽略，无需手动 `remove`。

## 附录 F：并发安全分析

### F.1 `uploaded_hashes` 的线程安全

`PushExecutor::push_messages_batch` 接收 `Arc<RwLock<HashSet<String>>>` 作为参数。其并发安全分析如下：

| 操作 | 锁类型 | 并发数 | 安全分析 |
|------|--------|--------|----------|
| 读取 `uploaded_hashes` | `read().await` | 多并发 | 多个 `read` 锁可同时持有，无阻塞 |
| 写入 `uploaded_hashes` | `write().await` | 单并发 | 与所有 `read` 互斥，但仅在上传成功后短暂持有 |
| `HashSet::insert` | 无（在 `write` 锁内） | N/A | `HashSet` 非并发安全，但锁保护确保了串行访问 |

**潜在风险**：如果附件上传的并发度（当前为 3）未来提高到很高（如 100），`RwLock` 的写锁饥饿可能导致读取端长期等待。但在当前参数下，此风险可忽略。

### F.2 `DbWriteQueue` 的线程安全

`DbWriteQueue` 通过 `tokio::sync::mpsc` 通道实现单消费者模型：

```
[多个生产者: PullExecutor / 前端逻辑]
              |
              v
       mpsc::channel / unbounded_channel
              |
              v
    [单消费者: DbWriteQueue 处理循环]
              |
              v
         SQLite WAL 写入
```

由于 SQLite 在同一时刻仅支持一个写入事务，`DbWriteQueue` 的单消费者设计天然符合 SQLite 的并发模型，无需额外的锁机制。

## 附录 H：`BatchPullResult` 与 `PushBatchResult` 结构详解

这两个结果是批量操作的部分失败隔离机制的核心数据结构。

### H.1 `BatchPullResult`

```rust
pub struct BatchPullResult {
    pub topic_id: String,    // Topic 唯一标识符
    pub success: bool,       // 该 Topic 是否整体处理成功
    pub parsed_count: usize, // 成功解析并提交的消息数
    pub failed_count: usize, // 解析失败的消息数（`TopicNDJSONFrame` 强类型反序列化后恒为 0）
    pub error: Option<String>, // 若 success=false，此处存放错误描述
}
```

**使用场景**：调用方遍历返回的 `Vec<BatchPullResult>`，筛选出 `success=false` 的 Topic，将其加入下一轮同步的重试清单。

### H.2 `PushBatchResult`

```rust
pub struct PushBatchResult {
    pub topic_id: String,    // Topic 唯一标识符
    pub success: bool,       // 该 Topic 是否上传成功
    pub error: Option<String>, // 若 success=false，此处存放服务端错误
}
```

**与 `BatchPullResult` 的差异**：Push 侧不区分"解析成功/失败"，因为消息在推送前已在本地验证过格式。失败仅来源于服务端拒绝（如权限、数据校验）。

## 附录 I：`DbWriteTask` 枚举与执行器的映射

`PullExecutor` 向 `DbWriteQueue` 提交的任务类型：

| 任务变体 | 提交来源 | 说明 |
|----------|----------|------|
| `DbWriteTask::Agent { id, dto }` | `pull_agent`, `pull_entities_batch` | 单 Agent 写入 |
| `DbWriteTask::Group { id, dto }` | `pull_group`, `pull_entities_batch` | 单 Group 写入 |
| `DbWriteTask::AgentTopic { topic_id, dto }` | `pull_agent_topic` | 单 Agent Topic 写入 |
| `DbWriteTask::GroupTopic { topic_id, dto }` | `pull_group_topic` | 单 Group Topic 写入 |
| `DbWriteTask::AgentTopicBatch { topics }` | `pull_entities_batch` | 批量 Agent Topic 写入（归口优化） |
| `DbWriteTask::GroupTopicBatch { topics }` | `pull_entities_batch` | 批量 Group Topic 写入（归口优化） |
| `DbWriteTask::TopicMessages { topic_id, messages, compressed_contents, render_bytes, content_hashes, skip_bubble }` | `process_topic_messages` | Topic 消息批量写入（含 Zstd 压缩内容） |
| `DbWriteTask::Avatar { owner_type, owner_id, bytes }` | `pull_avatar` | 头像二进制写入 |

`skip_bubble = true` 的含义：在同步期间，消息的大量写入会触发频繁的哈希冒泡计算，严重影响性能。通过设置 `skip_bubble`，`DbWriteQueue` 在批量写入消息时跳过冒泡，待同步结束后再统一计算一次父级哈希。
