# 桌面端插件架构

## 1. 引言

**VCPMobileSync**（VCP 移动端同步服务插件）是 VCPChat 桌面端与 VCPMobile 手机端之间的数据同步桥梁。该插件以 **Service Plugin**（服务插件）的形式挂载于 VCPDistributedServer（VCP 分布式服务器）之上，通过 HTTP REST API 与 WebSocket 双通道，为手机端提供双向、增量、安全的数据同步能力。

桌面端在同步架构中扮演**服务端（Server）**角色：它负责维护本地 `AppData` 目录下全部实体（Agent、Group、Topic）与消息历史的索引数据库，接收手机端发起的同步请求，并在文件变更时主动更新索引。手机端则作为**客户端（Client）**，通过三阶段增量同步协议（`owner_metadata` → `topic_metadata` → `messages`）拉取差异、推送本地变更。

本文档完整描述桌面端插件的内部架构、数据模型、安全策略与已知限制，是理解 VCP 跨端同步机制的关键参考。

## 2. 插件元数据

### 2.1 插件清单（plugin-manifest.json）

VCPDistributedServer 通过读取 `plugin-manifest.json` 识别并加载插件。该文件定义了插件的身份、入口与配置契约。

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | `string` | 插件唯一标识符，值为 `VCPMobileSync` |
| `displayName` | `string` | 人类可读名称：`VCP Mobile Sync Service` |
| `version` | `string` | 插件版本号，当前为 `0.9.13`，与手机端版本对齐 |
| `description` | `string` | 功能描述：为 VCPMobile 提供安全的双向同步 API |
| `author` | `string` | 作者：`Nova` |
| `pluginType` | `string` | 插件类型：`service`，表示无 UI 的后台服务插件 |
| `entryPoint.script` | `string` | 入口脚本：`index.js` |
| `communication.protocol` | `string` | 通信协议：`direct`，表示与宿主进程直接交互 |
| `configSchema.MobileSyncToken` | `string` | 用户配置的同步令牌类型，用于手机端连接鉴权 |
| `configSchema.MobileSyncPort` | `integer` | WebSocket 监听端口，默认 `5975` |

上述字段定义于 `plugin-manifest.json`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/plugin-manifest.json`）。

### 2.2 与 VCPDistributedServer 的集成方式

VCPDistributedServer 在启动时会扫描 `Plugin/` 目录，读取每个子目录中的 `plugin-manifest.json`。当检测到 `pluginType` 为 `service` 时，宿主进程会：

1. 解析 `configSchema`，从用户配置（`config.env`）中读取 `MobileSyncToken` 与 `MobileSyncPort`；
2. 调用插件入口 `index.js` 导出的 `registerRoutes(app, pluginConfig, projectBasePath)` 函数；
3. 将 Express 应用实例 `app`、插件配置 `pluginConfig` 与项目基础路径 `projectBasePath` 注入插件。

插件内部通过 `path.resolve(projectBasePath, "..", "AppData")` 推导实际的 `AppData` 目录（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

## 3. 插件生命周期

### 3.1 启动阶段总览

`registerRoutes` 是插件的唯一外部入口，由 VCPDistributedServer 在初始化阶段调用。整个生命周期严格按以下顺序执行：

| 阶段 | 函数/操作 | 说明 |
|------|-----------|------|
| ① 参数提取 | `registerRoutes(app, pluginConfig, projectBasePath)` | 提取 `MobileSyncToken`、`MobileSyncPort`，推导 `appDataPath` |
| ② 日志初始化 | `resetLogger()` | 创建新的 `SyncLogger` 会话，前缀为 `system` |
| ③ 数据库初始化 | `initDb(dbPath)` | 在插件目录创建/打开 `sync_state.db`，建立 5 张表 |
| ④ 索引扫描 | `reconcileLocalFiles(appDataPath)` | **阻塞式**全量扫描本地文件，建立初始索引 |
| ⑤ WebSocket 启动 | `startWsServer({ port, syncToken, onMessage })` | 在 `0.0.0.0:wsPort` 启动 WebSocket 服务 |
| ⑥ HTTP 路由注册 | `registerHttpRoutes(app, { syncToken, appDataPath })` | 挂载 `/api/mobile-sync/*` REST 路由 |
| ⑦ 文件监听启动 | `startFileWatcher(appDataPath)` | 使用 `chokidar` 监听文件变更事件 |
| ⑧ 定时清理 | `setInterval(cleanupOldDeletedRecords, 60min)` | 每小时清理 30 天前的软删除记录 |

各阶段调用关系与代码位置位于 `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`。

### 3.3 关键设计：先扫描后开放端口

第 ④ 步 `reconcileLocalFiles` 是**同步阻塞调用**，必须在 WebSocket 与 HTTP 服务启动之前完成。这一设计的目的在于：

- **防止手机端提前连接**：若手机端在索引尚未就绪时发起同步请求，服务端无法提供完整的 `manifest`（清单），会导致差异计算错误甚至数据丢失；
- **保证一致性基线**：只有在本地全部实体、话题、消息、附件的哈希指纹都已入库后，服务端才能准确响应手机端的 `SYNC_MANIFEST` 与 `SYNC_TOPIC_HASH_BATCH` 请求。

此设计决策体现在 `index.js` 中 `startWsServer` 与 `registerHttpRoutes` 被明确置于 `reconcileLocalFiles` 的 `await` 之后（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

## 4. SQLite 索引数据库

插件使用 `better-sqlite3` 驱动，在插件自身目录下维护一个名为 `sync_state.db` 的 SQLite 数据库。该数据库**不存储实际业务数据**（如消息正文、配置内容），仅存储用于快速差异比对的**索引信息**（ID、哈希值、文件路径、时间戳）。

### 4.1 数据库初始化

`initDb(dbPath)` 在启动时被调用，自动创建 5 张表并建立必要的二级索引（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/db.js`）。

### 4.2 五张核心表的完整 Schema

#### 表 1：entity_index（实体索引）

存储 Agent、Group、Topic 的元数据与哈希指纹。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `TEXT` | `NOT NULL, PK(1)` | 实体唯一标识 |
| `type` | `TEXT` | `NOT NULL, PK(2)` | 实体类型：`agent`、`group`、`topic` |
| `file_path` | `TEXT` | `NOT NULL` | 所属 `config.json` 的绝对路径 |
| `hash` | `TEXT` | `NOT NULL` | 当前配置 DTO 的 SHA-256 哈希 |
| `aggregated_hash` | `TEXT` | `NULLABLE` | 层级聚合哈希（子话题哈希的二次聚合） |
| `updated_at` | `INTEGER` | `NOT NULL` | 最后更新时间戳（毫秒） |
| `deleted_at` | `INTEGER` | `DEFAULT NULL` | 软删除时间戳，NULL 表示未删除 |
| **主键** | — | `PRIMARY KEY (id, type)` | 复合主键，同一 ID 在不同类型下可共存 |

#### 表 2：message_index（消息索引）

存储每条消息的哈希与归属关系。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `msg_id` | `TEXT` | `NOT NULL, PRIMARY KEY` | 消息唯一标识 |
| `topic_id` | `TEXT` | `NOT NULL` | 所属话题 ID |
| `hash` | `TEXT` | `NOT NULL` | 消息内容的哈希指纹 |
| `updated_at` | `INTEGER` | `NOT NULL` | 最后更新时间戳 |
| `deleted_at` | `INTEGER` | `DEFAULT NULL` | 软删除时间戳 |

**二级索引**：`idx_msg_topic` 建立在 `topic_id` 上，用于按话题快速检索消息列表。

#### 表 3：attachment_index（附件索引）

存储附件文件的哈希与物理路径映射。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `hash` | `TEXT` | `PRIMARY KEY` | 附件内容的 SHA-256 哈希（64 位十六进制） |
| `file_path` | `TEXT` | `NOT NULL` | 附件物理绝对路径 |
| `updated_at` | `INTEGER` | `NOT NULL` | 最后扫描时间戳 |
| `deleted_at` | `INTEGER` | `DEFAULT NULL` | 软删除时间戳（通过 `ALTER TABLE` 迁移添加） |

#### 表 4：avatar_index（头像索引）

存储 Agent 与 Group 的头像文件映射。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `owner_id` | `TEXT` | `NOT NULL, PK(1)` | 头像所属实体 ID |
| `owner_type` | `TEXT` | `NOT NULL, PK(2)` | 所属类型：`agent`、`group`、`user` |
| `file_path` | `TEXT` | `NOT NULL` | 头像文件绝对路径 |
| `hash` | `TEXT` | `NOT NULL` | 头像二进制内容的哈希 |
| `updated_at` | `INTEGER` | `NOT NULL` | 最后更新时间戳 |
| `deleted_at` | `INTEGER` | `DEFAULT NULL` | 软删除时间戳 |
| **主键** | — | `PRIMARY KEY (owner_id, owner_type)` | 复合主键 |

#### 表 5：message_attachments（消息附件关联）

与手机端对等设计，记录消息与附件的多对多关系。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `msg_id` | `TEXT` | `NOT NULL, PK(1)` | 消息 ID |
| `hash` | `TEXT` | `NOT NULL` | 附件哈希 |
| `attachment_order` | `INTEGER` | `NOT NULL, PK(2)` | 附件在消息中的顺序 |
| `display_name` | `TEXT` | `NOT NULL` | 附件展示名称 |
| `created_at` | `INTEGER` | `NOT NULL` | 创建时间戳 |
| **主键** | — | `PRIMARY KEY (msg_id, attachment_order)` | 复合主键 |

### 4.3 核心 CRUD 函数

`core/db.js` 封装了全部数据库操作，采用 `better-sqlite3` 的预编译语句（Prepared Statement）机制，保证高性能与防注入（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/db.js`）。

| 函数 | 操作类型 | 说明 |
|------|----------|------|
| `upsertEntityIndex(id, type, filePath, hash, updatedAt)` | Upsert | 插入或更新实体索引；`filePath` 为 `null` 时仅更新哈希与时间戳 |
| `upsertMessageIndex(msgId, topicId, hash, updatedAt)` | Upsert | 插入或更新消息索引；哈希变化时才更新时间戳 |
| `upsertAttachmentIndex(hash, filePath, updatedAt)` | Upsert | 插入或更新附件索引；冲突时恢复 `deleted_at` 为 NULL |
| `upsertAvatarIndex(ownerId, ownerType, filePath, hash, updatedAt)` | Upsert | 插入或更新头像索引 |
| `upsertMessageAttachment(msgId, hash, order, displayName, createdAt)` | Upsert | 插入或更新消息-附件关联 |
| `getEntityIndex(id, type)` | Read | 查询单条实体；支持 `agent_topic`/`group_topic` 回退到 `topic` 类型匹配 |
| `getEntitiesByType(type)` | Read | 按类型批量查询全部实体 |
| `getMessagesByTopic(topicId)` | Read | 按话题 ID 查询消息列表 |
| `softDeleteEntityIndex(id, type, deletedAt)` | Update | 软删除实体（设置 `deleted_at`） |
| `softDeleteMessageIndex(msgId, deletedAt)` | Update | 软删除消息 |
| `softDeleteAvatarIndex(ownerId, ownerType, deletedAt)` | Update | 软删除头像 |
| `cleanupOldDeletedRecords()` | Delete | 硬删除 30 天前的软删除记录 |

### 4.4 Upsert 语义细节

所有 `upsert*` 函数均使用 SQLite 的 `ON CONFLICT(...) DO UPDATE` 语法，且遵循**哈希变化才更新时间戳**的优化策略。以 `upsertEntityIndex` 为例：

```
ON CONFLICT(id, type) DO UPDATE SET
  hash = excluded.hash,
  updated_at = CASE WHEN entity_index.hash <> excluded.hash
                    THEN excluded.updated_at
                    ELSE entity_index.updated_at END
```

该语义确保：当文件被重新扫描但内容未变时，`updated_at` 不会无意义地刷新，从而减少差异比对中的 false positive（伪阳性）。

## 5. 索引扫描机制

### 5.1 reconcileLocalFiles 完整扫描流程

`reconcileLocalFiles(appDataPath)` 是插件启动时的核心初始化步骤，负责对桌面端本地全部数据进行一次性全量扫描，并将关键元数据写入 `sync_state.db`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

扫描按以下**严格顺序**执行：

| 顺序 | 扫描目标 | 目录/文件 | 处理方式 |
|------|----------|-----------|----------|
| 1 | 附件文件 | `AppData/UserData/attachments/` | 遍历目录下全部文件；优先以文件名（64 位 hex）作为哈希，格式不符则读取文件内容计算二进制哈希 |
| 2 | 用户头像 | `AppData/UserData/user_avatar.png` | 读取二进制并计算哈希，写入 `avatar_index`，`owner_type="user"` |
| 3 | Agent 实体 | `AppData/Agents/*/` | 遍历子目录，读取 `config.json`，提取 DTO 并计算哈希；同时索引子话题与头像 |
| 4 | Group 实体 | `AppData/AgentGroups/*/` | 同上，类型为 `group` |
| 5 | 消息历史 | `AppData/UserData/*/topics/*/history.json` | 遍历全部历史文件，逐条消息哈希入库 |
| 6 | 聚合指纹 | — | 为每个 Agent/Group 计算子话题的层级聚合哈希 |

### 5.2 SYSTEM_FOLDERS 白名单机制

在扫描 Agent/Group 目录以及 UserData 下的历史记录时，插件使用 `SYSTEM_FOLDERS` 数组作为**排除白名单**，跳过与业务无关的系统目录与文件。

| 类别 | 排除项示例 |
|------|-----------|
| 系统数据目录 | `UserData`、`AppData`、`avatarimage`、`canvas`、`DesktopData`、`DesktopWidgets` |
| 缓存与临时目录 | `MusicCoverCache`、`ResampleCache`、`tts_cache`、`WallpaperThumbnailCache` |
| 模块数据目录 | `Notemodules`、`Translatormodules`、`systemPromptPresets`、`generated_lists` |
| 附件相关目录 | `attachments`、`notes_attachments_agent`、`notes_attachments_group` |
| 配置文件 | `user_avatar.png`、`forum.config.json`、`settings.json`、`model_favorites.json` 等 |
| IDE/版本控制 | `.git`、`.idea`、`.vscode`、`node_modules` |

该白名单定义于 `index.js` 的 `SYSTEM_FOLDERS` 常量（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。任何名称匹配该白名单的目录在 `scanEntities` 与 `scanHistory` 中都会被直接跳过，防止将非实体目录误判为 Agent/Group。

### 5.3 扫描顺序的深层原因

扫描顺序并非随意排列，而是基于**依赖关系**设计：

1. **附件优先**：附件哈希是消息 DTO 的组成部分，先建立附件索引可保证后续消息哈希计算时引用一致；
2. **实体次之**：Agent 与 Group 是话题的父级容器，必须先入库才能验证话题归属；
3. **历史随后**：消息历史依赖话题 ID 的存在性，且消息数量通常最大，放在后面可利用前面已建立的实体映射；
4. **聚合最后**：`aggregated_hash` 依赖所有子话题的 `hash` 与 `aggregated_hash`，必须在全部话题索引完成后才能计算。

### 5.4 为什么必须先完成扫描才开放端口

这一决策是桌面端插件架构的**核心安全约束**：

- **数据完整性**：手机端连接后立即发送 `SYNC_MANIFEST`，如果服务端索引尚未就绪，返回的 manifest 将为空或不完整，导致手机端误判为"桌面端无数据"，进而执行全量 PUSH（把手机数据全部推到空桌面），可能引发灾难性覆盖；
- **竞争避免**：`reconcileLocalFiles` 执行期间会大量读写 `sync_state.db`，若此时同时接收手机端请求，会产生数据库锁竞争与状态不一致；
- **用户体验**：阻塞扫描通常在 1~3 秒内完成（取决于数据量），完成后日志输出 `"索引扫描完成"` 与 `"WebSocket 同步总线已启动"`，用户可明确判断服务就绪。

## 6. 文件监听机制

### 6.1 chokidar 监听范围

在索引扫描完成后，插件启动基于 `chokidar` 的文件监听器，持续监听 `AppData` 目录下的文件变更（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 监听路径 | `appDataPath` | 即 `AppData` 根目录 |
| `persistent` | `true` | 保持进程活跃，持续监听 |
| `ignoreInitial` | `true` | 忽略启动时的初始文件列表事件，防止重复扫描 |
| `depth` | `5` | 最大递归深度 5 层，覆盖 `Agents/<id>/config.json` 与 `UserData/<id>/topics/<topicId>/history.json` |

`chokidar` 是可选依赖（`try/catch` 包裹 `require`），若未安装则跳过文件监听，不影响同步服务本身，但会导致桌面端本地变更无法实时反映到索引中。

### 6.2 事件过滤与路径限制

监听器通过多层过滤，只处理真正关心的变更：

| 过滤层 | 规则 | 目的 |
|--------|------|------|
| 文件名过滤 | 仅 `history.json` 或 `config.json` | 忽略附件、头像、临时文件等无关变更 |
| 路径过滤 | 必须包含 `Agents/`、`AgentGroups/` 或 `UserData/` | 防止监听 `node_modules`、缓存目录等 |
| ID 清理 | `sanitizeId(id)` | 去除非法字符，防止路径遍历与注入 |
| 写入锁检查 | `isWriteLocked(id)` | 跳过当前正在被同步写入的实体，防止自冲突 |

只有同时通过上述四层过滤的事件，才会触发后续的索引更新操作。

### 6.3 writeIntentLock 防止冲突

`writeIntentLock` 是一个全局 `Set` 实例，用于标记当前正在被**同步协议**写入的实体 ID（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`）。

**加锁时机**：当手机端通过 `upload-entity` 或 `upload-entities-batch` 接口上传实体/话题变更时，`uploadEntity` 函数首先将目标 ID 加入 `writeIntentLock`。

**解锁时机**：写入完成后（无论成功或失败），通过 `setTimeout(() => writeIntentLock.delete(safeId), 1000)` 延迟 1 秒解锁。

**冲突场景**：如果在写入期间（或 1 秒延迟期内），用户通过桌面端 UI 修改了同一实体的配置，`chokidar` 会触发文件变更事件。此时 `isWriteLocked(id)` 返回 `true`，文件监听器直接跳过该事件，避免以下竞争：

```
手机端上传新 config → 写入 tmp → rename → 解锁
         ↑                              ↓
    chokidar 触发 ← 桌面端 UI 也改了文件 ─┘
```

若无 `writeIntentLock`，上述并发可能导致：
1. 手机端写入 `config.json.tmp` → `config.json`；
2. `chokidar` 检测到变更，读取刚写入的文件并重新索引；
3. 桌面端 UI 进程同时写入同一文件；
4. 最终文件内容可能是混合状态或损坏。

### 6.4 文件变更 → 索引更新的触发链

当文件事件通过全部过滤后，触发以下处理链：

| 文件类型 | 处理函数 | 更新内容 |
|----------|----------|----------|
| `config.json`（在 `Agents/` 或 `AgentGroups/` 下） | `ingestConfigToDb(configPath, type)` | 重新读取配置 → 提取 DTO → 计算哈希 → `upsertEntityIndex` → 重新索引子话题 → 触发 `computeAggregatedHashes` |
| `history.json`（在 `UserData/*/topics/` 下） | `ingestHistoryToDb(filePath, topicId)` | 解析消息数组 → 逐条计算哈希 → `upsertMessageIndex` |

`ingestConfigToDb` 在更新完实体自身索引后，会**显式调用 `computeAggregatedHashes`**，确保父级 Agent/Group 的 `aggregated_hash` 能反映子话题的变更。这是层级指纹一致性在运行时得以维持的关键机制。

## 7. 安全机制

### 7.1 Token 认证（双通道统一鉴权）

桌面端插件在 WebSocket 与 HTTP 两条通道上均实施基于 `MobileSyncToken` 的认证，且两种协议的认证逻辑相互独立、策略一致。

**WebSocket 认证**（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/websocket.js`）：

| 项 | 说明 |
|----|------|
| 认证位置 | 连接建立时的 URL Query 参数 |
| 参数名 | `token` |
| 示例 | `ws://192.168.1.100:5975/ws-sync?token=your_secret` |
| 失败行为 | 关闭连接，code `4001`，reason `Unauthorized` |
| 路径校验 | 仅允许 `/` 或 `/ws-sync`，其他路径返回 code `1008` |

**HTTP 认证**（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js`）：

| 项 | 说明 |
|----|------|
| 认证位置 | 请求头或 Query 参数 |
| 支持头 | `x-sync-token`、`Authorization: Bearer <token>` |
| 支持参数 | `?token=...` |
| 失败行为 | 返回 HTTP `401 Unauthorized` |
| CORS | `Access-Control-Allow-Origin: *`，允许跨域 |

两个通道的认证令牌来源相同：均由 VCPDistributedServer 从 `config.env` 读取并注入 `pluginConfig.MobileSyncToken`，最终传入插件。

### 7.2 ID 清理（sanitizeId）

所有来自手机端的 ID 参数在进入业务逻辑前，必须经过 `sanitizeId` 函数清洗（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`）。

| 输入 | 输出 | 说明 |
|------|------|------|
| `agent_123` | `agent_123` | 合法字符保留 |
| `agent/123` | `agent123` | 斜杠被移除，防止路径遍历 |
| `../../etc/passwd` | `etcpasswd` | 危险字符全部被过滤 |
| `<script>alert(1)</script>` | `scriptalert1script` | 注入字符被过滤 |
| `null` / `undefined` / 非字符串 | `""` | 空字符串，后续逻辑视为无效 ID |

清理规则为正则表达式 `id.replace(/[^a-zA-Z0-9_\-]/g, "")`，仅保留字母、数字、下划线与连字符。这一机制在以下场景中被调用：

- 手机端请求下载/上传实体时；
- 文件监听器从路径提取 Topic ID 时；
- WebSocket 收到 `SYNC_DELETE_NOTIFY` 时；
- 批量下载/上传实体时。

### 7.3 损坏配置保护

桌面端插件在**写入任何配置文件前**，都会执行严格的读取校验，防止因覆盖损坏文件而导致数据丢失（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`）。

**校验流程**：

| 步骤 | 检查内容 | 失败行为 |
|------|----------|----------|
| ① 读取文件 | `fs.readFile` 是否成功 | `ENOENT` 视为新文件，其他错误进入下一步 |
| ② 空文件检查 | `content.trim() === ""` | 抛出 `"Empty config file"`，拒绝写入 |
| ③ 结构校验 | `typeof config !== "object"` 或 `Array.isArray(config)` | 抛出 `"Invalid config structure"`，拒绝写入 |
| ④ ID 匹配校验（Group） | `config.id !== safeId` | 抛出 `"Config ID mismatch"`，拒绝写入 |
| ⑤ 数组拒绝写入 | `Array.isArray(config)` | 抛出 `"Refusing to write array as config"`，拒绝写入 |

上述校验确保：即使本地 `config.json` 因磁盘故障、进程崩溃等原因损坏，手机端同步也不会盲目覆盖，而是返回明确的错误信息，提示用户人工检查桌面端数据。

### 7.4 原子写入（tmp → rename 模式）

所有涉及配置文件（`config.json`）与历史文件（`history.json`）的写入操作，均采用**原子写入**策略，防止并发或异常导致文件内容为空或部分写入（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`、`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/message.js`）。

**原子写入步骤**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| ① 生成临时路径 | `${targetPath}.tmp_${Date.now()}_${random}` | 包含时间戳与随机后缀，确保唯一性 |
| ② 写入临时文件 | `fs.writeFile(tmpPath, content, "utf-8")` | 全部内容一次性写入临时文件 |
| ③ 原子重命名 | `fs.rename(tmpPath, targetPath)` | 利用操作系统原子的 `rename` 语义替换目标文件 |

**原子性保证**：
- 若步骤 ② 失败（进程崩溃、磁盘满），原 `config.json` 完全不受影响；
- 若步骤 ③ 失败（极罕见，通常因文件系统错误），临时文件残留，原文件仍然完整；
- 由于 `rename` 在主流文件系统（NTFS、ext4、APFS）上为原子操作，其他进程永远不会读到半写入状态。

使用该模式的场景包括：

| 场景 | 源文件 |
|------|--------|
| 单实体上传 | `config.json` |
| 批量实体上传 | `config.json` |
| 头像上传后更新 Group 配置 | `config.json` |
| Topic 删除后更新父配置 | `config.json` |
| 消息批量上传 | `history.json` |

### 7.5 版本校验

手机端在建立 WebSocket 连接后，通常会发送 `VERSION_CHECK` 消息，携带自身版本号。桌面端插件读取 `plugin-manifest.json` 中的 `version` 字段并返回（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

| 消息方向 | 类型 | 载荷 |
|----------|------|------|
| 手机端 → 桌面端 | `VERSION_CHECK` | `{ mobileVersion: "0.9.13" }` |
| 桌面端 → 手机端 | `VERSION_ACK` | `{ version: "0.9.13" }` |

版本校验的主要目的不是强制拦截，而是**在日志中记录双方版本**，便于排查因版本不匹配导致的协议差异问题。

### 7.6 幂等性控制

HTTP 上传接口（`/upload-entity`）支持通过请求头 `x-idempotency-key` 实现幂等性（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/idempotency.js`）。

| 机制 | 说明 |
|------|------|
| 存储结构 | `Map<opId, { timestamp, result }>` |
| TTL | `300000` 毫秒（5 分钟） |
| 自动清理 | 每分钟遍历一次，移除过期记录 |
| 命中行为 | 直接返回上一次的操作结果，不再执行写入 |

该机制防止手机端因网络超时重试而导致同一实体被重复上传、重复覆盖。

## 8. 已知限制清单

以下限制是桌面端插件在当前版本（`0.9.13`）中的已知行为约束，运维与开发中需特别注意。

### 8.1 Agent 配置不写入 `id` 字段

桌面端 `Agents/<id>/config.json` 的规范是：**`id` 字段不写入配置文件**，实体 ID 由目录名推导（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/config/defaults.js`）。

| 对比项 | Agent | Group |
|--------|-------|-------|
| 配置是否包含 `id` | **否** | 是（必须） |
| ID 来源 | 目录名 | `config.json` 中的 `id` 字段 |
| 校验逻辑 | 上传时不校验 ID 匹配 | 上传时强制校验 `config.id === safeId` |
| 新建逻辑 | `createAgentConfig(id, dto)` 不写入 `id` | `createGroupConfig(id, dto)` 显式写入 `id` |

这一差异源于 VCPChat 桌面端的历史设计：Agent 配置一直由目录名隐含标识。手机端同步时必须注意，上传 Agent DTO 时桌面端会忽略 `id` 字段，仅以 URL/路径中的 ID 为准。

### 8.2 首次同步前必须完成索引扫描

如第 5.4 节所述，插件在 `reconcileLocalFiles` 完成前**不会**开放 WebSocket 与 HTTP 端口。这意味着：

- 如果用户在 VCPDistributedServer 启动后立即尝试手机端同步，可能会遇到连接被拒绝；
- 日志中必须出现 `"WebSocket 同步总线已启动"` 后，手机端才能成功连接；
- 数据量极大时（数万条消息、数百个 Agent），扫描可能需要数秒，用户需耐心等待。

### 8.3 多手机连接风险

虽然 WebSocket 服务器在技术上允许**多个客户端同时连接**（`ws` 模块的 `wss.clients` 是一个集合），但插件**未实现多客户端会话隔离**与**并发写入排队**机制。

| 风险场景 | 后果 |
|----------|------|
| 手机 A 与手机 B 同时修改同一 Agent | 后完成的写入会覆盖前者，导致数据丢失 |
| 手机 A 同步时手机 B 也发起同步 | 两边同时拉取/推送，可能产生状态不一致 |
| 文件监听与同步写入并发 | `writeIntentLock` 只能防护单一实体 1 秒窗口，跨实体无保护 |

**官方建议**：一次只连接一台手机。如需更换设备，请先断开旧手机的连接，或重启桌面端后再连接新手机。

### 8.4 附件路径限制

桌面端附件索引使用**绝对路径**存储（`attachment_index.file_path`）。这意味着：

- 如果桌面端的 `AppData` 目录被移动（如从 `C:\VCPChat\AppData` 迁移到 `D:\VCPChat\AppData`），已有附件索引将全部失效；
- 附件文件名以哈希值命名（`<hash>.<ext>`），但实际查找依赖 `file_path` 字段；
- 若附件文件在扫描后被手动移动或重命名，索引与实际文件脱节，手机端下载时会返回 `404`。

**缓解措施**：更新插件或迁移数据时，**务必删除 `sync_state.db`**，强制插件重新扫描并重建路径索引。

### 8.5 更新插件需删除 `sync_state.db`

`sync_state.db` 是插件的本地缓存数据库， Schema 与索引策略可能随版本升级而变更。当前版本未实现自动数据库迁移（除 `attachment_index` 的 `deleted_at` 字段通过 `try/catch ALTER TABLE` 补丁添加外）。

| 升级场景 | 必须操作 | 原因 |
|----------|----------|------|
| 插件版本升级 | 删除 `sync_state.db` 后重启 | 新字段、新索引策略、新哈希算法可能导致旧数据不兼容 |
| `AppData` 路径变更 | 删除 `sync_state.db` 后重启 | 附件路径绝对化，旧路径全部失效 |
| 发现索引异常 | 删除 `sync_state.db` 后重启 | 强制全量重建索引，修复潜在的不一致 |
| 首次安装 | 无需操作 | 数据库会自动创建 |

`sync_state.db` 位于插件目录下（`VCPMobileSync/sync_state.db`），删除后插件在下次启动时会自动重新执行 `reconcileLocalFiles`，重建完整索引。

### 8.6 其他已知约束

| 约束 | 说明 | 相关文件 |
|------|------|----------|
| `chokidar` 为可选依赖 | 未安装时文件监听不启动，桌面端本地变更不会实时同步到索引，需等待下次重启重扫 | `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js` |
| 附件单文件 100MB 限制 | HTTP 路由 `express.raw({ limit: "100mb" })`，超过该限制的附件上传会被拒绝 | `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js` |
| 历史消息 JSON 5MB 限制 | 单实体上传路由 `express.json({ limit: "5mb" })`，批量上传为 `10mb` | `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js` |
| 日志文件无自动轮转 | `SyncLogger` 每个会话生成一个 `.log` 文件，无按大小/日期轮转机制，长期运行可能积累大量日志 | `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/logger.js` |
| 删除记录 30 天硬清理 | `cleanupOldDeletedRecords` 每 60 分钟执行一次，删除 `deleted_at < now - 30天` 的记录，已删除数据超过 30 天无法恢复 | `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/db.js` |

## 9. WebSocket 消息协议

### 9.1 消息类型总览

桌面端插件通过 WebSocket 接收手机端发送的同步指令，所有消息均采用 JSON 格式。`index.js` 中的 `onMessage` 回调是 WebSocket 层的中央分发器（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

| 消息类型（Type） | 方向 | 功能 | 响应类型 |
|-----------------|------|------|----------|
| `SYNC_MANIFEST` | 手机 → 桌面 | 发送手机端全量 manifest，请求差异比对 | `SYNC_DIFF_RESULTS`（由 `handleSyncManifest` 返回） |
| `GET_MESSAGE_MANIFEST` | 手机 → 桌面 | 请求指定话题的消息 manifest | 消息差异结果 |
| `SYNC_TOPIC_HASH_BATCH` | 手机 → 桌面 | V1 批量话题哈希比对 | 差异指令集 |
| `SYNC_TOPIC_HASH_BATCH_V2` | 手机 → 桌面 | V2 批量话题哈希比对（支持聚合哈希） | 差异指令集 |
| `SYNC_MESSAGE_DIFF_BATCH` | 手机 → 桌面 | 批量消息差异同步请求 | 消息差异结果 |
| `PHASE_START` | 手机 → 桌面 | 通知桌面端某阶段开始 | `PHASE_ACK` |
| `PHASE_COMPLETED` | 手机 → 桌面 | 通知桌面端某阶段完成 | `PHASE_ACK` |
| `SYNC_ENTITY_UPDATE` | 手机 → 桌面 | 通知桌面端某实体已被更新 | `SYNC_ACK` |
| `VERSION_CHECK` | 手机 → 桌面 | 版本校验请求 | `VERSION_ACK` |
| `SYNC_DELETE_NOTIFY` | 手机 → 桌面 | 通知桌面端某实体/消息已被删除 | `SYNC_ACK` |

### 9.2 阶段确认机制

`PHASE_START` 与 `PHASE_COMPLETED` 是手机端向桌面端报告同步阶段状态的信令。桌面端收到后仅返回 `PHASE_ACK` 作为确认，**不再返回冗余的 manifest 数据**。这是因为所有 manifest 数据已在 `SYNC_MANIFEST` 阶段由手机端主动发送并处理完毕。

| 字段 | 说明 |
|------|------|
| `phase` | 阶段名称，如 `owner_metadata`、`topic_metadata`、`messages` |
| `PHASE_ACK` 载荷 | `{ type: "PHASE_ACK", phase }` |

### 9.3 删除通知处理

`SYNC_DELETE_NOTIFY` 是手机端向桌面端传播删除操作的通道。桌面端根据 `dataType` 执行不同的删除策略：

| `dataType` | 处理逻辑 |
|-----------|----------|
| `message` | 调用 `deleteMessage`，对 `message_index` 执行软删除 |
| `avatar` | 解析 `id` 格式（`type:ownerId`），对 `avatar_index` 执行软删除 |
| 其他（`agent`/`group`/`topic`） | 调用 `deleteEntity`，对 `entity_index` 执行软删除，并删除物理目录（Agent/Group）或从父配置中移除（Topic） |

所有删除操作均先经过 `sanitizeId` 清洗，防止非法 ID 注入。

## 10. DTO 提取与默认值系统

### 10.1 默认值配置中心

`config/defaults.js` 是桌面端插件的**默认值配置中心**，定义了新建 Agent、Group、Topic 时的完整默认属性（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/config/defaults.js`）。

| 配置对象 | 用途 | 关键字段 |
|----------|------|----------|
| `AGENT_DEFAULTS` | 新建 Agent 的默认配置 | `name`、`model`、`temperature`、`contextTokenLimit`、`maxOutputTokens`、`streamOutput` |
| `GROUP_DEFAULTS` | 新建 Group 的默认配置 | `name`、`mode`、`members`、`groupPrompt`、`invitePrompt`、`useUnifiedModel` |
| `AGENT_TOPIC_DEFAULTS` | 新建 Agent Topic 的默认配置 | `locked`、`unread`、`creatorSource` |

### 10.2 Agent 配置的特殊设计

`createAgentConfig(id, dto)` 函数体现了 Agent 配置的两大特殊设计：

| 设计点 | 说明 |
|--------|------|
| **不写入 `id`** | 返回的配置对象中不包含 `id` 字段，Agent ID 完全由目录名推导 |
| **动态系统提示词** | 若手机端未提供 `systemPrompt`，默认生成 `"你是 ${name}。"` |
| **桌面端特有字段注入** | `disableCustomColors: true`、`useThemeColorsInChat: true` 强制注入，保证桌面端 UI 一致性 |
| **topics 数组初始为空** | 新建 Agent 的 `topics` 初始化为 `[]`，话题由 Phase 2 独立同步 |

### 10.3 Group 配置的 ID 强制校验

与 Agent 相反，Group 配置**必须包含 `id` 字段**，且上传时严格校验匹配性：

```
if (type === "group" && config.id !== safeId) {
  throw new Error(`Config ID mismatch for ${safeId}`);
}
```

此校验防止手机端因 ID 混淆而将 Group A 的数据误写入 Group B 的配置文件中。

### 10.4 附件对象的跨端映射

`createDesktopAttachment(dto, desktopPath, ext)` 函数负责将手机端的附件 DTO 转换为桌面端标准附件结构：

| 字段映射 | 手机端 DTO | 桌面端结构 |
|----------|-----------|-----------|
| 路径格式 | 相对路径或无 | 统一添加 `file://` 前缀 |
| 内部文件名 | 无 | `${hash}${ext}` |
| `_fileManagerData` | 无 | 完整填充 `id`、`name`、`internalPath`、`hash` 等元数据 |

桌面端附件的物理存储位置固定为 `AppData/UserData/attachments/<hash><ext>`，该路径在 `createDesktopAttachment` 中作为 fallback（兜底）路径硬编码。

## 11. 日志系统（SyncLogger）

### 11.1 三层日志架构

桌面端日志系统 `SyncLogger` 采用与手机端对齐的三层架构：Session（会话）→ Phase（阶段）→ Operation（操作）（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/logger.js`）。

| 层级 | 方法 | 作用 |
|------|------|------|
| Session | `startSession(prefix)` / `endSession()` | 标记一次完整同步的生命周期，生成独立的 `.log` 文件 |
| Phase | `startPhase(phase, expected)` / `completePhase(phase)` | 标记同步的一个大阶段，如 `owner_metadata`、`topic_metadata`、`messages` |
| Operation | `logOperation(...)` / `logInfo(...)` | 记录单个操作的结果，如实体下载、文件变更、错误信息 |

### 11.2 日志输出通道

`SyncLogger` 同时向三个通道输出日志：

| 通道 | 目的地 | 格式 | 用途 |
|------|--------|------|------|
| 控制台 | `console.log` | `[VCPMobile] [HH:mm:ss] message` | 开发调试 |
| 文件 | `logs/sync/<prefix>_<timestamp>.log` | ISO 时间戳 + 原始消息 | 持久化排查 |
| WebSocket 广播 | `wss.clients` | JSON `{ type, level, phase, message, ts }` | 实时推送到手机端展示 |

### 11.3 阶段自动收尾

`SyncLogger` 具备**阶段自动收尾**机制：

- 当调用 `startPhase(newPhase)` 时，若上一个阶段尚未 `completePhase`，会自动补调用 `completePhase(prevPhase)`；
- 当调用 `endSession()` 时，会自动遍历并收尾所有未完成的阶段；
- 日志文件流在 `endSession()` 后延迟 500ms 关闭，确保最后一批日志落盘。

### 11.4 进度报告

`logOperation` 内部维护每个阶段的 `processed`、`success`、`errors` 计数器。当 `processed % 100 === 0` 时，自动输出进度摘要：

```
[PROGRESS] messages: 500/? (OK:498 ERR:2)
```

此设计使得处理大批量数据（如万级消息同步）时，用户可通过日志实时感知进度。

## 12. HTTP 路由详述

### 12.1 路由挂载点

所有 HTTP 路由挂载在 `/api/mobile-sync/` 前缀下（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/transport/routes.js`）。完整 URL 示例：

```
http://192.168.1.100:5974/api/mobile-sync/download-entity?id=xxx&type=agent
```

### 12.2 路由清单

| 方法 | 路径 | 功能 | 请求体/参数限制 |
|------|------|------|----------------|
| `GET` | `/download-entity` | 下载单个实体 DTO | Query: `id`, `type` |
| `POST` | `/download-entities` | 批量下载实体 DTO | JSON `{ requests: [{id, type}] }` |
| `POST` | `/upload-entity` | 上传单个实体 | JSON `{ id, type, data }`，Limit: 5MB，支持幂等键 |
| `POST` | `/upload-entities-batch` | 批量上传实体 | JSON `{ items: [{id, type, data}] }`，Limit: 10MB |
| `POST` | `/download-messages-stream` | 流式批量下载消息 | JSON `{ requests: [{topicId, msgIds?}] }`，NDJSON 响应 |
| `POST` | `/upload-messages-batch` | 流式批量上传消息 | NDJSON 请求流，Raw body |
| `POST` | `/upload-attachment` | 上传附件 | Raw binary，Query: `hash`, `name`, `type`，Limit: 100MB |
| `GET` | `/download-attachment` | 下载附件 | Query: `hash`，返回文件流 |
| `GET` | `/download-avatar` | 下载头像 | Query: `id`, `type`，返回文件流 |
| `POST` | `/upload-avatar` | 上传头像 | Raw binary，Query: `id`, `type`，Limit: 10MB |
| `POST` | `/delete-entity` | 删除实体 | JSON `{ id, type, deletedAt }` |
| `POST` | `/delete-message` | 删除消息 | JSON `{ msgId, deletedAt }` |

### 12.3 批量下载优化

`/download-entities` 实现了**按文件路径分组**的优化策略：

- 多个 Topic 可能属于同一个 `config.json`（同一 Agent/Group 下的多个话题）；
- 路由处理时先按 `file_path` 分组，每个 `config.json` 只读取一次；
- 然后从已解析的配置中提取多个 Topic DTO，显著减少磁盘 I/O。

### 12.4 流式消息同步

`/download-messages-stream` 与 `/upload-messages-batch` 采用 **NDJSON（Newline Delimited JSON）** 格式进行流式传输，解决万级消息同步时的内存溢出问题：

| 特性 | 说明 |
|------|------|
| 请求格式 | 每行一个独立 JSON 对象，以 `\n` 分隔 |
| 响应格式 | 同 NDJSON，支持边处理边返回 |
| 内存优势 | 无需在内存中加载全部消息数组，处理一条输出一条 |
| 错误处理 | 若流已开始，通过 `_stream_error` 帧返回错误，避免中断连接 |

## 13. 性能与数据一致性

### 13.1 预加载与 N+1 消除

`computeAggregatedHashes` 函数在计算层级聚合指纹时，采用了**预加载分组**策略，消除了典型的 N+1 查询问题（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/index.js`）。

| 优化前（N+1） | 优化后 |
|--------------|--------|
| 遍历每个 Agent/Group → 查询其子话题 | 一次性读取全部 Topic → 按 `parentId` 分组存入 `Map` → 遍历 Agent/Group 时直接查 `Map` |
| 时间复杂度 O(N×M) | 时间复杂度 O(N+M) |

### 13.2 文件读取批量优化

`downloadEntities` 与 `uploadEntitiesBatch` 均实现了**按配置文件分组**的批量读取：

- `downloadEntities`：按 `file_path` 将请求分组，每个 `config.json` 仅 `fs.readFile` 一次；
- `uploadEntitiesBatch`：同样按 `configPath` 分组，每个文件执行一次"读取 → 修改 → 原子写入"周期。

### 13.3 数据一致性保障

桌面端插件通过多层机制保证索引与实际文件系统的一致性：

| 机制 | 一致性保障 | 触发场景 |
|------|-----------|----------|
| 启动全量扫描 | 重建完整索引基线 | 每次 VCPDistributedServer 启动 |
| 文件监听增量更新 | 实时同步本地变更 | 桌面端 UI 修改配置或历史 |
| 原子写入 | 防止半写入导致文件损坏 | 任何配置/历史写入操作 |
| 写入意图锁 | 防止同步协议与文件监听的并发冲突 | 手机端上传实体时 |
| 层级聚合哈希 | 父级哈希反映全部子级状态 | 每次 Topic 变更后 |
| 定期清理 | 移除过期软删除记录，防止索引膨胀 | 每小时 + 启动时 |

### 13.4 哈希计算策略

桌面端插件使用多种哈希算法对应不同数据类型：

| 数据类型 | 哈希算法 | 来源 |
|----------|----------|------|
| 二进制文件（附件、头像） | SHA-256 | `computeBinaryHash`（`core/hash.js`） |
| 配置 DTO（Agent/Group/Topic） | SHA-256 | `computeDtoHash`（`core/hash.js`） |
| 层级聚合 | SHA-256 | `computeAggregatedHash`（`core/hash.js`） |
| 消息内容 | SHA-256 | 消息文本规范化后计算 |

所有哈希值均以 64 位十六进制字符串存储，与手机端算法严格对齐，确保跨端差异比对时哈希可直接比较。
