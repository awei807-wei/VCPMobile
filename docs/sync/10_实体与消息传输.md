# 实体与消息传输

## 1. 引言

### 1.1 实体传输与消息传输的区别

VCPMobile 同步协议将数据分为两大类别：**实体（Entity）** 和 **消息（Message）**。两者在数据形态、传输策略和 I/O 模型上存在本质差异。

| 维度 | 实体传输 | 消息传输 |
|------|---------|---------|
| 数据载体 | `config.json`（Agent/Group 配置） | `history.json`（Topic 消息历史） |
| 单次量级 | 轻量（KB 级 JSON） | 重（万级消息可达数十 MB） |
| 传输模式 | 请求-响应（Request-Response） | 流式（NDJSON Stream） |
| 并发策略 | 幂等键（Idempotency Key）防重 | 写意图锁（Write Intent Lock）隔离 |
| 批量优化 | 按 `file_path` / `configPath` 分组 | 按 Topic 归口合并 |
| 相关文件 | `entity.js` | `message.js` |

### 1.2 为什么需要两套传输体系

实体（Agent、Group、Topic 元数据）具有以下特征：
- 结构扁平，字段有限（Agent/Group 配置通常小于 50 KB）
- 更新频率低，但要求强一致性
- 需要支持**新建**（目录创建）和**局部更新**（DTO 局部覆盖）

消息历史则完全相反：
- 单 Topic 消息数可达 **10,000+**，整体 JSON 序列化后体积巨大
- 若使用传统 JSON 数组整体传输，服务端和客户端均需将完整内容缓冲在内存，极易触发 **OOM（Out of Memory，内存溢出）**
- 消息追加为主，更新为辅，适合增量流式合并

因此，实体传输采用传统的 REST 风格批量接口，消息传输采用 **NDJSON（Newline Delimited JSON，换行分隔 JSON）** 流式协议。

### 1.3 术语对照表

| 英文术语 | 中文翻译 | 说明 |
|---------|---------|------|
| DTO | 数据传输对象（Data Transfer Object） | 移动端与桌面端交换的数据子集 |
| NDJSON | 换行分隔 JSON | 每行一个独立 JSON 对象，适合流式处理 |
| OOM | 内存溢出 | 进程内存耗尽导致的崩溃 |
| Write Intent Lock | 写意图锁 | 防止文件监听器与同步写入冲突的标记机制 |
| Idempotency Key | 幂等键 | 确保同一操作只执行一次的令牌 |
| Flush | 刷新缓冲区 | 将已缓冲数据立即写入网络/磁盘 |



## 2. 实体 CRUD 完整流程

实体 CRUD 操作的入口由 `transport/routes.js` 暴露为 HTTP 端点，业务逻辑全部委托给 `sync/entity.js`。所有实体类型共享同一套核心流程，但在 DTO（数据传输对象）提取与合并策略上按类型分化。

支持的实体类型：

| 类型标识 | 实际含义 | 存储位置 | DTO 处理函数 |
|---------|---------|---------|------------|
| `agent` | 智能体 | `Agents/{id}/config.json` | `extractAgentDTO` / `applyAgentDTO` |
| `group` | 群组 | `AgentGroups/{id}/config.json` | `extractGroupDTO` / `applyGroupDTO` |
| `topic` / `agent_topic` | Agent 话题 | 嵌入父级 `config.json` 的 `topics` 数组 | `extractAgentTopicDTO` / `applyAgentTopicDTO` |
| `group_topic` | 群组话题 | 嵌入父级 `config.json` 的 `topics` 数组 | `extractGroupTopicDTO` / `applyGroupTopicDTO` |
| `avatar` | 头像 | `Agents\|AgentGroups/{id}/avatar.png` | 独立二进制传输 |

### 2.1 单实体下载（downloadEntity）

`downloadEntity` 负责将桌面端的本地配置转换为移动端可消费的 DTO。流程如下：

```
输入: { id, type }
  ↓
sanitizeId(id) —— 过滤非法字符（仅保留 a-zA-Z0-9_-）
  ↓
查询 entity_index 获取 file_path
  ↓
读取 file_path → JSON.parse → config 对象
  ↓
判断归属:
  ├─ type ∈ {topic, agent_topic, group_topic}
  │   → 在 config.topics 中按 id 查找 topic 对象
  │   → isGroup ? extractGroupTopicDTO() : extractAgentTopicDTO()
  │
  ├─ type == agent → extractAgentDTO(config)
  └─ type == group → extractGroupDTO(config)
  ↓
返回 DTO 或 null（实体不存在/读取失败时）
```

**关键实现细节**（`sync/entity.js`）：

| 细节 | 说明 |
|------|------|
| Phase 标记 | Topic 元数据标记为 `"topic_metadata"`，Agent/Group 标记为 `"owner_metadata"`，用于日志分类 |
| Owner 推导 | `ownerId = config.id \|\| path.basename(path.dirname(file_path))` —— 兼容旧配置无显式 `id` 字段的情况 |
| 失败静默 | 任何读取异常均返回 `null`，不抛异常中断批量流程 |

### 2.2 批量实体下载（downloadEntities）的分组策略

当移动端需要拉取数十个实体时，若逐个调用 `downloadEntity`，会造成同一 `config.json` 被反复读取（例如一个 Agent 有 10 个 Topic，则该 Agent 的 `config.json` 会被读 10 次）。`downloadEntities` 通过 **按 `file_path` 预分组** 解决此问题。

**分组算法**：

```
输入: requests[] = [{id, type}, ...]

fileGroups = Map<file_path, { isGroup, reqs[] }>

for each req in requests:
    row = getEntityIndex(safeId, req.type)
    if row not exists: continue
    
    if fileGroups 不包含 row.file_path:
        fileGroups.set(row.file_path, {
            isGroup: row.file_path.includes("AgentGroups"),
            reqs: []
        })
    
    fileGroups.get(row.file_path).reqs.push({ req, safeId })

// 遍历分组，每个 file_path 只读一次
for each [filePath, {isGroup, reqs}] of fileGroups:
    content = fs.readFile(filePath)
    config = JSON.parse(content)
    ownerId = config.id || path.basename(path.dirname(filePath))
    
    for each {req, safeId} in reqs:
        // 在同一 config 对象内提取多个 DTO
        dto = extractXXXTDTO(...)
        if dto: results.push({ id: req.id, type, data: dto })
```

**优化效果**：

| 场景 | 不分组 I/O | 分组后 I/O |
|------|-----------|-----------|
| 10 个 Topic 属于同一 Agent | 10 次读取 | 1 次读取 |
| 跨 5 个 Agent 共 20 个 Topic | 20 次读取 | 5 次读取 |



### 2.3 单实体上传（uploadEntity）

`uploadEntity` 是移动端向桌面端推送实体变更的核心函数，支持**新建**、**更新**和**损坏保护**。流程复杂度远高于下载，因为涉及目录创建、配置合并和索引更新。

**整体流程**：

```
输入: { id, type, data, appDataPath }
  ↓
sanitizeId(id)
  ↓
判断类型: isTopic = type ∈ {topic, agent_topic, group_topic}
  ↓
查找或推导 configPath:
  ├─ row = getEntityIndex(safeId, type)
  ├─ row 存在 → configPath = row.file_path
  ├─ row 不存在且 !isTopic → 新建 Agent/Group 目录 + config.json
  └─ row 不存在且 isTopic → 根据 data.ownerId + data.ownerType 推导父级路径
      → 验证父级 config.json 存在性（不存在则拒绝）
  ↓
加 writeIntentLock(safeId) —— 1 秒后自动释放
  ↓
读取现有配置:
  ├─ 成功 → fileReadSuccess = true
  ├─ ENOENT → fileReadSuccess = false（允许新建）
  └─ 损坏/空/非对象 → 抛出异常，拒绝覆盖
  ↓
按类型合并:
  ├─ isTopic → handleTopicUpload()（局部更新 topics 数组）
  ├─ agent → handleAgentUpload()（DTO 局部覆盖，保留 topics）
  └─ group → handleGroupUpload()（DTO 局部覆盖，强制写入 id）
  ↓
写入前校验:
  ├─ config 不能是数组
  └─ group 的 config.id 必须等于 safeId
  ↓
原子写入: tmp → rename 模式
  ↓
更新索引:
  ├─ Topic → updateTopicIndex()（重新计算 DTO 哈希）
  └─ Agent/Group → upsertEntityIndex()（提取 DTO 后计算哈希）
  ↓
返回 { success: true } 或 { success: false, error }
```

**Agent 上传逻辑**（`handleAgentUpload`）：

| 条件 | 行为 |
|------|------|
| `!fileReadSuccess && isNewEntity` | 调用 `createAgentConfig(id, data)` 全新创建 |
| `!fileReadSuccess && !isNewEntity` | 索引与实际文件不一致，使用 DTO 重建 |
| `fileReadSuccess` | 调用 `applyAgentDTO(config, data)` 局部覆盖，**绝不触碰 `topics` 数组** |

**Group 上传逻辑**（`handleGroupUpload`）：

与 Agent 类似，但 Group 配置**必须包含显式 `id` 字段**（因为 Group 目录名可能与 id 不同），因此在更新时会强制写入 `config.id = id`。

**Topic 上传逻辑**（`handleTopicUpload`）：

```
输入: { config, id, entityType, data, configPath, appDataPath }

if config 无效（数组/null/非对象）:
    抛出异常

if !Array.isArray(config.topics):
    config.topics = []

topicIdx = config.topics.findIndex(t => t.id === id)

if topicIdx > -1:
    // 更新现有 Topic
    config.topics[topicIdx] = isGroupTopic
        ? applyGroupTopicDTO(existing, data)
        : applyAgentTopicDTO(existing, data)
else:
    // 新建 Topic
    newTopic = isGroupTopic ? createGroupTopic(data) : createAgentTopic(data)
    config.topics.push(newTopic)
    
    // 确保 history.json 存在（空数组）
    historyDir = appDataPath/UserData/{parentId}/topics/{id}
    mkdir(historyDir, recursive)
    writeFile(historyDir/history.json, "[]")

// 按 createdAt 降序排序
config.topics.sort((a, b) => (b.createdAt || 0) - (a.createdAt || 0))

return config
```

**Topic 排序规则**：桌面端要求 `topics` 数组按 `createdAt` 降序排列，因此每次上传后都会重新排序。

### 2.4 批量实体上传（uploadEntitiesBatch）的 Topic 归口优化

`uploadEntitiesBatch` 是 V2 引入的批量接口，专为**大量 Topic 元数据同步**设计。与循环调用 `uploadEntity` 相比，核心差异在于**按 `configPath` 归口**，将属于同一父级实体的多个 Topic 合并为一次文件读写。

**归口算法**：

```
输入: items[] = [{id, type, data}, ...]

fileGroups = Map<configPath, { items[] }>

for each item in items:
    safeId = sanitizeId(item.id)
    isTopic = type ∈ {topic, agent_topic, group_topic}
    
    // 解析 configPath
    row = getEntityIndex(safeId, type)
    if row:
        configPath = row.file_path
    else if isTopic && data.ownerId:
        parentBaseDir = (type == "group_topic" || data.ownerType == "group")
            ? "AgentGroups" : "Agents"
        configPath = appDataPath / parentBaseDir / data.ownerId / config.json
    else if !isTopic:
        // 批量上传暂不支持新建 Agent/Group
        results.push({ id, success: false, error: "Cannot resolve config path" })
        continue
    
    fileGroups.getOrCreate(configPath).items.push({ id: safeId, type, data })

// 每个 configPath 只执行一次 读取→修改→写入
for each [configPath, group] of fileGroups:
    try:
        config = 读取 configPath（ENOENT 则初始化为 {}）
        for each item in group.items:
            config = handleTopicUpload() / handleAgentUpload() / handleGroupUpload()
            results.push({ id, success: true })
        
        原子写入 configPath
        
        // 批量更新索引
        for each item in group.items:
            updateTopicIndex() 或 upsertEntityIndex()
    catch e:
        // 文件级错误，整组标记失败
        for each item in group.items:
            results.push({ id, error: "File error: ..." })
```

**限制说明**：

| 限制项 | 说明 |
|--------|------|
| 不支持新建 Agent/Group | 批量接口仅优化 Topic 更新，新建 Agent/Group 需调用单实体接口 |
| 原子性粒度 | 以 `configPath` 为原子单位，同一文件内的所有 Topic 要么全部成功，要么全部失败 |
| 索引更新时机 | 文件写入成功后批量更新索引，避免中间状态 |

### 2.5 实体删除（deleteEntity）的软删除 + 物理删除

`deleteEntity` 采用**软删除优先、物理删除兜底**的策略，确保多端同步时删除事件可被追踪。

**删除行为矩阵**：

| 实体类型 | 索引操作 | 物理操作 | 备注 |
|---------|---------|---------|------|
| `agent` / `group` | `softDeleteEntityIndex()` | `fs.rm(entityDir, recursive)` | 删除整个目录 |
| `agent_topic` / `group_topic` | `softDeleteEntityIndex()` | 从父级 `config.topics` 中过滤掉该 Topic | 原子重写父级 config |
| `avatar` | `softDeleteAvatarIndex()` | 无 | 仅标记删除，不删文件 |

**Topic 删除的特殊处理**：

```
if type ∈ {agent_topic, group_topic}:
    if row && row.file_path:
        content = fs.readFile(row.file_path)
        config = JSON.parse(content)
        if Array.isArray(config.topics):
            config.topics = config.topics.filter(t => t.id !== safeId)
            原子写入 row.file_path
```

**物理删除失败不阻断**：无论 `fs.rm` 是否成功，索引层面的软删除已经生效。下次同步时，若物理文件仍存在但索引已删除，系统会通过一致性对账机制发现差异。



## 3. NDJSON 流式消息传输

### 3.1 为什么使用 NDJSON

在 Phase 2 及之前的实现中，消息历史通过标准 JSON 数组传输：服务端将整个 `history.json` 读入内存，序列化为巨型 JSON 字符串后一次性响应。当单 Topic 消息数达到 **10,000+** 时，该模型暴露出严重缺陷：

| 问题 | 传统 JSON 数组 | NDJSON 流式 |
|------|-------------|------------|
| 服务端内存峰值 | 完整 history.json × Topic 数 | 单 Topic 消息数组（流式逐帧输出） |
| 客户端内存峰值 | 完整响应体 | 逐行消费，可边读边写 SQLite |
| 失败影响范围 | 单点失败导致整批重试 | 单 Topic 失败只影响自身 |
| 响应首字节时间（TTFB） | 需等待全部序列化完成 | 首 Topic 就绪后立即输出 |
| 可恢复性 | 无，失败需完整重传 | 客户端可记录已成功的 Topic，断点续传 |

**NDJSON 格式定义**：每行是一个独立的 JSON 对象，行与行之间以 `\n` 分隔。解析器通过逐行读取即可消费，无需等待整体闭合括号。

```ndjson
{"topicId":"topic_a","messages":[{"id":"m1",...},{...}]}
{"topicId":"topic_b","messages":[],"_error":"topic not found"}
{"topicId":"topic_c","messages":[{"id":"m2",...}]}
```

### 3.2 流式下载（downloadMessagesStreamRaw）

`downloadMessagesStreamRaw` 是桌面端向移动端推送消息的核心函数，直接操作 Express 的 `response` 对象进行流式输出。

**HTTP 头设置**：

| 响应头 | 值 | 作用 |
|--------|-----|------|
| `Content-Type` | `application/x-ndjson` | 标识 NDJSON 流格式 |
| `Transfer-Encoding` | `chunked` | 启用分块传输，无需预计算 Content-Length |

**逐 Topic 处理流程**：

```
输入: requests[] = [{ topicId, msgIds[] }], appDataPath, res

res.setHeader("Content-Type", "application/x-ndjson")
res.setHeader("Transfer-Encoding", "chunked")
res.flushHeaders()  // 立即发送头，让客户端尽早开始消费

for each { topicId, msgIds } in requests:
    safeTopicId = sanitizeId(topicId)
    try:
        row = getEntityIndex(safeTopicId, "topic")
        if !row:
            res.write(JSON.stringify({ topicId, messages: [], _error: "topic not found" }) + "\n")
            continue
        
        parentId = path.basename(path.dirname(row.file_path))
        historyPath = appDataPath / UserData / parentId / topics / safeTopicId / history.json
        
        history = 读取 historyPath（失败则视为空数组）
        
        // 过滤：若 msgIds 非空，只返回指定消息；否则返回全部
        idSet = new Set(msgIds)
        filtered = idSet.size == 0 ? history : history.filter(m => idSet.has(m.id))
        
        // 拍平附件结构：将桌面端的 _fileManagerData 提取到顶层
        flattened = filtered.map(msg => {
            if msg.attachments:
                msg.attachments = msg.attachments.map(att => ({
                    type: att.type,
                    name: att.name,
                    size: att.size,
                    hash: att._fileManagerData?.hash || null,
                    extractedText: att._fileManagerData?.extractedText || null,
                    imageFrames: att._fileManagerData?.imageFrames || null,
                    createdAt: att._fileManagerData?.createdAt || null,
                }))
            return msg
        })
        
        res.write(JSON.stringify({ topicId, messages: flattened }) + "\n")
        
    catch e:
        // 单 Topic 失败只写错误帧，不中断流
        res.write(JSON.stringify({ topicId, messages: [], _error: e.message }) + "\n")

res.end()
```

**附件拍平（Flatten）说明**：桌面端在消息中存储附件时，使用 `_fileManagerData` 对象包裹额外元数据（哈希、提取文本、视频帧等）。移动端不需要知晓 `_fileManagerData` 的存在，因此服务端在输出前将其拍平为扁平字段。

### 3.3 移动端消息拉取 DTO（MessagePullSyncDTO）

v0.9.14 引入 `MessagePullSyncDTO` 作为 NDJSON 流中的网络契约类型，隔离桌面端消息格式与移动端内部 `ChatMessage` 实体：

```rust
pub struct MessagePullSyncDTO {
    pub id: String,
    pub role: String,
    pub name: Option<String>,
    pub content: String,
    pub timestamp: u64,
    pub is_thinking: bool,
    pub agent_id: Option<String>,
    pub group_id: Option<String>,
    pub topic_id: String,
    pub is_group_message: bool,
    pub finish_reason: Option<String>,
    pub attachments: Option<Vec<Attachment>>,
    pub content_hash: Option<String>,
    pub avatar_color: Option<String>,
}
```

| 字段 | 说明 |
|------|------|
| `content_hash` | 若桌面端已计算消息指纹，移动端直接复用，跳过本地 fingerprinting |
| `avatar_color` | 头像主色调，用于前端消息列表的彩色边框渲染 |
| `is_thinking` | 保留字段，但移动端已不依赖此值进行渲染决策 |

**转换策略**：`impl From<MessagePullSyncDTO> for ChatMessage` 将 DTO 转换为内部实体时，`blocks` 被显式设为 `None`（注释："同步下载阶段不再执行耗时预渲染，直接设为 None，由 Lazy Render 闭环接管"）。这确保同步下载阶段只做最小必要的解析与入库，预渲染推迟到用户实际浏览时触发。

### 3.4 流式上传（uploadMessagesBatchRaw）

`uploadMessagesBatchRaw` 处理移动端向桌面端推送消息的 NDJSON 流。与下载相反，此函数读取请求流而非写入响应流。

**请求消费模型**：

```
输入: req (Express Request Stream), appDataPath, res

res.setHeader("Content-Type", "application/x-ndjson")
res.setHeader("Transfer-Encoding", "chunked")
res.flushHeaders()

rl = readline.createInterface({ input: req, terminal: false })

for await (const line of rl):
    if line 为空: continue
    
    try:
        { topicId, messages } = JSON.parse(line)
        safeTopicId = sanitizeId(topicId)
        
        row = getEntityIndex(safeTopicId, "topic")
        if !row:
            res.write({ topicId, success: false, error: "topic not found" } + "\n")
            continue
        
        writeIntentLock.add(safeTopicId)
        result = await doUploadSingleTopic(safeTopicId, messages, appDataPath, row)
        res.write(JSON.stringify({ topicId, ...result }) + "\n")
        processedTopicIds.push(safeTopicId)
        
    catch e:
        logger.logOperation("messages", "upload_batch_stream", "line_parse", "error", e.message)

// 所有 Topic 处理完毕后，统一重建索引
for each safeTopicId in processedTopicIds:
    row = getEntityIndex(safeTopicId, "topic")
    if row:
        ingestHistoryToDb(historyPath, safeTopicId, "batch_push")

// 统一触发聚合哈希更新
computeAggregatedHashes(db, logger)

// 释放所有 writeIntentLock
for each tid in processedTopicIds:
    writeIntentLock.delete(tid)

res.end()
```

**为什么索引重建延迟到全部完成后**：`ingestHistoryToDb` 涉及大量 SQLite 写入和指纹计算。若在每个 Topic 处理后立即执行，会严重降低流式吞吐量。批量延迟重建是 CPU 与 I/O 的权衡。

### 3.5 单 Topic 上传逻辑（doUploadSingleTopic）

`doUploadSingleTopic` 被设计为**纯逻辑函数**，不包含幂等性检查、写锁管理和索引重建。这使得批量流和潜在的未来单 Topic 接口可以复用同一套合并逻辑。

**核心合并算法**：

```
输入: safeTopicId, messages[], appDataPath, row

parentId = path.basename(path.dirname(row.file_path))
isGroup = row.file_path.includes("AgentGroups")
historyDir = appDataPath / UserData / parentId / topics / safeTopicId
historyPath = historyDir / history.json

mkdir(historyDir, recursive)

localHistory = 读取 historyPath（失败则为 []）

// 使用 Map 合并：以消息 id 为键，移动端数据覆盖本地数据
msgMap = new Map(localHistory.map(m => [m.id, m]))
neededAttachmentHashes = new Set()

for each msg in messages:
    desktopMsg = reconstructMessage(
        msg, safeTopicId, parentId, isGroup,
        db, neededAttachmentHashes, appDataPath
    )
    msgMap.set(msg.id, desktopMsg)  // 存在则覆盖，不存在则新增

// 按 timestamp 升序排序（桌面端历史按时间正序存储）
finalHistory = Array.from(msgMap.values())
    .sort((a, b) => (a.timestamp || 0) - (b.timestamp || 0))

// 原子写入
原子写入(historyPath, JSON.stringify(finalHistory, null, 2))

return {
    success: true,
    neededAttachmentHashes: Array.from(neededAttachmentHashes)
}
```

**Map 合并的语义**：
- **幂等性保障**：同一消息多次上传，后到达的覆盖先到达的（因为移动端总是持有最新状态）
- **删除兼容性**：桌面端的消息删除通过 `status === "removed"` 标记实现。若移动端上传的消息列表中不包含某条已删除消息，该消息会在合并后保留（因为 `msgMap` 初始包含本地历史）。这是设计上的选择：删除操作应通过独立的 `delete-message` 端点同步，而非消息列表的缺失来推断。

### 3.6 消息格式重建（reconstructMessage）

`reconstructMessage` 是移动端消息结构与桌面端消息结构之间的**适配器（Adapter）**。两者字段不完全一致，需要在写入前完成转换。

**基础字段映射**：

| 移动端字段 | 桌面端字段 | 默认值/处理 |
|-----------|-----------|------------|
| `msg.id` | `desktopMsg.id` | 直接透传 |
| `msg.role` | `desktopMsg.role` | `"user"` |
| `msg.name` | `desktopMsg.name` | `"User"`（user 角色）/ `"Assistant"`（其他） |
| `msg.content` | `desktopMsg.content` | 直接透传 |
| `msg.timestamp` | `desktopMsg.timestamp` | 直接透传 |
| `msg.isThinking` | `desktopMsg.isThinking` | `false`（仅非 user） |
| `msg.finishReason` | `desktopMsg.finishReason` | `"completed"`（仅非 user） |
| `msg.agentId` | `desktopMsg.agentId` | 单聊时默认 parentId |
| `msg.avatarColor` | `desktopMsg.avatarColor` | `"rgb(128, 128, 128)"` |

**群组消息特殊字段**：

| 字段 | 赋值规则 |
|------|---------|
| `isGroupMessage` | `true`（仅当 isGroup 时） |
| `groupId` | `msg.groupId \|\| parentId` |
| `topicId` | `msg.topicId \|\| topicId` |

**附件处理逻辑**：

```
if role == "user" && msg.attachments 非空:
    desktopMsg.attachments = msg.attachments.map(att => {
        if att.hash:
            attRow = db.prepare("SELECT file_path FROM attachment_index WHERE hash = ?").get(att.hash)
            if !attRow:
                neededAttachmentHashes.add(att.hash)  // 桌面端缺失此附件
            desktopPath = attRow ? attRow.file_path : ""
            ext = desktopPath ? path.extname(desktopPath) : getExtensionFromType(att.type)
            return createDesktopAttachment(att, desktopPath, ext)
        else:
            return att  // 无哈希，透传原始结构
    })
```

**头像路径解析**：

对于 Assistant 角色消息，桌面端需要 `avatarUrl` 字段指向本地头像文件：

```
avatarPath = appDataPath / Agents / msgAgentId / avatar.png
desktopMsg.avatarUrl = `file://${avatarPath}`
```

注意：此处代码中存在一个已知特征——无论 `isGroup` 为何值，路径均指向 `Agents` 目录（`isGroup ? "Agents" : "Agents"`）。这是当前版本的实际行为。

**Extra 字段透传**：

```
if msg.extra 是对象:
    Object.assign(desktopMsg, msg.extra)
```

移动端可通过 `extra` 对象传递桌面端特有的扩展字段，实现前向兼容。



## 4. 附件传输

附件（Attachment）是消息同步体系中的独立数据类型，不与消息历史强耦合。附件通过**内容寻址（Content-Addressed）**模型管理：以文件内容的 SHA-256 哈希作为唯一标识，相同内容的附件全局只存一份。

### 4.1 附件上传（uploadAttachment）

**路径生成规则**：

```
attachmentsDir = appDataPath / UserData / attachments
filePath = attachmentsDir / {hash}{ext}
```

其中 `ext` 通过 MIME 类型推导：`getExtensionFromType(type)`。例如：

| MIME 类型 | 哈希（前 16 位） | 最终文件名 |
|-----------|-----------------|-----------|
| `image/png` | `a1b2c3d4e5f6...` | `a1b2c3d4e5f6...7890.png` |
| `application/pdf` | `b2c3d4e5f6a7...` | `b2c3d4e5f6a7...8901.pdf` |
| `video/mp4` | `c3d4e5f6a7b8...` | `c3d4e5f6a7b8...9012.mp4` |

**上传流程**：

```
输入: { hash, data (Buffer), name, type, appDataPath }

attachmentsDir = path.join(appDataPath, "UserData", "attachments")
await fs.mkdir(attachmentsDir, { recursive: true })

ext = getExtensionFromType(type)
filePath = path.join(attachmentsDir, `${hash}${ext}`)

await fs.writeFile(filePath, data)
upsertAttachmentIndex(hash, filePath)

return { success: true }
```

**HTTP 端点约束**（`transport/routes.js`）：

| 属性 | 值 |
|------|-----|
| 方法 | `POST /api/mobile-sync/upload-attachment` |
| Body 解析 | `express.raw({ type: "*/*", limit: "100mb" })` |
| 参数传递 | Query String：`?hash=...&name=...&type=...` |
| 大小限制 | 100 MB |

### 4.2 附件下载（downloadAttachment）

附件下载为反向流程：移动端提供哈希，桌面端查询 `attachment_index` 返回物理路径，再通过 `res.sendFile()` 发送。

```
输入: hash

row = db.prepare("SELECT file_path FROM attachment_index WHERE hash = ?").get(hash)
if !row: return null

return { filePath: row.file_path }
```

**HTTP 端点**：

| 属性 | 值 |
|------|-----|
| 方法 | `GET /api/mobile-sync/download-attachment?hash=...` |
| 响应方式 | `res.sendFile(result.filePath)` |

### 4.3 头像上传与下载的特殊处理

头像虽然本质也是二进制文件，但在业务上被视为**实体元数据的一部分**，因此拥有独立的上传/下载端点和索引表。

**头像存储位置**：

| 所有者类型 | 目录 |
|-----------|------|
| Agent | `Agents/{id}/avatar.png` |
| Group | `AgentGroups/{id}/avatar.png` |

**头像上传特殊逻辑**（`uploadAvatar`）：

```
输入: { id, type, data, appDataPath }

safeId = sanitizeId(id)
isGroup = type === "group"
baseDirName = isGroup ? "AgentGroups" : "Agents"
entityDir = appDataPath / baseDirName / safeId
avatarPath = entityDir / avatar.png

await fs.mkdir(entityDir, { recursive: true })
await fs.writeFile(avatarPath, data)

hash = computeBinaryHash(data)
upsertAvatarIndex(safeId, type, avatarPath, hash)

// Group 头像需额外更新 config.json 的 avatar 字段
if isGroup:
    configPath = entityDir / config.json
    config = JSON.parse(await fs.readFile(configPath))
    config.avatar = "avatar.png"
    原子写入(configPath)
    
    // 更新 Group 实体索引
    groupHash = computeDtoHash(extractGroupDTO(config), GROUP_SYNC_FIELDS)
    upsertEntityIndex(safeId, "group", configPath, groupHash)
```

**关键差异**：Agent 头像无需更新 `config.json`（桌面端直接按固定路径 `avatar.png` 读取），但 Group 头像需要在配置中显式记录 `avatar` 字段。

### 4.4 neededAttachmentHashes 机制

在消息上传流程中，`reconstructMessage` 会生成一个 `neededAttachmentHashes` 集合。这是桌面端实现**懒附件同步**的核心机制。

**触发条件**：当移动端消息包含附件哈希，但桌面端 `attachment_index` 中查无此哈希时，该哈希被加入 `neededAttachmentHashes`。

**响应回传**：

```
// uploadMessagesBatchRaw 的单 Topic 响应帧
{
    "topicId": "topic_a",
    "success": true,
    "neededAttachmentHashes": ["a1b2c3d4...", "b2c3d4e5..."]
}
```

**移动端后续行为**：
1. 解析所有 Topic 的响应帧
2. 合并 `neededAttachmentHashes` 去重
3. 对缺失哈希逐个调用 `uploadAttachment` 推送二进制
4. 附件补齐后，可再次推送消息（或依赖桌面端在渲染时懒加载）

**设计意图**：消息历史同步与附件传输解耦。在弱网环境下，优先保证文本消息的快速同步，附件可在后台异步补齐。



## 5. 原子写入实现细节

桌面端使用文件系统作为持久化存储，而非数据库。在并发写入场景下（同步推送 + 文件监听器 + 用户操作），直接覆盖写入可能导致文件内容为空或半写入状态。VCPMobile 采用三重保护机制确保写入原子性。

### 5.1 tmp → rename 模式

所有涉及 `config.json` 和 `history.json` 的写入操作均遵循 **"先写临时文件，再原子重命名"** 模式。

**伪代码**：

```
function atomicWrite(filePath, content):
    tmpPath = `${filePath}.tmp_${Date.now()}_${Math.random().toString(36).slice(2, 7)}`
    
    // 步骤 1: 写入临时文件
    await fs.writeFile(tmpPath, content, "utf-8")
    
    // 步骤 2: 原子重命名（POSIX/Windows 均保证此操作为原子）
    await fs.rename(tmpPath, filePath)
```

**为什么这样是安全的**：

| 阶段 | 崩溃影响 |
|------|---------|
| `writeFile` 过程中崩溃 | 临时文件可能不完整，原文件完全未受影响 |
| `writeFile` 完成 → `rename` 前崩溃 | 临时文件完整，原文件未受影响，下次启动可清理 |
| `rename` 过程中崩溃 | 操作系统保证 `rename` 为原子操作，不会出现半状态 |
| `rename` 完成后崩溃 | 写入已成功，数据完整 |

**临时文件命名规则**：`.tmp_{时间戳}_{5位随机字符}`。随机字符使用 `Math.random().toString(36).slice(2, 7)` 生成，避免同一毫秒内并发冲突。

### 5.2 writeIntentLock 防止文件监听器冲突

桌面端运行着**文件系统监听器（File System Watcher）**，当 `config.json` 或 `history.json` 被修改时，监听器会触发重新加载和索引重建。若监听器在同步写入的半中间状态读取文件，可能摄入不一致数据。

**writeIntentLock 机制**：

```
// 全局锁集合
const writeIntentLock = new Set()

function isWriteLocked(id):
    return writeIntentLock.has(sanitizeId(id))
```

**使用场景**：

| 函数 | 加锁时机 | 释放时机 |
|------|---------|---------|
| `uploadEntity` | 写入前 `writeIntentLock.add(safeId)` | `finally` 块中 `setTimeout(() => delete, 1000)` |
| `uploadMessagesBatchRaw` | 每个 Topic 处理前 `writeIntentLock.add(safeTopicId)` | 全部 Topic 完成后统一删除 |

**为什么需要 1 秒延迟释放**：文件系统的 `mtime`（修改时间）精度与监听器的触发存在竞态窗口。延迟 1 秒确保监听器在锁释放后才真正开始摄入，避免读取到 `rename` 之前的缓存状态。

**监听器侧的配合**（此文档范围外，但需知晓）：监听器在检测到文件变更后，应先检查 `isWriteLocked(entityId)`，若锁定则跳过本次摄入，等待下次事件或轮询。

### 5.3 损坏配置保护

`uploadEntity` 在读取现有配置时，对损坏文件实施**拒绝写入**策略，防止数据丢失扩散。

**损坏判定条件**：

| 条件 | 处理 |
|------|------|
| `content.trim() === ""` | 视为空文件，抛出异常 |
| `JSON.parse` 失败（非 ENOENT） | 视为损坏 JSON，抛出异常 |
| 解析结果为数组 | 抛出异常（配置必须是对象） |
| 解析结果为 `null` / 非对象 | 抛出异常 |

**行为差异**：

```
读取异常:
  ├─ code === "ENOENT"（文件不存在）
  │   → fileReadSuccess = false
  │   → 允许新建配置（新建 Agent/Group 或重建 Topic 父级）
  │
  └─ 其他异常（损坏/空/无效）
      → 抛出异常
      → 返回 { success: false, error: "Cannot upload to corrupted config: ..." }
      → 日志记录完整错误信息
```

此策略的核心假设是：**如果本地配置已损坏，同步不应以覆盖方式修复，而应由用户手动处理或从备份恢复**。自动覆盖可能将移动端的旧数据误写入损坏配置，导致双端数据同时丢失。



## 6. 批量操作的 I/O 优化

VCPMobile 同步模块在批量场景下实施了多层 I/O 优化，核心思想是**减少磁盘读取次数**和**减少文件写入次数**。这些优化对 SSD 寿命和同步速度均有显著影响。

### 6.1 按 file_path 分组减少磁盘读取

**应用位置**：`downloadEntities`（`sync/entity.js`）

**问题描述**：假设移动端请求下载 20 个 Topic，这 20 个 Topic 分属于 3 个 Agent。如果不分组，每个 Topic 都需要独立读取其父级的 `config.json`，共 20 次磁盘 I/O。

**优化方案**：

```
// 预分组阶段
fileGroups = new Map()

for each request in requests:
    row = getEntityIndex(safeId, type)
    if !fileGroups.has(row.file_path):
        fileGroups.set(row.file_path, { isGroup, reqs: [] })
    fileGroups.get(row.file_path).reqs.push({ req, safeId })

// 执行阶段：每个 file_path 只读一次
for each [filePath, {isGroup, reqs}] of fileGroups:
    content = await fs.readFile(filePath, "utf-8")  // 1 次读取
    config = JSON.parse(content)
    
    for each {req, safeId} in reqs:
        // 在已加载的 config 对象上直接提取 DTO
        dto = extractXXXTDTO(...)
```

**效果量化**：

| 场景 | Topic 数 | 涉及 Agent 数 | 未优化读取 | 优化后读取 | 减少比例 |
|------|---------|--------------|-----------|-----------|---------|
| 单个 Agent 多 Topic | 15 | 1 | 15 | 1 | 93.3% |
| 分散多 Agent | 30 | 10 | 30 | 10 | 66.7% |
| 极端集中 | 50 | 2 | 50 | 2 | 96.0% |

### 6.2 按 configPath 分组减少写入次数

**应用位置**：`uploadEntitiesBatch`（`sync/entity.js`）

**问题描述**：若移动端推送 30 个 Topic 更新，逐一调用 `uploadEntity` 会导致每个 Topic 触发一次父级 `config.json` 的读取-修改-写入，共 30 次写入（若这些 Topic 属于同一 Agent，则同一文件被写 30 次）。

**优化方案**：

```
// 归口阶段
fileGroups = Map<configPath, { items[] }>

for each item in items:
    // 解析该 item 应写入的 configPath
    configPath = resolveConfigPath(item)
    fileGroups.getOrCreate(configPath).items.push(item)

// 执行阶段：每个 configPath 只写一次
for each [configPath, group] of fileGroups:
    config = 读取 configPath（或 {}）
    
    for each item in group.items:
        // 在内存中依次应用所有变更
        config = handleTopicUpload(config, item)
    
    // 最终一次原子写入
    原子写入(configPath, JSON.stringify(config, null, 2))
    
    // 批量更新索引
    for each item in group.items:
        updateTopicIndex(...)
```

**效果量化**：

| 场景 | Topic 更新数 | 涉及 config 文件数 | 未优化写入 | 优化后写入 | 减少比例 |
|------|-------------|-------------------|-----------|-----------|---------|
| 同一 Agent 批量更新 | 20 | 1 | 20 | 1 | 95.0% |
| 同一 Group 批量更新 | 20 | 1 | 20 | 1 | 95.0% |
| 分散 5 个父级 | 25 | 5 | 25 | 5 | 0%（已最优） |

### 6.3 消息历史的流式 I/O 优势

虽然消息历史的 NDJSON 流式传输主要解决内存问题，但在 I/O 层面同样带来优化：

| 优化维度 | 传统 JSON | NDJSON 流式 |
|---------|----------|------------|
| 响应序列化 | 需将所有 Topic 数据构建为完整 JSON 树后序列化 | 每个 Topic 独立 `JSON.stringify` + `res.write` |
| 网络发送 | 等待全部数据就绪后才发送首字节 | 首 Topic 就绪后立即发送，利用 TCP 滑动窗口 |
| 客户端解析 | `JSON.parse` 一次性解析，CPU 峰值高 | 逐行 `JSON.parse`，CPU 负载平滑 |
| 磁盘读取 | 服务端可能并行读取多个 history.json | 顺序读取，磁盘队列深度可控 |

### 6.4 索引批量更新策略

**实体索引**：`uploadEntitiesBatch` 在每个 `configPath` 写入成功后，遍历该组所有 item 批量调用 `updateTopicIndex` 或 `upsertEntityIndex`。这避免了逐条提交 SQLite 事务的开销。

**消息索引**：`uploadMessagesBatchRaw` 将所有 Topic 的索引重建延迟到流结束后统一执行。虽然牺牲了一定的实时性，但将 `ingestHistoryToDb` 的调用次数从 N 次降至 1 次（外加一次 `computeAggregatedHashes`）。

**权衡说明**：

| 策略 | 优点 | 缺点 |
|------|------|------|
| 逐条更新索引 | 实时性强，崩溃后已同步数据立即可查 | 事务开销大，吞吐量低 |
| 批量延迟更新 | 吞吐量高，磁盘 I/O 少 | 若在批量更新前崩溃，已写入文件的数据未被索引 |

当前版本选择批量延迟更新，因为：
1. 文件已原子写入，数据不丢失
2. 下次启动时可通过一致性对账重建缺失索引
3. 同步吞吐量的优先级高于索引实时性
