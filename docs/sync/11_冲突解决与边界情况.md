# 冲突解决与边界情况

## 1. 引言

在分布式双端同步架构中，冲突（Conflict）是不可避免的物理现实。移动端与桌面端各自维护独立的 SQLite / JSON 存储，任何一方的网络延迟、用户操作或后台任务都可能在同一时刻修改同一实体。VCP Mobile 的同步协议采用**最终一致性（Eventual Consistency）**模型，通过三阶段流水线（Phase 1 Owner Metadata → Phase 2 Topic Metadata → Phase 3 Messages）将冲突检测与解决限制在可控的粒度内。

冲突解决的核心策略是 **LWW（Last Write Wins，后写胜出）**，辅以**软删除（Soft Delete）双向传播**、**幂等性机制（Idempotency）**和**双哈希比对（Dual-Hash Validation）**作为保险层。本文档将逐层拆解这些机制在代码中的具体实现、决策矩阵与边界行为。

### 1.1 冲突的类型与检测粒度

VCP Mobile 的同步冲突按检测粒度分为三层：

| 层级 | 检测对象 | 冲突判定依据 | 仲裁位置 |
|------|---------|-------------|---------|
| 配置层（Config） | Agent / Group / Topic 元数据 | `config_hash` + `updated_at` | 桌面端 `manifest.js` |
| 内容层（Content） | Topic 下的消息集合 | `content_hash`（聚合哈希） | 桌面端 `diff.js` |
| 资源层（Asset） | 附件二进制 | `attachment.hash` | 按需拉取 / 推式上传 |

配置层冲突由 Phase 1/2 的 Manifest Diff 解决；内容层冲突由 Phase 3 的 Message Diff Batch 解决；资源层不执行主动冲突仲裁，而是"存在即正确"——附件以首次上传的哈希为准，后续同名文件通过哈希去重避免重复存储。

（`src-tauri/src/vcp_modules/sync_types.rs`、`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js`）

## 2. LWW（Last Write Wins）策略

### 2.1 时间戳比较的完整规则

差异比对的核心逻辑位于桌面端插件的 `handleSyncManifest` 函数（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）。当移动端发送 `SYNC_MANIFEST` 后，桌面端对每一条实体执行以下比较链：

```
1. 若 remote.deletedAt 存在：
     → 本地未删除或无记录： action = DELETE
     → 本地已删除：跳过
2. 若本地无记录：
     → action = PUSH（本地推给远端）
3. 若本地已删除但远端未删除：
     → action = PUSH_DELETE（通知远端本地已删）
4. 否则（双方均存在且未删除）：
     a. 比较 configHash：
        - remote.ts > local.ts  → action = PUSH
        - remote.ts ≤ local.ts  → action = PULL
     b. 若 dataType 为 agent/group，额外比较 contentHash：
        - 不匹配则标记 mismatchedContent = true
```

**关键点**：时间戳比较是单向的 `>` 而非 `>=`。当 `remote.ts == local.ts` 时，走 `else` 分支，决策为 **PULL**（远端拉取本地版本）。这意味着在极端时间戳碰撞场景下，本地版本拥有默认优先权。

（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）

### 2.2 deleted_at 参与比较的特殊处理

`deletedAt` 字段在比对链中拥有最高优先级。只要 `remote.deletedAt` 不为空，无论时间戳或哈希如何，一律触发 `DELETE` 动作。这是为了确保删除操作的传播不会被后续修改覆盖。

| 条件 | 桌面端决策 | 移动端行为 |
|------|-----------|-----------|
| `remote.deletedAt` 存在，本地无记录或本地未删除 | `DELETE` | `DeleteExecutor::soft_delete_xxx` |
| `local.deletedAt` 存在，远端未删除 | `PUSH_DELETE` | 软删除本地 + 发送 `SYNC_ENTITY_DELETE` WS 通知 |
| 双方 `deletedAt` 均存在 | 跳过 | 无操作 |

软删除的实现采用**时间戳标记**而非物理删除：
- 移动端：`UPDATE agents SET deleted_at = ? WHERE agent_id = ?`（`src-tauri/src/vcp_modules/sync_executor/delete_executor.rs`）
- 桌面端：`softDeleteEntityIndex(safeId, type, deletedAt)`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`）

### 2.3 判定矩阵

以下矩阵以**远端状态**为行、**本地状态**为列，展示 `handleSyncManifest` 的完整决策：

| 远端＼本地 | 不存在 | 存在且未删除 | 存在且已删除 |
|-----------|--------|-------------|-------------|
| **不存在** | — | `PULL`（若本地已删则 `PUSH_DELETE`） | `PUSH_DELETE` |
| **存在且未删除** | `PUSH` | 比较 `ts` 与 `hash`：<br>`remote.ts > local.ts` → `PUSH`<br>`remote.ts ≤ local.ts` → `PULL` | `PUSH_DELETE` |
| **存在且已删除** | `DELETE` | `DELETE` | 跳过（双方一致） |

**补充说明**：
- `PUSH` 表示"移动端应向桌面端推送该实体"。
- `PULL` 表示"移动端应从桌面端拉取该实体"。
- `DELETE` 表示"移动端应软删除该实体"。
- `PUSH_DELETE` 表示"移动端应通知桌面端该实体已在本地删除"。

（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）

### 2.4 V2 双哈希的演进动机

早期 V1 协议仅使用单哈希（`hash`）表征实体状态，导致以下问题：
- 桌面端修改 Topic 标题（配置变更）会改变整棵树的哈希，迫使移动端在 Phase 3 拉取全部消息；
- 移动端接收新消息（内容变更）会反向迫使桌面端重新比对配置。

V2 引入 `config_hash` 与 `content_hash` 分离后，配置修改与消息修改互不干扰。`manifest.js` 中的比对逻辑也相应扩展：

```javascript
const remoteConfig = remote.configHash || remote.hash;
const remoteContent = remote.contentHash || "";
const localConfig = local.configHash || local.hash;
const localContent = local.contentHash || "";
```

只有当 `localConfig !== remoteConfig` 时，才进入 LWW 时间戳仲裁；`contentHash` 的差异仅标记 `mismatchedContent`，用于引导 Phase 2.5 的 Topic Validation，而非直接触发 Pull/Push。

（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`）

## 3. 时间戳碰撞的 Hash 仲裁

### 3.1 什么情况下会碰撞

时间戳碰撞（Timestamp Collision）发生在以下场景：

1. **同分钟内重复操作**：用户在桌面端和移动端几乎同时修改同一 Agent 配置，双方 `updated_at` 均落在同一毫秒或同一分钟内。
2. **离线后同步**：设备 A 离线期间修改实体，设备 B 同时在线修改，重新连接时双方时间戳恰好相同。
3. **NTP 回拨或时钟漂移**：移动端系统时钟异常导致 `updated_at` 与桌面端完全重合。

### 3.2 当前实现行为

在 `handleSyncManifest` 中，时间戳比较代码为：

```javascript
if (remote.ts > local.ts) {
    action = PUSH;
} else {
    action = PULL;
}
```

**当 `remote.ts == local.ts` 时，决策固定为 `PULL`。** 这意味着移动端版本在平局时拥有隐性优先权。系统**未采用显式的 Hash 字典序仲裁**，而是依赖以下两层机制避免真正的内容冲突。

### 3.3 双哈希作为间接仲裁（V2）

V2 协议引入 **configHash + contentHash** 的双哈希结构，将"配置元数据"与"内容数据"解耦：

| 层级 | 移动端字段 | 桌面端字段 | 作用 |
|------|-----------|-----------|------|
| 配置层 | `config_hash` | `hash` | Agent/Group/Topic 的 DTO 序列化指纹 |
| 内容层 | `content_hash` | `aggregated_hash` | 消息聚合指纹（MerkleRoot 风格） |

在 Phase 2.5（Topic Validation）中，桌面端对双哈希执行严格相等判断：

```javascript
if (localConfig === remoteConfig && localContent === remoteContent) {
    // 完全一致，跳过
} else {
    changedTopics.push(topicId);
}
```

（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js`）

**一致性保障**：聚合哈希的计算规则为"将所有子哈希按 ID 字典序排列后连接，再计算总 Hash"（`src-tauri/src/vcp_modules/sync_types.rs`）。该规则在双端共享，确保同一数据集必然产生同一指纹，从而避免"双方都认为自己是新版本"的死锁。

## 4. 软删除双向传播

### 4.1 DELETE vs PUSH_DELETE

`DELETE` 与 `PUSH_DELETE` 是两个截然不同的语义动作：

| 动作 | 发起端 | 接收端行为 | 通信方式 |
|------|--------|-----------|---------|
| `DELETE` | 桌面端（manifest diff） | 移动端软删除对应实体 | 通过 `SYNC_DIFF_RESULTS` 返回，移动端异步执行 |
| `PUSH_DELETE` | 桌面端（manifest diff） | 移动端软删除 + 通过 WS 发送 `SYNC_ENTITY_DELETE` 通知桌面端 | 移动端先执行删除，再通知远端 |
| `SYNC_ENTITY_DELETE` | 移动端（用户主动删除） | 桌面端通过 WS 接收，软删除本地索引 | 实时 WebSocket |

在 `sync_service.rs` 中，对 `DELETE` 与 `PUSH_DELETE` 的处理逻辑如下：

```rust
if action == "DELETE" || action == "PUSH_DELETE" {
    // 执行软删除
    match data_type {
        Agent => DeleteExecutor::soft_delete_agent(...),
        Group => DeleteExecutor::soft_delete_group(...),
        Topic => DeleteExecutor::soft_delete_topic(...),
        Avatar => DeleteExecutor::soft_delete_avatar(...),
    }
    if action == "PUSH_DELETE" {
        tx.send(SyncCommand::NotifyDelete { data_type, id });
    }
}
```

（`src-tauri/src/vcp_modules/sync_service.rs`）

### 4.2 deleted_at 字段在 Diff 中的角色

`deleted_at` 在 Manifest 条目中的传递路径：

```
移动端 DB (agents.deleted_at)
  → Phase1Metadata::build_agent_manifest (SELECT 查询)
  → SYNC_MANIFEST WS 消息 (remoteItems)
  → handleSyncManifest (remote.deletedAt)
  → SYNC_DIFF_RESULTS (action: DELETE/PUSH_DELETE)
  → DeleteExecutor::soft_delete_agent (UPDATE deleted_at = now)
```

**设计意图**：`deleted_at` 不参与 LWW 时间戳竞争，而是作为独立的状态位。一旦存在，即触发删除传播链，防止"删除后被旧版本覆盖"的幽灵数据问题。

### 4.3 删除后的恢复机制

当前协议**未实现自动恢复**。被软删除的实体保留在数据库中，仅通过 `deleted_at IS NOT NULL` 过滤。恢复需要：

1. 用户在源端重新创建同名实体（生成新 ID）；
2. 或通过 `DeleteExecutor::cleanup_old_deleted_records` 定期清理（默认按天数阈值物理删除）。

清理函数签名：
```rust
pub async fn cleanup_old_deleted_records(app, days: i64)
```

（`src-tauri/src/vcp_modules/sync_executor/delete_executor.rs`）

### 4.4 HashAggregator 冒泡机制

软删除不仅影响被删实体本身，还会通过 HashAggregator 向上冒泡（Bubble），影响父级实体的 `content_hash`：

```
soft_delete_topic(T1)
  → bubble_topic_hash(T1)     // 重新计算 Topic 的 content_hash
  → bubble_agent_hash(A1)     // 重新计算所属 Agent 的 content_hash
  → bubble_group_hash(G1)     // 或所属 Group 的 content_hash
```

冒泡的触发点在 `delete_executor.rs` 中显式调用：

```rust
sqlx::query("UPDATE topics SET deleted_at = ? WHERE topic_id = ?")
    .bind(now).bind(topic_id).execute(&db.pool).await?;

if let Some(row) = parent_row {
    let mut tx = db.pool.begin().await?;
    if owner_type == "agent" {
        HashAggregator::bubble_agent_hash(&mut tx, &owner_id).await?;
    } else if owner_type == "group" {
        HashAggregator::bubble_group_hash(&mut tx, &owner_id).await?;
    }
    tx.commit().await?;
}
```

**设计意图**：删除操作必须立即反映在父级哈希中，否则 Phase 2.5 的双哈希比对将无法感知 Topic 数量的变化，导致下一轮同步仍然认为双方一致。

（`src-tauri/src/vcp_modules/sync_executor/delete_executor.rs`、`src-tauri/src/vcp_modules/sync_hash.rs`）

## 5. 幂等性机制

### 5.1 幂等 Key 生成公式

移动端在上传实体（Agent / Group）时，通过 `PushExecutor` 生成幂等键：

```
idempotency_key = SHA256(action + entity_type + id + minute_timestamp)
```

Rust 实现：

```rust
fn generate_idempotency_key(action: &str, entity_type: &str, id: &str) -> String {
    let mut hasher = Sha256::new();
    hasher.update(action.as_bytes());
    hasher.update(entity_type.as_bytes());
    hasher.update(id.as_bytes());
    let now = chrono::Utc::now().timestamp() / 60; // 按分钟截断
    hasher.update(now.to_string().as_bytes());
    format!("{:x}", hasher.finalize())
}
```

（`src-tauri/src/vcp_modules/sync_executor/push_executor.rs`）

**按分钟截断（minute_timestamp）的设计**：允许同一分钟内因网络重试导致的重复请求通过幂等检查，但跨分钟的修改会被视为新操作。

### 5.2 5 分钟 TTL 策略

桌面端 `core/idempotency.js` 维护内存中的操作缓存：

```javascript
const recentOperations = new Map();
const IDEMPOTENCY_TTL = 300000; // 5 分钟
```

检查流程：
1. 收到 `/upload-entity` 请求，提取 `x-idempotency-key`；
2. 调用 `checkIdempotency(opId)`；
3. 若存在且未过期（< 5 分钟），直接返回上次结果，不执行写入；
4. 若不存在或已过期，执行 `uploadEntity`，成功后 `recordOperation(opId, result)`。

（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/idempotency.js`）

### 5.3 在 HTTP /upload-entity 上的应用

`PushExecutor::push_agent` 的 HTTP 请求构造：

```
POST /api/mobile-sync/upload-entity
Headers:
  x-sync-token: <token>
  Authorization: Bearer <token>
  x-idempotency-key: <sha256_key>
Body:
  { id, type: "agent", data: AgentSyncDTO }
```

**应用范围**：幂等性目前仅覆盖单实体上传接口（`/upload-entity`）。批量接口（`/upload-entities-batch`、 `/upload-messages-batch`）未携带幂等键，因为批量操作的重复检测成本较高，且底层 NDJSON 流式传输天然降低了重试概率。

### 5.4 清理任务与内存泄漏防护

桌面端幂等缓存基于内存 `Map`，存在两项防护机制：

1. **TTL 过期自动清理**：每 60 秒执行一次扫描，删除超过 5 分钟的记录；
2. **进程重启清空**：缓存仅存于 Node.js 堆内存，插件重启后全部失效。

潜在风险：若用户在 5 分钟窗口内连续执行超过缓存容量的操作（理论上无上限，但内存受限），可能导致 RSS 增长。实际场景中，单次同步的 PUSH 操作数通常 < 1000，内存开销可忽略。

（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/idempotency.js`）

## 6. 具象同步场景

### 场景 1：新建 Agent（首次同步）

**初始状态**：
- 桌面端：已存在 Agent `A1`，`updated_at = T1`，`config_hash = H1`
- 移动端：空数据库（首次安装）

**操作序列**：
1. 用户点击"开始同步"。
2. 移动端 `Phase1Metadata::build_phase1_manifests` 查询本地 agents，返回空数组。
3. 移动端发送 `SYNC_MANIFEST`（`dataType: agent`，`items: []`）。
4. 桌面端 `handleSyncManifest`：远端无记录，本地存在 `A1`。
5. 桌面端返回 `SYNC_DIFF_RESULTS`：`[{ id: "A1", action: "PULL" }]`。

**Diff 结果**：

| 实体 | 动作 | 理由 |
|------|------|------|
| A1 | PULL | 移动端无记录，需从桌面端拉取 |

**最终状态**：移动端通过 `PullExecutor::pull_agent` 下载 DTO，经 `DbWriteQueue` 写入 SQLite，并触发 Hash 初始化。

**伪代码映射**：

```rust
// 移动端：构建空 Manifest
let manifests = Phase1Metadata::build_phase1_manifests(&db.pool).await?;
// → [{ data_type: Agent, items: [] }]

// 桌面端：handleSyncManifest
const localItems = getLocalManifest("agent", null);
// → [{ id: "A1", hash: "H1", ts: T1 }]
for (const local of localItems) {
    if (!remoteItems.find(r => r.id === local.id)) {
        results.push({ id: "A1", action: "PULL" });
    }
}

// 移动端：执行 PULL
PullExecutor::pull_agent(&app, &client, &url, &token, "A1", &write_queue).await?;
```

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_pipeline/phase1_metadata.rs`
- `src-tauri/src/vcp_modules/sync_executor/pull_executor.rs`
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`

### 场景 2：双向修改冲突

**初始状态**：
- 桌面端 Agent `A1`：`updated_at = 1000`，`config_hash = H_old`
- 移动端 Agent `A1`：`updated_at = 2000`，`config_hash = H_new`

**操作序列**：
1. 用户在桌面端修改 `A1`（`updated_at = 1000`）。
2. 用户在移动端修改同一 Agent（`updated_at = 2000`）。
3. 同步启动，移动端发送 Manifest。
4. 桌面端比对：`remote.ts = 2000 > local.ts = 1000`。

**Diff 结果**：

| 实体 | 动作 | 理由 |
|------|------|------|
| A1 | PUSH | 移动端时间戳更新，桌面端应接受移动端版本 |

**最终状态**：桌面端 `uploadEntity` 接收移动端 DTO，通过 `applyAgentDTO` 局部覆盖配置，保留桌面端特有字段（如 `topics` 数组）。移动端无需拉取。

**最终状态细节**：
桌面端 `entity.js` 中的 `handleAgentUpload` 执行局部合并：

```javascript
if (!fileReadSuccess) {
    config = createAgentConfig(id, data);  // 新建
} else {
    applyAgentDTO(config, data);           // 局部覆盖，保留 topics
}
```

**关键约束**：`applyAgentDTO` 明确不触碰 `config.topics` 数组，Topic 同步由独立 Phase 2 负责，防止配置同步与消息同步交叉污染。

**涉及代码**：
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`
- `src-tauri/src/vcp_modules/sync_executor/push_executor.rs`

### 场景 3：时间戳碰撞仲裁

**初始状态**：
- 桌面端 Agent `A1`：`updated_at = 1000`，`config_hash = H_d`
- 移动端 Agent `A1`：`updated_at = 1000`，`config_hash = H_m`（`H_m ≠ H_d`）

**操作序列**：
1. 双端几乎同时修改，时间戳均为 `1000`。
2. 同步启动，桌面端比对：`remote.ts = 1000`，`local.ts = 1000`。
3. 进入 `else` 分支。

**Diff 结果**：

| 实体 | 动作 | 理由 |
|------|------|------|
| A1 | PULL | `remote.ts ≤ local.ts`，默认本地（桌面端）胜出 |

**最终状态**：移动端拉取桌面端版本，`H_d` 覆盖 `H_m`。用户在一次同步中"丢失"了移动端的修改。这是 LWW 单向比较的已知限制，系统通过以下方式降低影响：
- 同步触发频率较高（连接即同步），减少离线碰撞窗口；
- V2 双哈希在 Phase 2.5 进一步校验内容一致性。

**风险说明**：
该场景是 LWW 策略的**已知缺陷**。若用户仅在移动端修改且时间戳恰好与桌面端相等，修改将被静默覆盖。缓解措施包括：
- 移动端在写入数据库时强制 `updated_at = max(old_ts + 1, now)`，确保单调递增；
- 高频同步减少离线编辑窗口。

**涉及代码**：
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`

### 场景 4：删除传播

**初始状态**：
- 桌面端：Agent `A1` 已软删除，`deleted_at = 5000`
- 移动端：Agent `A1` 仍存在，`deleted_at = null`

**操作序列**：
1. 用户在桌面端删除 `A1`。
2. 移动端尚未同步。
3. 同步启动，移动端发送 Manifest（含 `A1`，`deletedAt: null`）。
4. 桌面端比对：`remote.deletedAt = null`，但 `local.deletedAt = 5000`。
5. 桌面端返回 `PUSH_DELETE`（因为本地已删除而远端未删除）。

**Diff 结果**：

| 实体 | 动作 | 理由 |
|------|------|------|
| A1 | PUSH_DELETE | 本地已删除，需通知移动端 |

**最终状态**：
1. 移动端执行 `DeleteExecutor::soft_delete_agent(A1)`，设置 `deleted_at = now`；
2. 移动端发送 `SYNC_ENTITY_DELETE` WS 消息回桌面端（冗余确认）；
3. 桌面端忽略重复删除通知。

**消息传播细节**：
`SYNC_ENTITY_DELETE` 的 WS 发送逻辑位于 `sync_service.rs`：

```rust
SyncCommand::NotifyDelete { data_type, id } => {
    let msg = json!({
        "type": "SYNC_ENTITY_DELETE",
        "id": id,
        "dataType": data_type
    });
    let _ = ws_stream.send(Message::Text(msg.to_string().into())).await;
}
```

桌面端接收后同样执行软删除索引更新，但**不删除物理文件**（Agent/Group 目录在 `deleteEntity` 中通过 `fs.rm` 移除，Topic 仅从父级 `config.json` 的 `topics` 数组中移除）。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_service.rs`
- `src-tauri/src/vcp_modules/sync_executor/delete_executor.rs`
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/manifest.js`

### 场景 5：带附件的消息同步

**初始状态**：
- Topic `T1` 在桌面端有消息 `M1`，含附件 `att.jpg`（`hash = H_att`）
- 移动端 `T1` 无 `M1`

**操作序列**：
1. Phase 3 启动，移动端发送 `SYNC_MESSAGE_DIFF_BATCH`，包含 `T1` 的本地消息哈希映射。
2. 桌面端 `handleSyncMessageDiffBatch` 比对：发现 `M1` 在移动端不存在。
3. 桌面端返回 `{ T1: { toPull: ["M1"], toPush: false } }`。

**Diff 结果**：

| Topic | toPull | toPush | 理由 |
|-------|--------|--------|------|
| T1 | [M1] | false | 移动端缺少 M1 |

**最终状态**：
1. 移动端 `PullExecutor::pull_messages_batch` 流式拉取 `M1`；
2. `process_topic_messages` 解析消息时，发现附件 `hash = H_att`；
3. 批量查询 `attachments` 表，发现本地无 `H_att`；
4. 消息写入 `messages` 表，附件路径标记为默认占位符 `file://attachments/H_att`；
5. 后续用户点击附件时，触发按需下载；或在同步周期中由独立附件同步任务补齐。

**附件路径的占位策略**：
`process_topic_messages` 在解析每条消息的附件时，执行批量路径查询：

```rust
let query = format!(
    "SELECT hash, internal_path FROM attachments WHERE hash IN ({})",
    placeholders
);
```

若本地缺失该附件，则填充默认占位符：
- `internalPath` = `"attachments/{hash}"`
- `src` = `"file://attachments/{hash}"`
- `status` = `"ready"`（乐观标记，实际尚未下载）

这种"先写消息、后补附件"的策略确保消息同步不被大文件阻塞，同时前端可通过 `status` 字段展示加载占位图。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_executor/pull_executor.rs`
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js`

## 7. 边界情况清单

### 7.1 空数据库首次连接

**现象**：移动端全新安装，首次连接桌面端。

**行为**：
- Phase 1：所有 Agent / Group / Avatar 均走 `PULL`；
- Phase 2：所有 Topic 均走 `PULL`；
- Phase 3：所有消息均走 `PULL`。

**性能保障**：
- 实体批量下载：`PullExecutor::pull_entities_batch` 一次 HTTP 请求拉取最多 50 个 Agent/Group 或 1000 个 Topic；
- 消息流式下载：`download-messages-stream` 以 NDJSON 格式逐 Topic 返回，移动端 Semaphore(20) 并发处理。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_executor/pull_executor.rs`

### 7.2 同步中途网络中断

**现象**：WebSocket 在 Phase 2 或 Phase 3 中途断开。

**行为**：
1. `sync_service.rs` 外层 loop 捕获连接错误；
2. 指数退避重试：首次 500ms，每次翻倍，上限 5 秒，最多重试 3 次；
3. 若重试耗尽，发布 `vcp-sync-status` = `error`，前端提示用户；
4. 重新连接后，整个同步流水线从头开始（Manifest 重新发送），但幂等性键保证重复上传不会导致数据重复。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_service.rs`

### 7.3 同分钟内重复操作

**现象**：用户在同一分钟内多次修改同一实体，或网络抖动导致同一请求重复到达。

**行为**：
- 移动端生成相同的 `idempotency_key`（因 `minute_timestamp` 相同）；
- 桌面端 `checkIdempotency` 在 5 分钟 TTL 内命中缓存，直接返回上次结果；
- 实体写入仅执行一次。

**限制**：批量上传接口（`/upload-entities-batch`）未携带幂等键，重复请求理论上可能导致重复 Topic。实际中因 WS 信令层按序执行，且批量请求由单次 Manifest diff 触发，重复概率极低。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_executor/push_executor.rs`
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/idempotency.js`

### 7.4 损坏配置的保护

**现象**：桌面端 `config.json` 文件损坏（空文件、非法 JSON、数组而非对象）。

**行为**：
`uploadEntity` 在写入前执行严格校验：

```javascript
try {
    const content = await fs.readFile(configPath, "utf-8");
    config = JSON.parse(content);
    if (typeof config !== "object" || config === null || Array.isArray(config)) {
        throw new Error("Invalid config structure");
    }
    fileReadSuccess = true;
} catch (e) {
    if (e.code !== "ENOENT") {
        // 文件存在但损坏，拒绝覆盖
        throw new Error(`Cannot upload to corrupted config: ${e.message}`);
    }
}
```

**后果**：损坏配置拒绝接受同步更新，防止合法数据被空/错误结构覆盖。修复需用户手动清理桌面端对应目录。

**涉及代码**：
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/entity.js`

### 7.5 多设备同时在线（已知限制）

**现象**：用户拥有多台移动设备（如手机 + 平板），同时连接同一桌面端。

**当前限制**：
- 协议为**双端设计**（桌面端 ↔ 单个移动端），未实现多移动端仲裁；
- 若设备 A 和设备 B 同时修改同一实体，桌面端按"最后到达的 Manifest"处理，可能导致 A 与 B 的最终状态不一致；
- 消息级同步通过 Topic Hash 快速路径减少冲突，但无法完全消除写丢失（Write Loss）。

**缓解措施**：
- 高频率同步（连接即触发）缩小冲突窗口；
- 双哈希比对确保内容级不一致至少能被检测到（`changedTopics` 机制）；
- 建议用户避免在多台设备上同时编辑同一 Agent/Topic。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_service.rs`
- `VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/diff.js`

### 7.6 同步看门狗死锁恢复

**现象**：Phase 1 或 Phase 2 中，桌面端返回 `SYNC_DIFF_RESULTS` 后，某些 Pull/Push 任务因网络超时或异常未能正确递减 `pending_tasks`，导致同步永远停滞。

**行为**：
`sync_service.rs` 内置看门狗协程，每 10 秒检查一次 `pending_tasks`：

```rust
tokio::spawn(async move {
    let mut last_pending = pending_wd.load(Ordering::SeqCst);
    let mut stuck_count = 0;
    loop {
        tokio::time::sleep(Duration::from_secs(10)).await;
        let current_pending = pending_wd.load(Ordering::SeqCst);
        if current_pending == last_pending {
            stuck_count += 1;
            if stuck_count >= 6 {
                // 60 秒无进展，强制进入下一阶段
                if current_phase_wd == 1 {
                    tx.send(SyncCommand::StartTopicMetadata);
                } else if current_phase_wd == 2 {
                    tx.send(SyncCommand::StartTopicValidation);
                }
                break;
            }
        } else {
            stuck_count = 0;
            last_pending = current_pending;
        }
    }
});
```

**阈值含义**：
- `stuck_count >= 1`：发出警告日志"同步进度缓慢"；
- `stuck_count >= 6`（约 60 秒）：判定为死锁，强制触发阶段过渡。

**副作用**：强制过渡可能导致部分任务未完成，但由于幂等性和双哈希校验，下一轮同步会自动补齐剩余差异。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_service.rs`

### 7.7 超大消息负载分批

**现象**：某个 Topic 包含数万条消息（如长期聊天记录），单次 WS 消息可能超过 2 MB，导致移动端内存压力或 WS 帧截断。

**行为**：
移动端在 Phase 3 启动时，将 Topic 按消息数量分批次发送：

```rust
const MAX_MESSAGES_PER_BATCH: usize = 10000;

fn build_diff_batches(topic_states) -> VecDeque<Map<String, Value>> {
    // 每批最多 10000 条消息，约 1.5-2 MB JSON
    if current_msg_count > 0 && current_msg_count + msg_count > MAX_MESSAGES_PER_BATCH {
        batches.push_back(current_batch);
        current_batch = Map::new();
        current_msg_count = 0;
    }
}
```

桌面端返回 `SYNC_DIFF_RESULTS_BATCH` 后，移动端处理完当前批次才发送下一批：

```rust
let mut pending = pending_diff_batches.lock().await;
if let Some(next_batch) = pending.pop_front() {
    let msg = json!({ "type": "SYNC_MESSAGE_DIFF_BATCH", "topics": next_batch });
    let _ = ws_stream.send(Message::Text(msg.to_string().into())).await;
}
```

**内存保护**：
- 10,000 条消息的批次上限确保单次 JSON 解析在移动端可控；
- 断线重连后，`pending_diff_batches` 会被清空，防止发送过时数据。

**涉及代码**：
- `src-tauri/src/vcp_modules/sync_service.rs`
