# 性能优化与资源管理

## 1. 引言

在移动端同步场景中，性能（Performance）与资源管理（Resource Management）直接决定用户体验与设备稳定性。VCP Mobile 面向的是物理 Android 终端，其约束远比桌面端严苛：

| 约束维度 | 移动端限制 | 对同步的影响 |
|---------|-----------|------------|
| 内存（Memory） | 典型 4-8GB，应用可用仅数百 MB | 禁止全量缓冲，必须流式处理 |
| 网络（Network） | WiFi/5G/4G 频繁切换，高延迟抖动 | 需要自适应并发控制与指数退避 |
| 存储 IO（Storage IO） | SQLite 在闪存上的 WAL 写入存在锁竞争 | 必须串行化写入并合并事务 |
| CPU（Central Processing Unit） | 大核小核架构，后台任务受限 | 预渲染等 CPU 密集型任务需控制并行度 |

本文档系统阐述 VCP Mobile 同步引擎在 Rust 后端实现的六大性能优化策略，涵盖并发控制、数据库写入队列、批量尺寸控制、NDJSON（Newline Delimited JSON，换行分隔 JSON）流式传输、附件去重以及冷启动特殊处理。

## 2. NetworkAwareSemaphore

### 2.1 实现原理

`NetworkAwareSemaphore` 是同步引擎的核心并发阀门，基于 `tokio::sync::Semaphore` 实现（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。其设计目标是：**在 IO 密集型同步任务中最大化吞吐，同时避免移动端因并发过高导致 UI 卡顿或 OOM（Out Of Memory，内存溢出）**。

### 2.2 默认并发数

并发数并非硬编码，而是根据设备物理核心数动态计算：

```
concurrency = clamp(cores × 1.5, 6.0, 12.0)
```

| 设备类型 | 核心数 | 计算结果 | 最终并发数 |
|---------|--------|---------|-----------|
| 中低端手机 | 4 核 | 6.0 | 6 |
| 主流旗舰 | 8 核 | 12.0 | 12 |
| 高端平板 | 8 核以上 | >12.0 | 12（上限截断）|

在典型 8 核设备上，默认并发数为 **10**，这是 IO 密集型任务的理论平衡点（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。

### 2.3 基于网络类型的自适应调节

当前版本采用**设备能力优先**策略：并发上限由 CPU 核心数决定，而非实时网络测速。其背后的权衡是：

- **WiFi/5G 场景**：10-12 并发可充分利用带宽；
- **4G/弱网场景**：高并发会导致大量连接超时，依赖上层指数退避（Exponential Backoff）重试；
- **未来演进方向**：可接入 Android `ConnectivityManager` 实时调节 `Semaphore` 的许可数量。

### 2.4 信号量获取与释放的完整流程

```
┌─────────────────┐     acquire()      ┌──────────────────┐
│  SyncService    │ ─────────────────> │ NetworkAware     │
│  (主事件循环)    │                    │ Semaphore        │
└─────────────────┘                    └──────────────────┘
                                              │
                                              ▼
                                    ┌──────────────────┐
                                    │ tokio::sync::    │
                                    │ Semaphore::new() │
                                    └──────────────────┘
                                              │
        ┌─────────────────────────────────────┘
        ▼
┌─────────────────┐     spawn task       ┌──────────────────┐
│  异步任务 (PULL) │ ─────────────────>  │ 持有 Permit      │
│  或 PUSH        │                      │ 直至 await 结束  │
└─────────────────┘                      └──────────────────┘
```

伪代码如下：

```rust
// 初始化（sync_service.rs）
let network_semaphore = Arc::new(NetworkAwareSemaphore::new());

// 在 WS 消息处理中（sync_service.rs）
tauri::async_runtime::spawn(async move {
    let _permit = sem.acquire().await;  // 获取许可
    match data_type {
        Agent => PullExecutor::pull_agent(...).await,
        Group => PullExecutor::pull_group(...).await,
        Topic => PullExecutor::pull_agent_topic(...).await,
    }
    // Permit 在此处自动释放
});
```

**关键设计**：`SemaphorePermit` 与异步任务的生命周期绑定，任务完成即自动释放，无需手动调用 `release()`（`src-tauri/src/vcp_modules/sync/sync_service.rs`）。

## 3. DB 写队列（DbWriteQueue）

### 3.1 为什么需要串行化写入

SQLite 在 WAL（Write-Ahead Logging，预写日志）模式下虽然支持读并发，但**写操作仍需获取独占锁**。在同步高峰期，多个并发任务同时写入会导致 `database is locked` 错误。`DbWriteQueue` 通过**单工作线程 + 批量事务合并**彻底消除锁竞争（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）。

### 3.2 队列结构和工作线程

```
┌─────────────────────────────────────────────────────────────┐
│                     DbWriteQueue 架构                        │
├─────────────────────────────────────────────────────────────┤
│  生产者线程 (N个)                                             │
│       │                                                      │
│       ▼                                                      │
│  ┌─────────────┐   容量 256   ┌─────────────┐              │
│  │ mpsc::Sender │ ─────────> │ mpsc::Receiver│              │
│  │  (async)    │            │   (async)     │              │
│  └─────────────┘            └──────┬──────┘              │
│                                    ▼                        │
│                         ┌─────────────────┐                │
│                         │  Worker Task    │                │
│                         │  (tokio::spawn) │                │
│                         └────────┬────────┘                │
│                                  ▼                          │
│                         ┌─────────────────┐                │
│                         │ spawn_blocking  │                │
│                         │ + rusqlite      │                │
│                         └─────────────────┘                │
└─────────────────────────────────────────────────────────────┘
```

工作线程的核心行为（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）：

| 参数 | 值 | 目的 |
|-----|-----|------|
| 通道容量 | 256 | 背压（Backpressure）控制，防止内存无限增长 |
| 最大任务数/事务 | 200 | 平衡事务大小与提交频率 |
| 最大消息数/事务 | 5000 | 防止单事务 SQL 参数超限（SQLite 上限 999/语句） |
| 批处理超时 | 50ms | 低延迟与高开销之间的折中 |

### 3.3 写操作的优先级策略

`DbWriteTask` 枚举定义了所有可写入的实体类型（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）：

| 任务变体 | 实体 | 批量支持 |
|---------|------|---------|
| `Agent` | 智能体配置 | ❌ |
| `Group` | 群组配置 | ❌ |
| `Avatar` | 头像二进制 | ❌ |
| `AgentTopic` / `GroupTopic` | 话题元数据 | ✅ `AgentTopicBatch` |
| `TopicMessages` | 消息列表 + 预渲染字节 | ✅（内部 Chunked Insert） |
| `Flush` | 强制落盘信号 | — |

**批处理合并逻辑**：

```rust
while tasks_in_this_tx.len() < 200 && total_msg_count < 5000 {
    match timeout(50ms, rx.recv()).await {
        Ok(Some(Flush)) => { flush_tx_opt = Some(tx); break; }
        Ok(Some(task)) => { tasks_in_this_tx.push(task); }
        _ => break,
    }
}
```

`Flush` 任务具有**穿透优先级**：一旦收到，立即结算当前批次并返回确认（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）。

### 3.4 与同步流程的交互点

`DbWriteQueue` 在同步生命周期中的三个关键交互时机（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

| 同步阶段 | 交互行为 | 目的 |
|---------|---------|------|
| Phase 1 → Phase 2 过渡 | `write_queue.flush().await` | 确保所有 Agent/Group 元数据已落盘 |
| Phase 2 → Phase 2.5 过渡 | `write_queue.flush().await` | 确保 Topic 元数据已落盘 |
| Phase 3 → Finalize 过渡 | `write_queue.flush().await` | 确保所有消息已落盘，方可计算 Hash |

`flush()` 使用 `oneshot::channel` 实现同步等待，保证相位切换的强一致性（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）。

## 4. 批量操作的尺寸控制

### 4.1 实体批次的最大数量

在 HTTP 批量拉取（Batch Pull）和推送（Batch Push）中，不同实体类型采用不同的批次尺寸（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

| 数据类型 | 批次大小 | 拉取端点 | 理由 |
|---------|---------|---------|------|
| Agent / Group | 50 | `download-entities` | 配置数据复杂，单条体积大 |
| Topic（元数据） | 1000 | `download-entities` | 元数据轻量，可大规模并发 |
| 其他 | 100 | `download-entities` | 默认值 |

### 4.2 消息批次的最大数量

Phase 3（消息同步）采用**双层级批次控制**（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

**第一层：WebSocket Diff 批次**

```rust
const MAX_MESSAGES_PER_BATCH: usize = 10000;
```

- 每批最多包含 **10,000 条消息** 的哈希映射；
- 典型 Payload 约 1.5-2MB JSON，在移动端 WebSocket 安全范围内；
- `build_diff_batches` 函数按 `topic` 粒度累加，不会拆散单个 topic 的消息映射。

**第二层：HTTP NDJSON 流式批次**

- 拉取端：`download-messages-stream` 以 topic 为单位逐行返回；
- 处理端：`Semaphore(20)` 控制同时处理中的 topic 数量（`src-tauri/src/vcp_modules/sync/sync_executor/pull_executor.rs`）。

### 4.3 附件大小限制

| 方向 | 限制 | 代码位置 |
|-----|------|---------|
| 上传（Upload） | 20 MB | `file_manager.rs`（`store_file`） |
| 下载/读取（Download/Read） | 50 MB | `file_manager.rs`（`read_local_file_base64`） |

附件上传采用**3 并发限制**（`src-tauri/src/vcp_modules/sync/sync_executor/push_executor.rs`）：

```rust
const MAX_CONCURRENT_UPLOADS: usize = 3;
for chunk in hashes_to_upload.chunks(3) {
    let futures = chunk.iter().map(|h| upload_attachment(...));
    let results = futures_util::future::join_all(futures).await;
}
```

## 5. NDJSON 流式的内存优势

### 5.1 与传统 JSON 批量的内存对比

| 对比维度 | 传统 JSON 批量 | NDJSON 流式 |
|---------|--------------|------------|
| 内存占用 | 正比于总数据量（全量缓冲） | 正比于单个 topic（行缓冲） |
| 10,000 条消息 | ~50-100MB 峰值 | ~2-5MB 峰值 |
| 失败恢复 | 全部重试 | 单 topic 失败，不影响其他 |
| 进度反馈 | 请求结束后统一反馈 | 逐 topic 实时反馈 |

NDJSON（Newline Delimited JSON）的核心优势在于：**解析器无需等待整个响应体下载完成，只需维护一个行级缓冲区**（`src-tauri/src/vcp_modules/sync/sync_executor/pull_executor.rs`）。

### 5.2 流式读取的实现机制

移动端 `pull_messages_batch` 的实现（`src-tauri/src/vcp_modules/sync/sync_executor/pull_executor.rs`）：

```
┌─────────────────────────────────────────────────────────────┐
│              NDJSON 流式解析与并发处理管线                     │
├─────────────────────────────────────────────────────────────┤
│  HTTP Response Stream                                         │
│       │                                                      │
│       ▼                                                      │
│  ┌─────────────┐     chunk (bytes)     ┌─────────────┐     │
│  │ bytes_stream│ ───────────────────> │  Buffer Vec │     │
│  │ (futures)   │                     │  (行级累积)  │     │
│  └─────────────┘                     └──────┬──────┘     │
│                                             ▼              │
│                                   ┌─────────────────┐      │
│                                   │ 逐行查找 '\n'   │      │
│                                   │ drain(..=end)   │      │
│                                   └────────┬────────┘      │
│                                            ▼               │
│                                   ┌─────────────────┐      │
│                                   │  spawn 异步任务  │      │
│                                   │  Semaphore(20)  │      │
│                                   │  process_topic  │      │
│                                   └─────────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

关键代码逻辑：

```rust
let mut stream = res.bytes_stream();
let mut buffer: Vec<u8> = Vec::new();

while let Some(chunk) = stream.next().await {
    buffer.extend_from_slice(&chunk);
    
    while let Some(line_end) = buffer.iter().position(|&b| b == b'\n') {
        let line = buffer.drain(..=line_end).collect::<Vec<_>>();
        let topic_data: Value = serde_json::from_slice(&line)?;
        // 解析 topicId 和 messages，spawn 并发处理
    }
}
```

**Chunk 边界跨越处理**：由于 TCP 分片，`chunk` 可能截断在任意字节位置。`buffer` 累积机制确保不丢失半行数据（`src-tauri/src/vcp_modules/sync/sync_executor/pull_executor.rs`）。

### 5.3 Flush 策略

桌面端 `downloadMessagesStreamRaw` 的 Flush 策略（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/message.js`）：

```javascript
res.setHeader("Content-Type", "application/x-ndjson");
res.setHeader("Transfer-Encoding", "chunked");
res.flushHeaders();

for (const request of requests) {
    const history = await fs.readFile(historyPath, "utf-8");
    const messages = JSON.parse(history);
    res.write(JSON.stringify({ topicId, messages }) + "\n");
    // Node.js 自动 flush chunked 数据
}
res.end();
```

桌面端每处理完一个 topic 立即 `res.write()`，利用 HTTP Chunked Transfer Encoding 将数据推送到网络层。移动端因此可以**在完整响应到达之前就开始处理首批 topic**。

## 6. 附件去重策略

### 6.1 基于 Hash 的物理文件去重

附件在数据库中以 `hash` 为主键（Primary Key），采用 `INSERT ... ON CONFLICT(hash) DO UPDATE` 语义（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）：

```sql
INSERT INTO attachments (
    hash, mime_type, size, internal_path, extracted_text, 
    image_frames, thumbnail_path, created_at, updated_at
) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)
ON CONFLICT(hash) DO UPDATE SET
    mime_type = excluded.mime_type,
    size = excluded.size,
    ...
```

同一文件无论被多少条消息引用，物理存储仅保留一行记录。`message_attachments` 关联表维护消息与附件的多对多关系（`src-tauri/src/vcp_modules/persistence/db_write_queue.rs`）。

### 6.2 neededAttachmentHashes 机制

在 Push 场景中，移动端上传消息后，桌面端返回**仍需补充的附件哈希列表**（`src-tauri/src/vcp_modules/sync/sync_executor/push_executor.rs`）：

```javascript
// 桌面端（message.js）
const neededAttachmentHashes = new Set();
for (const msg of messages) {
    for (const att of msg.attachments) {
        const attRow = db.prepare("SELECT file_path FROM attachment_index WHERE hash = ?").get(hash);
        if (!attRow) {
            neededAttachmentHashes.add(hash);  // 桌面端缺失此附件
        }
    }
}
return { success: true, neededAttachmentHashes: Array.from(neededAttachmentHashes) };
```

移动端收集所有 topic 的 `neededAttachmentHashes`，去重后批量上传（`src-tauri/src/vcp_modules/sync/sync_executor/push_executor.rs`）：

```rust
let unique_hashes: Vec<String> = all_needed_hashes.into_iter()
    .filter(|h| seen.insert(h.clone()))
    .collect();
```

### 6.3 Orphan 附件清理

当前架构中，附件的清理依赖以下机制：

1. **消息删除时**：`message_attachments` 关联记录被级联删除，但 `attachments` 表中物理记录保留；
2. **未来演进方向**：可定期执行 `VACUUM` 或扫描无引用的 `attachments.hash`，清理孤儿文件。

## 7. 首次同步（冷启动）的特殊处理

### 7.1 空数据库首次连接时的全量拉取策略

当移动端首次安装或清空数据后连接桌面端，数据库中所有 `config_hash` 和 `content_hash` 均为空值。此时同步引擎的行为：

| Phase | 行为 | 结果 |
|-------|------|------|
| Phase 1 | 桌面端对比发现所有 Agent/Group/Avatar 均缺失 | 返回全量 PULL |
| Phase 2 | 所有 Topic 元数据缺失 | 返回全量 PULL |
| Phase 2.5 | 双哈希批量比对发现所有 Topic 消息不一致 | 进入 Phase 3 |
| Phase 3 | 全量消息哈希 diff | 分批拉取所有消息 |

### 7.2 HashInitializer::ensure_all_*_hashes 的作用

在 WebSocket 连接成功后、Phase 1 开始前，`SyncService` 主动调用（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

```rust
HashInitializer::ensure_all_agent_hashes(&db.pool).await?;
HashInitializer::ensure_all_group_hashes(&db.pool).await?;
```

`HashInitializer` 的职责（`src-tauri/src/vcp_modules/sync/sync_hash.rs`）：

| 方法 | 扫描条件 | 执行动作 |
|-----|---------|---------|
| `ensure_all_agent_hashes` | `config_hash = '' OR NULL OR 'PENDING'` | 逐行加载 Agent 配置，计算确定性哈希并回写 |
| `ensure_all_group_hashes` | `config_hash = '' OR NULL OR 'PENDING'` | 逐行加载 Group 配置，计算确定性哈希并回写 |

**关键设计**：冷启动时所有 hash 为空，`HashInitializer` 在**本地事务内**完成所有种子哈希的计算，避免将空 hash 发送到桌面端导致比对异常。

### 7.3 初始种子流程

```
┌─────────────────┐
│  WS 连接成功     │
└────────┬────────┘
         ▼
┌─────────────────────────┐
│ ensure_all_agent_hashes  │
│ ensure_all_group_hashes  │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Phase 1: Owner Metadata │
│  (全量 PULL Agent/Group) │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Phase 2: Topic Metadata │
│  (全量 PULL Topics)      │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Phase 2.5: Hash Validation│
│  (双哈希比对 → 全部 changed)│
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Phase 3: Messages       │
│  (分批 NDJSON 流式拉取)   │
└─────────────────────────┘
```

## 8. 性能监控指标

### 8.1 SyncMetrics 的结构

虽然代码中没有名为 `SyncMetrics` 的单一结构体，但同步引擎通过 `SyncLogger` 和多个原子计数器实现了等价的性能观测体系（`src-tauri/src/vcp_modules/sync/sync_logger.rs`）：

| 结构/类型 | 字段 | 含义 |
|----------|------|------|
| `SyncPhaseMetrics` | `started_at: Instant` | 阶段开始时间戳 |
| `SyncPhaseMetrics` | `expected_count: AtomicU32` | 预期操作总数 |
| `SyncPhaseMetrics` | `success_count: AtomicU32` | 成功操作计数 |
| `SyncPhaseMetrics` | `error_count: AtomicU32` | 失败操作计数 |
| `PhaseSummary` | `duration_ms: u64` | 阶段耗时（毫秒） |
| `ErrorAggregator` | `errors: HashMap<phase, Vec>` | 按阶段聚合的错误（上限 1000 条/阶段）|

### 8.2 各阶段的耗时统计

`SyncLogger` 在每个 Phase 的开始和结束节点记录耗时（`src-tauri/src/vcp_modules/sync/sync_logger.rs`）：

```rust
pub fn start_phase(&mut self, phase: &str, expected: u32) {
    let metrics = Arc::new(SyncPhaseMetrics {
        started_at: Instant::now(),
        expected_count: AtomicU32::new(expected),
        ...
    });
}

pub fn complete_phase(&mut self, phase: &str) -> Option<PhaseSummary> {
    let duration = metrics.started_at.elapsed().as_millis() as u64;
    // 输出: "Phase completed in 1250ms | expected=100, success=98, errors=2"
}
```

各 Phase 的监控粒度：

| Phase | 监控内容 | 日志示例 |
|-------|---------|---------|
| `owner_metadata` | Agent/Group/Avatar 的 PULL/PUSH/DELETE | `Diff: pull=15 push=2 delete=0` |
| `topic_metadata` | Topic 配置拉取 | `Sending batch of 500 topics` |

### 8.3 前端可观测的指标

同步引擎通过 Tauri Event 向前端推送实时进度（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

| Event 名称 | 触发时机 | Payload 字段 |
|-----------|---------|-------------|
| `vcp-sync-progress` | 每完成一个 topic 或批次 | `phase`, `total`, `completed`, `message` |
| `vcp-sync-status` | 连接状态变化 | `status`, `message`, `source` |
| `vcp-sync-completed` | 同步会话结束 | `agentsChanged`, `groupsChanged`, ... |
| `vcp-log` | 每条同步日志 | `level`, `category`, `message` |

**Phase 3 进度计算**：使用 `Phase3Tracker` 替代简单的 `AtomicU32` 计数器，防止同一 topic 被重复标记完成导致进度跳变（`src-tauri/src/vcp_modules/sync/sync_service.rs`）：

```rust
struct Phase3Tracker {
    completed: tokio::sync::Mutex<HashSet<String>>,
    modified: tokio::sync::Mutex<HashSet<String>>,
    total: std::sync::atomic::AtomicUsize,
}
```

`mark_completed` 方法通过 `HashSet::insert` 的返回值判断是否为首次完成，确保进度条线性增长。

## 10. v0.9.14 新增优化

### 10.1 PullExecutor `spawn_blocking`  offload

v0.9.14 将 `PullExecutor::process_topic_messages` 中的消息处理循环整体包裹在 `tokio::task::spawn_blocking` 中：

```rust
tokio::task::spawn_blocking(move || {
    for msg in dtos {
        // fingerprinting + Zstd compression + optional pre-rendering
    }
}).await?
```

**收益**：
- 彻底消除 CPU 密集型任务（fingerprinting、Zstd 压缩、AST 编译）对 Tokio 异步运行时同线程 IO 任务的阻塞；
- 同步吞吐量在高并发消息场景下提升显著；
- 内置 `std::time::Instant` 性能剖析（`[ProfileDetail]` / `[ProfileSummary]`），便于诊断瓶颈。

### 10.2 预渲染解耦（Prerender Decoupling）

v0.9.14 引入 `syncPrerenderEnabled` 设置（前端设置面板 + 后端 `settings_manager.rs`）：

| 设置状态 | 同步下载行为 | `render_cache` 写入 | 消息加载行为 |
|---------|-------------|---------------------|-------------|
| 启用（默认） | `MessagePullSyncDTO` → `ChatMessage` 时执行预渲染，`blocks` 填充 | 批量写入 | 命中 render_cache 直接反序列化 |
| 禁用 | `blocks` 显式设为 `None`，跳过预渲染 | **完全跳过**（条件写入） | 懒渲染缓存闭环在加载时编译 |

**设计意图**：
- 首次全量同步时消息量极大，若强制预渲染所有消息，同步时间会显著延长；
- 禁用预渲染后，同步下载阶段仅需解压和入库，预渲染推迟到用户实际浏览消息时按需触发；
- `render_cache` 条件写入：批量插入时过滤掉 `render_bytes` 为空的条目，`PARAMS_PER_MSG` 不变，但 `render_cache` INSERT 批次可能为空而被跳过。

### 6.4 去重对网络流量的影响

以一次典型的 Push 会话为例：假设用户发送了 10 条消息，每条消息附带同一张 5MB 图片。传统方案下，该图片会被上传 10 次，总流量 50MB。基于 hash 的去重机制确保：

1. 首次上传后，`uploaded_hashes` 集合记录该 hash；
2. 后续所有引用同 hash 的消息不再触发上传；
3. 桌面端 `neededAttachmentHashes` 仅返回缺失的 hash；
4. 实际网络流量降至 5MB + 极小元数据开销。

## 9. 性能优化最佳实践

### 9.1 移动端侧

- **避免在同步期间触发大量 UI 重绘**：`vcp-sync-completed` 事件已聚合所有变更，前端应在收到该事件后批量刷新；
- **监听 `vcp-sync-progress` 而非轮询**：进度事件以 topic/批次为粒度推送，轮询会额外消耗 CPU 与电池；
- **日志清理策略**：`clear_old_sync_logs` 命令可按保留天数清理，建议设置为 7 天。

### 9.2 桌面端侧

- **NDJSON 流的逐 topic flush**：`downloadMessagesStreamRaw` 中每处理完一个 topic 立即 `res.write()`，不可缓存到数组后一次性 `res.json()` 输出，否则会失去流式内存优势；
- **`writeIntentLock` 的及时释放**：批量上传完成后必须遍历释放所有 topic 的锁，否则文件监控器（File Watcher）会停滞（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/sync/message.js`）。
