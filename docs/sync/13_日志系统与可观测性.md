# 日志系统与可观测性

## 1. 引言

在分布式同步系统中，日志（Log）不仅是调试（Debugging）的核心依据，更是运行时可观测性（Observability）的唯一数据源。VCP Mobile 的同步链路横跨 Rust 移动端、Node.js 桌面服务端与 Vue 前端三层，任何一层的状态漂移都可能导致数据不一致。通过结构化的三级日志体系——Session（会话）→ Phase（阶段）→ Operation（操作）——开发者可以精准定位一次同步请求在全链路中的执行轨迹，快速诊断网络延迟、哈希冲突、数据库锁竞争等典型问题。

本文档同时覆盖移动端（Rust）与桌面端（Node.js）的日志实现，并基于真实日志样本解读关键模式。

---

## 2. Session → Phase → Operation 三级结构

VCP Mobile 的日志系统采用严格嵌套的三级树形结构，两端实现保持一致。

### 2.1 概念定义

| 层级 | 英文术语 | 中文释义 | 生命周期 | 标识符示例 |
|------|----------|----------|----------|------------|
| 一级 | Session | 同步会话 | 一次完整的同步连接（WebSocket 握手到断开） | `sync_1778537156425` |
| 二级 | Phase | 同步阶段 | 会话内的逻辑分段，按数据类型划分 | `owner_metadata`、`messages` |
| 三级 | Operation | 单个操作 | 阶段内的最小原子单元（单条实体处理） | `diff:agent`、`upload_messages_batch_stream` |

### 2.2 嵌套关系与约束

- 一个 **Session** 包含多个 **Phase**，Phase 按固定顺序串行执行：`owner_metadata` → `topic_metadata` → `messages`。
- 一个 **Phase** 包含任意数量的 **Operation**，Operation 在 Phase 内可并行或串行执行。
- Phase 具备自动收尾机制：桌面端（`logger.js`）在启动新 Phase 时，若检测到前一个 Phase 未完成，会自动调用 `completePhase` 强制关闭（`src-tauri/src/vcp_modules/sync_logger.rs` 与 `logger.js` 均实现了此语义）。
- 每个 Phase 维护独立的计数器：`expected`（预期总量）、`success`（成功数）、`errors`（错误数）。

### 2.3 标识符与追踪

| 属性 | 来源 | 说明 |
|------|------|------|
| Session ID | `Date.now()` 时间戳 | 桌面端由 `startSession(prefix)` 生成；移动端由外部调度器传入日志目录 |
| Phase 名称 | 硬编码枚举 | 两端共享同一组阶段名，确保日志可交叉比对 |
| Operation ID | 实体类型 + 实体 ID | 如 `topic_1773214887121`、`agent_xxx` |

## 3. Rust 端 SyncLogger

### 3.1 结构与核心方法

`SyncLogger`（`src-tauri/src/vcp_modules/sync_logger.rs`）是移动端的日志核心，采用纯内存状态 + 可选文件持久化的混合模式：

| 结构成员 | 类型 | 职责 |
|----------|------|------|
| `log_level` | `LogLevel` | 过滤阈值，低于此级别的日志被丢弃 |
| `phases` | `HashMap<String, Arc<SyncPhaseMetrics>>` | 各阶段的运行时指标（原子计数器） |
| `error_aggregator` | `ErrorAggregator` | 按 Phase 聚合错误，上限 1000 条/Phase |
| `log_file` | `Option<std::fs::File>` | 可选的日志文件句柄 |
| `log_path` | `Option<PathBuf>` | 日志文件路径 |

**核心方法清单：**

| 方法 | 级别 | 作用 |
|------|------|------|
| `new_session(level, log_dir)` | — | 创建会话，按时间命名日志文件 |
| `start_phase(phase, expected)` | Info | 注册新 Phase，记录预期处理量 |
| `log_operation(phase, type, id, success, detail)` | Debug/Error | 记录单个操作结果，自动累加计数器 |
| `complete_phase(phase)` | Info | 输出阶段耗时与统计摘要 |
| `end_session()` | Info | 收尾并刷盘 |

### 3.2 日志级别定义

Rust 端定义三级枚举（`src-tauri/src/vcp_modules/sync_logger.rs`）：

| 级别 | 数值 | 用途 |
|------|------|------|
| `Debug` | 0 | 高频操作日志（单条实体处理），默认关闭 |
| `Info` | 1 | 阶段起止、连接事件、统计摘要 |
| `Error` | 2 | 操作失败、数据库锁、网络异常 |

**级别过滤规则**：仅当 `level >= self.log_level` 时写入。生产环境建议设为 `Info`，调试时降为 `Debug`。

### 3.3 日志格式

Rust 端采用 ISO-8601 带毫秒偏移的格式：

```
[2026-05-12T05:09:48.123+08:00] [INFO] [messages] Phase started (expected=18)
```

字段依次为：**时间戳**、**级别**、**Phase 名**、**消息体**。

### 3.4 与前端事件的集成

Rust 端通过 Tauri 的 `emit` 机制向前端广播同步状态，日志系统本身不直接管理事件，但日志内容与事件 payload 保持语义一致。前端监听的三类核心事件如下：

| 事件名称 | 触发时机 | payload 关键字段 |
|----------|----------|------------------|
| `vcp-sync-status` | 阶段状态变更 | `{ phase, status, progress }` |
| `vcp-sync-completed` | 全量同步成功结束 | `{ duration_ms, summary }` |
| `vcp-sync-error` | 任意层级异常 | `{ phase, error, retryable }` |

移动端日志目录通常由外部调用方（如 `sync_service.rs`）传入，路径遵循应用配置目录下的 `logs/sync/`。

## 4. Node 端 SyncLogger

### 4.1 结构与核心方法

桌面端的 `SyncLogger`（`VCPChat/VCPDistributedServer/Plugin/VCPMobileSync/core/logger.js`）在语义上与 Rust 端对齐，但增加了 **WebSocket 广播** 与 **控制台可视化边框** 能力：

| 结构成员 | 类型 | 职责 |
|----------|------|------|
| `currentSession` | `Object` | 当前会话元数据与 Phase Map |
| `currentPhase` | `String` | 当前活跃 Phase，默认 `"system"` |
| `logDir` | `String` | 固定为 `../logs/sync/` |
| `logStream` | `WriteStream` | 文件写入流 |
| `logFilePath` | `String` | 当前会话的日志文件绝对路径 |

**核心方法清单：**

| 方法 | 作用 |
|------|------|
| `startSession(prefix)` | 清理旧流，创建新会话，输出 ASCII 边框横幅 |
| `startPhase(phase, expected)` | 注册 Phase，自动结束上一个未完成的 Phase |
| `logOperation(phase, type, id, result, detail)` | 记录操作，每 100 条输出进度简报 |
| `logInfo(phase, message, level)` | 纯文本信息，不更新计数器 |
| `completePhase(phase)` | 关闭 Phase，输出耗时与统计 |
| `endSession()` | 强制完成所有 Phase，输出结束横幅 |
| `broadcast(payload)` | 通过 WebSocket 向所有在线客户端广播 JSON 事件 |

### 4.2 日志格式规范

Node 端在控制台与文件中采用两种格式：

**控制台输出（带可视化边框）：**
```
[VCPMobile] [22:05:56] ╔══════════════════════════════════════════════════════════════════════════════╗
[VCPMobile] [22:05:56] ║ SESSION START: sync_1778537156425                                            ║
[VCPMobile] [22:05:56] ╚══════════════════════════════════════════════════════════════════════════════╝
[VCPMobile] [22:05:56]
[VCPMobile] [22:05:56] ┌─── PHASE START: [owner_metadata] (expected: 0) ──────────────────────────────
[VCPMobile] [22:05:56] │ [owner_metadata] diff:agent -> success (push=0 pull=0 delete=0 skip=6)
[VCPMobile] [22:05:56] └─── PHASE COMPLETE: [owner_metadata] duration: 31ms (Total:3 OK:3 ERR:0)
```

**文件持久化（精简格式）：**
```
[2026-05-11T22:05:56.426Z] === SESSION START: sync_1778537156425 ===
[2026-05-11T22:05:56.426Z] [websocket] connection:100.68.180.95 -> success (token=ok, path=/)
[2026-05-11T22:05:56.448Z] --- PHASE START: [owner_metadata] (expected: 0) ---
[2026-05-11T22:05:56.459Z] [owner_metadata] diff:agent -> success (push=0 pull=0 delete=0 skip=6)
```

| 字段 | 控制台 | 文件 |
|------|--------|------|
| 前缀 | `[VCPMobile] [HH:mm:ss]` | ISO-8601 时间戳 |
| Phase 标记 | `│ [phase_name]` | `[phase_name]` |
| 边框线 | `╔═╗╚═╝┌─┐└─┘` | 无 |
| 进度简报 | `│ [PROGRESS]` | `[PROGRESS]` |

### 4.3 WebSocket 日志广播机制

桌面端通过 `setWss(wss)` 注入 WebSocket 服务器引用，在关键节点广播结构化事件：

| 广播类型 | 触发方法 | payload 字段 | 用途 |
|----------|----------|--------------|------|
| `DESKTOP_PHASE_START` | `startPhase()` | `{ type, phase, ts }` | 通知前端某阶段开始 |
| `DESKTOP_PHASE_COMPLETE` | `completePhase()` | `{ type, phase, ts }` | 通知前端某阶段结束 |
| `SYNC_LOG_EVENT` | `logOperation()` / `logInfo()` | `{ type, level, phase, message, ts }` | 实时透传操作详情 |

广播范围覆盖所有 `readyState === 1`（OPEN）的 WebSocket 客户端，发送失败时静默吞掉异常，避免影响主同步流程。

## 5. 日志文件管理

### 5.1 文件命名规则

| 端 | 命名模板 | 示例 |
|----|----------|------|
| 移动端（Rust） | `{timestamp}_sync.log` | `20260512_050948_sync.log` |
| 桌面端（Node） | `{prefix}_{timestamp}.log` | `sync_1778537156425.log` |

时间戳语义：Rust 使用本地时间的 `YYYYMMDD_HHMMSS`；Node 使用 Unix 毫秒 `Date.now()`。

### 5.2 目录结构

桌面端的日志目录固定为插件根目录下的相对路径：

```
VCPMobileSync/
├── core/
│   └── logger.js
└── logs/
    └── sync/
        ├── sync_1778537156425.log       # 常规同步会话
        ├── sync_1778536310477.log       # 常规同步会话
        └── system_1778533765692.log     # 系统/索引扫描会话
```

移动端日志目录由 Tauri 的 `app_config_dir` 推导，通常位于：
- Android：`/data/data/com.vcp.avatar/files/logs/sync/`

### 5.3 日志分类

| 分类 | 前缀 | 典型内容 | 生命周期 |
|------|------|----------|----------|
| 同步会话日志 | `sync_*` | WebSocket 握手、三阶段 diff、消息上传下载 | 单次同步连接 |
| 系统日志 | `system_*` | 数据库初始化、轻量级索引扫描（reconcile） | 服务启动时 |

### 5.4 文件大小与轮转策略

当前版本（0.9.13）**未实现自动日志轮转（Log Rotation）**，依赖以下被动机制控制体积：

| 机制 | 说明 |
|------|------|
| 按会话分文件 | 每个 Session 独占一个文件，连接断开即关闭句柄 |
| 进度采样 | Node 端每 100 条 Operation 才输出一次 `PROGRESS` 简报，降低高频场景下的行数 |
| 错误上限 | Rust 端 `ErrorAggregator` 单 Phase 最多保留 1000 条错误明细 |
| 手动清理 | 需运维人员定期删除旧日志，或扩展 `cleanupOldLogs` 方法 |

## 6. 前端事件监听

Vue 前端通过 Tauri 的 `listen` API 订阅同步状态事件，实现 UI 进度条、Toast 提示与异常弹窗。

### 6.1 事件总览

| 事件名称 | 来源端 | 频率 | 典型消费者 |
|----------|--------|------|------------|
| `vcp-sync-status` | Rust（Tauri emit） | 高频（Phase/Operation 级别） | 同步进度页、状态栏 |
| `vcp-sync-completed` | Rust（Tauri emit） | 低频（每 Session 一次） | 全局 Toast、徽章清零 |
| `vcp-sync-error` | Rust（Tauri emit） | 中频（仅异常时） | 错误弹窗、重试按钮 |
| `DESKTOP_PHASE_START` | Node（WebSocket） | 中频 | 桌面端调试面板 |
| `DESKTOP_PHASE_COMPLETE` | Node（WebSocket） | 中频 | 桌面端调试面板 |

### 6.2 事件 Payload 结构

**`vcp-sync-status`**
```typescript
interface SyncStatusEvent {
  phase: "owner_metadata" | "topic_metadata" | "messages";
  status: "started" | "in_progress" | "completed";
  progress?: { expected: number; success: number; errors: number };
  detail?: string;  // 可选的操作级描述
}
```

**`vcp-sync-completed`**
```typescript
interface SyncCompletedEvent {
  session_id?: string;
  duration_ms: number;
  summary: {
    owner_metadata?: PhaseSummary;
    topic_metadata?: PhaseSummary;
    messages?: PhaseSummary;
  };
}
```

**`vcp-sync-error`**
```typescript
interface SyncErrorEvent {
  phase: string;
  error: string;
  entity_id?: string;
  retryable: boolean;  // 关键字段，决定是否展示"重试"按钮
}
```

### 6.3 前端与日志的对应关系

前端事件并非日志的逐行镜像，而是日志数据的聚合抽象。例如：
- Rust 端连续调用 18 次 `log_operation(..., success=true)` 对应前端 `vcp-sync-status` 中 `progress.success` 从 0 递增到 18。
- 当 `log_operation` 的 `success=false` 且 `detail` 包含 `"database is locked"` 时，`vcp-sync-error` 的 `retryable` 被设为 `true`，UI 可提示用户稍后重试。

## 7. 实际日志样本分析

以下分析基于桌面端真实日志文件 `sync_1778537156425.log`（2026-05-12 06:05:57 生成）与 `system_1778533765692.log`。

### 样本 1：会话建立与版本协商

```
[2026-05-11T22:05:56.426Z] [websocket] connection:100.68.180.95 -> success (token=ok, path=/)
[2026-05-11T22:05:56.432Z] [websocket] version_check:mobile -> info (mobileVersion=0.9.13, pluginVersion=0.9.13)
```

**解读**：
- 移动端 IP 为 `100.68.180.95`，WebSocket 连接成功并通过 Token 校验。
- 版本双向校验通过（两端均为 `0.9.13`）。若此处版本不匹配，后续会触发兼容性降级或拒绝连接。

### 样本 2：Owner Metadata 阶段的零变更场景

```
[2026-05-11T22:05:56.448Z] --- PHASE START: [owner_metadata] (expected: 0) ---
[2026-05-11T22:05:56.459Z] [owner_metadata] diff:agent -> success (push=0 pull=0 delete=0 skip=6)
[2026-05-11T22:05:56.460Z] [owner_metadata] diff:group -> success (push=0 pull=0 delete=0 skip=2)
[2026-05-11T22:05:56.469Z] [owner_metadata] diff:avatar -> success (push=0 pull=0 delete=0 skip=8)
[2026-05-11T22:05:56.479Z] --- PHASE COMPLETE: [owner_metadata] ---
```

**解读**：
- `expected: 0` 表示桌面端未预设具体数量，依赖实际 diff 结果动态统计。
- 三类实体（agent、group、avatar）均无推拉删操作，全部 `skip`，说明两端元数据已完全一致。
- Phase 耗时约 31ms，属于典型的快速路径（Fast Path）。

### 样本 3：Topic Metadata 阶段的批量哈希比对

```
[2026-05-11T22:05:56.521Z] [topic_metadata] diff:topic -> success (push=1 pull=0 delete=0 skip=130)
[2026-05-11T22:05:56.576Z] [topic_metadata] diff_batch_v2:summary -> success (total=131 match=113 changed=18)
```

**解读**：
- 首次 diff 发现 1 个 topic 需要 push（移动端向桌面端推送）。
- 随后进入 `SYNC_TOPIC_HASH_BATCH_V2` 流程，对 131 个 topic 进行批量哈希比对。
- 结果：113 个完全匹配（match），18 个存在差异（changed）。这 18 个 topic 将进入下一阶段的 messages 级 diff。

### 样本 4：Messages 阶段的上传与下载

```
[2026-05-11T22:05:56.756Z] [messages] diff:topic_1773214887121 -> success (toPull=2 toPush=false)
[2026-05-11T22:05:56.769Z] [messages] diff:topic_1778526713775 -> success (toPull=0 toPush=true)
[2026-05-11T22:05:56.829Z] [messages] download_messages_stream:batch -> success (topics=1 success=1 error=0)
```

**解读**：
- `topic_1773214887121` 需要从桌面端拉取 2 条消息（`toPull=2`）。
- `topic_1778526713775` 需要向桌面端推送消息（`toPush=true`）。
- `download_messages_stream:batch` 表明实际执行了流式下载，1 个 topic 成功，0 错误。
- 注意第 51 行的 `upload_messages_batch_stream` 出现 `error=1`，说明上传批次中存在个别失败，但 HTTP 状态码仍为 200，属于部分成功的典型模式。

### 样本 5：系统级索引扫描

```
[2026-05-11T21:09:25.692Z] === SESSION START: system_1778533765692 ===
[2026-05-11T21:09:25.703Z] [reconcile] 正在执行轻量级索引扫描...
[2026-05-11T21:09:29.030Z] [reconcile] summary:reconcile -> success (agents=6 groups=2 topics=175 messages=4507 attachments=398 aggregated=2)
```

**解读**：
- `system_*` 前缀表示这不是客户端触发的同步，而是桌面端服务启动后的自检流程。
- `reconcile` Phase 耗时约 3.3 秒，扫描了 4507 条消息与 398 个附件。
- `aggregated=2` 表示有 2 条消息被聚合处理（可能是重复合并或碎片化整理）。

## 8. 调试技巧

### 8.1 如何开启详细日志

| 端 | 操作方式 | 效果 |
|----|----------|------|
| Rust 移动端 | 修改 `SyncLogger::new_session` 的 `log_level` 参数为 `LogLevel::Debug` | 输出所有单条 Operation，文件体积显著增大 |
| Node 桌面端 | 日志固定为全量输出，控制台天然可见 `│` 前缀的详细操作 | 通过 `logs/sync/` 目录直接查看文件 |
| Vue 前端 | 在 DevTools Console 过滤 `vcp-sync-*` 事件 | 实时观察事件流与 payload |

### 8.2 如何定位特定实体的同步历史

假设需要追踪某个 topic（如 `topic_1773214887121`）的完整同步轨迹：

1. **grep 单实体 ID**：在 `logs/sync/` 目录下搜索 `topic_1773214887121`，可快速找到涉及该实体的所有 Operation。
2. **grep Phase 边界**：搜索 `PHASE START: [messages]` 与 `PHASE COMPLETE: [messages]`，定位该实体所属会话的上下文。
3. **结合 Session ID**：确认 `sync_{timestamp}` 文件名，确保不跨会话混淆。
4. **移动端补充**：Android 端通过 ADB 拉取日志文件：`adb pull /data/data/com.vcp.avatar/files/logs/sync/`。

### 8.3 常见日志模式与问题对应关系

| 日志模式 | 可能的问题 | 诊断方向 |
|----------|------------|----------|
| `database is locked` | SQLite WAL 模式下的并发写入冲突 | 检查是否有其他线程/进程持有数据库连接 |
| `toPull>0` 但 `download_messages_stream` 缺失 | 差异检测后未执行实际传输 | 检查 Phase ACK 是否丢失，WebSocket 是否提前断开 |
| `error=1` 但 HTTP `status=200` | 批次内部部分失败 | 展开批次详情，定位具体失败的 topic/消息 ID |
| Phase 耗时 > 30s | 网络延迟或大数据量传输 | 检查 `expected` 数量，确认是否处于首次全量同步 |
| `skip=N` 但 `N` 远小于预期总数 | 哈希索引未覆盖全部实体 | 检查 reconcile 扫描是否完整，索引是否损坏 |
| `VERSION_ACK` 版本不匹配 | 两端协议不兼容 | 升级移动端或桌面端插件至同一版本 |

### 8.4 性能分析速查

通过日志中的 `duration` 字段可进行轻量级性能剖析：

| 指标 | 正常范围 | 异常阈值 | 优化建议 |
|------|----------|----------|----------|
| `owner_metadata` Phase | < 100ms | > 500ms | 减少 avatar/agent 数量，或启用增量哈希 |
| `topic_metadata` Phase | < 1s | > 5s | 检查 topic 总数是否超过千级，考虑归档旧 topic |
| `messages` Phase（无变更） | < 500ms | > 2s | 优化哈希比对算法，减少不必要的批次请求 |
| `messages` Phase（有变更） | 取决于消息量 | 单条消息 > 200ms | 检查附件大小，启用流式分片传输 |
