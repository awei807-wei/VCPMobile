# 双端精确对照手册

## 1. 引言

本文档是 **VCP Mobile 同步模块** 的**快速查阅手册（Quick Reference）**，面向需要同时理解移动端（Rust）与桌面端（Node.js）同步实现细节的开发者。

文档采用**表格为主、文字为辅**的编排方式，旨在建立以下精确对照关系：

- **文件级映射**：Rust 模块 ↔ JS 模块的物理对应关系
- **函数级映射**：核心算法的函数签名对照
- **类型级映射**：跨语言的数据结构字段与命名风格转换
- **协议级映射**：WebSocket 消息类型与 HTTP 端点的双端处理链路
- **存储级映射**：移动端 SQLite 表 ↔ 桌面端文件/目录布局

**命名风格提示**：Rust 侧采用 `snake_case`（蛇形命名），JS 侧采用 `camelCase`（驼峰命名）。JSON 序列化时通过 `#[serde(rename_all = "camelCase")]` 或手动标注实现自动转换。

## 3. 函数级映射表

### 3.1 Hash 计算

| 功能 | Rust 函数签名 | Node.js 函数签名 | 说明 |
|---|---|---|---|
| 稳定 JSON 序列化 | `stable_stringify(value: &Value) -> String` | `stableStringify(obj, key = "")` | 按 key 字典序排序后序列化，消除字段顺序差异 |
| 确定性哈希 | `compute_deterministic_hash<T: Serialize>(data: &T) -> String` | `computeDtoHash(dto, fields)` | Rust 侧基于 `stable_stringify` + SHA-256；JS 侧基于 `stableStringify` + `crypto.createHash` |
| Merkle Root | `compute_merkle_root(hashes: Vec<String>) -> String` | `computeAggregatedHash(hashes)` | 哈希排序后连接再 SHA-256；空数组返回 `""` |
| 消息指纹 | `HashAggregator::compute_message_fingerprint(content: &str, attachment_hashes: &[String]) -> String` | `computeMessageFingerprint(msg)` | 基于 content + 排序后的附件 hash 数组 |
| Agent 配置哈希 | `HashAggregator::compute_agent_config_hash(dto: &AgentSyncDTO) -> String` | `computeDtoHash(dto, AGENT_SYNC_FIELDS)` | temperature 统一格式化为 2 位小数 |
| Group 配置哈希 | `HashAggregator::compute_group_config_hash(dto: &GroupSyncDTO) -> String` | `computeDtoHash(dto, GROUP_SYNC_FIELDS)` | 直接对 DTO 计算 |
| Agent Topic 元数据哈希 | `HashAggregator::compute_agent_topic_metadata_hash(dto: &AgentTopicSyncDTO) -> String` | `computeDtoHash(dto, AGENT_TOPIC_SYNC_FIELDS)` | 字段：id, name, createdAt, locked, unread |
| Group Topic 元数据哈希 | `HashAggregator::compute_group_topic_metadata_hash(dto: &GroupTopicSyncDTO) -> String` | `computeDtoHash(dto, GROUP_TOPIC_SYNC_FIELDS)` | 字段：id, name, createdAt |
| Topic 内容哈希 | `HashAggregator::compute_topic_root_hash(tx, topic_id) -> Result<String, String>` | `ingestHistoryToDb` 内计算 | 汇总消息 fingerprint 后计算 Merkle Root |
| Agent 内容哈希 | `HashAggregator::compute_agent_root_hash(tx, agent_id) -> Result<String, String>` | `computeAggregatedHashes` 内计算 | 汇总子 topic 的 config_hash + content_hash |
| Group 内容哈希 | `HashAggregator::compute_group_root_hash(tx, group_id) -> Result<String, String>` | `computeAggregatedHashes` 内计算 | 同 Agent 内容哈希 |
| 二进制哈希 | `HashAggregator::compute_avatar_hash(bytes: &[u8]) -> String` | `computeBinaryHash(buffer)` | 直接对二进制数据计算 SHA-256 |
| Hash 冒泡（Topic） | `HashAggregator::bubble_topic_hash(tx, topic_id)` | `ingestHistoryToDb` → `computeAggregatedHashes` | 更新 topic 的 content_hash + config_hash |
| Hash 冒泡（Agent） | `HashAggregator::bubble_agent_hash(tx, agent_id)` | `computeAggregatedHashes` 内更新 | 更新 agent 的 content_hash |
| Hash 冒泡（Group） | `HashAggregator::bubble_group_hash(tx, group_id)` | `computeAggregatedHashes` 内更新 | 更新 group 的 content_hash |
| Hash 初始化（全部 Agent） | `HashInitializer::ensure_all_agent_hashes(pool)` | `reconcileLocalFiles` 内扫描时 | 对缺失 config_hash 的 agent 补全 |
| Hash 初始化（全部 Group） | `HashInitializer::ensure_all_group_hashes(pool)` | `reconcileLocalFiles` 内扫描时 | 对缺失 config_hash 的 group 补全 |

## 4. 类型级映射表

### 4.1 核心枚举

| Rust Enum | JS 等价 | 说明 |
|---|---|---|
| `SyncDataType` (`agent`/`group`/`avatar`/`topic`/`message`) | 字符串字面量 `"agent"` 等 | Rust 侧使用 `#[serde(rename_all = "lowercase")]` |
| `LogLevel` (`Debug`/`Info`/`Error`) | 字符串 `"debug"`/`"info"`/`"error"` | Rust 侧为 enum，JS 侧为字符串 |
| `PipelinePhase` (`Idle`/`Phase1Metadata`/`Phase2TopicMetadata`/`Phase3Messages`/`Completed`/`Failed`) | 无显式枚举 | JS 侧通过消息类型隐式处理 Phase |
| `SyncCommand` | 无直接对应 | Rust 侧内部命令枚举，用于 mpsc 信道调度 |
| `PipelineCommand` | 无直接对应 | Rust 侧内部流水线命令枚举 |

### 4.2 核心结构体

| Rust Struct | JS Object Shape | 字段差异说明 |
|---|---|---|
| `EntityState { id, hash, config_hash, content_hash, ts, deleted_at, owner_type }` | `{ id, hash, configHash, contentHash, ts, deletedAt, ownerType }` | snake_case ↔ camelCase。`config_hash`/`content_hash` 为 V2 新增双哈希字段 |
| `SyncManifest { data_type, items }` | `{ dataType, data: items }` | WS 消息中 JS 侧使用 `data` 承载 `items` 数组 |
| `AgentSyncDTO { name, system_prompt, model, temperature, context_token_limit, max_output_tokens, stream_output }` | `{ name, systemPrompt, model, temperature, contextTokenLimit, maxOutputTokens, streamOutput }` | 字段名完全对应，风格差异 |
| `GroupSyncDTO { name, members, mode, member_tags, group_prompt, invite_prompt, use_unified_model, unified_model, tag_match_mode, created_at }` | `{ name, members, mode, memberTags, groupPrompt, invitePrompt, useUnifiedModel, unifiedModel, tagMatchMode, createdAt }` | `member_tags` 为 `Option<Value>`；JS 侧默认 `{}` |
| `AgentTopicSyncDTO { id, name, created_at, locked, unread, owner_id }` | `{ id, name, createdAt, locked, unread, ownerId }` | `locked` 默认 `true`，`unread` 默认 `false` |

## 9. 同步协议 Phase 对照

| Phase | 名称 | 移动端主导动作 | 桌面端主导动作 | 关键 WS 消息 |
|---|---|---|---|---|
| Phase 0 | 握手 | 发送 `VERSION_CHECK` | 返回 `VERSION_ACK` | `VERSION_CHECK` → `VERSION_ACK` |
| Phase 1 | Owner Metadata | 构建并发送 Agent/Group/Avatar 清单 | 比对后返回 `SYNC_DIFF_RESULTS` | `SYNC_MANIFEST` → `SYNC_DIFF_RESULTS` |
| Phase 1.5 | 实体传输 | 根据 diff 结果执行 pull/push/delete | 被动响应 HTTP 请求 | HTTP 请求 |
| Phase 2 | Topic Metadata | 对变更 owner 构建定向 Topic 清单发送 | 比对后返回 `SYNC_DIFF_RESULTS` | `SYNC_MANIFEST` (phase=2) → `SYNC_DIFF_RESULTS` |
| Phase 2.5 | Topic Validation | 发送 `SYNC_TOPIC_HASH_BATCH_V2` | 比对后返回变更 topic 列表 | `SYNC_TOPIC_HASH_BATCH_V2` → `SYNC_TOPIC_HASH_RESULTS` |
| Phase 3 | Messages | 发送 `SYNC_MESSAGE_DIFF_BATCH` | 比对后返回 `SYNC_DIFF_RESULTS_BATCH` | `SYNC_MESSAGE_DIFF_BATCH` → `SYNC_DIFF_RESULTS_BATCH` |
| Phase 3.5 | 消息传输 | 根据 diff 执行 push/pull 消息 | 被动响应 HTTP 流式请求 | HTTP 流式请求 |
| Phase 4 | Finalize | 发送 `PHASE_COMPLETED` 并关闭 WS | 确认后等待断开 | `PHASE_COMPLETED` |

## 12. 错误处理与容错机制对照

| 机制 | 移动端（Rust） | 桌面端（Node.js） | 说明 |
|---|---|---|---|
| 连接失败重试 | `MAX_RETRIES = 3`，指数退避 500ms→5s | 无（服务器被动等待） | Rust 侧 `tokio_tungstenite::connect_async` 失败时触发 |
| 流式错误帧 | 检测 `{"_stream_error": ...}` 后中断 | 写入 `{"_stream_error": e.message}` + `\n` | 双端 NDJSON 流级错误处理 |
| 单 Topic 错误帧 | 检测 `{"_error": ...}` 后标记失败 | 写入 `{"topicId", "messages": [], "_error": ...}` | 不中断整体流 |
| 消息解析失败 | 统计 `failed_count`，继续处理其他消息 | 无直接对应 | Rust 侧 `process_topic_messages` 内 `catch_unwind` + 错误统计 |
| 看门狗（Deadlock 检测） | 每 10s 检查 pending 计数，stuck ≥ 6 次强制推进 | 无直接对应 | Rust 侧 Phase 1/2 专用，防止 manifest diff 任务死锁 |
| 幂等性控制 | `generate_idempotency_key`（SHA-256，分钟级时间戳） | `checkIdempotency(opId)` + `recordOperation(opId, result)` | JS 侧基于内存，重启后失效 |
| 数据库锁定重试 | `detail.contains("database is locked")` 标记 retryable | 无（better-sqlite3 同步写入） | Rust 侧 `SyncLogger::log_operation` 内判断 |
| 版本不兼容 | 断开连接，提示下载最新插件 | 返回 `VERSION_ACK` 携带版本号 | 期望版本 `EXPECTED_PLUGIN_VERSION = "0.9.13"` |
| 配置损坏保护 | 无（SQLite 事务保证一致性） | 读取失败时拒绝覆盖，抛出异常 | JS 侧 `uploadEntity` 内 `content.trim() === ""` 检测 |
| 空数组写入保护 | 无 | `Array.isArray(config)` 时拒绝写入 | JS 侧防止数组覆盖对象配置 |
| ID 不匹配保护 | 无 | Group 上传时校验 `config.id === safeId` | JS 侧 `uploadEntity` 内校验 |

## 14. 数据流时序对照

### 14.1 单次完整同步时序（移动端视角）

| 步骤 | 移动端动作 | 桌面端动作 | 通信方式 |
|---|---|---|---|
| 1 | 读取设置，构建 WS URL（`?token=`） | — | 本地 |
| 2 | `connect_async(&ws_url)` | WS Server 监听 `0.0.0.0:port` | WS |
| 3 | 发送 `VERSION_CHECK` | 验证 token，返回 `VERSION_ACK` | WS |
| 4 | `HashInitializer::ensure_all_*_hashes()` | — | 本地 SQLite |
| 5 | 发送 `SYNC_MANIFEST` (phase=1, dataType=agent/group/avatar) | `handleSyncManifest()` 比对索引 | WS |
| 6 | 接收 `SYNC_DIFF_RESULTS` | 返回 diff 结果 | WS |
| 7 | 派发 pull/push/delete 任务 | 被动响应 HTTP 下载/上传请求 | HTTP |
| 8 | 发送 `PHASE_COMPLETED` (phase=owner_metadata) | 返回 `PHASE_ACK` | WS |
| 9 | `Pipeline::on_owner_metadata_done()` → Phase 2 | — | 本地 mpsc |
| 10 | 发送 `SYNC_MANIFEST` (phase=2, dataType=topic, targetedOwners) | `handleSyncManifest()` 定向比对 | WS |
| 11 | 接收 `SYNC_DIFF_RESULTS`，派发 Topic pull/push | 被动响应 | WS + HTTP |
| 12 | 发送 `PHASE_COMPLETED` (phase=topic_metadata) | 返回 `PHASE_ACK` | WS |
| 13 | `Pipeline::on_topic_metadata_pull_done()` → Phase 2.5 | — | 本地 mpsc |
| 14 | 发送 `SYNC_TOPIC_HASH_BATCH_V2` | `handleSyncTopicHashBatchV2()` 比对 | WS |
| 15 | 接收 `SYNC_TOPIC_HASH_RESULTS` | 返回变更 topic 列表 | WS |
| 16 | `Pipeline::on_topic_validation_done()` → Phase 3 | — | 本地 mpsc |
| 17 | 分批发送 `SYNC_MESSAGE_DIFF_BATCH` | `handleSyncMessageDiffBatch()` 比对消息 | WS |
| 18 | 接收 `SYNC_DIFF_RESULTS_BATCH` | 返回每 topic 的 toPull/toPush | WS |
| 19 | 先 `push_messages_batch()`，再 `pull_messages_batch()` | 被动响应 NDJSON 流 | HTTP (NDJSON) |
| 20 | 每批次完成后发送下一批 `SYNC_MESSAGE_DIFF_BATCH` | — | WS |
| 21 | 所有 topic 标记完成 | — | 本地 |
| 22 | `Pipeline::on_messages_done()` → Finalize | — | 本地 mpsc |
| 23 | 全局 Hash 冒泡（`bubble_topic_hash` → `bubble_agent/group_hash`） | — | SQLite 事务 |
| 24 | 发送 `PHASE_COMPLETED`，关闭 WS | 返回 `PHASE_ACK` | WS |
| 25 | 发射 `vcp-sync-completed` | — | Tauri Event |

### 14.2 实时变更推送时序

| 场景 | 移动端动作 | 桌面端动作 | 通信方式 |
|---|---|---|---|
| Agent/Group/Topic 本地变更 | `SyncCommand::NotifyLocalChange` → 发送 `SYNC_ENTITY_UPDATE` | `upsertEntityIndex(id, dataType, null, hash, ts)` | WS |
| 消息本地变更 | 由应用层触发同步（或等待下次全量同步） | 文件监听器 `chokidar` 检测 history.json 变更 | 文件系统 |
| 本地删除 | `SyncCommand::NotifyDelete` → 发送 `SYNC_ENTITY_DELETE` | `deleteEntity()` / `deleteMessage()` | WS |
| 桌面端删除 | 接收 `SYNC_DELETE_NOTIFY` → `DeleteExecutor::soft_delete_*` | 用户操作触发删除 → 发送 `SYNC_DELETE_NOTIFY` | WS |

## 19. 版本演进标记（V1 → V2）

| 特性 | V1（旧版） | V2（当前 v0.9.13） | 涉及文件 |
|---|---|---|---|
| Topic 哈希 | 单 `content_hash` | `config_hash` + `content_hash` 双哈希 | `sync_types.rs`, `sync_hash.rs`, `manifest_builder.rs`, `diff.js`, `manifest.js` |
| Agent/Group 哈希 | 单 `hash` | `config_hash` + `content_hash` 双哈希 | 同上 |
| Topic 同步范围 | 全量 Topic 清单 | 定向 `targetedOwners` 清单 | `sync_service.rs`, `manifest_builder.rs`, `manifest.js` |
| 消息 Diff | 单 topic 请求 | 批量 topic + 分批发送 | `sync_service.rs`, `diff.js` |
| 消息传输 | 整 JSON 响应 | NDJSON 流式传输 | `pull_executor.rs`, `push_executor.rs`, `message.js` |
| 实体下载 | 逐个 HTTP GET | 批量 HTTP POST | `pull_executor.rs`, `entity.js` |
| 实体上传 | 逐个 HTTP POST | 批量 HTTP POST | `push_executor.rs`, `entity.js` |
| 索引聚合 | 简单 hash 汇总 | 层级 `computeAggregatedHashes` | `sync_hash.rs`, `index.js` |
| 配置文件写入 | 直接覆盖 | 原子写入（tmp → rename） | `entity.js`, `message.js` |
| 消息预渲染 | 无 | `MessageRenderCompiler` 并发预渲染 | `pull_executor.rs` |
| 附件路径查询 | N+1 查询 | 批量查询 + HashMap 缓存 | `pull_executor.rs` |
