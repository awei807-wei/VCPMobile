# 03. 会话状态与消息流

## 1. 概述

### 1.1 领域定位

core/stores/ 中围绕聊天的四个 Pinia Store 构成了 VCP Mobile 前端**对话引擎的状态底座**。它们负责管理：

- **谁在对谈**：当前选中的 Agent/Group 与话题（chatSessionStore）
- **谈了什么**：消息列表的加载、分页、追加与修改（chatHistoryStore）
- **正在说什么**：Backend-Driven Streaming 架构下的流式消息生命周期（chatStreamStore）
- **要带什么**：用户待发送的附件暂存与预处理（ttachmentStore）

这四个 Store 通过 Tauri IPC 与 Rust 后端交互，并通过 Vue 3 的响应式系统驱动 ChatView.vue 及其子组件的渲染。

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **Backend-Driven Streaming** | 流式消息由后端 SSE 事件驱动。前端**不再**预创建 thinking 占位消息；`StreamEvent::thinking` 才触发 UI 初始化。 |
| **双 Store 边界** | `chatStreamStore` 管理流式期间的响应式状态；流结束后内容移交 `chatHistoryStore`。两者职责严格分离。 |
| **全局流消息池** | `activeStreamMessages`（`Map<string, ChatMessage>`）存储所有正在生成的消息，保证前台/后台切换时流式链路不断裂。 |
| **Object Hydration** | 从 DB 加载历史时，若发现某消息正处于活跃流式中，则用全局池中"活的"响应式对象替换骨架，实现无缝接续。 |
| **Channel 流式加载** | `load_chat_history_streamed` 通过 Tauri `Channel<HistoryChunk>` 逐条推送消息，避免大话题全量加载导致 UI 阻塞。 |
| **会话一致性校验** | 所有异步回调（Channel onmessage）均校验 `currentTopicId` 是否仍与请求时一致，防止话题切换导致旧数据污染。 |

### 2.3 与 Agent/Group 的关系

`chatSessionStore` 不直接存储 Agent/Group 的完整列表，而是消费 `useAssistantStore`（`assistant.ts`）中的 `agents` 和 `groups` 数组：

- 判定 `ownerType` 时扫描 `assistantStore.agents.some(a => a.id === itemId)`
- 构造 `currentSelectedItem` 时从 `assistantStore` 中查找对应对象并注入 `type` 字段

这意味着 Agent/Group 的增删改由 `assistantStore` 统一管理，`chatSessionStore` 只持有**选中态**的副本。

#### 核心加载逻辑（loadHistory）

> 函数位置：`src/core/stores/chatHistoryStore.ts` 第 83–187 行

使用 Tauri `Channel<HistoryChunk>` 实现**流式逐条加载**：

```
前端调用 load_chat_history_streamed
       │
       ▼
Rust 后端逐条查询消息，通过 Channel 推送
       │
       ▼
channel.onmessage = (chunk) => {
    ┌─────────────────────────────────────────┐
    │ 1. 会话一致性校验                       │──> 若 currentTopicId ≠ requestedTopicId，丢弃
    └─────────────┬───────────────────────────┘
                  ▼
    ┌─────────────────────────────────────────┐
    │ 2. Object Hydration (关键修复)          │──> 从 streamStore.activeStreamMessages
    │    若消息正在活跃生成，取出活的响应式对象  │    获取活跃消息替换骨架
    └─────────────┬───────────────────────────┘
                  ▼
    ┌─────────────────────────────────────────┐
    │ 3. 追加到列表或缓冲池                    │──> offset===0 ? push 到 currentChatHistory
    │                                         │    offset>0  ? push 到 buffer（旧消息前置）
    └─────────────┬───────────────────────────┘
                  ▼
    ┌─────────────────────────────────────────┐
    │ 4. is_last 时完成合并                    │──> offset>0: buffer 前置到 currentChatHistory
    │    更新 historyOffset / hasMoreHistory   │    offset===0: 直接设置偏移量
    └─────────────────────────────────────────┘
}
```

**Object Hydration 机制**：

当用户切换到一个正在进行流式生成的话题时，从 DB 拉回来的消息是"死"的 Plain Object。`loadHistory` 在收到每条 `HistoryChunk` 时，检查 `streamStore.activeStreamMessages.get(chunk.message.id)`：

- 若命中 → 使用全局池中**已 reactive** 的对象替换骨架，UI 自动接续流式动画
- 若未命中 → 使用 chunk 中的原始消息对象

这保证了"后台生成 → 切换话题 → 前台展示"的链路中，消息的响应式引用不丢失。

## 4. 流式状态（chatStreamStore）

### 4.1 Backend-Driven Streaming 架构

VCP Mobile 在 `8888d85` 架构变更后，全面转向 **Backend-Driven Streaming**：

| 旧架构 | 新架构 |
|--------|--------|
| 前端预创建 thinking 消息（带临时 ID） | 前端**不预创建** thinking 消息 |
| 前端分配 `thinking_message_id` | 后端自主生成 Assistant 消息 ID |
| `StreamEvent::content` 直接更新历史列表 | `StreamEvent::thinking` 触发 `chatStreamStore` 初始化气泡 |
| 流式期间状态分散在多处 | 流式期间状态统一由 `activeStreamMessages` 管理 |

这一变更大幅简化了 `chatHistoryStore`：它只需关心"何时将完成的消息追加到列表"，而不再参与 thinking 状态的维护。

### 4.2 流式消息生命周期

#### 核心状态字段

> 文件位置：`src/core/stores/chatStreamStore.ts`

```typescript
const streamingMessageId = ref<string | null>(null);  // 当前主活跃流消息 ID

// 按会话索引的活跃流集合
const sessionActiveStreams = ref<Record<string, string[]>>({}); // "itemId:topicId" -> messageId[]

// 全局活跃流消息池：messageId -> Reactive<ChatMessage>
const activeStreamMessages = reactive<Map<string, ChatMessage>>(new Map());
```

#### 事件处理流水线（processStreamEvent）

> 函数位置：`src/core/stores/chatStreamStore.ts` 第 141–277 行

```
Rust Channel 推送 StreamEvent
       │
       ▼
┌─────────────────────────────────────────┐
│ 提取 actualMessageId, type, context     │──> 从 event 中解构
│ 校验 messageId / topicId / itemId 存在   │
└─────────────┬───────────────────────────┘
              ▼
┌─────────────────────────────────────────┐
│ 查找或创建消息对象                       │──> msg = activeStreamMessages.get(id)
│ 若为新流 (isNewStream):                  │    若不存在，创建 reactive<ChatMessage>
│   - role="assistant"                     │    推入 activeStreamMessages
│   - isThinking = (type === "thinking")   │    调用 onMessageCreated 回调
│   - 异步 append_single_message 入库骨架   │    (确保后台切换回来后能触发 Hydration)
└─────────────┬───────────────────────────┘
              ▼
      按 type 分支处理:
      ┌─────────┬─────────┬─────────┬─────────┐
      ▼         ▼         ▼         ▼         ▼
   thinking    data     aurora    end      error
```

**各事件分支详解**：

| 事件类型 | 行为 | 状态变更 |
|----------|------|----------|
| `thinking` | 后端开始生成 thinking 内容 | `msg.isThinking = true`；加入 `sessionActiveStreams`；设置 `streamingMessageId` |
| `data` | SSE 增量文本到达 | `msg.isThinking = false`；追加 `chunk.choices[0].delta.content` 到 `msg.content`；同步 `tailContent` |
| `aurora` | Aurora 语义沉淀更新 | `msg.content = aurora.content`；`msg.tailContent = aurora.tail`；`msg.blocks = aurora.stableBlocks`；`msg.tailBlock = aurora.tailBlock` |
| `end` | 流式正常结束 | `msg.tailContent = ""`；`removeSessionStream()`；调用 `process_message_content` 编译 blocks；触发 `onStreamFinished` |
| `error` | 流式异常结束 | 同 `end`，但追加错误文本到 content；`finishReason = "error"` |

此异步调用的意义在于：即使用户在流式生成中途切走，骨架消息也已入库。当用户切回时，`loadHistory` 能读到这条消息，并通过 Object Hydration 将其替换为全局池中"活的"响应式对象，实现**无缝接续**。

```
┌─────────────────────────────────────────────────────────────────────┐
│                         严格边界示意图                               │
│                                                                     │

│   └──────┬──────┘                                                   │
│          │                                                          │
│          ▼                                                          │
│   ┌─────────────┐     Backend-Driven 分界线                         │
│   │  Rust 后端   │     ─────────────────────────────                │
│   │             │     前端不再预创建 thinking 消息                   │
│   └──────┬──────┘                                                   │
│          │ StreamEvent::thinking                                     │
│          ▼                                                          │
│   ┌─────────────┐                                                   │
│   │chatStream   │  创建 reactive msg → activeStreamMessages         │
│   │  Store       │  通知 UI 插入新消息 (onMessageCreated)            │

└─────────────────────────────────────────────────────────────────────┘
```

**边界规则**：

1. **流式期间**：`chatStreamStore` 是 `msg.content`、`msg.blocks`、`msg.isThinking` 的**唯一写入者**
2. **流结束后**：`chatStreamStore` 将 `isThinking` 置为 `false`，编译最终 `blocks`，随后通知 `onStreamFinished`
3. **数据库最终持久化**：由 Rust 后端在流结束后调用 `patch_single_message` 完成（见 `agent_chat_application_service.rs` 5.3.3）
4. **全局池清理**：`removeSessionStream` 在流结束后 1 秒延迟从 `activeStreamMessages` 中删除，确保 finalize 期间组件仍能访问对象

## 8. 设计决策与注意事项

### 8.1 为何采用 Backend-Driven Streaming？

旧架构中，前端需要在 `sendMessage` 时预分配一个 `thinking_message_id`，构造占位消息推入列表，然后等待后端响应。这导致：

1. **ID 冲突风险**：前后端 ID 生成策略不一致时，可能出现重复或覆盖
2. **状态同步复杂**：前端 thinking 消息与后端实际消息在 `content`、`blocks`、`finishReason` 上需要双向同步
3. **重生成场景混乱**：重新生成时，前端需要手动替换旧消息，容易出错

新架构将 thinking ID 的生成权完全交予后端（`agent_chat_application_service.rs` 第 6 步），前端仅作为事件的**被动消费者**。`chatHistoryStore` 从 400+ 行简化为只管"追加/删除/编辑"，流式复杂度全部收敛到 `chatStreamStore`。

### 8.2 全局流消息池的 OOM 防御

`activeStreamMessages` 是 `reactive<Map>`，理论上可以无限增长（用户快速切换多个话题，每个话题都在后台生成）。防御措施：

- **上限**：`MAX_STREAM_MESSAGES = 100`
- **清理策略**：按插入顺序（Map 保持插入顺序），删除最旧的**非活跃**消息（不在当前 `activeStreamingIds` 中）
- **延迟清理**：流结束后延迟 1 秒才从池中删除，确保 `finalizeStream` 和 UI 收尾能拿到对象

### 8.3 Object Hydration 的必要性

没有 Hydration 时，话题切换的时序问题如下：

```
T0: 用户发送消息，AI 开始流式生成 (msg_001)
T1: 用户切换到话题 B
T2: AI 继续推送 msg_001 的 data 事件 (msg_001 在 activeStreamMessages 中)
T3: 用户切回话题 A，loadHistory 从 DB 读取 msg_001 (死的 Plain Object)
T4: AI 继续推送 msg_001 的 data 事件
    → 若使用 DB 骨架：UI 不会更新（非响应式）
    → 若使用 Hydration：用 activeStreamMessages 中的 reactive 对象替换，UI 自动更新
```

Hydration 是 Backend-Driven Streaming 能在多话题场景下正常工作的**关键补丁**。

### 8.4 会话一致性校验的两次防线

`loadHistory` 在 `Channel.onmessage` 中设置了两道防线：

1. **逐条校验**：每条 chunk 到达时检查 `currentTopicId !== requestedTopicId`，若已切换则直接 `return` 丢弃
2. **完成校验**：`completePromise`  resolve 后再次检查，若已切换则打印警告并提前返回，避免后续 `resolveMessageAssets` 做无用功

同理，`chatStreamStore.processStreamEvent` 中的 `onMessageCreated` 回调在 `chatHistoryStore` 侧也校验 `tid === sessionStore.currentTopicId`，防止后台流的 UI 插入污染当前列表。

### 8.5 附件路径解析的安全策略

`attachmentStore.resolveMessageAssets` 负责将 Rust 返回的本地物理路径转换为 WebView 可访问的 URL：

- 仅对 `type.startsWith("image/")` 的图片执行转换
- 排除 `http://` / `https://` / `data:` 开头的已有 URL
- 使用 Tauri 的 `convertFileSrc(sourcePath)` 生成 `asset://` 或 `http://asset.localhost/` 协议 URL
- `resolvedSrc` 为**纯运行时字段**，不持久化到 DB，避免跨端同步时路径失效

### 8.6 编辑重发的乐观更新与持久化一致性

当用户长按某条用户消息选择"编辑重发"时：

1. `editingOriginalMessageId` 被设为该消息 ID，输入框回填内容
2. 用户修改内容后点击发送，`sendMessage` 检测到 `editingOriginalMessageId` 存在
3. 调用 `truncate_history_after_timestamp` 从后端删除该时间戳之后的全部消息
4. 前端乐观截断 `currentChatHistory`：`slice(0, targetIndex + 1)`
5. 调用 `triggerGeneration(targetMsg)` 以修改后的消息重新启动生成

这一流程确保了**前端状态与后端数据库在截断操作上的原子一致性**：前端在调用后端截断成功后，才更新本地数组。
