# 04. UI状态与覆盖层管理

## 1. 概述

### 1.1 领域定位

`core/stores/` 中的 `overlay`、`layout`、`theme`、`notification` 以及 `appLifecycle` 构成了 VCP Mobile 前端的** UI 状态中枢**。它们负责管理所有非内容层的视觉状态：虚拟页面栈（SlidePage）、左右抽屉（Sidebar）、全局模态（Prompt/ContextMenu/Editor）、主题变量注入、Toast/通知中心，以及应用启动生命周期。

该领域的核心使命是：
- 将物理返回键（手势/硬件按钮）正确引导至"关闭覆盖层"而非"退出应用"
- 提供统一的层级秩序（Z-Index），杜绝 `z-[999]` 式的魔法数字灾难
- 在 Hash 单页路由（仅 `/chat`）的前提下，实现"多页面"的覆盖层体验
- 将后端系统事件转化为前端可消费的 Toast/通知/状态指示

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **覆盖层优先于路由** | VCP Mobile 本质上是单路由应用（`/#/chat`）。所有"页面"（设置、Agent 设置等）均为覆盖层，由 `overlayStore.pageStack` 管理，而非 Vue Router |
| **语义化层级** | 所有 Z-Index 必须通过 `layers.ts` 常量或 UnoCSS 快捷类（`z-drawer`、`z-page`）引用，禁止裸露数字 |
| **物理返回键三层消费** | 第一层：`modalStack` 关闭覆盖层；第二层：重置聊天会话到欢迎页；第三级：双击退出到后台。此模型由 `App.vue handleExitRequest` 实现 |
| **状态即 API** | Store 不提供复杂业务逻辑，只提供原子状态变更与计算属性。业务编排（如启动序列）下沉到 `appLifecycleStore` |
| **静默降级** | IPC 调用失败不阻断 UI 流程（如心跳设置、主题更新），仅打印日志 |

## 2. 覆盖层状态（overlayStore）

> 文件位置：`src/core/stores/overlay.ts`

`overlayStore` 是 VCP Mobile 前端**所有覆盖层状态的唯一真相源**。它将传统分散的 `isSettingsOpen`、`isPromptOpen` 等布尔标志重构为统一的栈结构，支持多层 SlidePage 叠加与返回手势的正确消费。

### 2.1 虚拟页面栈（SlidePage）

#### 2.1.1 栈结构

```ts
interface PageStackItem {
  type: string;      // 页面类型标识，如 'settings'、'agentSettings'
  id?: string;       // 可选的业务 ID，如 agentId、groupId
  modalId: string;   // 在 modalStack 中注册的唯一标识，如 'Page:agentSettings:agent_123'
}

const pageStack = ref<PageStackItem[]>([]);
```

`pageStack` 是一个**后进先出（LIFO）**的虚拟导航栈。每打开一个 SlidePage，就向栈顶压入一个 `PageStackItem`；关闭时从栈顶弹出。栈顶项始终可通过 `pageStackTop` 计算属性获取。

#### 2.1.2 支持的页面类型

| 类型 | 用途 | 是否需 ID | 特殊管理 |
|------|------|----------|----------|
| `settings` | 全局设置页 | 否 | 标准栈 |
| `agentSettings` | Agent 设置页 | 是（agentId） | 标准栈 |
| `groupSettings` | Group 设置页 | 是（groupId） | 标准栈 |
| `syncSession` | 同步会话页 | 否 | **独立管理**（需联动 `syncSessionStore`） |
| `rebuildSession` | 重建任务页 | 否 | **独立管理**（需联动 `rebuildSessionStore`） |

#### 2.1.3 与 Vue Router 的区别

```
传统多页应用 (Vue Router):
  /chat  →  /settings  →  /agents/:id
   ↑         ↑            ↑
  路由切换   路由切换      路由切换
  页面重建   页面重建      页面重建

VCP Mobile (SlidePage 覆盖层):
  /chat  ────────────────────────► 始终不变（Hash 路由）
    │
    ├─ overlayStore.pushPage('settings')
    │   → SettingsView SlidePage 滑入 (z-index: 40)
    │
    ├─ overlayStore.pushPage('agentSettings', 'agent_123')
    │   → AgentSettingsView SlidePage 滑入 (z-index: 41)
    │
    └─ overlayStore.popPage()
        → 关闭栈顶 SlidePage，下方页面保持 DOM 常驻
```

**关键差异**：
- Vue Router 切页会导致组件卸载/重建；SlidePage 保持组件常驻（通过 `FeatureOverlays.vue` 中 `v-if`/`isMounted` 控制），支持平滑的进入/离开动画
- 路由只负责 `/#/chat`，所有"导航"通过覆盖层模拟

### 2.2 模态历史栈（Operation Aegis）

#### 2.2.1 架构定位

Operation Aegis 是 VCP Mobile 针对移动端物理返回键（手势/硬件按钮）的**统一拦截与分发系统**，实现在 `src/core/composables/useModalHistory.ts` 中。其核心目标是：**确保用户在覆盖层打开时按返回键，关闭的是覆盖层而非退出应用**。

#### 2.2.2 modalStack 结构

```ts
interface ModalInstance {
  id: string;       // 唯一标识，如 'LeftDrawer'、'Page:settings:'、'Prompt'
  close: () => void; // 关闭回调（由注册方提供）
}

const modalStack = ref<ModalInstance[]>([]);
```

`modalStack` 是一个**全局单例**的响应式数组（`ref`），跨所有 Store 与组件共享。它不存储任何 UI 状态，只存储"如何关闭"的回调引用。这种设计确保了：
- 同一时刻多个覆盖层（如 Drawer + SlidePage + Prompt）可以共存
- 返回键总是关闭**最上层**的覆盖层（LIFO）
- 关闭逻辑由注册方自行定义，Aegis 只负责调度

#### 2.2.3 useModalHistory() 提供的 API

| API | 签名 | 说明 |
|-----|------|------|
| `registerModal` | `(id: string, closeHandler: () => void) => void` | 注册一个模态到历史栈，同时 `pushState` 到浏览器历史 |
| `unregisterModal` | `(id: string) => void` | 从栈中移除，若非 popstate 触发则执行 `history.back()` |
| `closeTopModal` | `() => boolean` | 关闭栈顶模态，返回是否成功关闭 |
| `initRootHistory` | `() => void` | 压入 dummy state `{ vcpRoot: true, vcpMain: true }`，作为防护盾底 |
| `modalStackLength` | `() => number` | 返回当前栈深度 |

#### 2.2.4 内部状态机

```
┌─────────┐    popstate 触发    ┌──────────────────┐
│  IDLE   │ ──────────────────► │ POPSTATE_HANDLING│
└─────────┘                     └────────┬─────────┘
       ▲                                 │
       │         close 完成              │ 调用 topModal.close()
       └─────────────────────────────────┘

┌─────────┐    UI 主动关闭触发  ┌──────────────────┐
│  IDLE   │ ──────────────────► │  INTERNAL_BACK   │
└─────────┘                     └────────┬─────────┘
       ▲                                 │
       │         history.back() 完成     │ 调用 window.history.back()
       └─────────────────────────────────┘
```

- `IDLE`：正常状态
- `POPSTATE_HANDLING`：正在处理浏览器 `popstate` 事件（物理返回手势）
- `INTERNAL_BACK`：由 `unregisterModal` 触发的 `history.back()`，需要在 `handlePopState` 中识别并忽略，避免递归

#### 2.2.5 与 App.vue 中 handleExitRequest 的集成（三层消费模型）

> 文件位置：`src/App.vue` 第 87–148 行

当用户按物理返回键且 `modalStack` 为空时，`handlePopState` 会触发 `vcp-exit-requested` 自定义事件。`App.vue` 的 `handleExitRequest` 监听此事件，执行三层消费：

```
物理返回键按下 / vcp-exit-requested 事件触发
           │
           ▼
┌───────────────────────────────┐
│ 第一层：Modal Stack 消费       │
│ closeTopModal()               │
│ ──> 若 modalStack 非空，关闭  │
│     最上层覆盖层并 return     │
└───────────────┬───────────────┘
            No  │
                ▼
┌───────────────────────────────┐
│ 第二层：会话状态消费           │
│ 若 lifecycleStore.state ===   │
│   'READY' 且当前有选中 Agent/ │
│   Group，重置会话到欢迎页     │
│   (currentSelectedItem = null)│
└───────────────┬───────────────┘
            No  │
                ▼
┌───────────────────────────────┐
│ 第三层：物理退出消费           │
│ 首次按：显示 Toast"再按一次     │
│        退出应用"，2s 倒计时    │
│ 再次按：invoke move_task_to_  │
│        back（退到后台）        │
└───────────────────────────────┘
```

三层模型的关键是**严格优先级**：只要还有覆盖层打开，就不可能误触退出；只要还在聊天会话中，返回键先回到主页而非直接退出。

#### 2.2.6 路由避让策略

> 文件位置：`src/core/composables/useModalHistory.ts` 第 47–54 行

```ts
const currentPath = window.location.hash.replace(/^#/, '') || '/';
if (currentPath !== '/chat' && currentPath !== '/') {
  return; // 不拦截，交给 Vue Router
}
```

当 URL Hash 不是 `/chat` 或 `/` 时（如某些临时路由页），`handlePopState` 直接放行，让 Vue Router 处理导航。同时，`App.vue` 通过 `router.afterEach` 在每次路由切换后自动重新注入防护盾：

```ts
router.afterEach(() => {
  initRootHistory();
});
```

这确保了从任何路由返回主界面后，dummy state 始终位于栈顶，防护盾不会意外失效。

### 2.3 动态 Z-Index 计算

#### 2.3.1 计算逻辑

```ts
const getPageZIndex = (type: string) => {
  const index = pageStack.value.findIndex(p => p.type === type);
  if (index === -1) return LAYER_PAGE_BASE;
  return LAYER_PAGE_BASE + Math.min(index, LAYER_PAGE_MAX_OFFSET);
};
```

- 基础值 `LAYER_PAGE_BASE = 40`
- 每深一层（索引 +1），z-index 增加 1
- 封顶值 `LAYER_PAGE_MAX_OFFSET = 9`，即最高为 49，不会突破到 toast（50）及以上层级

#### 2.3.2 实际层级示例

| 栈状态 | settings (index=0) | agentSettings (index=1) | syncSession (index=2) |
|--------|-------------------|------------------------|----------------------|
| z-index | 40 | 41 | 42 |

这保证了后打开的页面始终压在前一个页面上方，形成自然的视觉堆叠。

### 2.4 页面打开/关闭的完整流程

#### 2.4.1 打开页面（以 AgentSettings 为例）

```
用户点击 Agent 设置按钮
           │
           ▼

           │
           ▼
┌─────────────────────────────┐
│ pushPage('agentSettings',   │
│   'agent_123')              │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐

│    'Page:agentSettings:     │
│     agent_123'              │

```

#### 2.4.2 关闭页面

关闭有两种路径：

**路径 A：UI 主动关闭（点击返回按钮）**
```
用户点击返回按钮
           │
           ▼
overlayStore.closeAgentSettings()
           │
           ▼
popPage()  ──> unregisterModal(modalId)
           │
           ▼
  执行 history.back()
  从 pageStack 弹出
```

**路径 B：物理返回手势**
```
用户滑动手势返回
           │
           ▼
browser popstate 事件
           │
           ▼
handlePopState() 捕获
           │
           ▼
modalStack 有内容 ──> 调用 topModal.close()
                    （即 popPageInternal）
                    从 pageStack 弹出
                    从 modalStack 移除
```

两条路径最终都收敛到 `pageStack.pop()`，确保状态一致性。

#### 2.4.3 传统 Modal API

除 SlidePage 外，`overlayStore` 还直接管理三种轻量模态：

| 模态 | 状态字段 | 打开方法 | 关闭方法 | Z-Index |
|------|---------|---------|---------|---------|
| Prompt（输入框） | `promptConfig` | `openPrompt(config)` | `closePrompt()` | `z-dialog` (60) |
| ContextMenu（菜单） | `contextMenuConfig` | `openContextMenu(actions, title?)` | `closeContextMenu()` | `z-dialog` (60) |
| FullScreenEditor | `editorConfig` | `openEditor(config)` | `closeEditor()` | `z-viewer` (70) |

这三种模态同样通过 `registerModal`/`unregisterModal` 接入 Operation Aegis，因此物理返回键也能正确关闭它们。

## 3. 布局状态（layoutStore）

> 文件位置：`src/core/stores/layout.ts`

### 3.1 左右抽屉状态

`layoutStore` 管理两个布尔状态：`leftDrawerOpen` 与 `rightDrawerOpen`，分别控制左侧 Agent 导航抽屉和右侧通知中心抽屉。

#### 3.1.1 互斥策略

```ts
const setLeftDrawer = (open: boolean) => {
  if (!open) { leftDrawerOpen.value = false; unregisterModal('LeftDrawer'); return; }
  setRightDrawer(false);   // 打开左抽屉时强制关闭右抽屉
  leftDrawerOpen.value = true;
  registerModal('LeftDrawer', () => { leftDrawerOpen.value = false; });
};

const setRightDrawer = (open: boolean) => {
  if (!open) { rightDrawerOpen.value = false; unregisterModal('RightDrawer'); return; }
  setLeftDrawer(false);    // 打开右抽屉时强制关闭左抽屉
  rightDrawerOpen.value = true;
  registerModal('RightDrawer', () => { rightDrawerOpen.value = false; });
};
```

左右抽屉**不能同时打开**。这种设计避免了移动端屏幕空间被双向抽屉同时挤压的混乱局面。

#### 3.1.2 与 Operation Aegis 的集成

每个抽屉打开时都会向 `modalStack` 注册自身。因此：
- 物理返回键优先关闭当前打开的抽屉
- 抽屉关闭时自动 `unregisterModal`，同步浏览器历史栈

## 4. 主题状态（themeStore）

> 文件位置：`src/core/stores/theme.ts`

### 4.1 主题列表与当前主题

#### 4.1.1 主题数据来源

VCP Mobile 的主题不再通过传统 CSS 文件静态引入，而是采用 **Vite 动态导入 + TypeScript 主题模块** 的架构：

```ts
const themeModules = import.meta.glob('../../assets/themes/*.ts') 
  as Record<string, () => Promise<ThemeModule>>;
```

每个主题是一个独立的 `.ts` 文件，导出一个 `ThemeModule` 对象：

```ts
interface ThemeModule {
  meta: { fileName: string; name: string };
  variables: { dark: Record<string, string>; light: Record<string, string> };
  extraCss?: string;  // 非变量样式，如 .tool-bubble 的特殊样式
}
```

#### 4.1.2 主题加载时机

```
appLifecycleStore.bootstrap()
           │
           ▼
themeStore.initTheme()
           │
           ├──► applyThemeFile(savedTheme)   // 优先加载当前主题，确保首屏不闪烁
           │
           └──► requestIdleCallback(fetchThemes)  // 空闲时扫描全部主题元数据
```

- **首屏优化**：`initTheme` 先加载当前主题并立即注入 CSS 变量，确保壁纸和基础色瞬间呈现
- **后台扫描**：通过 `requestIdleCallback`（或 `setTimeout` 回退）在浏览器空闲时异步加载全部主题列表，用于设置页的主题选择器

#### 4.1.3 主题缓存与查找

```ts
const fileNameToLoader = new Map<string, () => Promise<ThemeModule>>();

const findThemeLoader = (fileName: string): (() => Promise<ThemeModule>) | undefined => {
  // Dev mode (HMR): 每次重新扫描，因为 Vite 可能已替换 loader
  // Prod mode: 优先读缓存
  for (const [path, loader] of Object.entries(themeModules)) {
    if (path.endsWith(tsFileName)) { cache and return; }
  }
};
```

生产环境下通过 `Map` 缓存 loader 避免重复遍历；开发模式下每次重新扫描以支持 HMR。

### 4.2 明暗模式解析

#### 4.2.1 三种模式

```ts
export type ThemeMode = 'light' | 'dark' | 'system';
```

| 模式 | 行为 |
|------|------|
| `light` | 强制使用 light 变量集，无视系统偏好 |
| `dark` | 强制使用 dark 变量集，无视系统偏好 |
| `system` | 监听 `prefers-color-scheme: dark`，跟随系统 |

#### 4.2.2 isDarkResolved 的计算逻辑

```ts
const applyTheme = (newMode: ThemeMode) => {
  const isDark =
    newMode === 'dark' ||
    (newMode === 'system' && window.matchMedia('(prefers-color-scheme: dark)').matches);

  isDarkResolved.value = isDark;
  document.documentElement.classList.toggle('dark', isDark);
  document.body.classList.toggle('light-theme', !isDark);
};
```

- 当 `mode` 为 `system` 时，实时查询 `matchMedia` 结果
- 系统主题变化通过 `mediaQuery.addEventListener('change', ...)` 监听，自动重新应用

#### 4.2.3 切换防抖

```ts
const MODE_SWITCH_DEBOUNCE_MS = 100;

const setMode = (newMode: ThemeMode) => {
  if (now - lastModeSwitchAt.value < 100) return;
  if (mode.value === newMode) return;
  lastModeSwitchAt.value = now;
  mode.value = newMode;
};
```

100ms 的防抖窗口防止用户在设置页快速点击导致的闪烁。

### 4.3 CSS 变量注入机制

#### 4.3.1 变量注入流程

```ts
const injectVariables = (vars: Record<string, string>) => {
  // 1. 清除旧主题的残留变量
  for (const key of lastAppliedVarKeys.value) {
    document.documentElement.style.removeProperty(key);
  }
  // 2. 写入新变量
  for (const [key, value] of Object.entries(vars)) {
    document.documentElement.style.setProperty(key, value);
  }
  // 3. 记录本次写入的 keys，供下次清理
  lastAppliedVarKeys.value = Object.keys(vars);
};
```

**为什么要清除旧变量？** 不同主题可能定义不同的变量集合（如某主题有 `--chat-wallpaper-light` 而另一主题没有）。不清除会导致变量泄漏，产生混合状态。

#### 4.3.2 extraCss 注入

```ts
let styleTag = document.getElementById('vcp-custom-theme');
if (!styleTag) {
  styleTag = document.createElement('style');
  styleTag.id = 'vcp-custom-theme';
  document.head.appendChild(styleTag);
}
styleTag.textContent = mod.extraCss || '';
```

`extraCss` 用于注入非变量类样式（如 `.tool-bubble` 的特殊边框），通过动态 `<style>` 标签追加到 `<head>`。

#### 4.3.3 与 uno.config.ts 的联动

> 文件位置：`uno.config.ts`

```ts

    zIndex: {
      content: '0',
```

| 语义名 | TS 常量 | CSS 变量 | UnoCSS 类 |
|--------|---------|---------|-----------|

## 5. 通知与 Toast（notificationStore）

> 文件位置：`src/core/stores/notification.ts`

### 5.1 通知队列与去重

#### 5.1.1 双轨存储

```ts
const historyList = ref<VcpNotification[]>([]);   // 通知中心历史（最多 100 条）
const activeToasts = ref<VcpNotification[]>();     // 当前悬浮显示的 Toast
const unreadCount = ref(0);                        // 未读计数
```

通知系统采用**双轨并行**架构：一条轨道进入历史记录列表（通知中心），另一条轨道弹出悬浮 Toast。两者可以独立控制。

#### 5.1.2 单例抑制逻辑（P0 优化）

```
addNotification(payload)
           │
           ▼
┌─────────────────────────────┐
│ 是否提供固定 id?            │
└─────────────┬───────────────┘
         Yes  │
              ▼
┌─────────────────────────────┐
│ 1. 检查 activeToasts        │
│    同 id 存在? ──► 原地更新  │
│    内容 + 时间戳，重置计时器 │
│    return                   │
└─────────────┬───────────────┘
         No   │
              ▼
┌─────────────────────────────┐
│ 2. 检查 historyList         │
│    同 id 在 30s 内出现过?   │
│    Yes ──► 只更新历史时间戳 │
│            抑制新 Toast      │
│    No  ──► 正常入队          │
└─────────────────────────────┘
```

单例抑制解决了以下问题：
- 同步连接状态反复变化时，Toast 不会疯狂堆叠
- 同一条消息可以在悬浮中**原地更新**（如从"连接中..."变为"已连接"）
- 30 秒冷却窗口避免已处理过的错误反复弹出

#### 5.1.3 历史列表上限与未读计数

```ts
if (historyList.value.length > 100) historyList.value.pop();
unreadCount.value++;
```

历史列表上限 100 条，超过时从尾部淘汰（最旧的）。每条非 `toastOnly` 通知入队时 `unreadCount` 增加。

### 5.2 Toast-only 模式

#### 5.2.1 三种显示模式

| 标志 | 行为 | 典型场景 |
|------|------|----------|
| 无标志（默认） | 同时入历史列表 + 弹出 Toast | 普通通知 |
| `toastOnly: true` | 仅弹出 Toast，不进入历史 | "设置更新成功"等瞬时反馈 |
| `historyOnly: true` | 仅进入历史，不弹出 Toast | 同步连接失败等不希望打断用户的错误 |
| `silent: true` | 完全静默，两者都不进入 | 内部状态同步 |

#### 5.2.2 Toast 抽屉抑制

```ts
if (!isDrawerOpen.value && !payload.historyOnly) {
  activeToasts.value.push(notification);
}
```

当通知中心抽屉打开时，新通知只进入历史列表，不再弹出悬浮 Toast。这避免了用户在浏览历史时被新 Toast 遮挡内容。

#### 5.2.3 自动移除与幽灵清理

**正常移除**：
```ts
if (notification.duration !== 0) {
  setTimeout(() => {
    activeToasts.value = activeToasts.value.filter(t => t.id !== id);
  }, notification.duration || 3000);
}
```

**幽灵清理**（兜底机制）：
```ts
const ghostCleanupInterval = setInterval(() => {
  activeToasts.value = activeToasts.value.filter(toast => {
    if (toast.duration === 0) return true;  // 审批类通知不清理
    const duration = toast.duration || 3000;
    return now - toast.timestamp < duration + 5000; // 冗余 5s
  });
}, 30000);
```

每 30 秒扫描一次，强制清理那些因 Vue 渲染异常或计时器失效而残留的"幽灵 Toast"。

### 5.3 与 VCP System Event 的集成

> 文件位置：`src/core/composables/useNotificationProcessor.ts`

#### 5.3.1 事件入口

`App.vue` 在 `onMounted` 中注册 Tauri 事件监听：

```ts
unlistenLog = await listen("vcp-system-event", (event: any) => {
  const payload = event.payload;
  const processed = processPayload(payload);
  if (processed && !processed.silent) {
    notificationStore.addNotification(processed);
  }
});
```

#### 5.3.2 后端 Payload 类型映射

`useNotificationProcessor` 将后端原始 JSON 解析为前端统一的 `VcpNotification` 结构：

| 后端类型 | 前端类型 | 处理逻辑 |
|----------|---------|----------|
| `vcp-log-status` | 静默 | 更新 `vcpStatus` 状态槽，不弹 Toast |
| `vcp-core-status` | `error` / 静默 | 错误时强制弹窗（`duration: 0`） |
| `vcp-log-message` | `tool` / `error` / `success` | 解析 tool_name、status、content、MaidName、时间戳 |
| `tool_approval_request` | `warning` | 生成审批标题与消息，附加 action 按钮（允许/拒绝） |
| `video_generation_status` | `info` | 解析时间戳与原始输出 |
| `daily_note_created` | `success` / `info` | 遗留兼容 |
| `connection_ack` | 默认 | 连接成功确认，通常被过滤规则拦截 |

#### 5.3.3 过滤引擎

```ts
const builtInRules: FilterRule[] = [
  { name: 'Heartbeat/Ping/Pong Noise Reduction', action: 'hide', ... },
  { name: 'Redundant Connection Success', action: 'hide', ... },
  { name: 'Important Error Duration Extension', action: 'show', duration: 15000 },
  { name: 'DistPluginManager Noise Reduction', action: 'hide', ... },
];
```

- **心跳降噪**：拦截所有 heartbeat/ping/pong 消息，避免后台保活消息淹没通知中心
- **冗余连接成功**：首次连接成功后，后续的重复确认被静默
- **错误延长**：错误类通知显示时长延长至 15 秒，确保用户有足够时间阅读

#### 5.3.4 工具审批 Action 回传

```ts
const executeAction = async (notificationId: string, action) => {
  if (item.type === 'warning' && item.rawPayload?.type === 'tool_approval_request') {
    const response = {
      type: 'tool_approval_response',
      data: { requestId, approved: action.value }
    };
    await invoke('send_vcp_log_message', { payload: response });
  }
};
```

用户点击"允许"或"拒绝"后，通过 `send_vcp_log_message` 将审批结果回传到 Rust 后端，完成端到端的工具权限管控。

## 8. 设计决策与注意事项

### 8.1 为何不使用 Vue Router 管理"页面"？

VCP Mobile 在架构上被设计为**单路由应用**（仅 `/#/chat`）。若使用 Vue Router 管理设置页、Agent 设置页等，会带来以下问题：
1. **路由切换动画复杂**：Vue Router 的 `<Transition>` 在移动端返回手势下难以实现从右向左滑入、从左向右滑出的自然体验
2. **状态丢失**：路由切换会导致组件卸载，表单草稿、滚动位置等本地状态丢失
3. **返回键语义混乱**：物理返回键在路由模式下会触发 `history.back()`，与覆盖层关闭逻辑冲突

SlidePage 覆盖层方案的优势：
- 组件**常驻 DOM**（`FeatureOverlays.vue` 中 SettingsView 等不使用 `v-if`），支持 leave 动画
- 返回键通过 Operation Aegis **精确控制**，不会误触路由导航
- 多层叠加自然（设置页之上再打开 Agent 设置页）

### 8.2 Operation Aegis 的 dummy state 机制

`window.history.pushState({ vcpRoot: true, vcpMain: true }, '')` 是一个**无 URL 变更**的历史记录注入。它的作用是：
- 当用户连续按返回键清空所有 modalStack 后，下一次 popstate 会弹出这个 dummy state
- 弹出 dummy state 后触发 `vcp-exit-requested`，进入双击退出逻辑
- `THE BOUNCE`（第 78 行）立即重新压入 dummy state，确保下一次返回键仍然触发 popstate 而非直接退出应用

### 8.3 主题系统的 Vite HMR 支持

```ts
if (import.meta.hot) {
  setTimeout(() => {
    if (currentTheme.value) {
      applyThemeFile(currentTheme.value);
    }
  }, 100);
}
```

开发模式下修改主题 `.ts` 文件时，Vite 会热替换模块并冒泡到 `theme.ts`。100ms 的 `setTimeout` 是为了等待 Vite 完成模块替换后再重新执行 `applyThemeFile`，实现**无刷新实时预览**。

### 8.4 抽屉的桌面端降级策略

```css
@media (min-width: 768px) {
  .vcp-drawer {
    position: relative;
    transform: translateX(0) !important;
    width: 280px;
    z-index: var(--layer-local);
  }
}
```

在 `md` 断点（768px）以上，左右抽屉从 `fixed` 绝对定位降级为 `relative` 相对布局，始终可见，且层级从 `z-drawer` 降为 `z-local`。这确保了在平板或桌面浏览器中，侧边栏作为常驻导航而非覆盖层存在。

### 8.5 通知中心的 `isDrawerOpen` 状态桥接

`RightSidebar.vue` 中通过 `watch(props.isOpen)` 将抽屉的打开状态同步到 `notificationStore.isDrawerOpen`：

```ts
watch(() => props.isOpen, (isOpen) => {
  store.isDrawerOpen = isOpen;
  if (isOpen) { store.markAllRead(); }
}, { immediate: true });
```

这种" props → store "的桥接看似冗余，实则必要：`notificationStore.addNotification()` 需要根据 `isDrawerOpen` 判断是否抑制 Toast，而 `layoutStore` 只管理抽屉开关，不感知通知业务。Store 间的直接解耦避免了循环依赖。
