# 06. 组合式函数与类型系统

## 1. 概述

### 1.1 领域定位

`src/core/composables/` 是 VCP Mobile 前端 Vue 3 层的**全局组合式函数（Composable）仓库**，承担跨组件复用逻辑的职责。与 `features/<domain>/composables/` 中按领域隔离的局部组合式函数不同，`core/composables/` 中的 15 个函数全部是**应用级全局能力**，被 `App.vue`、`ChatView.vue`、`BootScreen.vue` 等顶层组件直接消费，或作为基础设施被其他局部 Composable 依赖。

`src/core/types/` 与 `src/core/utils/` 分别提供**跨层共享类型契约**和**纯工具函数**，三者共同构成前端核心层的"类型-逻辑-工具"三角。

### 1.2 模块构成表

| 目录 | 文件数 | 总职责 |
|------|--------|--------|
| `src/core/composables/` | 15 | 全局状态ful/stateless逻辑复用单元 |
| `src/core/types/` | 2 | 前端类型定义，严格对齐 Rust DTO |
| `src/core/utils/` | 1 | 纯函数工具（AST渲染、缓存管理） |

### 1.4 核心设计原则

1. **功能共置（Co-location）与全局分层**：仅在逻辑被 2 个以上独立领域消费时，才升格到 `core/composables/`；否则留在 `features/<domain>/` 内部。
2. **Hybrid Pipeline**：内容处理采用"流式前端正则 + 静态后端 AST"双轨策略（`useContentProcessor`），流式场景避免 IPC 往返，静态场景依赖 Rust 解析器保证准确性。
3. **引用计数与生命周期对齐**：设备能力类（`useScreenKeeper`）使用显式引用计数，确保 Android Activity 重建后状态可恢复。
4. **状态机替代布尔标志**：`useChatScroll` 使用 4 状态场景机（`initial/following/free/loading-top`）替代杂乱的布尔组合，根治滚动回弹与加载错位 Bug。
5. **单层覆盖防御**：`useContentProcessor` 的 CSS 注入器将 `html/body/:root` 选择器强制降维到 `[data-message-id]` 气泡内部，防止用户生成内容污染全局样式。
6. **权限预申请策略**：涉及麦克风等敏感权限的 Composable（`useAudioRecorder`、`useSpeechRecognition`）在调用 Web API 前先通过 `plugin:vcp-mobile|request_android_permission` 触发原生弹窗，避免 WebView 权限弹窗被系统拦截。

## 3. 重点 Composable 详解

### 3.1 useModalHistory — 模态历史栈

**源码位置**：`src/core/composables/useModalHistory.ts`

#### 3.1.1 问题背景

Android 物理返回键/侧滑手势在 Tauri WebView 中默认映射为 `window.history.back()`。若应用为单页 Hash 路由且只有 `/chat` 一条主路由，返回手势会直接退出应用。`useModalHistory` 通过**历史栈劫持**将返回手势转化为"逐层关闭覆盖层"。

#### 3.1.2 状态机

```
                    ┌─────────────┐
         popstate   │    IDLE     │◄────────────────┐
        ┌──────────►│  (正常状态)  │                 │
        │           └──────┬──────┘                 │
        │                  │ registerModal          │
        │                  ▼                        │
        │           ┌─────────────┐   closeTopModal │
        └───────────┤POPSTATE_    │─────────────────┘
                    │ HANDLING    │
                    └──────┬──────┘
                           │ unregisterModal
                           ▼
                    ┌─────────────┐
                    │ INTERNAL_   │─────► history.back()
                    │   BACK      │
                    └─────────────┘
```

#### 3.1.3 核心 API

| API | 签名 | 职责 |
|-----|------|------|
| `initRootHistory` | `() => void` | App 启动时压入带有 `vcpMain: true` 标记的 dummy state，确保栈底始终有拦截锚点 |
| `registerModal` | `(id: string, closeHandler: () => void) => void` | 模态框打开时调用：压入 `history.state` + 加入 LIFO 栈 |
| `unregisterModal` | `(id: string) => void` | 模态框 UI 关闭时调用：触发 `history.back()` 并移除栈记录 |
| `closeTopModal` | `() => boolean` | 程序化关闭栈顶模态（如点击遮罩层），直接从栈中移除并调用 `closeHandler` |
| `modalStackLength` | `() => number` | 返回当前栈深度 |
| `cleanupModalHistory` | `() => void` | 清理全局 `popstate` 监听器（测试/热重载用） |

#### 3.1.4 与 Layout Store 的协作

与 `useModalHistory` 直接协作的是 **`useLayoutStore`**（`src/core/stores/layout.ts`），而非 `overlayStore`。`useLayoutStore` 将左右抽屉的开关语义映射为模态注册/注销：

```typescript
// src/core/stores/layout.ts (节选)
const setLeftDrawer = (open: boolean) => {
  if (!open) {
    leftDrawerOpen.value = false;
    unregisterModal('LeftDrawer');
    return;
  }
  setRightDrawer(false);
  leftDrawerOpen.value = true;
  registerModal('LeftDrawer', () => { leftDrawerOpen.value = false; });
};
```

当用户侧滑或点击物理返回键时，`handlePopState` 按以下优先级处理：

1. 若 `state === 'INTERNAL_BACK'`（由 `unregisterModal` 触发），直接忽略。
2. 若当前路由非 `/chat` 或 `/`，交给 `vue-router` 处理真实页面返回。
3. 若 `modalStack` 非空，关闭栈顶模态框。
4. 若已到栈底，触发 `vcp-exit-requested` 自定义事件（由 `App.vue` 实现双击退出）。

#### 3.1.5 THE BOUNCE 机制

当返回手势到达栈底时，`handlePopState` 会重新 `pushState` 注入 dummy state，确保下一次返回手势依然能触发 `popstate` 事件，而不是直接杀死应用进程：

```typescript
window.dispatchEvent(new CustomEvent('vcp-exit-requested'));
window.history.pushState({ vcpRoot: true, vcpMain: true }, '');
```

#### 3.1.6 自校准状态检测

`initRootHistory` 在每次调用时检查当前 `history.state` 是否带有 `vcpMain` 标记。若不存在（如 WebView 重建后状态丢失），立即压入新的 dummy state，确保防护盾始终有效：

```typescript
const state = window.history.state;
if (!state || !state.vcpMain) {
  window.history.pushState({ vcpRoot: true, vcpMain: true }, '');
}
```

```
              ▼                         ▼
      ┌───────────────┐        ┌────────────────┐
      │ 前端正则替换   │        │invoke(process_ │
      │ THOUGHT/TOOL  │        │message_content)│
      │ ROLE_DIVIDER  │        └───────┬────────┘
      └───────┬───────┘                │
              │                        ▼
              │               ┌────────────────┐
              │               │ Rust 后端 AST  │
```

#### 3.2.2 流式模式正则替换规则

| 正则模式 | 替换目标 | 说明 |
|----------|----------|------|
| `\[--- VCP元思考链: ... ---\]...\[--- 元思考链结束 ---\]` | 折叠式思考块（灰色背景 + 🧠 图标） | 含主题名提取，默认主题为"元思考链" |
| `<think(ing)?>...</think(ing)?>` | 标准思考块 | 兼容 `<think>` / `<thinking>` 标签 |
| `<<<\[TOOL_REQUEST\]>>>` | 工具调用块（蓝色/琥珀色边框） | 含 `DailyNote` 特殊样式分支 |
| `<<<\[(END_)?ROLE_DIVIDE_(SYSTEM\|ASSISTANT\|USER)\]>>>` | 角色分界指示器 | 流式模式下保留占位，改善 UX |

> **注意**：MATH 块（`$...$` / `$$...$$`）的正则替换已完全迁移至 Rust 后端 `parse_content`，流式模式下由 `MessageRenderer` 直接消费后端推送的 AST blocks，不再做前端正则替换。

#### 3.2.3 CSS 注入与沙箱防御

静态模式下，若后端返回 `type: "style"` 的 `ContentBlock`，`injectScopedCss` 会将 CSS 强制限定在 `[data-message-id="{id}"]` 内部：

```typescript
const scopeSelector = `[data-message-id="${messageId}"]`;
const scopedCss = css.replace(
  /(^|\})\s*([^{]+)\s*\{/g,
  (_, prefix, selectors) => {
    const scopedSelectors = selectors
      .split(",")
      .map((s: string) => {
        const sel = s.trim();
        // 允许动画关键帧直接通过
        if (sel.match(/^(@|from|to|\d+%)/)) return sel;
        // 核心修复：绝对禁止 html, body 污染全局，强行降维到当前气泡
        if (sel.match(/^(html|body|:root)/i)) return `${scopeSelector}`;
        if (sel === "#vcp-root") return `${scopeSelector} ${sel}`;
        if (sel.startsWith("::") || sel.startsWith(":"))
          return `${scopeSelector}${sel}`;
        return `${scopeSelector} ${sel}`;
      })
      .join(", ");
    return `${prefix} ${scopedSelectors} {`;
  },
);
```

这确保 AI 生成的 HTML 预览消息不会污染全局主题或破坏应用布局。同时 `removeScopedCss` 在消息卸载时清理对应的 `<style>` 标签，防止内存泄漏。

#### 3.2.4 静态模式的后处理流程

```typescript
// 2. Call Rust backend to parse the text into AST blocks (Static Mode)
let blocks: ContentBlock[] = [];
try {
  blocks = await invoke("process_message_content", { content: text });

  // 3. Post-process blocks for side-effects (CSS injection)
  if (options.messageId) {
    for (const block of blocks) {
      if (block.type === "style") {
        injectScopedCss(block.content, options.messageId);
      }
    }
  }
} catch (error) {
  console.error("[useContentProcessor] Failed to parse content via Rust:", error);
  blocks = [{ type: "markdown", content: text }];
}
```

失败时优雅降级为单一大 `markdown` block，确保用户始终能看到内容。

### 3.3 useChatScroll — 聊天滚动控制

**源码位置**：`src/core/composables/useChatScroll.ts`

#### 3.3.1 场景状态机

```
                    ┌─────────────┐
                    │   initial   │◄── 话题切换/消息清空
                    │  (首屏加载)  │
                    └──────┬──────┘
                           │ messageCount > 0 && !loading
                           ▼
              ┌────────────────────────┐
              │       following        │◄── 用户滚动到底部
              │  (自动跟随新消息置底)   │
              └───────┬────────┬───────┘
                      │        │
        scroll 离开底部│        │scroll 到底部
                      ▼        │
              ┌─────────────┐  │
              │     free    │──┘
              │  (自由浏览)  │
              └──────┬──────┘
                     │ scrollTop < 100 && hasMoreHistory
                     ▼
              ┌─────────────┐
              │ loading-top │
              │(加载历史分页)│
              └──────┬──────┘
                     │ 加载完成
                     └──────► restore anchor → free
                              no anchor → following
```

#### 3.3.2 锚定元素机制

加载历史分页前，`prepareLoadAnchor` 记录当前视口最顶部可见消息的 `messageId` 及其相对于容器顶部的偏移量。加载完成后，`restoreScrollByAnchor` 根据该元素的新 `offsetTop` 恢复滚动位置，确保用户视觉不跳动：

```typescript
const prepareLoadAnchor = () => {
  const list = messageListRef.value;
  if (!list) return;
  const messages = list.querySelectorAll("[data-message-id]");
  const listRect = list.getBoundingClientRect();
  for (const el of Array.from(messages)) {
    const rect = el.getBoundingClientRect();
    if (rect.top >= listRect.top) {
      loadAnchor = {
        messageId: el.getAttribute("data-message-id")!,
        offsetFromTop: rect.top - listRect.top,
      };
      break;
    }
  }
};

const restoreScrollByAnchor = () => {
  if (!loadAnchor || !messageListRef.value) return;
  const el = messageListRef.value.querySelector(`[data-message-id="${loadAnchor.messageId}"]`);
  if (el) {
    messageListRef.value.scrollTop = (el as HTMLElement).offsetTop - loadAnchor.offsetFromTop;
  }
  loadAnchor = null;
};
```

#### 3.3.3 高性能保护罩

- **首屏渲染保护罩** (`isInitialRendering`)：首屏加载期间（头像、图片异步排版导致高度持续增长），屏蔽 scroll 事件的状态机迁移，防止状态机提前叛逃为 `free`。
- **物理高度守卫**：`handleContentChange` 中若 `scrollHeight === lastScrollHeight`，直接拦截退出，避免 RAF 高频回调浪费 CPU。
- **200ms 防抖续载**：图片/头像加载引起的高度增长会重置防抖定时器，推迟自动续载判定，等待布局彻底稳定：

```typescript
const triggerAutoLoadMoreWithDebounce = () => {
  if (loadMoreDebounceId) {
    clearTimeout(loadMoreDebounceId);
  }
  loadMoreDebounceId = setTimeout(() => {
    loadMoreDebounceId = null;
    evaluateAutoLoadMore();
  }, 200) as unknown as number;
};
```

#### 3.3.4 ResizeObserver + RAF 双重节流

```
DOM 尺寸变化
    │
    ▼
ResizeObserver 回调
    │
    ▼
requestAnimationFrame (合并单帧高频变动)
    │
    ▼
handleContentChange (场景分发)
```

优先监听 `.messages-inner-container`（无条件渲染的内部容器），若不存在则降级监听外层列表。这种设计确保 Observer 目标始终存在，避免空引用错误。

#### 3.3.5 监听 isLoadingHistory 兜底

`watch(isLoadingHistory)` 提供双重保险：当 `ResizeObserver` 因某些原因未及时触发时，在 `loading` 变为 `false` 后，通过 `nextTick + 双重 RAF` 确保布局稳定后再执行一次 `handleContentChange`：

```typescript
watch(isLoadingHistory, async (loading) => {
  if (loading) return;
```

### 3.4 useNotificationProcessor — 通知处理

**源码位置**：`src/core/composables/useNotificationProcessor.ts`

#### 3.4.1 输入与输出

**输入**：Rust 后端通过 Tauri Event `vcp-system-event` 发射的原始 JSON payload，可能携带多种子类型：

| payload.type | 来源 | 说明 |
|--------------|------|------|
| `vcp-log-status` | `vcp_log_service.rs` | 连接底层状态指示（已静默，不渲染 Toast） |
| `vcp-core-status` | `lifecycle_manager.rs` | 核心引擎状态变更（ready/error） |
| `vcp-log-message` | `vcp_log_service.rs` | VCP 工具执行日志、连接事件 |
| `tool_approval_request` | `vcp_log_service.rs` | 工具调用审批请求 |
| `video_generation_status` | `vcp_log_service.rs` | 视频生成进度 |
| `daily_note_created` | 遗留兼容 | 日记创建状态 |

**输出**：`Partial<VcpNotification>` 对象，由 `useNotificationStore` 消费，决定是否展示 Toast、进入历史列表或完全静默。

#### 3.4.2 处理流程

```
vcp-system-event payload
        │
        ▼
┌───────────────┐
│  类型路由分支  │
│ (vcp-log-msg /│
│ tool_approval /│
│  ... )        │
└───────┬───────┘
        │
        ▼
┌───────────────┐
│  字段提取与    │
│  规范化(title,│
│  message,type) │
└───────┬───────┘
        │
        ▼
┌───────────────┐
│ 全局过滤引擎   │
│ (降噪/增强)   │
└───────┬───────┘
        │
   ┌────┴────┐
   ▼         ▼
 hide      show ──► notificationStore.addNotification()
```

#### 3.4.3 内置过滤规则

| 规则名 | 命中条件 | 动作 |
|--------|----------|------|
| Heartbeat/Ping/Pong Noise Reduction | `type` 或内容含 heartbeat/ping/pong | `hide` |
| Redundant Connection Success | `connection_ack` + successful | `hide` |
| Important Error Duration Extension | title 含 error 或 message 含 failed | `show`, duration=15000ms |
| DistPluginManager Noise Reduction | source=DistPluginManager + heartbeat | `hide` |

过滤规则按顺序匹配，命中即返回，后续规则不再执行。

#### 3.4.4 VCP 日志深层解析

`vcp-log-message` 类型的 payload 往往包含嵌套 JSON。`processPayload` 执行以下解析：

1. **时间戳提取**：若 `inner.timestamp` 存在且长度 >= 16，提取 `HH:MM` 附加到标题。
2. **MaidName 提取**：若存在 `inner.MaidName`，追加到标题。
3. **original_plugin_output 提取**：优先展示结构化输出，对象类型做 `JSON.stringify(_, null, 2)` 格式化。
4. **DailyNote Fallback**：若 `DailyNote` 成功但无有效输出，显示固定文案"日记内容已成功记录到本地知识库"。
5. **错误模式处理**：若 `status === 'error'` 且内容含 `{`，尝试提取嵌套 JSON 中的 `plugin_error` / `error` / `message`。

#### 3.4.5 单例抑制与状态栏分离

- **单例抑制**：固定 ID（如 `vcp_sync_connection_status`）的通知若在 30s 冷却期内重复到达，仅更新历史记录时间戳，不弹出新 Toast。
- **状态栏分离**：`vcp-log-status` 和 `vcp-core-status` 不再渲染为悬浮 Toast，而是分别写入 `notificationStore.vcpStatus` 和 `notificationStore.vcpCoreStatus`，由状态栏组件独立订阅。
- **Sync 错误静默**：`vcp_sync_connection_status` + `error` 设置为 `historyOnly`，仅进入通知中心，不弹出悬浮 Toast，避免同步重连期间频繁打扰用户。

#### 3.4.6 审批请求处理

`tool_approval_request` 类型生成带操作按钮的通知：

```typescript
actions = [
  { label: '允许', value: true, color: 'bg-green-500 shadow-lg shadow-green-500/20' },
  { label: '拒绝', value: false, color: 'bg-red-500 shadow-lg shadow-red-500/20' }
];
duration = 0; // 永不自动消失
```

用户点击后，`notificationStore.executeAction` 通过 `send_vcp_log_message` Command 将审批结果回传后端。

### 3.5 useKeyboardInsets — 键盘适配

**源码位置**：`src/core/composables/useKeyboardInsets.ts`

#### 3.5.1 三层降级策略

Tauri Android WebView 中 `visualViewport` 在键盘弹起时不会正确更新（tauri-apps/tauri#10631、#13479），因此必须构建多层降级：

```
┌─────────────────────────────────────────────────────────────┐
│ 策略 1: Android 原生注入事件 (vcp-keyboard-inset)            │
│         通过 evaluateJavascript 注入 CustomEvent             │
│         最可靠，返回物理像素，需除以 DPR                     │
│  ─────────────────────────────────────────────────────────  │
│ 策略 2: Virtual Keyboard API (Chrome 94+)                   │
│         navigator.virtualKeyboard.overlaysContent = true     │
│         geometrychange 事件直接给出逻辑高度                   │
│  ─────────────────────────────────────────────────────────  │
│ 策略 3: focusin/focusout + scrollHeight 差值估算            │
│         focusin 后延迟 300ms 测量 scrollHeight - innerHeight │
│         focusout 后延迟 150ms 检查 activeElement 是否仍编辑   │
└─────────────────────────────────────────────────────────────┘
```

#### 3.5.2 防闪烁设计

- `focusin` 延迟 200ms 启动 fallback：若原生事件已到达则跳过估算，避免 footer 双重跳动。
- `focusout` 延迟 150ms 检查：处理从一个 `input` 切换到另一个 `input` 的场景，防止键盘高度先归零再恢复的闪烁。

#### 3.5.3 返回值

```typescript
interface UseKeyboardInsetsReturn {
  keyboardHeight: Ref<number>;    // 键盘高度（CSS 逻辑像素）
  isKeyboardOpen: Ref<boolean>;   // 键盘是否可见
  safeAreaBottom: Ref<number>;    // 底部安全区高度
  forceRecalculate: () => void;   // 强制重算（兜底）
}
```

```typescript
const LAST_CHECK_KEY = 'vcp_last_update_check';
```

`download_update` 使用 Tauri v2 的 `Channel<DownloadProgress>` 实现从 Rust 到前端的实时字节流传输，前端将其格式化为 `MB/KB` 并计算百分比：

```typescript
const formatBytes = (bytes: number) => {
  if (bytes === 0) return '0 B';
  const k = 1024;
  const sizes = ['B', 'KB', 'MB', 'GB'];
  const i = Math.floor(Math.log(bytes) / Math.log(k));
  return parseFloat((bytes / Math.pow(k, i)).toFixed(2)) + ' ' + sizes[i];
};
```

下载进度 Toast 使用固定 ID `vcp_update_download_progress`，由 `notificationStore` 的单例抑制机制实现原地更新，避免进度通知堆积。

## 5. 工具函数（core/utils/）

### 5.1 工具函数一览

**源码位置**：`src/core/utils/astRenderer.ts`

| 函数 | 签名 | 职责 |
|------|------|------|
| `renderMarkdownNodes` | `(nodes, messageId, blockHash?) => string` | 将 Rust 预渲染的 AST 节点树转换为 HTML 字符串，带 LRU 缓存 |
| `clearHtmlCache` | `() => void` | 全局清理 AST HTML 缓存（重建/同步后调用） |
| `clearMessageCache` | `(messageId: string) => void` | 清理单条消息的 AST 缓存（消息编辑后调用） |
| `escapeHtml` | `(text: string) => string` | HTML 实体转义（`& < > " '`） |

### 5.2 使用场景

#### 5.2.1 AST 渲染缓存

`renderMarkdownNodes` 使用 `Map<string, string>` 作为缓存，键生成策略：
- 若后端提供 `blockHash`，使用 `${messageId}:${blockHash}`（精确且稳定）
- 否则回退到 `${messageId}:len-${nodes.length}`（粗略，依赖后端逐步推进 hash 覆盖）

缓存上限为 **500 条**，超限后整表清空（实际命中模式为批量命中/失效，LRU 精细置换的收益不高）。

#### 5.2.2 代码高亮兼容层

`renderNode` 在渲染 `code_block` 时兼容旧 AST 的嵌套 `<pre><code>` 结构：若 `highlighted_html` 内部再包裹一层 `<pre>`，则提取单层，防止样式异常：

```typescript
const nestedPreMatch = html.match(/<pre[^>]*>\s*<code>([\s\S]*?)<\/code>\s*<\/pre>/i);
if (nestedPreMatch && nestedPreMatch[1].trim().startsWith('<pre')) {
  const innerMatch = nestedPreMatch[1].match(/<pre[^>]*>([\s\S]*?)<\/pre>/i);
  if (innerMatch) {
    html = `<pre class="vcp-code-block vcp-scrollable">${innerMatch[1]}</pre>`;
  }
}
```

#### 5.2.3 Asset 路径转换

`link` 和 `image` 类型的 `InlineNode` 若标记 `needs_asset_conversion: true`，则通过 `convertFileSrc()` 将 Rust 返回的本地文件路径转换为 WebView 可访问的 `asset://` URL：

```typescript
const href = node.needs_asset_conversion && node.href
  ? convertFileSrc(node.href)
  : escapeHtml(node.href || '');
```

#### 5.2.4 行内数学公式渲染

`inline_math` 节点根据 `display_mode` 决定渲染为块级（`vcp-math-block`）或行内（`vcp-math-inline`），并将 LaTeX 源码写入 `data-latex` 属性，供后续 MathJax/KaTeX 渲染：

```typescript
const cls = isDisplay ? 'vcp-math-block no-swipe' : 'vcp-math-inline no-swipe';
return `<span class="${cls}" data-latex="${escapeHtml(node.content || '')}">${escapeHtml(node.content || '')}</span>`;
```

## 7. 设计决策与注意事项

1. **为什么 `useModalHistory` 不直接使用 `overlayStore`，而是与 `useLayoutStore` 耦合？**
   历史原因：`overlayStore` 管理 SlidePage、BottomSheet、Dialog 等语义化覆盖层；而左右侧边栏（Drawer）是布局级组件，其开关状态由 `useLayoutStore` 独立维护。`useLayoutStore` 内部引入 `useModalHistory` 的 `registerModal`/`unregisterModal`，将 Drawer 也纳入返回手势管理体系，但保持 Store 间的单向依赖（Layout → ModalHistory，无反向依赖）。

2. **为什么 `useChatScroll` 放弃 IntersectionObserver？**
   `IntersectionObserver` 的 `rootMargin` 在动态加载历史场景下过于模糊，且 `nextTick` 无法准确预测图片/头像异步加载后的布局稳定时机。改用 `ResizeObserver + RAF + 物理高度守卫` 后，滚动回弹 Bug 完全消失，CPU 占用显著降低。

3. **为什么 `useContentProcessor` 的流式模式仍保留前端正则？**
   流式场景下每条增量都进行 IPC 往返（`invoke("process_message_content")`）的延迟不可接受（>50ms）。前端轻量正则可在 1ms 内完成 `THOUGHT`/`TOOL` 占位渲染，静态模式再交由 Rust 做精确 AST 解析。

4. **为什么 `useSpeechRecognition` 同时维护原生和 Whisper 两条路径？**
   国内 Android 设备普遍缺少 Google 语音服务框架，`SpeechRecognition` API 会抛出 `service-not-allowed`。Whisper 云端降级路径确保语音输入在所有设备上可用，但依赖用户配置 `vcpServerUrl` + `vcpApiKey`。

5. **关于 `useDocumentProcessor` 的职责边界**
   该 Composable 是**遗留兼容层**。当前实际生产中 PDF/DOCX/XLSX/PPTX 的文本提取已全部迁移至 Rust 后端（`file_extractor.rs`），前端仅保留 txt/md/csv/json 的简单读取能力。

6. **表情包修复为什么使用 `window.__vcpFixEmoticon` 全局函数？**
   `img` 标签的 `onerror` 事件处理器需要在 HTML 字符串中以内联形式引用。将修复逻辑挂载到 `window` 上，使得 Rust 后端渲染的 HTML 中可以直接写 `onerror="window.__vcpFixEmoticon(this)"`，无需前端再遍历 DOM 绑定事件。

7. **useUpdateDownloader 的 Channel 进度为什么是 ToastOnly？**
   下载进度通知使用 `toastOnly: true`，确保进度条仅作为悬浮 Toast 展示，不进入通知中心历史列表，避免历史列表被大量进度更新污染。

8. **useKeyboardInsets 为什么需要 focusout 延迟 150ms？**
   在表单场景中，用户可能从一个输入框切换到另一个输入框。若 focusout 立即将键盘高度归零，会造成 footer 短暂下沉再弹起的闪烁。150ms 延迟足够让新的 focusin 事件到达并重新设置高度。

## 10. 常见问题与排障指南

### Q1: 返回手势无法关闭侧边栏，直接退出应用？
**排查**：检查 `App.vue` 是否在 `onMounted` 中调用了 `initRootHistory()`；检查 `useLayoutStore` 是否正确调用了 `registerModal`/`unregisterModal`。

### Q2: 聊天滚动频繁回弹或自动加载历史失效？
**排查**：检查 `messageListRef` 是否正确绑定到滚动容器；确认消息元素带有 `data-message-id` 属性（锚定机制依赖此属性）。

### Q3: 键盘弹出时底部输入框被遮挡？
**排查**：确认 Android 插件的 `KeyboardInsetsManager` 正常运行；检查是否有其他 CSS（如 `100vh`）与键盘 insets 冲突。

### Q4: 语音输入按钮点击无响应？
**排查**：检查 `microphone` 权限是否已授予；检查 `settingsStore` 中是否配置了 `vcpServerUrl` 和 `vcpApiKey`（Whisper 降级模式需要）。

### Q5: 通知中心堆积大量 heartbeat/ping 消息？
**排查**：检查 `useNotificationProcessor` 是否被正确初始化；确认 `checkMessageFilter` 中的 Heartbeat Noise Reduction 规则未被意外移除。

## 11. 术语速查表

| 术语 | 说明 |
|------|------|
| **Aurora** | VCP 流式管道的语义沉淀机制，将流式增量聚合为稳定的 `stableBlocks` + 尾随 `tailBlock` |
| **ContentBlock** | 消息内容的结构化块，类型包括 markdown、tool-use、thought、style 等 |
| **THE BOUNCE** | `useModalHistory` 在栈底重新注入 dummy history state 的机制，防止返回手势直接杀死应用 |
| **Hybrid Pipeline** | 内容处理双轨策略：流式走前端正则，静态走 Rust AST |
| **MessageShell** | 消息外壳预计算属性（头像色、气泡边框、显示名），避免渲染时重复计算 |
| **StreamBlock** | 流式增量块，字段较 ContentBlock 精简，用于高频增量推送 |
| **Tail Block** | Aurora 管道中的尾随未稳定块，高频变动但不触发整消息重渲染 |
| **VCP System Event** | Rust 后端向前端发射的统一事件通道，payload 通过 `type` 字段自描述 |
| **Ephemeral State** | 纯前端运行时状态（如 `tailContent`、`isThinking`），不进行持久化 |
| **Asset Conversion** | 将 Rust 本地文件路径通过 `convertFileSrc` 转换为 WebView 可用的 `asset://` URL |
| **DPR** | Device Pixel Ratio，设备物理像素与 CSS 逻辑像素之比 |
| **Raw JNI** | Rust 侧通过 `jni` crate 直接调用 Android Activity 方法，绕过 Tauri 标准插件通道 |
| **Channel** | Tauri v2 的前后端双向流式通信机制，用于下载进度等大频率事件 |
