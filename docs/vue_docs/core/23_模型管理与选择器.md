# 23. 模型管理与选择器

## 1. 概述

### 1.1 领域定位

模型管理与选择器是 VCP Mobile 前端**基础设施层**的核心子系统，负责 LLM 模型元数据的缓存治理、收藏/热门交互排序，以及统一的模型选择 UI（`ModelSelector.vue`）。该子系统横跨 Pinia 状态管理、Vue 组件渲染与 Tauri IPC 三层，是 Agent 设置、群组设置、话题总结等多个功能模块的共享依赖。

领域边界严格限定为：

- **模型列表缓存与刷新**：从 Rust 后端获取模型元数据，实施 10 分钟 TTL 与锁频防护
- **收藏与热门系统**：维护用户收藏模型 ID 列表与后端使用统计驱动的热模型排行
- **统一选择器 UI**：提供 BottomSheet 风格的模型选择交互，支持搜索、厂商过滤、虚拟滚动

该子系统**不**涉及：
- 模型推理与参数调优（由 `agent_chat_application_service.rs` 负责）
- 模型权重文件管理
- 服务端模型可用性探测（由 Rust 端 `refresh_models` 负责网络请求）

### 1.2 模块构成表

| 文件 | 行数 | 职责 |
|------|------|------|
| `src/core/stores/modelStore.ts` | 154 | Pinia Composition API 风格的模型状态管理：缓存、排序、收藏交互 |
| `src/components/ModelSelector.vue` | 420 | 全局 BottomSheet 模型选择器：虚拟滚动、搜索、手势下拉、厂商过滤 |

> **被消费者**：
> - `src/features/agent/AgentSettingsView.vue` — Agent 模型参数选择
> - `src/features/agent/GroupSettingsView.vue` — 群组统一模型选择
> - `src/features/settings/SettingsView.vue` — 话题总结专用模型选择

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **缓存优先** | 前端与后端均实施双层缓存（前端内存 `ref` + 后端 `RwLock` + SQLite），避免重复网络请求 |
| **锁频防护** | `isLoading` 标志 + 10 分钟 TTL，防止用户快速点击导致的并发刷新与无意义网络开销 |
| **排序即产品** | `sortedModels` computed 将收藏、热门、字母序的优先级编码为可直接渲染的顺序，减少前端过滤逻辑 |
| **虚拟滚动** | 模型列表可能达数百条，使用 `useVirtualList` 固定高度虚拟滚动，保证打开 BottomSheet 的零延迟 |
| **手势原生感** | 顶部拉手条支持下拉关闭（120px 动力学阈值），配合 `translate3d` GPU 加速与遮罩透明度联动 |

## 2. 模型状态（modelStore）

`modelStore` 是 VCP Mobile 前端模型数据的**唯一可信源**（Single Source of Truth），采用 Pinia Composition API 风格定义。

> 文件位置：`src/core/stores/modelStore.ts`

### 2.1 核心状态字段

| 字段 | 类型 | 初始值 | 说明 |
|------|------|--------|------|
| `models` | `Ref<ModelInfo[]>` | `[]` | 完整模型列表（来自后端缓存或刷新） |
| `hotModels` | `Ref<string[]>` | `[]` | 热门模型 ID 列表，按使用频次降序 |
| `favorites` | `Ref<string[]>` | `[]` | 收藏模型 ID 列表 |
| `isLoading` | `Ref<boolean>` | `false` | 锁频标志，防止并发请求 |
| `lastRefreshed` | `Ref<number>` | `0` | 上次成功刷新时间戳（毫秒） |

```typescript
// src/core/stores/modelStore.ts 第 14–19 行
const models = ref<ModelInfo[]>([]);
const hotModels = ref<string[]>([]);
const favorites = ref<string[]>([]);
const isLoading = ref(false);
const lastRefreshed = ref(0);
```

`ModelInfo` 接口严格对应 Rust 端结构与 OpenAI `/v1/models` 响应格式：

```typescript
// src/core/stores/modelStore.ts 第 6–11 行
export interface ModelInfo {
  id: string;        // 模型唯一标识，如 "gemini-2.5-flash"
  object: string;    // 固定为 "model"
  created: number;   // Unix 时间戳（秒）
  owned_by: string;  // 提供方，如 "google"、"openai"
}
```

### 2.2 模型列表获取与缓存

`fetchModels(force?)` 是模型数据加载的**唯一入口**，实现了多级缓存策略与锁频防护。

> 函数位置：`src/core/stores/modelStore.ts` 第 44–98 行

**缓存命中判断逻辑**：

```
输入: force (boolean)
       │
       ▼
┌─────────────────────┐
│ isLoading == true?  │──> 是 → 直接返回（锁频防护）
└──────────┬──────────┘
           ▼
┌─────────────────────────────────────────────┐
│ !force && models.length > 0 &&              │──> 是 → 直接返回（10min TTL）
│ Date.now() - lastRefreshed < 600_000        │
└──────────┬───────────────┬──────────────────┘
      No   │               │ Yes
           ▼               ▼
    继续加载          跳过网络请求
```

**加载时序（三层渐进）**：

1. **L1 缓存预热**：若 `models` 为空，先调用 `get_cached_models` 快速填充内存，实现"秒开"
2. **L2 条件刷新**：仅在 `force = true` 或 `models` 仍为空时，调用 `refresh_models` 从服务器拉取
3. **元数据并行加载**：无论是否刷新网络，均并行调用 `get_hot_models` 与 `get_favorite_models`

```typescript
// src/core/stores/modelStore.ts 第 54–77 行
// 先尝试获取缓存
if (models.value.length === 0) {
  models.value = await invoke<ModelInfo[]>('get_cached_models');
}

// 只有在明确要求或没缓存时才刷新
if (force || models.value.length === 0) {
  models.value = await invoke<ModelInfo[]>('refresh_models');
  lastRefreshed.value = Date.now();
  // ... 成功/失败通知
}

await Promise.all([
  fetchHotModels(),
  fetchFavorites(),
]);
```

### 2.3 收藏与热门系统

收藏与热门是两个独立的数据维度，分别持久化在 Rust 后端的不同表中：

| 维度 | 后端表 | 前端状态 | 排序权重 |
|------|--------|----------|----------|
| **收藏** | `model_favorites` | `favorites: string[]` | 最高（二进制：1/0） |
| **热门** | `model_usage_stats` | `hotModels: string[]` | 次高（按 index 排序） |
| **其他** | — | — | 最低（字母序） |

**收藏切换**（`toggleFavorite`）：

```typescript
// src/core/stores/modelStore.ts 第 117–128 行
const toggleFavorite = async (modelId: string) => {
  const isFav = await invoke<boolean>('toggle_favorite_model', { modelId });
  if (isFav) {
    if (!favorites.value.includes(modelId)) favorites.value.push(modelId);
  } else {
    favorites.value = favorites.value.filter(id => id !== modelId);
  }
};
```

> 注意：前端在调用成功后立即乐观更新本地 `favorites` 数组，无需等待下次全量拉取。

**使用记录**（`recordUsage`）：

```typescript
// src/core/stores/modelStore.ts 第 130–138 行
const recordUsage = async (modelId: string) => {
  await invoke('record_model_usage', { modelId });
  fetchHotModels(); // 异步更新热门列表
};
```

当前 `recordUsage` 在聊天流程中由后端自动调用，前端保留此接口供未来扩展（如手动标记常用模型）。

### 2.4 综合排序计算属性（sortedModels）

`sortedModels` 是 `modelStore` 最核心的计算属性，将收藏、热门、字母序编码为单一的排序规则，供 `ModelSelector.vue` 直接消费。

> 定义位置：`src/core/stores/modelStore.ts` 第 24–39 行

```typescript
const sortedModels = computed(() => {
  return [...models.value].sort((a, b) => {
    // 1. 收藏优先（二进制权重）
    const aFav = favorites.value.includes(a.id) ? 1 : 0;
    const bFav = favorites.value.includes(b.id) ? 1 : 0;
    if (aFav !== bFav) return bFav - aFav;

    // 2. 热门次优先（按 hotModels 数组 index 排序）
    const aHot = hotModels.value.indexOf(a.id);
    const bHot = hotModels.value.indexOf(b.id);
    if (aHot !== -1 && bHot !== -1) return aHot - bHot;
    if (aHot !== -1) return -1;
    if (bHot !== -1) return 1;

    // 3. 兜底：字母序（稳定可预期）
    return a.id.localeCompare(b.id);
  });
});
```

**排序规则详解**：

| 优先级 | 规则 | 说明 |
|--------|------|------|
| P0 | 收藏 > 非收藏 | 被收藏的模型始终排在顶部，收藏之间保持原列表顺序 |
| P1 | 热门 > 非热门 | 两个都被收藏或都未被收藏时，按 `hotModels` 数组索引排序 |
| P2 | 字母升序 | 既非收藏也非热门的模型，按 `id` 字母升序排列，保证稳定性 |

**`isFavorite` 查询器**：

```typescript
// src/core/stores/modelStore.ts 第 41 行
const isFavorite = computed(() => (modelId: string) => favorites.value.includes(modelId));
```

返回一个柯里化函数，便于模板中直接调用：`modelStore.isFavorite(item.data.id)`。

## 3. 缓存与性能策略

### 3.1 10 分钟 TTL

前端缓存过期策略采用**时间戳驱动**的 TTL（Time-To-Live）机制：

```typescript
// src/core/stores/modelStore.ts 第 47–48 行
if (!force && models.value.length > 0 && Date.now() - lastRefreshed.value < 1000 * 60 * 10) {
  return;
}
```

| 参数 | 值 | 说明 |
|------|-----|------|
| TTL | 600,000 ms (10 min) | 模型列表变更频率低，10 分钟是移动端网络与新鲜度的平衡 |
| 条件 | `models.length > 0` | 空列表视为未初始化，不受 TTL 保护 |
| 覆盖 | `force = true` | 用户手动下拉刷新时强制穿透 |

后端同样实施缓存（内存 `RwLock` + SQLite `settings` 表），但前后端缓存独立管理：前端 TTL 决定何时触发 IPC 调用，后端缓存决定 IPC 调用是否触发网络请求。

### 3.2 锁频防护（isLoading）

```typescript
// src/core/stores/modelStore.ts 第 45 行
if (isLoading.value) return;
```

`isLoading` 是一个简单的布尔锁：

- **加锁时机**：`fetchModels` 入口立即置 `true`
- **解锁时机**：`finally` 块中置 `false`，保证异常场景下也能释放
- **UI 反馈**：`ModelSelector.vue` 的刷新按钮通过 `:class="{ 'animate-spin': modelStore.isLoading }"` 绑定旋转动画

### 3.3 强制刷新机制

当用户点击 ModelSelector 顶部的刷新按钮时，触发 `fetchModels(true)`，执行以下额外行为：

1. **穿透 TTL**：忽略时间戳检查
2. **网络拉取**：调用 `refresh_models` 从 VCP 服务器重新获取
3. **成功/失败通知**：通过 `notificationStore` 弹出 Toast（成功显示模型数量，失败提示检查网络/密钥）
4. **最低转圈时长**：为保证视觉连续性，若实际耗时不足 800ms，主动延迟到 800ms 后才熄灭旋转动画

```typescript
// src/core/stores/modelStore.ts 第 89–96 行
if (force) {
  const elapsed = Date.now() - startTime;
  const minDuration = 800;
  if (elapsed < minDuration) {
    await new Promise((resolve) => setTimeout(resolve, minDuration - elapsed));
  }
}
```

## 4. 模型选择器（ModelSelector.vue）

`ModelSelector.vue` 是 VCP Mobile 中**唯一**的模型选择 UI，以全局 BottomSheet 形式呈现，通过 `Teleport to="body"` 挂载到文档根节点，避免被父组件的 `overflow: hidden` 截断。

> 文件位置：`src/components/ModelSelector.vue`

### 4.1 触发方式（BottomSheet）

ModelSelector 通过 `v-model`（布尔值）控制显隐，属于受控组件：

```typescript
// src/components/ModelSelector.vue 第 16–22 行
const props = defineProps<{
  modelValue: boolean;      // 是否打开
  currentModel?: string;    // 当前已选中的模型 ID（高亮用）
  title?: string;           // 标题覆盖
}>();

const emit = defineEmits(['update:modelValue', 'select']);
```

**层级定位**：

- 遮罩层与抽屉内容均使用 `z-sheet`（语义化层级，对应 `LAYER_SHEET = 50`）
- 详见 `src/core/constants/layers.ts` 与 `docs/UI_LAYER_ARCHITECTURE.md`

**打开时机**：

- `ModelSelector` 挂载时（`onMounted`）自动调用 `modelStore.fetchModels()`，确保列表预加载
- 父组件通过 `v-model` 控制显隐，无需重复触发数据加载

### 4.2 列表结构与分组

ModelSelector **不**在渲染层面硬编码"收藏组/热门组/其他组"的分组标题，而是依赖 `modelStore.sortedModels` 的全局排序。视觉上所有模型处于同一扁平列表，但通过以下元素区分优先级：

| 视觉元素 | 出现条件 | 含义 |
|----------|----------|------|
| 左侧蓝色 Accent Bar（2px） | `currentModel === item.data.id` | 当前选中模型 |
| 右侧火焰图标（`Flame`） | `hotModels.includes(item.data.id)` | 热门模型 |
| 右侧黄色实心星标 | `isFavorite(item.data.id)` | 已收藏 |
| 右侧灰色空心星标 | 非收藏 | 可点击收藏 |
| 右侧蓝色对勾 | `currentModel === item.data.id` | 选中标记 |

**厂商快捷分类 Tags**：

除搜索外，ModelSelector 提供横向惯性滚动的厂商标签栏，动态从模型列表中提取 `owned_by` 字段并去重：

```typescript
// src/components/ModelSelector.vue 第 33–46 行
const manufacturerTags = computed(() => {
  const tags = ['全部', '收藏', '热门'];
  const counts: Record<string, number> = {};
  modelStore.models.forEach((m: ModelInfo) => {
    if (m.owned_by) {
      counts[m.owned_by] = (counts[m.owned_by] || 0) + 1;
    }
  });
  const sortedManufacturers = Object.keys(counts).sort((a, b) => counts[b] - counts[a]);
  return [...tags, ...sortedManufacturers];
});
```

标签按该厂商拥有的模型数量降序排列，确保常用厂商（如 `google`、`openai`）出现在前面。

### 4.3 收藏/取消收藏交互

每个模型项右侧有一个星标按钮，点击时：

1. **阻止事件冒泡**：`e.stopPropagation()`，防止触发模型选择
2. **调用 Store Action**：`modelStore.toggleFavorite(modelId)`
3. **乐观更新**：`modelStore` 内立即修改 `favorites` 数组，`sortedModels` computed 自动重排
4. **后端持久化**：`toggle_favorite_model` Command 异步写入 SQLite

```typescript
// src/components/ModelSelector.vue 第 173–176 行
const toggleFavorite = (e: Event, modelId: string) => {
  e.stopPropagation();
  modelStore.toggleFavorite(modelId);
};
```

### 4.4 搜索过滤

搜索支持按**模型 ID** 或**提供方**（`owned_by`）匹配：

```typescript
// src/components/ModelSelector.vue 第 62–67 行
const query = searchQuery.value.toLowerCase().trim();
return list.filter((m: ModelInfo) =>
  m.id.toLowerCase().includes(query) ||
  m.owned_by.toLowerCase().includes(query)
);
```

搜索与标签过滤**串联生效**：先按标签切片，再在结果中搜索。切换标签或修改搜索词时，自动将虚拟滚动容器重置回顶部，防止越界空白：

```typescript
// src/components/ModelSelector.vue 第 83–96 行
const selectTag = (tag: string) => {
  activeTag.value = tag;
  if (scrollContainerRef.value) {
    scrollContainerRef.value.scrollTop = 0;
  }
};

watch(searchQuery, () => {
  if (scrollContainerRef.value) {
    scrollContainerRef.value.scrollTop = 0;
  }
});
```

### 4.5 与 modelStore 的双向绑定

ModelSelector 与 `modelStore` 的关系是**消费单向 + 动作委托**：

- **读取**：`modelStore.sortedModels`、`modelStore.isLoading`、`modelStore.hotModels`、`modelStore.isFavorite()`
- **写入**：`modelStore.toggleFavorite()`、`modelStore.fetchModels(true)`（刷新按钮）
- **选择回调**：通过 `@select` 事件将选中模型 ID 抛给父组件，自身不修改任何业务状态

### 7.2 事件监听表

模型管理子系统**不**依赖 Tauri 事件通道（`listen` / `emit`），全部通过**同步 Request-Response 式 IPC**（`invoke`）交互。这是有意的设计选择：

- 模型列表变更频率低，无需实时推送
- `ModelSelector` 打开时主动拉取，关闭后无持续监听开销
- 收藏切换通过乐观更新 + 异步持久化实现即时反馈

## 9. 设计决策与注意事项

### 9.1 为何 ModelSelector 不做硬分组？

常见的模型选择器会显示"收藏"、"热门"、"其他"三个分组标题。VCP Mobile 选择**扁平列表 + 全局排序**的策略，原因如下：

1. **收藏与热门可能重叠**：一个模型既被收藏又属于热门，硬分组会导致重复显示或增加去重复杂度
2. **虚拟滚动兼容性**：分组标题会打断固定高度的虚拟滚动计算，需引入可变高度逻辑，增加渲染开销
3. **移动屏幕空间宝贵**：分组标题占用纵向空间，而扁平排序通过视觉元素（Accent Bar、火焰图标、星标）即可传达同样的优先级信息

### 9.2 为何使用 `useVirtualList` 而非原生滚动？

模型列表在 VCP Server 上可能包含数百个模型（尤其当接入聚合网关时）。`useVirtualList` 的配置：

```typescript
// src/components/ModelSelector.vue 第 71–74 行
const { list, containerProps, wrapperProps } = useVirtualList(filteredModels, {
  itemHeight: 62,
  overscan: 10,
});
```

- `itemHeight: 62`：固定高度确保虚拟滚动的精确计算（`py-3.5` + `text-15` + `text-11` + 间距 ≈ 62px）
- `overscan: 10`：视口上下各多渲染 10 条，防止快速滑动时出现空白

实际测试中，即使列表包含 500+ 模型，BottomSheet 打开与滚动的首帧时间仍保持在 16ms 以内。

### 9.3 下拉手势关闭的 120px 阈值

ModelSelector 实现了移动端原生的下拉关闭手势：

```typescript
// src/components/ModelSelector.vue 第 138–140 行
if (currentTranslateY.value > 120) {
  close();
}
```

- **120px**：经过实验，在 375–430px 宽度的屏幕上，120px 约等于 1/3 屏幕高度的下拉距离，符合用户"甩出"直觉
- **GPU 加速**：使用 `translate3d(0, ${deltaY}px, 0)` 开启硬件加速，避免内部子元素重排闪烁
- **遮罩联动**：下拉过程中遮罩透明度 `opacity = max(0, 1 - deltaY / 400)`，形成视觉深度反馈

### 9.4 搜索按 `owned_by` 而非显示名称

搜索框提示文字为"搜索模型名称"，但实际匹配逻辑包含 `m.owned_by.toLowerCase().includes(query)`。这是有意的设计：

- 模型 ID（如 `gemini-2.5-flash-preview-05-27`）通常比人类可读的"名称"更具辨识度
- 用户更常按厂商搜索（输入 "google" 过滤出所有 Gemini 模型）
- 若未来引入独立的中文显示名称字段，可扩展匹配逻辑而不破坏现有行为

### 9.5 最低转圈时长的 UX 考量

`fetchModels(true)` 在 `finally` 块中检查实际耗时，不足 800ms 时主动延迟：

- **防止"闪一下"**：若网络极快或命中本地缓存，旋转动画可能只显示 50ms，用户感知为 UI 抖动
- **心理预期管理**：800ms 是人类对"刷新动作"的最短心理预期时长，低于此值会让用户怀疑是否真的执行了刷新
