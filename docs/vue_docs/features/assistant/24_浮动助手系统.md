# 24. 浮动助手系统（Floating Assistant System）

## 1. 概述

### 1.1 领域定位

VCP Mobile 的浮动助手系统是一个**独立于主应用的 Mini-App**。它运行在 Android WindowManager 创建的悬浮 WebView 中，通过 WebSocket 与本地 Rust HTTP 服务器通信，实现一个完整的、不离开当前前台应用的 AI 对话界面。

与主聊天应用（`ChatView`、`chatManager` 等）的关键区别：
- **无 Tauri invoke 依赖**：浮动窗口不运行在主 Activity 上下文中，无法调用 `invoke()`，改用 WebSocket
- **无 Pinia Store 共享**：浮动窗口是一个独立的 WebView 实例，不与主应用的 Vue 实例共享内存
- **独立 Vite 入口**：`floating.html` 是独立的构建入口，产出一个轻量级独立 Bundle
- **临时会话模式**：对话默认不持久化，用户可选择"归档"将对话同步到主话题列表

### 1.2 在整体架构中的位置

```
┌──────────────────────────────────────────────────────────────┐
│                    Vite 双入口构建                            │
│  ┌─────────────────────┐    ┌─────────────────────────────┐  │
│  │   index.html         │    │   floating.html             │  │
│  │   主应用入口          │    │   浮动助手 Mini-App 入口     │  │
│  │   所有 Pinia Stores  │    │   仅 floatingAssistant     │  │
│  │   invoke() IPC       │    │   WebSocket IPC            │  │
│  └──────────┬───────────┘    └──────────────┬──────────────┘  │
│             │                               │                 │
│             ▼                               ▼                 │
│  ┌─────────────────────┐    ┌─────────────────────────────┐  │
│  │  ChatView.vue        │    │  AssistantView.vue          │  │
│  │  (主聊天界面)        │    │  (浮动聊天界面)             │  │
```

## 2. Mini-App 模式详解

### 2.1 双入口 Vite 构建

`vite.config.ts` 配置了 `build.rollupOptions.input` 包含两个 HTML 入口：

```
build.rollupOptions.input
    ├── index.html      → 主应用 bundle (ChatView + 全部功能)
    └── floating.html   → 浮动助手 bundle (AssistantView 独立)
```

两个入口**共享大部分公共 chunk**（Vue、UnoCSS、Pinia），但核心区别在于：

| 属性 | index.html | floating.html |
|------|-----------|---------------|
| 挂载组件 | `App.vue` → 完整路由/布局 | `AssistantView.vue` 独立 |
| Pinia Stores | 全部 18 个 Store | 仅 `floatingAssistant` |
| IPC 方式 | `invoke()` Tauri Commands | WebSocket (`ws://127.0.0.1:14202/ws`) |
| 持久化 | SQLite（后端托管） | 无（临时内存） |
| 密码管理器 | 全部共享 | 仅 `floatingAssistant` store |
| Toast | `notificationStore` | `floatingAssistant.toasts` 内置 |

### 2.2 为什么使用 Mini-App 模式

传统方案（浮动 WebView 加载 `/chat` 路由）的问题：
1. **依赖污染**：浮动窗口会加载全部 Store（session、history、settings 等），这些 Store 的初始化依赖 `invoke()`，而 invoke 在浮动上下文中不可用
2. **状态冲突**：两个 Vue 实例同时写入同一个 SQLite 会导致数据竞争
3. **资源浪费**：加载完整应用在浮动窗口的小屏幕上显示 UI 布局臃肿

Mini-App 模式的解决方案：
1. `floating.html` 仅包含 `AssistantView.vue`，砍掉路由、侧边栏、设置等全部冗余 UI
2. `floatingAssistant` Store 自包含——不依赖任何其他 Store，不使用 `invoke()`
3. 所有数据流通过 WebSocket 串行化，状态只在浮动窗口内存中

## 3. floatingAssistant Store

> 文件位置：`src/core/stores/floatingAssistant.ts` (352 行)
> Store ID: `"floatingAssistant"`

### 3.1 核心状态

| 状态 | 类型 | 说明 |
|------|------|------|
| `messages` | `ChatMessage[]` | 当前临时会话的消息列表（不持久化） |
| `inputText` | `string` | 输入框文本 |
| `isGenerating` | `boolean` | 是否正在流式生成 |
| `currentStreamingMessageId` | `string \| null` | 当前流式输出的消息 ID |
| `internalSettings` | `AppSettings \| null` | 内部缓存的应用设置（WebSocket config 推送） |
| `toasts` | `Toast[]` | 内置 Toast 通知列表 |
| `isFloatingMode` | `boolean` | 是否为浮动模式（检测 URL） |
| `ws` | `WebSocket \| null` | WebSocket 连接实例 |
| `wsReady` | `boolean` | WebSocket 连接已建立 |
| `wsConfigured` | `boolean` | 已收到 initial_config 且设置加载完成 |

### 3.2 浮动模式检测

```typescript
const isFloatingMode = ref(
  window.location.pathname.includes("floating") ||
  window.location.search.includes("mode=floating"),
);
```

此双路径检测允许主应用在特定场景（如调试/开发时）通过 `?mode=floating` 参数模拟浮动模式行为。在正式部署中，`/floating` 路径由 `floating.html` 入口保证始终为浮动模式。

### 3.3 WebSocket 生命周期

```
initWebSocket()
    │
    ├── 检查 isFloatingMode && ws 不存在
    │       ↓ (仅在浮动模式下初始化)
    ├── 创建 new WebSocket("ws://127.0.0.1:14202/ws")
    │
    ├── onopen:
    │     ├── wsReady = true
    │     └── 发送 { action: "get_initial_config" }
    │            获取 VCP URL、API Key、Agent ID 等配置
    │
    ├── onmessage:
    │     └── handleWsMessage(data)   ← 见 §3.4
    │
    ├── onerror:
    │     └── console.error 打印错误
    │
    └── onclose:
          ├── wsReady = false
          └── ws = null
```

### 3.4 消息处理（`handleWsMessage`）

```
接收 WebSocket 消息 (JSON)
    │
    ├── data.type === "initial_config"
    │     └── 将 data.settings 存储到 internalSettings
    │         设置 wsConfigured = true
    │
    ├── data.type === "archive_success"
    │     └── Toast 通知"归档成功"
    │         clearSession() 清空消息列表
    │
    ├── data.type === "thinking"
    │     └── 查找以 _assistant_temp 结尾的占位消息
    │         将 messageId 赋给占位消息 → 流式写入选定
    │
    ├── data.type === "data"
    │     └── 查找 currentStreamingMessageId 或 isThinking 的消息
    │         追加 data.chunk / data.chunk.choices[0].delta.content 到 content
    │         isThinking = false
    │
    ├── data.type === "end"
    │     └── isGenerating = false
    │         消息 isThinking = false
    │         currentStreamingMessageId = null
    │
    └── data.type === "error"
          └── isGenerating = false
              消息 content += "\n\n[错误]: ..."
              currentStreamingMessageId = null
```

### 3.5 双模发送（`sendMessage`）

`sendMessage` 是 Store 最核心的方法，根据 `isFloatingMode` 选择不同的通信方式：

```
sendMessage(content)
    │
    ├── 1. resolveSettings() → 获取 VCP URL + API Key + Agent ID
    │       │
    │       ├── 浮动模式: 使用 internalSettings (WebSocket config 推送)
    │       └── 主应用模式: invoke("read_settings") 获取
    │
    ├── 2. 校验: agentId != null && vcpUrl != "" && vcpApiKey != ""
    │
    ├── 3. 创建消息:
    │     ├── userMsg: { role: "user", content, id: "msg_<now>_user_temp" }
    │     └── aiMsg:  { role: "assistant", content: "", id: "msg_<now+1>_assistant_temp", isThinking: true }
    │
    ├── 4. isGenerating = true
    │
    └── 5. 发送:
          │
          ├── if (isFloatingMode && ws):
          │     ws.send(JSON.stringify({
          │       action: "handle_assistant_chat_stream",
          │       payload: { agentId, tempMessages, vcpUrl, vcpApiKey }
          │     }))
          │     ↑ 浮动模式：通过 WebSocket 发送
          │
          └── else:
                const channel = new Channel<any>()
                channel.onmessage = (event) => handleWsMessage(event)
                invoke("handle_assistant_chat_stream", {
                  payload, streamChannel: channel
                })
                ↑ 主应用模式：通过 Tauri invoke + Channel 发送
```

**为何主应用也使用 floatingAssistant Store？** Store 内置了双模支持，主应用中用户可以打开"划词助手"面板（例如通过浮动入口），此时 Store 以"fallback 到 Tauri invoke"的方式工作。这确保同一套 AssistantView.vue UI 代码在主应用和浮动窗口中均可复用。

### 3.6 会话归档（`archiveSession`）

```
archiveSession()
    │
    ├── 过滤 messages: 仅保留 (role === "user" || role === "assistant") && content 非空
    │       │
    │       ├── 若无有效消息 → return null
    │       └── 有消息 → 继续
    │
    ├── resolveAgentId() → 获取绑定的 Agent ID
    │
    ├── if (isFloatingMode && ws):
    │     ws.send(JSON.stringify({
    │       action: "archive_assistant_chat",
    │       payload: { ownerId, ownerType: "agent", tempMessages }
    │     }))
    │     return "WS_PENDING"  (等待 archive_success WebSocket 回调)
    │
    └── else (主应用模式):
          invoke("archive_assistant_chat", { ownerId, ownerType, tempMessages })
          .then(topicId => {
            addToast("success", ...)
            clearSession()
            return topicId
          })
```

### 3.7 Toast 通知系统

Store 内置了独立的 Toast 系统，不依赖 `notificationStore`（浮动窗口不存在该 Store）：

```typescript
interface Toast {
  id: string;        // 随机 ID
  title: string;     // 标题
  message: string;   // 详情
  type: "info" | "success" | "warning" | "error";
  timestamp: number; // 创建时间戳
}
```

**管理逻辑**：
- `addToast(type, title, message)` 创建 Toast
- 上限 5 个（`if (toasts.length > 5) toasts.shift()`）
- 每条 Toast 3 秒后自动移除（`setTimeout(filter, 3000)`）

## 4. AssistantView.vue

> 文件位置：`src/features/assistant/AssistantView.vue` (267 行)

### 4.1 组件结构

```
AssistantView.vue
    ├── Header (连接状态 + 标题 + 关闭按钮)
    ├── Message List (AssistantMessageCard × N)
    │     └── Empty State (首次打开引导)

    └── Input Area (Textarea + Send Button)
```

### 4.2 Header — 连接状态指示

```html
<div class="w-1.5 h-1.5 rounded-full"
     :class="
       wsConfigured || !isFloatingMode
         ? 'bg-green-500'    <!-- 完全就绪 -->
         : wsReady
           ? 'bg-yellow-500'  <!-- WS 已连接，等待 config -->
           : 'bg-red-500'     <!-- WS 未连接 -->
     ">
```

| 灯色 | 含义 | 触发条件 |
|------|------|----------|
| 绿色 (Green) | 就绪 | `wsConfigured === true` 或非浮动模式（直接 IPC） |
| 黄色 (Yellow) | 配置中 | WebSocket 已连接但 `initial_config` 尚未返回 |
| 红色 (Red) | 断开 | WebSocket 尚未建立连接 |

### 4.3 消息列表 — 流式自动滚动

```typescript
watch(
  () => {
    const msgs = floatingStore.messages;
    if (msgs.length === 0) return "";
    return msgs[msgs.length - 1].content || "";
  },
  () => {
    requestAnimationFrame(() => scrollToBottom());
  },
);
```

监听最后一条消息 `content` 的变化（每次追加 chunk 时触发），使用 `requestAnimationFrame` 确保在 DOM 更新后执行滚动，避免 `scrollTop` 与 `scrollHeight` 不一致导致抖动。

### 4.4 输入区域 — 连接状态感知

```html
<textarea
  :placeholder="isFloatingMode && !wsConfigured
    ? '正在连接助手服务...'
    : '问问 VCP...'"
  @keydown.enter.prevent="handleSend"
/>
```

`Enter` 键直接发送（无修饰键），发送按钮在以下条件禁用：
- 输入为空
- 正在生成 (`isGenerating`)
- 浮动模式下 WebSocket 尚未配置完成 (`isFloatingMode && !wsConfigured`)

### 4.5 AndroidBridge 集成

```typescript
const closeAssistant = () => {
  if (floatingStore.isFloatingMode) {
    if ((window as any).AndroidBridge) {
      (window as any).AndroidBridge.closeWindow();
    }
  }
};

const tryFillClipboard = () => {
  if ((window as any).AndroidBridge) {
    const text = (window as any).AndroidBridge.getClipboard();
    if (text && text.trim()) {
      inputText.value = text.trim();
    }
  }
};
```

- **closeWindow()**：调用原生 `FloatingWindowManager.hideAssistantWindow()` 收起助手窗口
- **getClipboard()**：读取系统剪贴板，在打开浮动窗口时自动填入输入框，实现"复制即问"的工作流

`AndroidBridge` 仅在 Android 浮动 WebView 环境中注入（见 §5），主应用模式中上述调用静默跳过。

### 4.6 Toast TransitionGroup

```html
<TransitionGroup name="toast" tag="div" ...>
  <div v-for="toast in floatingStore.toasts" :key="toast.id"
       :class="toast.type === 'success' ? 'bg-green-500' :
               toast.type === 'error' ? 'bg-red-500' :
               'bg-black/80 dark:bg-white/80'">
    {{ toast.title }}
  </div>
</TransitionGroup>
```

- **进场动画**：`opacity 0 → 1` + `translateY(-8px → 0)`，300ms ease
- **退场动画**：`opacity 1 → 0` + `translateY(0 → -4px)`，200ms ease
- **变色规则**：`success` = 绿色, `error` = 红色, 其余 = 黑白半透明

## 6. WebSocket 协议表

### 6.1 客户端 → 服务器

| action | 用途 | payload 关键字段 |
|--------|------|-----------------|
| `get_initial_config` | 获取设置（VCP URL / API Key / Agent ID） | 无 |
| `handle_assistant_chat_stream` | 发起 Agent 流式对话 | `agentId`, `tempMessages`, `vcpUrl`, `vcpApiKey` |
| `archive_assistant_chat` | 归档临时会话到主话题列表 | `ownerId`, `ownerType`, `tempMessages` |

### 6.2 服务器 → 客户端

| type | 用途 | 关键字段 |
|------|------|---------|
| `initial_config` | 返回全局设置 | `settings` (含 `vcpServerUrl`, `vcpApiKey`, `assistantAgentId`) |
| `thinking` | Agent 开始思考 | `messageId`, `message_id` |
| `data` | 流式内容块 | `messageId`, `chunk` / `chunk.choices[0].delta.content` |
| `end` | 流式输出完成 | `messageId` |
| `error` | 错误发生 | `error` (错误描述) |
| `archive_success` | 归档成功 | `topicId` (新创建的话题 ID) |

| 特性 | 主应用 (ChatView) | 浮动助手 (AssistantView) |
|------|------------------|------------------------|

## 8. 设计决策与血训

### 8.1 为什么 WS 连接在 `initWebSocket` 中立即发送 `get_initial_config`

WebSocket `onopen` 阶段即发送 `get_initial_config`，而不是等待用户首次输入。原因：
- 用户从点击悬浮球到看到 UI 的间隙（约 200-500ms）正好被用来传输配置
- 如果等到发送消息时才获取配置，首次对话会有滞后感
- `initial_config` 响应体量小（< 1KB JSON），不占用显著带宽

### 8.2 为什么 floatingAssistant 在 `onMounted` 中调用 `tryFillClipboard()`

`AssistantView.onMounted()` 中立即调用 `AndroidBridge.getClipboard()` 获取剪贴板文本。这是"划词助手"的核心理念：用户在其他应用选中文本、分享到 VCP 时，剪贴板已有内容（某些 Android 版本分享不经过剪贴板，则由 ShareIntent 的 JS 注入路径覆盖）。该操作在 WebSocket 配置完成前即可执行，文本已就位，等待连接就绪即可发送。

### 8.3 双模 fallback 的价值

Store 的 `isFloatingMode` 分支使同一份组件代码可在两个完全不同的运行环境中复用。这使 `AssistantView.vue` 在开发阶段可直接在主 `index.html` 中加载并测试，无需每次构建 APK + 部署到真机，大幅加速前端调试循环。

### 8.4 临时消息 ID 命名规范

- User 消息：`msg_{timestamp}_user_temp`
- AI 消息：`msg_{timestamp+1}_assistant_temp`
- `_temp` 后缀标记为临时（未持久化）消息
- `_assistant_temp` 在 `thinking` 事件时被替换为后端返回的正式 messageId
