# 08. 对话引擎总览

## 1. 概述

### 1.1 领域定位

`features/chat/` 是 VCP Mobile 前端**对话引擎**的核心领域目录，承载用户与 Agent/Group 进行交互的完整前端链路。该领域负责：

- 聊天主视图的布局渲染与生命周期管理
- 消息历史的分页加载、流式接收与状态同步
- 用户输入的多模态采集（文本、语音、附件）
- 附件的暂存、预览、预处理与上传
- 群聊模式下的多 Agent 并发流式控制

该领域**不**涉及：
- 模型推理与 SSE 解析本身（由 Rust 后端 `vcp_client.rs` / `agent_chat_application_service.rs` 负责）
- 消息持久化的通用逻辑（由 Rust `message_service.rs` 负责）
- Agent/Group 的配置管理（由 `features/agent/` 及 Rust `agent_service.rs` 负责）

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **Backend-Driven Streaming** | Thinking 消息 ID 由后端生成并通过 SSE 事件推送，前端不再预创建占位消息。`chatStreamStore` 作为全局流消息池，保证前后台切换时流式状态不丢失。 |
| **Store 垂直拆分** | 会话状态 (`chatSessionStore`)、历史消息 (`chatHistoryStore`)、流式状态 (`chatStreamStore`)、附件暂存 (`attachmentStore`) 四者物理隔离，避免单一 God Store。 |
| **场景化滚动状态机** | `useChatScroll` 采用 `"initial" → "following" ↔ "free" → "loading-top"` 四状态机，替代 IntersectionObserver 的模糊边界判定，根治滚动回弹与误置底。 |
| **乐观更新 + 骨架接续** | 用户消息立即追加到 `currentChatHistory`，流式响应通过 `activeStreamMessages` 全局池维护；历史加载时通过 Object Hydration 将池中的"活"对象替换 DB 回流的骨架，实现无缝接续。 |
| **多模态输入统一收口** | 文本、语音(STT)、音频附件、文件附件、相机/相册全部收敛到 `InputEnhancer.vue`，最终通过 `historyStore.sendMessage()` 统一发射。 |

## 2. 聊天主视图（ChatView.vue）

> 文件位置：`src/features/chat/ChatView.vue`

### 2.1 组件结构

ChatView 采用 **Header + 消息区 + 输入区** 的三段式纵向 Flex 布局：

```
┌──────────────────────────────────────┐
│  Header (固定高度, 适配刘海屏)        │  <-- vcp-header-fixed
│  [☰] [Avatar] [Name+Status] [☀] [🔔] │
├──────────────────────────────────────┤
│                                      │
│  消息展示区 (flex-1, overflow-y-auto) │  <-- messageListRef
│  ┌────────────────────────────────┐  │
│  │  零数据引导状态 (绝对定位)      │  │
│  │  或 MessageRenderer 列表        │  │
│  └────────────────────────────────┘  │
│                                      │
├──────────────────────────────────────┤
│  [↓ 一键置底] (绝对定位, z-local)     │
├──────────────────────────────────────┤
│  InputEnhancer (固定底部)             │  <-- vcp-input-footer
│  [安全区留白]                         │
└──────────────────────────────────────┘
```

**关键 DOM 结构**（第 142–272 行）：

1. **容器层** (`chatViewContainerRef`)：全高 Flex 列，监听 `@touchmove` 以阻止非交互空白区域的 WebView 视口拖动（橡皮筋效应）。
2. **Header**：固定高度，使用 `calc(var(--vcp-safe-top, 24px))` 动态适配刘海屏；展示当前选中项头像、名称、`CoreStatusIndicator`、主题切换与通知按钮。
3. **消息列表** (`messageListRef`)：`flex-1 overflow-y-auto`，内部使用 `.messages-inner-container` 作为 `ResizeObserver` 的观测目标。
4. **一键置底按钮**：绝对定位，`z-local` 层级，动态计算 `bottom` 以避让输入框、键盘高度与扩展菜单。
5. **输入区** (`vcp-input-footer`)：固定底部，背景使用 `color-mix` 实现毛玻璃质感；内部渲染 `InputEnhancer`。

### 2.2 与 Store 的绑定关系

ChatView 是 4 个核心 Store 的**消费端协调器**，自身不持有业务状态，仅做组合与生命周期联动：

| Store | 绑定方式 | 用途 |
|-------|----------|------|
| `chatSessionStore` | 直接解构 | 读取 `currentSelectedItem` / `currentTopicId` 驱动 Header 展示与话题切换监听 |
| `chatHistoryStore` | 直接解构 | 读取 `currentChatHistory` 渲染消息列表；调用 `loadMoreHistory` 加载历史 |
| `chatStreamStore` | 直接解构 | 读取 `activeStreamingIds` 控制自动滚动启停 |
| `topicStore` | 直接解构 | 话题切换时 `markTopicAsRead`；零数据引导态判断 |

**话题切换监听**（第 62–81 行）：

```typescript
watch(
  [() => sessionStore.currentTopicId, () => sessionStore.currentSelectedItem],
  ([newTopicId, newSelectedItem]) => {
    showScrollToBottom.value = false;
    resetChatScroll();
    if (newTopicId && newSelectedItem) {
      topicStore.markTopicAsRead(newTopicId);
      historyStore.loadHistoryPaginated(
        newSelectedItem.id,
        newSelectedItem.type,
        newTopicId
      );
    } else {
      historyStore.currentChatHistory = [];
    }
  },
  { immediate: true }
);
```

关键行为：
- 话题切换时立即重置滚动状态机（`resetChatScroll` → `"initial"`）
- 若目标话题/选中项有效，触发分页历史加载（初始 limit = 5）
- 若无效（如未选中 Agent），清空当前历史数组

**空态处理（零数据引导）**（第 221–242 行）：

当 `historyStore.currentChatHistory.length === 0` 时，消息列表区展示绝对定位的引导层：

```
┌────────────────────────────────────────┐
│  大图标 (渐变背景 + 对话气泡 SVG)       │
│  "开启智能对话"                        │
│                                        │
│  条件文案 A:                            │
│    "当前话题暂无消息，开始发送第一条吧！" │
│  条件文案 B:                            │
│    "请从左侧列表选择一个助手..."        │
│                                        │
│  [立即开始] 按钮 (未选中 Agent 时展示)   │
└────────────────────────────────────────┘
```

- 文案 A：已选中 Agent/Group 但当前话题无消息
- 文案 B：未选中任何 Agent/Group（侧边栏未展开或首次进入）
- "立即开始"按钮调用 `layoutStore.toggleLeftDrawer()` 打开左侧抽屉
- 引导层使用 `rotate-3` 微旋转与 `shadow-2xl` 大投影，在极简布局中形成视觉焦点

**底部留白策略**：

`.messages-inner-container` 末尾固定插入 `<div class="h-20"></div>`（第 253 行），确保最后一条消息不会被输入框遮挡，同时为一键置底按钮留出呼吸空间。

### 2.3 生命周期钩子

**`onMounted`**（第 121–129 行）：
1. 注册 `window.addEventListener("vcp-button-click", ...)` —— 接收来自 VcpButton 组件的外部文本注入（如快捷指令）。
2. 在容器上监听 `focusin`，触发 `forceRecalculate` 以重新计算键盘 Insets（应对 Android 软键盘弹出时的布局变化）。
3. 调用 `checkAndLoadMore()` 触发首屏自动加载判定。

**`onUnmounted`**（第 131–139 行）：
1. 清理 `vcp-button-click` 事件监听。
2. 清理 `focusin` 监听。
3. 调用 `disposeChatScroll()` 彻底断开 `ResizeObserver` 与 `scroll` 事件。

**流式状态联动**（第 103–105 行）：
```typescript
watch(isStreamingActive, (active) => {
  active ? startAutoScroll() : stopAutoScroll();
});
```
当 `streamStore.activeStreamingIds` 从空变非空时启动自动滚动；流结束时停止。在新版 `useChatScroll` 架构中，`startAutoScroll` / `stopAutoScroll` 为空函数（保留 API 兼容），实际滚动由 `ResizeObserver` 自动驱动。

**键盘高度联动**（第 107–119 行）：
```typescript
watch(keyboardHeight, (height) => {
  chatViewContainerRef.value?.style.setProperty("--keyboard-offset", `${height}px`);
  if (height > 0 && historyStore.currentChatHistory.length > 0) {
    scrollToBottom(true); // 软键盘弹出时平滑置底
  }
});
```
将键盘高度写入 CSS 变量，供置底按钮与输入区安全区留白动态计算。

### 2.4 虚拟滚动与性能优化

ChatView **不采用传统虚拟滚动**（如 `vue-virtual-scroller`），而是基于以下策略保证长列表性能：

1. **分页加载**：初始仅加载 5 条最新消息（`loadHistoryPaginated`），用户上滑到顶时再以 10 条为增量加载旧消息（`loadMoreHistory`）。
2. **`useChatScroll` 场景状态机**：封装于 `src/core/composables/useChatScroll.ts`，详见下文。
3. **ResizeObserver + RAF 节流**：监听 `.messages-inner-container` 的物理尺寸变化，而非依赖 `nextTick` 的"猜时机"。
4. **滚动条隐藏**：通过 CSS `scrollbar-width: none` / `::-webkit-scrollbar { display: none }` 消除原生滚动条视觉干扰。

**消息列表的 DOM 结构细节**：

```vue
<div ref="messageListRef" class="flex-1 overflow-y-auto relative no-rubber-band">
  <!-- 零数据状态 (条件渲染) -->
  <div v-if="historyStore.currentChatHistory.length === 0" class="absolute inset-0 ...">
    ...
  </div>
  
  <!-- 消息列表容器 (始终渲染，ResizeObserver 的观测目标) -->
  <div class="messages-inner-container py-4 space-y-2 flex flex-col min-h-full">
    <MessageRenderer
      v-for="msg in historyStore.currentChatHistory"
      :key="msg.id"
      :message="msg"
      :agent-id="sessionStore.currentSelectedItem?.id"
      :data-message-id="msg.id"
    />
    <div class="h-20"></div>  <!-- 底部留白 -->
  </div>
</div>
```

- `:data-message-id="msg.id"` 是 `useChatScroll` 锚定机制的关键属性，用于在分页加载时识别视口顶部消息
- `.messages-inner-container` 必须始终存在于 DOM 中，因为它是 `ResizeObserver` 的观测目标；若使用 `v-if` 条件渲染，Observer 会在空态与列表态切换时断开
- `space-y-2` 提供消息间的紧凑间距，符合"高密度线性布局"规范

**`useChatScroll` 四状态机**：

| 状态 | 触发条件 | 行为 |
|------|----------|------|
| `initial` | 话题切换/消息清空后 | 首屏加载期间，滚动行为由 `ResizeObserver` 自动接管，首次内容到达后 `scrollToBottom(false)` 并迁移到 `following` |
| `following` | 用户位于底部 150px 内 | 新消息/流式内容到达时自动置底；持续评估是否需要自动续载历史 |
| `free` | 用户主动上滑离开底部 | 不再自动置底；允许用户自由浏览历史；上滑到顶部 100px 内触发 `loading-top` |
| `loading-top` | 上滑到顶触发分页加载 | 记录锚定元素（`prepareLoadAnchor`），加载完成后通过锚定偏移恢复滚动位置（`restoreScrollByAnchor`），回到 `free` |

**关键优化：加载锚定**

```typescript
const prepareLoadAnchor = () => {
  // 记录视口中最顶部可见消息的 data-message-id 及其距顶偏移
  loadAnchor = {
    messageId: el.getAttribute("data-message-id")!,
    offsetFromTop: rect.top - listRect.top,
  };
};
```

当旧消息 prepend 到列表后，通过锚定元素的 `offsetTop - offsetFromTop` 精确恢复用户视线位置，避免"加载后突然跳转"的糟糕体验。

**关键优化：物理高度守卫**

```typescript
const currentScrollHeight = list.scrollHeight;
if (currentScrollHeight === lastScrollHeight) return; // 高度无变化，直接拦截
lastScrollHeight = currentScrollHeight;
```

此守卫从物理上秒杀了用户手动上滑时的误置底无限回弹 Bug，同时极大释放 CPU 性能。

**输入区 4 种互斥状态**：

| 状态 | 条件 | 展示内容 | 用户操作 |
|------|------|----------|----------|
| 键盘输入 | `!isAudioMode && !isLongPressRecording` | `<textarea>` 多行文本框 | 打字、粘贴、快捷键发送 |
| 按住说话 (待机) | `isAudioMode && !isSTTActive` | 灰色大条，文字"按住 说话" | 触摸开始 STT |

| 长按录音中 | `isLongPressRecording` | 蓝色脉冲指示器 + 录音时长 | 上滑取消，松手以附件直发 |

**输入框自动增高**（第 306–318 行）：

```typescript
const autoResize = () => {
  if (textareaRef.value) {
    textareaRef.value.style.height = 'auto'; // Reset
    const scrollHeight = textareaRef.value.scrollHeight;
    textareaRef.value.style.height = `${scrollHeight}px`;
  }
};

watch(input, () => {
  nextTick(() => { autoResize(); });
});
```

- `rows="1"` 配合 `height: auto → scrollHeight` 实现从单行到多行的平滑扩展
- `max-height: 114px` 限制最大高度，超出后内部滚动
- `leading-[1.25]` 保持行高紧凑，符合项目"高密度线性布局"美学宪法

**Placeholder 逻辑**：
- `disabled` 为真时显示 `"请先选择话题以开启对话"`
- 正常状态显示 `"说点什么..."`
- `disabled` 由父组件 `ChatView` 通过 `:disabled="!sessionStore.currentTopicId"` 传入

**交互 A：STT（按住说话 → 实时转文字）**

### 3.4 长文本粘贴处理（useLongTextPaste）

> 文件位置：`src/features/chat/composables/useLongTextPaste.ts`

**问题背景**：Android 端直接粘贴数千字甚至上万字的长文本到 `<textarea>` 会导致：
1. 输入框严重卡顿甚至 ANR
2. 后端上下文爆炸，Token 消耗失控
3. WebView 内存压力骤增

**三层拦截策略**：

```
Layer 1: ClipboardEvent (paste)
         │
         ├── 文件类型? ──→ 放行，由 attachmentStore 处理
         ├── 图片类型? ──→ 放行，由 attachmentStore 处理
         └── 纯文本且 > 1000 字符? ──→ event.preventDefault() → stageTextAsFile()
         
Layer 2: InputEvent (beforeinput)
         │
         └── inputType === 'insertFromPaste' 且 data.length > 1000?
             ──→ event.preventDefault() → stageTextAsFile()
             
Layer 3: watch(input) 突变检测
         │
         └── delta > 1000 字符?
             ──→ 回退旧值 (input.value = oldVal)
             ──→ stageTextAsFile(增量文本)
```

阈值常量：`LONG_TEXT_THRESHOLD = 1000`（字符数，非字节数）。

**`stageTextAsFile` 流程**：

```
用户粘贴长文本
    │
    ▼
┌─────────────────────┐
│ 校验话题已选择       │──> 无选中项则 Toast 提示并退出
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 生成 .txt 文件名     │──> `pasted_text_YYYY-MM-DD-HH-mm-ss.txt`
│ 编码为 Uint8Array    │
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 插入 loading 占位    │──> stagedAttachments.unshift({ status: 'loading', ... })
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ invoke('store_file') │──> Rust 后端持久化到附件沙盒
│ 传递 bytes + mime    │
└──────────┬──────────┘
           ▼
    ┌──────┴──────┐
   成功           失败
    │              │
    ▼              ▼
┌────────┐   ┌────────────┐
│更新为   │   │移除占位卡片 │
│done状态 │   │Toast 错误提示│
└────────┘   └────────────┘
```

关键设计：长文本被转换为 `.txt` 附件后，随用户下一条消息一起发送，由后端 `message_service` 在组装上下文时提取文本内容。这既保护了前端输入体验，也保留了文本的完整语义。

**为何需要三层拦截？**

不同平台与输入法的粘贴行为差异极大：
- **桌面浏览器**：标准 `paste` 事件可靠，Layer 1 即可拦截
- **Android Gboard/搜狗**：某些版本使用 `beforeinput` 而非标准 `paste`，需 Layer 2
- **第三方输入法（如微信键盘）**：可能直接突变 DOM 值而不触发上述任一事件，需 Layer 3 的 `watch(input)` 做最终兜底

三层呈**递进防御**关系，而非互斥。任何一层成功拦截都会阻止长文本直接进入输入框。

### 5.2 附件预处理与上传

附件在发送前经历**两道处理关卡**：

**关卡 1：选择阶段上传**

***Android 原生选择器的双轨事件监听***（`attachmentStore.ts` 第 79–248 行）：

Android 端由于 WebView 文件选择器的限制，采用原生 Kotlin 插件实现文件选择。前端通过**双轨监听**（Invoke 返回值 + CustomEvent 事件流）确保可靠性：

```
invoke('plugin:vcp-mobile|pick_file')
    │
    ├── 轨道 A: Invoke 返回值 (Promise  resolve)
    │       └── 正常流程：返回 { path, name, mime, size, hash, thumbnailPath }
    │
    └── 轨道 B: CustomEvent 事件流 (Kotlin → WebView evaluateJavascript)
            ├── vcp-mobile-file-start  → 插入 loading 占位
            ├── vcp-mobile-file-progress → 更新进度条
            └── vcp-mobile-file-picked  → 更新为完成状态
```

双轨设计的必要性：
1. **大文件场景**：Kotlin 侧的文件拷贝与哈希计算可能耗时数秒，若仅依赖 Invoke 返回值，前端在这段时间内没有任何反馈，用户会以为点击未生效
2. **事件丢失容错**：若 WebView 错过了 `file-start` 事件，`file-progress` 回调中包含文件名与 MIME 类型，可"自我修复"补建卡片（第 114–125 行）
3. **5 分钟熔断**：`setTimeout(300000)` 防止 Kotlin 侧异常崩溃后前端永远挂起

### 5.4 乐观更新与错误回滚

**乐观更新**：用户消息在调用任何 IPC 之前就已经 `currentChatHistory.push(userMsg)`（第 329 行）。这保证了无论网络延迟多大，用户都能立即在列表底部看到自己的消息。

**流式骨架持久化**：在 `processStreamEvent` 中收到 `thinking` 事件时（第 183–203 行）：

此异步调用将空内容的骨架消息写入 SQLite，使得用户中途切换会话后再返回，历史加载能通过 Object Hydration 从 `activeStreamMessages` 池中找回"活"的响应式对象，流式动画无缝接续。

**`chatHistoryStore` 的分页加载策略**：

```typescript

  historyOffset.value = 0;
  hasMoreHistory.value = true;
  await loadHistory(ownerId, ownerType, topicId, 5, 0);  // limit=5, offset=0
};

// 加载更多（用户上滑到顶时）

  if (!hasMoreHistory.value || isLoadingHistory.value) return;
  await loadHistory(ownerId, ownerType, topicId, 10, historyOffset.value);
};
```

- 初始 limit = 5：仅加载最新消息，确保首屏快速渲染
- 增量 limit = 10：平衡加载速度与请求频率
- `historyOffset` 跟踪已加载总数，作为下次请求的 offset
- `hasMoreHistory` 在返回消息数 < limit 时置为 false，阻止后续空请求

**会话一致性校验**：

在 `loadHistory` 的 Channel `onmessage` 回调中（第 104–108 行）：
```typescript
channel.onmessage = (chunk) => {
  if (sessionStore.currentTopicId !== requestedTopicId && requestedTopicId !== null) {
    return; // 用户已切换话题，丢弃 stale 数据
  }
  ...
};
```

此守卫防止"快速切换话题"导致的旧数据污染新会话。由于历史加载是异步的，用户可能在加载完成前切换到另一个话题，此时后续 chunk 应被静默丢弃。

**错误处理**：
- 若 `processStreamEvent` 收到 `type === "error"`，将错误信息追加到消息内容末尾（`msg.content += errorText`），并标记 `finishReason = "error"`。
- 若 `process_message_content` 调用失败，消息仍保留原始文本内容，仅 blocks 回退为空数组。
- 用户可通过点击"重新渲染"按钮触发 `reRenderMessage` 重试 Block 解析。

**编辑重发分支**：

当 `editingOriginalMessageId.value` 非空时（第 290–308 行），`sendMessage` 进入编辑重发模式：

```
用户点击"编辑重发"
    │
    ▼
┌────────────────────────────────────────┐
│ 1. 清空 editingOriginalMessageId       │
│ 2. 找到原消息在 currentChatHistory 中的索引│
│ 3. 更新原消息的 content 与 blocks       │
│ 4. invoke('truncate_history_after_timestamp')│
│    → Rust 删除该消息时间戳之后的所有记录  │
│ 5. currentChatHistory.splice(targetIndex+1)│
│    → 前端同步截断                       │
│ 6. topicStore.decrementTopicMsgCount() │
│ 7. triggerGeneration(targetMsg)        │
│    → 以修改后的消息作为最后一条用户消息   │
│      重新触发 AI 生成                   │
└────────────────────────────────────────┘
```

此机制实现了"修改历史消息并重新生成后续回复"的能力，是 VCP Mobile 对话体验的核心交互之一。

**Channel 数据结构的演进**：

Tauri v2 的 `Channel<T>` 是前端与 Rust 之间双向流式通信的核心机制。在对话引擎中使用了两种 Channel 类型：

1. **`Channel<HistoryChunk>`**（历史加载）：
   ```typescript
   interface HistoryChunk {
     message: ChatMessage;
     index: number;      // 在批次中的索引
     is_last: boolean;   // 是否为最后一条
   }
   ```
   - Rust 通过 `channel.send(chunk)` 逐条推送历史消息
   - 前端在 `onmessage` 中根据 `offset` 决定是 `push`（初始加载）还是 `buffer` + `prepend`（分页加载）
   - `is_last` 触发 `resolveComplete`，释放加载完成的 Promise

2. **`Channel<StreamEvent>`**（流式响应）：
   事件类型包括 `thinking`、`data`、`aurora`、`end`、`error` 五种，由 `chatStreamStore.processStreamEvent` 统一调度。

两种 Channel 的共同特点：
- 均为**单向推送**（Rust → 前端），前端无法通过同一 Channel 回传数据
- 需要在前端显式创建 `new Channel<T>()` 并作为参数传入 `invoke`
- Channel 的生命周期与单次 IPC 调用绑定，调用结束后自动关闭

## 8. 设计决策与注意事项

### 8.1 为何放弃 IntersectionObserver 改用 scroll 事件状态机？

早期版本使用 `IntersectionObserver` 配合哨兵元素实现无限滚动与自动置底，但存在三个致命问题：
1. `rootMargin` 在移动 WebView 中行为不一致，导致"即将到底"的判定飘忽
2. `nextTick` 后操作滚动位置时，异步图片加载会导致布局持续变化，时机难以预测
3. 用户手动上滑浏览历史时，新消息到达后的自动置底会造成"回弹"，严重干扰阅读

`useChatScroll` 采用 **scroll 事件精确几何检测 + ResizeObserver 物理高度守卫 + 四状态机** 的组合，从根本上解决了上述问题。

### 8.2 为何 `sendMessage` 中先做乐观更新再调用 IPC？

用户消息的乐观追加（第 329 行）发生在 `triggerGeneration` 之前。这带来了两个好处：
1. **零感知延迟**：用户点击发送后立刻在列表底部看到消息，无需等待 Rust 持久化完成
2. **流式连贯性**：用户消息与后续的 thinking 骨架在列表中相邻，视觉上是连续的

代价：若 Rust `append_single_message` 失败，前端存在"已展示但未持久化"的消息。当前策略是依赖 Rust 侧的错误日志与后续加载时的状态校准，暂不实现前端主动回滚。

### 8.3 语音交互的双轨设计

InputEnhancer 实现了**两种语音输入模式**，彼此独立又共享同一物理按钮：

| 模式 | 触发方式 | 产物 | 适用场景 |
|------|----------|------|----------|
| **STT 语音转文字** | Tap 切换为语音模式 → 按住说话大条 | 文本填入输入框 | 用户想说一段话，但想编辑后再发 |
| **音频附件直发** | 长按左侧麦克风图标 (350ms+) | `.webm` 附件 + 立即发送 | 用户想发语音消息，不转文字 |

核心修复（第 178–184 行）：当处于语音模式时，点击麦克风图标判定为"切换回键盘"而非"开始长按录音"，彻底解决了 touchstart.prevent 导致 click 无法触发的"点不回去"手势缺陷。

### 8.4 全局流消息池的 OOM 防御

`chatStreamStore.activeStreamMessages` 是一个全局 `Map<string, ChatMessage>`，理论上无界增长会导致内存泄漏。防御策略：

1. **上限保护**：`MAX_STREAM_MESSAGES = 100`（第 76 行）
2. **按插入顺序清理最旧非活跃消息**（`enforceStreamPoolLimit`，第 78–89 行）
3. **延迟清理**：流结束后 1 秒才从池中删除（第 130–135 行），确保 `finalizeStream` 期间仍能通过 `messageId` 获取对象

### 8.5 附件大小分流的阈值选择

- **2MB**：`store_file` IPC 与 `prepare_vcp_upload` 高速链路的分界线
  - <2MB：IPC 直接传递 `Uint8Array`，Tauri v2 支持二进制跨层零拷贝
  - ≥2MB：XHR 流式上传，避免 IPC payload 过大导致序列化阻塞与内存峰值
- **10MB**：非 GIF 图片的拒绝阈值，防止 WebView 解码超限图片导致 OOM
- **8K×8K**：非 GIF 图片的分辨率拒绝阈值，超出此范围直接 Toast 提示压缩后重试

### 8.6 为何 `chatStreamStore` 需要全局流消息池？

在 Backend-Driven Streaming 架构之前，流式消息直接插入 `chatHistoryStore.currentChatHistory`，存在两个致命问题：

1. **前后台切换断流**：用户切到别的 Agent 再切回来，历史重新加载时会得到 DB 中的旧版本消息对象，失去响应式链接，流式动画停止
2. **群聊并发混乱**：多个 Agent 同时生成时，各自的增量更新相互覆盖

`activeStreamMessages: Map<string, ChatMessage>` 作为全局池解决了上述问题：
- 任何组件、任何会话都可以通过 `messageId` 获取到同一个响应式对象
- 历史加载时的 Object Hydration（第 113–114 行）用池中活对象替换骨架，保证流式状态不丢失
- 群聊模式下每个 Agent 的回复拥有独立的 `messageId`，天然隔离

### 8.7 `useChatScroll` 的 RAF 双重节流机制

`useChatScroll` 在两个层面使用 `requestAnimationFrame` 进行性能保护：

1. **ResizeObserver 回调内**：`scrollRafId = requestAnimationFrame(() => handleContentChange())`
   - 合并单帧内的高频尺寸变动（如批量图片加载完成时的连续重排）
2. **scroll 事件内**：`scrollThrottleId = requestAnimationFrame(() => { ... })`
   - 将 scroll 事件从"每像素触发"降频到"每帧触发"，在 60fps 下相当于 16ms 节流

此外，`triggerAutoLoadMoreWithDebounce` 使用 200ms 的 `setTimeout` 防抖，等待图片、头像与异步排版彻底稳定后再执行高度判定，避免"加载过早"导致的连续请求。
