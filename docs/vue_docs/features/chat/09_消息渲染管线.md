# 09. 消息渲染管线

## 1. 概述

### 1.1 领域定位

消息渲染管线是 VCP Mobile 前端 **Chat 领域** 的视觉核心层，负责将 Rust 后端返回的 `ChatMessage` 数据对象转化为用户可见的交互式消息界面。其设计目标是在移动端 WebView 的有限性能预算下，支撑高密度对话列表、流式增量渲染、富文本排版（KaTeX/Mermaid/代码高亮）与复杂长按交互。

该管线采用 **"单消息级渲染"** 架构：`ChatView.vue` 作为列表容器维护 `v-for` 循环，每条消息由独立的 `MessageRenderer.vue` 实例负责自包含渲染。这种设计将列表级的滚动管理（`useChatScroll`）与消息级的内容排版（Markdown AST 渲染、重内容懒加载）解耦，避免单文件膨胀，同时保证单条消息的增删改不会触发全列表重排。

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **单消息自包含** | 每条消息由独立 `MessageRenderer` 实例渲染，状态局部化，避免列表级重排 |
| **响应式对象劫持 (Hydration)** | 从 DB 加载的骨架消息若处于活跃流中，会被全局流池中的"活的"响应式对象替换，保证流式动画不中断 |
| **重内容惰性渲染** | KaTeX / Mermaid / 表情符号仅在消息离开活跃流后执行一次，流式期间完全跳过，防止 O(n) 遍历拖垮性能 |
| **Block 级不可变缓存** | `v-memo=[blockKey]` 保证已稳定的 Block 零开销：Vue 缓存 VNode 子树，不触碰 DOM |
| **Native Virtual Scrolling** | 使用 CSS `content-visibility: auto` + `contain-intrinsic-size` 实现浏览器原生虚拟滚动，而非 JS 虚拟列表 |
| **场景驱动滚动管理** | `useChatScroll` 采用 "initial / following / free / loading-top" 四场景状态机，替代模糊逻辑的 IntersectionObserver |

## 2. 消息列表（MessageRenderer.vue 与 ChatView.vue 的协作）

### 2.1 组件结构

需要首先澄清一个常见的误解：`MessageRenderer.vue` **不是**消息列表本身，而是**单条消息的自包含渲染器**。真正的列表容器是 `ChatView.vue`。

> 文件位置：`src/features/chat/ChatView.vue` 第 244–254 行

```vue
<div class="messages-inner-container py-4 space-y-2 flex flex-col min-h-full">
  <MessageRenderer
    v-for="msg in historyStore.currentChatHistory"
    :key="msg.id"
    :message="msg"
    :agent-id="sessionStore.currentSelectedItem?.id"
    :data-message-id="msg.id"
  />
  <div class="h-20"></div>  <!-- 底部留白，避免输入框遮挡 -->
</div>
```

`ChatView.vue` 维护一个 `overflow-y-auto` 的滚动容器（`messageListRef`），内部通过 `v-for` 遍历 `historyStore.currentChatHistory` 渲染全部消息。`MessageRenderer.vue` 接收单条 `ChatMessage` 作为 `props`，负责该消息的完整视觉呈现。

### 2.2 与 chatHistoryStore / chatStreamStore 的绑定

消息列表的数据源有两条路径：

**路径 A：历史消息加载（冷数据）**

```
用户切换话题
    │
    ▼
┌─────────────────────┐
│ ChatView.vue        │──> watch(sessionStore.currentTopicId)
│                     │    触发 historyStore.loadHistoryPaginated()
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ chatHistoryStore    │──> 调用 Rust: load_chat_history_streamed
│                     │    通过 Channel<HistoryChunk> 逐条接收
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Channel.onmessage   │──> 将 chunk.message push 到
│                     │    currentChatHistory.value[]
└─────────────────────┘
```

**路径 B：流式消息注入（热数据）**

```
SSE StreamEvent
    │
    ▼
┌─────────────────────┐
│ chatStreamStore     │──> processStreamEvent()
│                     │    创建 reactive<ChatMessage> 存入
│                     │    activeStreamMessages (全局流池)
└──────────┬──────────┘
           │
           ├───> onMessageCreated 回调 ───> currentChatHistory.push(msg)
           │
           └───> 流式期间 content / blocks / tailContent 增量更新
                      │
                      ▼
              Vue 响应式自动触发对应 MessageRenderer 重渲染
```

### 2.3 虚拟滚动与性能优化

项目**未采用 JS 虚拟列表**（如 `vue-virtual-scroller`），而是使用 CSS 原生机制实现等效的渲染优化：

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 501–506 行

```css
.vcp-message-item {
  content-visibility: auto;
  contain-intrinsic-size: auto 100px;
}
```

| 机制 | 作用 |
|------|------|
| `content-visibility: auto` | 浏览器自动跳过视口外元素的渲染与布局计算，效果等同于虚拟滚动 |
| `contain-intrinsic-size: auto 100px` | 为被跳过的元素提供预估尺寸，防止滚动条跳动 |

此外，`MessageRenderer.vue` 在 Block 级别使用 `v-memo` 做细粒度缓存：

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 431–432 行

```vue
<div v-memo="[getBlockKey(block, index)]">
  <!-- Block 内容 -->
</div>
```

`getBlockKey` 优先使用 `block.hash`（由 Rust AST 解析器生成的稳定哈希），若不存在则回退到 `index`。这意味着一旦某个 Block 的 `hash` 不变，Vue 将完全跳过该子树的 Diff 与 DOM 操作。

### 2.4 消息分组与日期分隔

当前版本（v0.9.14）的消息列表**不实现传统的日期分隔线**。消息按时间戳顺序线性排列，时间信息以微缩字体显示在每条气泡的 footer 区域。这种设计符合 VCP Mobile "高密度线性布局" 的 UI 美学宪法。

## 3. 消息气泡（ChatBubble.vue）

### 3.1 Props 与 Emits

> 文件位置：`src/features/chat/components/ChatBubble.vue`

```ts
const props = defineProps<{
  isUser: boolean;
  isStreaming: boolean;
  bubbleStyle?: Record<string, string>;
}>();
```

| Prop | 类型 | 说明 |
|------|------|------|
| `isUser` | `boolean` | 是否为用户消息，决定左右对齐与气泡样式 |
| `isStreaming` | `boolean` | 是否处于流式状态，控制流光边框动画 |
| `bubbleStyle` | `Record<string, string>` | 动态 CSS 变量注入，用于 Dominant Color 边框阴影 |

`ChatBubble.vue` 是一个**纯布局容器**，不接收消息内容本身。内容通过默认插槽传入，footer（时间戳）通过命名插槽 `#footer` 传入。

### 3.2 左右布局（用户 vs AI）

左右布局通过 Flexbox 实现，而非 margin 偏移：

> 文件位置：`src/features/chat/components/ChatBubble.vue` 第 16–29 行

```vue
<div class="w-full min-w-0 flex flex-col" :class="[
  isUser ? 'items-end' : 'items-start',
  isStreaming ? 'streaming' : '',
]">
  <div
    class="vcp-bubble-container message-bubble rounded-2xl ..."
    :class="[
      isUser ? 'p-3 w-fit max-w-[85%] vcp-bubble-user'
             : 'p-1.5 w-fit max-w-[100%] min-w-[1rem] vcp-bubble-agent'
    ]"
    :style="mergedStyle"
  >
    <slot />
  </div>
  <slot name="footer" />
</div>
```

- 外层 `flex flex-col` + `items-end` / `items-start` 控制整列（气泡 + footer）的左右对齐
- 用户消息：`max-w-[85%]`，内边距 `p-3`，内容密度较低（用户输入通常为短文本）
- AI 消息：`max-w-[100%]`，内边距 `p-1.5`，允许更宽以容纳代码块、表格等富内容

### 3.3 状态样式（发送中、失败、编辑中）

当前版本中，消息没有显式的 "发送中" / "失败" 状态样式。用户消息在 `sendMessage()` 中立即以完整对象推入 `currentChatHistory`，UI 瞬时呈现，无需等待网络确认。

流式状态的视觉反馈由 `isStreaming` 驱动：

> 文件位置：`src/features/chat/components/ChatBubble.vue` 第 59–80 行

```css
.streaming .vcp-bubble-container::before {
  content: "";
  position: absolute;
  inset: -1px;
  padding: 1px;
  border-radius: inherit;
  background: linear-gradient(90deg,
      transparent 25%,
      var(--highlight-text, #3b82f6) 50%,
      transparent 75%);
  background-size: 200% 100%;
  -webkit-mask: ...;
  animation: vcp-border-flow 4s linear infinite;
  opacity: 0.6;
}
```

流光边框使用 `::before` 伪元素 + `-webkit-mask` 实现镂空渐变边框效果。颜色 fallback 为 `#3b82f6`（Tailwind Blue-500），动画周期 4 秒，无限循环。

### 3.4 长按菜单交互

长按菜单由 `v-longpress` 自定义指令触发：

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 404–405 行

```vue
<div v-longpress="showMessageContextMenu" ...>
```

`v-longpress` 指令（`src/core/directives/longpress.ts`）绑定了一系列 Touch/Mouse 事件：

| 事件 | 行为 |
|------|------|
| `touchstart` / `mousedown` | 启动 600ms 计时器 |
| `touchend` / `mouseup` / `touchcancel` / `mouseleave` | 取消计时器 |
| `touchmove` / `mousemove` | 标记为移动状态，取消计时器 |
| `contextmenu` | 拦截桌面端原生右键菜单，直接触发回调 |

触发时若 `navigator.vibrate` 可用，执行 50ms 震动反馈。

`showMessageContextMenu` 构建的操作列表：

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 271–371 行

| 操作 | 图标 | 条件 | 行为 |
|------|------|------|------|
| 中止回复 | `StopCircle` | `isStreaming && !isUser` | `streamStore.stopMessage(id)` |
| 复制内容 | `Copy` | 始终可用 | `navigator.clipboard.writeText(fullText)` |
| 重新渲染 | `RotateCcw` | `!isStreaming` | `historyStore.reRenderMessage(id, topicId)` |
| 编辑消息 | `Edit2` | `!isStreaming` | `overlayStore.openEditor()` → `historyStore.updateMessageContent()` |
| 重新生成 | `RotateCcw` | `!isStreaming && !isUser` | `historyStore.regenerateResponse(id)` |
| 编辑重发 | `Edit2` | `!isStreaming && isUser` | 将内容填入输入框，标记 `editingOriginalMessageId` |
| 删除消息 | `Trash2` | 始终可用 | `confirm()` → `historyStore.deleteMessage(id)` |

操作列表通过 `overlayStore.openContextMenu(actions, side)` 弹出全局上下文菜单。

### 4.2 时间戳格式化

时间戳不在 `MessageHeader.vue` 中显示，而是由 `MessageRenderer.vue` 通过 `ChatBubble` 的 `#footer` 插槽注入：

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 491–496 行

```vue
<template #footer>
  <div class="text-[9px] mt-1.5 px-1 opacity-50 font-mono tracking-tighter w-full"
```

`formatTime` 使用 `Date.toLocaleTimeString` 格式化为 `HH:MM:SS`，配合 `font-mono` 保证等宽数字对齐。

### 4.3 群聊模式下的角色标识

群聊模式下，`MessageHeader` 展示的是发言 Agent 的个体信息，而非群组名称。`displayName` 和 `avatarDominantColor` 来自 `message.shell`，而该 `shell` 在流式初始化时由 `chatStreamStore.computeShell()` 根据 `agentId` 动态计算：

> 文件位置：`src/core/stores/chatStreamStore.ts` 第 31–53 行

```ts
function computeShell(msg: { role: string; agentId?: string; name?: string }): MessageShell {
  if (msg.role === "user") { /* 用户固定红色 */ }
  const agent = msg.agentId
    ? assistantStore.agents.find((a) => a.id === msg.agentId)
    : undefined;
  return {
    avatarColor: agent?.avatarCalculatedColor || "",
    displayName: msg.name || agent?.name || "AI",
    isUser: false,
    ...
  };
}
```

因此，群聊中每条 AI 消息的气泡边框颜色、头像边框发光色、名称文本色均与该 Agent 的 `avatarCalculatedColor` 保持一致，实现视觉上的 "谁发言、谁配色"。

## 5. 头像系统（ChatAvatar.vue）

### 5.1 头像加载与缓存

`ChatAvatar.vue` 是一个**桥接代理组件**，它不直接加载头像，而是将参数转译为 `VcpAvatar.vue` 能理解的格式。

> 文件位置：`src/features/chat/components/ChatAvatar.vue`

```ts
const ownerInfo = computed(() => {
  // 1. 优先使用显式参数
  if (props.ownerType && props.ownerId) {
    return { type: props.ownerType, id: props.ownerId };
  }
  // 2. 兼容旧代码：解析 vcp-avatar:// 协议 URL
  if (props.imageUrl && props.imageUrl.startsWith('vcp-avatar://')) {
    const parts = props.imageUrl.replace('vcp-avatar://', '').split('/');
    ...
  }
  return null;
});
```

真正的头像加载发生在 `VcpAvatar.vue` 中，它通过 `avatarStore.getAvatarUrl()` 获取 Blob URL：

> 文件位置：`src/components/ui/VcpAvatar.vue` 第 54–80 行

```ts
watchEffect(async () => {
  const key = `${props.ownerType}:${props.ownerId}`;
  // 同步检查缓存，命中则立即显示
  const existing = avatarStore.cache.get(key);
  if (existing && (reqVersion === 0 || existing.version >= reqVersion)) {
    avatarUrl.value = existing.blobUrl;
    imgExists.value = true;
    return;
  }
  // 缓存未命中，异步获取
  const url = await avatarStore.getAvatarUrl(...);
  ...
});
```

`avatarStore`（`src/core/stores/avatar.ts`）使用 `reactive(Map)` 缓存 Blob URL，缓存键为 `{ownerType}:{ownerId}`。缓存上限为 50 条，超出时按 FIFO 淘汰并 `URL.revokeObjectURL` 释放内存。

### 5.2 Dominant Color 背景

`dominantColor` 贯穿整个头像视觉系统：

1. **Rust 后端提取**：头像上传时通过 `extract_dominant_color_from_bytes()` 计算（512-bin 直方图 + HSV 过滤）
2. **数据库存储**：`avatars.dominant_color` 字段持久化
3. **前端同步缓存**：`avatarStore.dominantColors` Map 提供同步读取能力
4. **UI 应用**：
   - `VcpAvatar.vue` 中作为 fallback 背景色（无头像时首字母占位块的底色）
   - `MessageHeader.vue` 中作为名称文本色
   - `ChatBubble.vue` 中作为 `--dynamic-color` 注入气泡阴影
   - `ChatAvatar.vue` 中作为头像边框色

### 5.3 首字母占位 Avatar

当头像图片不存在或加载失败时，`VcpAvatar.vue` 展示首字母占位：

> 文件位置：`src/components/ui/VcpAvatar.vue` 第 95–101 行

```vue
<div class="absolute inset-0 flex items-center justify-center text-white font-bold select-none"
     :style="{ backgroundColor: fallbackBg }">
  {{ initial }}
</div>
```

`fallbackBg` 优先使用 `props.dominantColor`，若未提供则通过 `ownerId` 的字符串哈希从 5 色板中确定性地选取一种，保证同一实体的占位色始终一致。

### 6.3 与 chatStreamStore 的联动

流式指示器的显示条件在 `MessageRenderer.vue` 中计算：

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 57–72 行

```ts
const isStreaming = computed(() => {
  if (shell.value?.isUser) return false;

  const isGroup = !!props.message.isGroupMessage || ...;
  const itemId = isGroup
    ? (props.message.groupId || sessionStore.currentSelectedItem?.id)
    : (props.message.agentId || props.agentId);

  const topicId = sessionStore.currentTopicId;
```

关键设计：`isStreaming` 不仅检查消息是否在**任意**活跃流中（`isMessageInActiveStream`），还限定为**当前视口话题**的活跃流。这确保了多话题并发流式时，切换话题后旧话题的流式指示器不会继续显示。

### 7.3 Object Hydration（对象劫持）详解

> 文件位置：`src/core/stores/chatHistoryStore.ts` 第 110–114 行

```ts
const activeMsg = streamStore.activeStreamMessages.get(chunk.message.id);
const msgToUse = activeMsg || chunk.message;
```

当用户切换话题时，`loadHistory` 从 SQLite 通过 Channel 流式加载历史消息。若某条消息正处于活跃流中（用户发送后快速切走又切回），DB 返回的是一个**死的**普通对象。此时 `chatStreamStore.activeStreamMessages` 中已存在一个**活的** `reactive<ChatMessage>` 对象。

`Object Hydration` 就是用活的响应式对象替换死的骨架对象，确保：

1. 流式增量更新继续生效（响应式链路未断）
2. `isThinking` / `tailContent` 等运行时状态得以保留
3. Vue 无需重新创建组件实例，DOM Diff 效率最大化

## 9. 设计决策与注意事项

### 9.1 为何不使用 JS 虚拟列表？

VCP Mobile 的聊天场景具有以下特征，使得传统 JS 虚拟列表（如只渲染视口内元素）的收益有限：

1. **消息高度不可预估**：Markdown 富文本、代码块、Mermaid 图表、图片附件导致每条消息高度差异巨大
2. **滚动方向以底部为主**：用户大部分时间关注最新消息，向上翻查历史的频率较低
3. **CSS `content-visibility` 足够高效**：现代 Chromium WebView（Android 系统 WebView 120+）对该属性支持良好，能以原生性能跳过离屏渲染

因此项目选择 `content-visibility: auto` 而非引入第三方虚拟滚动库，在保持代码简洁的同时获得接近虚拟滚动的性能。

### 9.2 为何流式期间跳过重内容渲染？

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 167–245 行

`renderHeavyContent` 负责 KaTeX 数学公式、Mermaid 图表和表情符号替换。这三项操作的共同特征是：

- **计算密集**：KaTeX 需要解析 LaTeX 语法并生成 DOM；Mermaid 需要完整的图表布局算法
- **IO 密集**：两者均为动态导入（`await import('katex')`、`await import('mermaid')`），涉及 JS 代码下载与解析
- **高频触发风险**：流式期间每收到一个 chunk 都可能触发 `watch`，若执行重渲染则每次都要扫描全部 DOM

因此代码在流式期间完全短路：

```ts
if (isMessageInActiveStream.value) return;
```

只有在流式真正结束后（`isMessageInActiveStream` 从 `true` 变为 `false`），才执行一次性的完整重渲染。这种"延迟批处理"策略将 O(n×chunks) 的复杂度降为 O(n)。

### 9.3 v-memo 的 Hash 策略

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 149–155 行

```ts
function getBlockKey(block: ContentBlock, index: number): string {
  if (block.hash !== undefined && block.hash !== null) {
    return `${block.type}-${String(block.hash)}`;
  }
  return `${block.type}-idx-${index}`;
}
```

Rust 后端在解析 Markdown 时会为每个 Block 计算一个稳定哈希（通常基于内容的 SHA-256 前缀或语法树指纹）。`v-memo` 依赖此哈希实现 Block 级缓存。若使用 `index` 作为 key，则数组插入会导致后续所有 Block 的缓存失效；而使用 `block.hash` 则只影响发生变化的 Block。

### 9.4 流式尾部高画质推测渲染（Speculative Rendering）

> 文件位置：`src/features/chat/MessageRenderer.vue` 第 466–480 行

```vue
<div v-if="isStreaming && message.tailBlock" class="streaming-tail opacity-90">
  <div v-if="isPlainBlock(message.tailBlock.type)" v-html="renderBlockHtml(message.tailBlock)" />
  <ThoughtBlock v-else-if="message.tailBlock.type === 'thought'" ... />
</div>
<div v-else-if="isStreaming && message.tailContent" class="opacity-70 italic animate-pulse">
  {{ message.tailContent }}
</div>
```

Aurora 流式管道将内容分为 **stableBlocks**（已确认稳定的结构化块）和 **tailBlock/tailContent**（仍在变动的尾部）。尾部以较低透明度（`opacity-90` 或 `opacity-70`）叠加在稳定内容下方，给用户"正在逐字生成"的实时感知，同时避免对稳定区域的频繁重排。

### 9.5 useChatScroll 的四场景状态机

> 文件位置：`src/core/composables/useChatScroll.ts`

```
┌─────────┐    首屏加载完成      ┌───────────┐
│ initial │─────────────────────>│ following │
└────┬────┘    (scrollToBottom)   └─────┬─────┘
     │                                  │
     │ 无消息                            │ 用户手动上滑
     │                                  │ (scrollTop 远离底部)
     ▼                                  ▼
┌─────────┐    用户滑回底部          ┌─────────┐
│  free   │<─────────────────────────│  free   │
│ (空状态)│    (nearBottom)          │(自由浏览)│
└─────────┘                          └────┬────┘
     ▲                                    │
     │ 分页加载完成                       │ 上滑到顶部
     │ (restoreScrollByAnchor)            │ (nearTop + hasMore)
     │                                    ▼
     └────────────────────────────┐  ┌───────────┐
                                  └──┤loading-top│
                                     │(加载历史中)│
                                     └───────────┘
```

- **initial**：首屏加载中，ResizeObserver 检测到内容变化时自动置底
- **following**：用户位于底部，新消息/流式内容自动跟随滚动
- **free**：用户手动上滑离开底部，停止自动跟随，显示"置底按钮"
- **loading-top**：用户上滑到顶部触发历史加载，记录锚点元素以恢复滚动位置

### 9.6 群聊停止按钮的设计

> 文件位置：`src/features/chat/components/GroupStopAllButton.vue`

`GroupStopAllButton` 是独立于 `MessageRenderer` 的悬浮组件，直接挂载在 `ChatView.vue` 的消息列表区域上方。它同时监听两个条件：

```ts
const isVisible = computed(() => {
  return streamStore.isGroupGenerating && streamStore.activeStreamingIds.size > 0;
});
```

- `isGroupGenerating`：确保当前会话类型为 group
- `activeStreamingIds.size > 0`：确保有活跃流在进行

点击后调用 `streamStore.stopGroupTurn(topicId)`，该函数先向 Rust 发送 `interruptGroupTurn` 命令，再对前端本地所有活跃流调用 `stopMessage()` 做乐观状态清理。
