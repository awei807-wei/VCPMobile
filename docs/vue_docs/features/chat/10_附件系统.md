# 10. 附件系统

## 1. 概述

### 1.1 领域定位

`attachment/` 是 VCP Mobile 前端 `features/chat/` 模块下的**独立子领域**，负责附件的完整生命周期管理：从用户选择文件、类型分类、上传到消息内的预览渲染，再到全屏查看与外部应用打开。该子领域采用**插件化架构（Plugin Architecture）**，所有附件类型通过注册表动态登记，渲染器根据类型自动调度，新增类型无需修改核心逻辑。

领域边界严格限定为：

- **类型分类**：基于 MIME 类型与文件扩展名的双轨分类策略
- **组件注册与调度**：注册表模式管理 7 种明确类型 + 1 个兜底类型的渲染组件
- **预览与渲染**：消息内缩略图预览（含加载态、删除按钮）、全屏查看器
- **上传链路**：Android 原生 File Picker → Kotlin 沙盒拷贝 → Rust 注册；非 Android HTML Input → IPC / 高速 TCP 双轨分流

该领域**不**涉及：
- 文件物理存储本身（由 Rust `file_manager.rs` 负责）
- 文档内容提取（由 `useDocumentProcessor` 负责，仅在消息发送前 JIT 调用）
- 消息持久化逻辑（由 `message_service.rs` 负责）

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **插件化注册** | 附件类型与渲染组件解耦，通过 `AttachmentRegistry` 动态映射。新增类型只需：①定义枚举值 ②创建 Vue 组件 ③调用 `register()` |
| **懒加载渲染** | 全部 7 种类型组件通过 `defineAsyncComponent` 懒加载，避免首屏打包体积膨胀 |
| **双轨分类** | 分类器优先信任 MIME 类型前缀（`image/`, `video/`, `audio/`），MIME 为空或通用时回退到扩展名白名单 |
| **平台适配上传** | Android 走原生 File Picker（绕过 WebView 文件选择限制）；非 Android 走标准 HTML `<input>`，小文件 IPC、大文件高速 TCP |
| **内容寻址去重** | 后端以 SHA-256 哈希作为内部文件名，相同内容物理只存一份 |
| **渐进式预览** | 图片优先展示缩略图（`thumbnailPath`），文本文件流式读取前 128KB 避免内存爆炸 |

## 2. 附件类型系统

### 2.1 AttachmentType 枚举与接口

前端附件类型枚举定义极度精简，仅包含 7 个值 + 1 个兜底：

> 文件位置：`src/features/chat/attachment/types/AttachmentType.ts`

```typescript
export enum AttachmentType {
  IMAGE = 'image',
  VIDEO = 'video', 
  AUDIO = 'audio',
  DOCUMENT = 'document',
  CODE = 'code',
  TEXT = 'text',
  OTHER = 'other'
}
```

| 枚举值 | 字符串 | 典型文件 | 渲染策略 |
|--------|--------|----------|----------|
| `IMAGE` | `image` | jpg, png, gif, webp, svg | 缩略图填充卡片，点击全屏查看 |
| `VIDEO` | `video` | mp4, webm | 文件图标卡片，点击调用系统播放器 |
| `AUDIO` | `audio` | mp3, wav, ogg, flac | 文件图标卡片，点击调用系统播放器 |
| `DOCUMENT` | `document` | pdf, docx, xlsx, pptx | 文件图标 + 扩展名标签卡片 |
| `CODE` | `code` | js, ts, py, rs, java, vue | 代码图标 + 扩展名标签卡片 |
| `TEXT` | `text` | txt, md, csv, json, xml | 文本图标卡片，点击可全屏预览 |
| `OTHER` | `other` | 任何未匹配类型 | 通用文件图标卡片 |

### 2.2 与后端 Attachment 类型的映射

前端 `Attachment` 接口（`src/core/types/chat.ts` 第 73–88 行）与 Rust 侧的 `AttachmentSyncDTO` / `AttachmentData` 保持字段对齐：

| 前端字段 | Rust 对应 | 说明 |
|----------|-----------|------|
| `id` | 纯前端字段 | UI 稳定性标识，`att_${timestamp}_${random}` |
| `type` | `mime_type` | MIME 类型字符串，如 `image/jpeg` |
| `name` | `original_name` | 原始文件名 |
| `size` | `size` | 字节数 |
| `src` | `internal_path` | 物理存储路径，前端通过 `convertFileSrc` 转为 `asset://` URL |
| `resolvedSrc` | 运行时动态生成 | `asset://` 转换后的安全 URL，**不持久化** |
| `hash` | `hash` | SHA-256 十六进制，内容寻址键 |

## 3. 附件注册表（AttachmentRegistry）

### 3.1 注册表设计模式

`AttachmentRegistry` 是实现插件化架构的核心设施，采用**静态类 + 私有 Map** 模式：

> 文件位置：`src/features/chat/attachment/AttachmentRegistry.ts`

```typescript
export class AttachmentRegistry {
  private static componentMap = new Map<AttachmentType, Component>();

  static register(type: AttachmentType, component: Component): void
  static registerMultiple(mappings: Record<AttachmentType, Component>): void
  static getComponent(type: AttachmentType): Component | null
  static hasComponent(type: AttachmentType): boolean
  static unregister(type: AttachmentType): void
  static clear(): void
  static get count(): number
}
```

关键设计决策：
- **静态类而非实例**：注册表是全局单例，整个应用生命周期内类型映射不变
- **`defineAsyncComponent` 懒加载**：所有组件在注册时即包装为异步组件，仅在首次渲染时触发网络/磁盘加载
- **运行时可扩展**：`register` / `unregister` 支持动态增删，为未来插件市场预留接口

### 3.2 8 种附件类型的注册

注册表在模块导入时自动完成默认组件注册（文件底部自执行代码）：

```typescript
AttachmentRegistry.registerMultiple({
  [AttachmentType.IMAGE]: defineAsyncComponent(() => import('./types/ImageAttachment.vue')),
  [AttachmentType.VIDEO]: defineAsyncComponent(() => import('./types/VideoAttachment.vue')),
  [AttachmentType.AUDIO]: defineAsyncComponent(() => import('./types/AudioAttachment.vue')),
  [AttachmentType.DOCUMENT]: defineAsyncComponent(() => import('./types/DocumentAttachment.vue')),
  [AttachmentType.CODE]: defineAsyncComponent(() => import('./types/CodeAttachment.vue')),
  [AttachmentType.TEXT]: defineAsyncComponent(() => import('./types/TextAttachment.vue')),
  [AttachmentType.OTHER]: defineAsyncComponent(() => import('./types/OtherAttachment.vue')),
});
```

### 3.3 运行时类型解析

渲染调度时序：

```
AttachmentRenderer.vue 接收 props.file
            │
            ▼
    classifyAttachment(file.type, file.name)
            │
            ▼
    返回 AttachmentType (如 IMAGE)
            │
            ▼
    AttachmentRegistry.getComponent(IMAGE)
            │
            ▼
    返回 ImageAttachment.vue 异步组件
            │
            ▼
    <component :is="..."> 动态挂载
```

> 兜底策略：若注册表中找不到对应类型（理论上不会发生，因为 `OTHER` 始终已注册），回退到 `AttachmentType.OTHER` 组件。

## 5. 附件分类器（AttachmentClassifier）

### 5.1 MIME 类型分类逻辑

`classifyAttachment` 是前端类型判定的**唯一收敛点**，与 Rust `file_manager.rs:get_refined_mime_type` 保持逻辑一致：

> 文件位置：`src/features/chat/attachment/utils/AttachmentClassifier.ts`

分类优先级：
1. **强制覆盖**：扩展名为 `mp3` 强制归类为 `AUDIO`（匹配 Rust 侧特殊处理）
2. **MIME 前缀匹配**：`image/`, `video/`, `audio/`, `text/` → 直接映射
3. **MIME 子串匹配**：`pdf`, `word`, `excel`, `powerpoint`, `msword`, `sheet`, `presentation` → `DOCUMENT`
4. **MIME 代码检测**：`javascript`, `ecmascript`, `json`, `xml` → `CODE`
5. **扩展名 Fallback**：当 MIME 为空或 `application/octet-stream` 时，按扩展名白名单分类
6. **最终兜底**：`OTHER`

### 5.2 文件扩展名 Fallback

当 MIME 不可靠时，分类器维护了一份详尽的扩展名白名单：

| 类型 | 扩展名 |
|------|--------|
| TEXT | txt, md, json, xml, csv, html, css |
| DOCUMENT | pdf, doc, docx, xls, xlsx, ppt, pptx |
| IMAGE | jpg, jpeg, png, gif, svg, webp |
| AUDIO | wav, ogg, flac, aac, aiff |
| VIDEO | mp4, webm |
| CODE | js, mjs, bat, sh, py, java, c, cpp, h, hpp, cs, go, rb, php, swift, kt, kts, ts, tsx, jsx, vue, yml, yaml, toml, ini, log, sql, jsonc, rs, dart, lua, r, pl, ex, exs, zig, hs, scala, groovy, d, nim, cr |

### 5.3 分类决策树

```
输入: mimeType, fileName
           │
           ▼
    扩展名 === 'mp3'?
           │
      Yes ─┴─► AUDIO
           No
           ▼
    mime === '' 或 'application/octet-stream'?
           │
      Yes ─┴────────────────────────┐
           No                        ▼
           ▼              ┌─────────────────────┐
    mime 前缀匹配          │ 按扩展名白名单分类   │
           │              │ txt/md → TEXT       │
      ┌────┴────┐         │ pdf/docx → DOCUMENT │
      ▼         ▼         │ jpg/png → IMAGE     │
   image/    video/       │ mp4/webm → VIDEO    │
      │        │          │ js/ts/py → CODE     │
   IMAGE     VIDEO        └─────────┬───────────┘
      │                             │

      │

      │

      │

      │
```

## 6. 附件渲染器

### 6.1 AttachmentRenderer.vue（调度器）

渲染调度器是附件子系统的**中央路由**，职责仅为：分类 → 查表 → 挂载。

> 文件位置：`src/features/chat/attachment/AttachmentRenderer.vue`

```vue
<template>
  <component
    :is="currentComponent"
    :file="file"
    :index="index"
    :show-remove="showRemove"
    @remove="emit('remove', index)"
  />
</template>
```

核心逻辑：
```typescript
const attachmentType = computed(() => 
  classifyAttachment(props.file.type, props.file.name)
);

const currentComponent = computed(() => {
  const component = AttachmentRegistry.getComponent(attachmentType.value);
  return component || AttachmentRegistry.getComponent(AttachmentType.OTHER);
});
```

### 6.2 各类型渲染器详解

所有 7 种类型渲染器（不含 `Other`）共享统一的 Props 接口：

```typescript
interface Props {
  file: Attachment;
  index: number;
  size?: 'small' | 'medium' | 'large' | 'auto';
  showRemove?: boolean;
}
```

并通过 `<AttachmentPreviewBase>` 获取统一的视觉外壳（圆角、Hover 效果、加载遮罩、删除按钮）。

#### 6.2.1 ImageAttachment

> 文件位置：`src/features/chat/attachment/types/ImageAttachment.vue`

| 事件名 | 方向 | 触发源 | 载荷 | 前端处理 |
|--------|------|--------|------|----------|
| `vcp-mobile-file-start` | Rust/Kotlin → 前端 | Kotlin `onPickFileResult` | `{ name, size, mime }` | `attachmentStore` 创建 loading 卡片 |

## 10. 设计决策与注意事项

### 10.1 为何同时存在 AttachmentFactory 和 AttachmentRegistry？

`AttachmentFactory`（64 行）是早期实现，功能已冻结；`AttachmentRegistry`（106 行）是后续增强版，新增了 `registerMultiple`、`unregister`、`clear`、`count` 等运行时管理能力。当前渲染调度（`AttachmentRenderer.vue`）使用的是 `AttachmentRegistry`，`AttachmentFactory` 保留为兼容层和潜在的外部调用入口。二者内部的 `componentMap` 相互独立，但注册的组件集合完全一致。

### 10.2 为何视频/音频不内嵌播放器？

移动端消息列表中内嵌 `<video>` 或 `<audio>` 控件会带来以下问题：
1. **布局抖动**：视频加载后改变消息气泡高度，导致列表滚动位置漂移
2. **自动播放限制**：移动端 WebView 普遍禁止自动播放有声视频
3. **全屏冲突**：视频元素自带的全屏按钮与应用的 `AttachmentViewer` 层级体系冲突
4. **性能**：同时存在多个视频元素会消耗大量 GPU 内存

因此采用**文件卡片 + 系统播放器**策略：消息内展示图标+文件名，点击后调用 `open_file` 唤起系统默认应用。

### 10.3 为何文本预览限制 128KB？

`AttachmentViewer.vue` 在预览文本文件时：
1. 使用 `fetch` + `ReadableStream` 流式读取
2. 设置 128KB 硬上限（`const LIMIT = 128 * 1024`）
3. 超过上限时停止读取，展示截断提示条

这是为了防止用户打开一个 100MB 的日志文件导致 WebView OOM。128KB 约等于 3–4 万汉字或 10 万行代码（以每行平均 80 字节计），足以覆盖绝大多数日常文本文件的预览需求。

### 10.4 缩略图策略

| 场景 | 缩略图来源 | 说明 |
|------|-----------|------|
| Android 图片 | Kotlin 端 `generateNativeThumbnail` | 使用 Android 硬件加速解码，生成压缩后的 JPEG 缩略图 |
| 非 Android 图片 | 无 | 直接使用原图 `src`，依赖浏览器自身的解码与缩放 |
| 视频/音频/文档/代码/文本 | 无 | 不生成缩略图，使用 SVG 图标代替 |

`ImageAttachment.vue` 的 `safeSrc` computed 优先级：`thumbnailPath` > `src` > 空字符串。这意味着 Android 端图片始终以缩略图展示，显著降低消息列表内存占用。

### 10.5 孤儿附件清理策略

当用户删除暂存附件或清空暂存区时，前端不会立即物理删除文件，而是调用 `cleanup_single_orphaned_attachment`：

```
前端: invoke("cleanup_single_orphaned_attachment", { hash })
        │
        ▼
Rust: 查询 message_attachments 表
        │
        ├──► hash 被引用? ──► 返回"跳过清理"
        └──► hash 未被引用?
                │
                ├──► 删除 attachments 表记录
                ├──► 删除物理文件
                └──► 删除缩略图文件（若存在）
```

这种设计防止了竞态条件：若用户快速删除暂存附件后消息实际上已被发送，物理文件不应被删除。

### 10.6 高速 TCP 链路的必要性

Tauri v2 的 IPC 通道（`invoke`）在传递大体积二进制数据时存在以下限制：
1. WebView ↔ Rust 之间的序列化/反序列化开销随数据量线性增长
2. 部分平台对单次 IPC 消息大小有隐式上限
3. 大文件无法展示流式进度

高速 TCP 链路（`high_speed_channel.rs`）的解决方案：
- Rust 启动本地 `TcpListener`（`127.0.0.1:0` 随机端口）
- 前端通过 `XMLHttpRequest` 以原生 TCP 流式上传
- `xhr.upload.onprogress` 提供精确的逐帧进度回调
- Rust 端边接收边写入临时文件，无需完整缓冲

阈值设为 2MB，这是经验值：低于此值的文件 IPC 开销可接受，高于此值时 TCP 链路的优势显著。
