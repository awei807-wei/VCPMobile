# 11. 消息块与富媒体渲染

## 1. 概述

### 1.1 领域定位

本模块负责将 Rust 后端解析后的结构化消息内容（`ContentBlock[]`）渲染为可视化的 UI 组件。它是 VCP Mobile 聊天体验的核心呈现层，覆盖 Markdown 文本、代码高亮、数学公式、Mermaid 图表、HTML 预览、思考链、工具调用结果等全部富媒体形态。

区别于传统聊天应用的前端 Markdown 解析方案，VCP Mobile 采用 **Backend-Driven Rendering** 架构：Markdown 解析、AST 生成、代码高亮、哈希指纹计算全部在 Rust 后端完成，前端仅执行 AST → HTML 的轻量转换与 DOM 挂载。这一决策显著降低了前端主线程负担，尤其在移动端 WebView 环境下避免了复杂的排版计算导致的掉帧。

### 1.4 核心设计原则

1. **Backend-Driven Rendering**：消息内容的生命周期由后端驱动。静态消息经 Rust `parse_content` 预编译为 `ContentBlock[]` + `MarkdownNode[]` AST；流式消息经 `StreamBlockParser` 增量沉淀为 `StreamBlock[]`。前端**不做 Markdown 全量解析**，只做 AST → HTML 的轻量转换。

2. **双模管线（Hybrid Pipeline）**：
   - **静态模式**：`useContentProcessor.processMessageContent` → `invoke("process_message_content")` → Rust 全量解析 → 返回 `ContentBlock[]`。
   - **流式模式**：前端 `transformSpecialBlocksForStream` 做轻量正则替换，将特殊块（thought/tool）转为内联 HTML 占位，避免流式期间频繁跨 IPC 调用 Rust。

3. **渲染性能优先**：
   - `astRenderer.ts` 对 AST → HTML 结果做 LRU 缓存（上限 500 条）。
   - `MessageRenderer.vue` 对稳定块使用 `v-memo`，确保已闭合块零开销重渲染。
   - KaTeX / Mermaid / 表情替换等重操作仅在消息**离开活跃流后**统一执行一次。
   - 消息列表项使用 `content-visibility: auto` 延迟离屏消息的渲染与布局。

4. **安全沙箱**：HTML 预览块使用 `DOMPurify` 净化 + `iframe sandbox` 隔离，禁止 `applet`/`embed`/`object` 等危险标签；`<style>` 块通过选择器前缀化限制在单条消息作用域内。

## 2. 消息块类型系统

### 2.1 ContentBlock 接口详解

前端类型定义位于 `src/core/types/chat.ts`（第 41–68 行）：

```ts
export interface ContentBlock {
  type:
    | "markdown"
    | "tool-use"
    | "tool-result"
    | "diary"
    | "thought"
    | "button-click"
    | "html-preview"
    | "role-divider"
    | "style"
    | "math";
  content?: string;              // 原始文本或 HTML 字符串
  nodes?: MarkdownNode[];        // 仅 markdown / diary / thought 携带 AST
  tool_name?: string;            // tool-use / tool-result 的工具名
  is_complete?: boolean;         // thought / tool-use 的闭合状态
  status?: string;               // tool-result 的执行状态文本
  details?: Array<{ key: string; value: string }>; // tool-result 键值详情
  footer?: string;               // tool-result 尾部备注
  maid?: string;                 // diary: 女仆名
  date?: string;                 // diary: 日期
  theme?: string;                // thought: 思考主题
  role?: string;                 // role-divider: 角色名
  is_end?: boolean;              // role-divider: 是否为结束标记
  display_mode?: boolean;        // math: 是否为块级公式
  highlighted_content?: string;  // html-preview: 后端高亮的 HTML 代码
  hash?: string | number;        // 稳定哈希，用于缓存键与 v-memo
}
```

Rust 后端对应定义位于 `src-tauri/src/vcp_modules/chat/content_parser.rs`（第 8–84 行），结构完全一致，通过 `#[serde(tag = "type")]` 实现 tagged enum 序列化。

### 2.2 Block 类型枚举

| 类型 | 来源标记（原始文本） | 关键字段 | 渲染方式 |
|------|----------------------|----------|----------|
| `markdown` | 默认/无标记 | `nodes: MarkdownNode[]` | `astRenderer.renderMarkdownNodes` → `v-html` |
| `tool-use` | `<<<[TOOL_REQUEST]>>>` | `tool_name`, `content`, `is_complete` | `ToolBlock.vue`（type=tool-use） |
| `tool-result` | `[[VCP调用结果信息汇总:` | `tool_name`, `status`, `details[]`, `footer` | `ToolBlock.vue`（type=tool-result） |
| `thought` | `[--- VCP元思考链:主题---]` 或 `<think>` | `theme`, `content`, `nodes[]`, `is_complete` | `ThoughtBlock.vue` |
| `diary` | `<<<DailyNoteStart>>>` | `maid`, `date`, `content`, `nodes[]` | `MessageRenderer` 内联模板 |
| `html-preview` | ````html` 或 `<!DOCTYPE html>` | `content`, `highlighted_content` | `HtmlPreviewBlock.vue` |
| `role-divider` | `<<<[ROLE_DIVIDE_XXX]>>>` | `role`, `is_end` | `MessageRenderer` 内联模板 |
| `style` | `<style>...</style>` | `content`（CSS 文本） | 注入 `<head>` 作为作用域样式 |
| `button-click` | `[[点击按钮:...]]` | `content` | `MessageRenderer` 内联模板 |

### 2.3 与后端 Block 类型的映射

```
前端 ContentBlock        Rust ContentBlock (content_parser.rs)      Rust StreamBlock (stream_block_parser.rs)
─────────────────────────────────────────────────────────────────────────────────────────────────────────────
markdown               ←  ContentBlock::Markdown { nodes, hash }  ←  StreamBlock::Markdown { nodes, hash }
tool-use               ←  ContentBlock::ToolUse { tool_name, ... } ←  StreamBlock::Tool { tool_name, ... }
tool-result            ←  ContentBlock::ToolResult { ... }        ←  StreamBlock::ToolResult { ... }
thought                ←  ContentBlock::Thought { theme, ... }     ←  StreamBlock::Thought { theme, ... }
diary                  ←  ContentBlock::Diary { maid, date, ... } ←  StreamBlock::Diary { ... }
html-preview           ←  ContentBlock::HtmlPreview { ... }       ←  StreamBlock::HtmlPreview { ... }
role-divider           ←  ContentBlock::RoleDivider { ... }       ←  StreamBlock::RoleDivider { ... }
style                  ←  ContentBlock::Style { ... }             ←  StreamBlock::Style { ... }
button-click           ←  ContentBlock::ButtonClick { ... }       ←  StreamBlock::ButtonClick { ... }
```

**关键差异**：
- `ContentBlock` 用于**已持久化**的消息历史，`hash` 为 `Option<u64>`（由 `compute_hashes_recursively` 生成）。
- `StreamBlock` 用于**流式增量推送**，`hash` 为 `String`（由 `HashAggregator::compute_content_hash` 计算）。
- 流式期间未闭合的块不会进入 `stable_blocks`，而是作为 `tail_content` / `tail_block` 进行**推测渲染（Speculative Rendering）**。

## 3. 思考块（ThoughtBlock.vue）

### 3.1 职责定位

`ThoughtBlock.vue` 负责渲染模型的思考过程（Reasoning Chain）。支持两种来源：
- **VCP 元思考链**：`[--- VCP元思考链: 主题 ---]...内容...[--- 元思考链结束 ---]`
- **标准 Think 标签**：`<think>...</think>` 或 `<thinking>...</thinking>`

两者在 Rust 端均被归一化为 `ContentBlock::Thought` / `StreamBlock::Thought`，`theme` 字段分别为自定义主题或默认 `"思维链"`。`is_complete` 字段标记该思考链是否已闭合，用于流式期间的视觉反馈。

### 3.2 折叠/展开交互

```vue
<!-- src/features/chat/blocks/ThoughtBlock.vue -->
<div class="vcp-thought-header" @click="toggleExpand">
  <span class="vcp-thought-icon">🧠</span>
  <span class="vcp-thought-label">
    {{ block.theme || "元思考链" }}
    <Loader2 v-if="!block.is_complete" :size="10" class="animate-spin" />
  </span>
  <component :is="isExpanded ? ChevronUp : ChevronDown" />
</div>

<div v-show="isExpanded" class="vcp-thought-content animate-slide-down">
  <div class="thought-body"
    v-html="block.nodes?.length > 0
      ? renderMarkdownNodes(block.nodes, messageId)
      : escapeHtml(block.content || '')" />
</div>
```

- **默认折叠**：思考块默认收起，仅展示头部标签，避免占用过多垂直空间。用户点击 Header 切换展开状态。
- **未完成状态指示**：`!block.is_complete` 时显示旋转的 `Loader2` 图标，提示思考仍在进行。
- **内容渲染优先级**：优先使用后端预渲染的 `nodes`（AST）以支持 Markdown 格式（加粗、列表、代码等）；若 `nodes` 为空则回退到 HTML 转义后的纯文本，防止 XSS。

### 3.3 样式与动画

- **暗色适配**：通过 `html.dark .vcp-thought-block` 切换为低对比度背景，符合 UI 美学宪法「灰度优先」原则。
- **动画曲线**：`cubic-bezier(0.4, 0, 0.2, 1)` 为 Material Design 标准减速曲线，展开时自然平滑。
- **视觉层级**：`thought-body` 使用 `font-style: italic` + `opacity: 0.8`，与主内容形成明显区分。

### 3.4 流式期间的思考块渲染

在流式模式下，思考块不会立即以独立 `ThoughtBlock` 组件渲染。前端 `useContentProcessor.transformSpecialBlocksForStream` 将其替换为内联 HTML 占位（`useContentProcessor.ts` 第 91–111 行）：

```ts
// 流式正则替换
processed = processed.replace(
  /\[--- VCP元思考链(?::\s*([^\]]*?))?\s*---\]([\s\S]*?)\[--- 元思考链结束 ---\]/gs,
  (_, theme, content) => {
    const displayTheme = theme ? theme.trim().replace(/"/g, "") : "元思考链";
    return `\n<div class="my-2 p-3 bg-black/5 dark:bg-white/5 rounded-xl ...">
      <div class="flex items-center gap-2 mb-2 opacity-70 font-bold">
        <span class="grayscale">🧠</span> <span>${displayTheme} (已完成)</span>
      </div>
      <div class="italic opacity-80">${content}</div>
    </div>\n`;
  }
);
```

**原因**：流式期间消息尚未持久化，没有完整的 `ContentBlock[]` 结构。待流结束后，Rust `stream_block_parser.rs` 的 `finalize` 将全文重新解析为正规 `ContentBlock[]`，此时思考块才以真正的 `ThoughtBlock.vue` 组件形态渲染。

## 4. 工具块（ToolBlock.vue）

### 4.1 职责定位

`ToolBlock.vue` 是工具调用生态的 UI 呈现层，区分两种形态：
- **`tool-use`**：模型发起工具调用的请求，展示工具名称与参数。
- **`tool-result`**：工具执行后的结构化结果，展示状态、键值详情与尾部备注。

### 4.2 工具调用结果展示

工具结果的解析由 Rust `content_parser.rs` / `stream_block_parser.rs` 中的 `parse_tool_result` 完成（第 713–772 行）。其语法为：

```
[[VCP调用结果信息汇总:
- 工具名称: xxx
- 执行状态: success
- 可访问URL: https://...
- 其他键: 其他值
（支持多行值，后续非 KV 格式的行会被追加到当前值）
尾部备注文本（非 KV 格式的行）
VCP调用结果结束]]
```

解析后映射为 `details: Array<{ key, value }>` + `footer: string`。`ToolBlock.vue` 的渲染逻辑（第 86–104 行）：

```vue
<div v-for="item in block.details" :key="item.key" class="text-xs flex flex-col sm:flex-row">
  <span class="detail-key font-bold mr-2 whitespace-nowrap">{{ item.key }}:</span>
  <div class="flex-1 min-w-0">
    <!-- 图片值直接渲染为 img，其他值走纯文本 -->
    <template v-if="item.value && isImageValue(item.key, item.value)">
      <a :href="item.value" target="_blank" rel="noopener noreferrer">
        <img :src="item.value" class="max-w-full rounded-lg" loading="lazy" />
      </a>
    </template>
    <div v-else class="whitespace-pre-wrap">{{ item.value || '' }}</div>
  </div>
</div>
<div v-if="block.footer" class="mt-2 pt-2 border-t ... whitespace-pre-wrap">
  {{ block.footer }}
</div>
```

**图片自动检测**：`isImageValue` 函数检测键名（`可访问URL` / `返回内容` / `url` / `image`）与值格式（HTTP URL 结尾为图片扩展名，或 base64 data URI），自动渲染为 `<img>` 标签并支持点击放大。

### 4.3 展开/折叠与复制

与 `ThoughtBlock` 相同，`ToolBlock` 默认折叠，点击 Header 展开。额外特性：
- **`tool-use` 未完成态**：`!block.is_complete` 时显示旋转 `Loader2`，Header 使用蓝青渐变流动背景动画（`vcp-bubble-background-flow-kf`，20s 循环）。
- **流动边框**：`tool-use` 使用 `::after` 伪元素 + `mask-composite: exclude` 实现 2px 渐变流动边框，7s 线性循环。
- **IntersectionObserver 性能优化**：当工具块滚出视口时，通过 `vcp-animation-paused` 类暂停无限 CSS 动画，节省 GPU 资源。

### 4.4 与 ToolInteractionOverlay 的关系

`ToolBlock.vue` 与 `src/features/distributed/ToolInteractionOverlay.vue` **当前无直接耦合**。

- `ToolBlock` 负责**展示**工具的文本化请求与结果，属于**被动信息渲染**。
- `ToolInteractionOverlay` 是 Phase 3 骨架，负责**交互式工具**（如相机、生物识别）的模态覆盖层，通过监听 `tool-ui-request` Tauri 事件触发。属于**主动用户交互**。

两者的分界线：若工具需要用户实时操作（拍照、确认），走 `ToolInteractionOverlay`；若工具只是返回结构化数据（查询天气、生成图片 URL），走 `ToolBlock`。

## 5. HTML 预览块（HtmlPreviewBlock.vue）

### 5.1 职责定位

`HtmlPreviewBlock.vue` 用于渲染模型生成的 HTML 代码。它不仅是一个代码展示器，还提供了**实时预览**能力。触发条件（Rust `content_parser.rs` 第 256–263 行）：
- Markdown 代码围栏 ````html`
- 独立的 HTML 文档（`<!DOCTYPE html>` 或 `<html>` 开头）
- 块级 HTML 容器（`<div>`/`<section>`/`<article>` 等）

Rust `content_parser.rs` 将上述内容提取为 `ContentBlock::HtmlPreview`，并预先用 `syntect` 对 HTML 代码进行语法高亮，存入 `highlighted_content` 字段。前端优先展示后端高亮结果，若为空则回退到客户端 HTML 转义。

### 5.2 iframe / WebView 沙箱策略

HTML 预览采用**双层渲染策略**：

**第一层：DOMPurify 净化**
```ts
const cleanHtml = DOMPurify.sanitize(content, {
  USE_PROFILES: { html: true, svg: true, mathMl: true },
  ADD_TAGS: ['style', 'iframe', 'canvas', 'script', 'link', 'meta'],
  ADD_ATTR: ['*'],
  FORBID_TAGS: ['applet', 'embed', 'object'],
  ALLOW_UNKNOWN_PROTOCOLS: true,
  WHOLE_DOCUMENT: true,
  RETURN_DOM: false
});
```

- 允许 `script`/`style`/`iframe`/`canvas` 以支持交互式 HTML Demo。
- 显式禁止 `applet`/`embed`/`object` 等历史高危标签。
- `ALLOW_UNKNOWN_PROTOCOLS: true` 允许 `data:` URI 等自定义协议（用于 base64 图片）。

**第二层：iframe sandbox**
```vue
<iframe
  class="vcp-inline-iframe w-full h-full border-none"
  sandbox="allow-scripts allow-modals allow-forms allow-popups"
  loading="lazy"
  :srcdoc="getSandboxHtml(content)"
/>
```

- `allow-scripts`：允许内部 JS 运行（用于交互式 Demo）。
- **未包含** `allow-same-origin` 和 `allow-top-navigation`：防止 iframe 内脚本操作父级 DOM 或跳转顶层页面。
- 注入的 VCP 样式覆盖滚动条、背景透明、响应式图片尺寸，并拦截 `javascript:` 伪协议链接点击。

### 5.3 全屏预览模式（z-editor 层级）

```vue
<Teleport to="body">
  <Transition
    enter-active-class="transition duration-300 ease-out"
    enter-from-class="translate-y-10 opacity-0"
    enter-to-class="translate-y-0 opacity-100">
    <div v-show="isFullScreen" class="fixed inset-0 z-editor flex flex-col">
      <!-- 全屏 Header：返回按钮 + 刷新/复制 + Code/Preview Tab -->
      <!-- 全屏内容区：代码模式（pre+code）/ 预览模式（iframe 沙箱） -->
    </div>
  </Transition>
</Teleport>
```

- **Teleport to body**：避免被父级 `overflow: hidden` 裁剪，确保全屏覆盖整个视口。
- **z-editor 层级**：根据 `docs/UI_LAYER_ARCHITECTURE.md` 的语义化层级体系，`z-editor`（`--layer-editor: 80`）高于 `z-viewer`（70），低于 `z-toast`（90），确保全屏编辑器不会被 Toast 遮挡，但也不会覆盖 BootScreen/PermissionGate。
- **双模式 Tab**：全屏模式下支持「代码」与「预览」切换，状态与普通视图同步（`watch(isPreviewing)` + `watch(fullScreenTab)` 双向绑定）。
- **刷新机制**：`refreshPreview` 通过清空 `iframe.srcdoc` 再延时重置，强制重新加载 iframe 内容。

### 5.4 安全策略（DOMPurify、CSP）

| 层级 | 机制 | 实现 |
|------|------|------|
| 输入净化 | DOMPurify | 过滤危险标签与属性，保留结构化 HTML |
| 执行隔离 | iframe sandbox | 限制脚本能力，禁止同源访问与顶层导航 |
| 链接拦截 | JS 注入 | 拦截 `href="javascript:..."`、空链接与 `#` 点击 |

**与 AttachmentViewer 的区别**：

| 特性 | HtmlPreviewBlock | AttachmentViewer |
|------|------------------|------------------|

## 6. 代码块渲染

### 6.1 代码高亮实现

VCP Mobile 的代码高亮**完全由 Rust 后端在预渲染阶段完成**，前端不引入 `prismjs` 或 `highlight.js` 等前端高亮库。

**后端实现**（`src-tauri/src/vcp_modules/chat/pre_renderer/code_highlighter.rs`）：

```rust
use syntect::highlighting::ThemeSet;
use syntect::html::highlighted_html_for_string;
use syntect::parsing::SyntaxSet;

lazy_static! {
    static ref SYNTAX_SET: SyntaxSet = SyntaxSet::load_defaults_newlines();
    static ref THEME_SET: ThemeSet = ThemeSet::load_defaults();
}

pub fn highlight_code_block(code: &str, lang: &str) -> Option<String> {
    let lang_lower = lang.to_lowercase();
    let syntax = SYNTAX_SET
        .find_syntax_by_token(&lang_lower)
        .or_else(|| SYNTAX_SET.find_syntax_by_token(lang))
        .or_else(|| SYNTAX_SET.find_syntax_by_extension(lang))
        .unwrap_or_else(|| {
            // 语言未知时回退到 JavaScript（通用类 C 语法高亮）
            SYNTAX_SET.find_syntax_by_token("JavaScript")
                .unwrap_or_else(|| SYNTAX_SET.find_syntax_plain_text())
        });

    let theme = &THEME_SET.themes["base16-ocean.dark"];
    let html = highlighted_html_for_string(code, &SYNTAX_SET, syntax, theme).ok()?;

    // 统一添加 vcp-code-block 和 vcp-scrollable 类
    let fixed = if html.starts_with("<pre") {
        html.replacen("<pre", "<pre class=\"vcp-code-block vcp-scrollable\"", 1)
    } else {
        format!("<pre class=\"vcp-code-block vcp-scrollable\">{}</pre>", html)
    };
    Some(fixed)
}
```

- **库**：`syntect`（Rust 版 TextMate 语法高亮引擎，与 VS Code 使用相同的 `.sublime-syntax` 定义）。
- **主题**：固定使用 `base16-ocean.dark`，与 `HtmlPreviewBlock.vue` 的暗色 GitHub 风格配色一致。
- **回退策略**：语言未知时回退到 JavaScript，再未知时回退到纯文本。避免无高亮的糟糕体验。

**前端消费**（`astRenderer.ts` 第 62–75 行）：

```ts
case 'code_block': {
  let html = node.highlighted_html;
  if (html) {
    // 兼容旧 AST：如果 highlighted_html 是 <pre><code> 包裹内层 <pre> 的嵌套结构，提取单层
    const nestedPreMatch = html.match(/<pre[^>]*>\s*<code>([\s\S]*?)<\/code>\s*<\/pre>/i);
    if (nestedPreMatch && nestedPreMatch[1].trim().startsWith('<pre')) {
      const innerMatch = nestedPreMatch[1].match(/<pre[^>]*>([\s\S]*?)<\/pre>/i);
      if (innerMatch) {
        html = `<pre class="vcp-code-block vcp-scrollable">${innerMatch[1]}</pre>`;
      }
    }
    return html;
  }
  // 无高亮时回退到纯文本转义
  return `<pre class="vcp-code-block vcp-scrollable"><code>${escapeHtml(node.code || '')}</code></pre>`;
}
```

### 6.2 语言检测与标识

语言标识来自 Markdown 代码围栏的 info string：

## 7. Markdown / 文本块渲染

### 7.1 Markdown 解析策略

VCP Mobile 采用**前后端分离的 Markdown 解析架构**：

| 阶段 | 执行者 | 输出 |
|------|--------|------|
| Markdown 解析 | Rust `pre_renderer::parse_markdown_to_ast` | `MarkdownNode[]` + `InlineNode[]` |
| AST → HTML | 前端 `astRenderer.ts` | HTML 字符串 |
| 重内容渲染 | 前端 `MessageRenderer.renderHeavyContent` | KaTeX / Mermaid / 表情 DOM 操作 |

**为何不纯前端解析？**
1. **性能**：Rust 的 `pulldown-cmark` 解析速度远高于前端 JS 库，且后端可利用多核/异步优势。
2. **一致性**：前后端使用同一套 AST 定义（`markdown_ast.rs`），避免渲染差异导致的历史消息重排。
3. **预计算**：代码高亮、哈希指纹在 Rust 端一次完成，前端只做轻量拼接，无需加载庞大的前端高亮库。

**AST 节点类型**（`src-tauri/src/vcp_modules/chat/pre_renderer/markdown_ast.rs`）：

**块级节点（MarkdownNode）**：
- `paragraph`：段落
- `heading`：标题（level 1–6）
- `code_block`：代码块（含 `lang`, `code`, `highlighted_html`）
- `blockquote`：引用块
- `list`：有序/无序列表
- `table`：表格（含 `header`, `rows`, `wrapper_class`）
- `thematic_break`：水平分割线
- `raw_html`：原始 HTML
- `mermaid`：Mermaid 图表占位

**行内节点（InlineNode）**：
- `text`：纯文本
- `strong` / `emphasis` / `strikethrough`：粗体/斜体/删除线
- `code`：行内代码
- `link` / `image`：链接/图片（支持 `needs_asset_conversion`）
- `inline_math`：行内/块级数学公式
- `quoted_text` / `highlight_tag` / `alert_tag`：VCP 魔法标记
- `raw_html_inline`：原始行内 HTML

### 7.2 KaTeX 公式渲染

**AST 表示**：Rust 解析器识别 `$...$`（行内）和 `$$...$$`（块级）为 `InlineNode::InlineMath`，序列化为：

```ts
// astRenderer.ts 第 147–151 行
case 'inline_math': {
  const isDisplay = node.display_mode || false;
  const cls = isDisplay ? 'vcp-math-block no-swipe' : 'vcp-math-inline no-swipe';
  return `<span class="${cls}" data-latex="${escapeHtml(node.content || '')}">
    ${escapeHtml(node.content || '')}
  </span>`;
}
```

**动态渲染时机**：`MessageRenderer.vue` 的 `renderHeavyContent` 函数（第 166–245 行）：

```ts
const renderHeavyContent = async () => {
  if (isMessageInActiveStream.value) return; // 流式期间完全跳过
  await nextTick();
  const mathElements = messageContentRef.value.querySelectorAll(
    '.vcp-math-inline[data-latex], .vcp-math-block[data-latex]'
  );
  const katex = (await import('katex')).default;
  mathElements.forEach((el) => {
    if (el.querySelector('.katex')) return; // 已渲染则跳过
    const latex = el.getAttribute('data-latex');
    const isDisplay = el.classList.contains('vcp-math-block');
    katex.render(latex, el as HTMLElement, {
      throwOnError: false,
      strict: false,
      displayMode: isDisplay,
    });
  });
};
```

- **按需加载**：`import('katex')` 动态导入，减少首包体积。
- **流式保护**：消息处于活跃流中时**完全不执行** KaTeX 渲染，避免流式期间每帧触发昂贵的排版计算。
- **样式**：`main.ts` 全局引入 `katex/dist/katex.min.css`；`message-blocks.css` 提供横向滚动条美化与溢出保护。

### 7.3 Mermaid 图表渲染

Mermaid 图表在 AST 中被表示为 `MarkdownNode::MermaidPlaceholder`，前端渲染逻辑同样在 `renderHeavyContent` 中：

```ts
const mermaidPlaceholders = messageContentRef.value.querySelectorAll('.mermaid-placeholder');
const mermaid = (await import('mermaid')).default;
if (!mermaidInitialized) {
  mermaid.initialize({ startOnLoad: false, theme: 'dark' });
  mermaidInitialized = true;
}
for (const el of Array.from(mermaidPlaceholders)) {
  if (placeholder.querySelector('svg')) continue; // 已渲染则跳过
  const codeKey = placeholder.innerHTML;
  if (mermaidCache.has(codeKey)) {
    placeholder.innerHTML = mermaidCache.get(codeKey)!;
    continue;
  }
  renderingMermaids.add(codeKey);
  await mermaid.run({ nodes: [placeholder] });
  mermaidCache.set(codeKey, placeholder.innerHTML);
  renderingMermaids.delete(codeKey);
}
```

- **缓存机制**：以图表源码的 HTML 转义文本为键，`Map<string, string>` 缓存已渲染的 SVG，避免重复解析。
- **并发保护**：`renderingMermaids` Set 防止同一图表被并发渲染多次。
- **暗色主题**：固定初始化为 `theme: 'dark'`，与应用暗色模式一致。

### 7.4 链接与表情处理

**链接**（`astRenderer.ts` 第 127–132 行）：

```ts

  const href = node.needs_asset_conversion && node.href
    ? convertFileSrc(node.href)   // Tauri asset:// 协议转换

- `needs_asset_conversion`：Rust 端标记本地文件路径（如图片附件），前端通过 `convertFileSrc` 转换为 WebView 安全的 `asset://` URL。

**表情（Emoticon）**：`MessageRenderer.vue` 的 `renderHeavyContent` 最后调用 `processEmoticonsInContainer`，将特定文本标记替换为 `<img>` 标签（由 `useEmoticonFixer` composable 管理）。表情图片最大尺寸 110×110px，带悬停放大效果。

## 10. 设计决策与注意事项

### 10.1 为什么用 Block 而非纯文本渲染？

在 v0.9.x 之前的版本中，前端使用正则表达式在 `v-html` 中直接替换特殊标记。该方案存在以下问题：
1. **XSS 风险**：正则替换后的 HTML 难以彻底净化，特殊标记内可能包含恶意脚本。
2. **渲染不一致**：前后端 Markdown 解析器不同，导致同一条消息在不同端显示差异。
3. **流式闪烁**：纯文本增量追加时，未闭合的 Markdown 标记会导致整段内容反复重排。
4. **性能劣化**：长消息的正则替换是 O(n) 操作，每帧执行导致主线程阻塞。

**Block 架构的优势**：
- 每个 Block 是独立的渲染单元，已闭合块通过 `v-memo` 彻底冻结，不受后续追加影响。
- Rust 端统一解析，前端只做 AST → HTML 的确定性转换，零解析歧义。
- 特种块（tool/thought/html）获得独立组件级的交互与样式能力，不再受限于内联 HTML。

### 10.2 HtmlPreviewBlock 的 iframe 策略

**为何不直接用 `v-html` 渲染 HTML？**
- 模型生成的 HTML 可能包含 `<script>`，直接注入 DOM 会获得与主应用同源的 JS 执行权限，存在严重安全风险。
- iframe 的 `sandbox` 属性提供浏览器原生的执行隔离，即使允许 `allow-scripts`，脚本也无法访问 `parent.document` 或操作顶层路由。

**为何允许 `allow-scripts`？**
- VCP Mobile 的使用场景中，模型常生成交互式图表（ECharts）、小游戏、Canvas 动画或动态 Demo。完全禁止脚本会严重削弱产品价值。
- `allow-scripts` + DOMPurify + 无 `allow-same-origin` 的组合提供了功能与安全的平衡：脚本可运行，但无法逃逸沙箱。

### 10.3 代码高亮的性能优化

**为何选择 syntect 后端高亮而非前端库？**
- `syntect` 使用 TextMate 语法定义，覆盖 100+ 语言，质量与 VS Code 一致。
- 在 Rust 端完成高亮后，前端接收的是纯 HTML 字符串，无需加载前端高亮库、无需运行时解析语法定义。
- 对于长代码块（如数百行），后端高亮避免了前端主线程阻塞，保持 60fps 滚动体验。
- `HtmlPreviewBlock.vue` 同样受益于后端高亮：Rust 在生成 `ContentBlock::HtmlPreview` 时即调用 `highlight_code_block(&content, "html")`。

### 10.4 流式期间跳过重渲染的铁律

`MessageRenderer.vue` 中有多处 `if (isMessageInActiveStream.value) return;` 保护：

1. **KaTeX 渲染**：公式排版涉及大量 DOM 测量与字体计算，流式期间每帧触发会导致严重掉帧。
2. **Mermaid 渲染**：图表渲染是异步 SVG 生成，流式期间触发会造成 DOM 结构竞争与布局抖动。
3. **表情替换**：涉及 DOM 遍历与节点替换，与 Vue 的虚拟 DOM 更新可能冲突。

**正确性保证**：消息离开活跃流后，`watch(isMessageInActiveStream)` 确保统一执行一次 `renderHeavyContent`，补全所有重内容渲染。这一设计使得流式期间的消息更新路径极轻量：仅 Vue 响应式 Patch AST → HTML，无第三方库介入。

### 10.5 CSS 作用域隔离

`style` 类型的 Block 允许模型注入自定义 CSS。为防止样式污染全局，前端通过 `injectScopedCss`（`useContentProcessor.ts` 第 44–78 行）将 CSS 选择器强制前缀化为 `[data-message-id="xxx"] `：

```css
/* 原始 CSS */
h1 { color: red; }
body { background: black; }

/* 注入后 */
[data-message-id="msg_123"] h1 { color: red; }
[data-message-id="msg_123"] { background: black; }  /* body 被降级 */
```

- `html`/`body`/:root 选择器被强制降级为 `[data-message-id]` 本身，杜绝全局污染。
- 动画关键帧（`@keyframes` / `from` / `to` / 百分比）直接通过，不做修改。
- 伪类选择器（`:hover` / `::before` 等）正确附加在作用域选择器后。
- 消息卸载时调用 `removeScopedCss` 清理对应的 `<style>` 标签，避免内存泄漏。

### 10.6 AST HTML 缓存策略

`astRenderer.ts` 实现了两级缓存机制：

```ts
const htmlCache = new Map<string, string>();
const MAX_CACHE_SIZE = 500;

function getCacheKey(nodes: MarkdownNode[], messageId: string, blockHash?: string | number): string {
  if (blockHash !== undefined && blockHash !== null) {
    return `${messageId}:${String(blockHash)}`;
  }
  return `${messageId}:len-${nodes.length}`; // Fallback
}
```

- **缓存键**：优先使用后端计算的 `blockHash`（SHA-256 或 DefaultHasher 结果），确保内容不变时绝对命中；无 hash 时回退到节点长度。
- **LRU 保护**：超限时直接 `clear()` 而非逐条淘汰。实际场景中，缓存命中模式是「批量命中 / 批量失效」（如切换话题时全部失效），全清比逐条淘汰更高效。
- **单消息清理**：`clearMessageCache(messageId)` 用于编辑消息后强制重渲染，无需清空全局缓存。

### 10.7 消息块的 v-memo 优化

`MessageRenderer.vue` 中每个 Block 包裹在 `v-memo` 中：

```vue
<div v-memo="[getBlockKey(block, index)]">
  <ToolBlock v-else-if="block.type === 'tool-use' || block.type === 'tool-result'" ... />
  <ThoughtBlock v-else-if="block.type === 'thought'" ... />
  <!-- ... -->
</div>
```

- `v-memo` 的依赖是 `getBlockKey`，其基于 `block.hash`（稳定）或 `type-idx-${index}`（回退）。
- 已闭合的 Block 其 `hash` 不变，Vue 将完全跳过该子树的虚拟 DOM Diff 与 DOM 操作，实现**零开销稳定块**。
- 只有在 Block 新增（流式期间 `stable_blocks` 增长）或 `hash` 变化（编辑后重解析）时，对应子树才会重新渲染。

## 11. 术语速查表

| 术语 | 定义 |
|------|------|
| **ContentBlock** | 后端预解析的消息内容块，前端直接渲染的结构化单元。 |
| **StreamBlock** | 流式增量解析的轻量块，与 ContentBlock 同构但 hash 为字符串类型。 |
| **MarkdownNode** | 块级 AST 节点（paragraph / heading / code_block / table / list / mermaid 等）。 |
| **InlineNode** | 行内 AST 节点（text / strong / link / image / inline_math / highlight_tag 等）。 |
| **Aurora Pipeline** | Rust 流式语义沉淀管道，负责将 SSE chunk 解析为 stable + tail 块并推送到前端。 |
| **Speculative Rendering** | 流式期间对未闭合尾部（tail）的推测性 Markdown 预渲染，提升视觉连贯性。 |
| **syntect** | Rust 代码高亮库，基于 TextMate 语法定义，与 VS Code 高亮质量一致。 |
| **DOMPurify** | 前端 HTML 净化库，用于 HTML 预览块的输入消毒。 |
| **v-memo** | Vue 3 编译器指令，缓存 VNode 子树，在依赖未变时跳过重渲染与 Diff。 |
| **content-visibility** | CSS 属性（`auto`），用于 MessageRenderer 的列表项，延迟离屏消息的渲染与布局。 |
| **LAYER_EDITOR** | UI 层级常量（z-index: 80），HtmlPreviewBlock 全屏预览使用的语义化层级。 |
| **needs_asset_conversion** | AST 节点标记，指示链接/图片是否需要通过 `convertFileSrc` 转为 asset 协议 URL。 |
| **Backend-Driven Rendering** | 后端驱动渲染架构：解析、高亮、哈希全部在 Rust 完成，前端仅做轻量转换。 |
| **Hybrid Pipeline** | 双模管线：静态模式走 Rust 全量解析，流式模式走前端正则替换。 |
| **VCP 魔法标记** | 项目特有的文本标记语法（如 `#标签`、`!告警`、引用高亮），在 AST 中以特殊 InlineNode 表示。 |
