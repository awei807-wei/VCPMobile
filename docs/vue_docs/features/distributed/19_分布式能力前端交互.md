# 19. 分布式能力前端交互

## 1. 概述

### 1.1 领域定位

`src/features/distributed/` 是 VCP Mobile Vue 前端中负责**分布式节点 UI 层**的领域目录。它将 Rust 后端 `src-tauri/src/distributed/` 暴露的 13 个设备能力工具、WebSocket 连接状态与传感器数据流，转化为用户可感知、可配置、可交互的界面元素。

该目录遵循**功能共置（Feature Co-location）**原则，组件、组合式函数与类型定义集中在同一目录下：

```
src/features/distributed/
├── DistributedView.vue               # 分布式功能面板（814 行，三标签页 UI）
├── DistributedSettingsSection.vue    # 设置页入口：开关、状态、连接控制
├── ToolInteractionOverlay.vue        # 交互式工具覆盖层（Phase 3 骨架）
└── composables/
    └── useDistributed.ts             # 分布式状态组合式函数
```

### 1.2 模块构成表

| 文件 | 类型 | 职责 | 是否有 UI |
|------|------|------|-----------|
| `DistributedView.vue` | 组件 | 分布式功能面板：连接管理、插件列表（启用/禁用+遥测）、占位符配置（三标签页） | 是（SlidePage 全屏滑出面板） |
| `DistributedSettingsSection.vue` | 组件 | 分布式总开关、节点名称编辑、连接状态展示、手动重连 | 是 |
| `ToolInteractionOverlay.vue` | 组件 | 监听后端事件，展示交互式工具覆盖层；处理通知与剪贴板事件 | 是（条件渲染） |
| `useDistributed.ts` | 组合式函数 | 封装 Tauri 调用与事件监听，提供响应式连接状态 | 否 |

### 1.4 核心设计原则

1. **Android 原生采集模式**：传感器数据不再通过前端 Vue 组件采集，而是由 Android 原生 `SensorStatusManager.kt` 通过 `LocationManager`/`SensorManager` 直接采集，经 Plugin IPC 通道（`plugin:vcp-mobile|get_sensor_data`）传递给 Rust 工具层。前端 `DistributedView.vue` 通过 IPC 拉取遥测数据用于展示。
2. **状态单一来源**：`useDistributed.ts` 中的 `status` ref 是前端唯一的分布式连接状态源。所有组件通过调用该组合式函数读取状态，而非各自维护副本。
3. **事件驱动而非轮询**：前端通过 Tauri `listen()` 被动接收 Rust 推送的状态更新，仅在挂载时执行一次 `get_distributed_status` 查询初始状态。
4. **设置即连接**：`DistributedSettingsSection` 中的开关不仅持久化设置，还直接触发 WebSocket 连接的生命周期（`start()` / `stop()`），避免"保存设置后再手动连接"的两步操作。
5. **Phase 渐进式实现**：ToolInteractionOverlay 当前为 Phase 3 骨架，预留了相机、生物识别等交互式工具的 UI 插槽，现有事件通道已就绪。

### 2.2 分布式功能开关

开关使用 `SettingsSwitch` 组件，绑定逻辑为：

- **显示状态**：`status.connected`（来自 Rust 的实时状态），而非 `settings.distributedEnabled`。这保证了即使自动重连成功，开关也会即时反映真实连接状态。
- **禁用条件**：`loading` 为 true（连接中），或 `derivedWsUrl` 与 `derivedVcpKey` 均未配置且当前未连接。
- **切换动作**：`toggleConnection()` —— 若已连接则调用 `stop()` 并禁用；若未连接则校验 URL/Key 后调用 `start()`。

```typescript
// 连接信息派生逻辑（非用户直接输入）
const derivedWsUrl = computed(() => {
  const logUrl = props.settings.vcpLogUrl || "";
  return logUrl.replace(/\/+$/, "");   // 复用 VCPLog 的 WS 地址
});
const derivedVcpKey = computed(() => props.settings.vcpLogKey || "");
```

> 设计意图：分布式节点复用已有的 VCPLog 服务器地址与密钥，避免用户重复配置。若未设置核心连接，开关自动禁用，并提示"请先设置核心连接 → VCPLog URL"。

### 2.3 节点状态展示

状态展示通过 `statusDisplay` 计算属性实现四级状态机：

| 状态 | `type` | 展示文案 | 触发条件 |
|------|--------|----------|----------|
| 连接中 | `loading` | `连接中...` | `loading.value === true` |
| 已连接 | `success` | `已连接 · {server_id} · {registered_tools} 个工具` | `status.connected === true` |
| 错误 | `error` | `{last_error}` | `last_error` 非空 |
| 未连接 | `null` | `未连接` | 默认态 |

`SettingsInlineStatus` 根据 `type` 渲染不同颜色：success（绿色）、error（红色）、loading（灰色脉冲）。

**自动连接机制**：组件通过 `watch(immediate: true)` 监听 `settings.distributedEnabled`。若持久化设置中开关为开启状态，且当前未连接、URL/Key 有效，则自动调用 `start()` 恢复连接。这使得应用重启后无需手动重新开启节点。

## 3. 分布式功能面板（DistributedView.vue）

> 源码：`src/features/distributed/DistributedView.vue`（814 行）

### 3.1 职责定位

`DistributedView` 是分布式功能的**全功能 UI 面板**，取代了原来的 Headless `SensorCollector`。它提供三标签页布局（基础连接、插件列表、占位符），通过 `SlidePage` 全屏滑出面板挂载于应用右侧。

**核心职责**：

- **连接管理**：展示 WebSocket 连接状态、URL、Key，提供连接/断开按钮
- **插件发现与管理**：从 Rust 后端拉取所有已注册工具元数据，展示 13 个插件的启用/禁用状态，支持展开查看实时遥测数据
- **占位符配置**：列出所有占位符宏及其示例值，支持搜索筛选
- **状态同步**：通过 `useDistributed()` 组合式函数获取实时连接状态，通过 `invoke("get_registered_tools_metadata")` 拉取工具清单，通过 `invoke("update_disabled_tools")` 同步插件启用状态

### 3.3 插件列表交互

**插件元数据拉取**：

```typescript
const tools = await invoke('get_registered_tools_metadata');
// 返回: { name: string, description: string, tool_type: string, enabled: boolean }[]
```

- 首次加载时从 Rust 端 `ToolRegistry` 获取全部 13 个工具的元数据
- 支持搜索过滤（按名称或描述匹配）

**启用/禁用同步**：

```typescript
await invoke('update_disabled_tools', { disabledTools: ['MobileLocation', 'MobileMotion'] });
```

- 切换插件开关时立即同步到 Rust 端
- Rust 端维护 `DisabledTools` 集合，被禁用的工具在 `get_all_manifests()` 时自动排除

**展开实时遥测**：

展开插件卡片后，`DistributedView` 通过 Plugin IPC 命令（`plugin:vcp-mobile|get_sensor_data`）拉取实时数据：
- Streaming 工具（电池/CPU/GPU/内存/网络/存储）→ 通过 IPC 获取最新 sysfs 读数
- 传感器工具（位置/运动/环境光）→ 从 Android 原生 `SensorStatusManager.kt` 获取
- 若 IPC 数据不可用，回退到 Rust 端 `read_current()` 的快照值
- 对于非 Streaming 工具（如 `MobileDeviceInfo`、`MobileClipboard`），展示静态描述而非实时数据

#### 3.3.4 Root 检测 UI

`DistributedView` 的"插件列表"标签页中新增了 Root 访问状态检测 UI：

- **检测入口**：`invoke('plugin:vcp-mobile|check_root_access')` 调用 Rust 端 `check_root_access` 命令，底层通过 libsu 的 `Shell.getShell().isRoot` 检测
- **UI 展示**：在插件列表顶部显示 Root 状态指示器
  - 已 Root：绿色标签 `ROOT ✓` + 展开显示 "可执行 root 命令"
  - 未 Root：灰色标签 `ROOT ✗` + "设备未 Root，sysfs 读取仍可用"
- **Root 管理器启动**：提供"打开 Root 管理器"按钮，调用 `launch_root_manager` 命令，自动检测并启动 Magisk / KernelSU / APatch
- **Root 命令测试**：提供命令行输入框，可手动执行 `run_root_command` 测试（如 `cat /sys/class/thermal/thermal_zone0/temp`），结果实时展示

### 3.4 占位符展示

**占位符列表**：

列出全部 Streaming 工具的占位符宏，每个条目包含：
- 占位符名称（如 `{{MobileBattery}}`）
- 当前值或示例值（如 `"电量: 85%, 充电中, 温度: 32°C"`）
- 工具描述

**搜索与筛选**：支持按占位符名称或工具描述搜索，实时过滤列表。

### 3.5 SlidePage 挂载机制

`DistributedView` 使用 `SlidePage` 组件（全屏滑出面板）挂载于应用右侧，通过 overlay store 控制显示/隐藏：

```typescript
const overlay = useOverlayStore();
overlay.openSlidePage('distributed-panel');
```

- **层级**：`z-page`（高于 drawer 但低于 viewer）
- **动画**：从右侧滑入，带半透明背景遮罩
- **关闭方式**：点击遮罩或按返回键（Android 手势）

### 3.6 与 useDistributed 的联动

```typescript
const { status, loading, start, stop } = useDistributed();

// 连接状态变化时刷新插件列表
watch(() => status.value.connected, async (connected) => {
  if (connected) {
    await refreshTools();       // 重新拉取工具元数据
    await refreshPlaceholders(); // 刷新占位符值
  }
});
```

`useDistributed()` 提供的 `status` 是 `DistributedView` 的数据源。连接建立后自动刷新工具列表和占位符值；连接断开后所有实时遥测数据标记为"未连接"状态。

## 4. 工具调用交互覆盖层（ToolInteractionOverlay.vue）

> 源码：`src/features/distributed/ToolInteractionOverlay.vue`（108 行）

### 4.1 职责定位

`ToolInteractionOverlay` 是分布式工具体系中 **Interactive 工具模式** 的前端承载容器。当前版本为 Phase 3 骨架，已具备完整的事件监听与覆盖层渲染能力，但仅实现了以下功能：

- 接收并展示 `tool-ui-request` 事件（预留的交互式工具请求）
- 处理 `distributed-notification` 事件，调用系统通知 API
- 处理 `distributed-clipboard-write` 事件，调用前端剪贴板 API

组件通过 `<Teleport to="#vcp-feature-overlays">` 将覆盖层挂载到全局 overlay 容器，确保层级独立于路由视图。

### 4.2 触发时机

覆盖层的显示由 Rust 后端通过 Tauri 事件触发：

| 事件名 | 发射源（Rust） | 触发条件 | 前端动作 |
|--------|---------------|----------|----------|
| `tool-ui-request` | `InteractiveTool::execute()`（Phase 3 预留） | AI 调用需要用户确认/操作的 Interactive 工具 | 渲染模态覆盖层，等待用户交互 |
| `distributed-notification` | `notification.rs::NotificationTool::execute()` | 主服务器调用 `MobileNotification` | 调用 `new Notification(title, { body })` 或 console fallback |
| `distributed-clipboard-write` | `clipboard.rs::ClipboardTool::execute()` (action=write) | 主服务器调用 `MobileClipboard` 写入 | 调用 `navigator.clipboard.writeText(content)` |
| `distributed-clipboard-read` | `clipboard.rs::ClipboardTool::execute()` (action=read) | 主服务器调用 `MobileClipboard` 读取 | **当前未在前端实现监听**（Phase 3 完善） |

### 4.4 vcp_fold 块解析（新增）

`DistributedView` 在"占位符"标签页中新增了对 vcp_fold 渐进式内容展开协议的前端解析支持：

**解析逻辑**：
- 正则匹配 `[===vcp_fold: LEVEL ::desc: DESCRIPTION===]\nCONTENT` 块
- 提取 fold 级别（0.0-1.0 浮点数）、描述文本、内容
- 按级别分组展示：基础 (0.0-0.2)、详细 (0.2-0.4)、完整 (0.45+)

**UI 渲染**：
- 默认展开 level ≤ 0.2 的块（基础信息）
- Level > 0.2 的块以折叠状态显示，标题为 desc 描述文本
- 点击标题可展开/折叠详细内容
- 无 vcp_fold 标记的占位符值作为纯文本显示（向后兼容）

**数据来源**：
- 通过 `get_all_placeholder_values()` 定时轮询获取
- Streaming 工具（cpu_info、motion_sensor、ambient_sensor、device_status_summary）的 `read_current()` 返回已包含 vcp_fold 格式
- Android 原生 `SensorStatusManager.kt` 也输出 vcp_fold 格式

**代码示例**（TypeScript）：
```typescript
function parseFoldBlocks(text: string): FoldBlock[] {
  const regex = /\[===vcp_fold:\s*([\d.]+)\s*::desc:\s*([^=]+)===\]\n([\s\S]*?)(?=\[===vcp_fold:|$)/g;
  const blocks: FoldBlock[] = [];
  let match;
  while ((match = regex.exec(text)) !== null) {
    blocks.push({
      level: parseFloat(match[1]),
      description: match[2].trim(),
      content: match[3].trim()
    });
  }
  return blocks.length > 0 ? blocks : [{ level: 0, description: '', content: text }];
}
```

### 4.5 与 ToolBlock 的关系

`ToolBlock`（`src/features/chat/blocks/ToolBlock.vue`）与 `ToolInteractionOverlay` 是**互补但独立**的两个 UI 实体：

| 维度 | ToolBlock | ToolInteractionOverlay |
|------|-----------|------------------------|
| **定位** | 聊天消息中的内容块渲染器 | 全局覆盖层，请求用户实时操作 |
| **数据来源** | `message.blocks`（后端 SSE 流推送的 AST） | Tauri 事件 `tool-ui-request` |
| **展示内容** | 工具调用参数（tool-use）或执行结果（tool-result） | 需要用户确认的交互式工具界面 |
| **用户动作** | 展开/折叠查看详情 | 确认、拒绝、输入、拍照等 |
| **典型场景** | AI 已调用工具并返回结果，展示给人类阅读 | AI 请求调用相机/生物识别，需人类授权 |
| **层级** | 普通文档流（`z-content`） | `z-overlay`（固定层级 30） |

```
ChatView
├── MessageRenderer
│   └── ToolBlock          ← 展示 "MobileBatteryInfo 返回: 电量 85%"
│       └── 可折叠的结果卡片
│
└── #vcp-feature-overlays
    └── ToolInteractionOverlay  ← 展示 "AI 请求使用相机，请确认"
        └── 模态对话框（z-overlay）
```

> 关键区分：`ToolBlock` 用于**事后展示**已完成的工具调用结果；`ToolInteractionOverlay` 用于**事中拦截**需要人类参与的 Interactive 工具。两者不会同时针对同一个工具调用出现——OneShot/Streaming 工具走 ToolBlock 结果展示，Interactive 工具走 Overlay 实时交互。

## 5. 分布式组合式函数（useDistributed.ts）

> 源码：`src/features/distributed/composables/useDistributed.ts`（118 行）

### 5.1 API 与返回值

`useDistributed()` 是一个自包含的组合式函数，**不依赖任何现有 store**（chatManager、assistant、settings store 均不导入）。它通过模块级状态实现跨组件共享：

```typescript
export interface DistributedStatus {
  connected: boolean;           // WebSocket 是否已建立并收到 connection_ack
  server_id: string | null;     // VCPToolBox 主服务器分配的服务器 ID
  client_id: string | null;     // 本节点在该服务器上的客户端 ID
  registered_tools: number;     // 成功注册到主服务器的工具数量
  last_error: string | null;    // 最后一次连接/操作错误信息
}

export function useDistributed() {
  // 返回对象
  return {
    status: readonly(status),      // Ref<DistributedStatus> — 只读包装
    loading: readonly(loading),    // Ref<boolean> — 连接中状态
    start,                          // (wsUrl, vcpKey, deviceName) => Promise<void>
    stop,                           // () => Promise<void>
    refreshStatus,                  // () => Promise<void>
  };
}
```

#### 使用示例

```vue
<script setup>
import { useDistributed } from "@/features/distributed/composables/useDistributed";

const { status, loading, start, stop } = useDistributed();

// 在模板中直接读取
// status.connected → boolean
// status.registered_tools → number
</script>
```

### 5.2 与 Rust distributed 模块的封装

`useDistributed.ts` 是前端与 Rust `distributed/` 模块之间的**唯一官方入口**。它封装了 4 个 Tauri Command 和 1 个事件监听。这种单一入口设计避免了多个组件直接调用 `invoke()` 导致的调用点分散问题——所有分布式相关的 IPC 调用都收敛于此文件，便于后续迁移到 Tauri v2 新 API 或增加统一的请求拦截/日志逻辑。

| 前端 API | 对应 Rust Command | 参数 | 说明 |
|----------|-------------------|------|------|
| `start(wsUrl, vcpKey, deviceName)` | `start_distributed_node` | `wsUrl: string`, `vcpKey: string`, `deviceName: string` | 启动 WebSocket 连接，失败时抛出并记录 `last_error` |
| `stop()` | `stop_distributed_node` | 无 | 发送 shutdown 信号，abort 后台任务 |
| `refreshStatus()` | `get_distributed_status` | 无 | 查询当前状态快照，更新 `status` ref |
| `refreshTools()` | `get_registered_tools_metadata` | 无 | 拉取全部工具元数据（名称、描述、启用状态） |
| `toggleTool()` | `update_disabled_tools` | `disabledTools: string[]` | 同步禁用工具列表到 Rust 端 |

**事件监听机制**：

```typescript
// 模块级单例监听
unlisten = await listen<DistributedStatus>("vcp-distributed-status", (event) => {
  status.value = event.payload;
});
```

- 使用 `listenerCount` 计数器实现引用计数：第一个调用 `useDistributed()` 的组件挂载时建立监听，最后一个卸载时销毁监听。
- 这种设计避免了多个组件同时调用 `useDistributed()` 导致的重复监听或过早取消。

#### 5.2.1 引用计数监听器模式

`useDistributed` 使用 **模块级引用计数** 管理 Tauri 事件监听器的生命周期：

```
┌─────────────────────────────────────────────┐
│          useDistributed.ts (module scope)    │
│                                              │
│  let listenerCount = 0;  // 引用计数器       │
│  let unlisten: (() => void) | null = null;  │
│                                              │
│  export function useDistributed() {          │
│    if (listenerCount++ === 0) {             │
│      // 第一个消费者 → 建立监听             │
│      unlisten = await listen(...);           │
│    }                                         │
│                                              │
│    onUnmounted(() => {                       │
│      if (--listenerCount === 0) {           │
│        // 最后一个消费者卸载 → 销毁监听      │
│        unlisten?.();                         │
│        unlisten = null;                     │
│      }                                       │
│    });                                       │
│    // ...                                    │
│  }                                           │
└─────────────────────────────────────────────┘
```

**使用场景**：
- `DistributedView.vue` 和 `DistributedSettingsSection.vue` 可能同时挂载
- 引用计数确保只有一个 Tauri 事件监听器（避免重复接收事件）
- 最后一个组件卸载时才真正销毁监听器（避免过早断开）

**与普通 composable 的对比**：
- 普通 composable：每个组件实例独立，各自建立/销毁监听（造成 N 倍事件）
- 引用计数模式：模块级单例监听，组件间共享，零冗余事件

### 5.3 错误处理策略

| 操作 | 错误场景 | 处理行为 |
|------|----------|----------|
| `start()` | WebSocket 连接失败、认证错误 | `status.last_error = e.toString()`；`loading = false`；抛出异常供调用方捕获 |
| `stop()` | 后台任务已中止、state 读取失败 | `status.last_error = e.toString()`；静默失败（不抛出） |
| `refreshStatus()` | Command 调用失败 | `console.warn` 日志；不修改 `status`，避免覆盖已有的有效状态 |
| 事件监听 | Tauri 事件通道异常 | 由 Tauri 运行时处理，前端无额外逻辑 |

> 注意：`start()` 在失败时会抛出异常，因此 `DistributedSettingsSection` 中需用 `try/catch` 包裹，避免未处理 Promise 拒绝。
>
> 另：`refreshStatus()` 在失败时不抛出也不修改 `status`，这是一种保守策略——如果 Rust 侧的 `DistributedState` 尚未初始化（如应用启动极早期），静默失败比将状态重置为空白更安全。

### 6.2 Android 原生传感器协作详解

#### Phase 2 Android 原生传感器（3 个）

`MobileLocation`、`MobileMotion`、`MobileAmbient` 是典型的 **Android 原生传感器**。它们的数据流完全依赖 Android 原生 `SensorStatusManager.kt` 通过 Plugin IPC 通道（`plugin:vcp-mobile|get_sensor_data`）提供。Rust 侧每个工具通过不同的 `max_age_secs` 容忍陈旧度，反映了不同传感器数据的时效性差异：

| 工具 | `max_age_secs` | 设计理由 |
|------|----------------|----------|
| `MobileLocation` | 300 秒 | GPS 室内信号弱，更新慢；过短的容忍期会导致频繁报告"等待传感器采集..." |
| `MobileMotion` | 120 秒 | 运动状态变化较快，但 Burst 采样周期为 30 秒，2 倍余量足够 |
| `MobileAmbient` | 180 秒 | 环境光与气压变化缓慢，但传感器可能因省电被系统暂停 |

这三个工具的数据流图如下：

```
Android 原生 (SensorStatusManager.kt)
    ├── LocationManager ────► 经纬度、精度、海拔
    ├── SensorManager ──────► 加速度、陀螺仪
    └── 环境传感器 ──────────► 光照强度、气压
                │
                │ Plugin IPC: plugin:vcp-mobile|get_sensor_data
                ▼
        Rust distributed/ 工具层
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
location.rs  motion    ambient
            _sensor.rs  _sensor.rs
    │           │           │
    └───────────┴───────────┘
                │
    ┌───────────┴───────────┐
    ▼                       ▼
device_status    client.rs 30s 定时推送
_summary.rs              │
    │                       │
    └───────► update_static_placeholders ──► 主服务器
```

> **架构变更**：原 `frontend_bridge.rs` 已删除。传感器数据不再流经前端 Vue 组件或 `SENSOR_DATA` 内存缓存，而是直接从 Android 原生层通过 Plugin IPC 透传到 Rust 工具层的 `read_current()` 方法。`SensorStatusManager.kt` 在 Android 原生层使用突发采样（Burst Sampling）策略降低功耗，并维护传感器数据缓存供 IPC 调用即时返回。

#### OneShot 前端代理工具（2 个）

`MobileNotification` 和 `MobileClipboard` 虽归类为 OneShot，但因其需要调用浏览器/系统 API，实际通过**事件发射 → 前端执行**的代理模式工作：

```rust
// notification.rs
app.emit("distributed-notification", json!({"title": ..., "body": ...}));
```

```typescript
// ToolInteractionOverlay.vue
await listen("distributed-notification", (event) => {
  if ("Notification" in window && Notification.permission === "granted") {
    new Notification(event.payload.title, { body: event.payload.body });
  }
});
```

### 6.3 权限请求与引导

当前版本（v1.0.3）的权限处理遵循**被动降级**原则。传感器采集已全面迁移到 Android 原生层，权限检查由 `SensorStatusManager.kt` 在采集启动时执行：

| 权限 | 涉及工具 | 采集方式 | 权限检查点 | 降级行为 |
|------|----------|----------|-----------|----------|
| **定位权限** (`ACCESS_FINE_LOCATION` / `ACCESS_COARSE_LOCATION`) | MobileLocation | Android 原生 `LocationManager`（GPS + Network Provider） | `SensorStatusManager.startLocationListening()` 中 `ContextCompat.checkSelfPermission()` 双重检查 | 两项权限均未授予时，定位采集静默跳过，缓存值设为 `"位置信息: 未获得定位权限"`；运动和环境的采集不受影响 |
| **通知权限** | MobileNotification | 前端 `new Notification()` 浏览器 API | 首次调用前由浏览器自动弹窗 | 未授权时 fallback 到 `console.log` |
| **剪贴板权限** | MobileClipboard | 前端 `navigator.clipboard.writeText()` 浏览器 API | 由浏览器自动管理 | 写入失败时 `console.error` |
| **运动传感器**（加速度计） | MobileMotion | Android 原生 `SensorManager(TYPE_ACCELEROMETER)` 突发采样 | 无需运行时权限（Android 硬件传感器默认可用） | 设备无加速度计时，缓存值设为 `"运动状态: 设备无重力传感器"` |
| **环境传感器**（光 + 气压） | MobileAmbient | Android 原生 `SensorManager(TYPE_LIGHT, TYPE_PRESSURE)` 持续监听 | 无需运行时权限（Android 硬件传感器默认可用） | 传感器不可用时，缓存值设为 `"环境传感器: 设备不支持或权限未授予"` |

> **架构变更说明**：v1.0.3 将定位、运动、环境三类传感器从 Web API（`navigator.geolocation`、`devicemotion`、`AmbientLightSensor`）迁移到 Android 原生 `SensorStatusManager.kt` + Plugin IPC 通道。新的权限模型不再依赖 WebView 的权限弹窗机制，而是通过 Android 系统级权限检查实现。详见 `docs/plugins/06_权限与系统控制.md` §8--§10。
>
> Phase 3 计划：在 `ToolInteractionOverlay` 或独立 `PermissionGate` 中增加显式的权限引导 UI，对未授权的能力提供"去设置页开启"的跳转按钮。

## 10. 设计决策与注意事项

### D1. 为什么传感器采集迁移到 Android 原生层？

传感器采集从 Vue 前端迁移到 Android 原生 `SensorStatusManager.kt` + Plugin IPC 通道的优势：
- **可靠性**：Android 原生 `LocationManager`/`SensorManager` 比 WebView 的 `navigator.geolocation`/`devicemotion`/Generic Sensor API 更稳定，不受 WebView 权限弹窗限制
- **功耗优化**：Burst Sampling 策略在原生层实现更高效（30 秒周期中仅 2 秒活跃采样），且原生可精准控制传感器硬件启停
- **生命周期持久**：不受前端页面切换影响，即使 Vue 路由跳转或 WebView 被挂起，Android 原生传感器采集仍继续运行（通过前台服务）
- **IPC 延迟更低**：Rust 工具层直接通过 Plugin IPC 获取最新缓存值，省去 Vue → invoke → HashMap → Rust 的多次序列化/反序列化开销

### D2. 为什么 ToolInteractionOverlay 同时处理 notification 和 clipboard？

虽然通知和剪贴板在功能上差异很大，但它们共享同一模式：**Rust OneShot 工具需要调用前端浏览器 API**。将这些代理逻辑集中在一个全局覆盖层组件中，避免了为每个小功能创建独立的全局监听器组件，降低了挂载点复杂度。

### D3. 派生连接信息 vs. 独立配置

`DistributedSettingsSection` 没有独立的 URL/Key 输入框，而是派生自 `settings.vcpLogUrl` 和 `settings.vcpLogKey`。这是刻意的设计：
- **优点**：减少用户配置负担；分布式节点与日志服务器天然同源
- **限制**：未来若分布式网络与日志服务器分离，需改为独立字段

### D4. listenerCount 引用计数的必要性

`useDistributed.ts` 使用 `listenerCount` 而非简单的布尔标志，是因为 Vue 的组件树中可能存在**多个并发消费者**：

```
App.vue
├── DistributedView.vue          ──► useDistributed() → listenerCount++
├── DistributedSettingsSection   ──► useDistributed() → listenerCount++
└──（未来可能新增的）分布式仪表盘 ──► useDistributed() → listenerCount++
```

只有当最后一个消费者卸载时，Tauri 事件监听器才会真正销毁。这避免了"组件 A 卸载导致组件 B 收不到状态更新"的竞争问题。

### D4. 安全与隐私

- GPS 坐标、环境光、运动状态等敏感数据**仅存储在内存**（Android 原生 `SensorStatusManager` 缓存 + Rust `read_current()` 瞬时获取），不写入 SQLite 或本地文件
- 数据通过 WebSocket 直接发送至用户自托管的 VCPToolBox 主服务器，不经过第三方
- 传感器采集仅在用户主动开启"分布式节点"开关且成功连接后才开始

### D5. 已知限制（v1.0.3）

- `ToolInteractionOverlay` 的 Interactive 工具骨架尚未实现具体工具 UI（相机、生物识别等）
- `distributed-clipboard-read` 事件在前端没有监听器，`MobileClipboard` 的 `read` 操作目前返回占位符提示
- 网络接口 IP 枚举（`client.rs::report_ip`）的本地 IPv4 列表当前为空（TODO）
- 定位权限（`ACCESS_FINE_LOCATION` / `ACCESS_COARSE_LOCATION`）的主动请求尚未集成到现有 `request_android_permission` 流程中，当前依赖 Android 系统自动弹窗或用户手动授予。详见 `docs/plugins/06_权限与系统控制.md` §10

| 术语 | 英文/缩写 | 定义 |
|------|-----------|------|
| 分布式节点 | Distributed Node | 物理设备（Android 手机）作为 VCPChat 分布式网络中的计算与感知终端 |
| 主服务器 | Main Server | VCPToolBox 中的 `VCPDistributedServer`，负责管理节点注册、工具调度 |
| SlidePage 面板 | SlidePage Panel | 全屏滑出式覆盖面板，用于 `DistributedView` 等全功能 UI 的展示 |
| OneShot 工具 | OneShot Tool | 调用后立即返回结果的工具模式，无前端 UI 参与 |
| Interactive 工具 | Interactive Tool | 需要前端展示 UI、等待用户交互后才能返回结果的工具模式 |
| Streaming 工具 | Streaming Tool | 持续产生数据，通过占位符机制定期推送至主服务器的工具模式 |
| 占位符 | Placeholder | 形如 `{{MobileBattery}}` 的模板标记，主服务器在构造 LLM Prompt 时替换为实时值 |
| 突发采样 | Burst Sampling | Android 原生 `SensorStatusManager.kt` 中运动传感器的采集策略：2 秒高频采样 + 28 秒休眠 |
| Phase 1 传感器 | Phase 1 Sensors | 纯 Rust/sysfs 读取的硬件信息（电池、CPU、内存等），不依赖前端 |
| Phase 2 传感器 | Phase 2 Sensors | 依赖 Android 原生 API（LocationManager、SensorManager 等）并通过 Plugin IPC 通道传递的数据 |
| Phase 3 聚合 | Phase 3 Aggregation | 综合多源数据生成摘要的复合工具（如 `MobileStatusSummary`）及 Interactive 工具 UI |
| 插件 IPC 通道 | Plugin IPC Channel | `plugin:vcp-mobile\|get_sensor_data` 通道，连接 Android 原生 `SensorStatusManager.kt` 采集器与 Rust 工具 `read_current()` |
| 工具清单 | Tool Manifest | 向主服务器注册时提交的 JSON 元数据，含名称、描述、参数 Schema、类型 |
| VCP Key | VCP Key | 节点认证密钥，拼接在 WebSocket URL 中供主服务器校验 |
