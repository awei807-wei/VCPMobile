# 15. 设置系统总览

## 1. 概述

### 1.1 领域定位

`features/settings/` 是 VCP Mobile 前端**全局配置**的集中管理领域，负责将 Rust 后端持久化的设置项以高内聚、低耦合的方式呈现给用户，并协调设置变更在整个应用中的副作用传播。

该领域边界严格限定为：

- **设置视图容器**：`SettingsView.vue` 作为 SlidePage 级全屏页面，承载全部设置子页面
- **设置原子组件**：`src/components/settings/` 下 9 个可复用的 UI 原语（Card / Row / Switch / TextField / ActionButton / ActionWithStatus / InlineStatus / Disclosure / Section 等）
- **设置数据流**：`settingsStore` 管理设置状态的加载、缓存、增量更新与回写
- **主题子系统**：`ThemePicker.vue` + `themeStore` 负责视觉主题的动态切换与 CSS 变量注入

该领域**不**涉及：
- 智能体个体的配置编辑（由 `features/agent/` 的 `AgentSettings.vue` 负责）
- 群聊参数配置（由 `features/group/` 的 `GroupSettings.vue` 负责）
- 消息持久化本身（由 `features/chat/` 负责）

### 1.5 核心设计原则

1. **分层隔离**：SettingsView 只负责视图编排，不直接调用 `invoke()`；所有 IPC 调用下沉到 Section 组件或 Store 中。
2. **本地副本模式**：SettingsView 维护一份本地 `settings` ref，编辑操作只修改本地副本，仅在用户触发保存（返回/关闭/显式保存）时才回写后端。避免每个输入框的 `input` 事件都触发 IPC。
3. **增量更新优先**：Store 提供 `updateSettings`（增量 Patch）与 `saveSettings`（全量覆盖）两种 API，Section 内部简单变更使用 `saveSettings`，跨模块联动使用 `updateSettings`。
4. **主题与设置解耦**：主题状态（`themeStore`）独立于 `settingsStore`，仅在初始化时从 `localStorage` 恢复，不依赖后端设置表。壁纸预览通过 Vite 动态导入的 Theme TS 模块实时解析 CSS 变量生成。
5. **副作用后置**：设置回写成功后，Rust 层在 `internal_write_settings` 中判断 `vcp_log_url` / `vcp_log_key` 是否实际变更，若变更则异步触发 VcpLog 重连，避免前端关心连接生命周期。

## 2. 设置视图（SettingsView.vue）

### 2.1 打开方式（SlidePage）

`SettingsView` 并非独立路由页面，而是通过 `SlidePage.vue` 覆盖层组件从右侧滑入的全屏容器：

```vue
<!-- src/features/settings/SettingsView.vue (L145) -->
<SlidePage :is-open="props.isOpen" :z-index="props.zIndex">
```

- **打开触发**：由父组件（`App.vue` 或布局外壳）通过 `v-model` 或事件将 `isOpen` 置为 `true`
- **层级**：默认 `z-index = 50`，语义上对应 `z-sheet` 层（`LAYER_SHEET`），与 `BottomSheet`、`ModelSelector` 同级
- **关闭方式**：点击关闭按钮、`goBack()` 返回子页面、或外层 `SlidePage` 的遮罩点击/手势返回

### 2.2 组件结构

SettingsView 采用**双态结构**：主页（Category 列表）+ 子页面（Section 详情）。二者在同一 `SlidePage` 内通过 `currentSubPage` 状态切换，配合 `slide-subpage` Transition 实现原生级推入/推出动画。

Transition 使用 `cubic-bezier(0.32, 0.72, 0, 1)` 缓动曲线，进入耗时 350ms，离开耗时 300ms，与 iOS 标准导航推入动画手感接近：

```css
/* src/features/settings/SettingsView.vue */
.slide-subpage-enter-active {
  transition: transform 0.35s cubic-bezier(0.32, 0.72, 0, 1);
}
.slide-subpage-leave-active {
  transition: transform 0.3s cubic-bezier(0.32, 0.72, 0, 1);
}
.slide-subpage-enter-from,
.slide-subpage-leave-to {
  transform: translateX(100%);
}
```

### 2.4 与 settingsStore 的绑定

SettingsView 内部维护一份**本地副本** `settings` ref，而非直接绑定 `settingsStore.settings`：

```ts
// src/features/settings/SettingsView.vue (L42-L58)
const settings = ref<AppSettings>({
  userName: "User",
  vcpServerUrl: "",
  vcpApiKey: "",
  vcpLogUrl: "",
  vcpLogKey: "",
  syncServerUrl: "",
  syncHttpUrl: "",
  syncToken: "",
  adminUsername: "",
  adminPassword: "",
  fileKey: "",
  agentOrder: [],
  groupOrder: [],
  topicSummaryModel: "gemini-2.5-flash",
  syncLogLevel: "INFO",
});
```

**加载时**（`isOpen` 变为 `true` 或 `onMounted`）：

```ts
const loadSettings = async () => {
  await settingsStore.fetchSettings();           // 调用 read_settings
  if (settingsStore.settings) {
    settings.value = JSON.parse(JSON.stringify(settingsStore.settings)); // 深拷贝隔离
  }
};
```

**保存时**：子页面返回（`goBack`）或显式保存（`saveSettings`）调用 `settingsStore.saveSettings(settings.value)`，将本地副本全量回写后端。

> 设计意图：避免用户在输入过程中频繁触发后端写入，同时防止子组件直接修改全局 Store 状态导致数据不一致。

### 2.5 子页面的返回键管理

SettingsView 通过 `useModalHistory` 将子页面注册到全局模态栈中，确保 Android 物理返回键行为符合预期：

```ts
// src/features/settings/SettingsView.vue (L133-L141)
watch(currentSubPage, (val) => {
  if (val) {
    registerModal(SUBPAGE_MODAL_ID, () => {
      goBack();  // 按返回键时触发保存并退出子页面
    });
  } else {
    unregisterModal(SUBPAGE_MODAL_ID);
  }
});
```

当用户处于子页面时按下系统返回键，ModalHistory 会调用注册的回调 `goBack()`，而非直接关闭整个 SettingsView。

## 3. 设置数据流

### 3.1 应用启动时设置加载

应用启动时，`settingsStore` 不会自动预加载。首次加载发生在用户第一次打开 `SettingsView` 时（`isOpen` watch 触发 `loadSettings`）。这与 Agent 列表等热数据不同，属于**惰性加载**策略，降低启动时的 IPC 开销。

```
App.vue 启动
    │
    ▼
SettingsView.isOpen = false  (不加载)
    │
    ▼ (用户点击"设置")
SettingsView.isOpen = true
    │
    ▼
loadSettings()
    ├──► settingsStore.fetchSettings()
    │       └──► invoke("read_settings")
    │               └──► Rust: settings_manager::read_settings()
    │                       ├──► 查 SQLite cache
    │                       └──► 未命中则查 DB → 反序列化 → 回填 cache
    └──► JSON.parse(JSON.stringify(...)) → 本地 settings ref
```

### 3.2 设置变更的本地更新

各 Section 通过 `v-model` 直接绑定 `SettingsView` 的本地 `settings` ref。由于 `settings` 是对象引用传递，子组件的 `v-model` 变更会直接反应到父组件的 `settings.value` 上，无需显式事件上报：

```vue
<!-- VcpCoreSettingsSection.vue -->
<SettingsTextField v-model="settings.vcpServerUrl" label="VCP 服务器 URL" />
```

由于 `settings` 是对象引用，子组件通过 `defineProps<{ settings: AppSettings }>()` 接收后，`v-model` 的变更会直接反应到父组件的 `settings.value` 上。这是 Vue 的响应式引用传递特性，无需显式事件上报。

### 3.3 设置回写后端

回写有两种触发路径：

| 路径 | 触发场景 | 调用方法 | IPC 命令 |
|------|----------|----------|----------|
| 显式保存 | Section 内点击"保存"或验证连接前 | `settingsStore.saveSettings(settings.value)` | `write_settings`（全量覆盖） |
| 子页面返回 | 点击子页面左上角 ← | `goBack()` → `saveSettings()` | `write_settings`（全量覆盖） |
| 增量更新 | 外部模块（如 Agent 排序变更） | `settingsStore.updateSettings({ agentOrder: [...] })` | `update_settings`（增量 Patch） |

**显式保存 vs 增量更新的选择策略：**

- `saveSettings`：适用于用户在设置面板内完成一系列编辑后的一次性全量回写。SettingsView 内所有 Section 均采用此方式。
- `updateSettings`：适用于外部模块仅变更一个或少数几个字段的场景（如 Agent 拖拽排序后更新 `agentOrder`），减少序列化开销并避免覆盖用户正在编辑中的其他字段。

Rust 后端 `write_settings` 的执行流程：

1. 获取 `SettingsState.lock` 互斥锁（防止并发写入冲突）
2. 将 `Settings` 序列化为 JSON
3. `INSERT OR REPLACE INTO settings (key, value, updated_at) VALUES ('global', ?, ?)`
4. 比对旧 cache 与新设置中的 `vcp_log_url` / `vcp_log_key`
5. 若变更，异步 spawn VcpLog 重连任务
6. 更新内存 cache

### 3.4 设置变更的副作用

| 变更字段 | 副作用 | 触发位置 |
|----------|--------|----------|
| `vcpLogUrl` / `vcpLogKey` | 触发 VcpLog WebSocket 重连 | Rust `internal_write_settings` |
| `currentThemeMode` | 调用 `set_theme` 更新设置表；前端 `themeStore` 独立响应 | Rust `set_theme` command |
| `agentOrder` / `groupOrder` | 影响侧边栏 Agent/Group 列表排序 | 前端 `agentStore` / `groupStore` 读取 |
| `topicSummaryModel` | 影响话题总结请求的模型参数 | 前端 `topicStore` |
| `distributedEnabled` | 触发分布式节点 WS 连接或断开 | 前端 `useDistributed` composable |
| 任意字段 | Toast 通知 "设置更新成功" | 前端 `settingsStore.saveSettings` |

> 注意：当前版本**未实现**基于 `settings` 的 i18n 语言切换。语言为硬编码中文。`currentThemeMode` 虽然存在于 `AppSettings` 接口中，但实际主题切换由 `themeStore` 通过 `localStorage` 管理，与后端设置表弱关联。

### 3.5 设置加载态与错误处理

SettingsView 在 `loadSettings()` 期间显示全屏加载占位：

```vue
<div v-if="loading" class="flex-1 flex items-center justify-center ...">
  正在加载设置...
</div>
```

加载失败（`fetchSettings` 抛异常）时，前端仅打印日志，UI 仍然展示默认值的输入框，允许用户手动重新编辑。这种**降级可用**策略确保即使后端 DB 异常，用户仍可尝试修复配置（如修正错误的 VCP URL）。

## 4. 主题选择（ThemePicker.vue）

### 4.1 主题列表展示

`ThemePicker` 是一个纯展示组件，主题元数据完全由 `themeStore` 提供：

```ts
// src/features/settings/ThemePicker.vue (L12-L13)
const themeStore = useThemeStore();
const thumbnails = computed(() => themeStore.themeThumbnails);
```

主题列表来源为 Vite 的**动态导入**（`import.meta.glob`）：

```ts
// src/core/stores/theme.ts (L41)
const themeModules = import.meta.glob('../../assets/themes/*.ts') as Record<string, () => Promise<ThemeModule>>;
```

每个主题是一个独立的 `.ts` 文件，导出 `meta`（文件名、显示名）与 `variables`（暗色/亮色 CSS 变量映射）。`themeStore.fetchThemes()` 在组件 `onMounted` 时异步加载所有主题模块，构建 `availableThemes` 列表。

### 4.2 明暗模式切换

明暗模式由 `themeStore.mode`（`'light' | 'dark' | 'system'`）控制，通过 `watch(mode, applyTheme, { immediate: true })` 自动响应：

```ts
// src/core/stores/theme.ts (L191-L206)
const applyTheme = (newMode: ThemeMode) => {
  const isDark = newMode === 'dark' || (newMode === 'system' && window.matchMedia('(prefers-color-scheme: dark)').matches);
  isDarkResolved.value = isDark;
  document.documentElement.classList.toggle('dark', isDark);
  document.body.classList.toggle('light-theme', !isDark);
  // 若主题模块已加载，重新注入对应色系的 CSS 变量
  if (currentThemeModule) {
    const vars = isDark ? currentThemeModule.variables.dark : currentThemeModule.variables.light;
    injectVariables(vars);
  }
};
```

系统主题变化监听：通过 `window.matchMedia('(prefers-color-scheme: dark)').addEventListener('change', ...)` 实现，仅在 `mode === 'system'` 时生效。

### 4.3 壁纸预览与选择

主题卡片中的壁纸预览图并非独立功能，而是从主题模块的 CSS 变量中**自动提取**：

```ts
// src/core/stores/theme.ts (L119-L137)
for (const theme of themes) {
  const darkWp = theme.variables?.dark?.['--chat-wallpaper-dark'];
  const lightWp = theme.variables?.light?.['--chat-wallpaper-light'];
  let rawPath = darkWp || lightWp;
  if (rawPath && rawPath !== 'none') {
    const match = rawPath.match(/url\(['"]?(.*?)['"]?\)/);
    let filename = match ? match[1] : rawPath;
    filename = filename.replace(/^.*[\\\/]/, '').replace(/['"]/g, '');
    filename = filename.split('.')[0] + '.webp';
    thumbs[theme.fileName] = `/wallpaper/${filename}`;
  }
}
```

提取规则：
- 优先取暗色壁纸 `--chat-wallpaper-dark`， fallback 到亮色壁纸 `--chat-wallpaper-light`
- 从 CSS `url(...)` 语法中解析文件名
- 强制转换为 `.webp` 格式，映射到 `public/wallpaper/` 目录

主题卡片的视觉反馈：
- **未选中**：`border-transparent opacity-80`， hover 时背景图 `scale-110`
- **选中**：`border-primary shadow-lg scale-95`，左上角显示蓝色勾选图标
- **加载中/无预览图**：显示灰色占位块 + 主题名称文字

### 4.4 与 themeStore 的联动

用户点击主题卡片后：

```ts
// src/features/settings/ThemePicker.vue (L23-L25)
const selectTheme = (theme: ThemeInfo) => {
  themeStore.applyThemeFile(theme.fileName);
};
```

`applyThemeFile` 的执行链：

1. 更新 `currentTheme.value` 与 `localStorage['vcp-theme-name']`
2. 通过 `findThemeLoader` 定位 Vite 动态导入的加载器
3. 异步加载 ThemeModule（含 `variables` + `extraCss`）
4. 根据 `isDarkResolved` 选择 dark/light 变量集
5. `injectVariables`：先清除旧主题变量（防止混合态），再注入新变量到 `document.documentElement.style`
6. 将 `extraCss`（如 `.tool-bubble` 等非变量样式）写入 `<style id="vcp-custom-theme">`

> **Legacy 主题兼容**：`themeStore` 内置 `LEGACY_THEME_MAP`，将旧版中文文件名（如 `themes冰火魔歌.css`）映射到新版英文文件名，确保用户升级后主题不丢失。

### 4.5 主题初始化性能优化

`themeStore.initTheme()` 采用**两阶段加载**策略：

1. **首阶段**：立即 `applyThemeFile(savedTheme)` 加载当前主题，确保首屏背景和基础 CSS 变量瞬间呈现
2. **二阶段**：通过 `requestIdleCallback`（fallback 为 `setTimeout(cb, 1000)`）在浏览器空闲时异步 `fetchThemes()`，扫描全部主题模块元数据，用于 ThemePicker 的列表渲染

此设计避免在应用启动时同步加载全部主题模块导致的阻塞，尤其对包含大量壁纸缩略图的主题集合至关重要。

## 8. 设计决策与注意事项

### 8.1 为什么使用本地副本而非直接绑定 Store？

SettingsView 的 `settings` ref 是 `settingsStore.settings` 的**深拷贝**，编辑过程中子组件直接修改本地对象。这种设计的收益：

- **减少 IPC 频率**：避免每个输入字符都触发后端写入
- **可取消**：用户可返回/关闭而不保存未完成的修改
- **隔离副作用**：子页面内的临时变更不会立即扩散到全局状态

代价是需要在返回/关闭时显式调用 `saveSettings()`，已通过在 `goBack()` 中统一调用解决。

### 8.2 子页面返回时自动保存的可靠性

```ts
const goBack = async () => {
  try {
    await settingsStore.saveSettings(settings.value);
  } catch (e) {
    console.error("Failed to save settings:", e);
  }
  currentSubPage.value = null;
};
```

- 保存失败**不会阻塞**返回操作，防止用户因网络/DB 异常被困在子页面
- 失败仅打印日志，无额外错误提示（当前设计）。若需强一致性，可在此添加 Toast 警告

### 8.3 主题系统的双轨持久化

主题状态持久化在两个独立位置：

| 数据 | 存储位置 | 管理方 |
|------|----------|--------|
| 当前主题文件名 | `localStorage['vcp-theme-name']` | `themeStore` |
| 当前主题模式 | `localStorage['vcp-theme-mode']` | `themeStore` |
| `currentThemeMode` | SQLite `settings` 表 | `settings_manager`（通过 `set_theme`） |

前端主题切换**不**依赖后端设置表，确保离线可用和瞬时响应。`set_theme` 仅作为向后端同步当前主题的辅助通道。

### 8.4 `AiLogicSettingsSection` 的现状

`src/features/settings/components/AiLogicSettingsSection.vue` 当前为**占位组件**，仅展示两行"暂未启用"的 `SettingsRow`。Agent 的温度、Token 限制等 AI 逻辑参数实际在 `AgentSettings.vue`（智能体个体设置）中维护，不通过全局设置面板配置。该组件保留为未来全局默认 AI 参数预留扩展位。

### 8.5 关于页面的特殊头部处理

```vue
<header v-if="currentSubPage !== 'about'">
```

`about` 子页面使用沉浸式全屏设计（WebGL 流体背景 + 3D Logo），不共享 SettingsView 的标准头部。返回按钮内嵌于 `AboutSection` 自身，通过 `@back="currentSubPage = null"` 回调实现返回。

### 8.6 VcpLog 重连的副作用隔离

Rust 端在 `internal_write_settings` 中通过比对旧 cache 与新值来决定是否重连：

```rust
let should_reconnect = {
    let old_cache = state.cache.lock().await;
    if let Some(ref old) = *old_cache {
        old.vcp_log_url != settings.vcp_log_url || old.vcp_log_key != settings.vcp_log_key
    } else {
        !settings.vcp_log_url.is_empty() || !settings.vcp_log_key.is_empty()
    }
};
```

这避免了用户仅修改用户名或主题等无关字段时，意外断开并重建 WebSocket 连接。

### 8.7 设置原子组件的变体系统

`SettingsCard` 支持 4 种视觉变体，在不同 Section 中承担不同的语义角色：

| 变体 | 用途 | 使用场景 |
|------|------|----------|
| `default` | 标准卡片，半透明灰底 | 大部分设置分组（VcpCore、Sync 等） |
| `glass` | 高透明毛玻璃 | `UserProfileSection` 的头像卡片 |
| `danger` | 红色边框+底色 | 危险操作区域（预留） |
| `warning` | 黄色边框+底色 | 警告操作区域（预留） |

`SettingsActionButton` 同样支持 4 种变体（`primary`/`secondary`/`danger`/`ghost`）与 3 种尺寸（`sm`/`md`/`lg`），在 MaintenanceSection 中，`danger` 变体用于附件 GC，而 `primary` 变体用于缓存清理，形成视觉风险分层。
