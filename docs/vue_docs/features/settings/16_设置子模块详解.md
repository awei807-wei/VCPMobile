# 16. 设置子模块详解

## 1. 概述

### 1.1 领域定位

`features/settings/components/` 是 VCP Mobile 前端**全局设置中心**的原子 UI 单元集合。该目录下的 11 个 Vue 组件各自对应 `SettingsView` 中的一个功能子域，负责将用户输入转化为对 `settingsStore` 的写操作，或通过 Tauri IPC 触发 Rust 后端的维护/查询命令。

这些组件全部作为 **SettingsView 的子页面插槽**被引用，不独立存在于路由系统中。其设计严格遵循项目"高密度线性布局"美学宪法：列表重于卡片、灰度优先、2px 高亮侧边条反馈、内敛交互（`active:scale-95` 为主）。

设置子模块不涉及任何聊天业务逻辑，其职责边界清晰限定为：
- **配置读写**：用户身份、服务器连接、同步参数、主题等持久化配置
- **维护操作**：附件 GC、缓存清理、预渲染重建等低频但关键的后台任务
- **信息展示**：版本号、开源协议、电池优化引导、同步日志浏览

### 1.4 核心设计原则

1. **设置副本模式**：`SettingsView` 维护一份 `settings` 的本地响应式副本（`ref<AppSettings>`），子组件通过 `v-model` 绑定到副本字段。用户点击"返回"时才调用 `settingsStore.saveSettings()` 统一持久化，避免频繁写盘。
2. **操作即命令**：维护类操作（GC、缓存清理、重建）不修改设置副本，而是直接通过 `invoke` 调用 Rust Command，操作结果通过原子组件 `SettingsActionWithStatus` 回显。
3. **状态机隔离**：重建会话（`RebuildSessionView`）拥有独立的状态机（`idle | running | completed | error`），不耦合于 `settingsStore`，由 `useRebuildSessionStore` 统一管理。
4. **屏幕常亮守护**：所有可能长时间运行的维护操作（重建、GC）均通过 `withScreenKeep()` 包裹，防止 Android 息屏中断导致任务失败。
5. **前后台生命周期感知**：重建操作前后显式调用 `start_streaming_service` / `stop_streaming_service`，确保 Android 不会在后台中断长时间任务。

## 2. 核心设置模块

### 2.1 VcpCoreSettingsSection — VCP 核心配置

**职责一句话**：配置 VCP 后端 HTTP/HTTPS 接入点、WebSocket 日志通道及其鉴权密钥，并提供连接健康检查能力。

**源码位置**：`src/features/settings/components/VcpCoreSettingsSection.vue`

**Props**：

```ts
defineProps<{
  settings: AppSettings;
}>();

// Emit: save-request (触发父级先持久化当前输入)
defineEmits<{
  (e: 'save-request'): void;
}>();
```

**包含的表单字段**：

| 字段 | 组件 | 绑定字段 | 类型修饰 | 说明 |
|------|------|---------|---------|------|
| VCP 服务器 URL | `SettingsTextField` | `settings.vcpServerUrl` | — | HTTP/HTTPS 端点，如 `https://vcp-endpoint.com` |
| VCP API Key | `SettingsTextField` | `settings.vcpApiKey` | `isSecure` | 鉴权密钥，支持显隐切换 |
| VCP WebSocket URL | `SettingsTextField` | `settings.vcpLogUrl` | `mono` | 日志流地址，如 `ws://localhost:8024` |
| VCP WebSocket Key | `SettingsTextField` | `settings.vcpLogKey` | `isSecure` + `mono` | WebSocket 鉴权 Key |

**连接测试流程**：

1. 用户点击"验证连接"按钮。
2. 组件先 `emit('save-request')` 触发父级 `SettingsView.saveSettings()` 将当前输入持久化到磁盘。
3. 调用 `invoke('test_vcp_connection', { vcpUrl, vcpApiKey })`。
4. Rust 侧对齐桌面端 `fetchAndCacheModels` 逻辑：
   - 解析 URL 提取 `protocol://host:port`
   - 构造 `baseUrl/v1/models` 请求
   - 携带 `Authorization: Bearer {apiKey}` 请求头
   - 10 秒超时
5. 返回 `{ success: boolean, status: number, modelCount: number, models: any }`。
6. 前端通过 `vcpPingStatus` 响应式对象驱动 `SettingsActionWithStatus` 回显结果。

**关键交互细节**：
- 测试前若 URL 为空，直接本地报错（`type: 'error', message: '请先输入 VCP 服务器 URL'`），不发起 IPC，减少无效网络开销。
- 状态文案示例：
  - `loading`: "正在验证模型列表..."
  - `success`: "连接成功！拉取到 42 个可用模型"
  - `error`: "验证失败, HTTP 状态码: 401" / 原始异常字符串
- 使用了 `status-multiline` 属性，错误信息允许换行显示。

**样式特征**：字段间使用 `space-y-5` 垂直分布，组间以 `border-t border-black/5 dark:border-white/5` 细线分隔，保持视觉呼吸感。


**包含的表单字段与操作**：

| 字段/操作 | 组件 | 绑定/触发 | 修饰符 | 说明 |
|-----------|------|----------|--------|------|
| HTTP 服务 URL | `SettingsTextField` | `settings.syncHttpUrl` | `mono` | 同步 HTTP API 端点，如 `http://192.168.x.x:5974` |

**表情包修复库流程**：

```
用户点击 REFRESH 按钮
    │
    ▼
emoticonStatus = { type: 'loading', message: '正在从远程服务器获取...' }
    │
    ▼
invoke('regenerate_emoticon_library')
    │
    └──► Rust: emoticon_manager.rs
         ├── 远程获取表情包 JSON 索引
         ├── 本地 SQLite upsert 表情元数据
         └── 返回: number (表情总数)
    │
    ▼
emoticonStatus = { type: 'success', message: '同步成功：共计 156 个表情' }
    │
    └──► 3s 后自动清空状态
```

**全量神经同步入口**：
- 不直接发起 IPC，而是调用 `overlayStore.openSyncSession()`。
- 该操作会将 `syncSession` 类型推入 `overlayStore.pageStack`，打开独立的同步会话面板。
- 同步会话面板内部包含 `SyncLogBrowserCore` 组件用于浏览历史日志。

### 2.4 UserProfileSection — 用户资料

**职责一句话**：管理本地用户身份标识（头像、用户名）及管理员接口鉴权凭据。

**源码位置**：`src/features/settings/components/UserProfileSection.vue`

**Props**：

```ts
defineProps<{
  settings: AppSettings;
}>();
```

**包含的表单字段与交互**：

| 字段 | 组件 | 绑定字段 | 修饰符 | 说明 |
|------|------|---------|--------|------|
| 头像 | `VcpAvatar` + `AvatarCropper` | — | `owner-type="user"` | 点击触发文件选择 → 裁剪器 → Blob IPC |
| 用户名 | `SettingsTextField` | `settings.userName` | — | 本地显示名称 |
| 管理员账号 | `SettingsTextField` | `settings.adminUsername` | `mono` | Basic Auth 用户名 |
| 管理员密码 | `SettingsTextField` | `settings.adminPassword` | `isSecure` + `mono` | Basic Auth 密码 |

**头像上传完整时序**：

```
用户点击头像区域
    │
    ▼
触发隐藏 <input type="file" ref="fileInput" accept="image/*">
    │
    ▼
handleFileChange(event)
    ├── 取 files[0]
    ├── new FileReader()
    ├── reader.onload = (event) => {
    │     cropImg = event.target.result  // base64 DataURL
    │     isCropping = true
    │   }
    └── reader.readAsDataURL(file)
    │
    ▼
<AvatarCropper :img="cropImg" v-if="isCropping" />
    │
    ▼
用户在裁剪器中确认
    │
    ▼
onCropConfirm(blob: Blob)
    ├── isCropping = false
    ├── const arrayBuffer = await blob.arrayBuffer()
    ├── const bytes = new Uint8Array(arrayBuffer)
    ├── await assistantStore.saveAvatar(
    │     "user",           // ownerType
    │     "user_avatar",    // ownerId
    │     blob.type,        // mimeType
    │     Array.from(bytes) // 普通数组，绕过序列化限制
    │   )
    └── avatarVersion = Date.now()  // bust VcpAvatar 缓存
```

**关键点**：
- 头像通过 `assistantStore.saveAvatar` 经 IPC 写入 Rust `avatar_service.rs`，以 SHA-256 哈希命名文件存储于 `app_config_dir/avatars/`。
- `avatarVersion` 时间戳用于 bust 前端缓存，`VcpAvatar` 组件通过 `version` prop 变化重新加载图片。
- 管理员账号密码用于远程获取表情包库等管理接口的 Basic Auth 鉴权（见 `SyncSettingsSection` 的 `fileKey` 和表情包修复）。
- 卡片使用 `SettingsCard variant="glass"`，在亮色主题下呈现半透明白色毛玻璃质感。

## 3. 系统维护模块

### 3.1 MaintenanceSection — 维护与清理

**职责一句话**：提供附件库垃圾回收、WebView 缓存清理、全量预渲染重建入口三项系统维护操作。

**源码位置**：`src/features/settings/components/MaintenanceSection.vue`

**包含的操作按钮**：

| 操作 | 按钮文案 | 对应 Rust Command | 屏幕常亮 | 危险等级 |
|------|---------|------------------|----------|:--------:|
| 附件库垃圾回收 (GC) | 立即清理 | `cleanup_orphaned_attachments` | ✅ | 🔴 |
| 清理系统缓存 | 立即清理 | `clear_webview_cache` | ✅ | 🟡 |
| 全量预渲染重建 | 一键重建 | —（打开 RebuildSessionView） | — | — |

**附件 GC 详细逻辑**（Rust 侧 `maintenance_manager.rs`）：

```
前端: invoke('cleanup_orphaned_attachments')
    │
    └──► Rust:
         1. 获取 attachments_root_dir
         2. 若目录不存在 → RETURN "没有附件需要清理"
         3. SELECT hash, internal_path FROM attachments
         4. 若为空 → RETURN "索引库为空，无需清理"
         5. SELECT DISTINCT hash FROM message_attachments
            → used_hashes: HashSet<String>
         6. 遍历 all_indexed_hashes:
            IF hash NOT IN used_hashes:
              a. 物理删除 internal_path 文件
              b. 删除缩略图 thumbnails/{hash}_thumb.webp
              c. DELETE FROM attachments WHERE hash = ?
              d. 累计 deleted_count 与 freed_size
         7. RETURN "清理完成：删除了 N 个孤儿附件，释放了 X MB 空间"
```

**WebView 缓存清理**：
- 调用 Tauri v2 原生 API `webview.clear_all_browsing_data()`。
- 清除范围：HTTP Cache、图片缓存、LocalStorage、Cookies、表单数据等全部浏览数据。
- 该操作在 Android 上由系统 WebView 实现，执行时间通常 < 1s。

**状态管理**：每个操作拥有独立的 `ref` 状态对象：

```ts
const gcStatus = ref<{ 
  type: 'success' | 'error' | 'loading' | null; 
  message: string 
}>({ type: null, message: '' });
```

成功/错误状态在 5 秒后自动清空（`setTimeout(..., 5000)`），保持界面整洁。

**"全量预渲染重建"入口**：
- 该操作不直接调用 IPC，而是通过 `overlayStore.openRebuildSession('preRender')` 打开独占式重建页面。
- 原因：重建过程可能持续数分钟，需要阻止用户离开并显示实时进度。

**源码位置**：`src/features/settings/components/RebuildSessionView.vue`

**状态机**：

**各状态 UI 表现**：

| 状态 | 视觉 | 交互限制 |
|------|------|---------|

| `error` | 红色文字显示错误信息 + "关闭"按钮 | `canDismiss = true` |

**与 Rust 后端的完整交互链**：

```
startRebuild()  (useRebuildSessionStore.ts)
    │
    ├──► invoke('plugin:vcp-mobile|start_streaming_service', 
    │           { agentName: '[预渲染重建] VCP Mobile' })
    │      // 启动 Android 前台服务，显示高优先级通知
    │      // 防止 OEM 在重建过程中杀后台
    │
    ├──► registerListener('render_rebuild_progress')
    │      // 监听 Rust 进度推送事件
    │
    ├──► withScreenKeep(() => invoke('rebuild_all_pre_renders'))
    │      // 屏幕常亮包裹
    │      //
    │      └──► Rust: message_repository.rs
    │           1. 查询全部消息 ID 列表
    │           2. 分批遍历消息
    │           3. 对每条消息重新运行 AST 解析
    │           4. 应用代码高亮（syntect）
    │           5. 更新 message.pre_render_html
    │           6. 期间 emit('render_rebuild_progress', { current, total })
    │
    └──► invoke('plugin:vcp-mobile|stop_streaming_service',
                { agentName: '[预渲染重建] VCP Mobile' })
           // 无论成功或失败，均释放前台服务
```

**完成后刷新机制**：
- 重建成功后 `needsReload = true`。
- 用户点击关闭时，若 `needsReload === true`，弹 `confirm("重建已完成，数据已更新。点击确认立即刷新以生效。")`。
- 确认后调用 `useDataReload().performFullReload()`：
  1. `clearHtmlCache()` — 清理 AST HTML 渲染缓存
  2. `fetchAgents()` / `fetchGroups()` — 刷新智能体/群组元数据
  3. `invalidateAllTopicCaches()` — 使话题列表缓存失效
  4. `loadHistoryPaginated()` — 若在当前话题中，重新加载消息以获取最新 pre_render_html

**核心流程详解**：

**1. 检查更新**：
```ts
const info: UpdateInfo = await invoke('check_for_update');
```
- Rust 请求 `api.github.com/repos/MRiecy/VCPMobile/releases/latest`
- 携带 `User-Agent: VCPMobile` 请求头
- 若 `/latest` 返回 404（最新为 prerelease），降级到 `/releases?per_page=1`
- 在 release assets 中匹配 `*arm64-v8a.apk` 后缀
- 对比 `tag_name` 与 `app.package_info().version`

**2. 下载并安装**：
```ts
await useUpdateDownloader().downloadAndInstall(url);
```
- 创建 Tauri `Channel<DownloadProgress>`
- 回调中通过 `notificationStore.addNotification` 发送实时进度 Toast（`duration: 0` 常驻）
- Rust `download_update` 使用 `reqwest` 流式下载，通过 `tokio::io::AsyncWriteExt` 写入缓存目录
- 完成后调用 `install_update`，触发 Android `Intent.ACTION_VIEW` 安装 APK

**3. 三级降级策略**：
若本地安装器失败，依次尝试：
- `openUrl(info.downloadUrl)` — 浏览器直链下载
- `openUrl(info.releasePageUrl)` — 浏览器打开 Release 页面
- 最终报错

**UI 反馈矩阵**：

| 状态 | UI 表现 |
|------|---------|
| `checking` | `SettingsInlineStatus` loading，显示 "正在检查最新版本..." |
| `no-update` | `SettingsInlineStatus` success，显示当前版本号，4s 后自动消失 |
| `update-available` | 展开信息卡片：版本号（蓝色等宽字体）、APK 大小、Release Notes（最大高度 32，可滚动）、下载按钮 |

| `installing` | `SettingsInlineStatus` loading，显示 "正在启动安装器..." |

**1. 跟手模式（用户触摸/鼠标移动）**：
```ts
const handleMove = (e: MouseEvent | TouchEvent) => {
  animSessionId++;  // 打断任何正在进行的序列
  isMixing.value = true;
  transitionStyle.value = 'none';  // 0 延迟，极致跟手
  
  // 计算相对中心点的偏移
  const targetRotY = ((x - centerX) / centerX) * 150;
  const targetRotX = ((centerY - y) / centerY) * 80;
  
  // 硬限幅防止万向锁
  rotation.value.y = Math.max(-180, Math.min(180, targetRotY));
  rotation.value.x = Math.max(-90, Math.min(90, targetRotX));
  
  // 更新光晕位置
  glowPos.value = { x: (x / width) * 100, y: (y / height) * 100 };
};
```

**2. 自动动画序列（按下后 800ms 启动）**：
按下触发 8 段 CSS 变换序列：
- 向右缓慢旋转翻面（0→180°Y，2s）
- 向左缓慢旋转复原（180→0°Y，2s）
- 向左旋转 75° 展现左侧（1.2s）
- 快速向右翻转 75° 展现右侧（0.6s）
- 快速复原 + 扭动（0.4s + 3 次 100ms 微扭）
- 上下翻转 360° + 落地抖动（0.8s + 4 次修正）

通过 `animSessionId` 自增实现打断机制：用户再次触摸时立即取消序列，回归跟手模式。

### 6.4 与 UnoCSS 快捷类的配合

原子组件大量依赖 UnoCSS 原子类，同时也有少量自定义语义化类名（用于 `:deep()` 覆盖和 Scoped CSS）。典型配合模式：

```vue

  outline-none border transition-all duration-200
  focus:border-blue-500/50 focus:bg-black/10 dark:focus:bg-white/10" />

  border rounded-2xl shadow-sm px-3 py-4" />
```

组件内部不使用 UnoCSS 快捷类（如 `glass-panel`、`card`、`btn`），而是显式写出完整原子类。原因：
1. 设置组件需要精确控制每一个 Tailwind 原子类的响应式行为
2. 深色/亮色模式的 `dark:` 前缀需要显式声明，快捷类无法覆盖所有状态组合
3. `:deep()` 覆盖时需要匹配具体类名，快捷类的组合类会增加选择器复杂度

## 9. 设计决策与注意事项

1. **设置副本 vs 直接绑定 Store**：
   `SettingsView` 不直接让子组件 `v-model` 绑定 `settingsStore.settings`，而是维护本地 `ref` 副本。原因：避免用户在子页面输入时触发频繁持久化，只在点击"返回"时统一 `saveSettings()`。这种模式牺牲了即时保存的便利性，换取了更好的 I/O 性能和更清晰的撤销语义（返回即放弃未保存修改）。

2. **AiLogicSettingsSection 的占位策略**：
   该组件在 `v0.9.14` 中仅渲染两个"暂未启用"行。Agent 级别的 Temperature / Max Tokens 配置目前仍由桌面端 `config.env` 控制，移动端尚未实现全局覆盖逻辑。保留该组件是为了 UI 架构的完整性，避免未来新增字段时破坏布局或需要重构路由分类。

3. **RebuildSessionView 的独占式设计**：
   重建操作使用独立的 `SlidePage` 页面栈项（而非内联在 `MaintenanceSection` 中），是因为重建过程可能持续数分钟，需要阻止用户离开（`canDismiss = false` + 全局遮罩），且完成后可能触发全量数据重载，需要占据用户全部注意力。

4. **AboutSection 的深色/亮色深度覆盖**：
   关于页面使用 `WebGLFluidBackground` 作为全屏动态背景，其颜色与主题无关。因此该组件通过 `<style scoped>` 中的 `:deep()` 选择器，对子组件（`SettingsCard`、`SettingsRow`、`SettingsInlineStatus`、`SettingsActionButton`）进行深度样式覆盖，确保在亮色模式下文字可读、卡片有半透明毛玻璃质感。这是项目中少数允许深度选择器覆盖的场景。

5. **SyncLogBrowserCore 的亮色缺失**：
   该组件当前仅针对深色场景设计（固定 `text-white/70` 等色值）。若未来需要在亮色设置页中复用，需补充 CSS 变量适配或使用 `text-primary-text` 等语义化颜色类。

6. **UpdateSection 的下载降级链**：
   `downloadAndInstall` 并非直接返回失败，而是在本地安装器失败后尝试浏览器直链下载，最终再尝试打开 Release 页面。该三级降级确保用户在任何网络/权限环境下都能获取更新包。

7. **头像保存的流式处理**：
   `UserProfileSection` 将 `Blob` 转为 `ArrayBuffer` 再转为 `Uint8Array`，最终通过 `Array.from(bytes)` 序列化为普通数组后 IPC 传递。这是为了绕过 Tauri v2 对 `Uint8Array` 直接序列化的限制。Rust 侧接收 `Vec<u8>` 后写入文件系统。

8. **MaintenanceSection 的状态隔离**：
   每个维护操作拥有独立的 `ref` 状态对象（`gcStatus`、`cacheStatus`），而非共享一个全局状态。这种设计允许用户同时触发多个操作（虽然 UI 上通常不会这么做），且错误信息不会互相覆盖。

9. **BatteryOptimizationGuide 的纯前端实现**：
   该组件完全基于 `navigator.userAgent` 字符串匹配，不调用任何 Rust Command。其数据（品牌、步骤）硬编码于组件内部，因为 OEM 电池优化路径通常随系统版本变化缓慢，无需后端动态配置。

10. **SettingsTextField 的安全遮罩策略**：
    使用 `-webkit-text-security: disc` 而非 `<input type="password">`，是因为后者在移动端会自动触发键盘的"密码建议"和"自动填充"行为，干扰 API Key 等人工输入场景。`-webkit-text-security` 仅做视觉遮罩，不影响输入法行为。

## 10. 术语速查表

| 术语 | 说明 |
|------|------|
| **Section** | 设置子模块组件的统称，每个 Section 对应一个功能域 |
| **SettingsView** | 设置中心的根组件，以 SlidePage 形式从右侧滑入，z-page 层级 |
| **预渲染 (Pre-render)** | 将消息原始内容提前解析为带语法高亮的 HTML，存入数据库避免运行时重复解析 |
| **GC (Garbage Collection)** | 此处特指附件库垃圾回收，删除未被消息引用的孤立文件及其缩略图 |
| **Screen Keep** | 屏幕常亮锁定，通过 `tauri-plugin-vcp-mobile` 的 `setKeepScreenOn` 实现 |
| **前台服务 (Foreground Service)** | Android 机制，`StreamKeepaliveService` 以 `START_STICKY` + `IMPORTANCE_HIGH` 通知保活，降低 OEM 杀后台概率 |
