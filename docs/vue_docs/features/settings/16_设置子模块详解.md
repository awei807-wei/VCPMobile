# 16. 设置子模块详解

## 1. 概述

### 1.1 领域定位

`features/settings/components/` 是 VCP Mobile 前端**全局设置中心**的原子 UI 单元集合。每个 Section 组件对应 SettingsView 中的一个功能子域，负责将用户输入转化为对 `settingsStore` 的写操作，或通过 Tauri IPC 触发 Rust 后端的维护/查询命令。

该目录的组件全部作为 **SettingsView 的子页面插槽**被引用，不独立存在于路由系统中。其设计遵循"高密度线性布局"美学宪法：列表重于卡片、灰度优先、2px 高亮侧边条反馈。

### 1.4 核心设计原则

1. **设置副本模式**：`SettingsView` 维护一份 `settings` 的本地响应式副本（`ref<AppSettings>`），子组件通过 `v-model` 绑定到副本字段。用户点击"返回"时才调用 `settingsStore.saveSettings()` 统一持久化，避免频繁写盘。
2. **操作即命令**：维护类操作（GC、缓存清理、重建）不修改设置副本，而是直接通过 `invoke` 调用 Rust Command，操作结果通过原子组件 `SettingsActionWithStatus` 回显。
3. **状态机隔离**：重建会话（`RebuildSessionView`）拥有独立的状态机（`idle | running | completed | error`），不耦合于 `settingsStore`，由 `useRebuildSessionStore` 统一管理。
4. **屏幕常亮守护**：所有可能长时间运行的维护操作（重建、GC）均通过 `withScreenKeep()` 包裹，防止 Android 息屏中断。

## 2. 核心设置模块

### 2.1 VcpCoreSettingsSection — VCP 核心配置

**职责一句话**：配置 VCP 后端 HTTP/HTTPS 接入点、WebSocket 日志通道及其鉴权密钥，并提供连接健康检查能力。

**源码位置**：`src/features/settings/components/VcpCoreSettingsSection.vue`

**包含的表单字段**：

| 字段 | 组件 | 绑定字段 | 类型 | 说明 |
|------|------|---------|------|------|
| VCP 服务器 URL | `SettingsTextField` | `settings.vcpServerUrl` | 普通文本 | HTTP/HTTPS 端点，如 `https://vcp-endpoint.com` |
| VCP API Key | `SettingsTextField` | `settings.vcpApiKey` | `isSecure` | 鉴权密钥，支持显隐切换 |
| VCP WebSocket URL | `SettingsTextField` | `settings.vcpLogUrl` | `mono` | 日志流地址，如 `ws://localhost:8024` |
| VCP WebSocket Key | `SettingsTextField` | `settings.vcpLogKey` | `isSecure` + `mono` | WebSocket 鉴权 |

**连接测试流程**：

1. 用户点击"验证连接"按钮。
2. 组件先 `emit('save-request')` 触发父级 `SettingsView.saveSettings()` 将当前输入持久化。
3. 调用 `invoke('test_vcp_connection', { vcpUrl, vcpApiKey })`。
4. Rust 侧对齐桌面端 `fetchAndCacheModels` 逻辑：解析 URL → 构造 `/v1/models` 请求 → 携带 `Bearer` Token → 10s 超时。
5. 返回 `{ success, status, modelCount, models }`，前端回显成功/失败状态。

**关键交互**：
- 测试前若 URL 为空，直接本地报错，不发起 IPC。
- 状态通过 `SettingsActionWithStatus` 组件渲染，支持 `loading`（"正在验证模型列表..."）、`success`（"连接成功！拉取到 N 个可用模型"）、`error`。

### 2.4 UserProfileSection — 用户资料

**职责一句话**：管理本地用户身份标识（头像、用户名）及管理员接口鉴权凭据。

**源码位置**：`src/features/settings/components/UserProfileSection.vue`

**包含的表单字段**：

| 字段 | 组件 | 绑定字段 | 说明 |
|------|------|---------|------|
| 头像 | `VcpAvatar` + `AvatarCropper` | — | 点击触发文件选择 → FileReader → Base64 → 裁剪器 → Blob |
| 用户名 | `SettingsTextField` | `settings.userName` | 本地显示名称 |

**头像上传时序**：

```
用户点击头像区域
    │
    ▼
触发隐藏 <input type="file" accept="image/*">
    │
    ▼
FileReader.readAsDataURL(file)
    │
    ▼
cropImg = base64  ──►  <AvatarCropper :img="cropImg" />
    │
    ▼
用户确认裁剪 ──► onCropConfirm(blob)
    │
    ▼
```

## 3. 系统维护模块

### 3.1 MaintenanceSection — 维护与清理

**职责一句话**：提供附件库垃圾回收、WebView 缓存清理、全量预渲染重建入口三项系统维护操作。

**源码位置**：`src/features/settings/components/MaintenanceSection.vue`

**包含的操作按钮**：

| 操作 | 按钮文案 | 对应 Rust Command | 屏幕常亮 |
|------|---------|------------------|----------|
| 附件库垃圾回收 (GC) | 立即清理 | `cleanup_orphaned_attachments` | ✅ `withScreenKeep` |
| 清理系统缓存 | 立即清理 | `clear_webview_cache` | ✅ `withScreenKeep` |
| 全量预渲染重建 | 一键重建 | —（打开 RebuildSessionView） | — |

**附件 GC 逻辑**（Rust 侧 `maintenance_manager.rs`）：

1. 扫描 `attachments` 表获取全部 `(hash, internal_path)`。
2. 查 `message_attachments` 表获取**被引用**的 hash 集合。
3. 遍历差集，物理删除未被引用的附件文件及其缩略图（`*_thumb.webp`）。
4. 从 `attachments` 表删除对应记录。
5. 返回：`"清理完成：删除了 N 个孤儿附件，释放了 X MB 空间"`。

**WebView 缓存清理**：
- 调用 Tauri v2 原生 API `webview.clear_all_browsing_data()`。
- 清除 HTTP Cache、图片缓存、LocalStorage 等浏览数据。

### 3.2 RebuildSessionView — 重建会话

**职责一句话**：全量预渲染重建的独占式执行页面，通过状态机管理 idle → running → completed/error 生命周期。

**源码位置**：`src/features/settings/components/RebuildSessionView.vue`

**状态机**：

```
┌─────┐   点击"开始重建"   ┌─────────┐
│idle │ ─────────────────►│ running │
└─────┘                   └────┬────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
              ▼                ▼                ▼
        ┌─────────┐      ┌──────────┐    ┌─────────┐
        │completed│      │  error   │    │running* │
        └────┬────┘      └────┬─────┘    └─────────┘
             │                │
             ▼                ▼
        提示刷新页面      显示错误信息
```

**核心交互**：

- **idle**：显示播放图标 + "开始重建"按钮。
- **running**：进度条（`current / total`）+ 底部浮动提示"重建进行中 — 请勿退出"+ 全局遮罩阻止误触 + `canDismiss = false`。
- **completed / error**：显示结果文案 + "完成"/"关闭"按钮，`canDismiss = true`。

**与 Rust 后端的交互**：

```
startRebuild()
    │
    ├──► invoke('plugin:vcp-mobile|start_streaming_service')  // 前台保活
    │
    ├──► withScreenKeep(() => invoke('rebuild_all_pre_renders'))
    │         └──► Rust: message_repository.rs
    │              └── 遍历全部消息 → AST 重新解析 → 代码高亮 → 更新 pre_render_html
    │
    └──► invoke('plugin:vcp-mobile|stop_streaming_service')   // 释放前台服务
```

**事件监听**：
- 监听 Tauri Event `render_rebuild_progress`，实时更新 `progress.current / progress.total`。

**完成后刷新**：若 `needsReload === true`，关闭时弹 `confirm` 询问是否立即刷新。确认后调用 `useDataReload().performFullReload()`，清理 AST HTML 缓存并重新加载消息历史。

## 5. 更新与关于模块

### 5.1 UpdateSection — OTA 更新检查

**职责一句话**：查询 GitHub Release 最新版本，管理下载进度与安装触发。

**源码位置**：`src/features/settings/components/UpdateSection.vue`

**状态机**（`UpdateStatus`）：

```
 idle ──► checking ──┬──► no-update ──► (4s 后自动回到 idle)
                     │
                     └──► update-available ──► downloading ──┬──► idle (安装成功)
                                                               │
                                                               └──► error ──► idle
```

**核心流程**：

1. **检查更新**：`invoke('check_for_update')`
   - Rust 请求 `api.github.com/repos/MRiecy/VCPMobile/releases/latest`。
   - 若 `/latest` 404（最新为 prerelease），降级到 `/releases?per_page=1`。
   - 匹配 `*arm64-v8a.apk` 资源，返回 `UpdateInfo`。

2. **下载并安装**：`useUpdateDownloader().downloadAndInstall(url)`
   - 创建 Tauri `Channel<DownloadProgress>`，通过回调实时更新进度。
   - Rust `download_update` 下载 APK 到缓存目录，流式写入。
   - 完成后调用 `install_update`，触发 Android `Intent.ACTION_VIEW` 安装 APK。

3. **降级策略**：若本地安装器失败，回退到 `openUrl(info.downloadUrl)` 用浏览器下载。

**UI 反馈**：

| 状态 | UI 表现 |
|------|---------|

## 9. 设计决策与注意事项

1. **设置副本 vs 直接绑定 Store**：
   `SettingsView` 不直接让子组件 `v-model` 绑定 `settingsStore.settings`，而是维护本地 `ref` 副本。原因：避免用户在子页面输入时触发频繁持久化，只在点击"返回"时统一 `saveSettings()`。

2. **AiLogicSettingsSection 的占位策略**：
   该组件在 `v0.9.14` 中仅渲染两个"暂未启用"行。Agent 级别的 Temperature / Max Tokens 配置目前仍由桌面端 `config.env` 控制，移动端尚未实现全局覆盖逻辑。保留该组件是为了 UI 架构的完整性，避免未来新增字段时破坏布局。

3. **RebuildSessionView 的独占式设计**：
   重建操作使用独立的 `SlidePage` 页面栈项（而非内联在 `MaintenanceSection` 中），是因为重建过程需要阻止用户离开（`canDismiss = false` + 全局遮罩），且完成后可能触发全量数据重载，需要占据用户全部注意力。

4. **AboutSection 的深色/亮色深度覆盖**：
   关于页面使用 `WebGLFluidBackground` 作为全屏动态背景，其颜色与主题无关。因此该组件通过 `<style scoped>` 中的 `:deep()` 选择器，对子组件（如 `SettingsCard`、`SettingsRow`）进行深度样式覆盖，确保在亮色模式下文字可读、卡片有半透明毛玻璃质感。

5. **SyncLogBrowserCore 的亮色缺失**：
   该组件当前仅针对深色场景设计（固定 `text-white/70` 等色值）。若未来需要在亮色设置页中复用，需补充 CSS 变量适配。

6. **UpdateSection 的下载降级链**：
   `downloadAndInstall` 并非直接返回失败，而是在本地安装器失败后尝试浏览器直链下载，最终再尝试打开 Release 页面。该三级降级确保用户在任何网络/权限环境下都能获取更新包。

7. **头像保存的流式处理**：
   `UserProfileSection` 将 `Blob` 转为 `ArrayBuffer` 再转为 `Uint8Array`，最终通过 `Array.from(bytes)` 序列化为普通数组后 IPC 传递。这是为了绕过 Tauri v2 对 `Uint8Array` 直接序列化的限制。

## 10. 术语速查表

| 术语 | 说明 |
|------|------|
| **Section** | 设置子模块组件的统称，每个 Section 对应一个功能域 |
| **SettingsView** | 设置中心的根组件，以 SlidePage 形式从右侧滑入 |
| **预渲染 (Pre-render)** | 将消息原始内容提前解析为带语法高亮的 HTML，存入数据库避免运行时重复解析 |
| **GC (Garbage Collection)** | 此处特指附件库垃圾回收，删除未被消息引用的孤立文件 |
| **Screen Keep** | 屏幕常亮锁定，通过 `tauri-plugin-vcp-mobile` 的 `setKeepScreenOn` 实现 |
| **前台服务 (Foreground Service)** | Android 机制，`StreamKeepaliveService` 以高优先级通知保活，防止 OEM 杀后台 |
| **Channel** | Tauri v2 的双向通信通道，用于流式下载进度推送 |
| **OEM 电池优化** | 国内定制 Android 系统（ColorOS/MIUI/HarmonyOS 等）的后台进程冻结策略 |
| **AST** | Abstract Syntax Tree，消息内容结构化解析的中间表示 |
| **副本模式** | SettingsView 本地维护 `settings` 副本，返回时才持久化的设计模式 |
