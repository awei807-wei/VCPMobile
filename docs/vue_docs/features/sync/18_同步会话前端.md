# 18. 同步会话前端

## 1. 概述

### 1.1 领域定位

`sync/` 前端领域负责将 Rust 后端同步 V2 协议的运行状态以可视化方式呈现给用户，并提供配置入口与历史日志回溯能力。它是用户与"三阶段增量同步协议"交互的唯一 UI 界面，承担以下职责：

- **会话管理**：控制同步面板的打开/关闭、状态机流转、视图切换（实时/历史）
- **状态可视化**：WebSocket 连接状态灯、同步阶段指示器、进度条、终端式日志流
- **安全拦截**：同步启动前的电量与省电模式检测，防止低电量或省电状态下的大批量 IO
- **配置入口**：同步服务器地址、Token、预渲染开关的设置面板集成
- **日志浏览**：本地同步日志文件的分页浏览、关键字高亮、定期清理

该领域**不**涉及：
- 同步协议本身的哈希计算、差异比对、冲突仲裁（由 Rust `sync_service.rs` 负责）
- 网络层 WebSocket/HTTP 连接的物理维护（由 Rust `vcp_client.rs` 与 `sync_service.rs` 负责）
- 数据库写入与事务管理（由 Rust `db_manager.rs` 与 `db_write_queue.rs` 负责）

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **状态集中管理** | 所有同步相关运行时状态（连接状态、进度、日志）收敛于 `syncSessionStore`，视图组件纯渲染，不持有业务逻辑 |
| **事件驱动更新** | 后端通过 Tauri Event 通道向前端推送状态变更，前端无需轮询；`syncSessionStore` 负责事件的注册、过滤与清理 |
| **覆盖层统一管控** | `SyncSessionView` 纳入 `overlayStore.pageStack` 管理，复用 SlidePage 动画、Z-Index 语义化层级、返回键拦截 |
| **电量安全拦截** | 同步启动前强制检测电池电量与省电模式，低于 30% 或处于省电模式时智能拦截，避免数据损坏 |
| **屏幕常亮保障** | 同步期间通过 `useScreenKeeper` 引用计数锁保持屏幕常亮，同时启动 Android 前台服务降低后台被杀概率 |
| **日志窗口截断** | 内存中保留最近 200 条日志，DOM 仅渲染最近 100 条，防止长同步会话导致内存与渲染膨胀 |

## 2. 同步会话视图（SyncSessionView.vue）

### 2.1 打开方式（SlidePage）

`SyncSessionView` 是一个全屏覆盖层，通过 `SlidePage` 组件实现从右侧滑入的页面动画。

> 文件位置：`src/features/sync/SyncSessionView.vue`

**打开路径**：

```
用户点击"全量神经同步"按钮（SyncSettingsSection.vue）
           │
           ▼
   overlayStore.openSyncSession()
           │
           ▼
   ┌─────────────────────┐
   │ syncSessionStore    │
   │   .open()           │──> 重置状态、注册监听器
   └──────────┬──────────┘
              │
              ▼
   pageStack.push({ type: 'syncSession', ... })
              │
              ▼
   FeatureOverlays.vue 中 v-if="isSyncSessionOpen"
   渲染 <SyncSessionView :z-index="getPageZIndex('syncSession')" />
              │
              ▼
   SlidePage 播放滑入动画，页面呈现
```

**关闭路径**：

```
用户点击 X 或系统返回键
           │
           ▼
   overlayStore.closeSyncSession()
           │
           ▼
   syncSessionStore.close() ──> 清理监听器、释放屏幕锁、调用 stop_sync
           │
              ▼
   pageStack.pop()，SlidePage 播放滑出动画
```

**层级计算**：`SyncSessionView` 的 `z-index` 由 `overlayStore.getPageZIndex('syncSession')` 动态计算。`overlayStore` 维护 `pageStack` 数组，每个页面类型映射到基础层级 `LAYER_PAGE_BASE`（40）加上在栈中的索引偏移。当多个 SlidePage 同时打开时（如先打开设置页再打开同步面板），此机制确保后打开的页面层级更高，视觉覆盖正确。

> 详见 `docs/UI_LAYER_ARCHITECTURE.md` 中 "SlidePage 页面栈" 层级规范。

**按需加载**：`SyncSessionView` 在 `FeatureOverlays.vue` 中使用 `defineAsyncComponent` 按需异步加载：

```ts
const SyncSessionView = defineAsyncComponent(
  () => import('../features/sync/SyncSessionView.vue')
);
```

原因：同步会话属于低频操作，应用生命周期内大多数时间不会打开。异步加载将其 chunk 从主 bundle 中分离，降低应用冷启动时的 JS 解析与执行开销。

### 2.2 组件结构

`SyncSessionView` 通过 `SlidePage` 组件包裹内容。`SlidePage` 本身是一个极简的容器：

> 文件位置：`src/components/ui/SlidePage.vue`

```vue
<Transition name="slide-page">
  <div v-show="props.isOpen" class="fixed inset-0 pointer-events-auto"
       :style="{ zIndex: props.zIndex }">
    <slot />
  </div>
</Transition>
```

`SlidePage` 使用 Vue `<Transition>` 实现滑入/滑出动画：
- **进入**：`transform: translateX(100%) → 0`，持续时间 `0.35s`，缓动曲线 `cubic-bezier(0.32, 0.72, 0, 1)`
- **离开**：`transform: translateX(0) → 100%`，持续时间 `0.3s`，相同缓动曲线

这种曲线在移动端产生"弹性减速"的物理质感，与系统原生转场动画视觉对齐。

## 3. 同步状态（syncSessionStore）

### 3.1 核心状态字段

> 文件位置：`src/core/stores/syncSession.ts`

`syncSessionStore` 使用 Pinia Composition API 风格定义（`defineStore('syncSession', () => { ... })`），全部状态为 `ref`，共 7 个核心字段：

| 字段 | 类型 | 初始值 | 说明 |
|------|------|--------|------|
| `isOpen` | `boolean` | `false` | 面板是否打开，驱动 SlidePage 显隐 |
| `canDismiss` | `boolean` | `true` | 是否允许关闭（同步进行中禁止关闭） |
| `status` | `'idle' \| 'connecting' \| 'connected' \| 'error' \| 'completed'` | `'idle'` | 连接状态机当前值 |
| `activeTab` | `'live' \| 'history'` | `'live'` | 当前激活的 Tab |
| `needsReload` | `boolean` | `false` | 同步完成后数据已更新，需刷新标志 |
| `logs` | `Array<{id, level, message, time}>` | `[]` | 实时日志缓冲区 |
| `progressData` | `{phase, total, completed, message}` | `{phase:'initialization',total:0,completed:0,message:''}` | 当前进度 |

**日志缓冲区管理算法**：

```ts
const pushLog = (level: string, message: string) => {
  const id = `${Date.now()}_${Math.random().toString(36).substring(2, 9)}`;
  logs.value.push({ id, level, message, time: new Date().toLocaleTimeString() });
  if (logs.value.length > 200) logs.value.shift();
};
```

- **ID 生成**：时间戳 + 随机字符串，确保同一毫秒内多条日志的 `key` 唯一性（Vue `v-for` 需要）
- **FIFO 截断**：当缓冲区超过 200 条时，从头部移除最旧的日志
- **时间格式**：`toLocaleTimeString()` 使用系统本地时区，显示为 `HH:MM:SS`

**监听器生命周期管理**：

```ts
let unlistenFns: UnlistenFn[] = [];

const registerListeners = () => {
  cleanupListeners();
  listen('vcp-log', ...).then(fn => unlistenFns.push(fn));
  listen('vcp-sync-progress', ...).then(fn => unlistenFns.push(fn));
  listen('vcp-sync-status', ...).then(fn => unlistenFns.push(fn));
  listen('vcp-sync-completed', ...).then(fn => unlistenFns.push(fn));
};

const cleanupListeners = () => {
  unlistenFns.forEach(fn => fn());
  unlistenFns = [];
};
```

`registerListeners` 在 `open()` 时调用，`cleanupListeners` 在 `close()` 时调用。每次打开面板前强制清理旧监听器，防止重复注册导致事件回调多次触发。

### 3.2 连接生命周期（未连接/连接中/已连接/断开/错误）

```
                    ┌─────────────┐
         ┌─────────►│    idle     │◄─────────────┐
         │          │   (初始)     │              │
         │          └──────┬──────┘              │
         │                 │ 用户点击"开始同步"   │
         │                 ▼                     │
         │          ┌─────────────┐              │
         │          │ connecting  │              │
         │          │   (连接中)   │              │
         │          └──────┬──────┘              │
         │                 │ vcp-sync-status:    │
         │                 │   status = "open"   │
         │                 ▼                     │
         │          ┌─────────────┐              │
         │          │  connected  │              │
         │          │   (同步中)   │              │
         │          └──────┬──────┘              │
         │                 │ vcp-sync-completed  │
         │                 │ 或 vcp-sync-status: │
         │                 │   status = "error"  │
         │                 ▼                     │
         │     ┌─────────────────────────┐       │
         └─────┤       completed         │       │
               │       (已完成)           ├───────┘
               └─────────────────────────┘
                          │
                          │ vcp-sync-status:
                          │   status = "error"
                          ▼
                    ┌─────────────┐
                    │    error    │
                    │   (错误)     │
                    └─────────────┘
```

**状态机转换规则**：

| 触发条件 | 源状态 | 目标状态 | 副作用 |
|----------|--------|---------|--------|
| 用户点击"开始同步" | `idle` | `connecting` | 清空日志、获取屏幕锁、启动前台服务、调用 `start_manual_sync` |
| 收到 `vcp-sync-status` {status:"open"} | `connecting` | `connected` | `canDismiss = false`（禁止关闭） |
| 收到 `vcp-sync-completed` | `connected` | `completed` | `canDismiss = true`、`needsReload = true`、释放屏幕锁、停止前台服务 |
| 收到 `vcp-sync-status` {status:"error"} | `connecting` / `connected` | `error` | `canDismiss = true`、释放屏幕锁、停止前台服务 |
| `start_manual_sync` 调用异常 | `connecting` | `error` | `canDismiss = true`、释放屏幕锁、停止前台服务、打印错误日志 |
| 电量/省电模式拦截 | `connecting` | `error` | `canDismiss = true`、打印拦截原因日志 |

**自动重连策略**：当前实现中**无自动重连机制**。连接断开后，用户需手动关闭面板并重新点击"开始同步"。这是有意为之的设计决策：同步是一次性的全量/增量比对任务，而非长连接实时推送；断连后需要用户确认网络环境再重试，避免无效循环消耗电量。

### 3.3 同步进度与阶段

进度数据通过监听 `vcp-sync-progress` 事件更新：

```ts
listen('vcp-sync-progress', (event: any) => {
  progressData.value = event.payload;
}).then(fn => unlistenFns.push(fn));
```

`progressData` 直接绑定到 `SyncSessionView` 的进度条与阶段标签，无需额外转换（仅 `phase` 字符串进行中文化映射）。

进度百分比计算（前端 `computed`）：
```ts
const progressPercent = computed(() => {
  if (store.progressData.total <= 0) return 0;
  return Math.min(100, Math.round(
    (store.progressData.completed / store.progressData.total) * 100
  ));
});
```

### 3.4 冲突列表管理

当前前端实现中**无独立的冲突列表 UI**。同步 V2 协议的冲突仲裁由后端通过 LWW（Last-Write-Wins）策略自动完成，前端通过以下方式感知冲突结果：

1. **日志流**：后端将冲突解决过程以 `vcp-log` 事件推送到前端日志终端（如 `冲突解决：移动端版本较新，保留本地`）
2. **错误状态**：若冲突涉及无法自动仲裁的边界情况（如双端同时软删除后重新创建），后端将状态置为 `error` 并通过日志输出详细原因
3. **数据刷新**：同步完成后 `needsReload = true`，用户关闭面板时触发 `performFullReload()`，前端所有缓存重新加载，冲突结果已体现在新数据中

> 如需扩展独立冲突列表，需在 Rust `sync_service.rs` 中新增冲突详情事件，并在 `syncSessionStore` 中维护 `conflicts` 数组。详见 `docs/sync/11_冲突解决与边界情况.md`。

### 5.2 日志列表与筛选

**文件列表数据来源**：

```ts

  files.value = await invoke<LogFile[]>('list_sync_log_files');
};
```

返回的 `LogFile` 结构：

| 字段 | 类型 | 说明 |
|------|------|------|
| `filename` | `string` | 日志文件名（如 `sync_20260527_143022.log`） |
| `created_at` | `number` | Unix 时间戳（秒级） |
| `size_bytes` | `number` | 文件大小 |

列表项展示：`filename`（等宽字体截断）+ `created_at`（格式化）+ `size_bytes`（自动转换 KB/MB）。

**旧日志清理**：

```ts
const clearOldLogs = async () => {
  const removed = await invoke<number>('clear_old_sync_logs', { keepDays: 7 });
};
```

- 清理 7 天前的日志文件
- 清理后自动刷新文件列表
- 通过 `confirm()` 二次确认，防止误触

### 5.3 日志详情与高亮

**内容阅读态**：

1. **加载**：调用 `read_sync_log_file(filename)` 获取完整文本内容
2. **预处理**：按 `\n` 分割并过滤空行
   ```ts
   lines.value = content.split('\n').filter(l => l.trim());
   ```
3. **分页**：每 500 行为一页，计算 `totalPages`
   ```ts
   totalPages.value = Math.max(1, Math.ceil(lines.value.length / linesPerPage));
   ```
4. **可见行计算**：
   ```ts
   const visibleLines = () => {
     const start = currentPage.value * linesPerPage;
     return lines.value.slice(start, start + linesPerPage);
   };
   ```
5. **高亮规则**（基于行内容字符串匹配）：

| 匹配条件 | 高亮颜色 | 含义 |
|----------|---------|------|
| 包含 `[INFO]` 且包含 `success` | `text-green-400` | 成功信息 |
| 包含 `[Error]` / `failed` / `error` | `text-red-400` | 错误信息 |
| 包含 `WARN` | `text-yellow-400` | 警告信息 |
| 默认 | `text-white/70` | 普通日志 |

4. **操作**：支持整页复制到剪贴板（`navigator.clipboard.writeText`）

> 分页设计理由：同步日志文件在首次全量同步后可能达到数 MB（数万行），一次性渲染会导致 DOM 阻塞与内存峰值。500 行/页在移动端屏幕上约为 20–30 屏滚动量，是性能与阅读连续性的平衡点。

### 7.1 Tauri Commands 调用表

| 命令 | 调用方 | 输入 | 输出 | 触发时机 |
|------|--------|------|------|---------|
| `start_manual_sync` | `syncSessionStore.startSync()` | — | `void` / `Err(String)` | 用户点击"开始同步"，电量检测通过后调用 |
| `stop_sync` | `syncSessionStore.close()` | — | `void` | 用户关闭面板或同步异常时调用，终止当前同步会话 |
| `list_sync_log_files` | `SyncLogBrowserCore.loadFiles()` | — | `Array<{filename, created_at, size_bytes}>` | 历史 Tab 加载或清理后刷新时调用 |

| `get_sync_status` | —（当前前端未直接调用） | — | `SyncStatus` | 预留命令，可用于面板初始化时查询后端是否已有活跃会话 |

### 7.2 WebSocket 事件监听表

`syncSessionStore` 在 `open()` 时通过 `listen()` 注册 4 个 Tauri Event 监听器，在 `close()` 时统一清理（`cleanupListeners()`）。

| 事件名 | 来源（Rust 模块） | Payload 结构 | 前端处理逻辑 | 清理时机 |
|--------|-----------------|-------------|-------------|---------|
| `vcp-log` | `sync_service.rs`（`emit_sync_log`） | `{id, level, category, message, timestamp}` | 过滤 `category === 'sync'`，调用 `pushLog(level, message)` | `close()` |
| `vcp-sync-progress` | `sync_service.rs` / `sync_executor.rs` | `{phase, total, completed, message}` | 直接赋值给 `progressData.value` | `close()` |
| `vcp-sync-status` | `sync_service.rs`（连接状态变化） | `{status, message}` | `status="open"` → `connected`；`status="error"` → `error` | `close()` |
| `vcp-sync-completed` | `sync_service.rs`（会话完成） | `{source}` | `status="completed"`、`canDismiss=true`、`needsReload=true`、释放屏幕锁、停止前台服务 | `close()` |

> 注意：Rust 后端同时发射 `vcp-system-event`（统一系统事件通道），但 `syncSessionStore` 直接监听扁平化的 `vcp-sync-status` 与 `vcp-sync-completed`，避免在 Store 中解析嵌套事件结构，简化逻辑。

### 7.3 HTTP 端点调用

前端**不直接调用**任何同步相关的 HTTP 端点。所有与桌面端的 HTTP 通信（实体批量下载、消息 NDJSON 流式传输、附件下载）均由 Rust 后端代理完成。前端仅通过 Tauri Commands 与 Events 与后端交互。

这种设计的优势：
- **安全**：Token 与服务器地址仅存于 Rust 层，不暴露给前端 JavaScript
- **一致性**：网络重试、超时、并发控制由 Rust 统一处理（`NetworkAwareSemaphore`）
- **离线缓冲**：Rust 层可在连接恢复后自动重试，前端无需关心网络瞬断

> Rust 后端调用的 HTTP 端点完整参考详见 `docs/sync/附录B_HTTP端点完整参考.md`。

## 9. 设计决策与注意事项

### 9.1 为何使用 SlidePage 而非路由跳转？

同步会话是**临时性覆盖任务**，而非持久页面。使用 `SlidePage`（基于 `fixed inset-0` 的覆盖层）而非 Vue Router 导航的原因：

1. **上下文保持**：用户可在同步完成后立即回到之前的聊天或设置页面，无需重建页面状态
2. **动画一致性**：复用项目统一的 SlidePage 滑入/滑出动画体系，与 AgentSettings、GroupSettings 等行为一致
3. **层级管理**：通过 `overlayStore.pageStack` 统一管理 Z-Index 与返回键拦截，避免与底层页面的手势冲突

### 9.2 为何电量检测在前端而非后端？

电量检测逻辑位于 `syncSessionStore.startSync()` 中，而非 Rust `start_manual_sync` 内部：

1. **即时反馈**：前端可在用户点击后立即拦截并展示红色错误日志，无需等待 IPC 往返
2. **权限解耦**：电池状态通过 `plugin:vcp-mobile` 插件获取，前端直接调用插件命令比后端转发减少一次 IPC 层级
3. **UI 锁定**：拦截后前端立即将 `status` 置为 `error` 并解锁关闭按钮，避免用户困惑于"点击后无响应"

### 9.3 日志缓冲区的 200/100 截断策略

`syncSessionStore.logs` 内存中保留最近 200 条，但 `SyncSessionView` 的 DOM 仅渲染最近 100 条：

- **200 条内存保留**：支持用户点击"复制日志"时获取较完整的上下文，便于问题排查
- **100 条 DOM 渲染**：防止极长同步会话（如首次全量同步可能产生数千条日志）导致 Vue 响应式数组与 DOM 节点膨胀，避免低端 Android 设备卡顿或 OOM
- **折叠提示**：当内存中日志超过 100 条时，DOM 底部显示 `... N 条更早的日志已折叠（内存中保留最近 200 条）`，告知用户数据未丢失

### 9.4 同步进行中为何禁止切换 Tab？

`SyncSessionView` 中，当 `isSyncing`（即 `status === 'connected'`）时，"历史"Tab 按钮被 `disabled`：

- 同步进行中日志流是实时追加的，切换 Tab 会中断用户的实时观察
- 防止用户滑动到历史视图后，因 `watch(activeTab)` 驱动滚动而产生的意外视图跳动
- 完成/错误后自动解锁，允许用户自由查看历史日志

### 9.5 前台服务与屏幕常亮的双重保活

同步期间同时启用两种 Android 保活机制：

| 机制 | 前端控制 | 作用 | 失败策略 |
|------|---------|------|---------|
| 屏幕常亮 | `acquireScreenKeep()` / `releaseScreenKeep()` | 防止设备休眠导致 WebSocket 断开 | 失败仅 `catch(() => {})`，不阻断同步 |
| 前台服务 | `start_streaming_service` / `stop_streaming_service` | 显示高优先级通知，降低 OEM 杀后台概率 | 失败仅 `catch(() => {})`，不阻断同步 |

两者均采用**宽容策略**：调用失败仅静默忽略。因为它们是稳定性增强手段，而非同步功能的必要依赖。这种设计避免了在某些 Android 设备（如权限受限的 OEM 系统）上因保活失败导致同步无法启动的问题。

### 9.6 屏幕常亮锁的引用计数设计

`useScreenKeeper.ts` 使用引用计数管理屏幕常亮状态：

```ts
let refCount = 0;

export function acquireScreenKeep(): void {
  if (refCount === 0) {
    setKeepScreenOn().catch(() => {});
  }
  refCount++;
}

export function releaseScreenKeep(): void {
  if (refCount > 0) {
    refCount--;
    if (refCount === 0) {
      clearKeepScreenOn().catch(() => {});
    }
  }
}
```

此设计允许多个并发操作（如同步 + 流式聊天）同时请求屏幕常亮，只有最后一个操作释放时才真正清除标志。同步会话的 `acquireScreenKeep()` 与 `releaseScreenKeep()` 配对使用，确保不会干扰其他需要常亮的功能。

### 9.7 同步完成后为何需要 `performFullReload`？

同步完成后，Rust 后端已修改 SQLite 数据库中的 Agents、Groups、Topics、Messages 等表，但前端存在多层缓存：

1. **AST HTML 缓存**：`clearHtmlCache()` 清理消息内容的预渲染 HTML，确保重新渲染时基于最新数据
2. **Agent/Group 缓存**：`assistantStore.fetchAgents()` / `fetchGroups()` 重新从后端读取配置
3. **Topic 缓存**：`topicStore.invalidateAllTopicCaches()` 使话题列表缓存失效
4. **消息数组**：若当前在活跃话题中，`chatHistoryStore.loadHistoryPaginated()` 重新加载消息

`needsReload` 标志确保用户不会在同步中途因 UI 数据陈旧而产生困惑。关闭面板时的 `confirm()` 对话框提醒用户"数据已更新"，确认后执行全量刷新。

### 9.8 Tab 切换的双向绑定与防循环

`SyncSessionView` 的 Tab 切换采用"滚动驱动 Tab"与"Tab 驱动滚动"的双向绑定模式：

**Tab 驱动滚动**（`watch`）：
```ts
watch(() => store.activeTab, (newTab) => {
  if (!scrollContainer.value || isUserScrolling.value) return;
  const targetX = newTab === 'live' ? 0 : scrollContainer.value.clientWidth;
  scrollContainer.value.scrollTo({ left: targetX, behavior: 'smooth' });
});
```

**滚动驱动 Tab**（`@scroll`）：
```ts
const handleScroll = (e: Event) => {
  isUserScrolling.value = true;
  const scrollLeft = (e.target as HTMLElement).scrollLeft;
  const activeTab = scrollLeft < width / 2 ? 'live' : 'history';
  if (store.activeTab !== activeTab) store.switchTab(activeTab);
  setTimeout(() => { isUserScrolling.value = false; }, 100);
};
```

`isUserScrolling` 标志通过 100ms 延时重置，防止 `watch` 与 `@scroll` 形成无限循环：用户滑动 → 更新 `activeTab` → `watch` 触发滚动 → 触发 `@scroll` → 再次更新 `activeTab`... 标志位在此打断循环。
