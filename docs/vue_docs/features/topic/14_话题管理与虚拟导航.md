# 14. 话题管理与虚拟导航

## 1. 概述

### 1.1 领域定位

`features/topic/` 是 VCP Mobile 前端**话题生命周期管理**与**虚拟导航**的集中承载域。该领域负责：

- **话题 CRUD**：创建、读取、更新标题、切换锁定/未读状态、删除话题
- **话题列表渲染**：基于虚拟列表的高性能长列表展示，支持搜索过滤与状态角标
- **话题切换编排**：协调 `chatSessionStore`、`chatHistoryStore`、`chatStreamStore` 完成无闪烁的话题上下文迁移
- **虚拟页面栈（SlidePage）**：在单页 Hash 路由架构下，通过覆盖层堆叠模拟多页导航体验，并集成物理返回键拦截

该领域**不**涉及：
- 消息流式传输的协议解析（由 `chatStreamStore` 负责）
- 消息内容的结构化渲染（由 `MessageRenderer` 负责）
- 消息持久化的数据库操作（由 Rust `message_service.rs` 负责）

### 1.4 核心设计原则

| 原则 | 说明 |
|------|------|
| **流式加载优先** | 话题列表与历史记录均通过 `Channel` 流式获取，避免大列表阻塞主线程 |
| **乐观更新** | 创建/删除/锁定/未读状态在前端立即生效，后台命令异步同步 |
| **会话隔离** | `chatStreamStore` 按 `"itemId:topicId"` 隔离活跃流，切换话题天然不污染 |
| **虚拟导航** | 单路由 `/chat` 不变，所有"页面切换"通过 `SlidePage` 覆盖层堆叠实现 |
| **返回键消费** | Operation Aegis 保证返回手势先关闭 Modal → 再关闭 Page → 再退出应用 |

### 2.2 数据源与过滤

底层 `filteredTopics` 位于 `topicListManager.ts` 第 54–77 行：

```ts
const filteredTopics = computed(() => {
  const term = searchTerm.value.toLowerCase().trim();
  if (!term) return topics.value;
  return topics.value.filter((topic) => {
```

## 4. 虚拟页面栈（SlidePage）

VCP Mobile 采用 **Hash 路由 + SlidePage 覆盖层** 的混合导航架构。绝大多数用户感知的"页面切换"并非真实路由跳转，而是固定 DOM 的覆盖层堆叠。

### 4.1 与 Vue Router 的关系

```
真实路由（Vue Router Hash 模式）
    │
    ├── /         → redirect → /chat
    └── /chat     → ChatView.vue（唯一路由页）
    
虚拟导航（SlidePage 覆盖层）
    │
    ├── SettingsView        → pushPage('settings')
    ├── AgentSettingsView   → pushPage('agentSettings', id)
    ├── GroupSettingsView   → pushPage('groupSettings', id)
    ├── SyncSessionView     → openSyncSession()
    └── RebuildSessionView  → openRebuildSession()
```

**本质区别**：
- Vue Router 负责应用级别的**根上下文切换**（目前仅 `/chat`）
- SlidePage 负责**功能视图的层叠展示**，URL 始终为 `/#/chat`，不产生路由历史

### 4.2 页面打开/关闭机制

`overlayStore` 维护一个 `pageStack` 数组，记录当前打开的虚拟页面：

> 源码位置：`src/core/stores/overlay.ts` 第 22–66 行

```ts
interface PageStackItem {
  type: string;
  id?: string;
  modalId: string;
}

const pageStack = ref<PageStackItem[]>([]);
```

**打开页面**（`pushPage`）：

```ts
const pushPage = (type: string, id?: string) => {
  const modalId = `Page:${type}:${id || ''}`;
  const top = pageStack.value[pageStack.value.length - 1];
  if (top && top.type === type && top.id === id) return; // 去重

  pageStack.value.push({ type, id, modalId });
  registerModal(modalId, () => { popPageInternal(); });
};
```

**关闭页面**：
- **UI 关闭按钮** → `popPage()` → 同步移除栈顶 + `unregisterModal`
- **返回键关闭** → `popPageInternal()` → 由 `handlePopState` 触发回调

> 源码位置：`src/core/stores/overlay.ts` 第 49–82 行

**常驻 DOM 策略**：`SettingsView`、`AgentSettingsView`、`GroupSettingsView` 等使用 `v-show`（或 `:is-open` 控制内部 `SlidePage` 的 `v-show`），保持常驻 DOM 以保留表单草稿和确保 leave 动画完整。`SyncSessionView` 和 `RebuildSessionView` 使用 `v-if` 按需挂载（状态完全由 Store 托管）。

> 参考：`src/components/FeatureOverlays.vue` 第 36–69 行

### 4.3 动态 Z-Index 计算

SlidePage 的层级通过 `getPageZIndex` 动态计算，保证后打开的页面始终压在前一页之上：

> 源码位置：`src/core/stores/overlay.ts` 第 43–47 行

```ts
const getPageZIndex = (type: string) => {
  const index = pageStack.value.findIndex(p => p.type === type);
  if (index === -1) return LAYER_PAGE_BASE;
  return LAYER_PAGE_BASE + Math.min(index, LAYER_PAGE_MAX_OFFSET);
};
```

层级常量定义：

| 语义名 | 数值 | 用途 |
|--------|------|------|
| `LAYER_PAGE_BASE` | 40 | 页面栈基准层 |
| `LAYER_PAGE_MAX_OFFSET` | 9 | 最多支持 10 层堆叠（40–49） |

实际计算示例：

```
打开 SettingsView        → z-index: 40  (base + 0)
再打开 AgentSettingsView  → z-index: 41  (base + 1)
再打开 SyncSessionView    → z-index: 42  (base + 2)
```

> 源码位置：`src/core/constants/layers.ts` 第 25–35 行

### 4.4 模态历史栈（Operation Aegis）集成

Operation Aegis 是 VCP Mobile 的**全局模态历史栈**，解决移动端物理返回键的逐层关闭问题。

**核心机制**：

```
用户按物理返回键 / 手势返回
    │
    ▼
┌─────────────────────┐
│ popstate 事件触发   │
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 当前路径非 /chat ?   │──> 是 → 交给 Vue Router 处理（真实路由页）
│                     │──> 否 → 进入模态拦截逻辑
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ modalStack 有内容？  │──> 是 → 关闭栈顶 Modal（LIFO）
│                     │──> 否 → 触发 vcp-exit-requested（应用退出）
└─────────────────────┘
```

**`registerModal` 与 `pushState` 绑定**：

当 `overlayStore.pushPage()` 调用 `registerModal(modalId, closeCallback)` 时，Aegis 会执行：

```ts
window.history.pushState({ vcpRoot: true, vcpModalId: id }, '');
modalStack.value.push({ id, close: closeHandler });
```

这确保每一次打开虚拟页面都会在浏览器历史中压入一个 dummy state。用户按返回键时，浏览器先 pop 这个 dummy state，Aegis 拦截到 `popstate` 事件后调用对应的 `close` 回调，实现"按返回关闭页面"的原生体验。

> 源码位置：`src/core/composables/useModalHistory.ts` 第 110–119 行

### 4.5 物理返回键处理

App.vue 中注册了多级返回消费策略：

> 源码位置：`src/App.vue` 第 91–148 行

```
返回键触发
    │
    ▼
1. closeTopModal() ──> 关闭 SlidePage / Dialog / ContextMenu / Prompt
    │                    （Operation Aegis 的 LIFO 消费）
    ▼
2. 若当前在聊天中（READY 且已选中项）
   ──> 清空 currentSelectedItem + currentTopicId，退回零数据引导页
    │
    ▼
3. 已在引导页 → 双击退出到后台（move_task_to_back）
```

**路由后置守护**：每次路由切换后自动校准 Aegis 的 dummy state：

```ts
router.afterEach(() => {
  initRootHistory();
});
```

> 源码位置：`src/App.vue` 第 212–214 行

这保证了从其他真实路由页（如未来可能扩展的 `/settings`）返回 `/chat` 时，dummy state 始终处于正确位置。

## 5. 话题切换数据流

### 5.1 切换话题的完整时序

```
用户点击话题列表中的某一项
    │
    ▼
TopicList.vue: selectTopic(itemId, topicId, topicName)
    │
    ├── 1. 路由兜底
    │      if (router.currentRoute.value.path !== "/chat")
    │          await router.push("/chat");
    │
    ├── 2. 更新会话状态
    │      await sessionStore.selectTopicById(itemId, topicId);
    │      ├── currentTopicId = topicId        (实时响应，列表高亮)
    │      ├── lastActiveTopicMap[itemId] = topicId  (Pinia 持久化)
    │      ├── 设置 currentSelectedItem（头像/色调同步）
    │      └── 若有 loadHistoryCallback，则异步调用
    │
    ├── 3. 顶部栏显示话题标题
    │      sessionStore.currentSelectedItem.name = topicName;
    │
    ├── 4. 关闭侧边栏
    │      layoutStore.setLeftDrawer(false);
    │
    └── 5. emit("select-topic") 通知父组件
    │
    ▼
ChatView.vue: watcher 响应 [currentTopicId, currentSelectedItem]
    │
    ├── showScrollToBottom = false
    ├── resetChatScroll()          (重置滚动状态)
    ├── topicStore.markTopicAsRead(newTopicId)  (清空未读标记)
    └── historyStore.loadHistoryPaginated(itemId, type, topicId)
         │
         ├── historyOffset = 0      (重置分页)
         ├── hasMoreHistory = true
         └── loadHistory(..., limit=5, offset=0)
              │
              ├── currentChatHistory = []   (清空旧历史)
              ├── invoke("load_chat_history_streamed", { ..., onMessage: channel })
              │      └── Rust 侧通过 Channel 逐条推送消息
              ├── channel.onmessage
              │      ├── 会话一致性校验（中途切话题则丢弃）
              │      ├── Object Hydration（从 streamStore 取活跃消息对象）
              │      └── currentChatHistory.push(msgToUse)
              ├── 解析完成后 resolveComplete
              ├── attachmentStore.resolveMessageAssets(msg)  (附件解析)
              └── 消息列表重新渲染
    │
    ▼
chatStreamStore: activeStreamingIds 计算属性自动更新
    └── 新话题的 session key 无活跃流 → 返回空 Set
        （流式状态自然隔离，无需显式 reset）
```

> 相关源码：
> - `src/features/topic/TopicList.vue` 第 149–169 行
> - `src/core/stores/chatSessionStore.ts` 第 17–44 行
> - `src/features/chat/ChatView.vue` 第 62–81 行
> - `src/core/stores/chatHistoryStore.ts` 第 83–187 行

### 5.2 状态重置

话题切换时，以下状态被**显式重置**：

| 状态 | 重置点 | 说明 |
|------|--------|------|
| `currentChatHistory` | `loadHistory` (offset=0, index=0) | 清空旧消息数组 |
| `historyOffset` | `loadHistoryPaginated` | 重置为 0 |
| `hasMoreHistory` | `loadHistory` (offset=0, index=0) | 重置为 true |
| `showScrollToBottom` | `ChatView.vue` watcher | 隐藏置底按钮 |
| 滚动状态 | `resetChatScroll()` | 重置 IO 哨兵与 RAF 轮询 |
| 未读标记 | `markTopicAsRead(newTopicId)` | 清空当前话题未读数 |

**不**需要显式重置的状态：
- `chatStreamStore.activeStreamMessages`：按 `"itemId:topicId"` 键隔离，切换后 `activeStreamingIds` 自动为空
- `chatStreamStore.sessionActiveStreams`：同理，旧话题的流记录保留在 Map 中，不影响新会话

### 5.3 历史加载与缓存策略

#### 流式 Channel 加载

`loadHistory` 使用 Tauri v2 的 `Channel` 接收流式历史数据：

> 源码位置：`src/core/stores/chatHistoryStore.ts` 第 96–152 行

```ts
const channel = new Channel<HistoryChunk>();
channel.onmessage = (chunk) => {
  // 会话一致性校验
  if (sessionStore.currentTopicId !== requestedTopicId && requestedTopicId !== null) return;
  
  const activeMsg = streamStore.activeStreamMessages.get(chunk.message.id);
  const msgToUse = activeMsg || chunk.message; // Object Hydration
  
  currentChatHistory.value.push(msgToUse);
  // ...
};

await invoke('load_chat_history_streamed', {
  ownerId, ownerType, topicId, limit, offset, onMessage: channel,
});
```

#### Object Hydration（对象劫持）

这是话题切换时的**关键修复**：如果某条消息正在活跃生成中（存在于 `activeStreamMessages` 全局池），则从池中取出**活的响应式对象**替换 DB 拉回的骨架。这确保了用户切换话题后再切回，流式动画能无缝接续。

> 源码位置：`src/core/stores/chatHistoryStore.ts` 第 112–115 行

#### 分页策略

| 场景 | limit | offset | 说明 |
|------|-------|--------|------|
| 初始加载 | 15 | 0 | 首次进入话题，加载最近 15 条 |
| 话题切换 | 5 | 0 | `loadHistoryPaginated` 调用，轻量快速 |
| 加载更多 | 10 | historyOffset | 用户上拉加载旧消息 |

## 6. Hash 路由设计

### 6.1 路由表（仅 /chat）

> 文件位置：`src/core/router/index.ts`

```ts
const routes = [
  { path: '/', redirect: '/chat' },
  { path: '/chat', name: 'chat', component: ChatView },
];

export const router = createRouter({
  history: createWebHashHistory(),
  routes,
});
```

**路由表极简设计的原因**：

VCP Mobile 是**单页覆盖型应用**，绝大多数"导航"通过抽屉、BottomSheet、SlidePage 和 Teleport 实现。真实路由仅负责：
1. 启动时重定向到 `/chat`
2. 作为 ChatView 的挂载点
3. 与其他覆盖层协同时的路径校验（如 `useModalHistory` 中判断 `currentPath !== '/chat'`）

### 6.2 路由与覆盖层的分工

| 职责 | 路由（Vue Router） | 覆盖层（SlidePage / Drawer） |
|------|-------------------|------------------------------|
| **URL 变化** | 无（始终 `/#/chat`） | 无 |
| **历史记录** | 仅初始 `/chat` dummy state | Operation Aegis 管理 dummy state 栈 |
| **动画** | 无 | SlidePage 右入右出（cubic-bezier 缓动） |
| **层级** | 无 | 动态 z-index（40 + stackIndex） |
| **典型场景** | 应用启动、分享链接恢复 | 设置页、Agent 设置、同步会话 |

### 6.3 路由守卫与后置钩子

当前路由配置**无前置守卫**（`beforeEach`）。唯一的路由级干预是后置钩子：

```ts
router.afterEach(() => {
  initRootHistory();
});
```

**作用**：任何路由切换（包括重定向、参数变化、手动 `router.push`）完成后，自动检查并重新注入 Aegis 的 dummy state。这防止了以下边界情况：

- 用户通过分享链接进入 `#/chat` 时，dummy state 未初始化
- 路由跳转后浏览器历史栈状态不一致，导致返回键行为异常
- 真机调试时热重载导致 history state 丢失

> 源码位置：`src/App.vue` 第 207–214 行

## 8. 设计决策与注意事项

### 8.1 为何话题列表使用流式加载而非一次性返回？

`get_topics_streamed` 通过 `Channel` 分块推送话题数据，而非一次性返回完整数组。原因：

1. **首屏响应**：用户切换 Agent 后可立即看到前几条话题，无需等待全部加载
2. **大列表友好**：某些 Agent 可能拥有数百个话题，流式加载避免 JSON 序列化阻塞
3. **竞态防护**：`currentAgentId` 检查确保旧请求的结果不会覆盖新选中的 Agent 列表

> 源码位置：`src/core/stores/topicListManager.ts` 第 86–134 行

### 8.2 为何创建话题后需要 `topics.value = [...topics.value]`？

`useVirtualList` 来自 `@vueuse/core`，其对源数组的监听依赖 Vue 的响应式系统。由于话题操作（创建、删除、更新标题）使用 `unshift`、`splice` 或索引赋值，这些操作本身能触发响应式，但虚拟列表的**高度重计算**有时不会同步更新。强制重新赋值数组引用（`[...topics.value]`）可确保虚拟列表的 `totalHeight` 和 `scrollTo` 状态立即校准。

### 8.3 话题切换时为何不显式调用 `chatStreamStore.reset()`？

`chatStreamStore` 采用**会话键隔离**设计：`sessionActiveStreams` 的 key 为 `"${ownerId}:${topicId}"`。这意味着：

- 切换话题后，`activeStreamingIds` 计算属性自动返回新会话键对应的流集合（通常为空）
- 旧话题的流记录保留在 Map 中，用户切回时仍能恢复流式状态
- 全局池 `activeStreamMessages` 通过 `MAX_STREAM_MESSAGES` 和延迟清理机制控制内存上限

这种设计避免了"切换话题必须中断所有流"的粗暴策略，支持后台流式继续完成。

### 8.4 `Object Hydration` 的必要性

`load_chat_history_streamed` 从 DB 拉回的骨架消息对象**不是响应式的**。如果该消息正在流式生成中（存在于 `activeStreamMessages` Map），直接 push 骨架对象会导致：

1. 消息内容不随流式增量更新
2. `isThinking` 动画状态丢失
3. 同一条消息在列表中出现两个实例

通过 `activeMsg || chunk.message` 的劫持策略，DB 骨架被替换为全局池中**活的响应式对象**，实现无缝接续。

> 源码位置：`src/core/stores/chatHistoryStore.ts` 第 112–115 行

### 8.5 删除话题的双确认设计

话题删除采用**双确认对话框**：

```ts
if (window.confirm(`确定要删除话题 "${topic.name}" 吗？此操作不可逆转。`)) {
  if (window.confirm(`【最终确认】真的要永久删除 "${topic.name}" 吗？`)) {
    topicListStore.deleteTopic(...);
  }
}
```

这是**血训产物**：移动端误触率远高于桌面端，单次确认不足以阻止情绪化操作。双确认将误删概率降至可接受水平。

> 源码位置：`src/features/topic/TopicList.vue` 第 121–128 行
